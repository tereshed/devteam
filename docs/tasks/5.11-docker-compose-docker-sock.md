# 5.11 docker-compose: монтирование `/var/run/docker.sock`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** **`docker-compose.yml` в корне репозитория** (требование **`.cursor/rules/deploy.mdc`**, **`.cursor/rules/backend.mdc`** — единая точка входа вместе с **`Makefile`**); сервис **`app`** (контейнер бэкенда); при переносе из legacy-пути — **`Makefile`** (все вызовы **`docker-compose -f …`** на новый файл); при необходимости — **`README.md`** (локальная разработка, Docker Desktop / Linux, **`group_add`**).  
**Статус:** ✅ Выполнена

---

## Описание

Сервис **`DockerSandboxRunner`** (**5.5**) общается с Docker Engine через **HTTP API** к сокету демона. Когда API запускается **внутри контейнера** (**`docker-compose.yml`**, сервис **`app`**), у процесса по умолчанию **нет** доступа к сокету хоста — вызовы **`client.NewClientWithOpts(client.FromEnv)`** (или эквивалент) приводят к ошибкам подключения, и **sandbox-задачи не стартуют**.

**Цель задачи:** в **`docker-compose.yml`** для сервиса **`app`** явно смонтировать **сокет Docker хоста** в контейнер в режиме **чтения-записи** (**по умолчанию для bind-mount**, **без** суффикса **`:ro`**). Docker API для создания, запуска и остановки контейнеров **требует** возможности писать в unix-socket демона; монтирование **только для чтения** сокета **несовместимо** с мутирующими операциями и **не** допускается.

**Интерфейс локального запуска:** в репозитории окружение (в т.ч. сервис **`app`**) поднимают командой **`make up`** — **`Makefile`** задаёт вызов **`docker compose`** и путь к **`docker-compose.yml`**, что соответствует **`.cursor/rules/deploy.mdc`**. Проверку доступа к Docker Engine из контейнера бэкенда выполняют тем же способом.

**Границы задачи:** **compose в корне**, монтирование сокета, права на сокет в dev (**`group_add`** — см. ниже), минимальная документация в **`README`**. Если на момент PR файл ещё лежит в **`deployment/docker-compose.yaml`**, в рамках **5.11** необходимо **перенести** его в **`docker-compose.yml`** в корне и обновить **`Makefile`**, чтобы **`make up`**, миграции и остальные цели продолжали указывать на корректный **`-f`** — иначе нарушается **Compliance** по структуре репозитория. Не дублировать содержание **5.10** (env **`SANDBOX_*`**) — имена переменных остаются **единым источником правды** в коде конфига; в compose при необходимости **прокинуть** согласованные значения. Сборка образов sandbox — **5.12**; тесты раннера — **5.13**/**5.14**.

## Требования к `docker-compose.yml`

1. **Volume mount** для сервиса **`app`**:
   - **`/var/run/docker.sock:/var/run/docker.sock`** — **без** **`:ro`**; режим **rw** (дефолт bind-mount) **обязателен** для RPC к демону.
   - на **macOS / Windows** с Docker Desktop сокет обслуживается VM, но путь на **хосте**, с которого вызывают **`make up`**, тот же — **`/var/run/docker.sock`**; в файле compose монтирование **идентично** (одна фраза в **`README`** при необходимости).
2. **Права в контейнере (Linux, dev):** сокет на хосте часто принадлежит группе **`docker`** с **GID**, которого **нет** у пользователя процесса в образе **`app`** → **`permission denied`** при записи в сокет. **Рекомендуемый** подход: директива **`group_add`** в сервисе **`app`**, подставляя **GID группы `docker` с хоста** (например переменная окружения **`${DOCKER_GID}`**, которую разработчик задаёт один раз из **`getent group docker | cut -d: -f3`** или аналога). Так процесс **не** меняется через глобальное **`user:`**, что снижает риск сломать права на рабочую директорию, логи и временные файлы внутри контейнера. Жёсткий **`user:`** в compose — **не** предпочтительный дефолт для **5.11**; если в PR выбран **`user:`**, нужно **явно** обосновать влияние на файловую систему контейнера и права на **`/root/db/migrations`** и прочие mount’ы.
3. **Не** монтировать весь **`/var/run`** без необходимости — только **`docker.sock`**, чтобы сузить поверхность атаки.
4. Сохранить существующие **`volumes`** у **`app`** (например миграции **`../backend/db/migrations`** или путь после переноса compose — согласовать с фактической структурой репозитория) — сокет добавить **отдельной** строкой.

## Безопасность (обязательное предупреждение)

Монтирование **`docker.sock`** даёт контейнеру **`app`** по сути **root на хосте** (через API можно поднять привилегированный контейнер, смонтировать диск и т.д.). Для **локальной разработки** это приемлемый компромисс; для **production** compose-деплой с сокетом **не** является рекомендуемым паттерном без сетевой изоляции, auth proxy (**Docker over TCP + TLS**), Kubernetes, Nomad и т.п.

В **`README`** или в комментарии в **`docker-compose.yml`** (кратко): **«только для dev; в prod — отдельная стратегия доступа к Docker»**.

## Переменные окружения (опционально)

Если команда уже использует **`DOCKER_HOST`** для удалённого демона, **не** ломать сценарий: либо **не** задавать **`DOCKER_HOST`** в compose (дефолт — смонтированный socket), либо явно задокументировать приоритет. Конфликт «смонтирован socket + **`DOCKER_HOST=tcp://...`**» — описать ожидаемое поведение в PR.
