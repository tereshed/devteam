## Описание

Создать репозиторий для CRUD-операций с таблицей `conversations` (чаты пользователя с оркестратором в рамках проекта). Слой только для БД: интерфейс + приватная реализация, `context.Context`, GORM, sentinel-ошибки, безопасная пагинация и фильтрация. 

**Важно:** Загрузка сообщений чата (`conversation_messages`) выносится в **задачу 8.2**. Репозиторий должен поддерживать транзакции для атомарного создания чата и первого сообщения на уровне сервиса.

---

## Интерфейс

```go
type ConversationRepository interface {
    // WithTx возвращает копию репозитория, использующую переданную транзакцию
    WithTx(tx *gorm.DB) ConversationRepository

    Create(ctx context.Context, conv *models.Conversation) error
    
    // GetByID требует projectID для защиты от IDOR (Tenant Isolation)
    GetByID(ctx context.Context, projectID, id uuid.UUID) (*models.Conversation, error)
    
    ListByProjectID(ctx context.Context, projectID uuid.UUID, filter ConversationFilter) ([]*models.Conversation, int64, error)
    
    // Update использует Patch-семантику (только измененные поля) для предотвращения Race Conditions
    Update(ctx context.Context, projectID, id uuid.UUID, updates map[string]interface{}) error
    
    Delete(ctx context.Context, projectID, id uuid.UUID) error
}
```

## Методы и Краевые случаи

**Общее правило (Early Return):** Все методы должны возвращать ошибку (например, `ErrInvalidInput` или `ErrConversationNotFound`), если на вход передан `nil` указатель или `uuid.Nil`, чтобы не делать бессмысленных запросов в БД.

### Create

Вставляет строку в `conversations`. Обязательные поля: `project_id`, `user_id`.
- **Ошибки БД:** Проверять `SQLState` (код `23503` для `foreign_key_violation`) и имя констрейнта (`ConstraintName`), чтобы точно мапить ошибку в `ErrProjectNotFound` или `ErrUserNotFound` (переиспользовать из общего файла ошибок репозитория).
- Ошибки `CHECK` констрейнтов из БД (например, невалидный статус) должны перехватываться и мапиться в доменную `ErrInvalidConversationStatus`, чтобы не допускать утечки `pgconn.PgError` в слой сервиса.

### GetByID

Возвращает одну запись по `id` и `project_id` (защита от чтения чужих чатов).
- Возвращает `ErrConversationNotFound`, если записи нет (`gorm.ErrRecordNotFound`).

### ListByProjectID

Список разговоров проекта с пагинацией и фильтрами.
- **Пагинация:** `Count` + `Find` с `Limit`/`Offset`. 
  - *Примечание по производительности:* Если задан `Search`, точный `Count` может быть медленным. В будущем запланировать переход на Keyset Pagination (Cursor-based: `WHERE updated_at < ?`). Оставить `// TODO: migrate to cursor pagination`.
- **Возвращаемый тип:** `[]*models.Conversation` (указатели) для консистентности с `GetByID` и снижения аллокаций, если это соответствует конвенциям других репозиториев проекта.

### Update

Частичное обновление через `gorm.DB.Updates()` с передачей `map[string]interface{}`.
- **Никакого `gorm.Save`**, чтобы избежать Lost Update при конкурентных запросах.
- **Проверка RowsAffected:** Если `err == nil`, но `RowsAffected == 0`, метод обязан вернуть `ErrConversationNotFound`.

### Delete

Удаление строки `conversations` (с учетом `project_id`).
- **Проверка RowsAffected:** Аналогично `Update`, возвращать `ErrConversationNotFound` при `RowsAffected == 0`.
- **Soft/Hard Delete:** Уточнить по модели. Если используется `gorm.DeletedAt` (Soft Delete), каскадное удаление сообщений на уровне БД не сработает, это нужно будет обрабатывать на уровне приложения.

## Фильтры и пагинация

```go
type ConversationFilter struct {
    UserID   *uuid.UUID
    Status   *models.ConversationStatus
    Search   *string
    Limit    int
    Offset   int
    OrderBy  string
    OrderDir string
}
```

### Нормализация и Безопасность

- **Limit и Offset:** `Limit <= 0` → дефолт (например, 20), верхняя граница (100). `Offset < 0` → `0` (защита от синтаксических ошибок БД).
- **OrderBy Whitelist:** Использовать константы пакета (например, `const OrderByCreatedAt = "created_at"`). Никакой прямой подстановки строк от пользователя.
- **Search (ILIKE):** Экранирование спецсимволов (`%`, `_`, `\`) должно выполняться через общую утилиту (например, `database.EscapeLike`), чтобы избежать SQL Injection и ReDoS. Ограничить длину строки поиска.

## Конвенции реализации

1. **GORM Scopes:** Вместо лапши из `if` в методе `ListByProjectID`, вынести фильтры в переиспользуемые скоупы:
   ```go
   func FilterByStatus(status *models.ConversationStatus) func(db *gorm.DB) *gorm.DB
   ```
2. **Централизация ошибок:** Все sentinel-ошибки (`ErrConversationNotFound`, `ErrInvalidInput`) должны лежать в общем файле пакета `repository` (например, `errors.go`), а не дублироваться.
3. Во всех запросах использовать `r.db.WithContext(ctx)`.

## Тестирование

**Тег:** `//go:build integration`

### Изоляция и Стабильность (Flaky Tests)
- **Изоляция:** Каждый тест-кейс должен создавать *уникального* пользователя и проект, либо использовать откат транзакции (`tx.Rollback()`), чтобы параллельные тесты не влияли на `Count` друг друга.
- **Время:** При сравнении `created_at`/`updated_at` использовать `assert.WithinDuration(t, expected, actual, time.Millisecond)` или `.Truncate(time.Microsecond)`, так как Postgres обрезает наносекунды.

## Критерии приёмки

- [ ] Интерфейс поддерживает транзакции (`WithTx`) и защиту от IDOR (`projectID` в методах).
- [ ] `Update` использует `map[string]interface{}` (Patch) и проверяет `RowsAffected`.
- [ ] Быстрый возврат (early return) при `nil` или `uuid.Nil`.
- [ ] Ошибки БД (FK, CHECK) корректно мапятся в доменные ошибки (без утечки `pgconn.PgError`).
- [ ] `OrderBy` использует whitelist-константы, `Offset` защищен от отрицательных значений.
- [ ] Интеграционные тесты изолированы, не имеют состояния гонки и учитывают потерю точности времени.
