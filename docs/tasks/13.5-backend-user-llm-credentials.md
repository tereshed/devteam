# Backend: пользовательские LLM API keys (глобальные настройки)

**Sprint:** 13 (связка с фронтом)  
**Статус:** реализовано (backend)  
**Блокирует:** фронтенд [13.5-global-settings-screen.md](./13.5-global-settings-screen.md) (режим **A**).

**Этот файл — SSOT** для имён полей DTO (`set` / `clear`), формата `masked_preview`, лимитов, схемы БД и поведения ошибок/идемпотентности. Фронт **13.5** в режиме **A** обязан следовать таблицам и примерам ниже без дублирующих договорённостей.

---

## Цель

Дать аутентифицированному пользователю возможность хранить **свои** API-ключи для LLM-провайдеров (перекрытие/дополнение к серверным ключам из env — приоритет разрешения в рантайме **фиксируется отдельно** при интеграции с оркестратором; в рамках этой задачи достаточно CRUD «на пользователя» и безопасного хранения).

**Не входит в скоуп:** выбор `DefaultProvider`, вызовы LLM с подстановкой user-key (отдельные задачи на consumer), админские ручки `/api/v1/llm/*` (они остаются за `AdminOnlyMiddleware`), **`base_url`** в DTO (см. ниже — вынесено из MVP).

---

## Контракт API

**Канонический путь (единственный):** `GET` и `PATCH` **`/api/v1/me/llm-credentials`**. Дубликатов путей не вводить.

### Рассмотренные альтернативы (и отклонённые)

| Альтернатива | Почему не выбрана |
|--------------|------------------|
| **`PATCH /api/v1/me/llm-credentials/{provider}`** — один провайдер за запрос | Проще аудит «один запрос — одно действие» и не нужна мультистроковая транзакция, но **фронт 13.5** зафиксирован на **один** глобальный **Save** с несколькими карточками в одном теле; второй контракт раздвоил бы SSOT. |
| Оба контракта вместе: batch на **`/api/v1/me/llm-credentials`** и per-provider **`PATCH /api/v1/me/llm-credentials/{provider}`** | Дублирование поверхностей API без выигрыша для MVP. |

Текущий выбор: **batch-`PATCH`** + одна SQL-транзакция на запрос (см. ниже).

### Авторизация и IDOR (обязательная формулировка)

- **`user_id` для всех операций берётся только из JWT** (claim `sub` / тот же идентификатор, что уже кладётся в `gin.Context` после `AuthMiddleware`). **Запрещено:** читать `user_id` из тела запроса, path, query — даже если поле присутствует, оно **игнорируется** и не влияет на выбор строк БД.
- Токен пользователя **A** не должен иметь доступа к данным пользователя **B** при любой манипуляции телом; поведение при «подсунутом» `user_id` в JSON — то же, что при его отсутствии (работа только от JWT). **Тест:** см. раздел «Обязательные сценарии тестов».

### Лимиты размера (DoS и парсер)

Коды HTTP для лимитов **зафиксированы** (как в существующих ручках: `http.MaxBytesReader` + разбор `*http.MaxBytesError` → **`413`**, см. прецедент `backend/internal/handler/conversation_handler.go`).

| Что | Значение | HTTP при нарушении |
|-----|----------|---------------------|
| Максимальный размер **всего** тела **`PATCH`** | **8 KiB** (`http.MaxBytesReader` до `json.Decoder`, тот же паттерн, что в чатах — только другой лимит) | **`413 Request Entity Too Large`** |
| Максимальная длина **одного** значения `*_api_key` после **`trim()`** | **4096** символов (рун в UTF-8) | **`400 Bad Request`** |
| Минимальная длина нового ключа после **`trim()`** | **16** символов | **`400 Bad Request`** (короткий ключ + маска `****` + 4 символа раскрывает слишком большую долю секрета) |

## SSOT: имена полей `set` и `clear` (DTO)

Паттерн сброса: **`clear_<id>_key`** (булево). Установка: **`<имя>_api_key`** (строка). Исключение по имени: **`openrouter`** → поле **`openrouter_api_key`** (не `openrouter_key`).

| `id` (провайдер) | Поле установки ключа (`PATCH` body) | Поле сброса (`PATCH` body) |
|------------------|-------------------------------------|----------------------------|
| `openai` | `openai_api_key` | `clear_openai_key` |
| `anthropic` | `anthropic_api_key` | `clear_anthropic_key` |
| `gemini` | `gemini_api_key` | `clear_gemini_key` |
| `deepseek` | `deepseek_api_key` | `clear_deepseek_key` |
| `qwen` | `qwen_api_key` | `clear_qwen_key` |
| `openrouter` | `openrouter_api_key` | `clear_openrouter_key` |

**Конфликт в одном запросе:** для одного и того же `id` одновременно **`clear_<id>_key: true`** и непустое **`*_api_key`** для этого `id` → **`400`**, тело без утечки секретов. Обязательный тест — в чек-листе ниже.

**Семантика пустого `*_api_key`:** поле отсутствует или после `trim()` пустая строка → сохранённый секрет для этого провайдера **не меняется** (как во фронт-спеке **13.5**, раздел «SSOT: пустое поле ключа и сброс»).

**Идемпотентность `clear_*`:** `clear_<id>_key: true`, а ключа уже нет → **`200`**, no-op (не **`404`**), без ошибки — чтобы повторная отправка с клиента не ломала UX.

**Идемпотентность пустого `PATCH`:** `{}` или тело без изменений → **`200`**, данные не портятся.

**Конкуренция:** параллельные **`PATCH`** с двух клиентов без **`If-Match`** — допустима модель **last write wins** по завершении транзакции; недопустимы порча строки или «рваное» состояние между провайдерами внутри **одного** запроса (см. транзакции).

## SSOT: `masked_preview` (без `has_key`)

- В ответе **`GET`** и в теле **`PATCH` 200** используется **только** поле **`masked_preview`** на каждый провайдер. Поля **`has_key` в DTO нет** — дублирование семантики запрещено.
- **`masked_preview === null`** ⇔ для этого провайдера **нет** сохранённого ключа.
- Если ключ есть — **`masked_preview`** — **строго** строка длины **8**: префикс ровно **`****`** (четыре звёздочки) + **ровно последние 4 символа** ключа в исходном виде (после того как ключ принят сервером), без пробелов и без эллипсиса. Примеры: `****wxyz`, `****_7Ab` (если последние символы таковы). Фронт может опираться на длину 8 и префикс `****`.
- **Полный ключ никогда** не возвращается в JSON.

**`base_url` в MVP не входит в DTO** (ни `GET`, ни `PATCH`): исключает смешение «override пользователя» и «дефолт сервера» и утечку неоднозначной конфигурации. Отдельный endpoint/поле — только в следующей задаче с явным контрактом.

## Примеры JSON (нормативные)

### `GET /api/v1/me/llm-credentials` — **200 OK**

Тело — объект с **ровно шестью** ключами верхнего уровня — идентификаторы провайдеров; значения — объекты с единственным полем **`masked_preview`**.

```json
{
  "openai": { "masked_preview": "****3Mk9" },
  "anthropic": { "masked_preview": null },
  "gemini": { "masked_preview": null },
  "deepseek": { "masked_preview": "****xY12" },
  "qwen": { "masked_preview": null },
  "openrouter": { "masked_preview": null }
}
```

### `PATCH /api/v1/me/llm-credentials` — **тело запроса**

Частичное обновление; допустимы только ключи из таблицы SSOT выше + любые игнорируемые неизвестные поля **не** добавлять (неизвестные ключи → **`400`** для предсказуемости).

Пример: обновить OpenAI, сбросить Anthropic, остальное не трогать.

```json
{
  "openai_api_key": "sk-proj-................................",
  "clear_anthropic_key": true
}
```

### `PATCH` — **200 OK** (тело = актуальное состояние, тот же контракт, что `GET`)

После успешного применения примера выше:

```json
{
  "openai": { "masked_preview": "****3Mk9" },
  "anthropic": { "masked_preview": null },
  "gemini": { "masked_preview": null },
  "deepseek": { "masked_preview": "****xY12" },
  "qwen": { "masked_preview": null },
  "openrouter": { "masked_preview": null }
}
```

(значения масок — иллюстративные; сервер пересчитывает маску из сохранённого ключа.)

### `PATCH` — **400** (конфликт clear + set)

```json
{
  "openai_api_key": "sk-new-key-min-16-chars-here",
  "clear_openai_key": true
}
```

→ **`400`**, данные не менять.

## Транзакционность

Один запрос **`PATCH`** может затрагивать несколько провайдеров. Все изменения (удаления по `clear_*`, upsert зашифрованного blob по `*_api_key`) выполняются в **одной** SQL-транзакции: либо фиксируется весь набор, либо откат целиком при любой ошибке.

---

## Хранение (зафиксированная схема)

**Выбор:** таблица **`user_llm_credentials`** (имя можно уточнить в миграции, смысл — ниже):

- **`id`** `UUID` PK, `gen_random_uuid()`.
- **`user_id`** `UUID` NOT NULL, FK на `users(id)` **`ON DELETE CASCADE`** (при удалении пользователя строки credentials исчезают вместе с пользователем — требование GDPR/«право быть забытым» закрывается на уровне схемы; при soft-delete пользователя — отдельная явная очистка этой таблицы в сервисе удаления пользователя, если она предусмотрена в продукте).
- **`provider`** `VARCHAR` NOT NULL с **`CHECK (provider IN ('openai','anthropic','gemini','deepseek','qwen','openrouter'))`** (или доменный `ENUM` в Postgres/Yugabyte, если в проекте уже принят такой стиль для аналогичных полей). Дублирует валидацию сервиса и защищает от багов/инъекций в сырой SQL. Расширение списка провайдеров = новая миграция (`CHECK`/`ENUM` + код).
- **`encrypted_key`** `BYTEA` NOT NULL — ciphertext в формате **4.7** (version byte + nonce + sealed).
- **`created_at` / `updated_at`** — как в остальных таблицах проекта.
- **UNIQUE(`user_id`, `provider`)** — одна строка на пару; расширение на динамический список провайдеров = новые значения `provider` + миграция enum/check, без JSONB «все ключи в одной клетке».

**Почему не single-row+jsonb для всех секретов:** перезапись всего JSON при каждом `PATCH`, тяжёлый AAD, миграции при добавлении провайдера; строка на `(user_id, provider)` даёт естественный **AAD = `[]byte(rowUUID.String())`** по правилу [4.7-encrypt-credentials.md](./4.7-encrypt-credentials.md) (строковое представление UUID PK строки), одинаково при Encrypt/Decrypt.

**Ротация мастер-ключа (`ENCRYPTION_KEY`):** наследуется **байт версии формата** внутри blob (**4.7**, сейчас `0x01`). Новые записи пишутся текущей версией; при введении `0x02` в будущем — отдельная задача на миграцию данных и маппинг ключей; эта задача **не** вводит второй формат, только использует существующий `Encryptor`.

**Аудит (обязательно):** отдельная таблица **`user_llm_credential_audit`** (или эквивалент с префиксом проекта): **`user_id`**, **`provider`**, **`action`** (`set` | `clear`), **`created_at`**, **`ip`**, **`user_agent`** (как принято в проекте для HTTP-логов). **Без** хранения ключа или маски.

**Транзакционность аудита (MVP):** записи аудита на каждый успешный `set`/`clear` вносятся **в той же SQL-транзакции**, что и изменения `user_llm_credentials` (вставки в audit — дешёвые; commit один). Так нет окна «основной commit прошёл, процесс упал до audit». Отдельный **transactional outbox** в кодовой базе пока не является стандартом (см. TODO в других сервисах); если позже появится outbox для событий — пересмотр в отдельной задаче, **эта** спека требует согласованного commit для MVP.

**Логи приложения:** не логировать сырое тело **`PATCH`** и не логировать значения `*_api_key`. В структурных логах — `user_id`, `provider`, результат операции (`ok` / код ошибки).

## Rate limiting

**PATCH** `/api/v1/me/llm-credentials` — чувствительная поверхность. Зафиксировать лимит на уровне middleware или reverse proxy (ориентир: **30 запросов / минуту / пользователь** по `user_id` из JWT; при превышении **`429`** с заголовком `Retry-After` при наличии поддержки). Точная реализация — в коде, но в спеке лимит обязан быть не «тишина».

## Ошибки (кратко)

- **`401`** — нет/просрочен JWT.
- **`400`** — невалидный JSON, неизвестное поле верхнего уровня, конфликт `clear_*` + `*_api_key`, длина одного ключа **> 4096** или **< 16** символов после `trim()`, прочие ошибки валидации DTO.
- **`413`** — размер **всего** тела **`PATCH`** превысил **8 KiB** (типично `*http.MaxBytesError` при декодировании), в духе `conversation_handler`.
- **`429`** — rate limit.
- **`500`** — БД/шифрование; без stack trace в теле.

## Архитектура (Clean Architecture)

Поток: **`handler` → `service` → `repository`**. Handler не ходит в GORM напрямую; `Encryptor` внедряется в `service`.

| Слой | Ответственность |
|------|-----------------|
| `dto` | Запрос/ответ; теги `json` + Swagger |
| `handler` | Gin, **только** `user_id` из JWT, лимит размера тела, маппинг ошибок |
| `service` | Валидация, маска, шифрование, транзакция, аудит |
| `repository` | CRUD по `user_llm_credentials`, без HTTP |

**DI:** `cmd/api/main.go`; миграция — `make migrate-create` / цикл **`migrate-up` → `migrate-down` → `migrate-up`** локально.

## Goose `Down` и продакшен

Секция **`Down` обязательна** для локального цикла миграций. **`Down` в продакшене с `DROP TABLE` уничтожает пользовательские секреты** — не использовать как «откат релиза» без бэкапа и playbook. В документации PR для этой задачи: откат прод-схемы только через форвард-миграцию или восстановление из бэкапа, а не слепой `goose down` на живых данных.

## Swagger

- После реализации — **`make swagger`**.
- Для **`PATCH`:** в аннотациях указать **`@Failure 413`** (превышение размера тела) наряду с **`@Failure 400`**, чтобы фронт и SDK видели детерминированный контракт.
- Поля `*_api_key` в схеме запроса **`PATCH`:** пометить как **write-only** / **`format: password`** (или эквивалент в swag), чтобы автоген SDK и примеры не поощряли логирование «ответов» с секретом.

## MCP (жёсткое переопределение правила репозитория)

Для **`GET`/`PATCH` `/api/v1/me/llm-credentials`** **MCP-инструменты не регистрируются** — **никаких исключений** в рамках этой задачи и без «security review открывает дверь» в тексте спеки. Общее правило CLAUDE.md про MCP для новых ручек **здесь явно переопределено**: контур пользовательских LLM-секретов в MCP **не** попадает. Если когда-либо понадобится — **новая** задача + отдельный ADR/security review, не оговорка в текущем документе.

## Обязательные сценарии тестов (чек-лист)

| # | Сценарий |
|---|----------|
| 1 | **IDOR / изоляция по JWT:** подготовить пользователей **A** и **B**. Под токеном **A** выполнить успешный **`PATCH`** с установкой ключа хотя бы для одного провайдера; в БД в `user_llm_credentials` появляются строки **только** с `user_id = A` (assert по SQL или репозиторию). Под токеном **B** выполнить **`GET`**: все шесть `masked_preview` — **`null`** (или только данные **B**, но **никогда** не маски/ключи **A**). Дополнительно: в репозитории запросы строятся с **`WHERE user_id = $jwt_user_id`**, без подстановки идентификатора из тела/path/query. *(Лишний ключ в JSON, не входящий в whitelist DTO, по правилам этой спеки даёт **400** до «маскарада» под user_id — отдельный вектор; основной интеграционный тест — изоляция A/B.)* Если в будущем появится admin-bypass или service-token — эта ручка **не** должна их уважать без отдельной спеки. |
| 2 | **GET:** в JSON **нет** подстроки, совпадающей с сохранённым сырым ключом (golden/assert на отсутствие). |
| 3 | **Логи:** тест с перехватом логгера — после `PATCH` с известным ключом в логах **нет** этого ключа как подстроки. |
| 4 | **`clear_*`** при уже отсутствующем ключе — **`200`**, no-op. |
| 5 | **`clear_*` + `*_api_key`** для одного `id` — **`400`**, данные не меняются. |
| 6 | Тело **`PATCH`** > **8 KiB** — **`413`**, без panic. |
| 7 | Одно поле `*_api_key` > **4096** символов (тело в пределах лимита) — **`400`**, без panic. |
| 8 | Ключ короче **16** символов после `trim()` — **`400`**. |
| 9 | **Integration (по тегам проекта):** подмена ciphertext между двумя строками с разным `id` UUID → дешифрование **не** возвращает чужой plaintext (ошибка `Decrypt`). |
| 10 | Два последовательных успешных **PATCH** с разными ключами одного провайдера — финальное состояние соответствует второму; нет corrupt blob. |
| 11 | Один **PATCH** с двумя провайдерами; искусственный сбой после первого update в моке транзакции — откат, второй провайдер не «застревает» в полуприменённом виде. |
| 12 | Аудит: после успешного `set`/`clear` в audit-таблице есть строка с верным `user_id`, `provider`, `action` без секрета; запись в той же транзакции, что и изменение credentials (см. раздел «Аудит»). |

Плюс юнит-тесты сервиса с моками репозитория и encryptor по существующим правилам проекта.
