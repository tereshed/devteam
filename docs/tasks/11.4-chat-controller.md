## Описание

Presentation-слой: **один** `AsyncNotifier` (class-based `@riverpod`, как `CreateProjectController`) на пару **`projectId` + `conversationId`**, который:

1. **Загружает** метаданные чата и историю сообщений (см. § **Метаданные чата vs сообщения**).
2. **Загружает страницы** через `ConversationRepository.getMessages` (первая страница + **подгрузка старых** — offset-пагинация).
3. **Отправляет** сообщения с **`generateClientMessageId()`**, идемпотентный **ретрай** при транзиентах (§ **Идемпотентный ретрай**), обрабатывает **`SendMessageResult`** через **upsert** и пересортировку (§ ниже).
4. **Готовит контракт для стрима** реалтайма: публичный API/состояние, в которое **11.9** подставит подписку на `WebSocketService` **без** дублирования бизнес-логики в виджетах.

**Вне скоупа 11.4:** полноценная реалтайм-подписка и разбор `WsServerEvent` — **11.9**; разметка UI, markdown, поле ввода — **11.5** / **11.6** / **11.8**.

## Провайдер (Riverpod / codegen)

- **Класс:** **`@riverpod`** + **`class ChatController extends _$ChatController`** — базовый класс из codegen наследует **`AsyncNotifier<ChatState>`** (проверить в **`chat_controller.g.dart`** после генерации). Явная форма для чтения джуном:

```dart
@riverpod
class ChatController extends _$ChatController {
  @override
  FutureOr<ChatState> build({
    required String projectId,
    required String conversationId,
  }) {
    // см. § «Контракт build» ниже — не обязательно один большой await «всё сразу»
  }
}
```

Точная сигнатура **`build`** (именованные параметры family, **`FutureOr` vs `Future`**) — **как сгенерирует** `riverpod_generator`; при расхождении с примером править аннотацию и перегенерировать.

- **После codegen** имя провайдера — в **`chat_controller.g.dart`** (ожидаемо **`chatControllerProvider`**).
- **Family:** именованные параметры **`projectId`** и **`conversationId`** (**`String`**, UUID в строковом виде). Вызов из **11.5**: **`chatControllerProvider(projectId: …, conversationId: …)`** — порядок и имена параметров **строго** как в сгенерированном API.
- **Пустые family-параметры:** **`projectId.isEmpty` или `conversationId.isEmpty`** — в **`_loadInitial`** (или в самом начале **`build`** после валидации) установить **`AsyncError`** / **`ArgumentError`** без вызова репозитория (**11.3** тоже кинет **`ArgumentError`**, но позже). Начальный **`ChatState`** может успеть вернуться «успешным» на один кадр — допустимо; **11.5** показывает ошибку из **`AsyncValue`**. Роутер не должен открывать чат с пустыми id — граничный случай.

### Контракт **`build`** и § **Частичный state**

Паттерн **`Future<ChatState> build() async { final c = await getConversation(...); final m = await getMessages(...); return ChatState(...); }`** — **несовместим** с поэтапным AppBar: до завершения **`await`** провайдер в **`AsyncLoading`**, **`conversation`** недоступна для заголовка.

**Зафиксировать:** **`build`** возвращает **первичный** **`ChatState`** **синхронно** или через **`Future.value(ChatState(...))`** «мгновенно»: например **`conversation: null`**, **`messages: []`**, **`isLoadingInitial: true`**. Затем **`_loadInitial()`** (или эквивалент) запускается **после того, как `build` уже вернул значение** — иначе при **`unawaited(_loadInitial())` до `return`** первый **`await`** внутри **`_loadInitial`** может вызвать присвоение **`state = …`** до завершения инициализации **`AsyncNotifier`** в Riverpod (**`StateError`** / неопределённое поведение в зависимости от версии).

**Корректный старт фона:** **`Future.microtask(() { ... })`** или **`scheduleMicrotask(() { ... })`** (или **`addPostFrameCallback`** — на усмотрение PR). Внутри замыкания — **`unawaited(_loadInitial());`**, если **`_loadInitial`** возвращает **`Future<void>`** (tear-off **`Future.microtask(_loadInitial)`** годится только если **`_loadInitial`** — синхронная **`void`-функция**). **Не** запускать **`unawaited(_loadInitial())` непосредственно перед `return`** из **`build`**. Эквивалентно: **`return ChatState.initial();`** первым, затем в микротаске — загрузка и обновления **`state`** через **`copyWith`**. Так выполняется § **Частичный state**. Альтернатива с обязательным **`await`** всей цепочки в **`build`** отклоняет частичный UX — не использовать без явного решения «без AppBar до полной загрузки».

- **`keepAlive`:** в **11.4** включено **`@Riverpod(keepAlive: true)`**, чтобы при обращении только к **`.notifier`** (типичные unit-тесты и вызовы без `watch` на экране) не срабатывал **autoDispose** до завершения **`_loadInitial`** и не терялись **`pendingByClientId`**. Экран **11.5** по-прежнему должен **`watch`** провайдер для ребилда UI.

## Отмена запросов и `ConversationCancelledException`

В коде репозитория уже маппится отмена Dio → **`ConversationCancelledException`** (через **`parseDioApiError`**, **`isCancellation`** — см. **`ConversationRepository._handleDioError`**). ТЗ **11.3** синхронизировано: строка в таблице `_handleDioError` (§ ниже в **11.3**). Запросы **истории** отменяются **`_historyCancelToken`** (§ **`CancelToken`**); отмена **`sendMessage`** — только своим токеном / dispose, см. тот же §.

**В `ChatController`:** при **`ConversationCancelledException`** при загрузке/пагинации **не** считать фатальный сбой API; типичный случай — dispose / отмена токена. Не применять результат запроса к state, если контроллер уже отменил запрос (эпоха / токен). Для **send** после отмены — снять «ожидание» отправки, не показывать как сетевую ошибку (ориентир — как обработка **`ProjectCancelledException`** в **`CreateProjectController`**).

## Метаданные чата vs сообщения (AppBar в 11.5)

**Рекомендуемый вариант (а):** в **`build`** / первичной загрузке получить **`ConversationModel`** и первую страницу **`getMessages`** одним из согласованных ниже способов.

**Ошибки и параллельный запуск (зафиксировать один подход в PR):**

Семантика **`Future.wait`** критична — **нельзя** писать «после **`await Future.wait`** вызвать **`_historyCancelToken.cancel()`**, чтобы отменить второй запрос»: при **`eagerError: false`** (значение по умолчанию) **`Future.wait` ждёт завершения обоих** future; к моменту возврата оба HTTP уже завершились — **cancel бессилен** (no-op).

- **Вариант A — `Future.wait(..., eagerError: false)` без cancel после wait:** дождаться обоих; если есть ошибка — разобрать оба результата (например, не показывать **conversation** без успешных сообщений, или политика merge при частичном успехе — зафиксировать в PR). **Дополнительный `cancel` после `await` не нужен.**

- **Вариант B — `Future.wait(..., eagerError: true)` + cancel при первой ошибке:** `Future.wait` завершается с **первой** ошибкой, при этом второй future может **ещё выполняться** — в обработчике первой ошибки (или `whenComplete`) вызвать **`_historyCancelToken.cancel()`**, чтобы прервать второй inflight. Минус: успешный ответ второго запроса может прийти до отмены — игнорировать по **эпохе** / токену.

- **Вариант C — без `Future.wait`:** запустить оба future вручную (`unawaited` / два `await` в разной форме), при первом фейле — **`_historyCancelToken.cancel()`** и не применять устаревшие ответы.

**Последовательно:** **`getConversation` → `getMessages`** — проще всего, без гонки «двух параллельных»; заголовок раньше (**§ Частичный state**); лишний RTT допустим.

**Частичный state между запросами:** после успешного **`getConversation`** положить в **`ChatState`** **`conversation != null`**, **`messages`** — пустой список (или без изменений до merge), **`isLoadingInitial == true`** до завершения **`getMessages`**. AppBar (**11.5**) может показать заголовок сразу; область списка — индикатор загрузки. После первой страницы сообщений — **`isLoadingInitial = false`**. Итог: **`AsyncNotifier`** может оставаться в **`AsyncData`**, пока **`ChatState.isLoadingInitial`** отражает догрузку сообщений — не обнулять **`conversation`** целиком **`AsyncLoading`**, если метаданные уже есть (**избегать пустого AppBar** при доступном названии). Реализация см. § **Контракт build** в разделе **Провайдер**.

**Альтернатива (б):** отдельный **`conversationProvider(conversationId)`** (`FutureProvider`) только для метаданных; **`ChatController`** — только сообщения. Допустимо с явным обоснованием в PR.

**Поле `total` из `MessageListResponse`:** в **11.4** **не** хранить и не показывать; если **11.5** захочет «X сообщений» — расширение state отдельным пунктом/задачей по требованию UI.

|---------------|--------|

Канонический порядок — § **upsert / сортировка**; для **`ListView`** — переворот на presentation-слое. **`List`** сравнивает по identity при **`==`** в Riverpod — лишние перерисовки возможны; **`IList`** из **`fast_immutable_collections`** (если пакет есть в проекте) или неизменяемый список с предсказуемым **`==`** — улучшение на усмотрение PR; «умный» diff — **отдельная задача** при необходимости |
| `isLoadingInitial` / `isLoadingOlder` | Разделить первый запрос и подгрузку старых |
| `hasMoreOlder` | См. § **Пустая страница и hasNext** — не слепо с сервера |
| `olderOffset` | Смещение для следующего **`getMessages`**; см. § **`olderOffset`** |
| **`sessionEpoch`** | **`int`**, **приватное поле `ChatController`**, не в **`ChatState`**: инкремент в **`refresh()`** (и при смене «сессии» загрузки по политике PR). Каждый ответ **`getConversation` / `getMessages` / `loadOlder`** помечается ожидаемой эпохой; если **`sessionEpoch` ответа ≠ текущему** — **не** мержить в state (защита от ответа «в полёте» после **`cancel`**). **Альтернатива** — полагаться **только** на **`ConversationCancelledException`** и отбрасывать результат; для граничных случаев комбинация **эпоха + cancel** надёжнее. Новый инстанс Notifier после dispose начинает с **`sessionEpoch == 0`** — допустимо |
| **`pendingByClientId`** (`Map<String, PendingMessage>`, имя на усмотрение PR) | Идемпотентный ретрай (§ **Идемпотентный ретрай**): ключ — **`clientMessageId`**. **Хранить в `ChatState`**, не в приватном поле **`Notifier`** — переживает hot reload, доступно тестам и UI (**«Повторить»**). Очистка записи: успешный **created/duplicate**, явная отмена пользователем, логическая ошибка с переходом на новое сообщение (**новый** **`generateClientMessageId()`**) |
| **`sendError`** (опционально) | Ошибки **без** привязки к конкретному **`clientMessageId`**: валидация пустого ввода **до** генерации id, системные сбои «не относится ни к одному pending». **Per-message** ошибки (**502**, timeout) — в **`PendingMessage.lastError`** (**§ Минимальные поля `PendingMessage`**). Если **`lastError`** в pending используется полностью — поле **`sendError`** можно **не** вводить, чтобы не дублировать SnackBar и строку pending |

**`ChatState` (если выносится в Freezed):** **`abstract class ChatState with _$ChatState`** — обязательный синтаксис (**CLAUDE.md**, Frontend / Freezed). **`class ChatState` без `abstract`** — **запрещено**. Для методов/getter поверх сгенерированных полей — **`const ChatState._();`** в стиле **11.1** (без приватного конструктора нельзя добавить пользовательские члены в Freezed v3).

**Параллельные отправки:** **не** использовать глобальный **`_inFlight`**, блокирующий все **`send`** до завершения предыдущего. **Зафиксированный выбор — (а):** несколько **`send`** подряд разрешены (у каждого свой **`generateClientMessageId()`**); идемпотентность на сервере это поддерживает. Защита от двойного тапа по **одной** фразе — **UI (11.8**, debounce/throttle), не очередь «один запрос на всё» в контроллере.

**Альтернатива (б)** — FIFO **`Queue<PendingMessage>`** в контроллере — только если продукт явно требует строгой очереди; по умолчанию не использовать.

## `CancelToken` и повторный `build` (не `late final`)

**Не** объявлять **`late final CancelToken _cancelToken`**, если контроллер делает **`refresh()`** / повторные **`build`** без гарантии нового экземпляра нотифайера: повторное присвоение **`late final`** → **`LateInitializationError`**.

**Рекомендуемый паттерн:** поле **`CancelToken? _historyCancelToken`** (имя на усмотрение PR). В **начале каждого** `build` (или выделенном методе «начать сессию загрузки»): **`_historyCancelToken?.cancel()`**, затем **`_historyCancelToken = CancelToken()`**; в **`ref.onDispose`** — **`_historyCancelToken?.cancel()`**.

**Один токен на всю «историю» одной сессии:** все вызовы **`getConversation`**, **`getMessages`** (первая страница и **`loadOlder`**) в рамках этой сессии передают **один и тот же активный** **`_historyCancelToken`**. Иначе **`refresh()`** отменит только часть inflight-запросов, если **`loadOlder`** создаст локальный токен.

**`sendMessage` — отдельно от истории:** **не** передавать в **`sendMessage`** общий **`_historyCancelToken`**. Варианты: отдельный **`CancelToken`** на каждую inflight-отправку (отмена только явной «отменить отправку» / dispose) **или** **`cancelToken: null`**, если отмена отправки до dispose не нужна. Иначе **`refresh()`** отменит **`sendMessage`** → потеря пользовательского ввода и невозможность идемпотентного ретрая с тем же **`clientMessageId`**.

## `refresh()`: не только `invalidateSelf`

| Подход | Риск |
|--------|------|
| **`ref.invalidateSelf()`** | Новый нотифайер → новый токен, отмена старых запросов; но сбрасывается **вся** локальная ошибка (**sendError** и т.д.), возможны неожиданные побочные эффекты |
| Только ручной **`AsyncLoading` + await load** | Без отмены старого токена — гонки |

**Зафиксировать:** **`refresh()`** = явная отмена текущего **`_historyCancelToken`**, выдача **нового** токена истории (и при необходимости инкремент **эпохи** / токена сессии, чтобы ответы **старого** **`loadOlder`** не мержились в state), сброс offset/флагов пагинации, повторная загрузка **getConversation + getMessages** (первые страницы). Inflight-**`sendMessage`** при этом **не** отменяются общим токеном истории (см. § **`CancelToken`**). По желанию UX: сохранять **`sendError`** при refresh или очищать — описать в dartdoc одним предложением.

**Не** полагаться на **`invalidateSelf()`** как единственный механизм refresh, если нужно контролировать сохранение полей state.

## Пустая страница и `has_next: true` (анти-цикл)

Если ответ **`messages: []`** при **`has_next: true`** (рассинхрон бэкенда/фильтра), после **merge** число новых элементов **0**. Тогда **принудительно** выставить **`hasMoreOlder = false`** (или прекратить дальнейшие **`loadOlder`**), даже если в JSON **`has_next`**. Иначе **`loadOlder`** зациклится.

### Минимальные поля **`PendingMessage`** (Freezed в **`presentation/state/pending_message.dart`** или рядом)

Структура нужна, чтобы **11.5** / **11.8** различали timeout vs **502** и счётчик попыток; **не** договариваться «строка **`content`** без контекста ошибки».

```dart
@freezed
abstract class PendingMessage with _$PendingMessage {
  const factory PendingMessage({
    required String clientMessageId,
    required String content,
    Object? lastError,
    @Default(1) int attempts,
    required DateTime lastAttemptAt,
  }) = _PendingMessage;

  const PendingMessage._();
}
```

**`lastError`** — типично **`ConversationApiException`**, **`DioException`** или унифицированная обёртка после репозитория; расширение полей — по PR. Дубликат **`clientMessageId`** внутри factory не обязателен (ключ уже в **`Map`**), но допустим для удобства.

**`attempts`:** при первой записи pending после неудачного **`send`** — **`1`**. При каждом **`retrySend`** для того же **`clientMessageId`** — **`attempts + 1`** и обновить **`lastAttemptAt`**.

### Идемпотентный ретрай (транзиентные ошибки)

Смысл **`X-Client-Message-ID`** — не плодить дубликаты в БД при повторе запроса. **Первая** попытка **`send`** всегда получает **новый** id через **`generateClientMessageId()`**.

Если **`sendMessage`** упал с **транзиентной** ошибкой (**`ConversationApiException`** с **`statusCode == 502`**, таймаут, **`connectionError`** / сеть по классификации Dio в репозитории — единообразно с **`CreateProjectController`**), контроллер **сохраняет** пару **`clientMessageId` → pending** в **`ChatState.pendingByClientId`** (текст для **`SendMessageRequest`**) и предлагает ретрай через **`retrySend(clientMessageId)`**, который вызывает **`ConversationRepository.sendMessage`** с **тем же** **`clientMessageId`** и телом из карты. Повтор с **новым** id допустим только если: пользователь явно инициировал **новое** сообщение (новый вызов **`send`**, новый текст), **или** ошибка **логическая** (**400** / **422** / валидация контента) — тогда новая попытка = новый **`generateClientMessageId()`**, соответствующий pending удаляется.

**Автоматический** повтор запроса (интерceptor Dio, экспоненциальный backoff, фоновые ретраи) — **не входит в 11.4**. Триггер **`retrySend`** — только **действие пользователя** (**11.5** / **11.8**, кнопка «Повторить» и т.п.), иначе легко нарушить идемпотентность или задублировать логику отмены.

Без этого правила повтор по кнопке «Повторить» после **502** с новым UUID создаст **второе** сообщение на сервере.

### Гонка двух одновременных `loadOlder`

Флага **`isLoadingOlder`** недостаточно: два синхронных вызова (например, из **`ScrollController`**) оба могут прочитать **`false`** до установки **`true`** → два HTTP-запроса и двойной инкремент **`olderOffset`**.

**Зафиксировать один из эквивалентов:** **`Future<void>? _olderInFlight`**: если **`!= null`**, **`return _olderInFlight!`** (или **`await`** тот же **`Future`** — на усмотрение PR); в **`finally`** обнулить. Альтернатива — **`Completer`** / явная блокировка, если команда предпочитает.

### **`olderOffset`**

Инкрементировать **после успешного merge** на **`response.messages.length`** (фактическое число сообщений в ответе), **а не** на **`limit`**. Так корректна последняя усечённая страница и меньше расхождения при гонках после фикса выше.

### **`SendMessageRequest` и `trim`**

Внутри **`send` / `retrySend`**: **`final trimmed = raw.trim()`** (или сохранённый текст pending); если пусто — **return** без сети. В репозиторий: **`SendMessageRequest(content: trimmed)`** — явная связь с **11.1** / **`domain/requests.dart`**.

### **`SendMessageResult` — upsert по `id` и порядок списка**

И **`MessageSendStatus.created`**, и **`duplicate`** для UI — **upsert по `message.id`** (одна запись на id). Разница статусов — логи / метрики.

**При двух обновлениях одного и того же `id`** (например, поздний WS и HTTP): новое значение **полностью заменяет** старое (**last-write-wins** по всем полям **`ConversationMessageModel`**). Так детерминировано поведение при расхождении **`created_at`** из-за бага/реджекта на ином канале.

**После любого upsert или merge страниц** список для UI снова приводится к **каноническому порядку сообщений**: **`created_at` по возрастанию**, при равенстве **`created_at`** — **`id` по возрастанию** (согласовано с SQL на бэкенде для стабильного порядка). **Запрещено** вставлять «в конец» или «в начало» без пересортировки: событие **11.9** может прийти с **`created_at`** «из прошлого» (десинк часов, переподключение), **`loadOlder`** даёт пересечение по id.

**Реализация на 11.4:** простой **`List<ConversationMessageModel>`** и сортировка после каждого upsert/merge. Если в проде появятся жалобы на jank на очень длинных чатах и частых WS — вынести вторичный индекс (**`SplayTreeMap`** / map **`id`→позиция**) в **отдельную задачу оптимизации**.

Дальше из этого списка строится порядок для **`ListView`** (снизу вверх) — переворот **только** presentation-обёрткой над уже отсортированным каноном, без нарушения относительного порядка по **`created_at`/`id`**.

## Поведение до мержа **11.9** (UX)

После успешного **`sendMessage`** в списке появится **пользовательское** сообщение (ответ API). **Ответ ассистента** по HTTP в этом контракте **не гарантирован**; он ожидается из **WebSocket** в **11.9**. До **11.9** нормально видеть чат «только с репликой пользователя» без ответа ассистента. Полный сценарий «отправил → увидел ответ ассистента» — интеграционно проверяется после **11.9**. В критериях приёмки **11.4** это **ожидаемое** ограничение, не баг.

**Optimistic UI:** в **11.4** **не** добавлять в ленту фейковое сообщение с временным id до ответа **`sendMessage`** — иначе ломается дедуп по **`id`**. Текст в списке появляется **после** успешного ответа API (**created/duplicate**) или остаётся только в **`pendingByClientId`** для UI «Повторить» без строки в ленте при транзиентном фейле (**сознательный trade-off:** при медленной сети ввод «не виден» до ответа). Отдельная задача — optimistic placeholder с согласованной моделью id — при необходимости позже.

## Обработка ошибок

- Загрузка чата: **`AsyncValue`** с ошибкой при фатальных сбоях; **`ConversationCancelledException`** — см. § выше.
- Отправка: ошибки не должны затирать весь список; **`ConversationCancelledException`** на **отдельном** токене отправки (если используется) не считать «ошибкой сети» для SnackBar так же, как для истории.
- **`401` / `UnauthorizedException`:** текст для пользователя — через **`chatErrorTitle`** / **`l10n`** (**§ Локализация**). **Разлогин, очистка токена, редирект на login** — зона **глобального auth-interceptor / роутера / auth-слоя**, **не** реализация **11.4**; контроллер не дублирует глобальную auth-политику.
