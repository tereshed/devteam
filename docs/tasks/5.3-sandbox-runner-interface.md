# 5.3 Интерфейс `SandboxRunner`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)
**Файл:** `backend/internal/sandbox/runner.go`
**Статус:** ✅ Выполнена

---

## Описание

Определить Go-интерфейс **`SandboxRunner`** — контракт между слоем оркестрации/сервисов и **инфраструктурой изолированного выполнения** задач в Docker. Интерфейс описывает **жизненный цикл** sandbox-инстанса: создание и запуск (**`RunTask`**), **блокирующее ожидание завершения** (**`Wait`**, без поллинга), снимок статуса (**`GetStatus`**), стрим логов (**`StreamLogs`**), остановка (**`Stop`**) и гарантированная очистка с отменой ожиданий (**`Cleanup`**).

**Ключевой принцип:** Интерфейс — **только** абстракция над рантаймом (Docker сейчас, в будущем — Kubernetes / remote runner). Он **не** знает о доменных сущностях БД (`models.Task` и т.д.) на уровне сигнатур: идентификаторы задачи и проекта передаются как **строки/UUID в опциях** из вызывающего кода (`SandboxOptions` — задача **5.4**).

Реализация по умолчанию для MVP — **`DockerSandboxRunner`** (задачи **5.5–5.9**). Поведение контейнера внутри образа (`entrypoint.sh`, задача **5.2**) и формат артефактов согласуются с полями **`SandboxOptions`** и парсером результатов (**5.7**).

## Зависимости

**Требует:** **5.1**, **5.2** — готовый образ и контракт env/артефактов (ориентир для имён полей в **5.4**).

**Порядок с `types.go` (5.4):** В README **5.3** идёт перед **5.4**, но компилятору Go нужны типы в том же пакете `sandbox`. **Допустимо** в одном PR добавить **`runner.go` + `types.go`**, соблюдая нумерацию задач по смыслу: сначала зафиксировать контракт интерфейса (этот документ), затем структуры в **`types.go`**.

**Используется в:** **5.5** (`DockerSandboxRunner`), **5.6–5.9** (расширение реализации), **5.13–5.14** (тесты), **6.3** (`SandboxAgentExecutor` / оркестратор — инжект `SandboxRunner` через конструктор, без глобалов).

## Контекст использования

Согласно `main.mdc` (Sandbox Runner), Go-бэкенд:

1. Создаёт контейнер с параметрами репо, ветки, backend (`claude-code` / `aider`), лимитов и env (**`RunTask`**).
2. При необходимости ждёт завершения процесса в изоляции через **`Wait`** (блокирующее ожидание, без поллинга **`GetStatus`** в цикле).
3. При необходимости — один стрим логов (**`StreamLogs`**) на инстанс; раздача в БД и в WebSocket — в оркестраторе (см. «`StreamLogs`: один активный читатель»).
4. По завершении или ошибке забирает артефакты с диска контейнера (**5.7**).
5. При отмене пользователем вызывает **`Stop`**, затем **`Cleanup`**.

**Разделение слоёв (Clean Architecture):**

| Слой | Знание о Docker | Знание о `SandboxRunner` |
|------|-------------------|---------------------------|
| `handler` | ❌ | ❌ (только через сервисы) |
| `service` / orchestrator | ❌ | ✅ вызывает интерфейс |
| `internal/sandbox` | ✅ только в `DockerSandboxRunner` | ✅ определение + реализация |

## Интерфейс

### runner.go

Ниже — целевой контракт из концепции продукта; точные имена типов (`SandboxOptions`, `SandboxInstance`, `SandboxStatus`, `LogEntry`, `CodeBackendType` и т.д.) **фиксируются в задаче 5.4** в пакете `sandbox` (тот же модуль, без зависимости от `internal/models` в полях «доменного» уровня — только примитивы и enums пакета `sandbox`).

```go
package sandbox

import "context"

// SandboxRunner — абстракция изолированного выполнения задачи (Docker / иное).
// Реализация по умолчанию: DockerSandboxRunner (5.5).
//
// context.Context в каждом методе задаёт дедлайн/отмену для самого вызова API раннера
// (обращение к Docker Engine, ожидание ответа). Это не то же самое, что бизнес-таймаут задачи:
// см. раздел «Два уровня таймаутов».
type SandboxRunner interface {
	// RunTask создаёт и запускает изолированный runtime для одной задачи.
	// opts содержит образ, RepoURL, ветку, backend, SandboxOptions.Timeout (бизнес-таймаут задачи),
	// лимиты ресурсов, EnvVars для entrypoint.
	// Не дожидается завершения процесса в контейнере — см. Wait.
	// Возвращает SandboxInstance с идентификатором инстанса: см. раздел «Идентификатор sandboxID».
	RunTask(ctx context.Context, opts SandboxOptions) (*SandboxInstance, error)

	// Wait блокирует выполнение до завершения работы sandbox-инстанса (контейнер exited/stopped),
	// до отмены/дедлайна ctx этого вызова, либо до срабатывания бизнес-таймаута opts.Timeout из RunTask
	// (принудительный stop/kill контейнера — см. «Два уровня таймаутов»).
	// Не использует поллинг GetStatus в цикле — ожидание через нативные механизмы рантайма
	// (например docker.ContainerWait + внутренний таймер задачи).
	// Возвращает финальный SandboxStatus (включая ExitCode, признак успеха по политике 5.7) или ошибку ожидания.
	// При отмене только ctx вызова — возврат по правилам Go (context.Canceled / DeadlineExceeded);
	// контейнер при этом не должен остаться «сиротой» без политики завершения: действует opts.Timeout и/или Stop/Cleanup.
	Wait(ctx context.Context, sandboxID string) (*SandboxStatus, error)

	// GetStatus возвращает снимок текущего статуса (для UI, однократной диагностики).
	// Не является заменой Wait для ожидания завершения задачи (запрещён busy-poll в оркестраторе).
	GetStatus(ctx context.Context, sandboxID string) (*SandboxStatus, error)

	// StreamLogs открывает поток логов stdout/stderr контейнера в виде канала.
	//
	// Вариант А (MVP): на один sandboxID допускается не более одного активного стрима.
	// Повторный вызов, пока первый стрим не завершён (канал не закрыт, Cleanup не отменил чтение) —
	// ошибка ErrStreamAlreadyActive. Fan-out (БД + WebSocket) — ответственность оркестратора.
	// Контракт: канал закрывается при завершении стрима, отмене ctx или Cleanup; детали — 5.6.
	StreamLogs(ctx context.Context, sandboxID string) (<-chan LogEntry, error)

	// Stop инициирует мягкую остановку (SIGTERM / docker stop), ожидаемую entrypoint (5.2).
	Stop(ctx context.Context, sandboxID string) error

	// Cleanup удаляет контейнер и связанные ресурсы; идемпотентность желательна (повторный вызов — no-op или nil).
	// Обязанность по ресурсам: см. раздел «Контракт Cleanup и стримы».
	Cleanup(ctx context.Context, sandboxID string) error
}
```

**Замечания по семантике:**

- **`RunTask` + `Wait`:** **`RunTask`** только создаёт и запускает инстанс и **сразу** возвращает управление. Завершение работы агента в контейнере оркестратор ожидает вызовом **`Wait`**. Циклический опрос **`GetStatus`** для ожидания завершения **запрещён** как антипаттерн (лишняя нагрузка на CPU и Docker API). Точка реализации без busy-poll — **5.5** (`DockerSandboxRunner`).
- **`Stop`** и **`Cleanup`** разделены: остановка для отмены пользователем vs финальная уборка после сбора артефактов (**5.7**).

## `StreamLogs`: один активный читатель (**вариант А**)

**Проблема:** Оркестратор пишет логи в БД, а WebSocket-хендлер стримит в UI. Два параллельных **`StreamLogs(ctx, sameID)`** с двумя вызовами **`ContainerLogs`** дают дублирование горутин, неопределённое разбиение чанков между потребителями и гонки при чтении одного потока движка.

**Выбранный контракт (MVP) — вариант А:**

| Поведение | Требование |
|-----------|------------|
| Количество стримов на `sandboxID` | Не более **одного** активного одновременно |
| Повторный `StreamLogs` при уже активном стриме | Возврат **`ErrStreamAlreadyActive`** (без второго attach к Docker logs) |
| Несколько потребителей (БД + UI) | **Оркестратор** (или один координирующий сервис) читает **один** канал и **сам** рассылает копии событий (tee / broadcast на уровне приложения) |

**Вариант Б** (fan-out внутри раннера, несколько подписчиков на один источник) **не** входит в MVP; при необходимости post-MVP — отдельная задача и смена контракта (например `SubscribeLogs` + refcount).

Реализация блокировки «один стрим» и закрытие канала при **`Cleanup`** — **5.6**.

## Два уровня таймаутов: `context.Context` и `SandboxOptions.Timeout`

**Проблема:** У **`Wait`** (или **`RunTask`**) есть **`ctx`** с дедлайном 5 минут, а в **`SandboxOptions.Timeout`** указано 10 минут. Непонятно, что завершает контейнер и что означает отвал **`ctx`** HTTP-запроса: останется ли контейнер «сиротой».

**Разграничение:**

| Механизм | Зона ответственности | Поведение |
|----------|----------------------|-----------|
| **`context.Context`** в аргументах методов | Таймаут/отмена **конкретного вызова** к раннеру и низкоуровневых операций с Docker API (создать контейнер, один attach к логам, один RPC `wait` и т.д., по политике **5.5**) | Истёкший дедлайн или **`ctx.Done()`** означают: **вызывающий** больше не ждёт этот вызов — метод возвращает ошибку / отмену по правилам Go. Это **не** замена бизнес-ограничения «сколько может длиться задача». |
| **`SandboxOptions.Timeout`** (поле в opts, переданное в **`RunTask`**) | **Бизнес-таймаут** жизни задачи в изоляции | Раннер **обязан** учитывать его для **принудительной остановки контейнера** (SIGTERM / stop / kill по политике **5.8**), **независимо** от того, жив ли **`ctx`** того HTTP-запроса, который когда-то вызвал **`RunTask`** или **`Wait`**. Таймер стартует с момента, зафиксированным в **5.5** (типично: после успешного `start` контейнера). |

**Согласованное поведение при конфликте значений:**

1. Если **`ctx` у `Wait`** отменён раньше, чем сработал **`opts.Timeout`**: **`Wait`** возвращает (например **`context.Canceled`** / **`DeadlineExceeded`**), но **контейнер не обязан** немедленно исчезать только из‑за этого — он всё ещё ограничен **`opts.Timeout`** и должен быть остановлен раннером по внутреннему таймеру, либо позже через **`Stop`** / **`Cleanup`** оркестратора. Цель: **нет бессрочных сирот** при типичных коротких HTTP-ctx.
2. Если сработал **`opts.Timeout`** раньше дедлайна **`ctx`**: контейнер принудительно останавливается по бизнес-правилу; **`Wait`** завершается с финальным статусом / ошибкой по политике **5.5** (не маскировать как «только отмена ctx»).
3. **`RunTask(ctx, opts)`**: **`ctx`** ограничивает время **создания и старта** (pull образа, create, start); **`opts.Timeout`** не должен применяться к бесконечному ожиданию pull, если это отдельная политика — уточнить в **5.5** (граница: «таймер задачи» стартует после `start`).

Детали таймера, взаимодействие с **`Stop`** и сигналами — **5.5**, **5.8**.

## Идентификатор `sandboxID` (безопасность)

**Проблема:** Если строка `sandboxID` строится из пользовательского ввода, имён веток или произвольных полей задачи без нормализации, возможны **path traversal**, подстановка в аргументы хоста и **инъекции в вызовы Docker API** при формировании имён volume, bind-mount и т.д.

**Требования к контракту:**

1. **`RunTask`** возвращает в `SandboxInstance` **только** идентификатор, **сгенерированный или выданный доверенным рантаймом**, а не скопированный из пользовательского поля «как есть». Допустимые варианты:
   - идентификатор контейнера, возвращённый Docker Engine после успешного create (типично **hex**, длина по правилам движка);
   - либо внутренний **UUID** раннера с **детерминированным** маппингом на контейнер во внутренней таблице реализации (никогда не отдавать наружу сырой пользовательский slug как `sandboxID`).
2. Все методы с параметром **`sandboxID`** (**`Wait`**, **`GetStatus`**, **`StreamLogs`**, **`Stop`**, **`Cleanup`**) **обязаны** в начале выполнять **валидацию формата** строки (например: только **hex** допустимой длины для Docker ID, или строгий **UUID** для внутреннего идентификатора — политика фиксируется в **5.5** и в коде одной функцией `validateSandboxID(string) error`).
3. При несоответствии формату — возврат **`ErrInvalidSandboxID`** без передачи строки в Docker API и без использования в путях на хосте.
4. Поля в **`SandboxOptions`** (URL репо, имя ветки, env) остаются **внешним вводом**: их валидация и безопасная передача в контейнер — отдельные правила (entrypoint **`--`**, см. **5.2**); это **не** отменяет требований к **`sandboxID`**.

## Ошибки

Sentinel-ошибки в том же пакете (в `runner.go` или `errors.go`):

| Ошибка | Когда |
|--------|--------|
| `ErrInvalidSandboxID` | Строка `sandboxID` не прошла валидацию формата перед вызовом Docker API / доступом к ресурсам инстанса |
| `ErrSandboxNotFound` | Валидный ID, но инстанс неизвестен раннеру (уже удалён, никогда не существовал) |
| `ErrSandboxAlreadyStopped` | Повторная остановка в недопустимом состоянии (по политике реализации) |
| `ErrInvalidOptions` | Невалидные **`SandboxOptions`** после проверки в начале **`RunTask`** |
| `ErrStreamAlreadyActive` | Повторный **`StreamLogs`** для того же `sandboxID`, пока активен первый стрим (**вариант А**) |

**`ErrInvalidOptions` и отладка (early return):** Валидация **`SandboxOptions`** выполняется **в самом начале** **`RunTask`**, до обращения к Docker. Сообщение об ошибке **не** должно быть «голым» sentinel без контекста. **Обязательно** оборачивать с указанием поля/условия, например:

```go
if opts.Image == "" {
    return nil, fmt.Errorf("image: %w", ErrInvalidOptions)
}
// или fmt.Errorf("%w: empty image", ErrInvalidOptions) — единый стиль выбрать в пакете
```

Так оркестратор и логи показывают, **что именно** невалидно (пустой образ, отрицательный лимит памяти, нулевой таймаут и т.д.), сохраняя проверку `errors.Is(err, ErrInvalidOptions)` для маппинга на HTTP.

Маппинг на HTTP — **не** в пакете `sandbox`; на границе API — в `handler`/`apierror`.

## Контракт `Cleanup` и стримы (утечки горутин)

**Проблема:** Контейнер уже удалён, а горутины, которые ждут логи или «вечный» **`Wait`**, остаются висеть.

**Требования к реализации `Cleanup` (и согласованному внутреннему состоянию в **5.5–5.6**):**

1. **`Cleanup`** **обязан** принудительно завершить все **активные стримы логов** для данного `sandboxID`: отменить контекст чтения, закрыть канал **`StreamLogs`** с потребителем, дождаться завершения внутренних горутин раннера по этому инстансу (или эквивалент безопасной отмены — деталь **5.6**).
2. **`Cleanup`** **обязан** разблокировать любой ожидающий вызов **`Wait`** для этого `sandboxID` (завершить ожидание с ошибкой состояния / «инстанс удалён» или согласованным sentinel, **не** оставляя блокировку навсегда после удаления контейнера).
3. Порядок в типичном сценарии оркестратора: **`Stop`** (по необходимости) → сбор артефактов (**5.7**) → **`Cleanup`**; даже если **`Stop`** не вызывался, **`Cleanup`** не должен оставлять фоновые ожидания.

## Дизайн-решения

1. **DI:** Реализация передаётся в сервис оркестратора через конструктор; **запрещены** глобальные переменные и `init()`.
2. **Context:** Все методы принимают `context.Context` первым аргументом.
3. **Без Gin / HTTP** в интерфейсе.
4. **Согласование с entrypoint (5.2):** Имена env и путей артефактов в `SandboxOptions` должны совпадать с тем, что ожидает `entrypoint.sh` (см. задачу **5.2**, разделы контрактов).
5. **Ожидание завершения:** только **`Wait`**, не поллинг **`GetStatus`** (см. раздел «Замечания по семантике»).
6. **`sandboxID`:** доверенный идентификатор из **`RunTask`** + валидация во всех методах; см. раздел «Идентификатор `sandboxID`».
7. **`Cleanup`:** гарантированная отмена стримов и разблокировка **`Wait`**; см. раздел «Контракт `Cleanup` и стримы».
8. **`StreamLogs`:** вариант А — один активный стрим на ID, **`ErrStreamAlreadyActive`** при повторе; см. отдельный раздел.
9. **Таймауты:** `ctx` — границы вызовов API; **`SandboxOptions.Timeout`** — бизнес-лимит и принудительный stop контейнера; см. «Два уровня таймаутов».
