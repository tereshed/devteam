# 4.1 Интерфейс GitProvider

**Sprint:** 4 — Git Provider Integration (Backend)
**Файл:** `backend/pkg/gitprovider/provider.go`
**Статус:** ✅ Выполнена

---

## Описание

Определить универсальный Go-интерфейс `GitProvider`, абстрагирующий операции с git-репозиториями для разных провайдеров (GitHub, GitLab, Bitbucket, Local). Интерфейс — контракт для всех реализаций Sprint 4 (4.3 `GitHubProvider`, 4.4 `LocalGitProvider`) и точка интеграции для оркестратора (Sprint 6) и sandbox-раннера (Sprint 5).

**Ключевой принцип:** Интерфейс должен покрывать **минимально необходимый** набор операций для MVP-пайплайна:

```
[Sandbox: Clone → Branch → Agent → Commit → Push] → [Go: Create PR → Review → Merge]
```

Расширенные операции (Issues, Releases, Webhooks) — будущие спринты.

---

## Контекст использования

### ⚠️ Архитектурное решение: разделение ответственности Git-операций

Согласно `main.mdc` (раздел 4.3), **Docker-контейнер (sandbox) самостоятельно** выполняет git-операции через `entrypoint.sh`:

```bash
# Внутри контейнера (entrypoint.sh):
git clone --depth=50 "$REPO_URL" /workspace/repo
git checkout -b "$BRANCH_NAME"
# ... агент (Claude Code / Aider) работает ...
git push origin "$BRANCH_NAME"
```

Это означает, что **Orchestrator на Go НЕ вызывает** `Clone`, `CreateBranch`, `CommitAndPush` напрямую для основного пайплайна разработки. Контейнер получает `REPO_URL`, `BRANCH_NAME` и credentials через `SandboxOptions.EnvVars` и делает всё сам.

**Выбранный путь: Гибридная модель (Sandbox + API)**

| Зона ответственности | Кто выполняет | Через что |
|---|---|---|
| Clone, Branch, Commit, Push | **Docker-контейнер** (entrypoint.sh) | `git` CLI внутри sandbox |
| Validate Access, Create PR, Get PR, Diff, Review | **Go-бэкенд** (Orchestrator) | `GitProvider` interface (API) |
| Начальный импорт проекта | **Go-бэкенд** (ProjectService) | `GitProvider.Clone` + `ValidateAccess` |
| Cleanup (удаление веток) | **Go-бэкенд** (после merge PR) | `GitProvider.DeleteBranch` |

**Почему Clone/Branch/Commit/Push всё равно в интерфейсе:**
1. **ProjectService (4.6)** — при создании проекта из существующего репо: `Clone` для начальной индексации кода в векторку.
2. **LocalGitProvider (4.4)** — для `local` провайдера контейнер может не использоваться (лёгкие задачи без изоляции).
3. **Утилитарные сценарии** — скрипты миграции, batch-операции, тестирование.
4. **Будущее** — переход на «Путь А» (Orchestrator готовит workDir, монтирует volume) если entrypoint.sh станет узким местом.

### Полный пайплайн (MVP)

```
1. Пользователь создаёт задачу через чат
2. OrchestratorService определяет агента и создаёт ветку
3. OrchestratorService → SandboxRunner.RunTask(SandboxOptions{
       RepoURL:  project.GitURL,
       Branch:   "task/<id>",
       EnvVars:  {GIT_TOKEN: decrypted, REPO_URL: ..., BRANCH_NAME: ...},
   })
4. Docker-контейнер (entrypoint.sh):
   a. git clone → git checkout -b → агент работает → git commit → git push
   b. Результат: /workspace/result.json, /workspace/full.diff
5. Go-бэкенд забирает результат: docker.CopyFromContainer(...)
6. GitProvider.CreatePullRequest() → создаёт PR через API
7. Reviewer-агент: GitProvider.GetDiff() / GetPullRequest() / ListPRFiles()
8. GitProvider.AddPRComment() → комментарий ревьюера
9. После merge: GitProvider.DeleteBranch() → cleanup
```

### Кто использует GitProvider

| Потребитель | Операция | Sprint | Контекст |
|---|---|---|---|
| `ProjectService` (4.6) | `Clone`, `ValidateAccess` | 4 | Импорт проекта, валидация credentials |
| `OrchestratorService` (6.4) | `CreatePullRequest`, `GetDiff`, `ListBranches`, `DeleteBranch` | 6 | PR-менеджмент после работы sandbox |
| Reviewer Agent (6.2) | `GetPullRequest`, `GetDiff`, `ListPRFiles`, `AddPRComment` | 6 | Code review через API |
| `SandboxRunner` (5.5) | — (не использует GitProvider напрямую) | 5 | Git через entrypoint.sh внутри контейнера |
| Утилитарные сценарии | `Clone`, `Commit`, `Push`, `GetLocalDiff` | — | Скрипты, тесты, Local-провайдер без sandbox |

## Интерфейс

### provider.go

```go
package gitprovider

import (
    "context"
    "io"
)

// GitProvider — универсальный интерфейс для работы с git-репозиториями.
// Реализации: GitHubProvider (4.3), LocalGitProvider (4.4).
// В будущем: GitLabProvider, BitbucketProvider.
//
// Все типы, используемые в сигнатурах (CloneOptions, BranchOptions, PullRequest,
// PRFile, RepoInfo, Credentials и т.д.), определены в этом же пакете
// (pkg/gitprovider/types.go — задача 4.2). Пакет НЕ ЗАВИСИТ от internal/models.
type GitProvider interface {
    // --- Базовые операции ---

    // ValidateAccess проверяет, что credentials корректны и есть доступ к репозиторию.
    // Возвращает nil если доступ есть, иначе — ошибку с деталями.
    ValidateAccess(ctx context.Context, repoURL string) error

    // Clone клонирует репозиторий в указанную директорию.
    // repoURL — URL репозитория (источник). Остальные параметры — в CloneOptions.
    // ВАЖНО: Реализация обязана самостоятельно инжектить credentials (из Factory)
    // в процесс клонирования (например, через URL https://x-access-token:<token>@...
    // или git config), чтобы избежать зависания при запросе пароля в консоли.
    Clone(ctx context.Context, repoURL string, opts CloneOptions) error

    // --- Операции с ветками ---

    // CreateBranch создаёт новую ветку от указанной базовой.
    // workDir — директория с клоном репозитория.
    CreateBranch(ctx context.Context, workDir string, opts BranchOptions) error

    // ListBranches возвращает список веток репозитория (REMOTE API).
    // prefix фильтрует по началу имени (например "task/").
    // LocalGitProvider: возвращает ErrNotImplemented (используйте ListLocalBranches).
    ListBranches(ctx context.Context, repoURL string, prefix string) ([]string, error)

    // ListLocalBranches возвращает список веток в локальном клоне.
    // workDir — путь к директории с клоном репозитория на диске.
    // Поддерживается ВСЕМИ провайдерами.
    ListLocalBranches(ctx context.Context, workDir string, prefix string) ([]string, error)

    // DeleteBranch удаляет ветку (REMOTE API).
    // ВАЖНО: Если ветки уже нет (например, GitHub автоматически удалил её после мержа),
    // метод должен возвращать ErrBranchNotFound.
    // LocalGitProvider: возвращает ErrNotImplemented (используйте DeleteLocalBranch).
    DeleteBranch(ctx context.Context, repoURL string, branch string) error

    // DeleteLocalBranch удаляет ветку в локальном клоне.
    // workDir — путь к директории с клоном репозитория на диске.
    // Поддерживается ВСЕМИ провайдерами.
    DeleteLocalBranch(ctx context.Context, workDir string, branch string) error

    // --- Коммит и пуш ---

    // Commit создаёт локальный коммит в workDir без пуша в remote.
    // workDir — директория с клоном репозитория.
    // Возвращает SHA коммита и флаг hasChanges (false, если нечего коммитить).
    // Позволяет агенту делать несколько логических коммитов перед финальным пушем.
    // Author задаётся через CommitOptions.Author (структура { Name, Email }).
    // Files задаётся через CommitOptions.Files ([]string). Если пустой массив, выполняется `git add -A`.
    Commit(ctx context.Context, workDir string, opts CommitOptions) (commitSHA string, hasChanges bool, err error)

    // Push отправляет локальные коммиты в remote.
    // workDir — директория с клоном репозитория.
    // Отделён от Commit для поддержки retry при сетевых ошибках
    // и сценария "несколько коммитов → один пуш".
    Push(ctx context.Context, workDir string, opts PushOptions) error

    // CommitAndPush — convenience-метод: Commit + Push в одном вызове.
    // workDir — директория с клоном репозитория.
    // Для простых сценариев. Принимает и CommitOptions, и PushOptions,
    // так как для безопасного пуша необходимо явно знать имя ветки.
    // Возвращает SHA коммита и флаг hasChanges.
    CommitAndPush(ctx context.Context, workDir string, commitOpts CommitOptions, pushOpts PushOptions) (commitSHA string, hasChanges bool, err error)

    // --- Diff ---

    // GetDiff возвращает unified diff между двумя ветками через API провайдера (REMOTE).
    // Для GitHub: REST API compare endpoint. Для GitLab: аналогичный API.
    // LocalGitProvider: возвращает ErrNotImplemented (нет remote API; используйте GetLocalDiff).
    //
    // ВАЖНО: Вызывающая сторона ОБЯЗАНА закрыть возвращённый io.ReadCloser (defer rc.Close()).
    GetDiff(ctx context.Context, repoURL string, base, head string) (io.ReadCloser, error)

    // GetLocalDiff возвращает unified diff между двумя ref'ами в локальном клоне.
    // workDir — путь к директории с клоном репозитория на диске.
    // Работает через `git diff` в workDir (exec.Command).
    // Поддерживается ВСЕМИ провайдерами (операция локальная, не зависит от API).
    //
    // ВАЖНО: Вызывающая сторона ОБЯЗАНА закрыть возвращённый io.ReadCloser (defer rc.Close()).
    // Для LocalGitProvider реализация Close() ОБЯЗАНА вызывать cmd.Wait() для предотвращения zombie-процессов.
    GetLocalDiff(ctx context.Context, workDir string, base, head string) (io.ReadCloser, error)

    // --- Pull Requests ---

    // CreatePullRequest создаёт PR/MR в репозитории (REMOTE API).
    // repoURL — URL репозитория.
    // LocalGitProvider: возвращает ErrNotImplemented.
    CreatePullRequest(ctx context.Context, repoURL string, opts PRCreateOptions) (*PullRequest, error)

    // UpdatePullRequest обновляет существующий PR (заголовок, описание) (REMOTE API).
    // КРИТИЧНО: агент может изменить фокус задачи после ревью, и PR должен отражать это.
    // LocalGitProvider: возвращает ErrNotImplemented.
    UpdatePullRequest(ctx context.Context, repoURL string, number int, opts PRUpdateOptions) (*PullRequest, error)

    // GetPullRequest получает PR по номеру (REMOTE API).
    // LocalGitProvider: возвращает ErrNotImplemented.
    GetPullRequest(ctx context.Context, repoURL string, number int) (*PullRequest, error)

    // ListPullRequests возвращает список PR с фильтрацией (REMOTE API).
    // repoURL — URL репозитория.
    // Фильтры (состояние, ветка-источник, базовая ветка) задаются через PROptions.
    // КРИТИЧНО для идемпотентности: позволяет Оркестратору найти существующий PR
    // для ветки задачи, если он упал до сохранения номера PR в БД.
    // LocalGitProvider: возвращает ErrNotImplemented.
    ListPullRequests(ctx context.Context, repoURL string, opts PROptions) ([]PullRequest, error)

    // ListPRFiles возвращает список файлов, изменённых в PR (REMOTE API).
    // MVP: возвращает все совпадения без пагинации.
    // TODO (post-MVP): добавить пагинацию для гигантских PR.
    // LocalGitProvider: возвращает ErrNotImplemented.
    ListPRFiles(ctx context.Context, repoURL string, number int) ([]PRFile, error)

    // ListPRComments возвращает список всех комментариев (общих и review) к PR (REMOTE API).
    // КРИТИЧНО для агента-ревьюера: позволяет прочитать комментарии, оставленные человеком
    // или другими агентами (например, CI-ботами) напрямую в GitHub/GitLab, чтобы
    // учесть их контекст при следующем цикле ревью.
    // LocalGitProvider: возвращает ErrNotImplemented.
    ListPRComments(ctx context.Context, repoURL string, number int) ([]PRComment, error)

    // AddPRComment добавляет общий комментарий к PR (Issue Comment) (REMOTE API).
    // LocalGitProvider: возвращает ErrNotImplemented.
    AddPRComment(ctx context.Context, repoURL string, number int, body string) error

    // AddPRReviewComment добавляет комментарий к конкретной строке кода в PR (Review Comment).
    // Опции (файл, строка, SHA коммита) передаются через PRReviewCommentOptions.
    // LocalGitProvider: возвращает ErrNotImplemented.
    AddPRReviewComment(ctx context.Context, repoURL string, number int, opts PRReviewCommentOptions) error

    // SubmitPRReview отправляет итоговое решение по ревью (Approve, Request Changes, Comment) (REMOTE API).
    // КРИТИЧНО для агента-ревьюера: без этого метода агент может только спамить комментариями,
    // но не может заблокировать или разблокировать мерж.
    // Опции (Event, Body) передаются через PRReviewOptions.
    // LocalGitProvider: возвращает ErrNotImplemented.
    SubmitPRReview(ctx context.Context, repoURL string, number int, opts PRReviewOptions) error

    // MergePullRequest выполняет слияние PR (REMOTE API).
    // Опции слияния (CommitTitle, Message, MergeMethod, SHA) передаются через PRMergeOptions.
    // КРИТИЧНО: Передача SHA обязательна для защиты от race conditions (чтобы не смержить
    // чужой коммит, запушенный в ветку после нашего аппрува).
    // Возвращает ErrConflict, если есть конфликты слияния или проверки не пройдены.
    // LocalGitProvider: возвращает ErrNotImplemented.
    MergePullRequest(ctx context.Context, repoURL string, number int, opts PRMergeOptions) error

    // --- Метаданные ---

    // GetRepoInfo возвращает базовую информацию о репозитории (REMOTE API).
    // Для GitHub: REST API repos endpoint.
    // LocalGitProvider: возвращает ErrNotImplemented.
    GetRepoInfo(ctx context.Context, repoURL string) (*RepoInfo, error)

    // GetFileContent читает содержимое файла из указанной ветки через API провайдера (REMOTE).
    // Для GitHub: REST API contents endpoint.
    // LocalGitProvider: возвращает ErrNotImplemented (используйте GetLocalFileContent).
    //
    // ВАЖНО: Вызывающая сторона ОБЯЗАНА закрыть возвращённый io.ReadCloser
    // после чтения (defer rc.Close()), иначе произойдёт утечка ресурсов
    // (HTTP-соединение для remote).
    GetFileContent(ctx context.Context, repoURL string, branch string, path string) (io.ReadCloser, error)

    // GetLocalFileContent читает содержимое файла из указанного ref'а в локальном клоне.
    // workDir — путь к директории с клоном репозитория на диске.
    // ref — ветка, тег или SHA коммита.
    // Работает через `git show ref:path` в workDir (exec.Command).
    // Поддерживается ВСЕМИ провайдерами (операция локальная).
    //
    // ВАЖНО: Вызывающая сторона ОБЯЗАНА закрыть возвращённый io.ReadCloser
    // после чтения (defer rc.Close()).
    // Для LocalGitProvider реализация Close() ОБЯЗАНА вызывать cmd.Wait()
    // для предотвращения появления zombie-процессов (exec.Cmd).
    GetLocalFileContent(ctx context.Context, workDir string, ref string, path string) (io.ReadCloser, error)

    // ProviderType возвращает тип провайдера (github, gitlab, local и т.д.)
    ProviderType() string

    // SupportsPullRequests возвращает true, если провайдер поддерживает создание PR/MR.
    // Для LocalGitProvider возвращает false. Оркестратор использует это для адаптации пайплайна.
    SupportsPullRequests() bool
}
```

## Ошибки

### Sentinel-ошибки (provider.go)

```go
package gitprovider

import "errors"

var (
    // ErrAuthFailed — невалидные credentials или нет доступа.
    ErrAuthFailed = errors.New("gitprovider: authentication failed")

    // ErrRepoNotFound — репозиторий не найден (404).
    ErrRepoNotFound = errors.New("gitprovider: repository not found")

    // ErrBranchNotFound — ветка не найдена.
    ErrBranchNotFound = errors.New("gitprovider: branch not found")

    // ErrBranchAlreadyExists — ветка уже существует.
    ErrBranchAlreadyExists = errors.New("gitprovider: branch already exists")

    // ErrPRNotFound — Pull Request не найден.
    ErrPRNotFound = errors.New("gitprovider: pull request not found")

    // ErrPRAlreadyExists — Pull Request для этой ветки уже существует.
    // Важно для идемпотентности пайплайна оркестратора.
    ErrPRAlreadyExists = errors.New("gitprovider: pull request already exists")

    // ErrFileNotFound — файл не найден в репозитории.
    ErrFileNotFound = errors.New("gitprovider: file not found")

    // ErrConflict — конфликт при пуше или мерже.
    ErrConflict = errors.New("gitprovider: conflict detected")

    // ErrRateLimited — API rate limit exceeded (GitHub: 5000 req/hr).
    ErrRateLimited = errors.New("gitprovider: rate limit exceeded")

    // ErrPermissionDenied — есть auth, но нет прав на операцию (e.g. protected branch).
    ErrPermissionDenied = errors.New("gitprovider: permission denied")

    // ErrCloneFailed — ошибка клонирования (таймаут, диск, сеть).
    ErrCloneFailed = errors.New("gitprovider: clone failed")

    // ErrNotImplemented — операция не поддерживается для данного провайдера.
    // Например, LocalGitProvider не поддерживает CreatePullRequest.
    ErrNotImplemented = errors.New("gitprovider: operation not implemented for this provider")
)
```

**Маппинг на HTTP (для handler-уровня):**

| Ошибка | HTTP Status |
|---|---|
| `ErrAuthFailed` | 401 Unauthorized |
| `ErrRepoNotFound` | 404 Not Found |
| `ErrBranchNotFound` | 404 Not Found |
| `ErrBranchAlreadyExists` | 409 Conflict |
| `ErrPRNotFound` | 404 Not Found |
| `ErrPRAlreadyExists` | 409 Conflict |
| `ErrFileNotFound` | 404 Not Found |
| `ErrConflict` | 409 Conflict |
| `ErrRateLimited` | 429 Too Many Requests |
| `ErrPermissionDenied` | 403 Forbidden |
| `ErrCloneFailed` | 500 Internal Server Error |
| `ErrNotImplemented` | 501 Not Implemented |

## Дизайн-решения

### 1. Разделение ответственности: Sandbox vs GitProvider (⚠️ КРИТИЧНО)

**Проблема:** Согласно `main.mdc` (раздел 4.3), Docker-контейнер (`entrypoint.sh`) самостоятельно выполняет `git clone`, `git checkout -b`, `git push`. Если контейнер делает всё сам, зачем нужны `Clone`, `CreateBranch`, `CommitAndPush` в `GitProvider`?

**Решение: Гибридная модель.**

| Операция | Основной пайплайн (Developer agent) | Утилитарные сценарии |
|---|---|---|
| Clone | **Sandbox** (`entrypoint.sh`) | **GitProvider** (импорт проекта, индексация) |
| Branch | **Sandbox** (`entrypoint.sh`) | **GitProvider** (LocalGitProvider, скрипты) |
| Commit / Push | **Sandbox** (`entrypoint.sh`) | **GitProvider** (LocalGitProvider, batch) |
| Create PR | **GitProvider** (API после sandbox) | — |
| Get PR / Diff | **GitProvider** (API) | — |
| Validate Access | **GitProvider** (API) | — |

**Правило:**
- Для **sandbox-изолированных задач** (Developer, Tester): git CLI операции — внутри контейнера. Orchestrator передаёт `REPO_URL`, `BRANCH_NAME`, `GIT_TOKEN` через `SandboxOptions.EnvVars`.
- Для **API-операций** (PR, review, diff): Orchestrator использует `GitProvider` интерфейс.
- Для **ProjectService** (импорт, валидация): `GitProvider.Clone` + `ValidateAccess`.
- Для **LocalGitProvider** (без sandbox): все методы доступны напрямую.

**Почему не убираем Clone/Branch/Commit/Push:**
- `ProjectService.Create` вызывает `Clone` для импорта проекта.
- `LocalGitProvider` — работает без контейнера, нужны все операции.
- Будущее: если `entrypoint.sh` станет узким местом, Orchestrator может переключиться на «Путь А» (подготовка workDir + volume mount) без изменения интерфейса.

### 2. Granularity: один интерфейс vs несколько

**Выбор: один интерфейс `GitProvider`.**

**Альтернатива:** Разбить на `Cloner`, `BranchManager`, `PRManager`.

**Обоснование:** На этапе MVP все операции тесно связаны и всегда используются вместе (clone → branch → push → PR). Разбиение добавит сложность без выгоды. При необходимости — вынести sub-интерфейсы позже (`GitBrancher`, `GitPRManager`).

### 3. LocalGitProvider: чёткое разделение local/remote операций

`LocalGitProvider` работает через `exec.Command("git", ...)` и **не имеет доступа** к remote API.

**Работающие методы** (локальные, через git CLI):

| Метод | Реализация |
|---|---|
| `Clone` | `git clone` |
| `CreateBranch` | `git checkout -b` |
| `ListLocalBranches` | `git branch -a --list` |
| `DeleteLocalBranch` | `git branch -d` |
| `Commit` | `git add -A && git commit` |
| `Push` | `git push` |
| `CommitAndPush` | `Commit` + `Push` |
| `GetLocalDiff` | `git diff base..head` в workDir |
| `GetLocalFileContent` | `git show ref:path` в workDir |
| `ValidateAccess` | Проверка существования локальной директории и папки `.git` |
| `ProviderType` | `return "local"` |

**Методы, возвращающие `ErrNotImplemented`** (требуют remote hosting API):

| Метод | Причина |
|---|---|
| `GetDiff(repoURL, ...)` | Нет API; `repoURL` бесполезен для `git diff`. Используйте `GetLocalDiff(workDir, ...)` |
| `GetFileContent(repoURL, ...)` | Нет API; `repoURL` бесполезен для `git show`. Используйте `GetLocalFileContent(workDir, ...)` |
| `ListBranches(repoURL, ...)` | Нет API; используйте `ListLocalBranches(workDir, ...)` |
| `DeleteBranch(repoURL, ...)` | Нет API; используйте `DeleteLocalBranch(workDir, ...)` |
| `GetRepoInfo(repoURL)` | Нет API (GitHub REST /repos) |
| `CreatePullRequest` | Нет remote hosting |
| `UpdatePullRequest` | Нет remote hosting |
| `GetPullRequest` | Нет remote hosting |
| `ListPullRequests` | Нет remote hosting |
| `ListPRFiles` | Нет remote hosting |
| `ListPRComments` | Нет remote hosting |
| `AddPRComment` | Нет remote hosting |
| `AddPRReviewComment` | Нет remote hosting |
| `SubmitPRReview` | Нет remote hosting |
| `MergePullRequest` | Нет remote hosting |

**⚠️ Ключевое правило**: Методы с параметром `repoURL` — **remote-only** (API). Методы с параметром `workDir` — **local** (git CLI). Это не пересекается: `repoURL` нельзя использовать для `exec.Command`, `workDir` нельзя использовать для API.

### 4. Factory и Dependency Injection (DI)

Согласно правилам Clean Architecture (Правило 2.1.2), сервисы бизнес-логики не должны напрямую вызывать глобальные функции пакета. Поэтому в `pkg/gitprovider` определяется интерфейс фабрики:

```go
// В pkg/gitprovider/factory.go:
type Factory interface {
    Create(providerType string, creds Credentials) (GitProvider, error)
}
```

Credentials **не передаются в каждый вызов метода**. Вместо этого экземпляр `GitProvider` создаётся через инжектированную фабрику с уже расшифрованными credentials:

```go
// В ProjectService (фабрика передана через конструктор):
provider, err := s.gitFactory.Create(string(project.GitProvider), Credentials{
    Token: decryptedToken, // уже расшифрован
})
```

**⚠️ ВАЖНО:** Фабрика принимает `string`, а не `models.GitProvider`. Вызывающий код сам выполняет преобразование `string(project.GitProvider)`.

Это гарантирует, что `pkg/gitprovider` не импортирует `internal/models`.

Плюсы:
- Не загрязнять сигнатуры методов
- Переиспользовать один экземпляр для множества операций с одним репо
- Безопасно управлять жизненным циклом credentials

### 5. `workDir` vs `repoURL` — два класса операций + единый стандарт передачи

**Правило:** Параметр однозначно определяет тип операции.

| Параметр | Тип операции | Кто реализует | Методы |
|---|---|---|---|
| `workDir` (string) | Локальная (git CLI) | Все провайдеры | `Clone`*, `CreateBranch`, `ListLocalBranches`, `DeleteLocalBranch`, `Commit`, `Push`, `CommitAndPush`, `GetLocalDiff`, `GetLocalFileContent` |
| `repoURL` (string) | Удалённая (REST API) | GitHub, GitLab, Bitbucket | `ValidateAccess`, `ListBranches`, `DeleteBranch`, `GetDiff`, `CreatePullRequest`, `UpdatePullRequest`, `GetPullRequest`, `ListPullRequests`, `ListPRFiles`, `ListPRComments`, `AddPRComment`, `AddPRReviewComment`, `SubmitPRReview`, `MergePullRequest`, `GetRepoInfo`, `GetFileContent` |

\* `Clone` принимает `repoURL` первым аргументом (источник) + `DestPath` в Options (куда), после чего `DestPath` становится `workDir` для остальных операций.

**⚠️ КРИТИЧНО: Единый стандарт передачи идентификаторов ресурса.**

Идентификаторы «где выполняем операцию» (`repoURL`, `workDir`) **ВСЕГДА** передаются как **отдельные аргументы метода**, а не внутри Options-структур. Options содержат только полезную нагрузку (payload — параметры операции).

```
// ✅ ПРАВИЛЬНО: идентификатор — аргумент метода, payload — в Options
Clone(ctx, repoURL, opts)
CreateBranch(ctx, workDir, opts)
Commit(ctx, workDir, opts)
CreatePullRequest(ctx, repoURL, opts)
ListPullRequests(ctx, repoURL, opts)

// ❌ НЕПРАВИЛЬНО: идентификатор зашит внутрь Options
Clone(ctx, CloneOptions{RepoURL: ...})
CreateBranch(ctx, BranchOptions{WorkDir: ...})
```

**Исключение:** методы с 1-2 простыми параметрами (без Options-структуры) передают `repoURL`/`workDir` напрямую: `ValidateAccess(ctx, repoURL)`, `GetDiff(ctx, repoURL, base, head)`, `GetLocalDiff(ctx, workDir, base, head)`.

**⚠️ КРИТИЧНО:** Нельзя передать `repoURL` в `exec.Command("git diff", ...)` — нужен локальный путь. Нельзя передать `workDir` в GitHub REST API — нужен URL. Интерфейс **жёстко** разделяет эти два мира. `LocalGitProvider` возвращает `ErrNotImplemented` для всех `repoURL`-методов, кроме `ValidateAccess` (через `git ls-remote`).

### 6. Атомарные `Commit` и `Push` vs комбинированный `CommitAndPush`

**Выбор: оба варианта — атомарные `Commit` + `Push` И convenience `CommitAndPush`.**

**Сценарии, требующие разделения:**
- Агент (Claude Code) делает несколько логических коммитов в рамках одной задачи: `Commit` × N → `Push` × 1.
- Пуш упал из-за сети — retry только `Push`, не пересоздавая коммит.
- Промежуточный коммит для сохранения прогресса без немедленного пуша.

**`CommitAndPush` остаётся** для простого сценария (один коммит → один пуш), чтобы не заставлять вызывающий код делать два вызова в тривиальных случаях. Реализация: `CommitAndPush = Commit + Push`.

### 7. Remote vs Local пары: `GetDiff`/`GetLocalDiff` и `GetFileContent`/`GetLocalFileContent`

**Паттерн: для каждой операции чтения — две версии (remote API + local git CLI).**

| Remote (repoURL) | Local (workDir) | Реализация local |
|---|---|---|
| `GetDiff(ctx, repoURL, base, head)` | `GetLocalDiff(ctx, workDir, base, head)` | `git diff base..head` |
| `GetFileContent(ctx, repoURL, branch, path)` | `GetLocalFileContent(ctx, workDir, ref, path)` | `git show ref:path` |
| `ListBranches(ctx, repoURL, prefix)` | `ListLocalBranches(ctx, workDir, prefix)` | `git branch -a --list` |
| `DeleteBranch(ctx, repoURL, branch)` | `DeleteLocalBranch(ctx, workDir, branch)` | `git branch -d` |

**Почему не один метод с перегрузкой:**
- `repoURL` (e.g. `https://github.com/org/repo`) и `workDir` (e.g. `/tmp/clone-abc123`) — принципиально разные аргументы. Одна функция не может принять оба.
- `GetDiff(repoURL)` вызывает **GitHub REST API** (HTTP-запрос). `GetLocalDiff(workDir)` вызывает **`exec.Command("git", "diff")`** (процесс на диске). Смешивать семантику опасно.
- `LocalGitProvider` не имеет remote API → `GetDiff` и `GetFileContent` возвращают `ErrNotImplemented`. Если бы был один метод, LocalGitProvider не мог бы предоставить diff/file-reading вообще.

**Когда какой использовать:**

| Сценарий | Метод |
|---|---|
| Ревьюер анализирует PR **после пуша** | `GetDiff(repoURL, ...)` |
| Агент проверяет diff **до пуша** (self-review) | `GetLocalDiff(workDir, ...)` |
| Загрузка файла для контекста агента (из remote) | `GetFileContent(repoURL, ...)` |
| Чтение файла из локального клона (в sandbox/утилита) | `GetLocalFileContent(workDir, ...)` |

### 9. `ListBranches` и пагинация

**MVP:** `ListBranches` возвращает `[]string` без пагинации. Фильтрация по `prefix` — основной способ ограничения выборки (наш код создаёт ветки `task/<uuid>`, prefix `task/` достаточен).

**Post-MVP:** GitHub/GitLab API отдают ветки постранично. Когда потребуется поддержка больших репозиториев с тысячами веток, заменить на:
```go
ListBranches(ctx context.Context, opts ListBranchesOptions) (*BranchList, error)
```
где `ListBranchesOptions` содержит `RepoURL`, `Prefix`, `Limit`, `Cursor`, а `BranchList` — `Items []string` + `NextCursor string`.

### 10. `CommitOptions.Author` — структура, не строка

**Выбор: структурированный `Author { Name string; Email string }`.**

```go
// ❌ НЕПРАВИЛЬНО: строка, требует парсинга регулярками
Author: "DevTeam Bot <bot@devteam.io>"

// ✅ ПРАВИЛЬНО: структура, однозначно разбирается
Author: Author{Name: "DevTeam Bot", Email: "bot@devteam.io"}
```

**Обоснование:**
- Библиотека `go-git` (если будет использоваться под капотом) принимает `object.Signature{Name, Email}`.
- `git` CLI использует `--author="Name <email>"` — форматирование тривиально из структуры.
- Парсинг строки `"Name <email>"` ненадёжен (имя с `<>`, пробелы, unicode).
- Тип `Author` определён в `types.go` (задача 4.2).

### 11. `io.ReadCloser` контракт для чтения файлов и Diff

Методы чтения (`GetFileContent`, `GetLocalFileContent`) и получения Diff (`GetDiff`, `GetLocalDiff`) возвращают `io.ReadCloser` для экономии памяти при обработке больших объемов данных (например, огромных диффов при рефакторинге). **Контракт:**
- Вызывающая сторона **ОБЯЗАНА** закрыть поток (`defer rc.Close()`) после чтения.
- `GetFileContent` / `GetDiff`: незакрытый поток = утечка HTTP-соединения (GitHub API).
- `GetLocalFileContent` / `GetLocalDiff`: незакрытый поток = утечка файлового дескриптора / pipe.
- Комментарий в интерфейсе явно предупреждает об этом.

### Пример 1: Основной пайплайн (Sprint 6) — Sandbox + GitProvider API

```go
func (o *OrchestratorService) executeTask(ctx context.Context, task *models.Task) error {
    project, _ := o.projectSvc.GetByID(ctx, ..., task.ProjectID)

    // Использование инжектированной фабрики
    provider, err := o.gitFactory.Create(string(project.GitProvider), decryptedCreds)
    if err != nil {
        return fmt.Errorf("create git provider: %w", err)
    }

    branchName := fmt.Sprintf("task/%s", task.ID)

    // 1. Запускаем sandbox-контейнер.
    //    Clone, Branch, Commit, Push — делает entrypoint.sh ВНУТРИ контейнера.
    //    GitProvider НЕ участвует на этом этапе.
    instance, err := o.sandbox.RunTask(ctx, sandbox.SandboxOptions{
        TaskID:      task.ID.String(),
        Backend:     "claude-code",
        Image:       "devteam/sandbox-claude",
        RepoURL:     project.GitURL,
        Branch:      branchName,
        Instruction: task.Description,
        EnvVars: map[string]string{
            "REPO_URL":    project.GitURL,
            "BRANCH_NAME": branchName,
            "GIT_TOKEN":   decryptedToken,
            "AUTO_PUSH":   "true",
        },
        Timeout: 30 * time.Minute,
    })

    // 2. Ждём завершения, забираем результат
    result, err := o.sandbox.WaitAndCollect(ctx, instance.ID)

    // 3. Проверяем поддержку PR (для LocalGitProvider это false)
    if provider.SupportsPullRequests() {
        // Создаем PR через API (GitHub/GitLab)
        pr, err := provider.CreatePullRequest(ctx, project.GitURL, gitprovider.PRCreateOptions{
            Title:      task.Title,
            Body:       task.Description,
            HeadBranch: branchName,
            BaseBranch: project.GitDefaultBranch,
        })

        // 4. Reviewer-агент анализирует PR через API
        diffStream, _ := provider.GetDiff(ctx, project.GitURL, project.GitDefaultBranch, branchName)
        defer diffStream.Close()
        // ... чтение diffStream ...
        files, _ := provider.ListPRFiles(ctx, project.GitURL, pr.Number)

        // 5. Добавляем комментарий ревьюера к конкретной строке кода
        provider.AddPRReviewComment(ctx, project.GitURL, pr.Number, gitprovider.PRReviewCommentOptions{
            Path:      "main.go",
            Line:      42,
            Body:      "Нужно добавить обработку ошибок",
            CommitSHA: "abc1234",
        })

        // 6. Отправляем итоговое решение ревьюера (Approve/Request Changes)
        provider.SubmitPRReview(ctx, project.GitURL, pr.Number, gitprovider.PRReviewOptions{
            Event: "APPROVE",
            Body:  "Код выглядит отлично, можно мержить.",
        })

        // 7. Мержим PR (если все проверки пройдены)
        provider.MergePullRequest(ctx, project.GitURL, pr.Number, gitprovider.PRMergeOptions{
            CommitTitle:   "Merge task " + task.ID.String(),
            CommitMessage: task.Description,
            MergeMethod:   "squash", // или "merge", "rebase"
            SHA:           "abc1234", // Защита от race conditions (мержим только то, что ревьюили)
        })
    } else {
        // Альтернативный пайплайн для LocalGitProvider
        // Ревьюер анализирует локальный diff и оставляет комментарии в чате
        diffStream, _ := provider.GetLocalDiff(ctx, project.GitURL, project.GitDefaultBranch, branchName)
        defer diffStream.Close()
        // ... чтение diffStream ...
        // ... отправка комментария в TaskMessage ...
    }

    // 8. Cleanup после merge
    err = provider.DeleteBranch(ctx, project.GitURL, branchName)
    if err != nil && !errors.Is(err, gitprovider.ErrBranchNotFound) {
        // Логируем ошибку, но не падаем (ветка могла быть удалена GitHub-ом при мерже)
    }

    return nil
}
```

```go
func (s *projectService) Create(ctx context.Context, userID uuid.UUID, req dto.CreateProjectRequest) (*models.Project, error) {

    return project, nil
```

### Пример 3: Утилитарный сценарий — Local-провайдер без sandbox

```go
func runLocalTask(workDir string, instruction string) error {
    // В реальном коде фабрика инжектится
    provider, _ := factory.Create("local", gitprovider.Credentials{})

    if err := provider.CreateBranch(ctx, workDir, gitprovider.BranchOptions{
        BranchName: "fix/lint-errors",
        BaseBranch: "main",
    }); err != nil {
        return err
    }

    // Агент работает локально ...

    sha, hasChanges, err := provider.Commit(ctx, workDir, gitprovider.CommitOptions{
        Message: "fix: resolve lint errors",
        Author:  gitprovider.Author{Name: "DevTeam Bot", Email: "bot@devteam.io"},
    })
    if err != nil {
        return err
    }
    if !hasChanges {
        fmt.Println("No changes to commit")
        return nil
    }

    diffStream, _ := provider.GetLocalDiff(ctx, workDir, "main", "fix/lint-errors")
    defer diffStream.Close()
    diffBytes, _ := io.ReadAll(diffStream)
    fmt.Printf("Commit %s, diff:\n%s\n", sha, string(diffBytes))

    return provider.Push(ctx, workDir, gitprovider.PushOptions{
        Branch:  "fix/lint-errors",
    })
}
```
