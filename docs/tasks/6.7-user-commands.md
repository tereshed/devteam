# 6.7 Обработка пользовательских команд: `pause`, `cancel`, `resume`, `correct`

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/service/orchestrator_service.go`
**Статус:** ⬜ В планах

---

## Описание

Реализовать обработку пользовательских команд (`pause`, `cancel`, `resume`, `correct`) в `OrchestratorService`. Это позволит пользователю управлять процессом выполнения задач: приостанавливать работу агентов, полностью отменать выполнение, возобновлять после паузы, а также вносить корректировки (`correct`) в текущую задачу (например, изменять требования на лету).

---

## Зависимости

**Требует:**
- **6.4** — `OrchestratorService` (основной цикл управления).
- **7.4** — `EventBus` (подписка на команды управления от пользователя).
- **3.3** — `TaskService` для обновления статусов задач.

**Используется в:**
- **8.5** — `ConversationService` (передача команд от пользователя в оркестратор).

---

## Функциональные требования

### 1. Безопасность и авторизация (Security & RBAC)

- **Tenant isolation (защита от IDOR):** при обработке любой команды оркестратор обязан проверять право пользователя на управление задачей: сверять `user_id` из контекста аутентификации/метаданных команды с владельцем проекта или задачей в БД (или иной моделью RBAC/ABAC, принятой в продукте). Запрещено выполнять команды по чужому `task_id`, в том числе при событиях из `EventBus`.
- **Валидация ввода для `correct`:** команда принимает «новые вводные» — это вектор Prompt Injection и DoS (огромные строки → OOM). Требуется жёсткая валидация: лимит размера текста (на порядок ниже риска OOM, например сотни килобайт — конкретное значение зафиксировать в коде/конфиге), санитизация/экранирование перед передачей в промпт LLM и в логи.

### 2. Управление состоянием и ресурсами (State & Resource Management)

- **Pause:** при `pause` оркестратор ожидает завершения текущего атомарного шага (вызов LLM, выполнение в Sandbox). Если шаг не завершается в течение **таймаута graceful pause** (например, 30 с — значение конфигурируемое), оркестратор принудительно отменяет контекст выполнения (аналогично `cancel` на уровне шага), сохраняет промежуточное состояние в БД и переводит задачу в `paused`. Недопустимо бесконечное ожидание «зависшего» шага.
- **Cancel:** помимо отмены `context.Context`, требуется **гарантированное освобождение внешних ресурсов**: остановка и удаление Docker-контейнеров Sandbox, закрытие сетевых соединений, если `AgentExecutor` не обработал отмену — контекст сам по себе контейнеры не убивает. Задача переводится в `cancelled`.
- **Resume:** восстановление контекста задачи из БД и продолжение с сохранённой точки; учитывать актуальные артефакты и историю сообщений.
- **Correct:** принять валидированные вводные, обновить контекст/промпт текущей задачи и при необходимости перезапустить текущий этап (например, вернуть на стадию `Develop`).

### 3. Конкурентность и масштабирование (Concurrency & Scaling)

- **Не полагаться на `sync.Mutex` для статуса задачи:** мьютексы действуют только в пределах одного процесса; при нескольких инстансах оркестратора гонки при обновлении БД останутся.
- **Согласованность на уровне БД:** использовать **optimistic locking** (поле `version` или условное обновление по `updated_at`) либо **пессимистичные блокировки** (`SELECT … FOR UPDATE` в транзакции) при смене статуса задачи, чтобы исключить потерю обновлений при параллельной работе агента и пользователя.

### 4. Интеграция с EventBus и обработка ошибок

- **Подписка:** `OrchestratorService` подписан на топики/события команд управления; после успешной смены состояния публикуется событие для UI.
- **Idempotency:** повторная команда в уже достигнутом целевом состоянии не должна ломать систему (идемпотентная обработка или явный no-op).
- **Invalid state transitions:** если команда неприменима к текущему статусу (например, `resume` для `completed`), — игнорирование или понятная ошибка без паники; действия логируются.

---

## Критерии приёмки

- [ ] Реализована проверка прав (RBAC/ABAC): команды управления выполняются только для задач/проектов, к которым у пользователя есть право; попытка по чужому `task_id` отклоняется.
- [ ] Для `correct` задан лимит размера ввода и санитизация/экранирование перед LLM и логами.
- [ ] `pause` с таймаутом graceful pause: при превышении таймаута — принудительная отмена шага, сохранение состояния, переход в `paused`.
- [ ] При `cancel` гарантированно освобождаются внешние ресурсы (Sandbox: контейнеры; сетевые соединения), а не только отмена контекста.
- [ ] Смена статусов задачи согласована через optimistic locking или блокировки БД; не используется in-process mutex как единственная защита от гонок.
- [ ] Идемпотентность и корректная обработка недопустимых переходов без паник; структурное логирование без утечки секретов.
- [ ] Написаны **интеграционные тесты** (`//go:build integration`): конкурентное изменение статусов, отмена контекста и очистка ресурсов Sandbox, сценарий отклонения команды без прав.
- [ ] Написаны unit-тесты на валидацию ввода `correct` и переходы состояний там, где не требуется БД/EventBus.
