# 5.7 Сбор результата (`CopyFromContainer` → парсинг `status.json` + diff)

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/docker_runner.go` (точка вызова после завершения контейнера, например из **`containerWaitLoop`** / **`composeFinalStatus`**); при росте логики — вспомогательные файлы пакета **`sandbox`** (например `collect_artifacts.go`, `status_json.go`, `truncate.go`) **без** импорта `gin`, `handler`, GORM. Константы путей и **`CodeResult`** — **`backend/internal/sandbox/types.go`** (**5.4**).  
**Статус:** ✅ Выполнена

---

## Описание

После того как контейнер перешёл в **exited/stopped** (см. **`ContainerWait`** в **5.5**), раннер обязан **забрать артефакты** с файловой системы контейнера через Docker Engine API (**`client.CopyFromContainer`**, аналог CLI `docker cp`) и заполнить **`SandboxStatus.Result`** типом **`CodeResult`** (**5.4**), не нарушая лимиты памяти (**`CodeResultMaxArtifactBytes`**) и контракт ошибок пакета **`sandbox`**.

**Важно по именам файлов:** в строке README фигурирует «`result.json`» как продуктовый образец; **единая точка правды в репозитории** — машиночитаемый **`status.json`** по пути **`StatusJSONPath`** (`/workspace/status.json`), который гарантированно пишет **`deployment/sandbox/claude/entrypoint.sh`** (**5.2**) через **`finalize()`** на **`EXIT`** (в т.ч. при ошибках и SIGTERM). Отдельного **`result.json`** в текущем entrypoint **нет** — задача **5.7** описывает парсинг **`status.json`** + бинарных текстовых артефактов (**`full.diff`**, **`agent.log`** и при необходимости **`changes.txt`**).

**Границы задачи:** только **копирование из контейнера**, распаковка **tar**-архива ответа API, **парсинг JSON**, маппинг в **`CodeResult`**, усечение больших полей, санитизация путей для **`FilesChanged`**. Стриминг логов до завершения — **5.6**; политика SIGTERM/SIGKILL и бизнес-таймаут — **5.8**; лимиты CPU/Memory в **`HostConfig`** — **5.9**.

## Согласование успеха (`SandboxStatus.Status` vs `CodeResult.Success`)

Сейчас **`composeFinalStatus`** (**5.5**) выставляет **`SandboxStatus.Completed`** при **`exitCode == 0`** и не заполняет **`Result`**. После **5.7**:

1. **База:** Docker **`ExitCode`** и флаги **`OOMKilled`**, **`timedOut`**, **`stoppedByRunner`** остаются первичными для **`SandboxStatus.Status`** (как сейчас), за исключением явно описанных случаев.
2. **`status.json`:** если файл **прочитан и распарсен**, **`CodeResult.Success`** берётся из поля **`success`**, **но** при **`OOMKilled`**, **`SandboxStatusTimedOut`**, **`SandboxStatusStopped`** (отмена пользователем) итоговый **`SandboxStatus.Status`** **не** должен становиться **`completed`«только потому что** JSON сказал `success: true`» — политика: **инфраструктурный исход** (OOM/таймаут/стоп) **строже** маркетингового флага в JSON.
3. **Расхождение** `exit_code` из JSON и Docker: для API зафиксировать одно правило (например, **`SandboxStatus.ExitCode`** всегда код Docker; JSON — в **`CodeResult`** только как метаданные или в лог).
4. Если **`status.json` отсутствует** или JSON **битый** после нормального завершения контейнера: **`Result == nil`** или **`Result`** с **`Success: false`** (выбрать одно и покрыть тестом **5.13**); **`SandboxStatus.Status`** — **`failed`** или **`completed`** по политике «отсутствие артефактов = failed» (рекомендация: **`failed`**, чтобы оркестратор не считал задачу успешной без контрактных данных).

## Реализация: `CopyFromContainer`

### Вызов API

- Первый аргумент — **`context.Context`** с отменой/дедлайном (тот же **`ctx`**, что у вызывающего **`Wait`**, или короткий дочерний — зафиксировать; при отмене **`Wait`** сбор артефактов не должен зависнуть навсегда).
- Параметр **`srcPath`** — **только** литералы из констант **`types.go`** (**`StatusJSONPath`**, **`FullDiffPath`**, **`AgentLogPath`**), **не** строки из JSON/env пользователя.
- Ответ — **tar**-поток: итерировать записи через **`archive/tar`**. Для **каждой** записи обязательно смотреть **`Header.Typeflag`**:
  - обрабатывать **только** **`tar.TypeReg`** (обычный файл);
  - **`tar.TypeSymlink`**, **`tar.TypeLink`**, **`tar.TypeDir`** и прочие типы — **молча пропускать** (**`Next()`** без чтения содержимого по ссылке), чтобы исключить подмену путей и обход каталогов через symlink/hardlink внутри ответа движка;
  - **не** вызывать **`os.Readlink`** / **не** резолвить symlink в пути из заголовка.
- Для принятого **`TypeReg`:** проверка **`Header.Name`** на **path traversal** (компоненты **`..`**, абсолютные пути, выход за ожидаемый префикс внутри архива) — как и раньше; при нарушении — пропуск записи или фатальная ошибка с **`ErrSandboxDocker`** (одно поведение зафиксировать в коде и **5.13**).
- **Tar bomb (много записей):** опциональное усиление для PR — верхняя граница числа итераций по заголовкам (например порядка **10⁴**), после которой прервать разбор с ошибкой; иначе злонамеренный/битый слой может заставить CPU крутиться в цикле.

### Ошибки и частичные данные

- **`not found`** для отдельного файла: если **`status.json`** обязателен — фатально для «успешного результата»; если **`full.diff`** отсутствует — **`Diff` пустой**, без паники.
- Любая ошибка Docker SDK — обёртка **`fmt.Errorf("copy from container: …: %w", errors.Join(ErrSandboxDocker, err))`** (или согласованный префикс), **без** утечки внутренних путей демона в пользовательский JSON.

### Усечение (**`CodeResultMaxArtifactBytes`**) и закрытие потока (**deadlocks / утечки**)

- Читать тело записи **`TypeReg`** через **`io.LimitReader(body, limit)`**, где **`limit = min(Header.Size, CodeResultMaxArtifactBytes)`** для накопления в буфер **или** **`limit = Header.Size`** с последующим усечением строки — главное: **никогда** не читать **`Header.Size`** целиком в неограниченный **`[]byte`**.
- После чтения нужных **`N`** байт из записи размером **`Header.Size`**, **остаток** этой же записи в потоке **`tar.Reader` обязан быть дочитан до конца** (**`io.Copy(io.Discard, …)`** на хвост записи с известным оставшимся размером), иначе следующий **`Next()`** даст смещение и **коррупцию** разбора; это же снимает риск **недочитанного** HTTP/TCP к демону при преждевременном выходе из функции.
- Сразу после успешного **`CopyFromContainer`** — **`defer readCloser.Close()`** (или эквивалент «один **`Close`** на всех путях выхода»): **первой** строкой после присваивания **`ReadCloser`**, до долгого парсинга. При ошибке до **`defer`** — **`Close`** в ветке ошибки. Цель: не оставлять висящие соединения и FD при сценарии «файл 500 MiB, в память берём только первые **1 MiB**».
- Усечение до **`CodeResultMaxArtifactBytes`** для строк **`Diff`/`Output`:** после чтения, если нужно подрезать по рунам vs по байтам — зафиксировать одну политику в коде (как в черновике спеки).

### Нормализация строк из JSON (edge cases)

- Для всех строковых полей, мапящихся в **`CodeResult`** (**`BranchName`**, **`CommitHash`** и любые будущие), применять **`strings.TrimSpace`**; результат **только из пробелов** → пустая строка **`""`** (не **`" "`**).
- Для **`branch_name`:** пустое после нормализации — валидное «нет имени ветки в артефакте»; **не** считать это паникой JSON; оркестратор в **6.x** решает, показывать ли предупреждение в UI.

### Производительность

- Потоковое чтение с **`io.LimitReader`** и явный **drain** хвоста записи tar — целевой паттерн: **O(1)** по памяти относительно размера файла в контейнере (хост не загружает гигабайтные **`full.diff`** целиком).

### `FilesChanged` (**5.4**)

Контракт **`CodeResult.FilesChanged`:** надёжный источник — **`git diff --name-only -z`** внутри контейнера, записанный в **отдельный файл** (например будущая константа **`ChangedFilesPath`** и доработка **5.2**). Текущий **`changes.txt`** — это **`--stat`**, парсинг имён файлов из него **хрупкий** и **не** является обязательным MVP.

**Политика для MVP 5.7 (выбрать одну в PR и отразить в коде):**

| Вариант | Поведение |
|---------|-----------|
| **A** | **`FilesChanged == nil`** (или пустой слайс) до появления артефакта **`name-only`** в **5.2** |
| **B** | В той же задаче **5.7** (или малом доп. PR к **5.2**) добавить в entrypoint запись **`git diff --cached --name-only -z`** в файл и читать его в **5.7** |

В любом случае: каждый путь — **`filepath.Clean`** и проверка, что он **не** выходит за пределы логического корня репозитория (для отображения в UI), см. комментарий к **`CodeResult`** в **5.4**.

### Где вызывать в жизненном цикле

- **После** известного «контейнер не running» и **до** **`closeDone()`** в **`containerWaitLoop`**, либо централизованно из **`composeFinalStatus`** при наличии **`ctx`** и **`client`** — главное: **до** **`Cleanup`**, пока слой контейнера существует.
- **Не** дублировать тяжёлый сбор при каждом **`GetStatus`**: для **`GetStatus`** при **`exited`** допускается ленивый разовый кеш в **`instanceState`** (если уже собрано для **`Wait`**) — зафиксировать потокобезопасность с **`instanceState.mu`**.

## Безопасность

- **Path traversal:** не доверять полям **`artifacts.*`** из JSON для открытия путей в tar и на диске; только константы.
- **Tar: symlinks, hardlinks, каталоги:** см. раздел «Реализация: `CopyFromContainer`» — только **`tar.TypeReg`**; остальные **`Typeflag`** не обрабатывать содержимым.
- **Секреты в памяти и логах (`CodeResult`):** поля **`Output`** и **`Diff`** могут содержать API-ключи, URL с токенами, содержимое **`.env`**. Полагаться только на «не логировать в **sandbox**» **недостаточно**: любой будущий **`slog.Debug("sandbox", "result", result)`** внутри пакета утечёт секреты в **ELK/Kibana**.

  **Обязательное требование для PR:** для **`CodeResult`** реализовать **`slog.LogValuer`** (**`LogValue() slog.Value`**) — предпочтительный вариант в Go **1.21+**, чтобы **`slog`** при логировании всей структуры **автоматически** подставлял безопасное представление.

  Минимальное содержание **`LogValue`** (или эквивалента):

  - **`Success`**, **`TokensUsed`**, **`Duration`**, **`BranchName`**, **`CommitHash`**, **`len(FilesChanged)`** — как есть (ветка/commit после **`TrimSpace`** уже нормализованы);
  - **`Diff`** и **`Output`** — **не** сырые строки: либо **`"<redacted len=N>"`**, либо **жёстко усечённый** префикс (например **первые 256 рун/байт**) + суффикс **`…`**, **без** копирования полного секрета в **`slog.Value`** (достаточно длины и опционально хэша для корреляции);
  - **`FilesChanged`** — только **количество** и/или усечённый список имён (без полных путей длиной мегабайты), либо **`"<n paths>"`**.

  **Альтернатива / дополнение:** кастомный **`String() string`** с той же политикой маскирования/усечения для **`fmt`**, если команда избегает **`slog`** в пакете — но при использовании **`slog`** без **`LogValuer`** **`String()`** для структуры **не** вызывается автоматически, поэтому **`LogValuer` обязателен**, если **`CodeResult`** может попасть в **`slog`** как поле структуры.

  Документировать в комментарии к типу: **`fmt.Printf("%+v", result)`** и **`%#v`** по-прежнему обходят **`LogValue`** — для отладки использовать только явные безопасные хелперы.

- **Логи уровня INFO по умолчанию:** не печатать **`Diff`/`Output`** целиком даже вручную; маскирование на границе **`handler`/`service`** остаётся дополнительным слоём (**6.x**).
- **`ValidateSandboxID`** до любого RPC (уже в **`Wait`/`GetStatus`**).

## Границы слоёв и `backend.mdc`

Задача **5.7** — **только** пакет **`internal/sandbox`**. Публикация **`CodeResult`** в HTTP/Swagger/MCP — в задачах оркестратора/хендлеров (**6.x**), с отдельными DTO при необходимости.

### Декомпозиция (DRY, без God Object)

Логику **`CopyFromContainer`**, разбор tar (**`Typeflag`**, drain), парсинг **`status.json`**, нормализацию строк и построение **`CodeResult`** выносить из раздутого **`docker_runner.go`** в небольшие файлы того же пакета (**`collect_artifacts.go`**, **`status_json.go`**, при необходимости **`code_result_logvalue.go`**) с чёткими зависимостями; **`docker_runner.go`** оставляет только вызовы и wiring к **`instanceState`**.

## Критерии приёмки

- [ ] После завершения контейнера **`Wait`** возвращает **`SandboxStatus`** с заполненным **`Result`** (когда артефакты доступны и JSON валиден) или согласованной политикой при ошибках/отсутствии файлов.
- [ ] Используется **`CopyFromContainer`** с **`ctx`**; tar-ответ корректно разбирается; **`ReadCloser`** ответа: **`defer Close()`** сразу после получения; **`Close`** на всех путях выхода (**нет утечек FD** / висящих TCP).
- [ ] При усечении большого файла: чтение через **`io.LimitReader`** (или эквивалент); **остаток** записи tar **дочитан** (**`io.Discard`**) до следующего **`Next()`**; интеграционный или юнит-сценарий в **5.13** на «размер > лимита» не оставляет парсер в неконсистентном состоянии.
- [ ] В tar обрабатываются **только** **`tar.TypeReg`**; **`TypeSymlink`**, **`TypeLink`**, каталоги — пропуск без чтения по ссылке (**тест 5.13** с мок-архивом при наличии).
- [ ] **`Diff`** и **`Output`** никогда не превышают **`CodeResultMaxArtifactBytes`** в памяти после усечения; чтение **потоковое**, без загрузки всего файла в **`[]byte`**.
- [ ] **`CodeResult`** реализует **`slog.LogValuer`** с маскированием/усечением **`Diff`** и **`Output`**; есть тест (**5.13**): логирование через **`slog`** не содержит подстроки известного секрета, вложенного в мок **`Output`**/**`Diff`**.
- [ ] **`branch_name`** / **`commit_hash`**: нормализация **`strings.TrimSpace`**; строка из одних пробелов → **`""`** (**тест 5.13**).
- [ ] Парсинг **`status.json`** tolerant к дополнительным полям; битый JSON даёт предсказуемый исход (**тест 5.13**).
- [ ] Ошибки Docker обёрнуты с sentinel’ами **`sandbox`**, без «голого» SDK в сообщениях для потребителей вне пакета (согласовано с **5.5**/**5.6**).
- [ ] Зафиксирована политика **`FilesChanged`** (вариант **A** или **B** выше) и отражена в коде + тестах.
- [ ] Логика сбора вынесена из **`docker_runner.go`** в отдельные файлы пакета **`sandbox`** (нет «бога-объекта» на тысячи строк в одном файле).
- [ ] Комментарии в коде ссылаются на эту спецификацию и на **5.2**/**5.4**.
- [ ] В таблице README Sprint 5 у пункта **5.7** — ссылка «детали» на этот файл.
