# 7.4 Event Bus: внутренний pub/sub для уведомлений между сервисами

**Sprint:** 7 — WebSocket и Реалтайм (Backend)
**Файл:** `backend/internal/domain/events/eventbus.go` + `backend/internal/ws/hubbridge.go`
**Статус:** ⬜

---

## Описание

Реализовать **`EventBus`** — внутренний **in-process pub/sub**, через который **продюсеры** (`TaskService` 7.5, `SandboxRunner` 7.6, `OrchestratorService` 6.4) **публикуют доменные события**, а **подписчик** (мост на `ws.Hub` 7.1) **транслирует их в WebSocket-клиентов** в формате конверта **7.3**.

EventBus решает три проблемы одновременно:

1. **Развязывает сервисы и WS-инфраструктуру.** `TaskService` не должен знать про существование Hub'а или WebSocket'ов — он публикует «доменное событие изменения статуса». Это правильное направление зависимостей по Clean Architecture (`backend.mdc` §2.1).
2. **Унифицирует точку контроля** — маскирование секретов, whitelist `metadata`, преобразование доменных моделей в `ws.*Data` (7.3) происходят в **одном** месте — в подписчике-мосте, а не размазаны по всем продюсерам.
3. **Готовит миграционный путь к Redis Pub/Sub / NATS** — когда придёт горизонтальное масштабирование (несколько инстансов backend), интерфейс `EventBus` останется прежним, поменяется только реализация. **MVP — in-memory, без Redis.**

## Контекст использования

```
TaskService (7.5)        ─publish──┐
SandboxRunner (7.6)      ─publish──┤
OrchestratorService (6.4)─publish──┤
                                   ▼
                            ┌────────────┐
                            │  EventBus  │  (in-process pub/sub)
                            └─────┬──────┘
                                  │ subscribe
                                  ▼
                          HubBridge (this task)
                          ─ конвертирует Event → ws.Envelope (7.3)
                          ─ маскирует секреты, фильтрует metadata
                                  │
                                  ▼
                            ws.Hub.SendToProject (7.1)
                                  │
                                  ▼
                              WS-клиенты
```

**Важно (Clean Architecture):** `EventBus`, `DomainEvent` и структуры событий располагаются в **`backend/internal/domain/events/`** — независимый пакет, не зависящий от `ws/`. Это гарантирует правильное направление зависимостей: и `TaskService` (service), и `HubBridge` (ws) зависят от `domain/events`, но сервисы **не зависят** от `ws/`. Подробнее — `backend.mdc` §2.1.

Согласно `README.md` (Sprint 7): EventBus — связующее звено между сервисами и Hub'ом, **обязательное условие** для **7.5** и **7.6**.

## Доменные события (типы)

EventBus оперирует **доменными типами**, а не сразу `ws.Envelope`. Это важно: бизнес-сервис не должен знать про JSON-схему фронта.

```go
// backend/internal/domain/events/eventbus.go

// DomainEvent — маркер интерфейс. Реализуется конкретными событиями ниже.
type DomainEvent interface {
    domainEvent()    // приватный метод-маркер (нельзя реализовать снаружи пакета)
    GetProjectID() uuid.UUID
}

// TaskStatusChanged — переход статуса задачи (см. 3.6).
type TaskStatusChanged struct {
    ProjectID       uuid.UUID
    TaskID          uuid.UUID
    ParentTaskID    *uuid.UUID
    Previous        string  // task.StatusType
    Current         string
    AssignedAgentID *uuid.UUID
    AgentRole       string
    ErrorMessage    string  // только при Current == failed
    OccurredAt      time.Time
}

// TaskMessageCreated — новое сообщение в логе задачи (см. 3.2 / main.mdc §2.5).
type TaskMessageCreated struct {
    ProjectID   uuid.UUID
    TaskID      uuid.UUID
    MessageID   uuid.UUID
    SenderType  string
    SenderID    uuid.UUID
    SenderRole  string
    MessageType string
    Content     string
    Metadata    map[string]any   // raw, фильтрация — в подписчике
    OccurredAt  time.Time
}

// SandboxLogEmitted — одна строка лога из контейнера (см. 5.x).
type SandboxLogEmitted struct {
    ProjectID  uuid.UUID
    TaskID     uuid.UUID
    SandboxID  string
    Stderr     bool
    Line       string           // raw, маскирование — в подписчике
    Seq        int64
    Truncated  bool
    OccurredAt time.Time
}

// PipelineErrored — ошибка пайплайна / executor'а / sandbox'а (см. 6.x).
type PipelineErrored struct {
    ProjectID  uuid.UUID
    TaskID     *uuid.UUID
    Code       string           // из enum 7.3 error.code
    Message    string           // safe-string из константного перечня
    OccurredAt time.Time
}
```

`domainEvent()`-маркер запрещает реализовать `DomainEvent` извне пакета, что гарантирует **закрытый набор** типов и компилятор-проверяемую полноту обработки в подписчике.

## Интерфейс EventBus

```go
// EventBus — публикация и подписка на доменные события в одном процессе.
//
// Контракт:
//   - Publish: НЕБЛОКИРУЮЩАЯ. Если все подписчики заняты — событие дропается с инкрементом метрики
//     ws_eventbus_dropped_total. ОБЯЗАТЕЛЬНО логируется WARN с типом события.
//   - Subscribe: возвращает канал событий и функцию unsubscribe. Один subscriber = один канал.
//   - Close: останавливает Bus, закрывает все subscriber-каналы.
type EventBus interface {
    Publish(ctx context.Context, ev DomainEvent)
    Subscribe(buffer int) (<-chan DomainEvent, func())
    Close()
}

// NewInMemoryBus — реализация для одного инстанса backend.
// При переходе на Redis Pub/Sub интерфейс не меняется.

```

## Функциональные требования

### 1. Архитектура: один центральный fan-out

- Bus хранит **множество подписчиков**; каждый — буферизованный канал событий.
- На `Publish` Bus **итерируется** по подписчикам и **неблокирующе** пишет в каждый канал. Slow-subscriber **не должен** держать продюсеров.
- **НЕ использовать `sync.Map`** — производительность хуже `map + RWMutex` на наших паттернах.

### 2. Backpressure (slow subscriber isolation) — КРИТИЧНО

Если очередь подписчика переполнена:

- **Дроп события для этого подписчика** (а не для всех).
- Инкремент `ws_eventbus_dropped_total{subscriber=...,event_type=...}`.
- WARN-лог 1 раз в N сек (rate-limit чтобы не залить логи), с указанием subscriber-ID и типа события.
- **НЕ закрывать** канал подписчика — это сломает мост на Hub. Восстановиться через рестарт сервиса либо при переходе на персистентный Bus.

**Альтернатива (рассмотреть на ревью):** автоматическое отключение подписчика после N подряд дропов с пересоздание моста. В MVP — оставляем простой дроп с метрикой.

### 3. Неблокирующий Publish (Copy-on-Write)

На горячем пути (тысячи событий/сек от `SandboxRunner.StreamLogs`) **НЕ ДОПУСТИМЫ** аллокации в куче и мьютексы. Используем Copy-on-Write с `atomic.Pointer`:

```go
type inMemoryBus struct {
    subs   atomic.Pointer[[]*subscription] // nil-safe copy-on-write
    mu     sync.Mutex   // только для Subscribe/Unsubscribe/Close
    close  atomic.Bool
    metrics *Metrics
}

// Snapshot подписчиков — lock-free read на горячем пути
func (b *inMemoryBus) Publish(ctx context.Context, ev DomainEvent) {
    // ── 1. Early fail: валидация на входе ─────────────────────────────────
    if ev == nil || ev.GetProjectID() == uuid.Nil {
        // невалидное событие → WARN-лог и дроп (не паникуем, не маршрутизируем мусор)
        return
    }

    // ── 2. Lock-free чтение snapshot подписчиков ───────────────────────────
    subsPtr := b.subs.Load()
    if subsPtr == nil || b.close.Load() {
        return
    }

    // ── 3. Рассылка без блокировки ─────────────────────────────────────────
    for _, s := range *subsPtr {
        select {
        case s.ch <- ev:
        case <-ctx.Done():
            return
        default:
            // slow subscriber → drop с метрикой и rate-limited логом
            b.metrics.IncDropped(s.name, eventTypeName(ev))
        }
    }
}
```

**КРИТИЧНО:**
- `atomic.Pointer` позволяет читать snapshot без аллокаций и без мьютекса на горячем пути.
- `Subscribe`/`Unsubscribe` берут `sync.Mutex` и перезаписывают `atomic.Pointer` — редкие операции, там аллокация допустима.
- Под капотом Go гарантирует safe read через atomic.

### 4. Subscribe / Unsubscribe

- `Subscribe(buffer int)` создаёт канал размером `buffer` (рекомендация: **256 для моста на Hub**, **1024 для метрик-агрегатора**).
- Возвращает кортеж `(<-chan DomainEvent, unsubscribe func())`.
- `unsubscribe`:
  - Idempotent (повторный вызов — no-op).
  - Удаляет подписку из множества подписчиков (atomic swap). **НЕ закрывает канал** — это предотвращает панику `send on closed channel` при конкурентном Publish. Канал буферизованный, после удаления из списка на него нет ссылок от Bus, и GC соберёт его когда подписчик закончит чтение.
  - Закрытие каналов — **только** в `bus.Close()`, когда гарантировано что Publish больше не пишет.
  - **Подписчики ОБЯЗАНЫ** вызвать `unsubscribe` в `defer` своего цикла, иначе утечка горутины.

### 5. Graceful Close

- `Close()` отменяет внутренний контекст Bus, делает `b.close.Store(true)`, затем:
  - Для каждого подписчика: закрывает канал (`close(s.ch)`), чтобы разблокировать читающие горутины.
  - Очищает список подписчиков.
- Дальнейшие `Publish` — no-op (проверяют `b.close.Load()` и сразу возвращаются).
- Идемпотентен.

### 6. HubBridge — отдельный подписчик

`HubBridge` — **обязательная компонента этой задачи**. Это подписчик на `EventBus`, который:

1. Через `bus.Subscribe(256)` получает канал `<-chan DomainEvent`.
2. В отдельной горутине читает события, **матчит по type-switch** на `DomainEvent`, конвертирует в `ws.*Data` (7.3).
3. **Применяет sanitize-pipeline** (см. п. 7).
4. Сериализует через `ws.Marshal*` (7.3).
5. Отправляет в Hub: `hub.SendToProject(projectID, msgType, payload)`.

```go
type HubBridge struct {
    bus  EventBus
    hub  *Hub
    sec  SecretScrubber  // pkg/secrets — общий сканер секретов
    log  Logger
}

func NewHubBridge(bus EventBus, hub *Hub, sec SecretScrubber, log Logger) *HubBridge { /* ... */ }

// Run блокируется до закрытия контекста. Запускается в отдельной горутине из main.
func (b *HubBridge) Run(ctx context.Context) {
    ch, unsub := b.bus.Subscribe(256)
    defer unsub()

    for {
        select {
        case <-ctx.Done():
            return
        case ev, ok := <-ch:
            if !ok {
                return
            }
            b.dispatch(ev)
        }
    }
}
```

### 7. Sanitize-pipeline (КРИТИЧНО — Безопасность)

Все доменные события проходят через единые фильтры **внутри HubBridge** перед сериализацией:

- **`task_message.metadata`** — whitelist ключей (см. 7.3): `tokens_used`, `model`, `duration_ms`, `cost_usd`. Остальное — отбросить. **Значения** строковых ключей (например, `model`) тоже прогоняются через `SecretScrubber.Scrub()` — агент мог случайно вставить токен в кастомное поле.
- **`agent_log.line`** — `SecretScrubber.Scrub(line)`. Маски: `sk-…`, `ghp_…`, `xoxb-…`, `Authorization: Basic …`, JWT-подобные. Источник — общий пакет `pkg/secrets`, **не** дублировать.
- **`PipelineErrored.Message`** — должен быть из enum-перечня; иначе фоллбек на `internal_error` + WARN-лог. Не пускать «raw error» в WS.
- **`task_message.content`** тоже прогоняется через `SecretScrubber.Scrub` (агенты любят вставить ключ в коммит-комментарий «для удобства»).

### 8. Контекст и трейсинг

- `Publish(ctx, ev)` — продюсер передаёт `ctx` своего запроса. Bus уважает `ctx.Done()` (см. п. 3).
- Если в `ctx` есть trace-id — пробрасывать в WARN-логи дропов для корреляции.

### 9. Метрики (Prometheus)

| Метрика | Тип | Лейблы |
|---------|-----|--------|
| `ws_eventbus_published_total` | Counter | `event_type` |
| `ws_eventbus_dropped_total` | Counter | `event_type`, `subscriber` |
| `ws_eventbus_subscribers` | Gauge | (без лейблов) |

### 11. ЗАПРЕТЫ

- **Запрещено** напрямую звать `hub.SendToProject` из сервисов в обход EventBus — это вернёт нас к размазанной sanitize-логике.
- **Запрещено** хранить состояние/данные в EventBus (это шина, не БД). Persistence событий — отдельная задача (`task_messages` уже сохраняются в БД через `TaskService`, см. 3.2).
- **Запрещено** делать Bus глобальной переменной (см. `backend.mdc` §2.2).

## Дизайн-решения

1. **In-memory + интерфейс под Redis:** YAGNI для масштабирования прямо сейчас, но дешёвая страховка на завтра.
2. **Доменные события в `domain/events/`:** правильное направление зависимостей — `ws` знает про доменные типы, а **домен НЕ знает про `ws`**. Это позволит позже добавить других подписчиков (Slack-нотификации, аудит-лог) без изменения продюсеров.
3. **Единый sanitize-pipeline в HubBridge:** одна точка контроля = реальная защита от утечек. Размазанная фильтрация — путь к инциденту.
4. **Drop вместо back-pressure для продюсеров:** продюсеры (особенно `SandboxRunner.StreamLogs`) не должны блокироваться из-за того, что какой-то клиент тормозит. Real-time messaging — потеря событий допустимее, чем встанувший пайплайн.
5. **Copy-on-Write с `atomic.Pointer`:** lock-free read на горячем пути (Publish), мьютекс только на редких операциях (Subscribe/Unsubscribe). Ноль аллокаций и мьютексов при стриминге логов.
6. **Early fail валидация в Publish:** `nil`-события и события с `uuid.Nil` дропаются с WARN-логом. Не паникуем, не маршрутизируем мусор.
7. **Unsubscribe не закрывает канал:** только удаляет из списка. Закрытие — в `bus.Close()`. Предотвращает панику `send on closed channel`.
8. **Маркер-метод `domainEvent()`:** компилятор гарантирует closed-set type switch'а в HubBridge.
9. **Метрики с самого начала:** без них дроп событий — невидимый. Продакшн ляжет молча.
