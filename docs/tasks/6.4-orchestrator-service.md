# 6.4 Реализация: `OrchestratorService` — основной цикл управления

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/service/orchestrator_service.go`
**Статус:** ⬜ В планах

---

## Описание

Реализовать **`OrchestratorService`** — центральный координатор, управляющий жизненным циклом выполнения запроса пользователя. Оркестратор координирует работу различных агентов (Planner, Developer, Reviewer, Tester), создавая и назначая задачи, а также обрабатывая их результаты.

---

## Зависимости

**Требует:**
- **3.3** — `TaskService` для управления задачами.
- **6.1** — интерфейс `AgentExecutor`.
- **6.2** — `LLMAgentExecutor` для ролей Planner, Reviewer.
- **6.3** — `SandboxAgentExecutor` для ролей Developer, Tester.
- **Sprint 1** — модели `Project`, `Team`, `Agent`, `Task`.

**Используется в:**
- **8.5** — `ConversationService` (триггерит запуск оркестрации при новом сообщении).
- **7.4** — `EventBus` (публикация событий изменения статусов и подписка на команды управления).

## Функциональные требования

### 1. Декомпозиция и DRY (🏗️ Anti-God Object)
- **Координатор, а не исполнитель:** `OrchestratorService` должен делегировать логику специализированным компонентам.
- **Pipeline Engine / State Machine:** Вынести логику переходов между статусами и этапами в отдельный компонент. Оркестратор только вызывает переходы.
- **Context Builder:** Вынести сборку и фильтрацию контекста для агентов в отдельный хелпер.

### 2. Управление жизненным циклом (🚨 Resource Leaks & Context)
- **Context Propagation:** Во все методы **ОБЯЗАТЕЛЬНО** прокидывается `context.Context`.
- **Safe Cancellation:** Реализовать корректную обработку `<-ctx.Done()`. При отмене контекста оркестратор должен гарантировать остановку всех запущенных экзекуторов и очистку ресурсов (defer cleanup).
- **Recovery Mechanism (Zombie Tasks):** Реализовать механизм восстановления или перевода в `failed` "зависших" задач (например, при рестарте сервера), которые находятся в активном статусе дольше заданного таймаута (например, 1 час).

### 3. Безопасность и Маскирование (🔐 Security)
- **Secret Scrubbing:** Перед формированием `ExecutionInput` для LLM-агентов необходимо реализовать фильтрацию/маскирование секретов (API-ключи, пароли, токены).
- **Command/Flag Injection:** Пользовательский ввод и контекст должны экранироваться или передаваться в `SandboxAgentExecutor` исключительно безопасным путем (например, через запись в файл внутри контейнера), чтобы исключить выполнение произвольных команд.
- **Path Traversal Protection:** Строгая валидация путей файлов в артефактах, возвращаемых агентами (использование `filepath.Clean` и проверка изоляции внутри директории проекта). Запрещены пути вида `../../`.
- **Tenant Isolation (Defense in Depth):** На уровне сервиса должна выполняться проверка принадлежности задачи к проекту и наличия у пользователя прав на управление этим проектом (даже если проверка уже была в HTTP-хендлере).

### 4. Синхронизация и Потоки Событий (🏎️ Race Conditions & Event Flow)
- **Concurrency Control:** Использовать механизмы блокировок (мьютексы или optimistic locking в БД) для предотвращения состояния гонки при одновременном изменении статуса задачи.
- **Event Flow Control:** Оркестратор **публикует** события о смене статусов в `EventBus` (для UI), но **слушает только внешние команды** управления от пользователя (пауза, отмена), чтобы избежать бесконечных циклов обработки собственных событий.
- **Transactional Consistency:** Все изменения состояния задачи (смена статуса, сохранение артефактов, запись сообщений агентов) **ОБЯЗАНЫ** выполняться строго в рамках единой транзакции БД. В случае ошибки — полный Rollback.

### 5. Обработка ошибок и Валидация (🛡️ Resilience & Validation)
- **LLM Hallucinations & Invalid Output:** Строгая валидация ответов от `AgentExecutor` (JSON, структура). При ошибке парсинга — автоматический ретрай с указанием ошибки агенту.
- **Infrastructure Errors:** Обработка транзитных ошибок (429, 503, Docker issues) через паттерн Retry с экспоненциальным бэкоффом.
- **Panic Recovery:** Все асинхронные горутины (воркеры), запускаемые Оркестратором, должны содержать `defer recover()`. При панике задача переводится в `failed`, логируется стектрейс, приложение не крашится.

### 6. Лимиты и Производительность (⚡ Performance & Limits)
- **Iteration Limits:** Жесткий лимит на циклы исправления (`Review -> Develop`). При достижении лимита задача переходит в `failed`.
- **Context Window Management:** Лимитирование размера контекста (обрезка старых сообщений, ограничение количества чанков из векторной БД).
- **N+1 Query Problem:** Сбор истории сообщений и задач для `ContextBuilder` должен выполняться оптимизированными SQL-запросами (JOIN/IN).

### 7. Наблюдаемость (📊 Observability & Metrics)
- **Metrics Collection:** Сбор метрик выполнения (Prometheus/OTel): время нахождения задачи в каждом статусе, количество итераций (Review -> Develop), количество фейлов по типам, использование токенов на каждом шаге.
- **Structured Logging:** Использование `slog` с обязательными полями `project_id` и `task_id`. Запрещено логировать секреты.

## Критерии приёмки

- [ ] `OrchestratorService` декомпозирован: логика State Machine и сборки контекста вынесены в отдельные структуры.
- [ ] Во все методы прокидывается `context.Context`, реализована обработка `<-ctx.Done()` для предотвращения утечек ресурсов.
- [ ] Реализован механизм обработки "зависших" задач (Recovery/Timeout) при рестарте сервиса.
- [ ] Реализована защита от Race Conditions при параллельной смене статуса задачи агентом и пользователем.
- [ ] Отсутствуют бесконечные циклы событий: Оркестратор корректно публикует статусы в EventBus и слушает только внешние команды.
- [ ] Добавлена фильтрация/маскирование секретных данных перед отправкой в `ExecutionInput` и в логах.
- [ ] Пользовательский ввод экранируется/передается безопасно для исключения Command/Flag Injection.
- [ ] Реализована защита от Path Traversal при обработке путей к файлам из артефактов агентов.
- [ ] Реализована строгая валидация ответов LLM (JSON, структура) с механизмом автоматического ретрая при галлюцинациях.
- [ ] Сбор контекста (история, задачи) реализован без проблемы N+1 (оптимизированные запросы к БД).
- [ ] Реализована обработка инфраструктурных ошибок (Rate Limits, Network) через Retry с экспоненциальным бэкоффом.
- [ ] Реализован сбор метрик (длительность этапов, токены, ошибки) для мониторинга пайплайна.
- [ ] При достижении лимита итераций (например, 3-5 попыток `Review -> Develop`) задача корректно фейлится.
- [ ] Написаны Интеграционные тесты (`//go:build integration`) на реальной БД для проверки корректности переходов состояний Pipeline и работы блокировок.
- [ ] Используется структурное логирование (`slog`) с обязательными полями `project_id` и `task_id`.
- [ ] Реализована транзакционная целостность (смена статуса + артефакты + сообщения сохраняются в одной DB-транзакции).
- [ ] Добавлен `defer recover()` во все асинхронные горутины Оркестратора для защиты от краша всего приложения.
- [ ] Реализована проверка прав доступа (Tenant Isolation) на уровне сервиса перед выполнением любых действий с задачей.
