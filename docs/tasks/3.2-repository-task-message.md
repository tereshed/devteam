## Описание

Создать репозиторий для работы с сообщениями задач (`task_messages`). Таблица append-only: сообщения создаются и читаются, но не обновляются и не удаляются по одному (удаляются каскадно при удалении задачи). Основные сценарии: добавить сообщение от агента/пользователя, получить хронологический лог задачи, отфильтровать по типу сообщения или отправителю.

---

## Интерфейс

```go
type TaskMessageRepository interface {
    Create(ctx context.Context, msg *models.TaskMessage) error
    GetByID(ctx context.Context, id uuid.UUID) (*models.TaskMessage, error)
    ListByTaskID(ctx context.Context, taskID uuid.UUID, filter TaskMessageFilter) ([]models.TaskMessage, int64, error)
    ListBySender(ctx context.Context, senderType models.SenderType, senderID uuid.UUID, filter TaskMessageFilter) ([]models.TaskMessage, int64, error)
    CountByTaskID(ctx context.Context, taskID uuid.UUID) (int64, error)
}
```

---

## Методы

### Create

Добавляет сообщение в лог задачи. Append-only — основная операция записи.

| Параметр | Тип | Описание |
|----------|-----|----------|
| `msg` | `*models.TaskMessage` | Заполненная модель (TaskID, SenderType, SenderID, Content, MessageType обязательны) |

Возвращает ошибку при:
- Невалидном `task_id` (FK violation) → `ErrTaskNotFound`
- Невалидном `sender_type` / `message_type` (CHECK violation) → обёрнутая ошибка

### GetByID

Получает одно сообщение по UUID. Возвращает `ErrTaskMessageNotFound` при отсутствии.

### ListByTaskID

Основной метод чтения — хронологический лог сообщений задачи с пагинацией и фильтрами. Сортировка по `created_at ASC` (от старых к новым — лог).

### ListBySender

Все сообщения конкретного отправителя (полезно для аудита: "что написал агент X"). Сортировка по `created_at DESC`.

### CountByTaskID

Быстрый подсчёт сообщений задачи (для UI — badge на карточке задачи).

## Фильтры

```go
type TaskMessageFilter struct {
    MessageType *models.MessageType // Фильтр по типу (instruction/result/question/feedback/error)
    SenderType  *models.SenderType  // Фильтр по типу отправителя (user/agent)
    Limit       int                 // Размер страницы (default ставит Service: 50)
    Offset      int                 // Смещение
}
```

**Дефолты пагинации:** устанавливает Service/Handler (как в задаче 2.1). Репозиторий выполняет запрос с переданными значениями.

---

## Кастомные ошибки

```go
var (
    ErrTaskMessageNotFound = errors.New("task message not found")
)
```

- `gorm.ErrRecordNotFound` → `ErrTaskMessageNotFound`
- FK violation на `task_id` → `ErrTaskNotFound` (реимпорт из `task_repository.go` или определение в общем файле)

## Preload стратегия

Preload не используется — `task_messages` не имеет вложенных связей для загрузки. `Task` загружается отдельно в Service-слое при необходимости.

## Особенности реализации

### Append-only таблица

- **Нет метода `Update`** — сообщения иммутабельны после создания
- **Нет метода `Delete`** — удаление происходит каскадно через `ON DELETE CASCADE` при удалении задачи
- Это упрощает репозиторий и соответствует назначению таблицы (лог/аудит)

### Сортировка

| Метод | Сортировка | Причина |
|-------|-----------|---------|
| `ListByTaskID` | `created_at ASC` | Хронологический лог (от первого сообщения к последнему) |
| `ListBySender` | `created_at DESC` | Последние сообщения отправителя сверху |

Сортировка фиксирована в репозитории (не через фильтр) — для `task_messages` пользовательская сортировка не нужна, порядок определён семантикой.

## Конвенции

1. **Интерфейс + приватная struct** — `NewTaskMessageRepository(db *gorm.DB) TaskMessageRepository`
2. **`r.db.WithContext(ctx)`** — во всех запросах
3. **`fmt.Errorf("failed to ...: %w", err)`** — оборачивание ошибок
4. **Кастомные sentinel errors** — `ErrTaskMessageNotFound`
5. **Paging: `Count` + `Limit/Offset`** — аналогично `ProjectRepository`
