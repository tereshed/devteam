# 5.6 Стрим логов из контейнера (`ContainerLogs` → канал)

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/docker_runner.go` (метод **`StreamLogs`**, внутреннее состояние стрима в **`instanceState`**); при выносе логики — вспомогательные файлы пакета **`sandbox`** (например `stream_logs.go`, `stream_line_writer.go`) **без** импорта `gin`, `handler`, GORM. Константы и тип **`LogEntry`** — `backend/internal/sandbox/types.go` (или выделенный `logs.go` по рекомендации **5.4**).  
**Статус:** ✅ Выполнена

---

## Описание

Довести до полного контракта **5.3** реализацию **`DockerSandboxRunner.StreamLogs`**: открытие потока **`stdout`/`stderr`** контейнера через Docker Engine API (**`client.ContainerLogs`** с **`Follow: true`**, при необходимости **`Timestamps`**, **`Since`**), демультиплексирование потока движка (**`stdcopy.StdCopy`** или эквивалент), преобразование в канал **`<-chan LogEntry`**.

**Границы задачи:** только **стриминг логов** и согласованность с **`ErrStreamAlreadyActive`**, отменой **`context.Context`**, **`Cleanup`** и отсутствием утечек (**FD**, горутины). Сбор артефактов после завершения — **5.7**; политика SIGTERM/SIGKILL и бизнес-таймаут — **5.8**; тонкая настройка лимитов — **5.9**.

**Связь с 5.5:** в **5.5** допускался минимальный скелет **`StreamLogs`** (один attach, **`Close()`** на **`io.ReadCloser`**, заглушка **`ErrStreamAlreadyActive`**). Задача **5.6** закрывает пробелы: **backpressure** (переполнение буфера канала), явная семантика **дропа** vs **завершение с `LogEntry.Error`**, чанкирование длинных строк (**`LogEntryMaxLineBytes`**), согласованность **`Cleanup`** со стримом (отмена + закрытие канала + отсутствие гонок с горутиной-ридером), при необходимости — **tail** с момента последнего байта для реконнекта (post-MVP, зафиксировать в коде как «не делаем» или «делаем через `Since`**).

**Требует:** **5.3** (контракт **`SandboxRunner.StreamLogs`**), **5.4** (**`LogEntry`**, **`StreamLogsDefaultBuffer`**, **`LogEntryMaxLineBytes`**), **5.5** (**`DockerSandboxRunner`**, **`instanceState`**, **`ValidateSandboxID`**, доступ к инстансу по **`sandboxID`**).

**Используется в:** **5.13** (юнит-тесты: мок клиента, сценарии «второй `StreamLogs`», отмена **`ctx`**, **`Cleanup` во время стрима»), **5.14** (интеграция с реальным контейнером), **6.x** (оркестратор: tee в БД + WebSocket).

## Контракт API (согласование с 5.3)

### Один активный стрим (вариант А)

| Поведение | Требование |
|-----------|------------|
| Параллельные **`StreamLogs`** на один **`sandboxID`** | Второй и последующие — **`ErrStreamAlreadyActive`**, пока первый стрим не завершён (канал закрыт, внутреннее состояние сброшено). |
| Fan-out (БД + UI) | Вне раннера: один подписчик на канал раннера; **оркестратор** владеет буферами и политикой backpressure на стороне tee (второй потребитель, WebSocket и т.д.) — без неявного дублирования «ещё одного» ограниченного канала внутри **`sandbox`**. |

### Жизненный цикл канала

1. Успешный **`StreamLogs`** возвращает **только для чтения** канал; раннер — единственный отправитель.
2. Канал **закрывается** при: нормальном EOF логов, отмене **`streamCtx`** (в т.ч. из **`Cleanup`** через **`streamCancel`**), фатальной ошибке чтения после терминальной **`LogEntry`** с **`Error != nil`** (по политике ниже).
3. Потребитель **обязан** вычитывать канал до закрытия **или** отменить переданный в **`StreamLogs`** **`ctx`**, если больше не нужен поток (**5.3**).

### `LogEntry` и ошибки

- **`Error != nil`:** терминальная запись для данного инцидента; после неё канал может быть закрыт без дополнительных строк (политика: одна ошибка + `close` — зафиксировать в коде и тестах).
- **Текст и цепочка для потребителей вне `sandbox`:** в **`LogEntry.Error`** допускаются только **обёртки вокруг sentinel’ов пакета** (в первую очередь **`ErrSandboxDocker`**, при необходимости **`ErrSandboxStartup`**), вида `fmt.Errorf("log stream: %w", ErrSandboxDocker)` с опциональным **стабильным** коротким префиксом (`"log stream"`, `"container logs"`). **Запрещено** подставлять в цепочку, предназначенную для UI/JSON, **голый** `err` от Docker SDK (пути демона, длинные внутренние сообщения, стектрейсы плагинов): исходная ошибка остаётся только для **server-side** логов (**`slog`** + **`%v`/`%+v`** на DEBUG при флаге), не сериализуется в **`LogEntry`** как единственный смысл поля. **Недопустимо** считать «санитизированным» произвольный **`err.Error()`** от движка; если когда-либо понадобится человекочитаемая строка без `errors.Is` — завести **отдельное** поле в DTO API слоя **`handler`**, не расширять **`LogEntry`** произвольным текстом демона.

### Чанкирование

- **`len(Line) ≤ LogEntryMaxLineBytes`**; длинная логическая строка без `\n` или длиннее лимита — **несколько** **`LogEntry`** подряд с согласованными **`Stderr`**; **`Timestamp`:** ненулевой только у **первого** чанка логической строки, у продолжений после разбиения — нулевой (`stream_line_writer.go`).

### Семантика EOF и `LogEntry.Error` (согласование с тестами и UI)

- **EOF** от **`ContainerLogs`** (в т.ч. при ещё работающем контейнере — см. п.4 ниже): **тихое** завершение — **`close(ch)`** без обязательной последней записи с **`Error != nil`**.
- **`LogEntry.Error != nil`:** только при **сбое** чтения/демультиплексирования (SDK, обрыв не по штатному EOF), отмене, невозможности продолжить контракт стрима — чтобы **5.13** и UI не расходились («ошибка» ≠ «просто конец лога»).

## Реализация: Docker SDK

### `ContainerLogs`

- Первый аргумент — **`context.Context`** с поддержкой отмены (от **`Cleanup`**, от клиента).
- Опции: **`ShowStdout`**, **`ShowStderr`**, **`Follow: true`**; **`Timestamps`** — по конфигу (**5.10**) или разумный дефолт (**`false`**, чтобы не дублировать время, если **`LogEntry.Timestamp`** заполняется на стороне Go).
- **`Since`** / **`Until`** — в **5.6** не обязательны; если **не** реализуются — явный комментарий в коде «не поддерживается». **Когда параметры времени начнут приходить из пользовательского API** (query/body), валидация формата и разумных границ (защита от DoS бессмысленно огромными строками / нагрузкой на движок) — в **`handler`/`service`**, не в «тихом» пробросе в Docker (**инъекции в Engine маловероятны**, но бессмысленный ввод и объём — да).

### Демультиплексирование

- Использовать **`stdcopy.StdCopy`** (или эквивалент по формату multiplexed stream), чтобы разделить stdout/stderr в поле **`LogEntry.Stderr`**.

### Ресурсы

- Результат **`ContainerLogs`** — **`io.ReadCloser`**: на **всех** путях выхода горутины-ридера вызывать **`defer rc.Close()`** (уже зафиксировано в **5.5**; **5.6** проверяет отсутствие веток без **`Close`**).

## Backpressure и переполнение буфера

**Проблема:** если горутина-ридер **блокируется** на отправке в канал при полном буфере, она перестаёт читать из **`ContainerLogs`** → **`stdcopy`** блокируется → растёт буфер pipe **stdout/stderr** в контейнере → процесс агента может зависнуть на **`write`** (**5.3** требует явной политики).

**Схема «дроп» (вариант A, канал с буфером):** на каждую готовую порцию лога использовать **`select { case ch <- entry: default: }`**. Пока буфер канала **полон**, ветка **`default`** выполняется: **эта порция не попадает в канал** (теряется с точки зрения UI), но **чтение из `io.ReadCloser` продолжается** — ридер не блокируется на отправке. Дополнительно (рекомендуется): при первом дропе или раз в N дропов — одна синтетическая **`LogEntry`** с **`Line`** вроде `logs dropped: slow consumer` (без секретов) **или** только **`slog.Warn`** со счётчиком — выбрать одно и зафиксировать в коде. **Опционально (post-MVP / i18n на Flutter):** стабильный **машинно-читаемый** префикс в **`Line`** (например константа пакета `__DEVTEAM_LOGS_DROPPED__`) вместо «человеческой» строки, либо отдельное поле у **`LogEntry`** в задаче **5.4** — в этой итерации **не обязательно**.

| Вариант | Поведение |
|---------|-----------|
| **A — дроп (описан выше)** | Не блокировать ридер на `ch <-`; при полном буфере пропускать **входящую** порцию; опционально метка/лог о потере. |
| **B — завершение стрима** | При первом переполнении: неблокирующая попытка отправить **`LogEntry{Error: …}`** (по правилам раздела «`LogEntry` и ошибки») и **закрыть** канал; чтение из Docker остановить через **`Close(rc)`** + отмену **`ctx`**. Если в этот момент буфер канала **тоже полон** и запись не вошла в **`select`**: **один** из вариантов закрепить в коде — **`slog.Warn`** + закрыть канал без записи **или** **однократная** блокирующая отправка с **коротким** дедлайном отдельного контекста (не родительского HTTP-**`ctx`**). Контейнер может продолжить работу; логи остаются в **`json-file`** (**5.5**). |

Текущий код с **`default:`** без явной семантики потерь — **недостаточен**; **5.6** обязана реализовать **A** или **B** и продублировать выбор одной строкой в комментарии к **`StreamLogs`**.

## Потокобезопасность и `Cleanup`

1. **`StreamLogs`:** регистрация «активный стрим» и **`streamCancel`** под **`streamMu`** (или общий мьютекс инстанса — без дедлока с **`Cleanup`**).
2. **`Cleanup`:** как в **5.3** — вызвать **`streamCancel()``**, сбросить флаги, затем удалить контейнер. Порядок с **`ContainerRemove`:** отмена стрима **до** remove, чтобы горутина вышла из **`Read`** и закрыла **`rc`**.
3. **Гонка `StreamLogs` после `Cleanup`:** инстанс может быть удалён из **`instances`** — возврат **`ErrSandboxNotFound`**, **без** паники и без вызова **`ContainerLogs`** в противоречии состоянию.
4. **Одна сессия `ContainerLogs`:** в scope **5.6** **без** reconnect при обрыве стрима при ещё работающем контейнере (**`Follow: true`** иногда даёт **EOF** раньше остановки контейнера). **Семантика по умолчанию** (для тестов **5.13** и UI): **EOF → тихое закрытие канала** без терминальной **`LogEntry.Error`** (см. раздел «Семантика EOF и `LogEntry.Error`»); **`Error`** только на реальном сбое чтения. Автовторой **`ContainerLogs`** — **не** в **5.6** (отдельная задача/post-MVP).
5. **Завершение горутины-ридера после `Cleanup`:** верхняя граница по времени зависит от движка; **объективный критерий приёмки** — юнит-тест (**5.13**): после последовательности **`StreamLogs` → `Cleanup`** (и вычитки/отмены потребителем по контракту) **нет утечки горутин** (например **`go.uber.org/goleak`** в **`TestMain`** пакета **`sandbox`** или тест с контролем числа горутин до/после согласованным с runtime). Фраза «нет зависших горутин в типичном сценарии» **не** заменяет такой тест.

---

## Безопасность

- **`sandboxID`:** только после **`ValidateSandboxID`** до любого RPC (**5.3** / **5.5**).
- Логи могут содержать **секреты**, случайно выведенные агентом; раннер **не** логирует полное содержимое **`LogEntry.Line`** на уровне **INFO** по умолчанию (только метрики/длина; детали — флаг отладки).
- См. раздел «**`LogEntry` и ошибки**» про **`Error`**: не пробрасывать наружу сырой текст демона.

---

## Границы слоёв и `backend.mdc`

Задача **5.6** касается **только** внутреннего API пакета **`internal/sandbox`** (тип **`SandboxRunner`** и реализация). **Публичный HTTP-эндпоинт** для стриминга логов пользователю (если появится) **вне scope** этой задачи: для него потребуются отдельные **`handler`/`service`**, **Swagger** и при необходимости **MCP-инструмент** по правилам репозитория. Здесь достаточно не смешивать транспорт с **`ContainerLogs`**.

## Критерии приёмки

- [x] **`StreamLogs`** использует **`ContainerLogs`** с **`ctx`**, **`Follow: true`**, корректным демультиплексированием stdout/stderr; **один** вызов **`ContainerLogs`** на успешный стрим (**без** reconnect в **5.6**).
- [x] Повторный **`StreamLogs`** при активном первом — **`ErrStreamAlreadyActive`**.
- [x] **`io.ReadCloser`** от **`ContainerLogs`** всегда закрывается (**`defer Close`** на всех путях).
- [x] Канал закрывается при EOF, отмене **`ctx`**, после **`Cleanup`**; юнит-тесты (**5.13**) подтверждают **отсутствие** утечки горутин после **`StreamLogs` → `Cleanup`** (например **`goleak`**, см. раздел «Завершение горутины-ридера»).
- [x] Соблюдено **`LogEntryMaxLineBytes`** (чанкирование длинных строк и «хвоста» без `\n`).
- [x] Зафиксирована и реализована **одна** политика **backpressure** (раздел выше); есть тест в **5.13**, демонстрирующий поведение при медленном потребителе.
- [x] Ошибки чтения в **`LogEntry.Error`** — только цепочки с sentinel’ами **`sandbox`** (**`ErrSandboxDocker`** и т.д.), без голого SDK в сообщении для потребителя (см. «`LogEntry` и ошибки»); **EOF** — без обязательной **`Error`**-записи (см. «Семантика EOF»).
- [x] Комментарии к **`StreamLogs`** в коде ссылаются на **5.3** (вариант А) и эту спецификацию.
- [x] **Документация (PR):** в таблице README Sprint 5 у пункта **5.6** сохранена ссылка «детали» на актуальный файл спеки — проверка при слиянии, **не** замена автоматических тестов.
