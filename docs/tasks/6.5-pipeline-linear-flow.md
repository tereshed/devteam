# 6.5 Реализация: Pipeline — линейный поток `Plan → Develop → Review → Test`

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/service/orchestrator_service.go`
**Статус:** ⬜ В планах

---

## Описание

Реализовать базовую логику линейного пайплайна (цепочки выполнения) внутри `OrchestratorService`. Пайплайн должен обеспечивать последовательный переход задачи через этапы: планирование (Plan), разработка (Develop), ревью кода (Review) и тестирование (Test). 

Это "счастливый путь" (happy path) оркестрации, который является фундаментом для более сложных графов выполнения.

---

## Функциональные требования

### 1. Этап планирования (Plan)
- Оркестратор вызывает `Planner` (LLM) для декомпозиции высокоуровневой задачи.
- Результат: список конкретных шагов или подзадач.
- Статус задачи: `pending` -> `planning` -> `in_progress`.

### 2. Этап разработки (Develop)
- Оркестратор передает задачу `Developer` (Sandbox: Claude Code / Aider).
- Контекст включает: описание задачи, план, релевантный код из векторной БД.
- Результат: Diff изменений, список измененных файлов, коммит.
- Статус задачи: `in_progress`.

### 3. Этап ревью (Review)
- Оркестратор передает Diff и контекст `Reviewer` (LLM).
- Логика ветвления:
    - Если `Approve`: переход к этапу `Test`.
    - Если `Changes Requested`: возврат к этапу `Develop` (с учетом лимита итераций).
- Статус задачи: `review`.

### 4. Этап тестирования (Test)
- Оркестратор передает код и задачу `Tester` (Sandbox).
- Тестер запускает существующие тесты или генерирует/запускает новые.
- Логика ветвления:
    - Если `Success`: переход в `completed`.
    - Если `Failed`: возврат к этапу `Develop`.
- Статус задачи: `testing` -> `completed` / `failed`.

### 5. Управление потоком (Flow Control)
- **Линейность:** Строгое соблюдение порядка этапов.
- **Итерации:** Ограничение количества циклов `Review -> Develop` и `Test -> Develop` (настраиваемый параметр, например, макс. 3-5 итераций).
- **Транзакции:** Сохранение промежуточных результатов (артефактов) на каждом этапе.

### 6. Безопасность и Инъекции (🔐 Security)
- **Command/Prompt Injection Protection:** Перед передачей данных в `SandboxAgentExecutor` (этапы Develop и Test) обеспечить строгую валидацию и экранирование входных данных (описание задачи, план, контекст). Пользовательский ввод должен передаваться безопасным способом (например, через запись в файл внутри контейнера), чтобы исключить выполнение произвольных команд.
- **Input Sanitization:** Все строковые входные данные должны проверяться на наличие потенциально опасных последовательностей (shell-метасимволы, escape-последовательности).

### 7. Context & Cancellation (⏹️ Control Flow)
- **Context Propagation:** Пайплайн обязан принимать `context.Context` и прокидывать его во все вызовы `AgentExecutor` и `TaskService`. Это позволяет прерывать выполнение задачи по запросу пользователя (нажатие "Стоп").
- **Graceful Shutdown:** При получении сигнала отмены контекста (`ctx.Done()`) пайплайн должен немедленно прерываться (early return) и переводить задачу в статус `cancelled`.
- **Resource Cleanup:** При отмене должны освобождаться все связанные ресурсы (остановка песочницы, закрытие соединений).

### 8. Краевые случаи и Валидация (🛡️ Edge Cases)
- **Iteration Limit Handling:** При достижении лимита итераций (3-5 циклов `Review -> Develop` или `Test -> Develop`) задача должна переводиться в статус `failed` с сохранением соответствующего `error_message` ("Превышен лимит попыток исправления"). Альтернативно возможен переход в `paused` для ручного вмешательства пользователя.
- **Empty Response Handling:** Реализовать проверку на пустые/невалидные ответы от агентов. Если Planner вернул пустой план или Developer не произвел изменений — задача должна немедленно переводиться в `failed` (early fail).
- **System Error Handling:** Любые системные ошибки (падение песочницы с OOM, таймауты сети, ошибки Docker) должны корректно перехватываться и переводить задачу в статус `failed` с детальным логом ошибки.
- **Transactional Consistency:** Обновление статуса задачи и сохранение артефактов (результатов этапа) должно происходить строго в рамках одной транзакции БД. При ошибке — полный Rollback состояния.
- **Panic Recovery:** Все горутины, запускаемые в рамках пайплайна, должны содержать `defer recover()` для предотвращения краха всего приложения при панике.

## Критерии приёмки

- [ ] Реализован жесткий порядок переходов: `Plan -> Develop -> Review -> Test`.
- [ ] Оркестратор корректно вызывает соответствующие `AgentExecutor` для каждого этапа.
- [ ] Реализована логика возврата на этап `Develop` при получении `Changes Requested` от ревьюера.
- [ ] Реализована логика возврата на этап `Develop` при провале тестов.
- [ ] Внедрен счетчик итераций для предотвращения бесконечных циклов правок.
- [ ] Артефакты каждого этапа (план, diff, отчет о тестах) сохраняются в БД через `TaskService`.
- [ ] Статусы задачи в БД обновляются синхронно с переходом по этапам пайплайна.
- [ ] Написаны unit-тесты для `OrchestratorService`, проверяющие логику переходов (с использованием моков для экзекуторов).
- [ ] Интеграционный тест проходит по всему "счастливому пути" от `Plan` до `Test`.
- [ ] Во все методы пайплайна прокидывается `context.Context`, реализована корректная обработка его отмены (`ctx.Done()`).
- [ ] Добавлен тест на прерывание пайплайна по таймауту/отмене контекста.
- [ ] Добавлен тест на достижение лимита итераций (проверка перехода в статус `failed`).
- [ ] Обновление стейта задачи и запись артефактов в БД обернуты в транзакцию.
- [ ] Реализована санитизация входных данных перед передачей в `SandboxAgentExecutor`.
- [ ] Добавлен `defer recover()` в горутины пайплайна для защиты от паники.
