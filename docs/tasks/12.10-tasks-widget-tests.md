## Эмуляция WS в widget-слое (без сокета)

Паттерн по смыслу **11.11** § **WS-эмуляция**, но поверхность контроллеров задач (**12.3** / **12.9**) — **`applyWsTaskStatus`**, **`applyWsTaskMessage`**, **`applyRealtimeFailure`**, **`applyAuthFailure`**, **`requestRestRefetch`** — **не** поднимать **`WebSocketService`**.

**Список задач — обновление статуса с доски/ленты:**

```dart
import 'package:frontend/core/api/websocket_events.dart';
import 'package:frontend/features/tasks/presentation/controllers/task_list_controller.dart';
import 'package:frontend/features/tasks/presentation/screens/tasks_list_screen.dart';
// kTaskFixtureProjectId, tid — из task_fixtures / сценария

final container = ProviderScope.containerOf(
  tester.element(find.byType(TasksListScreen)),
);
final notifier = container.read(
  taskListControllerProvider(projectId: kTaskFixtureProjectId).notifier,
);
notifier.applyWsTaskStatus(
  WsTaskStatusEvent(
    ts: DateTime.utc(2026, 1, 1, 12),
    v: 1,
    projectId: kTaskFixtureProjectId,
    taskId: tid,
    previousStatus: 'pending',
    status: 'in_progress',
    parentTaskId: null,
    assignedAgentId: null,
    agentRole: null,
    errorMessage: null,
  ),
);
await tester.pump();
```

**Деталь задачи — новое сообщение по WS (`applyWsTaskMessage`):**

```dart
import 'package:frontend/core/api/websocket_events.dart';
import 'package:frontend/features/tasks/presentation/controllers/task_detail_controller.dart';
import 'package:frontend/features/tasks/presentation/screens/task_detail_screen.dart';

final container = ProviderScope.containerOf(
  tester.element(find.byType(TaskDetailScreen)),
);
final notifier = container.read(
  taskDetailControllerProvider(
    projectId: kTaskFixtureProjectId,
    taskId: tid,
  ).notifier,
);
notifier.applyWsTaskMessage(
  WsTaskMessageEvent(
    ts: DateTime.utc(2026, 1, 1, 12),
    v: 1,
    projectId: kTaskFixtureProjectId,
    taskId: tid,
    messageId: 'aaaaaaaa-aaaa-aaaa-aaaa-aaaaaaaaaaaa',
    senderType: 'agent',
    senderId: 'bbbbbbbb-bbbb-bbbb-bbbb-bbbbbbbbbbbb',
    senderRole: 'developer',
    messageType: 'result',
    content: 'Fixture WS message',
    metadata: const <String, dynamic>{},
  ),
);
await tester.pump();
```

**Деталь задачи** — **`applyWsTaskStatus`** на том же **`notifier`**, поля как в первом блоке. Сценарии фейла сессии — **`applyRealtimeFailure`**, **`applyAuthFailure`** по необходимости (**12.9**).

**Family `taskDetailControllerProvider` — autoDispose:** вызывать **`container.read(…).notifier`** и методы **`applyWs*`** только когда **`TaskDetailScreen`** уже смонтирован и найден (**`find.byType(TaskDetailScreen)`**); после **`pop`** / смены маршрута без **`pump`** family может пересоздаться — событие уйдёт «в пустой» инстанс, тест останется зелёным при баге в проде. Для списка **`taskListControllerProvider`** — **`keepAlive`** (**12.3**), та же ловушка реже, но проверять монтирование **`TasksListScreen`** перед **`read`** всё равно хорошая гигиена.

## Якоря верификации в прод-коде (без hedge-clauses)

Исполнитель **не** пропускает сценарий с формулировкой «нет в проде»: ниже привязка к **фактическим** символам в **`lib/`** на момент закрытия **12.4**–**12.8**.

### `TasksListScreen` (`tasks_list_screen.dart`)

- **Якорь в коде:** искать символы **`class _KanbanColumn`**, **`if (isWide)`**, **`Row`**, **`for (final status in taskStatuses)`**, **`_KanbanColumn`** (ориентир при текущей ревизии, из каталога **`frontend/`**: `grep -n '_KanbanColumn\\|isWide' lib/features/tasks/presentation/screens/tasks_list_screen.dart`). Два режима по ширине — **[12.4](./12.4-tasks-list-screen.md)** § **«Канон режима отображения (SSOT)»** и § **«UI: обязательные элементы»**. Из теста **нельзя** **`find.byType(_KanbanColumn)`** (library-private тип).
- **Обязательно:** (1) **narrow** и **wide** через **`useViewSize`**: при **непустом** списке на **wide** одновременно видны **≥ 2** заголовка колонок статусов (текст из **`taskStatusLabel`**, напр. **`find.text(l10n.taskStatusPending)`** и **`find.text(l10n.taskStatusCompleted)`**); на **narrow** — **одна** вертикальная лента **без** того же набора заголовков колонок Kanban; (2) **pull-to-refresh** при \< **600 dp** (**`RefreshIndicator`**); (3) при ≥ **600 dp** — **`IconButton(Icons.refresh)`** → **`notifier.refresh()`** — канон **[12.4](./12.4-tasks-list-screen.md)** § **«UI: обязательные элементы»**, пункт **6** (не путать с п. **6** из других разделов того же файла); эталон поведения — **10.4**; (4) **`loadMore`** у нижнего края и баннер **`loadMoreError`** (**`kTasksListLoadMoreErrorBannerKey`**) — § **«UI: обязательные элементы»** п. **7**–**8** в **12.4**; (5) **поиск и фильтры:** состояние «пусто из-за фильтра» — **`tasksEmptyFiltered`** + **`tasksEmptyFilteredClear`** (§ **«UI: обязательные элементы»** п. **4** в **12.4**); тап по очистке приводит к сбросу фильтра (**`setFilter(TaskListFilter.defaults())`** на нотифайере или тот же эффект через UI). **Поиск:** по **[12.4](./12.4-tasks-list-screen.md)** § **«UI: обязательные элементы»** п. **2** поле поиска на экране **обязательно** — минимальный сценарий **behavior**, не «нашли hint»: ввод текста в **`TextField`** поиска → после дебаунса **400 ms** (как **10.4**) проверить, что **`filter.search`** в **`TaskListState`** отражает введённую строку (например **`ref.read(taskListControllerProvider(...))`** после **`await tester.pump()`** и **`await tester.pump(const Duration(milliseconds: 400))`**, либо общая задержка ≥ **400 ms** серией **`pump`**). Одного **`find.text(l10n.tasksSearchHint)`** недостаточно.
