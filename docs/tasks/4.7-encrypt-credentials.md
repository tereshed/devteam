# 4.7 Шифрование credentials (AES-256-GCM)

**Sprint:** 4 — Git Provider Integration / Security (Backend)  
**Файлы:** `backend/pkg/crypto/` (конкретная реализация + тесты), `backend/internal/service/` (интерфейс `Encryptor`), `backend/internal/config/config.go`, `backend/cmd/api/main.go`  
**Статус:** ✅ Выполнена

---

## Описание

Реализовать безопасное шифрование и дешифрование чувствительных данных (учётные данные Git, API-ключи MCP и т.д.) алгоритмом **AES-256-GCM**. Решение должно соответствовать **Clean Architecture** и **DI**: сырой ключ не передаётся в каждый вызов; заложена **версия формата blob’а** для будущей ротации ключей; используется **AAD**, чтобы ciphertext нельзя было «переставить» между строками БД без ошибки дешифрования.

---

## Требования к реализации

### 1. Алгоритм

- **AES-256-GCM** (конфиденциальность + аутентификация ciphertext и AAD).

### 2. Размещение интерфейса и реализации (пуризм Go / Clean Architecture)

- **Интерфейс `Encryptor`** объявляется **там, где он потребляется** — в `backend/internal/service` (отдельный файл, например `encryptor.go`, или рядом с первым сервисом, который его использует). Так удобнее мокать зависимость в юнит-тестах сервисов.
- Пакет **`backend/pkg/crypto`** экспортирует **конкретный тип** (например `AESEncryptor`) и конструктор:

```go
func NewAESEncryptor(key []byte) (*AESEncryptor, error)
```

- Тип `*AESEncryptor` в `main` передаётся в сервисы как **`service.Encryptor`** (неявное удовлетворение интерфейсу). В `pkg/crypto` **не** дублировать интерфейс с тем же именем, если он уже определён в `internal/service`.

### 3. Сигнатуры методов и AAD (защита от swapping)

Без **Additional Authenticated Data (AAD)** злоумышленник или баг могут скопировать `encrypted_credentials` из записи пользователя A в запись пользователя B — GCM успешно расшифрует тот же plaintext, и секрет «привяжется» к чужой строке.

**Обязательно:** при шифровании и дешифровании передавать один и тот же **детерминированный** AAD, привязанный к сущности в БД, например:

- для MCP-конфига — UUID `project_id` или **`mcp_server_config.id`** (выбрать одно правило и держаться его во всех вызовах для этой таблицы);
- для Git credentials — **`git_credentials.id`** или связка, однозначно идентифицирующая строку.

Интерфейс в `internal/service`:

```go
type Encryptor interface {
    Encrypt(plaintext []byte, associatedData []byte) ([]byte, error)
    Decrypt(ciphertext []byte, associatedData []byte) ([]byte, error)
}
```

- **Стандарт AAD (без «зоопарка»):** для идентификаторов типа UUID из `github.com/google/uuid` передавать **строковое представление в байтах UTF-8**, например **`[]byte(id.String())`** (36 символов с дефисами, как в логах и в SQL). Один и тот же способ при `Encrypt` и `Decrypt` для одной логической записи. Так проще отлаживать, чем смешивать сырые 16 байт (`uuid.UUID[:]`) и `.String()` в разных местах.
- Сервис при сохранении/чтении формирует AAD; репозиторий по-прежнему только читает/пишет `[]byte` в `bytea`.

### 4. Формат blob’а в БД (версия ключа + nonce + ciphertext)

Зарезервировать **1 байт версии формата** для будущей **ротации ключей** без неоднозначности «каким ключом шифровали эту строку».

**Структура blob’а:**

`[1 byte Version] || [12 bytes Nonce] || [Ciphertext+Tag]`

| Часть | Размер | Описание |
|--------|--------|----------|
| Version | 1 | Сейчас **жёстко `0x01`**. При смене схемы или ключа можно ввести `0x02` и маппинг ключей в конфиге. |
| Nonce | 12 | Стандарт для GCM; каждый раз новый, **`crypto/rand`**. |
| Остальное | ≥ 16 байт | Вывод `cipher.AEAD.Seal`: ciphertext + **authentication tag** (у стандартного GCM в Go — **16 байт**). Даже при **пустом plaintext** в sealed будет только tag (**16 байт**). |

При **Decrypt**:

1. Проверить `len(blob) >= 29` (**1** version + **12** nonce + **16** минимум для tag), иначе `ErrInvalidCiphertext` (или эквивалент), **без panic** и **без** вызова `Open` — для более короткого blob’а дешифрование заведомо невалидно.
2. Прочитать `version := blob[0]`. Если `version != 0x01` — вернуть явную ошибку неизвестной версии (например `ErrUnsupportedCiphertextVersion`); в будущем здесь выбор ключа по версии.
3. Nonce = `blob[1:13]`, sealed = `blob[13:]`, вызов `Open` с тем же **AAD**, что при `Encrypt`.

### 5. Инициализация `cipher.AEAD` (без оверинжиниринга)

- В `NewAESEncryptor`: один раз создать блочный шифр, **`cipher.NewGCM`**, сохранить результат **`cipher.AEAD` в поле структуры**.
- **`cipher.AEAD` потокобезопасен** — его можно вызывать из нескольких горутин без дополнительной синхронизации.
- **Запрещено** в ТЗ описывать «ленивые кэши» или отложенную инициализацию GCM: это лишнее и создаёт риск гонок при неверной реализации.

### 6. Конфигурация (только HEX)

Сырые 32 байта в ENV задать нельзя.

**Обновить `internal/config`:**

- Заменить хранение ключа на **`[]byte`** после декодирования (например поле `Key []byte` в `EncryptionConfig` вместо текущего `Key string`).
- **`ENCRYPTION_KEY` — строго HEX: ровно 64 шестнадцатеричных символа → 32 байта.** Никакого Base64 и иных форматов в рамках этой задачи — один контракт, меньше «зоопарка» в коде и в эксплуатации.
- При `Load()`: декодировать HEX; если длина после декодирования не 32 — **ошибка старта** с понятным сообщением.

### 7. Безопасность и краевые случаи (без panic)

- Минимальная длина входа для `Decrypt`: **`>= 29`** (**1** version + **12** nonce + **16** tag GCM). Если **меньше** — сразу **`ErrInvalidCiphertext`** до любых срезов и до `Open`.
- Неизвестный `version` — ошибка, не panic.
- Неверный AAD при `Decrypt` (в т.ч. «переставленная» строка) — ошибка аутентификации GCM.
- Unit-тесты: пустой blob, **длины меньше 29 байт**, неверная версия, несовпадение AAD при том же ключе и nonce.

### 8. Слои (где вызывать Encryptor)

- **Только `internal/service`.** Сервис формирует `plaintext`, **AAD** из ID сущности, вызывает `Encryptor`, отдаёт в репозиторий уже **готовый blob** (version || nonce || sealed).
- **`internal/repository`**: только `[]byte` ↔ `bytea`, **без** импорта `pkg/crypto` и без вызовов шифрования.

Связь с данными (в последующих задачах):

- `mcp_server_configs.encrypted_credentials`
- `git_credentials.encrypted_value`

## Тестирование

Файл(ы) `*_test.go` в `backend/pkg/crypto/`:

- Round-trip с одинаковым AAD — plaintext совпадает.
- **Разный AAD** при `Encrypt` и `Decrypt` (имитация переноса blob’а в чужую строку) — ошибка, успешного plaintext нет.
- Неверная **версия** первого байта (например `0x02`) — ожидаемая ошибка, не panic.
- Повреждённый sealed-фрагмент — ошибка GCM.
- **Длины меньше 29 байт** — `ErrInvalidCiphertext`, без `slice bounds out of range`.
- Конфиг: валидный HEX 64 символа; невалидный HEX / неверная длина — ошибка `Load` (тесты на уровне `config`, если удобно вынести декодирование).
