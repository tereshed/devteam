## Доменный модуль `linked_task_snapshots` (DoD)

Один модуль **`linked_task_snapshots.dart`**:

### Семантика partial-patch (обязательна)

Ввести **`LinkedTaskSnapshotPatch`** (или эквивалент), где каждое поле (**`status`**, **`title`**, **`error_message`**, **`agent_role`**, …) имеет семантику **«отсутствует в патче»** vs **«явно задано, в т.ч. `null`»**:

| Ситуация | Поведение **`mergeTaskSnapshotPatch`** |
|----------|----------------------------------------|
| Поля **нет** в патче (ключ отсутствует / optional **absent**) | Старое значение в снимке **не меняется**. |
| Поле **есть** и значение **`null`** (**explicit null**, бэкенд передал очистку) | Поле в снимке **очищается** (в модели — **`null`** / пустая строка по контракту поля). |
| Поле **есть** и значение не-**null** | Заменить на новое значение. |

Реализация: **`Optional<String?>`** из пакета со «значение отсутствует» vs **`Some(null)`**, или тривиальный sealed-class **`Absent` / Present(T)`** — на выбор PR, но **две семантики обязательны**.

**Тесты доменного модуля:** минимум три кейса — absent не трогает; explicit **`null`** очищает **`error_message`**; обычная строка перезаписывает.

## Антишторм `refresh()` при `needsRestRefetch`

**Проблема:** серия **`stream_overflow`** может вызвать re-entrant **`refresh()`** и шторм REST.

**Дефолт (единый для всех PR по 11.9, без выбора «на усмотрение»):**

1. **`bool _historyRefreshInFlight`** (или семантически то же имя): пока выполняется **`refresh()`**, индуцированный **`needsRestRefetch`** (или совпадающая по смыслу загрузка первой страницы истории в этом потоке), **второй параллельный** **`refresh()`** из этого канала **не** запускается — последующие **`needsRestRefetch`** подавляются или ставятся в очередь «один раз после завершения текущего» (**одна** очередь на контроллер). Тест **T4** проверяет это поведение.
2. **`sessionEpoch`** (**11.4**): ответы REST «старой» эпохи **не** мержить — **обязательная** страховка.
3. **Debounce/throttle — только если нужно:** если после (1) в проде или тестах всё ещё фиксируется частый шторм, добавить **`static const kRestRefetchMinInterval`** (**`Duration`**) с комментарием «подключено из-за …» и усилить **T4** / добавить измерение интервала. Без измеримой проблемы константу **не** вводить.

Итог: «без бесконечных циклов» = **(1)+(2)** по умолчанию; **(3)** — опциональный слой поверх.

## Смена `projectId` и несколько чатов

**Контекст:** **`ChatController`** — family **`(projectId, conversationId)`**. **`WebSocketService`** сейчас привязан к **`webSocketServiceProvider`** (**`keepAlive: false`** — факт на коде).

**Кто вызывает `disconnect` (зафиксировано, не обсуждается в PR):**

- **`ChatController.ref.onDispose`** отменяет **только** локальную **`StreamSubscription`** на **`WebSocketService.events`**. **`disconnect()`** и **`dispose()`** на **`WebSocketService`** из **`ChatController`** **не** вызываются — иначе один закрытый чат обрушит сокет **всем** остальным экранам того же проекта, которые делят сервис.
- Закрытие TCP при уходе с **дашборда проекта** / смене **`projectId`** в роутере — зона **scope-провайдера** (**`keepAlive: true`** на оболочке проекта **или** явный **`disconnect`/`dispose`** при **`pop`** до корня проекта). Wiring описывается в PR один раз, но правило «контроллер чата не рвёт общий сервис» — **обязательное**.

| Риск | Уточнение |
|------|------------|
| Два контроллера / два проекта | Подписка на **`events`** — broadcast; фильтр **`event.projectId == family projectId`** в **`ChatController`** — **обязателен** (silent ignore). |
| Несколько чатов одного проекта | Общий **`WebSocketService`** на проект + только отмена подписки в **`onDispose`** каждого **`ChatController`** — совместимо с правилом выше. |
| Смена проекта A → B | Новый **`connect(B)`** на уровне провайдера проекта B; старый сервис проекта A закрывается **не** из **`ChatController`**, а из lifecycle оболочки A. |

Минимум в **11.9**: тест/ручная проверка «два чата одного проекта + уход с экрана одного чата не отключает WS второму» и «смена проекта — нет применения чужих **`WsTaskStatusEvent`**».

## Auth и выход из сессии

**Не** сводить **`WsServiceFailure.authExpired`** / **`WsClientEvent.authFailure`** только к строкам в **11.10** (там — **`l10n`**, не редирект).

**Зафиксировать один из вариантов (в PR явно):**

- **(А)** Глобальный слой (слушатель **`accessTokenProvider`** / auth **Notifier**) при терминальном событии делает **redirect на `/login`**; **`ChatController`** только поднимает типизированный флаг или вызывает зарегистрированный колбэк — **без** строк.
- **(Б)** Поле в **`ChatState`** с enum терминальных кейсов (**`authExpired`**, **`tooManyConnectionsTerminal`**, …); имя поля и тип задаёт PR (**иллюстрация:** `ChatRealtimeSessionFailure? sessionFailure` — **не существует в репозитории до мержа**, не импортировать как готовый тип).

**Наблюдаемый минимум (связан с тестом T9):** после **`authExpired`** отправка нового сообщения **не** уходит в **`ConversationRepository`** до восстановления сессии; пользователь видит блокировку или редирект — без «тихого» HTTP **401**.
