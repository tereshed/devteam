# 6.9 Агенты по умолчанию (YAML-конфиг)

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файлы:** `backend/agents/orchestrator.yaml`, `planner.yaml`, `developer.yaml`, `reviewer.yaml`, `tester.yaml`
**Статус:** ⬜

---

## Описание

Создать YAML-конфигурации агентов по умолчанию для пяти ролей пайплайна: **Orchestrator**, **Planner**, **Developer**, **Reviewer**, **Tester**. Конфигурации хранятся в каталоге `backend/agents/` и определяют параметры запуска агента: какую LLM-модель использовать, температуру, лимиты, привязку к промптам из `backend/prompts/` (**6.8**).

Цель — **единый источник конфигурации агентов**, декларативный и версионируемый.运行时 (`LLMAgentExecutor`, `SandboxAgentExecutor`) читают конфиг и запускают агент с заданными параметрами; изменение модели или лимитов делается правкой YAML без изменения кода.

**Граница с задачей 6.8:** в **6.8** — содержимое и композиция промптов (тексты, структура секций). В **6.9** — конфигурация агентов по умолчанию (модель, температура, лимиты и привязка к идентификаторам/путям промптов).

## Функциональные требования

### 1. Схема YAML-конфига агента

- **Единственный источник правды по структуре:** перечень полей и их типы задаются схемой (аналогично `prompt_schema.json` из 6.8). Файлы агентов обязаны соответствовать схеме.
- **Фиксированный набор ключей верхнего уровня:**
  - `name` — уникальный идентификатор агента (строка, kebab-case).
  - `role` — роль в пайплайне (`orchestrator`, `planner`, `worker`, `supervisor`).
  - `prompt_name` — ссылка на промпт из `backend/prompts/` (без `.yaml`, например `orchestrator_prompt` → `backend/prompts/orchestrator.yaml`).
  - `model_config` — параметры модели:
    - `model` — идентификатор модели (строка, например `gpt-4o`).
    - `temperature` — температура (число 0.0–2.0). **Строгая валидация границ** при парсинге: значения вне диапазона или нечисловые — приложение падает при старте с явной ошибкой, **не** передаётся мусор в API.
    - `max_tokens` — опционально, лимит токенов (число, положительное).
    - `top_p` — опционально (число 0.0–1.0).
  - `is_active` — флаг активации агента (boolean).
  - `limits` — опционально, лимиты рантайма (время, память и т.д.).
- **Запрещено** вводить новые ключи без изменения схемы.
- **Валидация `prompt_name` (КРИТИЧНО — Path Traversal):** поле должно содержать **только** символы `[a-zA-Z0-9_-]`. Категорически запрещены: слеши (`/`, `\`), точки (`.`), нуль-символ, и любые другие специальные символы. Валидация выполняется при парсинге конфига; некорректное значение = отказ запуска с понятной ошибкой. Это предотвращает атаки типа `prompt_name: "../../../etc/passwd"` или `prompt_name: "/absolute/path/to/secret"`.

### 2. Привязка к промптам и безопасность маппинга

- Каждый агент указывает `prompt_name`, который маппится на конкретный YAML-файл в `backend/prompts/` по соглашению: `backend/prompts/{prompt_name}.yaml`.
- **Orchestrator** → `orchestrator_prompt` → `backend/prompts/orchestrator.yaml`
- **Planner** → `planner_prompt` → `backend/prompts/planner.yaml`
- **Developer** → `developer_prompt` → `backend/prompts/developer.yaml`
- **Reviewer** → `reviewer_prompt` → `backend/prompts/reviewer.yaml`
- **Tester** → `tester_prompt` → `backend/prompts/tester.yaml`
- Загрузчик промптов (из 6.8) и загрузчик конфигов агентов должны быть согласованы по именованию.
- **Path Traversal (КРИТИЧНО):** маппинг `prompt_name` → файл **всегда** происходит внутри каталога `backend/prompts/`; выход за его пределы невозможен даже при некорректном `prompt_name` благодаря валидации из п. 1. Дополнительно: код загрузчика использует `filepath.Join` или аналог, но **не** конкатенацию строк с пользовательским вводом.

### 3. Параметры модели по ролям (разумные значения по умолчанию)

- **Orchestrator:** анализ и маршрутизация — `temperature: 0.1` (низкая, нужна точность).
- **Planner:** декомпозиция задач — `temperature: 0.2` (небольшая вариативность).
- **Developer:** генерация кода — `temperature: 0.2` (воспроизводимость).
- **Reviewer:** проверка кода — `temperature: 0.1` (консервативность).
- **Tester:** интерпретация результатов — `temperature: 0.1` (точность).
- Все агенты используют модель `gpt-4o` по умолчанию; возможность переопределения на уровне проекта — в будущих задачах.

### 4. Читаемость конфигов из кода и жизненный цикл

- Код оркестратора и executor'ов **ожидает** структуру конфига и маппит её на объекты/классы. Конкретный формат объектов — на усмотрение разработчика слоя конфигурации, но он должен соответствовать ключам из п. 1.
- **Кэширование при старте (Performance):** YAML-конфиги агентов загружаются, парсятся, валидируются **один раз** при инициализации приложения (на этапе start-up / инициализации сервисов). Результат хранится в памяти (in-memory cache). `LLMAgentExecutor` / `SandboxAgentExecutor` получают **готовые in-memory структуры**, **не** читают и не парсят YAML при каждом запуске агента. Это исключает лишние I/O-аллокации и просадку производительности.
- При запуске агента executor получает `model`, `temperature` и `prompt_name` из предзагруженной структуры; `prompt_name` используется для загрузки текста промпта через механизм из 6.8 (промпт также кэшируется).
- **Контракт для `is_active: false` (быстрый фейл):** если базовый агент пайплайна (Orchestrator, Planner, Developer, Reviewer, Tester) имеет `is_active: false`, пайплайн **должен падать с явной ошибкой на этапе валидации графа/пайплайна до начала выполнения**. Пропуск отключённого шага недопустим — это приводит к непредсказуемому стейту (например, пропуск Developer с последующим запуском Reviewer = непонятная ошибка дальше по пайплайну). Валидация `is_active` выполняется при загрузке конфигов на старте приложения.

### 5. Валидация в CI и Makefile (DRY)

- **Переиспользование валидатора из 6.8:** логика валидации YAML по JSON Schema **должна** быть переиспользована из задачи 6.8 (где делали то же самое для промптов). Никакого копипаста валидаторов — общий модуль/библиотека для работы со схемами.
- Проверка всех YAML-конфигов агентов против схемы.
- Падение валидации = падение сборки.
- Локально команда из Makefile воспроизводит проверку.

### 6. Согласованность с существующими конфигами

- Изучить уже существующие конфиги в `backend/agents/` (например, `coder.yaml`, `reviewer.yaml`).
- Новые конфиги для пяти ролей должны следовать тому же формату и соглашениям именования.
- При расхождениях — задокументировать и при необходимости обновить существующие конфиги.

### 7. Жизненный цикл конфигов и отказ при некорректных данных

- **Старт приложения = точка валидации:** все конфиги агентов загружаются и валидируются **до** начала работы executor'ов. Если любой конфиг некорректен (невалидна схема, `temperature` вне границ, `prompt_name` содержит запрещённые символы, отсутствует обязательный файл промпта), приложение **не запускается** — явный краш с понятным сообщением об ошибке.
- **Нет graceful degradation для конфигов:** мы не пытаемся "подняться без reviewer" или "использовать значения по умолчанию для сломанного temperature". Это гарантирует детерминированное поведение и упрощает debugging.
- **Все пять базовых агентов обязательны:** Orchestrator, Planner, Developer, Reviewer, Tester должны быть определены в конфигах и быть `is_active: true` для штатного запуска пайплайна.
