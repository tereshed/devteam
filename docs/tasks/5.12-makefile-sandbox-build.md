## Описание

Интеграционные тесты sandbox (**5.14**), ручные проверки **`DockerSandboxRunner`** (**5.5**) и сценарии с реальным Docker ожидают **локально собранный** (или pulled) образ с тегом, согласованным с **allowlist** в коде (например **`devteam/sandbox-claude:local`** — см. **`DefaultAllowedSandboxImages`** в **`docker_runner.go`** и тесты).

**Цель задачи:** зафиксировать в **`Makefile`** устойчивую схему сборки sandbox-образов, которая:

1. Следует **DRY**: одно шаблонное правило на все подкаталоги **`deployment/sandbox/<agent>/`**, без копипаста **`docker build`** на каждый новый backend.
2. Закрывает **инъекции флагов** при переопределении имён образов через переменные окружения / аргументы **`make`**.
3. Даёт **быстрый фейл** с понятным сообщением, если Docker Engine недоступен.
4. Явно согласована с **`.cursor/rules/deploy.mdc`**: почему сборка **не** обязана жить в **`docker-compose.yml`** (см. раздел **Compliance**).
5. **Обязательно** включает **BuildKit** для целей сборки sandbox.
6. Документирована в **`make help`** и в **`README`** (строка **5.12** — ссылка «**детали**» на этот файл).

**Границы задачи:** только **оркестрация сборки** через **`Makefile`**, вспомогательная цель **`check-docker`** и краткая документация. **Не** дублировать содержание **5.1**/**5.2**. **Не** подменять **5.11** (монтирование **`docker.sock`**).

## Compliance: `docker-compose.yml` vs `docker build` для sandbox

В **`.cursor/rules/deploy.mdc`** для локальной разработки обязателен **`docker-compose.yml`** и **`Makefile`** как точка входа. Это **не** означает, что **каждый** образ в проекте обязан объявляться сервисом в compose.

**Архитектурное решение (зафиксировать в PR и при необходимости в комментарии в `Makefile`):**

- **`docker-compose.yml`** описывает **долгоживущий стек** приложения (API, YugabyteDB и т.д.). Sandbox-образы — **эфемерные артефакты** для задач агентов: они тяжёлые, реже пересобираются, не являются зависимостью **`docker compose up`** для дневной работы с API.
- Включение **`sandbox-claude`** как сервиса «на каждый `up`» раздувает время старта и смешивает роли: **runtime стека** vs **инструмент CI/интеграционных тестов**.
- Поэтому **сознательно** используем **`docker build`** (через **`make sandbox-build`**) **вне** основного compose. Ревьюер **не** должен отклонять PR за «отсутствие sandbox в compose», если в **`Makefile`**/комментарии явно указано это обоснование и единая точка входа **`make`** сохранена.

Если в будущем команда решит перенести сборку в **`docker compose build`** отдельного профиля (**`profiles: [sandbox]`**), это отдельная задача: текущая **5.12** остаётся валидной до явной миграции.

## Безопасность: инъекция флагов и переменные `make`

**Критично:** любые переменные Make, попадающие в аргументы **`docker`** (в первую очередь **`-t`**, пути **`-f`**, контекст), **нельзя** подставлять без кавычек в shell-рецепте.

**Плохо (запрещено):**  
`docker build -t $(SANDBOX_CLAUDE_IMAGE):$(SANDBOX_IMAGE_TAG) ...`  
Вызов вида **`make sandbox-build SANDBOX_IMAGE_TAG='local --build-arg=FOO=bar'`** превращает одну «логическую» строку в **несколько argv** у **`docker`**, открывая вектор **flag injection**.

**Обязательно:**

- Весь ref для **`-t`** передавать как **один** аргумент shell, в **двойных кавычках**:  
  **`docker build -t "$(SANDBOX_CLAUDE_IMAGE):$(SANDBOX_IMAGE_TAG)" ...`**  
  либо, если ref собирается в одну переменную, **`docker build -t "$(SANDBOX_IMAGE_REF)" ...`**.
- То же для путей: **`"-f deployment/sandbox/$*/Dockerfile"`** и контекст **`"deployment/sandbox/$*"`**, если в рецепте есть риск пробелов (диск с пробелами редок, но кавычки — дешёвая страховка и сигнал ревьюеру).
- **Опционально:** переменные по умолчанию через **`?=`** для переопределения **полного** ref или пары имя/тег — только вместе с правилом выше и кратким комментарием «кавычки обязательны, не снимать для удобства».

Шаблонное правило ниже использует литерал **`devteam/sandbox-$*:local`** внутри **`"..."`**, что исключает подстановку произвольного ref из переменной для **дефолтного** сценария; при введении override-переменных — см. абзац про кавычки.

## DRY: шаблонное правило (Pattern Rule)

**Запрещено** плодить копии рецепта: **`sandbox-build`**, **`sandbox-build-aider`**, десять ручек на каждый новый агент.

**Обязательная схема (GNU Make):**

```makefile
# В начале файла или непосредственно перед блоком sandbox (см. раздел Performance):
export DOCKER_BUILDKIT := 1

.PHONY: check-docker sandbox-build sandbox-build-claude
# sandbox-build-aider — добавить в .PHONY после появления deployment/sandbox/aider/

check-docker:
	@docker info >/dev/null 2>&1 || (echo "Ошибка: Docker Engine недоступен (демон не запущен или нет прав). Запустите Docker и повторите." >&2 && exit 1)

# Единый рецепт: stem = имя подкаталога deployment/sandbox/<stem>/
sandbox-build-%: check-docker
	docker build -t "devteam/sandbox-$*:local" -f "deployment/sandbox/$*/Dockerfile" "deployment/sandbox/$*"

# Дефолтная сборка для большинства разработчиков и CI
sandbox-build: sandbox-build-claude ## Собрать дефолтный sandbox-образ (Claude)
```

**Пояснения для исполнителя:**

- Имена целей **`sandbox-build-claude`** и, после появления каталога, **`sandbox-build-aider`** добавляются в **`.PHONY`** явно; шаблон **`sandbox-build-%`** в **`.PHONY`** в GNU Make не объявляют — объявляют конкретные производные цели (**`sandbox-build-claude`** и т.д.).
- **STEM (`%`):** допускать только идентификаторы, совпадающие с именами каталогов в репозитории (**`claude`**, позже **`aider`**). При необходимости ограничить список через **`$(filter claude aider,$*)`** с **`$(error ...)`** для неизвестного stem — по согласованию в PR.
- **GNU Make 3.81 (macOS по умолчанию):** цель вида **`sandbox-build-claude`**, объявленная в **`.PHONY`**, **не** сопоставляется с обычным правилом **`sandbox-build-%`** — получается «Nothing to be done» без **`docker build`**. Обход: **статическое шаблонное правило** **`$(SANDBOX_BUILD_TARGETS): sandbox-build-%:`**, где **`SANDBOX_BUILD_TARGETS`** — список **`sandbox-build-<stem>`** для stem из allowlist; **`.PHONY`** перечисляет эти конкретные цели. Строка **`sandbox-build: sandbox-build-claude`** не должна содержать **`## ...`** на той же строке: в 3.81 это превращается в лишние prerequisites.
- Дефолт **`sandbox-build` → `sandbox-build-claude`** должен совпадать с **allowlist** и **`defaultSandboxImage`** в тестах.

## Краевые случаи: быстрый фейл (`check-docker`)

Перед **`docker build`** выполнять проверку доступности демона:

```makefile
check-docker:
	@docker info >/dev/null 2>&1 || (echo "Ошибка: Docker Engine недоступен ..." >&2 && exit 1)
```

**Обязательно:** все цели вида **`sandbox-build-%`** (и любые агрегирующие цели, которые вызывают **`docker build`** для sandbox) должны зависеть от **`check-docker`**, чтобы не выдавать новичку непарсируемый traceback Docker без контекста.

Сообщение — краткое, по-русски или по-английски единообразно с остальным **`Makefile`**, в **stderr**.

## Performance: BuildKit

**`DOCKER_BUILDKIT=1`** для целей сборки sandbox — **обязательное** требование, не рекомендация.

**Реализация:** в корневом **`Makefile`** задать **`export DOCKER_BUILDKIT := 1`** в области, где объявлены рецепты **`sandbox-build*`** (достаточно один раз на файл, если не конфликтует с осознанным отключением BuildKit в других целях; при конфликте — **`export`** только в рецепте через **`DOCKER_BUILDKIT=1 docker build ...`** для строк sandbox — предпочтительнее **не** ломать глобальные ожидания и обсудить в PR).

Итог: любой **`docker build`**, относящийся к **5.12**, выполняется с включённым BuildKit.

## Критерии приёмки

- [x] В **`Makefile`**: **`export DOCKER_BUILDKIT := 1`** (или эквивалент, гарантирующий BuildKit для sandbox **`docker build`**).
- [x] Цель **`check-docker`** с понятным сообщением и **`exit 1`**, если **`docker info`** недоступен.
- [x] Шаблонное правило **`sandbox-build-%`** с зависимостью от **`check-docker`**; рецепт **`docker build`** с **`-t`**, **`-f`** и контекстом в **кавычках** (минимум вокруг полного ref **`"devteam/sandbox-$*:local"`**). (Реализовано как **`$(SANDBOX_BUILD_TARGETS): sandbox-build-%:`** — см. пояснение про GNU Make 3.81 в разделе DRY.)
- [x] **`sandbox-build: sandbox-build-claude`** (или эквивалент); описание в **`make help`** (ручные **`@echo`**; **`##`** на той же строке не используется из‑за Make 3.81).
- [x] **`sandbox-build`**, **`check-docker`**, **`sandbox-build-claude`** в **`.PHONY`** (через **`$(SANDBOX_BUILD_TARGETS)`**); **`sandbox-build-aider`** — после **`deployment/sandbox/aider/`** и **`aider`** в **`SANDBOX_BUILDABLE_STEMS`**.
