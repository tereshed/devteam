## Описание

> **⚠️ Исключение из Правила 7 (backend.mdc):** Создание MCP-инструментов для задач (`task_list`, `task_get`, `task_create`, `task_update`) вынесено в отдельную задачу **3.9**. В рамках текущей задачи (3.5) реализовывать инструменты в `internal/mcp/` **НЕ НУЖНО**.

Создать HTTP-обработчик для задач. Это **самый крупный handler в приложении** — 10 эндпоинтов, включая CRUD, три state-transition действия (pause/cancel/resume) и вложенный ресурс messages. Handler — тонкий слой: парсинг запроса, извлечение auth-контекста, вызов Service, маппинг ошибок в HTTP-статусы, формирование ответа через DTO. Бизнес-логика, валидация state machine и ABAC — **только** в `TaskService`.

**Архитектурная особенность:** Tasks используют две группы роутов:
- **Вложенные в projects:** `POST/GET /projects/:project_id/tasks` — создание и список задач (проект определяется из path)
- **Самостоятельные:** `GET/PUT/DELETE /tasks/:id` и action-эндпоинты — задача уже содержит `project_id`, ABAC проверяется внутри Service

## Структура

```go
type TaskHandler struct {
    service service.TaskService
}

func NewTaskHandler(service service.TaskService) *TaskHandler
```

**Одна зависимость** — `TaskService`. Handler не знает о `ProjectService`, `TeamService`, `TaskRepository` — всё скрыто за сервисом.

## Методы

### Create (`POST /projects/:project_id/tasks`)

1. `requireAuth(c)` → `userID`, `userRole` — при ошибке: `return` (401 уже отправлен внутри `requireAuth`)
2. `ctx := c.Request.Context()`
3. `projectID, err := uuid.Parse(c.Param("id"))` — при `err != nil`: вернуть `400 Bad Request` ("Invalid project ID format")
4. `c.ShouldBindJSON(&req)` → `dto.CreateTaskRequest` — при `err != nil`: вернуть `400 Bad Request` с `err.Error()` (детали валидации binding-тегов)
5. `task, err := service.Create(ctx, userID, userRole, projectID, req)` — при `err != nil`: `writeTaskServiceError(c, err)` + return
6. `c.JSON(http.StatusCreated, dto.ToTaskResponse(task))`

### List (`GET /projects/:project_id/tasks`)

1. `requireAuth(c)` → `userID`, `userRole` — при ошибке: `return`
2. `ctx := c.Request.Context()`
3. `projectID, err := uuid.Parse(c.Param("id"))` — при `err != nil`: вернуть `400 Bad Request` ("Invalid project ID format")
4. `c.ShouldBindQuery(&req)` → `dto.ListTasksRequest` — при `err != nil`: вернуть `400 Bad Request`
5. **Нормализация пагинации перед вызовом Service:**
   ```go
   req.Limit, req.Offset = normalizeTaskListPagination(req.Limit, req.Offset)
   ```
   Если клиент не передал `limit` (парсится как `0` из query) → устанавливается `50`. Если `limit > 200` → обрезается до `200`. Мутируем спарсенный DTO **до** передачи в Service — Service получает уже валидные значения.
6. `tasks, total, err := service.List(ctx, userID, userRole, projectID, req)` — при `err != nil`: `writeTaskServiceError(c, err)` + return
7. `c.JSON(http.StatusOK, dto.ToTaskListResponse(tasks, total, req.Limit, req.Offset))`

**Примечание:** `req.Limit` и `req.Offset` уже нормализованы на шаге 5 — они передаются и в Service, и в response DTO, обеспечивая консистентность. Service дополнительно нормализует лимиты (defense in depth), но handler обязан передать корректные значения.

## Маппинг ошибок

Приватная функция `writeTaskServiceError` — маппинг Service-ошибок в HTTP-статусы.

```go
func writeTaskServiceError(c *gin.Context, err error) {
    switch {
    // 404 Not Found
    case errors.Is(err, service.ErrTaskNotFound):
        apierror.JSON(c, http.StatusNotFound, apierror.ErrNotFound, err.Error())
    case errors.Is(err, service.ErrProjectNotFound):
        apierror.JSON(c, http.StatusNotFound, apierror.ErrNotFound, err.Error())

    // 403 Forbidden
    case errors.Is(err, service.ErrProjectForbidden):
        apierror.JSON(c, http.StatusForbidden, apierror.ErrForbidden, err.Error())

    // 409 Conflict (state machine)
    case errors.Is(err, service.ErrTaskInvalidTransition):
        apierror.JSON(c, http.StatusConflict, apierror.ErrConflict, err.Error())
    case errors.Is(err, service.ErrTaskTerminalStatus):
        apierror.JSON(c, http.StatusConflict, apierror.ErrConflict, err.Error())

    // 422 Unprocessable Entity (бизнес-валидация)
    case errors.Is(err, service.ErrAgentNotInTeam):
        apierror.JSON(c, http.StatusUnprocessableEntity, apierror.ErrUnprocessable, err.Error())
    case errors.Is(err, service.ErrTaskParentNotFound):
        apierror.JSON(c, http.StatusUnprocessableEntity, apierror.ErrUnprocessable, err.Error())

    // 400 Bad Request (валидация полей)
    case errors.Is(err, service.ErrTaskInvalidTitle):
        apierror.JSON(c, http.StatusBadRequest, apierror.ErrBadRequest, err.Error())
    case errors.Is(err, service.ErrTaskInvalidPriority):
        apierror.JSON(c, http.StatusBadRequest, apierror.ErrBadRequest, err.Error())
    case errors.Is(err, service.ErrTaskInvalidStatus):
        apierror.JSON(c, http.StatusBadRequest, apierror.ErrBadRequest, err.Error())
    case errors.Is(err, service.ErrTaskMessageInvalidType):
        apierror.JSON(c, http.StatusBadRequest, apierror.ErrBadRequest, err.Error())

    // 500 Internal Server Error
    default:
        apierror.JSON(c, http.StatusInternalServerError, apierror.ErrInternalServerError, "Request failed")
    }
}
```

### Полная таблица маппинга

| Service error | HTTP Status | apierror code | Когда |
|---|---|---|---|
| `ErrTaskNotFound` | 404 | `not_found` | GetByID, Update, Delete, Pause/Cancel/Resume, Messages |
| `ErrProjectNotFound` | 404 | `not_found` | Create, List (проект не найден) |
| `ErrProjectForbidden` | 403 | `forbidden` | Любой — нет доступа к проекту |
| `ErrTaskInvalidTransition` | 409 | `conflict` | Update (status), Pause, Cancel, Resume |
| `ErrTaskTerminalStatus` | 409 | `conflict` | Pause, Cancel, Resume (задача в терминальном статусе) |
| `ErrAgentNotInTeam` | 422 | `unprocessable` | Create, Update (агент не в команде) |
| `ErrTaskParentNotFound` | 422 | `unprocessable` | Create (parent_task_id не существует) |
| `ErrTaskInvalidTitle` | 400 | `bad_request` | Create |
| `ErrTaskInvalidPriority` | 400 | `bad_request` | Create, Update |
| `ErrTaskInvalidStatus` | 400 | `bad_request` | Update (невалидный статус) |
| `ErrTaskMessageInvalidType` | 400 | `bad_request` | AddMessage |
| прочие | 500 | `internal_server_error` | Сбой БД и т.д. |

**Почему `409 Conflict` для state machine ошибок:** Клиент пытается выполнить действие, которое конфликтует с текущим состоянием ресурса (RFC 7231 §6.5.8). Это отличается от `400 Bad Request` (синтаксис) и `422 Unprocessable Entity` (семантика).

**Почему `422` для `ErrAgentNotInTeam` и `ErrTaskParentNotFound`:** Запрос синтаксически валиден, но семантически невыполним — связанная сущность не удовлетворяет бизнес-условию.

## Хелперы (переиспользование)

Функции `requireAuth`, `getUserID`, `getUserRole` уже определены в `project_handler.go`. Они доступны в пределах пакета `handler` — повторно объявлять не нужно.

Дополнительно для TaskHandler:

```go
func normalizeTaskListPagination(limit, offset int) (int, int) {
    if limit <= 0 {
        limit = 50
    }
    if limit > 200 {
        limit = 200
    }
    if offset < 0 {
        offset = 0
    }
    return limit, offset
}
```

**Как использовать:** Handler парсит `dto.ListTasksRequest` из query params (Gin `ShouldBindQuery`). Если клиент не передал `limit`, Gin оставляет его как `0` (zero value для `int`). Хелпер **мутирует** спарсенные из DTO значения **до** передачи в Service:

```go
var req dto.ListTasksRequest
if err := c.ShouldBindQuery(&req); err != nil {
    apierror.JSON(c, http.StatusBadRequest, apierror.ErrBadRequest, err.Error())
    return
}
req.Limit, req.Offset = normalizeTaskListPagination(req.Limit, req.Offset)
// Теперь req содержит валидные limit/offset → передаём в Service и в response DTO
```

Нормализованные `req.Limit` и `req.Offset` используются в двух местах:
1. Передаются в `service.List(ctx, ..., req)` — Service получает уже корректные значения
2. Передаются в `dto.ToTaskListResponse(tasks, total, req.Limit, req.Offset)` — response содержит фактические значения пагинации

**Почему отдельный хелпер, а не `normalizeListPagination` из project_handler:** Проекты используют дефолт 20/max 100, задачи — 50/200. Разные лимиты для разных ресурсов. В будущем можно рефакторить в generic-функцию с параметрами.

### Роуты

Задачи используют **две группы** роутов из-за разной семантики URL:

```go
// Tasks (вложенный ресурс проекта — создание и список)
projects.POST("/:id/tasks", deps.TaskHandler.Create)
projects.GET("/:id/tasks", deps.TaskHandler.List)

// Tasks (самостоятельный ресурс — CRUD по task ID)
tasks := api.Group("/tasks")
tasks.Use(authMW)
{
    tasks.GET("/:id", deps.TaskHandler.GetByID)
    tasks.PUT("/:id", deps.TaskHandler.Update)
    tasks.DELETE("/:id", deps.TaskHandler.Delete)

    // State transitions (action endpoints)
    tasks.POST("/:id/pause", deps.TaskHandler.Pause)
    tasks.POST("/:id/cancel", deps.TaskHandler.Cancel)
    tasks.POST("/:id/resume", deps.TaskHandler.Resume)

    // Messages (вложенный ресурс задачи)
    tasks.GET("/:id/messages", deps.TaskHandler.ListMessages)
    tasks.POST("/:id/messages", deps.TaskHandler.AddMessage)
}
```

**Важно: порядок роутов в projects группе.**

Вложенные роуты tasks (`/:id/tasks`) должны быть зарегистрированы в правильном месте:
```
/:id/team      ← уже есть (TeamHandler)
/:id/tasks     ← ДОБАВИТЬ (TaskHandler)
/:id           ← уже есть (ProjectHandler.GetByID)
```

Gin **не перепутает** `/:id/tasks` с `/:id`, потому что `/tasks` — фиксированный суффикс. Но для консистентности — вложенные ресурсы регистрируются **перед** `/:id`.

### Параметр path: `project_id` vs `id`

В группе `/projects`, Gin использует `:id` для project ID. В handler'е `Create` и `List` — `c.Param("id")` возвращает project ID. В группе `/tasks` — `c.Param("id")` возвращает task ID. Коллизии нет, потому что это разные route groups.

**Альтернатива (рассмотреть при реализации):** Если нужна явность, можно создать отдельную подгруппу `projectTasks` внутри projects с параметром `:project_id`. Но это усложняет Gin routing. Текущий подход (`:id` в обоих группах, различаемый контекстом) — проще и совпадает с паттерном `TeamHandler`.

7. **DTO конвертеры** — `dto.ToTaskResponse()`, `dto.ToTaskListResponse()`, `dto.ToTaskMessageResponse()`
8. **Один файл** — все 10 методов в одном файле `task_handler.go` (как `project_handler.go`)
9. **`writeTaskServiceError`** — приватная функция маппинга ошибок (аналог `writeProjectServiceError`)
10. **Action endpoints возвращают ресурс** — Pause/Cancel/Resume возвращают обновлённый `TaskResponse` (не просто `{"status": "ok"}`)
