# 5.9 Resource limits (CPU, Memory) при создании контейнера

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/docker_runner.go` — сборка **`containertypes.HostConfig`** / **`Resources`** при **`ContainerCreate`**; вспомогательные функции **`effectiveMemoryBytes`**, **`effectivePidsLimit`**, **`effectiveNanoCPUs`** (вынести в **`resource_limits.go`** по желанию PR, пакет **`sandbox`**); **`options_validate.go`** — границы и запреты на **`ResourceLimit`** до Docker; **`types.go`** — комментарии к полям **`ResourceLimit`**; после **5.10** — чтение полов/потолков из **`SandboxConfig`** и передача **политики** в конструктор **`NewDockerSandboxRunner`** (без глобалов).  
**Статус:** ✅ Выполнена

---

## Описание

Зафиксировать и довести до **продуктового** уровня применение ** cgroup-лимитов** при создании sandbox-контейнера: **память** (RAM + политика **swap**), **CPU** (**`NanoCPUs`** в терминах Docker/Moby), **лимит PID** (fork-bomb), опционально **диск** (**`DiskMB`** в **`ResourceLimit`** — см. раздел «Диск»). Минимальный «пол» и верхний **потолок** должны быть **однозначны**, согласованы с **5.4** (**`SandboxOptions.ResourceLimit`**) и **5.10** (конфиг приложения), без «тихого» игнорирования частично применённых лимитов со стороны движка там, где это **критично для безопасности** — с оговоркой для **swap-лимитов на ядрах без swapaccount** (см. раздел «Предупреждения демона»).

**Границы задачи:** только то, что попадает в **`HostConfig`** (и связанные поля **`ContainerCreate`**) **до** **`ContainerStart`**. Таймауты жизни контейнера и **SIGTERM/SIGKILL** — **5.8**; сбор артефактов — **5.7**; стрим логов — **5.6**.

**Связь с текущим кодом:** в **5.5** уже заложены ориентиры (**память ~1 GiB floor**, **~16 GiB ceil**, **`PidsLimit`**, **`MemorySwap == Memory`**, **`NanoCPUs`** только при **`NanoCPUs > 0`**). Задача **5.9** **закрывает** контракт: валидация входа (**в т.ч. защита от integer overflow**), **жёсткий дефолт CPU при `NanoCPUs <= 0`** (см. ниже — **не** оставлять безлимитный CPU для агентов), **`DiskMB`**, перенос констант в конфиг (**5.10**), **чистые** **`effective*`** с параметрами floor/ceil, тесты (**5.13**), документация и критерии приёмки.

## Маппинг `ResourceLimit` → Docker `HostConfig.Resources`

| Поле Go (`ResourceLimit`) | Docker / cgroup | Поведение |
|---------------------------|-----------------|-----------|
| **`MemoryMB`** | **`Resources.Memory`** (байты) | После **`effectiveMemoryBytes(..., floorMB, ceilMB)`** (см. «Чистые функции»): не ниже пола, не выше потолка. Ноль / не задано → пол. **Перед** умножением **`MB → байты`** — проверка на **overflow** и на **`MemoryMB ≤ верхней границы, безопасной для int64`** (отдельный подраздел). |
| *(политика swap)* | **`Resources.MemorySwap`** | **`MemorySwap == Memory`** — отключить использование swap поверх лимита RAM там, где ядро/cgroup это поддерживают. На хостах **без** поддержки учёта swap в cgroup (**`swapaccount`/`memsw`**) Docker часто возвращает **Warning** вида *«Your kernel does not support swap limit capabilities»* — см. раздел «Предупреждения демона» (**не** fail fast). |
| **`NanoCPUs`** | **`Resources.NanoCPUs`** | **`1_000_000_000` = 1 CPU** (контракт Docker). **Архитектурно зафиксировано:** если **`NanoCPUs <= 0`** (ноль или не задано), в **`HostConfig.Resources` обязано** попасть **`NanoCPUs = DefaultSandboxNanoCPUs`**, где **`DefaultSandboxNanoCPUs = 1_000_000_000`** (ровно **1 CPU**), если **5.10** не переопределит дефолт через конфиг. **Безлимитный CPU** для LLM-sandbox (Python/Node, бесконечные циклы) **запрещён** — это **DoS** всего хоста. Значение **> ceil** — clamp до ceil **или** **`ErrInvalidOptions`** — согласовать с **`MemoryMB`**, но **верхний потолок** обязателен. |
| **`PIDsLimit`** | **`Resources.PidsLimit`** | Floor/ceil через **`effectivePidsLimit(..., floor, ceil)`**; указатель на int64 в API контейнера — DRY одна реализация. При необходимости — та же дисциплина **«проверка до умножений/приведений»**, если появятся производные вычисления. |
| **`DiskMB`** | см. раздел «Диск» | На момент задачи может быть **0 = не задавать** или реализовать через **`StorageOpt`** / политику томов — зафиксировать один вариант в PR. |

## Integer overflow и безопасное приведение типов (**КРИТИЧНО**, Rule 3 **`review.mdc`**)

**Проблема:** поле **`MemoryMB`** (и любые будущие «мегабайты» в int) при значениях порядка **`math.MaxInt32`** / **`MaxInt64`** дают **`MemoryMB * 1024 * 1024`** с **целочисленным переполнением** в Go: результат становится **отрицательным или малым положительным**, проходит «валидацию» относительно ceil **неверно**, хост получает контейнер с **минимальной памятью** — обход защиты.

**Требование:**

1. **До** любого умножения **`MemoryMB * 1024 * 1024`** (или эквивалента) проверять **`MemoryMB`** на **верхнюю границу в единицах MB**, выведенную из **безопасного максимума байт** (например **`ceilBytes / (1024*1024)`**, с явной целочисленной арифметикой **без** overflow при делении).
2. Вычисление байт из MB выполнять в **`int64`** с **явной** проверкой: если после умножения значение **< 0** или **> ceilBytes** до clamp — считать это **ошибкой валидации** (**`ErrInvalidOptions`**), а не «clamp в пол».
3. Аналогичная дисциплина для любых других полей, где ввод умножается на константу для сравнения с лимитом (**`DiskMB`**, будущие поля).

## Предупреждения демона (`CreateResponse.Warnings`)

**Базовое требование:** при **`len(Warnings) > 0`** — **каждая** строка в лог уровня **WARN** (**5.5**).

### Swap / cgroup swap limits (**хрупкость хоста**)

**Проблема:** при **`MemorySwap == Memory`** на Linux **без** включённого учёта swap в cgroup (**`swapaccount=1`** и т.п.) Docker выдаёт предупреждение о **неподдержке swap limit**. Политика **«fail fast на любое Warning»** ломает запуск на типичных dev/stage хостах.

**Требование (зафиксировано):** предупреждения, указывающие на **отсутствие поддержки лимитов swap в ядре/cgroup** (в т.ч. текст про *«kernel does not support swap limit capabilities»* и согласованные формулировки в новых версиях движка после нормализации/подстроки в allowlist), **никогда** не переводят **`RunTask`** в ошибку: **только WARN в лог**, контейнер считается **созданным успешно** с этой точки зрения. Память по-прежнему ограничивается **`Memory`**; отсутствие swap-capability означает лишь **меньшую жёсткость** по swap, а не отказ в запуске.

### Прочие предупреждения о лимитах

Для **остальных** предупреждений (memory/cpuset/pids и т.д., **не** swap-kernel) допускается политика **fail fast** только для **явного подмножества** строк, занесённого в код с комментарием «почему фатально»; подмножество **не** должно включать swap-kernel класс выше. Если для класса нет уверенности — по умолчанию **WARN без abort**, пока не добавлен ADR.

## Архитектура: чистые функции **`effective*`** (**DRY**, Rule 2 **`review.mdc`**)

**Проблема:** **`effectiveMemoryBytes`** / **`effectiveNanoCPUs`** / **`effectivePidsLimit`**, читающие **глобальные константы** или **синглтон конфига**, усложняют тесты и нарушают явный DI (**backend.mdc**).

**Требование:**

1. Функции **чистые** (**pure**): результат зависит **только** от аргументов, **без** обращения к пакетным глобалам и **`init()`**-конфигу.
2. Сигнатуры в духе: **`effectiveMemoryBytes(requestedMB int, floorBytes, ceilBytes int64) int64`**, **`effectivePidsLimit(requested int, floor, ceil int64) int64`**, **`effectiveNanoCPUs(requestedNano int64, defaultWhenNonPositive, ceilNano int64) int64`** (имена на усмотрение PR, смысл обязателен: **floor/ceil/default** передаются снаружи).
3. Сборка политики (**числа из конфига 5.10** или дефолты) — в **`cmd/api`** / фабрике раннера, один раз в структуру **`ResourceLimitPolicy`** (или поля **`DockerSandboxRunner`**), затем вызов **`effective*`** из **`RunTask`** с этой структурой.

Так **`effective*`** покрываются **юнит-тестами без моков** конфига.

## Диск (`DiskMB`)

Docker **не** задаёт единый «лимит диска контейнера» через **`Resources`** так же, как память. Варианты:

1. **`StorageOpt`** на **`graphdriver`** (например **`size=`** для **overlay2** — зависит от storage driver и версии движка).
2. Оставить **`DiskMB`** зарезервированным для **будущего** (volume с квотой / отдельный volume driver) и в **`Validate`** разрешить только **`0`** до реализации.

**Требование задачи:** явно выбрать (1) или (2) в PR; при (2) — **`Validate`** / комментарий в **`types.go`**, чтобы вызывающий слой не ожидал cgroup-лимита диска без реализации.

## Валидация (`options_validate.go`)

- **`MemoryMB`:** сначала проверка на **overflow** и на **разумный верх в MB** (см. «Integer overflow»), **затем** сравнение с политикой ceil; при нарушении — **`ErrInvalidOptions`**. Не полагаться на «умножение и потом сравнение с ceil» без предшествующей границы.
- **`NanoCPUs`:** отрицательные — **`ErrInvalidOptions`**; ноль / не задано — **не ошибка**: на этапе маппинга в Docker подставляется **`DefaultSandboxNanoCPUs`** (**1 CPU**). Сверх ceil — политика согласована с **`MemoryMB`** (clamp **с** логом **или** ошибка — одно на весь тип лимитов).
- **`PIDsLimit`:** отрицательные / ноль — по политике floor; сверх ceil — как для **`NanoCPUs`**.
- Не логировать и не возвращать в ошибках **сырой** **`SandboxOptions`** целиком — **`LogSafe()`** (**5.4**).

---

## Конфигурация (**5.10**, сквозная связь)

После появления **`SandboxConfig`**: полы/потолки (**memory floor/ceil в байтах или MB**, **pids floor/ceil**, **`defaultNanoCPUs`**, **`nanoCPUsCeil`**) загружаются в **`cmd/api`** и передаются в **`ResourceLimitPolicy`**, откуда **`RunTask`** берёт аргументы для **`effective*`**.

До **5.10** допустимо собирать **`ResourceLimitPolicy`** из **констант** в одном месте (**`resource_limits_defaults.go`** или конструктор раннера), но **не** читать константы **внутри** тела **`effective*`** (нарушение чистоты).

---

## OOM и классификация в **`SandboxStatus`**

При **`OOMKilled`** в **`ContainerInspect`** / wait-ответе — статус должен оставаться согласованным с **5.5**/**5.7** (**`failed`** vs отдельный маркер — если меняется enum, синхронизировать **5.4** и документацию). **5.9** не обязана вводить новый **`SandboxStatusType`**, но **обязана** явно описать ожидаемое поведение в PR при изменении маппинга.

---

## Безопасность и DoS (**`review.mdc`**)

- Лимиты — **защита хоста** от одной задачи; верхний **ceil** — защита от запроса «512 GiB» через API.
- **CPU:** **`NanoCPUs <= 0` → 1 CPU по умолчанию** — обязательная мера против DoS от CPU-bound кода агента.
- Не ослаблять **floor** по памяти/PIDs без архитектурного ADR — регресс **5.5**.

## Критерии приёмки

- [ ] **`HostConfig.Resources`:** **`Memory`**, **`MemorySwap == Memory`** (где поддерживается), **`PidsLimit`**, **`NanoCPUs`** — в Docker **всегда** задан осмысленный CPU-лимит: при **`ResourceLimit.NanoCPUs <= 0`** в конфиг уходит **`1_000_000_000`** (или **`defaultNanoCPUs`** из **5.10**, **не** ниже **1 CPU** без ADR).
- [ ] **Overflow:** валидация **`MemoryMB`** (и аналоги) **до** умножения на **`1024*1024`**; вычисление байт в **`int64`** без молчаливого wrap-around.
- [ ] **`Warnings`:** класс *kernel/swap limits unsupported* — **только WARN**, **без** **`RunTask` error**; прочие — по явному allowlist в коде.
- [ ] **`effective*`:** **чистые** функции с параметрами floor/ceil/default; **нет** чтения глобального конфига внутри них.
- [ ] **`DiskMB`:** реализован лимит через driver **или** явно зарезервирован с валидацией **`0`** и комментарием в **`types.go`**.
- [ ] **`ResourceLimit`:** валидация в **`options_validate.go`**, согласованная с **`effective*`**.
- [ ] **DRY:** одна реализация политики; при **5.10** — без копипасты констант в оркестраторе.
- [ ] **5.13 — юнит-тесты `effective*` (обязательные кейсы):** **`0`** / **не задано** для CPU → **ровно** дефолт **1 CPU**; **отрицательные** вводы, где по контракту ошибка — **`ErrInvalidOptions`**; **> ceil**; **< floor** (в т.ч. «запрос ниже пола» → эффективное значение **не ниже floor**); **огромные `MemoryMB`** (например около **`MaxInt`**) — **нет** panic, **нет** «маленькой памяти из-за overflow», ожидаемая ошибка валидации или безопасный отказ; **граничные** значения ровно на floor/ceil.
- [ ] При необходимости — тест на моке **`ContainerCreate`**, что в **`HostConfig.Resources`** уходят ожидаемые числа для комбинации опций + политики.
- [ ] Нет утечки секретов в логах при ошибках валидации лимитов.
