## Описание

Задача заключается в реализации метода контекстного поиска по векторной базе данных (Weaviate). Метод `SearchContext` должен выполнять семантический поиск по чанкам кода конкретного проекта, фильтровать нерелевантные результаты и возвращать наиболее точные фрагменты для использования LLM-агентами.

---

## Зависимости

- `Weaviate` — векторная БД.
- `IndexerService` / `CodeIndexer` — интерфейс и его реализация.
- `Common Mapping Helpers` — переиспользуемые функции для парсинга ответов Weaviate.

---

## Функциональные требования

### 1. Интерфейс и Валидация (Early Returns)
- **Сигнатура:** `SearchContext(ctx context.Context, projectID uuid.UUID, query string, limit int) ([]Chunk, error)`.
- **Валидация Query:** 
    - Проверка на пустоту (`query == ""`).
    - **Защита от DoS:** Ограничение максимальной длины запроса (например, `len(query) > 4096`). При нарушении возвращать `ErrQueryTooLong`.
- **Валидация Limit:** Применять хардлимит. Если `limit <= 0` или `limit > 50`, устанавливать значение по умолчанию (например, 10) или возвращать ошибку валидации.
- **Контракт по результатам:** Если поиск не дал результатов (или всё отсеяно порогами), возвращать пустой инициализированный слайс `[]Chunk{}` и `nil` ошибку.

### 2. Безопасный и Умный Поиск (Weaviate)
- **Защита от инъекций:** Использовать исключительно **официальный GraphQL builder** клиента Weaviate.
- **Изоляция данных:** Строгая фильтрация по `project_id`.
- **Порог релевантности (Certainty/Distance):** Внедрить фильтрацию по порогу схожести, чтобы не возвращать "шум" (нерелевантный код), если семантическая близость слишком низкая.
- **Оптимизация выборки:** Запрашивать только необходимые поля (content, file_path, lines и т.д.). **Не запрашивать векторы (vectors/embeddings)**.

### 3. Архитектура и DRY
- **Переиспользование маппинга:** Логику преобразования `map[string]interface{}` из Weaviate в структуру `Chunk` вынести в отдельный хелпер, если она уже используется или будет использоваться в других методах (например, при получении чанков по ID).

### 4. Наблюдаемость (Observability)
- **Логирование Latency:** Обязательно логировать время выполнения запроса к Weaviate и количество найденных/отфильтрованных результатов (уровень INFO или DEBUG).
- **Контекст:** Прокидывать `ctx` во все вызовы клиента для корректной обработки таймаутов.

## Критерии приёмки

- [ ] Метод `SearchContext` реализован с поддержкой `limit`.
- [ ] Валидация отсекает пустые, слишком длинные запросы и некорректные лимиты.
- [ ] Используется безопасный builder и фильтрация по порогу релевантности.
- [ ] При отсутствии совпадений возвращается `[]Chunk{}`, а не `nil`.
- [ ] Логика маппинга не дублируется (используется хелпер).
- [ ] В логах фиксируется время выполнения (latency) каждого поиска.
- [ ] Юнит-тесты покрывают все краевые случаи и логику валидации.
