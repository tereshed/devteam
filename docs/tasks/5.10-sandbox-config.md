# 5.10 Конфигурация: `SandboxConfig` в `config.go`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/config/config.go` — структура **`SandboxConfig`**, загрузка из **env** в **`Load()`** (или вынесенный хелпер **`loadSandboxConfig`** в том же пакете); при необходимости — **`cmd/api/main.go`** (передача политики в фабрику **`NewDockerSandboxRunner`** / конструктор раннера); **`backend/internal/sandbox/`** — замена «жёстких» дефолтов **`DefaultResourceLimitPolicy()`** на значения из **`Config.Sandbox`** (без глобального синглтона).  
**Статус:** ⬜ Не начата

---

## Описание

Вынести **операционные параметры sandbox** из разрозненных констант и комментариев «до 5.10» в **единую секцию конфигурации приложения**: структура **`SandboxConfig`**, вложенная в **`Config`**, загрузка из **переменных окружения** с **понятными именами** и **безопасными дефолтами**, валидация на этапе **`config.Load()`** (fail fast при старте API).

**Цели:**

1. **Оператор** может настроить полы/потолки лимитов, дефолтный таймаут задачи, максимум параллельных контейнеров и т.п. **без пересборки** бинарника.
2. **Код sandbox** получает **`ResourceLimitPolicy`** (или эквивалент) **через DI** из уже загруженного **`Config`**, а не читает env напрямую и не полагается на пакетные глобалы (**`backend.mdc`**, **`review.mdc`**).
3. Текущие значения из **`DefaultResourceLimitPolicy()`** / **`resource_limits_defaults.go`** остаются **fallback по умолчанию**, если соответствующие env не заданы — **без изменения поведения** для существующих деплоев.

**Границы задачи:** только **конфиг и загрузка**; монтирование **`docker.sock`** — **5.11**; **`Makefile sandbox-build`** — **5.12**; юнит-тесты раннера — **5.13**; интеграционный тест — **5.14**. Логика **`effective*`** и **`HostConfig`** — **5.9** (задача **5.10** **подаёт числа**, а не дублирует маппинг в Docker).

## Поля `SandboxConfig` (минимальный контракт)

Имена полей и env — **ориентир** для PR; важен **смысл** и **однозначная валидация**.

| Смысл | Пример поля Go | Пример env | Заметки |
|--------|------------------|------------|---------|
| Политика памяти: пол в байтах | `MemoryFloorBytes` | `SANDBOX_MEMORY_FLOOR_BYTES` | Дефолт = текущий **`1 GiB`** из **`DefaultResourceLimitPolicy`** |
| Политика памяти: потолок в байтах | `MemoryCeilBytes` | `SANDBOX_MEMORY_CEIL_BYTES` | Дефолт = **`16 GiB`**; **`ceil ≥ floor`** |
| PID: пол / потолок | `PidsFloor`, `PidsCeil` | `SANDBOX_PIDS_FLOOR`, `SANDBOX_PIDS_CEIL` | Согласовать с **5.9** |
| CPU: дефолт при «не задано» в опциях | `DefaultNanoCPUs` | `SANDBOX_DEFAULT_NANO_CPUS` | Не ниже **`1_000_000_000`** без явного ADR (**5.9**) |
| CPU: верхний потолок | `NanoCPUsCeil` | `SANDBOX_NANO_CPUS_CEIL` | **`≥ DefaultNanoCPUs`** |
| Таймаут выполнения задачи по умолчанию | `DefaultTaskTimeout` | `SANDBOX_DEFAULT_TIMEOUT` | **`time.Duration`** через **`time.ParseDuration`** (как **`SERVER_READ_TIMEOUT`**) |
| Макс. параллельных sandbox на процесс | `MaxConcurrent` | `SANDBOX_MAX_CONCURRENT` | **≥ 1**; очередь воркеров может прийти позже — поле **резервирует** контракт для **Phase 2** / **6.x** |

Дополнительные поля (образ по умолчанию, сеть, registry pull policy) — **только** если уже есть потребитель в коде; иначе **не раздувать** задачу без отдельного пункта roadmap.

## Загрузка и валидация в `Load()`

- Парсинг **int64** для байт и **`NanoCPUs`** — через **`strconv.ParseInt`** с явной базой **10** и проверкой диапазона; избегать **`Atoi`** для значений, которые могут превысить **`MaxInt32`**.
- **Диапазоны:** отрицательные и нулевые там, где «ноль = дефолт» — **документировать** в комментарии к полю; противоречия (**`ceil < floor`**) — **`error`** из **`Load()`** с текстом **без** утечки секретов.
- **`DefaultTaskTimeout`:** пустой env → дефолт (например **`30m`**, согласовать с **5.8** / **`SandboxOptions.Timeout`** в домене); невалидная строка duration → **`error`**.
- **`MaxConcurrent`:** невалидное или **`< 1`** → **`error`** или clamp **только** если зафиксировано в PR; предпочтительно **явная ошибка** в prod-подобных окружениях.

## DI и границы пакетов

- Пакет **`config`** **не** импортирует **`internal/sandbox`** (циклы и нарушение слоёв). Конфиг — **«сырые» числа и duration**; сборка **`ResourceLimitPolicy`** — в **`cmd/api`** или в конструкторе **`NewDockerSandboxRunner(..., policy sandbox.ResourceLimitPolicy)`** из полей **`cfg.Sandbox`**.
- **`normalizeResourceLimitPolicy`** в **`sandbox`** может принимать **`ResourceLimitPolicy`** «сверху» и дополнять пропуски дефолтами — **без** чтения **`os.Getenv`** внутри **`sandbox`**.

## Критерии приёмки

- [ ] В **`Config`** добавлено поле **`Sandbox SandboxConfig`** (или **`Sandbox`** с тем же смыслом), заполняется в **`Load()`**.
- [ ] Все перечисленные параметры имеют **документированные env** и **безопасные дефолты**, совместимые с текущим **`DefaultResourceLimitPolicy()`** при пустом env.
- [ ] **`Load()`** возвращает **`error`** при противоречивых или заведомо опасных значениях (например **`ceil < floor`**, отрицательный **`MaxConcurrent`**).
- [ ] **`internal/sandbox`** не вызывает **`os.Getenv`** для этих параметров; политика передаётся **извне**.
- [ ] В **`README`** (таблица Sprint 5) у строки **5.10** — ссылка «**детали**» на этот файл.
- [ ] При необходимости — краткое упоминание ключевых env в **`README`** или в **`deployment/docker-compose`** в рамках **5.11** (не обязательно в рамках только **5.10**, но дублирования имён env быть не должно).
