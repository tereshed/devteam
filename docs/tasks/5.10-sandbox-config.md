# 5.10 Конфигурация: `SandboxConfig` в `config.go`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/config/config.go` — структура **`SandboxConfig`**, загрузка из **env** в **`Load()`** (или вынесенный хелпер **`loadSandboxConfig`** в том же пакете); **`backend/internal/config/config_test.go`** — **юнит-тесты** парсинга env, дефолтов и валидации (см. «Критерии приёмки»); при необходимости — **`cmd/api/main.go`** (передача политики в фабрику **`NewDockerSandboxRunner`** / конструктор раннера); **`backend/internal/sandbox/`** — замена «жёстких» дефолтов **`DefaultResourceLimitPolicy()`** на значения из **`Config.Sandbox`** (без глобального синглтона).  
**Статус:** ✅ Выполнена

---

## Описание

Вынести **операционные параметры sandbox** из разрозненных констант и комментариев «до 5.10» в **единую секцию конфигурации приложения**: структура **`SandboxConfig`**, вложенная в **`Config`**, загрузка из **переменных окружения** с **понятными именами** и **безопасными дефолтами**, валидация на этапе **`config.Load()`** (fail fast при старте API).

**Цели:**

1. **Оператор** может настроить полы/потолки лимитов, дефолтный таймаут задачи, максимум параллельных контейнеров и т.п. **без пересборки** бинарника.
2. **Код sandbox** получает **`ResourceLimitPolicy`** (или эквивалент) **через DI** из уже загруженного **`Config`**, а не читает env напрямую и не полагается на пакетные глобалы (**`backend.mdc`**, **`review.mdc`**).
3. Текущие значения из **`DefaultResourceLimitPolicy()`** / **`resource_limits_defaults.go`** остаются **fallback по умолчанию**, если соответствующие env не заданы — **без изменения поведения** для существующих деплоев.

**Границы задачи:** **конфиг, загрузка и юнит-тесты пакета `config`** для секции sandbox (**обязательно** **`config_test.go`** — см. раздел «Тесты» и критерии приёмки). Монтирование **`docker.sock`** — **5.11**; **`Makefile sandbox-build`** — **5.12**; юнит-тесты **`DockerSandboxRunner`** на моке Docker-клиента — **5.13**; интеграционный тест реального контейнера — **5.14**. Логика **`effective*`** и **`HostConfig`** — **5.9** (задача **5.10** **подаёт числа**, а не дублирует маппинг в Docker).

## Поля `SandboxConfig` (минимальный контракт)

Имена полей и env — **ориентир** для PR; важен **смысл** и **однозначная валидация**.

| Смысл | Пример поля Go | Пример env | Заметки |
|--------|------------------|------------|---------|
| Политика памяти: пол в байтах | `MemoryFloorBytes` | `SANDBOX_MEMORY_FLOOR_BYTES` | Дефолт = текущий **`1 GiB`** из **`DefaultResourceLimitPolicy`** |
| Политика памяти: потолок в байтах | `MemoryCeilBytes` | `SANDBOX_MEMORY_CEIL_BYTES` | Дефолт = **`16 GiB`**; **`ceil ≥ floor`** |
| PID: пол / потолок | `PidsFloor`, `PidsCeil` | `SANDBOX_PIDS_FLOOR`, `SANDBOX_PIDS_CEIL` | Согласовать с **5.9** |
| CPU: дефолт при «не задано» в опциях | `DefaultNanoCPUs` | `SANDBOX_DEFAULT_NANO_CPUS` | Не ниже **`1_000_000_000`** без явного ADR (**5.9**) |
| CPU: верхний потолок | `NanoCPUsCeil` | `SANDBOX_NANO_CPUS_CEIL` | **`≥ DefaultNanoCPUs`** |
| Таймаут выполнения задачи по умолчанию | `DefaultTaskTimeout` | `SANDBOX_DEFAULT_TIMEOUT` | **`time.Duration`** через **`time.ParseDuration`** (как **`SERVER_READ_TIMEOUT`**) |
| Макс. параллельных sandbox на процесс | `MaxConcurrent` | `SANDBOX_MAX_CONCURRENT` | **≥ 1**; очередь воркеров может прийти позже — поле **резервирует** контракт для **Phase 2** / **6.x** |

Дополнительные поля (образ по умолчанию, сеть, registry pull policy) — **только** если уже есть потребитель в коде; иначе **не раздувать** задачу без отдельного пункта roadmap.

## Загрузка и валидация в `Load()`

- Парсинг **int64** для байт и **`NanoCPUs`** — через **`strconv.ParseInt`** с явной базой **10** и проверкой диапазона; избегать **`Atoi`** для значений, которые могут превысить **`MaxInt32`**.
- **Диапазоны:** отрицательные и нулевые там, где «ноль = дефолт» — **документировать** в комментарии к полю; противоречия (**`ceil < floor`**) — **`error`** из **`Load()`** с текстом **без** утечки секретов.
- **`DefaultTaskTimeout`:** пустой env → **строго положительный** дефолт (например **`30m`**, согласовать с **5.8** / **`SandboxOptions.Timeout`** в домене); невалидная строка **`time.ParseDuration`** → **`error`**. После парсинга обязательно **`DefaultTaskTimeout > 0`**: значения **`0`**, **отрицательные** (**`-1h`**, **`-1ns`**) и любая трактовка «ноль = бесконечность / без таймаута» **запрещены** без отдельного явного флага в конфиге и ADR — иначе возможны **мгновенное** срабатывание таймера, **зависание** ожиданий или неочевидная связка с **5.8**.
- **`MaxConcurrent`:** невалидное или **`< 1`** → **`error`** или clamp **только** если зафиксировано в PR; предпочтительно **явная ошибка** в prod-подобных окружениях.

## Краевые случаи, DoS и sanity checks (upper bounds)

**Проблема:** в env может прийти астрономическое значение (например **`SANDBOX_MEMORY_CEIL_BYTES=9999999999999999999`**) — переполнение при парсинге/приведении типов, либо «формально корректное» гигантское число, которое приводит к **OOM на хосте**, нестабильности Docker-демона или к заведомо бессмысленной нагрузке (**DoS через конфиг**).

**Требование:**

1. Для полей в **байтах** (**`Memory*Bytes`** и аналоги), **`NanoCPUs`**, **`PidsFloor` / `PidsCeil`**, **`MaxConcurrent`** задать **верхние разумные пределы** (sanity ceiling для одного API-процесса на типичном сервере). Конкретные константы — на усмотрение PR, но они **должны** отсекать «космические» значения **до** передачи в раннер/Docker.
2. Значение **выше** sanity-limit → **`error`** из **`Load()`** (**не** молчаливый clamp без зафиксированного ADR), с сообщением, указывающим **имя поля/env** (без секретов).
3. Согласовать верхние пределы с порядком величин из **5.9** (например потолок памяти **16 GiB** как ориентир), но **глобальный** sanity-cap может быть чуть выше «продуктового» ceil, если нужно разрешить staging; главное — **нет** безлимитного роста через env.

## Ошибки и контекст (wrap)

Все ошибки парсинга и валидации секции sandbox при пробросе из **`Load()`** (или из **`loadSandboxConfig`**) **оборачиваются** в единый понятный контекст, например: **`fmt.Errorf("invalid sandbox config: %w", err)`** (допустим префикс **`config: sandbox: ...`** — главное **единообразие** в PR). Внутренняя причина (**`time.ParseDuration`**, **`strconv.ErrSyntax`**, собственные **`errors.New`**) остаётся доступной через **`errors.Is` / `errors.As`**. Цель (**Compliance**, **`review.mdc`**): при падении приложения на старте в логе однозначно видно, что проблема в **секции Sandbox**, а не в безликом **`invalid duration`**.

## Тесты (Compliance)

Новый функционал загрузки **`SandboxConfig`** **не** откладывается до **5.13**/**5.14**: **5.13** — про раннер и мок Docker-клиента, **5.14** — про реальный контейнер. Здесь же, в рамках **5.10**, обязательны **юнит-тесты** в **`backend/internal/config/config_test.go`**: успешная **`Load()`** с пустым env (дефолты), успешный парсинг типичных env, а также **все значимые краевые случаи** — **`ceil < floor`**, отрицательные/нулевой **`DefaultTaskTimeout`**, невалидная duration-строка, **`MaxConcurrent < 1`**, значения **выше sanity upper bound**, при необходимости — переполнение/слишком длинная строка для **`ParseInt`**. Табличные тесты приветствуются.

## DI и границы пакетов

- Пакет **`config`** **не** импортирует **`internal/sandbox`** (циклы и нарушение слоёв). Конфиг — **«сырые» числа и duration**; сборка **`ResourceLimitPolicy`** — в **`cmd/api`** или в конструкторе **`NewDockerSandboxRunner(..., policy sandbox.ResourceLimitPolicy)`** из полей **`cfg.Sandbox`**.
- **`normalizeResourceLimitPolicy`** в **`sandbox`** может принимать **`ResourceLimitPolicy`** «сверху» и дополнять пропуски дефолтами — **без** чтения **`os.Getenv`** внутри **`sandbox`**.

## Критерии приёмки

- [x] В **`Config`** добавлено поле **`Sandbox SandboxConfig`** (или **`Sandbox`** с тем же смыслом), заполняется в **`Load()`**.
- [x] Все перечисленные параметры имеют **документированные env** и **безопасные дефолты**, совместимые с текущим **`DefaultResourceLimitPolicy()`** при пустом env.
- [x] **`Load()`** возвращает **`error`** при противоречивых или заведомо опасных значениях (например **`ceil < floor`**, **`MaxConcurrent < 1`**, значения **выше sanity upper bound**).
- [x] **`DefaultTaskTimeout`:** после разбора duration выполняется проверка **`> 0`**; **`0`**, отрицательные и невалидные строки — **`error`**.
- [x] Ошибки секции sandbox при возврате из **`Load()`** обёрнуты в **понятный контекст** (например **`invalid sandbox config: %w`**), чтобы в логе старта было видно **именно sandbox-конфиг**.
- [x] **Sanity checks:** верхние разумные пределы для байт/**`NanoCPUs`**/**`Pids*`**/**`MaxConcurrent`**; «космические» значения из env отклоняются **ошибкой**, а не доходят до Docker без ограничений.
- [x] Написаны **юнит-тесты** в **`backend/internal/config/config_test.go`**, проверяющие: успешную загрузку **дефолтов** при пустом env; корректный **парсинг env**; **ошибки** при **`ceil < floor`**, при **`DefaultTaskTimeout <= 0`** (в т.ч. **`0`**, **`-1h`**), при невалидной duration-строке, при **`MaxConcurrent < 1`**, при значениях **выше** установленных sanity-limits (и прочие согласованные в PR краевые случаи).
- [x] **`internal/sandbox`** не вызывает **`os.Getenv`** для этих параметров; политика передаётся **извне**.
- [x] В **`README`** (таблица Sprint 5) у строки **5.10** — ссылка «**детали**» на этот файл.
- [ ] При необходимости — краткое упоминание ключевых env в **`README`** или в **`deployment/docker-compose`** в рамках **5.11** (не обязательно в рамках только **5.10**, но дублирования имён env быть не должно).
