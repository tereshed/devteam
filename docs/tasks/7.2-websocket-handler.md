# 7.2 WebSocket Handler: upgrade HTTP → WS, аутентификация через JWT

**Sprint:** 7 — WebSocket и Реалтайм (Backend)
**Файл:** `backend/internal/ws/handler.go`
**Статус:** ⬜

---

## Описание

Реализовать **HTTP-обработчик `WebSocketHandler.Connect`**, который выполняет **HTTP → WebSocket upgrade** для эндпоинта `GET /api/v1/projects/:id/ws` (см. **7.7**), аутентифицирует пользователя по **JWT**, **проверяет права доступа** к запрошенному `project_id` и регистрирует соединение в **Hub** (**7.1**) с запуском read/write pump'ов.

Handler — **единственное место** в стеке, где принимается решение о допуске пользователя к стриму событий конкретного проекта. Hub намеренно не валидирует доступы (см. контракт **7.1**), поэтому **все проверки прав ОБЯЗАНЫ выполняться здесь**, до вызова `hub.Register`.

## Архитектура: Clean Architecture

| Слой | Ответственность |
|------|------------------|
| `handler` (`ws.WebSocketHandler`) | HTTP upgrade, извлечение `userID`/`projectID`, делегирование проверки доступа в `service`, регистрация в Hub |
| `service` (`ProjectService.HasAccess`) | RBAC/ABAC: пользователь → проект |
| `ws.Hub` (7.1) | Хранение подключений, broadcast, slow-client isolation — **без проверок доступа** |

`handler` **НЕ** обращается к репозиторию напрямую (нарушение `backend.mdc` §2.1). Любая проверка прав идёт через `service`.

## Функциональные требования

### 1. Аутентификация (JWT)

- **Источник токена** в порядке приоритета:
  1. **Заголовок** `Authorization: Bearer <token>` — основной путь, если клиент умеет ставить заголовки на upgrade-запросе (мобильный Flutter, серверные клиенты).
  2. **Sec-WebSocket-Protocol** sub-protocol вида `bearer.<jwt>` — fallback для браузеров (нативный `WebSocket` JS API не позволяет ставить произвольные заголовки). Сервер ОБЯЗАН ответить тем же sub-protocol в `Sec-WebSocket-Protocol` после успешного upgrade — иначе браузер сразу закроет соединение.
  3. **(ЗАПРЕЩЕНО)** query-параметр `?token=...` — токен попадает в access-логи и в `Referer`. **Не используем.**
- Валидация — через `AuthMiddleware` (см. требование к доработке ниже); при ошибке — `401 Unauthorized`, **без** upgrade.
- `userID` извлекается из контекста (`middleware.GetUserID`) **до** upgrade.

> **⚠️ Требование к доработке `AuthMiddleware` (обязательная часть этого тикета):** текущая реализация `backend/internal/middleware/auth_middleware.go` умеет только `X-API-Key` → `Authorization: Bearer`. На WS-роуте **этого недостаточно** — браузерные клиенты передают JWT через `Sec-WebSocket-Protocol: bearer.<jwt>` и будут молча получать `401`, не доходя до handler'а.
>
> **Принятое решение — Вариант А (доработка middleware, НЕ вынос auth'а в handler):**
>
> - Логика сохраняет единое место проверки JWT (Clean Architecture + `backend.mdc` §4).
> - Роут `GET /api/v1/projects/:id/ws` остаётся под `projects.Use(authMW)` без ручного обхода.
> - Handler продолжает читать `userID` через `middleware.GetUserID(c)`.
>
> **Изменения в `AuthMiddleware`:**
>
> 1. После неудачного пути `X-API-Key` и пустого `Authorization` — **дополнительный fallback**:
>    ```
>    for _, p := range websocket.Subprotocols(c.Request) {   // парсит Sec-WebSocket-Protocol
>        if token, ok := strings.CutPrefix(p, "bearer."); ok && token != "" {
>            authenticateWithJWT(c, jwtManager, middleware.BearerPrefix + token)
>            return
>        }
>    }
>    ```
>    Если среди заявленных sub-protocol'ов есть один с префиксом `bearer.` — берём его, отрезаем префикс, валидируем как обычный JWT.
> 2. **Источник правды — только верифицированный токен.** Никаких других claims/ID из sub-protocol'а не читаем.
> 3. **Приоритет источников остаётся прежним:** `X-API-Key` → `Authorization: Bearer` → `Sec-WebSocket-Protocol: bearer.*`. Последний срабатывает **только** если два предыдущих пусты. Это предотвращает downgrade-атаки (клиент не может «подсунуть» sub-protocol, если у него уже есть битый `Authorization`).
> 4. **Ограничить область fallback'а WS-роутами.** Варианты:
>    - простая проверка `c.Request.Header.Get("Upgrade") == "websocket"` **до** попытки прочитать sub-protocol (sub-protocol вне WS-контекста = бессмыслица, игнорируем);
>    - ИЛИ отдельная middleware-конструктор `AuthWebSocketMiddleware(...)` на основе `AuthMiddleware`, вешается только на WS-роут.
>    В MVP — **первый вариант**, он проще и не плодит middleware.
> 5. **Unit-тесты `auth_middleware_test.go`** расширяются: JWT через `Sec-WebSocket-Protocol` → `userID` в контексте; без `Upgrade: websocket` sub-protocol игнорируется; `Authorization` имеет приоритет над sub-protocol'ом.
>
> **Вариант Б (ручная валидация в handler'е, вынос WS-роута из-под `authMW`) отклонён:** дублирование логики парсинга/валидации JWT, размазывание auth'а, нарушение `backend.mdc` §4. Остаётся в документе как «не делаем и почему».

> **⚠️ DevOps/Infra — обязательное предупреждение:** эхо sub-protocol'а означает, что **JWT улетает в заголовке ответа** `Sec-WebSocket-Protocol: bearer.<jwt>`. Это **стандартный компромисс** протокола, но он требует явной гигиены на уровне инфраструктуры:
>
> - **NGINX / балансировщик / ingress ОБЯЗАНЫ НЕ логировать** заголовок `Sec-WebSocket-Protocol` (и заголовок `Authorization`) в access-логах. Для NGINX — не включать `$http_sec_websocket_protocol` / `$http_authorization` в `log_format`.
> - **APM/трейсеры** (Sentry, Datadog, OpenTelemetry) должны маскировать этот заголовок в http-спанах (обычно достаточно дефолтного sanitizer'а, но проверить).
> - **Прокси-кэши** не должны кэшировать ответы upgrade'а (они и так не кэшируются из-за `101`, но фиксируем явно).
>
> Это требование **попадает в чек-лист деплоя Sprint 7** и отдельно проверяется при первом выкате.

### 2. Авторизация (доступ к `project_id`)

**КРИТИЧНО — IDOR:** Hub (7.1) не знает о пользователях. Если handler пропустит запрос без проверки прав, **любой авторизованный пользователь подпишется на чужой проект** и будет получать чужие диффы, логи агента и pull-токены LLM.

- `projectID` парсится из `:id` через `uuid.Parse`. Невалидный UUID → `400 Bad Request`.
- Перед upgrade вызывается `projectService.HasAccess(ctx, userID, projectID)`:
  - `false` → `403 Forbidden` (без раскрытия "проект не существует" vs "нет доступа", чтобы не сливать enumeration).
  - Ошибка БД → `500 Internal Server Error` через `apierror.AbortJSON`.
- Кросс-проектные подписки в MVP **не поддерживаются**: один WS-коннект = один `projectID`.

### 3. Upgrade HTTP → WebSocket (Gorilla)

- Использовать `gorilla/websocket.Upgrader`. Параметры:
  - `ReadBufferSize: 1024`, `WriteBufferSize: 1024` (буферизация TCP-фреймов).
  - `EnableCompression: true` (per-message-deflate, экономит трафик на JSON).
  - **`Subprotocols` НЕ задаём** (см. следующий пункт — почему).
- **Эхо `Sec-WebSocket-Protocol` (КРИТИЧНО — Gorilla string-match):** `gorilla/websocket.Upgrader.Subprotocols` делает **строгое** сравнение строк. Если мы поставим `Subprotocols: []string{"bearer"}`, а клиент пришлёт `Sec-WebSocket-Protocol: bearer.eyJhbG...` — совпадения **не будет**, Gorilla **не подтвердит** протокол в ответе, и нативный браузерный `WebSocket` **сразу закроет соединение** (WebSocket API требует, чтобы сервер подтвердил один из заявленных sub-protocol'ов, иначе handshake считается провальным).
  - **Решение:** `Subprotocols` в Upgrader **оставить пустым**. Парсим sub-protocol'ы из запроса вручную, находим тот, что начинается с `bearer.`, и **явно пишем его** в `Sec-WebSocket-Protocol` через `responseHeader` при `Upgrade()`:
  ```go
  var negotiated string
  for _, p := range websocket.Subprotocols(c.Request) { // парсит Sec-WebSocket-Protocol request header
      if strings.HasPrefix(p, "bearer.") {
          negotiated = p
          break
      }
  }
  respHdr := http.Header{}
  if negotiated != "" {
      respHdr.Set("Sec-WebSocket-Protocol", negotiated)
  }
  conn, err := h.upgrader.Upgrade(c.Writer, c.Request, respHdr)
  ```
  - **Порядок выбора:** первый `bearer.*` из списка клиента. Дополнительную валидацию формата здесь **не делаем** — JWT уже провалидирован на слое `AuthMiddleware`. Нам от строки нужен только сам факт её наличия, чтобы эхнуть.
  - Если клиент НЕ прислал `bearer.*` (мобильный/CLI-клиент, аутентифицированный через `Authorization: Bearer`) — `Sec-WebSocket-Protocol` в ответ **не ставим** вовсе. Это корректно: если запрошенного sub-protocol'а не было, серверу и нечего эхо-подтверждать.
- **`CheckOrigin` (КРИТИЧНО — CSWSH):** **запрещено** возвращать `true` без проверки, НО **нельзя** жёстко реджектить пустой `Origin` — иначе легитимные мобильные/десктопные клиенты (Flutter iOS/Android, CLI, MCP-боты) не подключатся: они не ставят `Origin`, т.к. не являются браузерами и понятия «кросс-сайтовый запрос» для них не существует. Правильная семантика:
  ```
  if Origin пустой          → РАЗРЕШИТЬ (не-браузер, CSWSH неприменим)
  if Origin ∈ whitelist     → РАЗРЕШИТЬ
  если Origin заполнен,
     но НЕ в whitelist      → ЗАПРЕТИТЬ (403, защита от CSWSH в браузере)
  ```
  - Whitelist берётся из `config.AllowedOrigins` (env-переменная). Содержит **только** домены веб-фронта. Дополнительно применять нормализацию (scheme+host+port, lower-case) перед сравнением.
  - Cross-Site WebSocket Hijacking возможен исключительно в браузере, т.к. CORS к WS не применяется — единственная защита это `Origin`-check (для браузерных клиентов) + обязательная проверка JWT на каждом upgrade (для всех клиентов).
- **Ошибка `Upgrade()` (КРИТИЧНО — двойная запись ответа):** `gorilla/websocket.Upgrader.Upgrade()` **самостоятельно пишет HTTP-ответ** в `http.ResponseWriter`, если upgrade невозможен (клиент не поддерживает WS, битый `Sec-WebSocket-Key`, чужой `Origin` и т.п.). **После этого вызывать `apierror.AbortJSON` / `c.JSON` / `c.AbortWithStatus` ЗАПРЕЩЕНО** — получим `http: superfluous response.WriteHeader call` и потенциальный panic на recovery. Правильный паттерн:
  ```go
  conn, err := h.upgrader.Upgrade(c.Writer, c.Request, nil)
  if err != nil {
      // Upgrader уже написал ответ клиенту. Нам — только лог и return.
      h.log.Warnw("ws upgrade failed", "err", err, "remote", c.ClientIP())
      return
  }
  ```
  - Если нужен **кастомный JSON-формат** для ошибок upgrade — прокидывать через `Upgrader.Error func(w, r, status, reason)`. В MVP **не городим** — стандартный текстовый 400/403 от Gorilla достаточен, у фронта есть JWT/origin-валидация и он знает, что ответ может быть не-JSON на этом эндпоинте.
- Все pre-upgrade проверки (JWT, UUID, `HasAccess`, Origin-whitelist для не-пустого Origin, лимит коннектов) выполняются **до** вызова `Upgrade()` через `apierror.AbortJSON`. После `Upgrade()` — только логирование.

### 4. Регистрация в Hub и запуск pump'ов

- После успешного upgrade:
  ```go
  client := ws.NewClient(uuid.NewString(), userID.String(), conn, hub)
  if ok := hub.RegisterIfUnderLimit(client, []string{projectID.String()}, h.cfg.MaxConnsPerUserProject); !ok {
      // Лимит проверяется атомарно внутри Run(), поэтому false ≠ race — это честный отказ.
      // Upgrade() УЖЕ случился → стандартным WS-механизмом шлём close-фрейм с кодом 1008 и закрываем соединение.
      _ = conn.WriteControl(
          websocket.CloseMessage,
          websocket.FormatCloseMessage(websocket.ClosePolicyViolation, "connection limit exceeded"),
          time.Now().Add(time.Second),
      )
      _ = conn.Close()
      return
  }
  go client.WritePump()
  go client.ReadPump()
  ```
  **Важно:** лимит всё же проверяется **дважды** — предварительно через `CountUserConnections` **до** `Upgrade()` (чтобы отдать `429` без upgrade-оверхеда) и финально через `RegisterIfUnderLimit` **после** `Upgrade()` (защита от гонки между pre-check и регистрацией). Финальный отказ после upgrade'а — уже через WS close-фрейм, не через HTTP.
- `Client.ID` — **серверный UUID**, не из запроса. Никогда не доверять идентификатору с клиента — иначе spoofing.
- `Client.UserID` — из **верифицированных JWT-claims** (`middleware.GetUserID`), а не из тела/query запроса.
- Возврат из handler'а сразу после запуска горутин — Gin не должен держать соединение в active-pool.
- **Pump'ы НЕ получают `c.Request.Context()`** (см. §7).

### 5. Read/Write deadlines и Pong-handler (live-detection)

- **`SetReadDeadline` + Pong handler:** Без этого мёртвые TCP-соединения (NAT timeout, потеря Wi-Fi) висят часами и копят клиентов в Hub. Стандартный паттерн Gorilla:
  - `pongWait = 60s`, `pingPeriod = 54s` (≈90% от pongWait).
  - `WritePump` каждые `pingPeriod` шлёт `websocket.PingMessage`.
  - `ReadPump` ставит `SetReadDeadline(now + pongWait)` и обновляет его в `SetPongHandler`.
- При срабатывании deadline `ReadPump.ReadMessage()` вернёт ошибку → `defer Unregister()` (см. **7.1**) → клиент чистится из Hub.
- **Реализация ping/pong и deadlines** живёт в `client.go` (расширение **7.1**), handler только запускает pump'ы.

### 6. Лимит одновременных подключений (DoS protection)

- Лимит на `(userID, projectID)` — например, **5 одновременных WS** (значение из `config`, дефолт 5).
- При превышении — `429 Too Many Requests` до upgrade.
- **Источник истины — ТОЛЬКО Hub.** Вводится новый метод (расширение **7.1**):
  ```go
  // CountUserConnections — число активных клиентов пары (userID, projectID).
  // Actor Model: обрабатывается в Run() через канал-запрос + resp-канал.
  func (h *Hub) CountUserConnections(userID, projectID string) int
  ```
  Hub — **единственный**, кто знает о реальных `Register`/`Unregister`, включая асинхронные отключения по pong-deadline из `ReadPump`. Любой «лёгкий счётчик в handler'е» рано или поздно рассинхронизируется с реальностью (handler не узнаёт об отключении клиента по таймауту) и навсегда вычерпает лимит пользователя. **Это явный запрет.**
- Чтобы Hub узнавал о пользователе, в `Client` (7.1) добавляется поле `UserID string`, которое заполняется **handler'ом** при `NewClient` и в Hub попадает через `RegisterMessage` (Actor Model: данные летят вместе с событием).
- **Race (Register после Count):** между `CountUserConnections` и `hub.Register` теоретически может проскочить 6-й коннект. Для MVP это допустимо (лимит ~strict), но корректнее — реализовать в Hub атомарный `RegisterIfUnderLimit(client, projectIDs, max int)`, возвращающий `(ok bool)` внутри одного тика `Run()`. **Рекомендуется второй вариант**; в критериях приёмки оформлен как обязательный.

### 7. Контекст и graceful shutdown (КРИТИЧНО — lifetime)

- `c.Request.Context()` (= `ctx` Gin-запроса) жив **только** пока handler исполняется. **Сразу** после `return` из handler'а Gin отменяет этот контекст.
- Правила использования `c.Request.Context()`:
  - ✅ **ДО** вызова `h.upgrader.Upgrade(...)` — валидно. Используется для `projectService.HasAccess(ctx, ...)` и других синхронных вызовов в `service` (см. `backend.mdc` §2.1.4).
  - ❌ **ПОСЛЕ** `Upgrade()` (в `NewClient`, `hub.Register`, `go WritePump()`, `go ReadPump()` и во всех фоновых горутинах) — **ЗАПРЕЩЕНО**. Этот контекст отменится через миллисекунды, как только handler вернёт управление, и любой код, завязанный на него, упадёт с `context canceled`.
- **Что использовать в pump'ах вместо запросного контекста:**
  - `Client` и его `ReadPump`/`WritePump` **не принимают** `context.Context` в MVP — lifecycle определяется событиями «закрытие `Send`-канала Hub'ом» и «ошибка чтения из TCP-соединения».
  - Если в будущем потребуется остановка pump'а по внешнему сигналу (например, принудительный kick пользователя) — использовать **отдельный `context.Context`, переданный в `Hub` при `NewHub`/`Run`** (это и есть rootCtx приложения из 7.7), **но не запросный ctx**.
- При остановке приложения Hub получает отмену rootCtx (см. **7.1** `shutdown` и **7.7**) — handler НЕ обязан дополнительно ничего делать: pump'ы умрут на закрытом `client.Send` / закрытом `Conn`.
- **Анти-паттерн, который категорически ловим на ревью:**
  ```go
  // ❌ ЗАПРЕЩЕНО — запросный ctx проживёт 1 мс после return
  client := ws.NewClient(id, conn, hub, c.Request.Context())
  go client.WritePumpCtx(c.Request.Context())
  ```

## Структура

```go
// backend/internal/ws/handler.go
package ws

import (
    "net/http"

    "github.com/devteam/backend/internal/middleware"
    "github.com/devteam/backend/internal/service"
    "github.com/devteam/backend/pkg/apierror"
    "github.com/gin-gonic/gin"
    "github.com/google/uuid"
    "github.com/gorilla/websocket"
)

// WebSocketHandler — HTTP-обработчик upgrade в WS + регистрация в Hub.
// ВСЯ проверка прав доступа к project_id происходит здесь (Hub их не валидирует).
type WebSocketHandler struct {
    hub            *Hub
    projectService service.ProjectService
    upgrader       websocket.Upgrader
    cfg            HandlerConfig
}

// HandlerConfig — настройки handler'а (allowed origins, лимиты, таймауты).
type HandlerConfig struct {
    AllowedOrigins        []string
    MaxConnsPerUserProject int
    ReadBufferSize        int
    WriteBufferSize       int
```

## Дизайн-решения

1. **JWT через заголовок или sub-protocol, НЕ через query.** Query-параметры утекают в логи NGINX/access.log и в `Referer` соседних страниц. Sub-protocol `bearer.<jwt>` — стандартный обходной путь для браузеров. Плата — JWT улетает в заголовке ответа `Sec-WebSocket-Protocol` (эхо-требование протокола), поэтому DevOps обязан исключить этот заголовок и `Authorization` из access-логов и APM-спанов (см. §1 «DevOps/Infra»).
2. **Origin-check обязателен, но с исключением для пустого Origin.** Без проверки возможен CSWSH: вредоносный сайт открывает `new WebSocket(...)` к нашему серверу, браузер шлёт куки/JWT (если в куки), сервер апгрейдит — атакующий читает поток. CORS на WS не работает. Но мобильные и CLI-клиенты **не ставят** `Origin` — жёсткий реджект пустого Origin сломает весь Flutter-фронт. Поэтому `empty Origin → allow` (для них CSWSH неприменим), `non-empty Origin → whitelist-check`.
3. **Проверка доступа в handler, НЕ в Hub.** Hub — низкоуровневый менеджер, не знает о доменной модели. Размазывание проверок прав по слоям — путь к дырам. Контракт зафиксирован в **7.1**.
4. **Один коннект = один проект.** Упрощает счётчик лимитов, IDOR-логику и unregister. Подписка на несколько проектов — отдельный коннект.
5. **Серверный `client.ID`.** Никогда не доверять ID от клиента — иначе spoofing и подмена unicast-адресата. `Client.UserID` также заполняется **сервером** из верифицированных JWT-claims, а не из тела запроса.
6. **Ping/pong и deadlines обязательны.** Без них Hub за неделю наберёт тысячи "зомби"-клиентов и упадёт по памяти/файловым дескрипторам.
7. **Лимит коннектов — только в Hub.** Hub — единственный, кто видит полный lifecycle соединения, включая асинхронные отключения по pong-deadline из `ReadPump`. Любой внешний счётчик (atomic map в handler'е, Redis-счётчик, middleware-счётчик) рано или поздно рассинхронизируется: handler декрементирует при upgrade-ошибке, но не узнает о отключении клиента через 2 часа по таймауту, и пользователь навсегда «съест» свои 5 слотов. Атомарный `RegisterIfUnderLimit` внутри `Run()` (Actor Model) — единственно корректное решение.
8. **Запросный ctx — только до `Upgrade()`.** `c.Request.Context()` отменяется Gin-ом сразу после `return` handler'а. Если протащить его в `Client`/`ReadPump`/`WritePump` — все фоновые горутины упадут через ~1 мс с `context canceled`. Это классическая мина замедленного действия в Gin+Gorilla коде. Lifecycle pump'ов управляется событиями `Send`-канала (закрыт Hub'ом при `removeClient` или `shutdown`) и ошибкой `Conn.ReadMessage()`, а не контекстами.
9. **Ошибка `Upgrade()` — только log+return.** `gorilla/websocket.Upgrader` сам пишет HTTP-ответ при ошибке (`400 Bad Request` без заголовка Upgrade, `403` если `CheckOrigin` вернул false, и т.д.). Повторная запись через `apierror.AbortJSON` = `http: superfluous response.WriteHeader call` + потенциальный panic в Gin-recovery middleware. Нестандартный JSON-формат для этих ошибок в MVP не нужен — фронт знает, что upgrade-эндпоинт может вернуть plain-text на failure.
10. **Sub-protocol эхо — динамическое, `Upgrader.Subprotocols` пустой.** `gorilla/websocket` делает **строгое** string-match поля `Subprotocols`. Клиентский `bearer.<jwt>` никогда не совпадёт с серверным `"bearer"` → Gorilla не подтвердит протокол → нативный браузерный `WebSocket` обрывает соединение. Поэтому: `Subprotocols` в `Upgrader` **не задаём вовсе**, а нужный `bearer.*` из клиентского запроса читаем вручную через `websocket.Subprotocols(c.Request)` и прокидываем в `responseHeader` третьим аргументом `Upgrade()`. Без `bearer.*` в запросе — в ответ sub-protocol не ставим (эхать нечего).
11. **Auth через доработку `AuthMiddleware`, а не через вынос auth'а в handler.** Стандартный middleware из `backend/internal/middleware/auth_middleware.go` умеет `X-API-Key` и `Authorization: Bearer`, но **не парсит** `Sec-WebSocket-Protocol: bearer.<jwt>` — браузерный клиент получит `401` до handler'а. Решение (Вариант А): расширить `AuthMiddleware` новым fallback-путём, который срабатывает **только** при `Upgrade: websocket` и **только** если `Authorization`/`X-API-Key` пусты. Так сохраняется единая точка проверки JWT (`backend.mdc` §4) и чистый роутинг. Вариант Б (ручная валидация в handler'е) явно отклонён.

## Обработка ошибок

| Сценарий | Код HTTP | Ошибка |
|----------|----------|--------|
| Нет JWT и нет `bearer.*` sub-protocol | 401 | `apierror.ErrTokenRequired` |
| JWT невалиден / истёк | 401 | `apierror.ErrInvalidToken` / `ErrTokenExpired` |
| `:id` — не UUID | 400 | `apierror.ErrInvalidRequest` |
| `HasAccess() == false` ИЛИ `service.ErrProjectNotFound` | 403 | `apierror.ErrForbidden` (одинаковое сообщение, см. п. 2) |
| Origin не в whitelist | 403 | `apierror.ErrForbidden` |
| Лимит коннектов превышен | 429 | `apierror.ErrTooManyRequests` |
| Ошибка `Upgrade()` | 400/403 (пишет Gorilla) | **Только** логируем + `return`. `apierror.AbortJSON` / `c.JSON` / `c.AbortWithStatus` после `Upgrade()` **ЗАПРЕЩЕНЫ** — Gorilla уже записал ответ, повторная запись → `http: superfluous response.WriteHeader call`. |
| Ошибка `service.HasAccess` (БД) | 500 | `apierror.ErrInternal` |

**Важно:**
- **До** успешного `Upgrade()` ответы — обычные JSON через `apierror.AbortJSON`.
- **После** успешного `Upgrade()` — общение только через WS-фреймы; Gin-ответ слать **ЗАПРЕЩЕНО** (panic: connection hijacked).
- **При ошибке `Upgrade()`** — Gorilla сама уже ответила клиенту; от handler'а только `log + return`.

## Критерии приёмки

- [ ] Создан `backend/internal/ws/handler.go` с `WebSocketHandler` и методом `Connect(c *gin.Context)`.
- [ ] Реализована **аутентификация JWT** через `Authorization: Bearer` **или** `Sec-WebSocket-Protocol: bearer.<jwt>`. Query-параметр `?token=` **не поддерживается**.
- [ ] **`AuthMiddleware` доработан (Вариант А):** в `backend/internal/middleware/auth_middleware.go` добавлен fallback-путь, который при `Upgrade: websocket` и пустых `Authorization`/`X-API-Key` парсит `Sec-WebSocket-Protocol` через `websocket.Subprotocols(r)`, берёт первый элемент с префиксом `bearer.`, отрезает префикс и валидирует как обычный JWT. Приоритет: `X-API-Key` → `Authorization: Bearer` → `Sec-WebSocket-Protocol: bearer.*`. Вне WS-контекста sub-protocol игнорируется. Роут `GET /api/v1/projects/:id/ws` остаётся под `authMW` без ручной валидации в handler'е.
- [ ] **Динамическое эхо sub-protocol'а (КРИТИЧНО):** `Upgrader.Subprotocols` **НЕ задан** (Gorilla делает строгий string-match и `"bearer"` != `"bearer.eyJ..."`). Handler читает `websocket.Subprotocols(c.Request)`, находит первый элемент с префиксом `bearer.` и явно пишет его в `Sec-WebSocket-Protocol` через `responseHeader` в третьем аргументе `Upgrade(w, r, respHdr)`. Если `bearer.*` нет — sub-protocol в ответ не ставим. Unit-тест проверяет, что браузерный клиент, прислав `bearer.<jwt>`, **получает** тот же sub-protocol в ответе (иначе `WebSocket` API закрыл бы соединение).
- [ ] Реализована **авторизация** через `projectService.HasAccess(ctx, userID, projectID)`; запрет доступа → `403`.
- [ ] **IDOR:** документировано и протестировано, что пользователь без доступа к проекту получает `403` ещё до upgrade и не видит ни одного события.
- [ ] **CSWSH (корректная семантика):** `Upgrader.CheckOrigin` возвращает `true` для **пустого** `Origin` (не-браузерные клиенты — Flutter iOS/Android, CLI, MCP-боты). Для **не-пустого** `Origin` — проверка по whitelist из конфига; чужой origin → `403`. Явный unit-тест на каждый из трёх случаев.
- [ ] **Sec-WebSocket-Protocol leakage:** в коде и в документации задеплоя зафиксировано, что NGINX/ingress/APM/трейсеры **не должны** логировать заголовки `Sec-WebSocket-Protocol` и `Authorization`. Пункт включён в чек-лист деплоя Sprint 7.
- [ ] **Ошибка `Upgrade()`:** handler делает **только** `log + return`. Любые `apierror.AbortJSON` / `c.JSON` / `c.AbortWithStatus` после вызова `Upgrade()` **запрещены** и явно помечены в коде комментарием. Unit-тест проверяет отсутствие `superfluous response.WriteHeader call` в логах.

- [ ] `hub.Register(client, []string{projectID})` вызывается **после** успешной проверки прав.

- [ ] **Ping/pong + deadlines:** `pongWait = 60s`, `pingPeriod = 54s`; pong-handler обновляет `SetReadDeadline`. Реализация лежит в `client.go` (расширение **7.1**).
