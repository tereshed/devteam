## Описание

Presentation-слой: **два** class-based **`@riverpod`**-контроллера (codegen, как **11.4** / **`CreateProjectController`**), которые инкапсулируют загрузку, пагинацию, обновление и мутации задач **только** через **`ref.read` / `ref.watch`(`taskRepositoryProvider`)**. **Запрещено:** собирать query списка вручную на экранах — только **`TaskListFilter`** + **`limit`/`offset`** из **12.2**.

## Минимальный контракт state (**обязательно**)

Не «опционально Freezed, если не влезает в **`AsyncValue`**»: именованный state, как **`ChatState`** в **11.4**.

| Поле | Тип / смысл |
|------|----------------|

**`AsyncNotifier`:** наружу **`AsyncValue<TaskListState>`** (или эквивалент), внутри — не «магические» поля только в **`Notifier`** без **`TaskListState`**.

**Шаблон Riverpod (**§ 56**):** как **11.4** — **`@Riverpod class … extends _$…`**, **`FutureOr<TaskListState> build()`** / **`FutureOr<TaskDetailState> build()`** с **синхронным** **`return` initial state** без **`await`**; ошибки первичной загрузки в **`_loadInitial`** — **`state = AsyncError(...)`** / **`AsyncValue.guard`**, чтобы матрицы (**`AsyncError`**, **`TaskNotFoundException`**) применимы. **`Notifier<TaskListState>`** без **`AsyncValue`** — **не** канон **12.3**.

## `TaskListItemModel` и `TaskModel` — разные типы (**канон**)

- **`TaskListItemModel`** — ответ **`GET …/projects/:id/tasks`** (подмножество полей).  
- **`TaskModel`** — **`GET /tasks/:id`**, **`pauseTask`**, **`updateTask`**, … (в т.ч. **`description`**, **`context`**, **`subTasks`**, **`messageCount`**, …).

**Запрещено** писать в мутациях «upsert **`TaskModel`** в **`items`**» без явного шага.

### `taskModelToListItem(TaskModel) → TaskListItemModel`

**Обязательна** top-level или статическая функция **`TaskListItemModel taskModelToListItem(TaskModel m)`** (имя на усмотрение PR), которая:

1. Копирует **только** поля, общие для обоих типов (**id**, **`projectId`**, **`parentTaskId`**, **`title`**, **`status`**, **`priority`**, **`assignedAgent`**, **`createdByType`**, **`createdById`**, **`branchName`**, **`startedAt`**, **`completedAt`**, **`createdAt`**, **`updatedAt`** — сверить с **`task_list_item_model.dart`** / **`task_model.dart`** **12.1**).  
2. **Не** заполняет «из воздуха» поля, которых нет в **`TaskModel`** на wire.  
3. В **`debugMode`** (или тестах) — **`assert`** / сравнение, что ни одно **required** поле **`TaskListItemModel`** не остаётся с factory-дефолтом, если в **`TaskModel`** значение было не **`null`**, но не попало в mapping (**регрессия DTO**).

**Альтернатива «без downcast»:** после HTTP-мутации **не** трогать **`items`**, а **`ref.invalidate(taskListControllerProvider(projectId))`** — тогда **`taskModelToListItem`** для этого пути **не** вызывается; **смешивать** в одном PR «invalidate для create, downcast для pause» без мотивации — **нельзя**.

## Пагинация списка задач (без `has_next`)

В **12.1**: у **`TaskListResponse`** нет **`has_next`**, только **`tasks`**, **`total`**, **`limit`**, **`offset`**.

**`hasMore`:**

```text
hasMore = offset + tasks.length < total
```

**Следующий `offset`:** **`offset + tasks.length`** после merge (не вслепую **`+ limit`**).

**`limit`:** **`kTaskListDefaultLimit`** (**50**) из **`task_repository.dart`**; верхняя граница **200** — как **12.2**.

**Анти-цикл (как **`_hasMoreAfterPage`** в **11.4**):** snapshot **`id`** в **`items`** до merge → если **0 новых** **`id`** в ответе при «теоретическом» **`hasMore`** — **`hasMore := false`**. Частный случай **`tasks.isEmpty`** при **`offset < total`** — тоже сброс.

**Дедуп:** **last-write-wins** по **`id`**; пересортировка по **`order_by`** из **`TaskListFilter`**.

## Пагинация сообщений задачи (**семантика ≠ чат**)

**Бэкенд:** **`created_at ASC`**. **UI задач:** лента **сверху вниз** по времени (**без** инверсии как в чате). Значит:

- **Не** использовать имя **`loadOlder`** / **`olderOffset`** из чата как копипасту.  
- **`loadMoreMessages`** — подгрузка **вперёд по времени** (не **`loadOlder`** чата).  
- **Следующий `messagesOffset` (канон, § 53):** **`messagesOffset += response.messages.length`** (как **`olderOffset += page.messages.length`** в **11.4**), **не** **`messages.length`** после merge: после **`_mergeTaskMessages`** / WS длина списка может разойтись с серверной страницей → **пропуск строк** при **`offset = messages.length`**. **Запрещено** оставлять «два эквивалентных варианта в PR».

Анти-цикл по snapshot **`message.id`** — как для списка задач.

## `CancelToken`, эпоха и `refresh`

**Почему одного `CancelToken` мало:** **`cancel()`** не отменяет уже завершённый **`Future`**; ответ может смержиться после **`refresh()`** — см. **`_sessionEpoch`** в **`chat_controller.dart`** ~360–424.

**Канон:** **`int _sessionEpoch`** (приватно); перед **`refresh` / смена фильтра`** — инкремент; каждый ответ помечен **`epochAtStart`**; перед merge — если **`epochAtStart != _sessionEpoch`** → **discard**. Плюс **`token.isCancelled`** в continuation. **Минимум одно** из двух для **`loadMore`** списка и деталей.

**`CancelToken?`:** в **`build`** — новый токен + **`ref.onDispose`**, при повторном **`build`** — отмена предыдущего; в **`refresh()`** — **`cancel`**, новый токен, эпоха. **Не** **`late final CancelToken`** при **`refresh`**.

**`Future<void>? _loadMoreInFlight`** для **`loadMore`** (список и сообщения) — как **`_olderInFlight`** ~443–474 в **`chat_controller.dart`** (**`whenComplete`**, повторный синхронный вызов не плодит второй HTTP).

**`TaskDetailController` — токены (**§ 83**):** ровно **два** — например **`_taskHistoryToken`** (**`getTask`**, перезагрузки «ветки задачи» после **`refresh`**) и **`_messagesHistoryToken`** (**`listTaskMessages`**, **`loadMoreMessages`**, смена фильтров сообщений). **Мутации** (**`pauseTask` / `resumeTask` / …` / `correctTask` / `sendTaskMessage` / `addTaskMessage`**) — **без** привязки к этим токенам (**как** **`ChatController.sendMessage`**, **`cancelToken: null`** ~570): иначе **`refresh()`** отменит inflight **`correctTask`** и даст гонку. **Не** плодить **5+** токенов на каждый метод — только разделение «история задачи» vs «история сообщений» + мутации вне cancel.

**`TaskListController` — мутации (**§ E**):** **`createTask`** и любые **другие HTTP-мутации на списке** (если появятся) — **также без** cancel-токена «истории списка» (**`listTasks` / `loadMore`** остаются на **`CancelToken`** списка): **`refresh()`** / смена фильтра **не** должны **отменять** inflight **`createTask`** — симметрия с **§ 83**.

## Мутации (HTTP)

1. Репозиторий (**12.2**).  
2. После успеха: **`TaskModel`** в **`TaskDetailController`**; для списка — **публичный** метод на нотифайере **`TaskListController`** (имя на усмотрение PR, напр. **`syncListFromHttpTask(TaskModel m)`** или **`applyTaskFromHttpMutation`**), **обязательно не приватный** (**§ 77**): его вызывает **`ref.read(taskListControllerProvider(…).notifier)`** из **другого** файла (**кросс-контроллер**, стр. кросс-шов ниже); **`_sync…` в том же library недоступен извне**. Внутри — **`taskModelToListItem(m)`** + § **«Мутации и `TaskListFilter`»**, **или** **`invalidate`** — по политике § **`TaskModel` → `TaskListItemModel`**.  
3. **`TaskCancelledException`** — не фатальная «сеть» при отмене.  
4. **`ArgumentError` до сети** (**12.2**): **`correctTask`** — пустой **`trim`**, **> 256 KiB** UTF-8; **`updateTask`** — **`assignedAgentId != null && clearAssignedAgent`**. Контроллер + **12.5/12.8** не должны слать такие тела; при провале — **не** маскировать как **`TaskApiException`**.

**Слияние сообщений (HTTP + WS):** одна функция **`_mergeTaskMessages`** — **upsert по `id`**, сортировка **`created_at ASC`**, **`id ASC`** (как **`_mergeMessagesCanonical`** в **`chat_controller.dart`** ~82–102, ~228–236).

### Блокировка мутаций при **`realtimeMutationBlocked` (**§ 66**)**

Как **`ChatController.send`** (**`chat_controller.dart`** ~546–558): при **`realtimeSessionFailure != null`** возвращается **`ChatSendOutcome.blockedByRealtimeSession`** — **HTTP не вызывается**, точка истины в контроллере.

**Канон 12.3:** для **`pauseTask` / `resumeTask` / `cancelTask` / `correctTask` / `updateTask` / `deleteTask` / `sendTaskMessage`** (и **`createTask`** на списке) — **порядок проверок (**§ 82**):** сначала **локальная валидация** (п. **4** выше: пустой **`trim`**, лимиты UTF-8, взаимоисключающие поля **`updateTask`**, …) → при провале **`TaskMutationOutcome.validationFailed`**, **без** HTTP и **без** проверки realtime; затем, если валидно — **`state.realtimeMutationBlocked == true`** → **`blockedByRealtime`**, **без** **`repo.*`**; иначе — HTTP → **`completed`** или исключение/**`AsyncError`**. **Сигнатуры (**§ 84**):** **все** мутации возвращают **`Future<TaskMutationOutcome>`** (или **`Future<void>`** + outcome через callback — **не** смешивать sync/async в одной фиче): даже если «ранний выход» только валидация, UI получает один awaitable контракт. **12.5**/**12.8** по-прежнему могут дизейблить кнопки по флагу для UX, **но** забытая проверка в UI **не** приводит к «тихому» HTTP.

### `addTaskMessage` / `sendTaskMessage`

**Идемпотентность (**§ 45**):** в **`TaskRepository.addTaskMessage`** **нет** **`clientMessageId`** / **`Idempotency-Key`** — **двойной тап «Отправить» = два сообщения** на сервере. Варианты, **один** должен быть явно выбран в PR и в **README** фичи:

- **(а)** только UI: debounce / блок кнопки до ответа (**12.5**/**12.8**);  
- **(б)** расширение **12.2** / бэкенда под заголовок идемпотентности — **отдельная** задача, не «молчаливый PR».

### `createTask` — optimistic и **`total`** (**§ 47–48 / § 57**)

**Рекомендация по умолчанию для 12.3:** **без** optimistic prepend — после **201** только **`refresh()`** первой страницы **или** **`invalidate`**; гонки с **`total`**, откат при **409/422** и зависимость от **`currentUserId`** остаются для **12.4 / 12.8** как отдельная мера UX, если понадобится.

Если в PR всё же выбран **optimistic prepend** **`TaskListItemModel`**:

- **`total`** с сервера **устаревает**; **канон UI:** либо **`items.length`** как локальный счётчик, либо **`total + 1`** до **`refresh`** — **один** вариант + явный откат при ошибке.  
- **`TaskListItemModel`** требует **`createdByType` / `createdById`**: **`currentUserId`** из auth (**`ref.watch(...)`**, § **«Импорты»**). Без auth — optimistic **запрещён**.

После **`pauseTask` / …`:** **`ref.read(taskListControllerProvider(projectId: …).notifier).syncListFromHttpTask(model)`** (или **`invalidate`** — по политике).

## Мутации и `TaskListFilter` / `deleteTask`

После мутации поля могут не проходить **`TaskListFilter`**.

**`TaskListFilter.search` и серверные фильтры (**§ 50**):** **`search`** — серверный полнотекст (**12.2** / бэкенд), **локально** не воспроизвести (`title.contains` ≠ **`tsvector`**). **Канон:** если **`filter.search != null && filter.search!.trim().isNotEmpty`** (или эквивалент «поиск активен») — после **любой** HTTP-мутации, влияющей на список (**`syncListFromHttpTask`**, **`createTask`**, …), **всегда** **`ref.invalidate(taskListControllerProvider(projectId))`** — **никогда** локальный upsert/insert по **`taskMatchesCurrentFilter`** без поиска. То же **обязательно**, если активен фильтр, для которого нужен **серверный** контекст дерева (**`parent_task_id`** / **`root_only`** и т.д. — по списку полей **12.2**), **если** команда не может доказать локальную эквивалентность — **консервативно `invalidate`**.

**`syncListFromHttpTask` и строка отсутствует в `items` (**§ 51**):** если **`id`** задачи **не** было в **`items`**, и **`total > items.length`** (на сервере есть **невыгруженный хвост**) — **не вставлять** новую строку из мутации (ломает порядок и **`offset`**); **дефолт:** **`invalidate`** (консервативно подтянуть **`total`/порядок**); **`ничего`** допустимо **только** с мотивацией в PR («мутация не меняла membership в списке»). Вставка разрешена **только** если **`total == items.length`** (все строки уже в памяти) **и** нет активного **`search`** / спорных серверных фильтров.

**Канон (а)** (когда **`search`** пуст и локальные фильтры однозначны): **`taskMatchesCurrentFilter`** по полям, совпадающим с **12.2**; не проходит — **удалить**; не было и не проходит — **не вставлять**; проходит — **upsert** + правила выше.

**Канон (б):** **`invalidate`** — trade-off scroll.

**`deleteTask`:** **204** → деталь «удалено» (**`AsyncError`** / флаг — **12.5**); в списке — удалить **`id == taskId`** или **`invalidate`** если инстанса списка нет.

**Свежесть HTTP `getTask` vs поздний WS-патч (полный `TaskModel`):** если после **`getTask`** пришёл **`TaskModel`** с **`updatedAt`**, а в state уже новее (из WS) — сравнить **`updatedAt`**; устаревший HTTP **не** затирает (**итерация 2**).

## Шов для **12.9** (WS → контроллер)

### Почему **не** `applyTaskPatch(TaskModel)` от **`WsTaskStatusEvent`**

**`WsTaskStatusEvent`** (**`websocket_events.dart`** ~150–162) несёт **`previousStatus`**, **`status`**, **`assignedAgentId`**, **`agentRole`**, **`errorMessage`**, **`ts`**, **`v`**, **`projectId`**, **`taskId`**, **`parentTaskId`** — **нет** **`title`**, **`description`**, **`updatedAt`** из **`TaskModel`**, нельзя собрать ни **`TaskModel`**, ни **`TaskListItemModel`** целиком (**§ 35**).

**Канон — вариант (а):** публичный **`applyWsTaskStatus(WsTaskStatusEvent e)`** на **`TaskListController`** и **`TaskDetailController`** (или один общий mixin — на усмотрение PR): **patch поверх текущего state**, без выдуманных полей.

**Вариант (б):** **`applyWsTaskStatus`** только триггерит **`getTask(e.taskId)`** и заменяет **`task`** — допустим при простоте; **trade-off:** лишний RTT. **Запрещено** **12.9** собирать **`TaskModel`** вручную из WS.

### FSM: **`previousStatus`** (**§ 36**)

Перед применением патча статуса:

```text
if state.task == null → return (silent)
if e.projectId != family.projectId → return
if e.taskId не в множестве «свои id» (см. § «Подзадачи») → return

if e.taskId == family.taskId:
  if state.task!.status != e.previousStatus
     → mismatch родителя: drop patch; guarded **`getTask(family.taskId)`** (**§ 64**)
  else применить к state.task поля из e: status, errorMessage,
       assignedAgent (см. § «Агент»); **не** писать **`task.updatedAt := e.ts`** (см. § **«HTTP vs WS: свежесть `updatedAt`»** ниже)
else:
  // подзадача: найти i, где subTasks[i].id == e.taskId (membership уже проверен выше)
  if subTasks[i].status != e.previousStatus
     → mismatch подзадачи (**§ 75**): drop patch; **дефолт (а)** guarded **`getTask(family.taskId)`**
       и заменить **`state.task` целиком** (с сервера придут актуальные **`subTasks`**);
       **не (б)** отдельный **`getTask(subTask.id)`** + маппер **`TaskModel` → `TaskSummaryModel`**
       без мотивации в PR
  else в **`subTasks[i]`** применить **только `e.status`** (**§ 76**); **`errorMessage`**, **`assignedAgentId`**, **`agentRole`** из события **не** маппить в **`TaskSummaryModel`** (полей нет) — **игнор без assert**; полный контекст агента/ошибки — только открыв подзадачу отдельным **`TaskDetailController`**
```

**`TaskListController` (**§ 65 / § 68**):** для строки **`items`**, где **`item.id == e.taskId`**, тот же тест **`item.status == e.previousStatus`**. При mismatch — **дефолт:** точечный **`getTask(e.taskId)`** + **`taskModelToListItem`** + upsert **одной** строки (**не** полный **`invalidate`** списка — теряется scroll Kanban); refetch под **тем же anti-storm guard**, что **§ 64** (например **`_wsRefetchInFlight`** + **`whenComplete`**, или debounce **~250 ms** на **`taskId`** — **один** механизм в PR). Если **`e.taskId`** **ни в одной** строке **`items`** (задача за пределами загруженных страниц при **`total > items.length`**) — **silent ignore** (строка появится/обновится при **`loadMore` / `refresh`**).

### Подзадачи (**§ 40 / § 75 / § 76**)

**`state.task.subTasks`** содержит **`TaskSummaryModel`** (**12.1** — **`id`**, **`title`**, **`status`**, **`priority`**). WS для подзадачи: **`e.taskId == subTask.id`**, не равный **`family.taskId`**. **Канон membership:** множество «своих» **`taskId`**: **`{ family.taskId } ∪ { sub.id for sub in state.task?.subTasks ?? [] }`**.

**FSM `previousStatus` для подзадачи** — симметрично родителю и строке **`items`**: см. ветку **`else`** в псевдокоде выше (**§ 75**): при совпадении **`subTasks[i].status == e.previousStatus`** — **только** **`subTasks[i] = subTasks[i].copyWith(status: e.status)`** (остальные поля **`TaskSummaryModel`** из WS **не** обновляются). **`errorMessage` / `assignedAgentId` / `agentRole`** в **`WsTaskStatusEvent`** для подзадачи в **`subTasks`** **не** применяются (**нет полей**); **не** падать и **не** прокидывать в **`metadata`** — при необходимости UI открывает подзадачу как отдельную деталь.

### Агент: **`assignedAgentId` + `agentRole`** без **`name`** (**§ 37**)

**Запрещено:** **`AgentSummaryModel(name: '', …)`** «для галочки».

**Канон (**§ 65, дефолт**):** если **`e.assignedAgentId`** отличается от **`state.task.assignedAgent?.id`** — **`unawaited(getTask(taskId))`** в **`TaskDetailController`** после патча (восстановить **`name`**). Для **`TaskListController`** при смене id агента на строке — **`unawaited(getTask(e.taskId))` + `taskModelToListItem` + upsert строки`** (те же причины, что mismatch: **не** **`invalidate`** всего списка по умолчанию). Полный **`invalidate`** — только с мотивацией в PR.

### HTTP vs WS: свежесть **`updatedAt`**

Для **полного `TaskModel`** после **`getTask`** vs последующего **`getTask`** (после WS-refetch) — сравнение **`updatedAt`** как в предыдущем ТЗ. **WS-патч и `updatedAt`:** **`e.ts`** — время на шине, **`task.updated_at`** в БД после COMMIT может быть **строго меньше** **`e.ts`**. **Канон:** при **`applyWsTaskStatus`** (успешный FSM-патч родителя) **не** присваивать **`state.task.updatedAt = e.ts`** — иначе следующий **`getTask`** вернёт **`updatedAt` из БД < state** и правило «устаревший HTTP не затирает» **отбросит** свежий ответ сервера. **`e.ts`** использовать только для логики порядка/отладки **или** в отдельном поле state (**`lastWsEventAt`?** — на усмотрение PR), **не** подменяя **`TaskModel.updatedAt`**. **Эвристика (если в PR уже записали `e.ts` в `updatedAt`):** при merge HTTP **`getTask`** — если **`http.updatedAt < state.updatedAt`**, но **`state.updatedAt - http.updatedAt ≤ 5 s`**, считать это drift шины/БД и **принять** **`http`** (задокументировать в PR); **дефолт для 12.3** — **не** писать **`e.ts`** в **`updatedAt`**.

Для **чистого WS-patch** опираться на FSM выше, **не** на сравнение **`updatedAt`** с **`e.ts`** в **`TaskModel`**.

### **`WsTaskMessageEvent` → `TaskMessageModel` (**§ 49**): не 1:1**

Сравнение полей (**`task_message_model.dart`** ~40–71 vs **`websocket_events.dart`** ~166–179): **`messageId` → `id`**, **`ts` → `createdAt`**; **`projectId`**, **`v`** в **`TaskMessageModel`** **не** попадают. **`senderRole`** в WS есть, в **`TaskMessageModel`** (**12.1**) **нет**.

**Канон:** одна явная функция (например **`TaskMessageModel wsTaskMessageToModel(WsTaskMessageEvent e, {required String taskId})`**) или метод **`applyWsTaskMessage`**, внутри — **только** эти переименования + **`taskId`** из family/события. **Запрещено** без отдельной задачи на расширение **12.1** класть **`senderRole`** в **`metadata['sender_role']`** или иной «тихий» канал — **12.5** не должен читать недокументированные ключи. **Два явных варианта в PR (один выбрать):** (а) **`senderRole` сознательно отбрасывается** (dartdoc: UI не показывает роль отправителя из WS); (б) отдельная задача на поле **`senderRole`** в **12.1** + UI — **не** смешивать с **12.3**.

Далее merge — тот же **`_mergeTaskMessages`**, что и для **HTTP 201**.

### **`realtimeMutationBlocked` и сбой realtime (**§ 52 / § 67**)**

**12.9** **не** пишет в **`TaskDetailState`** через **`copyWith(realtimeMutationBlocked: …)`** напрямую. **Обязателен** публичный **`applyRealtimeFailure(WsServiceFailure failure)`** — тип **`WsServiceFailure`** из **`package:frontend/core/api/websocket_events.dart`** (**~140** и далее), **не** «согласовывать с **`task_exceptions`**». **Дефолт (**§ 80**):** маппинг **`failure` → `realtimeSessionFailure` / `realtimeMutationBlocked` / snack** вынести в **`package:frontend/core/api/realtime_failure_mapper.dart`** (или соседний файл в **`core/api/`**) **одной** функцией/классом с **unit-тестом**; в **PR 12.3** вызывают **только** **`TaskListController`** и **`TaskDetailController`**. **Миграция** **`ChatController._handleWsServiceFailure` / `_applyTerminalServiceFailure`** на общий маппер — **отдельная** follow-up задача в спринте **12** (или согласование со спринтом **11**), **не** блокер **12.3**, чтобы не тащить регрессию чата в задачу задач. **`setRealtimeMutationBlocked(bool)`** — только тесты / edge-case; **12.9** в проде — **`applyRealtimeFailure`**.

### Сброс **`realtimeMutationBlocked`** после сбоя (**§ 79**)

Как в чате: **`_clearRealtimeTransientFailure`** (**`chat_controller.dart`** ~189–198) чистит **transient**, **не** «разблокирует» терминальный сценарий сам по себе. **Канон 12.3:** **`realtimeMutationBlocked == true`** после **terminal** **`applyRealtimeFailure`** (**`auth_lost` / `forbidden` / `connection_limit_exceeded`**, …) **не** сбрасывается автоматически при следующем **`WsServiceFailure.transient`** без явного шага — **сброс** (и повторная попытка мутаций) **только** после **успешного** **`refresh()`** контроллера (**аналог** **`ChatController.refresh`** ~332–357), который заново поднимает задачу/список с HTTP и выставляет state **без** блокировки **или** после явного пользовательского «повторить» (если появится в **12.5**). **`applyRealtimeFailure(transient)`** **никогда** не сбрасывает **`realtimeMutationBlocked`**, выставленный **terminal**-веткой маппера — **семантика 1:1** с **`ChatController._clearRealtimeTransientFailure`** (~189–198): transient чистит только «мягкие» поля, **не** terminal-блок мутаций.

### **`needsRestRefetch` / FSM mismatch / anti-storm (**§ 38 / § 59 / § 64**)**

**§ 59 — `needsRestRefetch`:** **12.9** **не** вызывает **`refresh()`** напрямую на списке. **Обязателен** публичный **`requestRestRefetch()`** на **`TaskListController`**: внутри **`if (_wsRefetchInFlight) return; _wsRefetchInFlight = true; unawaited(refresh().whenComplete(() => _wsRefetchInFlight = false))`** — как **`_requestWsOverflowRefresh`** в **`chat_controller.dart`** ~238–247, **~319–321**.

**§ 64 — FSM mismatch (`previousStatus`):** при буфере WS после реконнекта возможна **серия** mismatch подряд → каждый **`getTask`** без guard даёт шторм, **аналогичный** **`needsRestRefetch`**. **Минимум:** для **`TaskDetailController`** и **`TaskListController`** — **один** общий флаг **`_wsRefetchInFlight`** (или эквивалентное имя) на **оба** сценария — **`requestRestRefetch` / overflow** и **mismatch refetch** — чтобы **не** плодить два независимых inflight и **не** получить параллельный шторм «overflow не блокирует mismatch». **Допустимо** debounce **~250 ms** на **`taskId`** **внутри** того же guard — **один** механизм в PR. **Запрещено** полагаться только на anti-storm **`needsRestRefetch`**.

**§ 65 — дефолты (отклонения только с мотивацией):** список — **`@Riverpod(keepAlive: true)`**; mismatch в списке — **точечный refetch строки**, не **`invalidate`**; смена агента в списке — **`getTask` + `taskModelToListItem`**. **`requestRestRefetch`:** **обязательно** публичное зеркало на **`TaskDetailController`** (**тот же** **`_wsRefetchInFlight`** / общий хелпер).

**§ 78 — что делает зеркало на деталях:** **дефолт (а)** — тот же смысл, что **`refresh()`** после **`_requestWsOverflowRefresh`** в чате: **полная** перезагрузка ветки детали — **`getTask(family.taskId)`** + первая страница **`listTaskMessages`** (**`messagesOffset := 0`**, merge-политика как при cold load), под **тем же** guard **`_wsRefetchInFlight`**. **Не** только **`getTask`** без сообщений **(б)** и **не** только сообщения без задачи **(в)** — иначе после overflow останется рассинхрон. *Trade-off:* тяжелее, чем точечный refetch; отклонение — с мотивацией в PR.

### Список и подзадачи после WS (**§ 54**)

**Не** взаимоисключающие сценарии: одна и та же подзадача может быть **и** строкой **`items`** (**`TaskListController`**), **и** элементом **`state.task.subTasks`** (**`TaskDetailController`**), если фильтр списка не скрывает подзадачи.

**Канон:** **`TaskListController.applyWsTaskStatus`** — всегда патчит строку **`items`**, где **`item.id == e.taskId`** (с FSM **`previousStatus`**). **`TaskDetailController.applyWsTaskStatus`** — всегда, независимо от списка: если **`e.taskId == family.taskId`** — патч **`task`**; если **`e.taskId`** совпал с **`subTasks[i].id`** — патч этой записи в **`subTasks`**. **Оба** пути выполняются при совпадении; **не** выбирать «один вариант в PR» между списком и деталями.
