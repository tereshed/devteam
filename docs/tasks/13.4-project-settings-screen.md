## Предусловия перед началом

1. **Импорты** — только **`package:frontend/...`**.
2. **Presentation не импортирует `dio`** — различение ошибок по типам из **`project_exceptions.dart`** и **`api_exceptions.dart`** (см. колонку **«Зависит от»** выше и **`docs/rules/review.md`** §2).
3. **Локализация** — без хардкода строк в UI; новые ключи с префиксом **`projectSettings`** (или согласованный префикс после аудита **13.6** на дубликаты с общими **`common*`** / **`dataLoadError`**).
4. **`Color.withValues(alpha: …)`**, не **`withOpacity`** (если актуально для версии SDK).
5. **Мини-проверка API:** **`GET /api/v1/projects/:id`** и **`PUT /api/v1/projects/:id`** — тело **`UpdateProjectRequest`** сверить с **`frontend/lib/features/projects/domain/requests.dart`** и **`backend/internal/handler/dto/project_dto.go`**.
6. **Права (RBAC):** при **403** репозиторий бросает **`ProjectForbiddenException`** (как для **`getProject`**). В UI — не generic «ошибка». **Нюанс:** один и тот же **403** на **`reindex`** может означать **нет доступа к проекту** (`checkProjectAccess`) либо **`ErrGitCredentialForbidden`** (credential другого пользователя / запрет на использование) — **`httputil`** отдаёт оба как **403** (**`project_handler.go`**). До появления стабильного различения в UI: либо **один нейтральный** l10n (**например `projectSettingsActionForbidden`**) без текста «у вас нет прав на **этот проект**» как единственного варианта для всех **403** на reindex; либо ветвление по **`ProjectForbiddenException.apiErrorCode`**, если код из JSON **стабилен** и задокументирован в ответе API (**инвестиция в 13.6**, не молчаливый future-debt).
7. **401 / `UnauthorizedException`:** репозиторий уже мапит **401** в **`UnauthorizedException`** (**`dio_repository_error_map`**). В **`dio_client`** отдельного auto-redirect на логин при **401** **нет** (interceptor только пробрасывает **`handler.next(error)`**). На экране **13.4** — тот же UX, что на других фичах: текст **`l10n.errorUnauthorized`** (прецеденты **`create_project_controller.dart`**, **`projects_list_screen.dart`**), без введения **второго** глобального механизма только здесь; выход на экран логина / обновление токена — по существующему контракту **`go_router`** / auth слоя приложения.

## SSOT: частичное обновление (только dirty-патч)

**Зафиксирован ровно один сценарий:** перед каждым **`PUT`** собирается **`UpdateProjectRequest`** только из полей, которые пользователь **реально изменил** в данной форме (dirty-tracking по секциям или по полям). В JSON попадают **только** эти поля (плюс явные флаги **`remove_git_credential`**, **`clear_tech_stack`** — см. таблицу ниже).

**Запрещено:**
- отправлять «снимок всей модели проекта» целиком через один общий **`toJson()`** без dirty-логики;
- передавать **`git_credential_id`** в **13.4** (нет API списка credentials — см. **«Git credential»**);
- одновременно в одном запросе **`tech_stack`** и **`clear_tech_stack: true`** — бэкенд вернёт **400** (**`ErrUpdateProjectTechStackConflict`** в **`project_service.go`**);
- одновременно **`git_credential_id`** и **`remove_git_credential: true`** — бэкенд вернёт **400** (**`ErrUpdateProjectGitCredentialConflict`**);
- включать в **`PUT` тела 13.4** ключи **`name`**, **`description`**, **`status`**, **`settings`**, **`clear_settings`** — даже если они есть в **`UpdateProjectRequest`** (**`project_dto.go`**): эти поля **вне продуктового MVP** этой вкладки; случайная отправка (общий **`toJson`**, копипаст контроллера) класс багов **«поменяли не то»** — исключить на уровне спеки.

**Про `includeIfNull: false`:** при dirty-патче опциональные поля **просто не включаются** в `Map`, если не менялись — **`git_credential_id` не появляется в JSON** → бэкенд **не** трогает привязку credential (ветка `switch` в **`Update`** срабатывает только на **`RemoveGitCredential`** или ненулевой **`GitCredentialID`**). Отвязка возможна **только** через **`remove_git_credential: true`**.

**Инвариант смены провайдера на `local`:** если пользователь меняет **`git_provider`** на **`local`**, а в текущем **`ProjectModel`** есть привязанный **`gitCredential`**, в **том же `PUT`** обязательно включить **`remove_git_credential: true`** (вместе с **`git_provider`**). Иначе сервис вернёт **400** **`ErrGitCredentialNotSupportedForLocal`** (**`project_service.go:557–558`**) при неизменном **`GitCredentialsID`** в БД — «немой» фейл без подсказки.

**Бэкенд для `tech_stack`** (истина — **`project_service.go`**): если в запросе **нет** ни **`clear_tech_stack`**, ни **`tech_stack`** — поле в БД **не** меняется. Если **`clear_tech_stack: true`** — в БД пишется пустой JSON-объект **`{}`**. Если передан **`tech_stack`** — выполняется **полная замена** значения jsonb на тело запроса; **отдельного schema-checker на сервисе нет** — осознанно ограничиваем клиент **MVP**.

**Инвариант «удалил все пары в редакторе»:** если пользователь стёр **все** пары ключ→значение в UI (без отдельной кнопки «Очистить»), в **`PUT` отправлять только `clear_tech_stack: true`**, ключ **`tech_stack` в JSON не включать** (в т.ч. **не** слать **`tech_stack: {}`**: семантически близко к очистке, но это **другой** ключ в DTO и разъезжается с явной кнопкой «Очистить» и тестами).

### `vector_collection`: валидация и Weaviate

Бэкенд на **`PUT`** имя коллекции **не валидирует** — некорректное значение дойдёт до пайплайна индексации без раннего стопа.

**Клиентская валидация (обязательный выбор в PR — зафиксировать в описании PR один вариант):**

1. **Строгий режим (рекомендуется по умолчанию):** для **непустого** значения после **`trim()`** — **`RegExp(r'^[A-Z][A-Za-z0-9_]*$')`** (первая буква — заглавная латиница, далее латиница / цифры / **`_`**). Пустая строка после **`trim()`** — допустима **только** если продукт явно выравнивает это с контрактом создания проекта (**пустой `vector_collection` на create** — см. **10.5**); иначе — своя локальная политика «пусто нельзя» в форме.
2. **Отказ от клиентской валидации:** в описании PR явно: «имя коллекции не валидируем на клиенте, риск мусора и ошибок Weaviate — на эксплуатации / отдельная задача» — тогда в чек-листе мёржа отмечается этот **осознанный** отказ (без regex).

**Сиротство данных (footgun):** смена **`vector_collection`** в БД **не** переносит векторы из старой коллекции Weaviate в новую: старая коллекция остаётся с данными, новая — пустая, пока не выполнена переиндексация. В UI после успешного **`PUT`**, если **`vector_collection`** изменился относительно предыдущей загрузки: показать **баннер / callout** (l10n, например **`projectSettingsVectorCollectionRenamed`**) с явным текстом: нужно запустить **«Переиндексировать»** (или внешняя очистка старых коллекций — вне **13.4**). Не обязательно блокировать **`PUT`**, но пользователь **должен** увидеть предупреждение, а не «тихий» успех.

**Общее UX:** первая загрузка через **`ref.watch(projectProvider(projectId))`** — loading / error + **Retry**; **`ProjectNotFoundException`** (**`api_exceptions.dart`**) — согласовать с дашбордом.

**После успешного `PUT`:** **`ref.invalidate(projectProvider(projectId))`** **и** краткий **success feedback** (**SnackBar** или эквивалент): l10n **`projectSettingsSaved`** (или согласованное имя в **13.6**). Ошибки **`PUT`** — **SnackBar** / баннер по типу исключения (**502** / **400** / **403** / **409** — разные ключи, см. таблицы выше).

## Reindex: контракт и UI

**Эндпоинт:** **`POST /api/v1/projects/:id/reindex`**.

**Контракт `ProjectRepository`:** **`Future<void> reindex(String projectId, {CancelToken? cancelToken})`** — при **202** тело ответа не мапится в **`ProjectModel`**; успех = завершение **`Future`** без исключения. Имя метода и сигнатура — часть контракта PR (в т.ч. для моков в тесте №4).

### Таблица: HTTP и ошибки reindex

| HTTP | Когда (бэкенд) | Клиент (типично) | UI / l10n |
|------|----------------|------------------|-----------|
| **202** | Старт пайплайна | успех, затем **`invalidate(projectProvider)`** | Краткий SnackBar с **`projectSettingsReindexStarted`** (**13.6**) — **не** **`projectSettingsSaved`** (это только успех формы **`PUT`**); затем обновление **`status`**. |
| **400** | **`ErrProjectLocalCannotReindex`** ( **`local`** или пустой **`git_url`** ) и др. валидации из блока **400** в **`httputil`** | **`ProjectApiException`**, **`statusCode: 400`** | Уже частично покрыто **disabled** кнопки; если запрос всё же ушёл — отдельное сообщение. |
| **403** | Нет доступа к проекту **или** **`ErrGitCredentialForbidden`** | **`ProjectForbiddenException`** | Не утверждать единственный текст «нет прав на проект» для всех **403** на reindex — см. предусловие п.6. |
| **409** | **`ErrProjectIndexingConflict`** | **`ProjectConflictException`** | **`projectSettingsReindexConflict`** — **не** путать с **502** и **403**. |
| **502** | **`ErrGitValidationFailed`**, **`ErrGitCloneFailed`** после **`ValidateAccess`** (**`project_service.go:678`**) | **`ProjectApiException`**, **`statusCode: 502`** | Ключ **502** — см. **«Связано с: 13.6»** (один общий **`projectSettingsGitRemoteAccessFailed`** или два ключа в одном PR). |

**Важно:** **409** / **`ProjectConflictException`** — только про «уже индексируется» (и аналогичные конфликты). **401** (**`UnauthorizedException`**), **403**, **502** — другие типы и тексты; **не** подменять их сообщением **`projectSettingsReindexConflict`**.

**Инвариант отмены запроса:** **`CancelToken.cancel()`** отменяет **только HTTP**. После **202** индексация на бэкенде продолжается — **однострочный комментарий** в коде.

**Состояние кнопки «Переиндексировать»:**
- **`status == indexing`** — кнопка **disabled** + l10n (**например `projectSettingsReindexInProgress`**).
- **`git_provider == 'local'`** (строка в **`ProjectModel`**, как приходит с API) **или** «пустой URL для remote-логики reindex»: в **`ProjectModel`** поле **`gitUrl`** — **`required String`** (**не `String?`**); условие «пусто» для UI и для симметрии с бэкендом: **`gitUrl.trim().isEmpty`** (**не** `gitUrl?.isEmpty` — nullable нет). Вместе: **`gitProvider == 'local' || gitUrl.trim().isEmpty`** — **disabled** + **`projectSettingsReindexUnavailable`**.

**Анти-race:** **disabled** по **`GET`** — оптимизация UX, **не** замена обработке **409**: между загрузкой и кликом индексацию мог запустить другой клиент/вкладка — **`POST`** всё ещё может вернуть **409**; обработчик **обязателен** (тест №4 остаётся в силе).

**Пустой `vector_collection`:** отдельной проверки в **`Reindex`** нет; кнопку **не** блокируем только из-за этого (как ранее).

## Вне скоупа 13.4

- **Глобальные** API keys / LLM — экран **13.5**.
- **Редактирование `name` и `description` проекта** на этом экране — **вне 13.4** (поля есть в DTO **`Update`**, но продуктовый MVP вкладки — **git / tech_stack / vector / reindex**; отдельная задача, если переносим сюда карточку «О проекте»). Технически ключи **`name` / `description` / `status` / `settings` / `clear_settings`** в **`PUT` из этого экрана запрещены** — см. **SSOT**-таблицу и **«Запрещено»**.
- **Picker / привязка** Git credential, **сырой JSON** для **tech_stack**.
- **CRUD** Git credentials у пользователя.
- **WebSocket**-стрим прогресса индексации.
- **SEO / title** для web — отдельной задачёй.
