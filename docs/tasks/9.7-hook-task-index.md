# 9.7 Хук: индексация задачи при создании/обновлении

- `backend/internal/service/task_service_test.go`
**Статус:** ⬜ К выполнению

---

## Описание

Задача заключается в интеграции механизма индексации задач в жизненный цикл задачи в `TaskService`. Необходимо обеспечить автоматическую индексацию задачи в векторной базе данных (Weaviate) при её создании и при переходе в терминальные статусы (Completed, Failed, Cancelled). Это позволит агентам использовать историю выполненных задач как базу знаний для решения новых проблем.

---

## Зависимости

- `TaskIndexer` — сервис, отвечающий за формирование документа задачи и его сохранение в Weaviate.
- `TaskRepository` — для получения актуальных данных задачи.
- `TaskMessageRepository` — для включения обсуждения в индекс.

---

## Функциональные требования

### 1. Инъекция зависимости и Graceful Shutdown
- Добавить `TaskIndexer` (интерфейс) в структуру `taskService`.
- **Graceful Shutdown**: Внедрить `sync.WaitGroup` в `taskService` для отслеживания активных фоновых горутин индексации.
- Добавить в интерфейс `TaskService` (и реализовать) метод `Close()`, который будет дожидаться завершения всех фоновых задач (`wg.Wait()`).

### 2. Точки вызова и предотвращение Race Conditions (КРИТИЧНО)
Индексация должна триггериться в следующих методах:
- **`Create`**: Сразу после успешного создания задачи. **Важно**: вызов асинхронной индексации должен происходить **строго после завершения (commit) транзакции БД**, чтобы избежать `RecordNotFound` в горутине.
- **`Update` / `Transition`**: При переходе задачи в терминальный статус (`Completed`, `Failed`, `Cancelled`). Вызов также после коммита.
- **`Correct`**: Так как это действие добавляет важное пользовательское уточнение.
- **`Delete`**: Вызывать `DeleteTask` у индексатора.
- **Конкурентная индексация**: Обеспечить защиту от одновременного запуска нескольких горутин индексации для одной и той же задачи (например, если `Update` и `AddMessage` вызваны одновременно). Использовать дебаунс или полагаться на идемпотентный Upsert в самом `TaskIndexer` с проверкой версии/времени обновления.

### 3. Асинхронность, Надежность и Ресурсы
- **Фоновое выполнение**: Индексация выполняется асинхронно.
- **Предотвращение Data Race (КРИТИЧНО)**: В метод индексации передавать **глубокую копию** объекта `models.Task`. Копирование должно выполняться в основном потоке **ДО** запуска горутины, чтобы избежать конкурентного доступа (Data Race) при чтении/записи полей.
- **Early Return**: В начале метода `indexTaskAsync` добавить проверку на `nil`. Если объект не валиден — немедленный выход.
- **Контекст и Таймауты**: Использовать отвязанный контекст с таймаутом: `ctx := context.WithTimeout(context.WithoutCancel(reqCtx), 2*time.Minute)`. Убедиться, что `TaskIndexer` строго уважает этот контекст.
- **Retry-механизм**: Внутри `indexTaskAsync` реализовать механизм повторных попыток (Exponential Backoff, 3-5 попыток) для защиты от кратковременных сбоев сети или Weaviate.
- **Panic Recovery**: Внутри горутины обязателен `recover` с логированием стек-трейса.
- **Логирование**: Ошибки индексации (после исчерпания ретраев) не должны влиять на основной процесс (Fail Gracefully).

### 4. Безопасность: Маскирование секретов и Side Effects
- Перед отправкой данных в `TaskIndexer` (или внутри него) **обязательно** должен выполняться шаг санитаризации текста.
- **Изоляция изменений**: Санитаризация должна применяться только к локальной копии данных, предназначенной для индексатора. Недопустима мутация оригинального объекта `task` в памяти, которая может повлиять на другие части приложения.
- Маскировать токены, API-ключи и пароли, которые могут содержаться в описании задачи или сообщениях. Векторная БД не должна содержать чувствительных данных.

### 5. Оптимизация AddMessage (DDoS Protection)
- **Ограничение триггеров**: НЕ индексировать задачу на каждое сообщение.
- Вызывать переиндексацию только при добавлении сообщений с типами `MessageTypeResult`, `MessageTypeSummary` или `MessageTypeFeedback`.
- Игнорировать промежуточные логи и системные сообщения для экономии ресурсов БД и Weaviate.

## План реализации

| # | Шаг | Описание |
|---|-----|----------|
| 1 | `DI & Shutdown` | Внедрить `TaskIndexer` и `sync.WaitGroup`. Реализовать метод `Close()`. |
| 2 | `Internal Helper` | Создать метод `s.indexTaskAsync(ctx, task)`. Внутри: `wg.Add(1)`, `defer wg.Done()`, `recover`, механизм Retry и вызов индексатора. |
| 3 | `Sanitization` | Убедиться в наличии механизма маскирования секретов перед отправкой в Weaviate. |

| 6 | `Message Hook` | Добавить фильтрацию по типам сообщений в `AddMessage` и вызов индексации для разрешенных типов. |
| 7 | `Delete Hook` | Добавить вызов `DeleteTask` в `Delete`. |
| 8 | `Unit Testing` | Обновить тесты. Использовать `require.Eventually` или каналы в моках для предотвращения flaky-тестов при проверке асинхронных вызовов. |

## Критерии приёмки

- [ ] `TaskIndexer` интегрирован, реализован Graceful Shutdown через `WaitGroup`.
- [ ] Индексация запускается только после фиксации изменений в БД (нет Race Condition).
- [ ] Реализован механизм Retry для устойчивости к сетевым сбоям.
- [ ] Секреты маскируются перед попаданием в векторную БД.
- [ ] В `AddMessage` индексация срабатывает только на значимые типы сообщений.
- [ ] В горутину передается объект из памяти, минимизируя запросы к БД.
- [ ] Ошибки и паники в фоне изолированы и логируются.
- [ ] Тесты стабильны (не flaky) и подтверждают выполнение всех архитектурных требований.
