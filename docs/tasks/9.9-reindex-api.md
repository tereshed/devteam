# 9.9 API: `POST /api/v1/projects/:id/reindex` (полная переиндексация)

**Sprint:** 9 — Векторная индексация проекта (Backend)
**Файлы:**
- `backend/internal/handler/project_handler.go`
- `backend/internal/service/project_service.go`
- `backend/cmd/api/main.go` (или файл роутинга)
**Статус:** ⬜ К выполнению

---

## Описание

Задача заключается в реализации API-эндпоинта для принудительного запуска полной переиндексации проекта. Это необходимо в случаях, когда автоматическая индексация (задача 9.6) не сработала, либо когда требуется обновить индекс вручную после внешних изменений в репозитории.

Эндпоинт должен принимать ID проекта, проверять права доступа пользователя и запускать уже существующий пайплайн индексации (`runIndexingPipeline`) в фоновом режиме.

---

## Зависимости

- `ProjectService` — необходимо расширить интерфейс методом `Reindex`.
- `ProjectHandler` — необходимо добавить метод `Reindex` для обработки HTTP-запроса.
- `CodeIndexer` — используется внутри сервиса для выполнения самой индексации.

---

## Функциональные требования

### 1. API Endpoint
- **Метод:** `POST`
- **Путь:** `/api/v1/projects/:id/reindex`
- **Авторизация:** Требуется (Bearer Auth / ApiKey).
- **Ответ:** 
  - `202 Accepted` — если процесс успешно запущен.
  - `400 Bad Request` — если ID проекта имеет невалидный формат (UUID).
  - `403 Forbidden` — если у пользователя нет доступа к проекту.
  - `404 Not Found` — если проект не существует.
  - `409 Conflict` — если проект уже находится в статусе `Indexing`.

### 2. Service Layer (`ProjectService`)
- Добавить метод `Reindex(ctx context.Context, userID uuid.UUID, userRole models.UserRole, projectID uuid.UUID) error`.
- **Логика:**
  1. Получить проект из БД и проверить права доступа (`checkProjectAccess`).
  2. Проверить текущий статус проекта. Если статус `Indexing`, вернуть ошибку конфликта.
  3. Обновить статус проекта на `Indexing`.
  4. Запустить `runIndexingPipeline` асинхронно в фоновой горутине.
  5. Вернуть управление (не дожидаясь завершения индексации).

### 3. Работа с конкурентностью и контекстом (КРИТИЧНО)
- **Отвязка контекста и таймаут:** Для фоновой горутины **запрещено** использовать контекст из HTTP-запроса напрямую. Необходимо использовать `context.WithoutCancel(ctx)` (Go 1.21+) или создать новый `context.Background()`.
- **Предотвращение зависаний (Resource Leaks):** Чтобы избежать "вечного" статуса `Indexing` при зависании внешних сервисов (Git, VectorDB), контекст горутины **обязан** иметь таймаут. Например: `ctx, cancel := context.WithTimeout(context.WithoutCancel(reqCtx), 15*time.Minute)`. Обязательно вызывать `defer cancel()` внутри горутины.
- **Перехват паник (Panic Recovery):** Весь код внутри горутины должен быть обернут в `defer func() { if r := recover(); r != nil { ... } }()`. При панике необходимо залогировать стек-трейс и гарантированно обновить статус проекта.
- **Гарантированное обновление статуса:** Финальное обновление статуса проекта (на `Ready` или `IndexingFailed`) должно быть гарантированным при любом исходе. Рекомендуется использовать `defer` для сброса статуса в случае ошибок или паник, чтобы проект не остался заблокированным в состоянии `Indexing`.
- **Race Conditions:** Использовать `UpdateStatus` с условием `WHERE status != 'Indexing'`, чтобы гарантировать атомарность запуска.

## Критерии приёмки

- [ ] Эндпоинт `POST /api/v1/projects/:id/reindex` доступен и требует авторизации.
- [ ] Возвращается `400 Bad Request` при невалидном формате UUID в URL.
- [ ] Метод возвращает `202 Accepted` при успешном запуске.
- [ ] Индексация выполняется асинхронно в фоне с отвязанным контекстом и обязательным таймаутом (напр. 15 мин).
- [ ] Реализован гарантированный сброс статуса проекта (через `defer`) при любом исходе горутины.
- [ ] Реализована защита от повторного запуска, если индексация уже идет (`409 Conflict`).
- [ ] Проверяются права доступа пользователя к проекту.
- [ ] Статус проекта в БД корректно меняется на `Indexing` при старте и на `Ready`/`IndexingFailed` по завершении.
- [ ] Код покрыт юнит-тестами.
