## Контракт `CreateProjectRequest` (актуально для `requests.dart`)

Источник правды — **`frontend/lib/features/projects/domain/requests.dart`**. На момент ревью документа поля **`gitProvider`**, **`gitUrl`**, **`vectorCollection`** — **`required String`** (не `String?`, без `@Default('')` в Freezed). **`description`** имеет `@Default('')` — в конструкторе можно не указывать поле; при сериализации в JSON ключ **`description`** всё равно уходит на бэкенд как **`"description":""`** — это нормально и ожидаемо (пустая строка, не отсутствие ключа).

| Поле | Контракт для UI → запрос |
|------|--------------------------|
| `name` | После **`trim()`** — непустая строка. Строка из одних пробелов — **невалидна** (validator формы / ранний return в контроллере). Max длина согласована с API (255). |
| `description` | Пустой ввод в UI → в запросе пустая строка через дефолт фабрики Freezed или явно `description: ''`. В JSON при `toJson()` поле присутствует как **`"description":""`** — не трактовать как «убрать ключ из тела запроса». |
| `gitProvider` | **Всегда** одно из `gitProviders` — выбор пользователя, **required**. |
| `gitUrl` | **Всегда `String`, не `null`**. Для **`local`** передавать **`''`**. Для **github / gitlab / bitbucket** при принятом правиле продукта: пустой URL — **невалиден** (форма не отправляется); непустой — валидный URL (клиентская проверка + бэкенд `binding:"omitempty,url"`). |
| `vectorCollection` | **Всегда передать явно `''`**. Пользователь не вводит поле. Бэкенд принимает пустую строку: см. `backend/internal/handler/project_handler_test.go` (`VectorCollection: ""`). Индексация/имя коллекции на сервере — не задача экрана. |
| `gitDefaultBranch`, `techStack`, `settings`, `status` | **Не дублировать вручную**, если совпадают с **`@Default`** в Freezed — иначе шум в коде. Задавать только если нужно отличие от дефолта. |
| `gitCredentialId` | **Вне scope 10.5** — оставить **`null`** (следовать сериализации Freezed для опционального поля). |

## Предусловия перед началом

1. **Flutter SDK** — те же минимальные требования, что в задаче 10.4 (`Color.withValues` → Flutter ≥ 3.27), если используются те же паттерны темы.
2. **Единая санитизация сообщений API** — перед кодированием выполнить по репозиторию:  
   `rg -n 'sanitize|mask|userinfo|UserPassword' frontend/lib`  
   На момент обновления этого документа отдельной утилиты маскирования **нет**; дублировать логику в контроллере и репозитории **нельзя**. См. раздел **«Ошибки API и безопасность»** — одна реализация, вызываемая из **`ProjectRepository._handleDioError`** для всех методов.
3. **`CancelToken` для create** — в `ProjectRepository.createProject` токен уже передаётся в `_dio.post(..., cancelToken: cancelToken)`. В notifier создавать токен и **`ref.onDispose(() => token.cancel())`** — отмена при уходе с экрана работает без изменений репозитория.
4. **`git_credential_id`** — **вне scope** (нет UI выбора credentials).
5. **Локализация — одно решение:** все новые строки для экрана создания проекта добавляются **в этой задаче** в `app_en.arb` / `app_ru.arb`, затем `flutter gen-l10n`. Задача **10.8** не должна «пересобирать» ключи с нуля — там аудит/дозаполнение остального Sprint 10; дубли ключей для 10.5 недопустимы.
6. **Импорты** — только `package:frontend/...`, без относительных `../`.

## Контекст и текущее состояние

Слой данных (репозиторий, провайдеры чтения) описан в 10.2–10.3. Экран списка (10.4) даёт вход в раздел проектов и навигацию к созданию. В этой задаче реализуется **форма создания** и вызов `POST /projects` через уже существующий репозиторий.

Бэкенд (`CreateProjectRequest` в Go): имя обязательно; `git_url` валидируется как URL, если не пустой; провайдер парсится на стороне сервиса. Для remote-провайдера с непустым URL выполняется проверка доступа к репозиторию (может вернуться ошибка — см. **безопасность** ниже).

**Текущее поважное место в коде:** в `project_repository.dart` метод **`_handleDioError`** собирает `message` из `data['message']` / `data['error']` и передаёт в исключения **без** очистки (`project_repository.dart` ~176–196). Утечка userinfo из ответа сервера возможна **до** UI — санитизация обязательна **при формировании строки для исключения**, не только в SnackBar.

## Цель

- Экран с **AppBar** (назад + локализованный заголовок).
- Поля формы (первый релиз):
  - **Имя** — обязательное после **`trim()`**, пустые и пробельные — невалидны; `maxLength` согласован с API (255).
  - **Описание** — многострочное; пустое → дефолт модели и **`"description":""`** в JSON (см. контракт).
  - **Git URL** — в модели **всегда строка**; для **`local`** отправлять **`''`**; для remote — непустой валидный URL (см. контракт выше).
  - **Провайдер** — выбор из `gitProviders` (`project_model.dart`); подписи через `gitProviderDisplayLabel(context, value)`.
- Кнопка **Создать**: disabled при загрузке **и** контроллер обязан **игнорировать повторный submit**, пока операция в полёте (`state.isLoading` / внутренний флаг) — см. раздел «Архитектура».
- По успеху — инвалидация списка и переход на хаб проекта (путь ниже).
- Ошибки — **локализованные заголовки по типу исключения**; опционально короткий хвост уже из **исключения с сообщением, санитизированным в репозитории** (см. ниже).

## Архитектурные решения

### Разделение UI и мутации

| Компонент | Ответственность |
|-----------|-----------------|
| `CreateProjectScreen` | `Scaffold`, `AppBar`, встраивание формы |
| Форма (`Form` + `GlobalKey<FormState>`) | `validator`, **`autovalidateMode`** после первой попытки submit; **`trim()` имени** в validator или перед сборкой запроса |
| **`CreateProjectController` (отдельный файл)** | `createProject`, **guard от повторного вызова** при активном запросе, `CancelToken` + `ref.onDispose`, маппинг **типа** исключения → **l10n-ключи** (не повторная санитизация текста — сообщение в исключении уже безопасное) |

**`PopScope` / предупреждение при уходе с несохранённой формой — вне scope MVP 10.5.** Не добавлять в этой задаче. Если понадобится — отдельная подзадача после стабилизации экрана.

Использовать **class-based `AsyncNotifier`** или `@riverpod` notifier для мутации — по замечанию в 10.3 для задач 10.5+.

### Защита от двойного submit

Только `disabled` на кнопке недостаточно (повторный тап до перестроения). В контроллере: если уже идёт создание — **второй `submit` возвращает без вызова** `createProject`. Зафиксировать это unit-тестом.

### Условная валидация Git URL

- **`local`:** в запрос уходит **`gitUrl: ''`**.
- **`github` / `gitlab` / `bitbucket`:** пустой URL после trim — **ошибка валидации**; непустой — проверка формата URL на клиенте. Поведение покрыть тестами (см. ниже).

### Ошибки API и безопасность (утечка токенов)

Сообщения из `data['message']` / `data['error']` могут содержать URL с **`user:password@`** (ответ бэкенда про clone и т.д.). Сегодня **`_handleDioError`** кладёт эту строку в **`ProjectApiException.message`** как есть — утечка возможна **ещё в репозитории**.

**Инвариант (одна точка входа):**

1. Вынести функцию **`sanitizeUserFacingMessage(String input)`** в **`frontend/lib/core/utils/sanitize_user_facing_message.dart`** (или соседний модуль в `core/`), с unit-тестами: на вход `https://u:tok@host/repo` → на выходе строка **без** подстроки `tok@`, **без** пары `u:` перед `@`, сохраняя при необходимости хост/путь для диагностики без секретов.
2. В **`ProjectRepository._handleDioError`** присваивать во все ветки с текстом из ответа сервера:  
   `final safe = sanitizeUserFacingMessage(raw)`  
   и передавать **`safe`** в конструкторы исключений (`UnauthorizedException`, `ProjectForbiddenException`, `ProjectConflictException`, `ProjectApiException`, …). Так **все** методы репозитория (`createProject`, `getProject`, …) получают одно поведение.
3. **Контроллер экрана создания** не санитизирует повторно для безопасности; он только выбирает **локализованный заголовок** по `runtimeType` / `statusCode` и при необходимости показывает **короткий** `exception.message` (уже безопасный). Логирование полных тел ошибок с URL — запретить или маскировать тем же helper.

После появления второго репозитория с Dio — вынести общий **`sanitize`** в вызов из общего error-handler или дублировать вызов утилиты в каждом `_handleDioError` (не две разные реализации алгоритма).

### После успешного создания

1. `ref.invalidate(projectListProvider(...))` (актуальные аргументы — как на экране списка).  
2. `context.go('/projects/${created.id}')`.

### Адаптивность

На широком экране — `ConstrainedBox(maxWidth: 560–640)`; на узком — полная ширина с отступами из темы.

Разделить **слой санитизации / репозитория** и **UI**.

### Unit (репозиторий + `sanitizeUserFacingMessage`)

| Сценарий | Ожидание |

### Widget / controller (экран)

| Сценарий | Ожидание |
|----------|----------|
| Успешное создание | Один вызов `createProject`, затем навигация (mock `GoRouter`) или проверка вызова invalidate |
| **409** `ProjectConflictException` | **`find.byType(SnackBar)`** (или `MaterialBanner`) содержит **локализованный** текст «имя занято» / ключ из arb; не сырое тело ответа сервера |
