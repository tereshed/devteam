## Контракт HTTP — канон (зафиксировано)

### Каталог инструментов (листинг)

| Поле | Значение |
|------|-----------|
| Метод и путь | **`GET /api/v1/tool-definitions`** |
| Префикс | **`@BasePath /api/v1`** в Swagger; в аннотациях **`@Router`** — **`/tool-definitions [get]`** |
| Назначение | Глобальный каталог для UI: **активные** записи (**`is_active = true`**), без привязки к **`projectId`** |

**Регистрация роутов:** группа **`/api/v1`**, **тот же JWT-middleware** (`Use(authMW)`), что и для **`/projects`** — рядом с **`projects.GET`** в **`backend/internal/server/server.go`** (не внутрь **`projects/:id`**, ресурс плоский).

**Сортировка ответа:** стабильно **`category ASC`**, затем **`name ASC`**. Значение **`category`** в БД — произвольная строка-код (часто EN); **локализация подписей категорий на UI — вне скоупа 13.3.1**, на экране показывать **`category` как пришло с сервера** (однострочная оговорка для ревью).

**Preferred (как в 13.3):** **`PATCH /api/v1/projects/:id/team/agents/:agentId`** — то же тело **`PatchAgentRequest`**, расширенное полем **`tool_bindings`** (см. ниже).

## Часть A — Backend API

### A.1 Каталог: DTO и запрет утечки модели

- **`Запрещено`** возвращать **`models.ToolDefinition`** в HTTP — только **`dto`** (как **`team_dto.go`**).
- Ввести **`dto.ToolDefinitionListItemResponse`** (минимум: **`id`**, **`name`**, **`description`**, **`category`**, **`is_builtin`**).
- **`Запрещено`** отдавать **`parameters_schema`** в листинге каталога (**размер payload**, потенциально лишняя детализация для клиента). Отдельный **`GET /tool-definitions/:toolDefinitionId`** с полной схемой — **отдельная задача**, не 13.3.1.

### A.2 Чтение команды: `AgentResponse.tool_bindings` (один контракт)

**Зафиксировано:** в **`AgentResponse`** массив **`tool_bindings`**, элемент — **`dto.ToolBindingResponse`**, минимум:

- **`tool_definition_id`** (UUID string)
- **`name`**
- **`category`**

**Не** предлагать альтернативу «только UUID + merge на фронте» — для коллекции это даёт **N+1 / race** с каталогом и два разных **`fromJson`**.

**Репозиторий:** **`GetByProjectID`** (и при необходимости **`GetByID`** команды, если ответ переиспользуется) — **`Preload("Agents.ToolBindings.ToolDefinition")`** (или эквивалент), маппинг в DTO без прокидывания **`parameters_schema`**.

**Read vs write:** элемент **`tool_bindings` в ответе** и элемент **в теле PATCH** — **разные DTO**: Swagger **`ToolBindingResponse`** vs **`ToolBindingPatchItem`**; на фронте — **разные модели** (**read** для **`AgentModel`**, **write** — только поля PATCH). **Запрещено** переиспользовать одну Freezed-модель для **`toJson`** PATCH и **`fromJson`** ответа команды.

### A.3 PATCH: три состояния для `tool_bindings`

Расширить **`dto.PatchAgentRequest`** в духе **`patch_agent_request.go`** (три-состояние **omit / присутствует со значением / `null` отвергнуть**).

| JSON | Семантика |
|------|-----------|
| Ключ **`tool_bindings` отсутствует** | Не менять привязки инструментов |
| **`"tool_bindings": []`** | **Снять все** bindings у агента |
| **`"tool_bindings": [ … ]`** | **Заменить весь набор** на переданный (после нормализации) |
| **`"tool_bindings": null`** | **400** (как **`is_active: null`** в **`patch_agent_request_test.go`**) — явный отказ |

**Обязательно:** юнит-тесты в **`patch_agent_request_test.go`** на разбор JSON для **`tool_bindings`**: omit, пустой массив, непустой массив, **`null` → ошибка**.

### A.4 Тело элемента PATCH и поле `config`

**Зафиксировано для 13.3.1 — вариант (a):** на фронте **`config` не редактируется**; сервер при вставке строк **`agent_tool_bindings` всегда пишет **`config = '{}'`**. Расширение на редактируемый **`config`** — отдельная задача (в объект **`ToolBindingPatchItem`** позже добавят поля без поломки контракта массива объектов).

**Wire-форма элемента массива `tool_bindings` в PATCH (жёстко):** только объект **`dto.ToolBindingPatchItem`** с полем **`tool_definition_id`** (UUID string). **Запрещены** массивы сырых UUID-строк — иначе при появлении **`config`** на том же уровне контракт сломает старых клиентов.

### A.5 Стратегия записи bindings и конкурентность

**Порядок выполнения:** валидация существования и **`is_active`** (**A.7**) и лимит (**A.6**) — **до открытия транзакции**. Затем транзакция: **`DELETE … WHERE agent_id = ?`** → **`INSERT`** новых строк (**без дополнительных SELECT-проверок внутри tx**; TOCTOU допустим при **last writer wins**).

**Зафиксировано для 13.3.1:** **full replace** в одной транзакции: **`DELETE FROM agent_tool_bindings WHERE agent_id = ?`** + **`INSERT`** новых строк с **`config = '{}'`**.

**Обоснование:** при неизменяемом **`config`** потеря данных при «пере-выборе» того же tool не возникает; **diff / upsert** — когда появится редактируемый **`config`** (отдельная задача).

**Конкурентные PATCH одного агента:** уровень изоляции по умолчанию (**READ COMMITTED**); **пессимистических блокировок нет**; побеждает **последний успешно закоммиченный** запрос (**last writer wins**). Явно зафиксировать в PR-комментарии / описании — без **`SELECT FOR UPDATE`** в 13.3.1.

**Идемпотентность и `updated_at`:** повторная отправка **того же набора** **`tool_definition_id`** (после **dedup**) → **200**, состояние **`agent_tool_bindings`** консистентно (в т.ч. **`[A,A,B]`** эквивалентно **`[A,B]`** после нормализации). **`No-op detection`** (пропуск DELETE/INSERT при совпадении множества) — **вне скоупа 13.3.1**. Если ключ **`tool_bindings` присутствует в запросе и валидация прошла — **`updated_at` агента обновляется всегда**, даже если итоговое множество совпало бы с предыдущим (**упрощение реализации**, симметрично отсутствию no-op detection у остальных полей **`PatchAgentRequest`**).

### A.6 Лимиты (безопасность / DoS)

- **Максимум элементов** в **`tool_bindings`** в одном PATCH: **`50`** (после **dedup**). Превышение → **`400`**.
- **Dedup:** сервер **нормализует** входной массив к **уникальным** **`tool_definition_id`** (порядок после нормализации — стабильный, например по UUID string); **дубликаты в теле не считаются ошибкой** (composite PK и так не даст дубликат строки, но семантика должна быть предсказуемой).

### A.7 Валидация существования и `is_active` (без N+1)

**Запрещён** цикл **`GetByID`** по каждому UUID.

**Обязательно:** один запрос вида **«все id из запроса exist AND is_active»** (`WHERE id IN (…) AND is_active = true`), сравнение **количества найденных** с **количеством уникальных id** после **dedup**. Несовпадение → **`400`**.

### A.13 Деактивация `tool_definitions` и существующие bindings (**out of scope**)

Деактивация инструмента (**`is_active = false`**) **не каскадит** и **не удаляет** существующие **`agent_tool_bindings`**. Валидация **`is_active`** в **13.3.1** действует **только при PATCH** (нельзя добавить неактивный id). Восстановление консистентности «вручную» — повторный PATCH без деактивированных id. Поведение при **`DELETE`** из **`tool_definitions`** — см. FK в **`016_alter_agents.sql`**; в **13.3.1** миграции **не** менять.

**Канон как у `PromptsRepository` (13.3):** **`mapDioExceptionForRepository`**, **`CancelToken`** параметром запросов, иерархия **`Tools*Exception`**.

### B.2 Модели агента и обратная совместимость

**Wire `snake_case` ↔ Dart `camelCase`:** явные **`@JsonKey`**, как **`prompt_id` → promptId`** в **13.3** — не именовать Dart-поля **`tool_definition_id`** «в лоб».

- **`AgentModel`:** **`toolBindings`** + **`@JsonKey(name: 'tool_bindings')`** + **`@Default(<immutable>[])`**.
- **`ToolBindingResponseModel`** (только read / **`GET …/team`**): Dart **`toolDefinitionId`**, **`name`**, **`category`** с **`@JsonKey(name: 'tool_definition_id')`** для id ( **`name`** / **`category`** — при совпадении ключей JSON можно без лишнего **`@JsonKey`**).
- **`ToolBindingPatchItem`** (write): одно поле **`toolDefinitionId`** + **`@JsonKey(name: 'tool_definition_id')`** на wire **`PATCH`**. **`UpdateAgentPatch.toolBindings`** — **`List<ToolBindingPatchItem>`** (или эквивалент под sentinel PATCH). **Не** сериализовать read-модель в PATCH.

### B.3 PATCH и `_dirty`

- Три состояния **`tool_bindings`** на клиенте должны совпадать с сервером: **omit** если секцию tools не трогали; **`[]`** если сняли все; иначе — массив **`ToolBindingPatchItem`** (**Dart:** **`toolDefinitionId`** → wire **`tool_definition_id`**).
- Расширить **`UpdateAgentPatch`** / sentinel-патч (**13.3**) без **`Map` сырым**.

### B.4 Жизненный цикл каталога и race с Save (зеркало 13.3)

| Тема | Решение |
|------|---------|
| Провайдер каталога | **`autoDispose`**, скоуп диалога (как провайдеры промптов в **13.3**); **не** `keepAlive: true` без причины |
| Пока **`GET /tool-definitions`** в полёте | Секция инструментов: **скелетон / заблокированный блок**; **остальная форма доступна**; **Save разрешён** — если пользователь **не** менял tools, в PATCH **omit** ключа **`tool_bindings`** (аналог **`_promptDirty`** / «не трогали секцию») |
| Ошибка каталога | Секция в состоянии ошибки; **Save без изменений tools** → **omit** **`tool_bindings`** |
| **`CancelToken`** | **`cancel()` в `dispose`** тела диалога — как **`GET /prompts`** в **13.3** |
| Каталог перезагрузился с ошибкой **после** того, как пользователь менял выбор | **Локальный state выбранных bindings** остаётся источником истины для PATCH; ошибка reload **не сбрасывает** чекбоксы; Save отправляет текущий локальный набор (**если** секция tools «грязная»). |

### B.6 Widget-тесты — обязательная матрица

Базовые shell и **`find.byKey(Key('agentEditDialogBody'))`** — как **13.3**. Дополнительно:

1. Каталог загружен — список/чипы; уже выбранные из **`agent.toolBindings`** отмечены.
2. Loading каталога — секция в скелетоне; остальная форма доступна; Save без правок tools → **нет** ключа **`tool_bindings`** в PATCH.
3. Ошибка **`GET /tool-definitions`** — сообщение в секции; успешный Save остальных полей → **`tool_bindings` omit**.
4. Каталог успел загрузиться → пользователь изменил выбор → повторный **`GET /tool-definitions`** упал (например после пересоздания провайдера) → локальный выбор сохранён; Save с «грязной» секцией tools отправляет **`tool_bindings`** согласно локальному state. **Техника:** мок **`ToolsRepository.fetchToolDefinitions`** с двумя ответами подряд (**`mockSequence`**) **или** закрытие и повторное открытие диалога между состояниями.

9. Регресс **13.3** — поля модели, промпта, **`code_backend`**, **`is_active`** работают как раньше.
10. Секция прокручиваема на **обоих** shell (**`Size(400, …)`** и **`Size(800, …)`**), как релевантные сценарии из **13.3**.

3. **PATCH** поддерживает три состояния **`tool_bindings`**; пустой массив снимает все привязки; при наличии ключа **`tool_bindings`** после успешной валидации **`updated_at` агента** обновляется всегда (**A.5**); тесты **`patch_agent_request_test.go`** покрывают разбор JSON.
4. Валидация id — один batch-запрос **до** транзакции; лимит **50**; **MCP `tool_definitions_list`** + тест; инструмент возврата команды в MCP — см. **A.11**.
5. Фронт: **`ToolsRepository`** по путям **B.1**, **`ToolBindingResponseModel`** / **`ToolBindingPatchItem`**, **`CancelToken`**, сценарии **B.4** / **B.6**; l10n; контракт **400** без обязательного списка id (**A.8**).
