# 9.12 Unit-тесты: IndexerService

**Sprint:** 9 — Векторная индексация проекта
**Файл:** `backend/internal/service/indexer_service_test.go`
**Статус:** ⬜

---

## Что сделать

Реализовать комплексное unit-тестирование для `IndexerService`. Сервис является координатором процесса индексации, поэтому тесты должны сфокусироваться на правильном взаимодействии с под-индексаторами и векторной БД.

**Основные сценарии для тестирования:**
1. **Полная индексация проекта:**
   - Проверка **конкурентного** вызова `CodeIndexer`, `TaskIndexer` и `ConversationIndexer` (например, через `errgroup`).
   - Проверка отсутствия гонок данных (data races) при параллельном выполнении.
   - Проверка корректности передачи `projectID` и параметров индексации.
2. **Инкрементальное обновление:**
   - Проверка логики определения того, что именно требует обновления.
   - Тестирование частичного обновления только одного из типов данных (например, только задач).
3. **Обработка ошибок и краевые случаи:**
   - **Частичный сбой индексаторов:** при ошибке одного из под-индексаторов сервис должен залогировать ошибку, дождаться завершения остальных процессов и вернуть агрегированную ошибку.
   - Ошибки сохранения в векторную БД (ошибки репозитория).
   - Обработка пустых проектов (отсутствие файлов/задач/чатов).
   - **Отмена контекста (Context Cancellation):** проверка немедленного прерывания индексации при отмене `context.Context` (возврат `context.Canceled`).

## Зачем

`IndexerService` обеспечивает актуальность векторного представления проекта. Ошибки в этом сервисе приведут к тому, что Orchestrator (задача 9.11) будет получать нерелевантный или устаревший контекст, что критически снизит качество работы AI-агентов.

## Критерии приёмки

- [ ] Создан файл `backend/internal/service/indexer_service_test.go`.
- [ ] Все внешние зависимости заменены моками (`testify/mock`), сгенерированными с помощью `mockery` или `go generate` (ручное написание моков запрещено).
- [ ] Реализован тест на успешную конкурентную полную индексацию проекта с проверкой агрегации результатов.
- [ ] Реализован тест на успешное инкрементальное обновление.
- [ ] Сценарии с ошибками и краевыми случаями реализованы через **Table-Driven Tests**.
- [ ] Реализованы тесты на обработку частичных сбоев (ожидание всех горутин + агрегированная ошибка) и отмену контекста.
- [ ] Покрытие кода `IndexerService` составляет не менее 80%.
- [ ] Команда `go test -v -race ./internal/service/indexer_service_test.go` проходит успешно (флаг `-race` обязателен для проверки конкурентности).

## Ограничения

- **Никаких реальных соединений:** Тесты не должны требовать запущенного экземпляра Weaviate или доступа к файловой системе (все данные мокаются).
- **Скорость:** Тесты должны выполняться в памяти и завершаться за доли секунды.
