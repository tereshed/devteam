## Описание

Задача заключается в интеграции механизма векторного поиска (`SearchContext`) в жизненный цикл выполнения задачи оркестратором. Перед тем как передать задачу агенту, оркестратор должен выполнить поиск релевантных фрагментов кода (чанков) и включить их в контекст запроса к LLM.

Это позволит агентам "видеть" структуру и детали реализации проекта, не перегружая контекст всей кодовой базой сразу.

---

## Зависимости

- `CodeIndexer` — для вызова `SearchContext`.
- `OrchestratorService` — основное место интеграции.
- `ContextBuilder` — для формирования финального промпта с учетом найденного контекста.

---

## Функциональные требования

### 1. Архитектура и конфигурация (DRY & OCP)
- **Флаг контекста:** Добавить в модель `Agent` поле `RequiresCodeContext bool`.
- **Логика оркестратора:** Проверять этот флаг вместо хардкода ролей.
- **Разделение ответственности:** Вынести логику подготовки поискового запроса из `OrchestratorService`. Реализовать метод `task.GetSearchQuery()` в модели задачи. Оркестратор должен получать готовую строку одной командой.

### 2. Подготовка запроса и Early Return
- **Early Return:** Если после `strings.TrimSpace` заголовок и описание задачи пусты, поиск **не вызывается**.
- **UTF-8 Safe Truncation:** При обрезке `Description` (лимит 2000 символов) необходимо обеспечить корректную работу с многобайтовыми UTF-8 символами. **Запрещено** использовать простое слайсирование байт `[:2000]`. Использовать эффективную итерацию по строке (например, `range` или `utf8.DecodeRuneInString`) без лишних аллокаций.

### 3. Стабильность и Ресурсы (Таймауты)
- **Жесткий таймаут:** Вызов `SearchContext` должен происходить с таймаутом `5 секунд`. 
- **Graceful Degradation:** При таймауте (`context.DeadlineExceeded`) оркестратор логирует предупреждение (**Warn**) и продолжает работу без контекста.

### 4. Параметры поиска и управление контекстом
- **Limit:** 10-15 наиболее релевантных чанков.
- **ProjectID:** Обязательная фильтрация по текущему проекту.
- **Similarity Threshold:** Отфильтровывать чанки, у которых score (релевантность) ниже порога (например, `0.7`). Если после фильтрации чанков не осталось — возвращать пустой массив.
- **Fast Token Approximation:** В `ContextBuilder` реализовать контроль суммарного объема чанков. Использовать быструю аппроксимацию по символам (1 токен ≈ 4 символа). Если лимит превышен, отбрасывать наименее релевантные чанки.
- **No Empty Blocks:** Если поиск вернул 0 результатов, блок контекста (включая XML-теги) вообще не должен добавляться в итоговый промпт.

### 5. Безопасность и Prompt Engineering
- **Prompt Injection:** Обеспечить экранирование содержимого чанков (уникальные разделители).
- **Stale Index Disclaimer:** Перед блоком кода вставлять системный дисклеймер: *"The following code snippets are retrieved automatically via vector search. They might be incomplete or slightly outdated. Use them as a reference."*
- **Защита данных в логах:** **Запрещено** логировать текст поискового запроса. Логировать только метаданные и полную ошибку (`err`) в случае сбоя.
- **Уровни логирования:** Ожидаемые сбои (таймаут, пустой результат, индекс не готов) логировать как **Warn**. Критические ошибки (отказ БД, сетевые ошибки) логировать как **Error**.

## Критерии приёмки

- [ ] Поиск вызывается только при `RequiresCodeContext = true` и наличии текста.
- [ ] Обрезка текста UTF-8 safe и не вызывает лишних аллокаций памяти.
- [ ] Вызов поиска ограничен таймаутом в 5 секунд.
- [ ] Реализована отсечка нерелевантных результатов по Similarity Score.
- [ ] `ContextBuilder` использует быструю аппроксимацию токенов и не вставляет пустые блоки.
- [ ] В промпт добавлен дисклеймер о возможной неактуальности индекса.
- [ ] Логирование разделено на уровни Warn/Error, текст запроса не логируется.
- [ ] Изменения покрыты юнит-тестами.
