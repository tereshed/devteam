# 6.6 Реализация: `ResultProcessor` — обработка результатов агентов и маршрутизация

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/service/result_processor.go`
**Статус:** ⬜ В планах

---

## Описание

Реализовать **`ResultProcessor`** — отдельный компонент, отвечающий за обработку результатов выполнения агентов и принятие решений о дальнейшем маршруте задачи в пайплайне.

После завершения выполнения любого `AgentExecutor` ( Planner, Developer, Reviewer, Tester ), `OrchestratorService` делегирует анализ результата и определение следующего шага `ResultProcessor`. Это позволяет вынести логику ветвления пайплайна из основного оркестратора и сделать её тестируемой и расширяемой.

---

## Зависимости

**Требует:**
- **6.1** — интерфейс `AgentExecutor` и типы `ExecutionResult`.
- **6.4** — `OrchestratorService` (интегрирует `ResultProcessor`).
- **6.5** — логика пайплайна `Plan → Develop → Review → Test`.
- **3.6** — State Machine статусов задач.

**Используется в:**
- **6.4** — `OrchestratorService` вызывает `ResultProcessor` после получения результата от агента.
- **6.7** — возможное расширение для параллельного выполнения подзадач.

## Функциональные требования

### 1. Архитектура: Паттерн Strategy (🏗️ Anti-God Object)

**Проблема:** Логика обработки для 4+ разных ролей в одном методе приведёт к гигантскому switch/case.

**Решение:** Использовать паттерн **Strategy**. Основной `ResultProcessor` — только маршрутизатор, делегирующий логику конкретным обработчикам.

```go
// RoleProcessor — обработчик для конкретной роли
type RoleProcessor interface {
    Process(ctx context.Context, result *agent.ExecutionResult, iterations IterationCounters) (ProcessResult, error)
}

// ResultProcessor содержит map[string]RoleProcessor — маппинг роль → обработчик
// При добавлении новой роли достаточно реализовать RoleProcessor и зарегистрировать в мапе
```

**Требования:**
- Основной `ResultProcessor` только определяет роль и вызывает соответствующий `RoleProcessor`
- Каждый `RoleProcessor` (PlannerProcessor, DeveloperProcessor, ReviewerProcessor, TesterProcessor) — отдельная структура в своём файле
- Единый реестр обработчиков через конструктор:
  ```go
  func NewResultProcessor(cfg Config, processors map[string]RoleProcessor) *ResultProcessor
  ```

### 2. Архитектура и ответственность (🏗️ Single Responsibility)
- **Изоляция логики:** `ResultProcessor` и все `RoleProcessor` — чистая бизнес-логика без побочных эффектов. Не ходят в БД, не вызывают Docker, не делают HTTP-запросы.
- **Stateless:** Компонент не хранит состояние между вызовами. Все данные передаются через параметры.
- **Deterministic:** При одинаковых входных данных всегда возвращает одинаковый результат (идемпотентность).
- **Low Coupling:** Процессор не зависит от GORM-моделей. Используются только примитивы и DTO.

### 3. Конфигурация лимитов (🔄 Loop Prevention)

**Требование:** Лимиты итераций передаются через конфигурацию при инициализации, НЕ хардкодятся.

```go
type Config struct {
    MaxReviewIterations int  // default: 3
    MaxTestIterations   int  // default: 3
}

func NewResultProcessor(cfg Config, processors map[string]RoleProcessor) *ResultProcessor
```

**Логика счётчиков:**
- `review_iterations` — количество циклов `Review → Develop`
- `test_iterations` — количество циклов `Test → Develop`
- При превышении лимита — `DecisionFail` с `error_message`: "Превышен лимит попыток исправления"
- Проверка лимита происходит ДЕРЕВО принятия решения о возврате

### 4. Обработка результатов по ролям

#### 4.1. Результат Planner (LLM)
- **Успех (`success: true`):**
  - План получен и валиден → следующий шаг: `Developer`
  - Статус задачи: `planning` → `in_progress`
- **Неудача (`success: false` или план невалиден):**
  - Задача переходит в статус `failed`
  - `error_message` содержит описание проблемы

#### 4.2. Результат Developer (Sandbox)
- **Успех (`success: true`):**
  - Код написан, diff создан → следующий шаг: `Reviewer`
  - Статус задачи: `in_progress` → `review`
- **Неудача (`success: false`):**
  - Задача переходит в статус `failed`
  - Сохраняются логи и артефакты для диагностики

#### 4.3. Результат Reviewer (LLM)
- **Успех (`success: true`, `review_decision: approve`):**
  - Код одобрен → следующий шаг: `Tester`
  - Статус задачи: `review` → `testing`
- **Успех (`success: true`, `review_decision: changes_requested`):**
  - Требуются правки → **возврат к `Developer`** (при проверке лимита)
  - Статус задачи: `review` → `in_progress`
  - В `ContextAdditions` добавляются комментарии ревьюера
  - Инкрементируется `review_iterations`
- **Неудача (`success: false`):**
  - Задача переходит в статус `failed`

#### 4.4. Результат Tester (Sandbox)
- **Успех (`success: true`, `test_result: pass`):**
  - Все тесты пройдены → пайплайн завершён
  - Статус задачи: `testing` → `completed`
- **Успех (`success: true`, `test_result: fail`):**
  - Тесты не пройдены → **возврат к `Developer`** (при проверке лимита)
  - Статус задачи: `testing` → `in_progress`
  - В `ContextAdditions` добавляется отчёт о проваленных тестах
  - Инкрементируется `test_iterations`
- **Неудача (`success: false`):**
  - Инфраструктурная ошибка (Sandbox упал) → задача переходит в `failed`

### 5. Безопасность и валидация (🛡️ Validation & Security)

#### 5.1. Path Traversal Protection (КРИТИЧНО)
**Требования к валидации путей в артефактах (файлы из diff):**
- Использовать `filepath.Clean()` для нормализации пути
- Проверять, что очищенный путь строго начинается с разрешённой базовой директории:
  ```go
  cleaned := filepath.Clean(path)
  if !strings.HasPrefix(cleaned, workspaceRoot) {
      return error // Path traversal detected
  }
  ```
- Запрещены любые `../` или абсолютные пути вне `workspaceRoot`

#### 5.2. OOM Protection — Truncate First (КРИТИЧНО)
**Требования к обработке вывода агентов:**
- **Сначала truncation, потом обработка:** Ограничение длины (например, 10KB) ОБЯЗАНО выполняться ДО любых операций со строками (регулярки, парсинг, удаление ANSI-кодов)
- Если агент вернул 500MB логов — сначала усекаем до лимита, потом чистим
- Использовать `strings.Builder` с `Grow()` при необходимости объединения строк

#### 5.3. Валидация `ExecutionResult`
- Проверка обязательных полей (`Success`, `Output` при `Success == false`)
- Проверка структуры артефактов (если есть)
- Early return в начале метода при nil-указателях

### 6. Контракт и типы

```go
// ProcessResult — результат обработки, определяющий следующий шаг
// Возвращается по значению (не по указателю) — оптимизация аллокаций
type ProcessResult struct {
    Decision       PipelineDecision  // next_step / retry / fail / complete
    NextRole       string            // Роль следующего агента (если Decision == next_step)
    NewStatus      string            // Новый статус задачи
    Iterations     IterationCounters // Обновлённые значения счётчиков
    ErrorMessage   string            // При Decision == fail
    ContextAdditions map[string]string // Дополнительный контекст для следующего агента
}

type PipelineDecision string

const (
    DecisionNextStep PipelineDecision = "next_step"  // Переход к следующей роли
    DecisionRetry    PipelineDecision = "retry"      // Возврат к предыдущей роли (Develop)
    DecisionFail     PipelineDecision = "fail"       // Перевод задачи в failed
    DecisionComplete PipelineDecision = "complete"   // Завершение пайплайна
)

// IterationCounters — счётчики циклов возврата
type IterationCounters struct {
    ReviewIterations int
    TestIterations   int
}
```

## Интерфейс

```go
// ResultProcessor — основной маршрутизатор результатов
type ResultProcessor interface {
    // Process анализирует результат выполнения агента и возвращает решение о следующем шаге
    // Early Return: метод обязан начинаться с проверки на nil для всех ссылочных типов
    // Unknown Role: при неизвестной роли возвращается ошибка (не паника)
    Process(
        ctx context.Context,
        currentRole string,                    // Текущая роль (planner/developer/reviewer/tester)
        currentStatus string,                  // Текущий статус задачи
        executionResult *agent.ExecutionResult,
        iterations IterationCounters,
    ) (ProcessResult, error)  // Возвращается по значению, не *ProcessResult
}

// RoleProcessor — обработчик для конкретной роли (Strategy pattern)
type RoleProcessor interface {
    Process(ctx context.Context, result *agent.ExecutionResult, iterations IterationCounters) (ProcessResult, error)
}

// Конструктор
func NewResultProcessor(cfg Config, processors map[string]RoleProcessor) *ResultProcessor
```

## Критерии приёмки

### Архитектура
- [ ] Реализован паттерн Strategy: интерфейс `RoleProcessor` и отдельные обработчики для каждой роли.
- [ ] Основной `ResultProcessor` только маршрутизирует вызовы, не содержит switch/case на сотни строк.
- [ ] Каждый `RoleProcessor` находится в отдельном файле (`planner_processor.go`, `reviewer_processor.go` и т.д.).
- [ ] Лимиты итераций передаются через `Config` в конструкторе, не захардкожены.

### Безопасность
- [ ] Реализована защита от Path Traversal: используется `filepath.Clean()` + `strings.HasPrefix()` с проверкой базовой директории.
- [ ] Реализована защита от OOM: truncation до лимита (10KB) выполняется ДО любой обработки строк.
- [ ] Валидация `ExecutionResult` с early return при nil.

### Сигнатуры и типы
- [ ] Метод `Process` принимает `currentRole string` и `currentStatus string`, а не `*models.Task` (low coupling).
- [ ] Метод `Process` возвращает `ProcessResult` по значению, а не `*ProcessResult` (оптимизация GC).
- [ ] При неизвестной роли возвращается ошибка `ErrUnknownRole`, а не паника.

### Логика
- [ ] Реализована логика обработки для всех ролей: Planner, Developer, Reviewer, Tester.
- [ ] При `review_decision: changes_requested` — проверка лимита, затем `DecisionRetry` или `DecisionFail`.
- [ ] При `test_result: fail` — проверка лимита, затем `DecisionRetry` или `DecisionFail`.
- [ ] При `review_decision: approve` — `DecisionNextStep` к Tester.
- [ ] При `test_result: pass` — `DecisionComplete`.
- [ ] Счётчики итераций корректно инкрементируются при возвратах к Developer.

### Тестирование
- [ ] Написаны unit-тесты для каждого `RoleProcessor` (с использованием моков).
- [ ] Написаны unit-тесты для `ResultProcessor` (проверка маршрутизации).
- [ ] Тесты покрывают: approve, changes_requested, test pass, test fail, limit exceeded.
- [ ] Тесты покрывают краевые случаи: nil result, unknown role, empty artifacts, path traversal attempt.
- [ ] Добавлен тест на OOM: проверка truncation большого вывода агента.
- [ ] Компонент покрыт логированием (`slog`) с полями `task_id`, `role`, `decision`.
