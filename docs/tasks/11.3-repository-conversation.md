## Пагинация: зачем кламп на клиенте

Бэкенд (`backend/pkg/httputil/pagination.go`) **не** поджимает тихо значения: при `limit < 0` → **400** `limit must be positive`, при `limit > 100` → **400** `limit cannot exceed 100`, при `offset < 0` → **400** `offset must be positive`. То есть **нормализация на клиенте — не косметика**, а **защита от заведомо невалидного запроса** в production (плоский `limit=-1` / `offset=-5` сорвёт вызов).

**Правила (как в 10.2, семантика как `defaultLimit` / `maxLimit` на сервере):**

- `limit <= 0` → `20`
- `limit > 100` → `100`
- `offset < 0` → `0`

## HTTP: `Content-Type` на POST (критично)

В `Create` и `SendMessage` хендлер сравнивает `c.ContentType()` **строго** с `application/json`. Если Dio выставит `application/json; charset=utf-8`, сервер вернёт **415**.

**Обязательно:** для `createConversation`, `sendMessage` — `Options` с `contentType: 'application/json'` **без** charset.

**`DELETE`** без тела — отдельный `Options` не нужен.

## Дисамбигуация 404: проект vs чат

В JSON тело ошибки **одинаковое** по форме: `{"error":"not_found","message":"..."}` — **различать по HTTP-пути запроса**, а не по тексту `message` (хрупко).

| Методы | Path содержит | 404 означает | Исключение |
|--------|----------------|--------------|------------|
| `listConversations`, `createConversation` | `/projects/` … `/conversations` | проект не найден | **`ProjectNotFoundException`** — импорт из **`package:frontend/core/api/api_exceptions.dart`** (перенос в **предусловие**); **не** `ConversationNotFoundException` |
| `getConversation`, `deleteConversation`, `getMessages`, `sendMessage` | `/conversations/` | чат не найден | **`ConversationNotFoundException`** |

**11.4 / UX (ориентир):**

- `on ProjectNotFoundException` → «проект удалён или недоступен» → навигация к списку проектов.
- `on ConversationNotFoundException` → «чат удалён» → обновить список чатов / закрыть экран чата.

`_handleDioError` (в репозитории) получает из общего хелпера **`ApiErrorPayload`**, затем по **статус-коду** и **`requestPath`** решает, кинуть **`ProjectNotFoundException`** или **`ConversationNotFoundException`** (и т.д.). См. **предусловие** — общий слой **не** знает feature-типы.

### Порядок элементов в ответе (обязательный dartdoc на методе)

Цепочка бэкенда: `ConversationHandler.GetHistory` → `conversationService.GetHistory` передаёт в репозиторий **`MessageFilter` только с `Limit` и `Offset`** (`conversation_service.go`) — поля сортировки **не** задаются. ## Стабильные коды ответа API (`error`) и HTTP

Источник: `backend/pkg/apierror/apierror.go` — поле **`error`** в JSON — **стабильный машинный код**; поле **`message`** — для человека (**не использовать как ключ логики** — текст может меняться / локализоваться на бэкенде).

**Правило клиента:** маппинг ошибок — по **`statusCode`** (+ различение ресурса по **path** для 404); при разборе тела опционально заполнять **`apiErrorCode`** из JSON-ключа `error` для точечной обработки; **не** матчить по подстрокам **`message`**.

- В **`ConversationApiException`** — опциональное поле **`String? apiErrorCode`** (если в ответе нет стабильного кода — `null`).
- **`422`** → **`ConversationApiException`**, `statusCode: 422`, при наличии в теле — **`apiErrorCode`**.
- **`502`** + `error: external_service_error` — **`sendMessage`** / оркестратор → **`ConversationApiException`**, `statusCode: 502`, **`apiErrorCode`** при наличии.

**Прочие 4xx/5xx:** `ConversationApiException` с `statusCode`, санитизированным текстом, `originalError`, **`apiErrorCode` опционально**.

## Conversation-домен: таблица HTTP ↔ стабильный `error` ↔ источник

Маппинг централизован в **`backend/pkg/httputil/errors.go`** (`RespondError`) и частично прямым **`apierror.JSON`** в хендлере (см. 401 ниже). Для сценариев чата — ориентиры:

| HTTP | Поле `error` (стабильно) | Когда / источник (Go) |
|------|--------------------------|------------------------|
| `400` | `bad_request` | `ErrInvalidConversationTitle`, `ErrInvalidMessageContent` и др. валидация через `RespondError`; ручные 400 в хендлере (невалидный UUID, пустой title/content) — тот же стиль `apierror.ErrBadRequest` |
| `401` | **`access_denied`** | Прямой вызов в хендлере: `apierror.JSON(..., ErrAccessDenied, "Unauthorized")` — **не** `unauthorized` и **не** через `RespondError` (см. **§ 401**). |
| `403` | `forbidden` | `ErrConversationForbidden` |
| `404` | `not_found` | `ErrConversationNotFound` **или** `ErrProjectNotFound` (см. path) |
| `429` | `too_many_requests` | `ErrMessageRateLimit` |
| `500` | `internal_server_error` | Ветка `default` в `RespondError`: **message в JSON часто** `"Request failed"` — **без деталей**; не полагаться на содержательный `message`. |

**Дополнительно (редко в чате, но в общем `RespondError`):** `502` + `external_service_error`, `409` + `already_exists` / `conflict` — при появлении в ответах чата обрабатывать через `statusCode` + `apiErrorCode`.

## 401 и поле `error` (ловушка для матчинга по строке)

Все ошибки из сервиса обычно идут через **`httputil.RespondError`**, но **401 без пользователя** в conversation-хендлере отдаётся **напрямую** (`apierror.ErrAccessDenied`). В JSON приходит **`error: "access_denied"`**, а не `"unauthorized"`.

**Зафиксировать в реализации:** для неаутентифицированного пользователя ориентироваться на **`statusCode == 401`**, а не на конкретное значение `error`. Строковый матч `"unauthorized"` **не сработает**.

## Таблица HTTP-кодов и исключений (`_handleDioError`)

Все **тексты для пользователя** в исключениях — после **`sanitizeUserFacingMessage`** (общий хелпер из предусловия).

| Код | Когда | Исключение |
|-----|--------|------------|
| `400` | … | `ConversationApiException` |
| `401` | … | **`UnauthorizedException`** (матчить по **статус-коду**; в JSON часто `error: "access_denied"` — см. § выше); **`apiErrorCode`** из JSON |
| `403` | … | `ConversationForbiddenException` (**`apiErrorCode`** из JSON при наличии) |

- **`ConversationRateLimitedException`** — в `conversation_exceptions.dart` (рядом с репозиторием); поле **`apiErrorCode`**.
- Остальное: `ConversationRepositoryException`, `ConversationNotFoundException`, `ConversationForbiddenException`, `ConversationApiException` (поля **`apiErrorCode`** опционально; у **`ConversationApiException`** реализованы **`==`/`hashCode`** по `message` + `statusCode` + `apiErrorCode`, без `originalError`).
- **`ProjectNotFoundException`** — **`package:frontend/core/api/api_exceptions.dart`** (опционально **`apiErrorCode`**); не путать с `ConversationNotFoundException`.

Не пробрасывать `DioException` наружу.

### ⚠️ `getMessages`: offset + параллельные события WS

Эндпоинт истории — только **`limit` / `offset`** (хендлер не пробрасывает cursor). В репозитории сообщений на бэкенде **есть** cursor-логика для других сценариев (`ListByProjectID` с `lastID`), но **HTTP GetHistory** её **не** использует.

**Риск:** пока клиент увеличивает `offset` для подгрузки **старых** сообщений, в чат приходят **новые** сообщения (WebSocket / другой клиент) → смещается «окно» выборки → возможны **пропуски** или **дубликаты** при классической offset-пагинации.

- **11.3** это **не** исправляет (HTTP-only).
- **Митигация — 11.4:** дедупликация по **`message.id`** при слиянии REST-страниц с потоком WS; при необходимости перезапрос первой страницы после крупных вставок.
- **Перспектива:** cursor-режим в HTTP API (бэкенд) — отдельная задача; до тех пор поведение задокументировано здесь, чтобы **11.4** не лечила баг «в репозитории, потому что удобнее».
