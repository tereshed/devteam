## Контракт `send` / будущие вложения (**11.4** ↔ **11.5** ↔ **11.8**)

**Сейчас в коде (11.4):** `Future<void> send(String raw)` — **нормализация текста** (**`trim()`** и пустой ввод) — **только внутри контроллера**; пустая строка после trim — **no-op** (сеть не вызывается).

**UI (11.5 / 11.8):** достаточно **не вызывать** **`send`**, если **`text.trim().isEmpty`** (кнопка неактивна / ранний return). **Не** дублировать второй **`trim()`** «для отправки» — иначе создаётся впечатление двойной нормализации; в контроллер уходит **сырой** текст из поля, как сейчас в **`send(String raw)`**.

**`retrySend(String clientMessageId)`** — в **11.4**; повтор транзиентно упавшей отправки с тем же id и телом из **`pending`** — **не** дублировать логику на экране.

**Расширение под вложения (11.8):** ломать существующие вызовы **нельзя**. Целевой паттерн: расширить API в **11.4** именованными необязательными параметрами с дефолтами, например:

`Future<void> send(String raw, {List<Attachment> attachments = const []});`

— либо эквивалент, сохраняющий исходный сценарий «только текст». Экран **11.5** и виджет **11.8** вызывают **одну** точку входа; до появления типа **`Attachment`** параметр может быть заглушкой **`Never`** / пустым typedef в отдельном файле — зафиксировать в PR **11.8**, не в **11.5**.

## Цель

- **`Scaffold`** с **AppBar**: заголовок из **`state.conversation?.title`** при наличии; при **`conversation == null`** и отсутствии заголовка — **`l10n.chatScreenAppBarFallbackTitle`** (или согласованное имя из **11.10**; см. § предусловий). Скелетон в AppBar — вне scope **11.5**, если не оговорено иначе (согласуемо с **10.6**).
- **Список сообщений:**
  - Отображение **`ChatState.messages`** с учётом роли (**user** / **assistant** / **system**) — выравнивание/цвет **минимально**, детальная типографика — **11.6**.
  - **Низ чата = последние сообщения:** `ListView` с **`reverse: true`** **или** эквивалент — **один** выбранный подход в PR.
  - **Подгрузка старых сообщений (пагинация «вверх по времени»):**
    - При **`reverse: true`** нулевой offset в **`ScrollController`** — это **визуальный низ** (новейшие сообщения). **Старейшие** сообщения находятся у **визуального верха** прокрутки, что соответствует **`position.pixels`**, близким к **`position.maxScrollExtent`** (не к `0`).
    - **Зафиксировать в коде/PR:** триггер **`loadOlder()`** — условие вида **`pixels >= maxScrollExtent - threshold`**, и **`hasMoreOlder`**, и **`!isLoadingOlder`** (последнее — **явный гард**, чтобы не вызывать **`loadOlder()`** повторно, пока уже идёт загрузка страницы); debounce по необходимости.
    - **Неправильно:** использовать порог к **`pixels <= threshold`**, ожидая «верх списка» — при **`reverse: true`** это инвертирует поведение.
  - Индикатор **`isLoadingOlder`** в зоне **старейших** сообщений (например, первый item при `reverse: true`).
- **Поле ввода и отправка:**
  - Контроллер текста + кнопка отправки; если **`text.trim().isEmpty`** — **не** вызывать **`send`** (UX-гард; нормализация тела сообщения остаётся в контроллере — см. § **Контракт `send`**).
  - Вызов **`ref.read(chatControllerProvider(...).notifier).send(text)`** — см. § **Контракт `send`**.
- **`pendingByClientId` и ошибки отправки (логика в 11.4, отображение в 11.5):**
  - При **транзиентном** сбое **`send`** оставляет запись в **`pendingByClientId`** с **`lastError`** — см. **`PendingMessage`**.
  - При **фатальном** сбое pending для этой отправки **снимается**, ошибка пробрасывается — экран показывает **`AsyncError`** / SnackBar по политике PR.
  - **`retrySend(clientMessageId)`** — единственный канонический retry **той же** идемпотентной отправки; в UI для упавшего pending — кнопка «Повторить» / иконка, вызывающая **`retrySend`**. Поведение при **`ref.invalidate(chatControllerProvider(...))`**: провайдер пересобирается — **ожидается сброс pending** вместе с состоянием (новая сессия загрузки); **не** рассчитывать на сохранение pending через invalidate. Если нужен persist pending — отдельная задача на уровень контроллера/хранилища, **вне 11.5**.
  - Визуально: для pending — индикатор ожидания; при **`lastError != null`** — состояние ошибки + **retry** (не вечные «часы» без выхода).
- **Состояния `AsyncNotifier`:**
  - Первая загрузка: не «мигать» пустым экраном при частичном **`ChatState`** — см. **11.4**, § «Частичный state».
  - Ошибка: **`AsyncError`** + **Retry** через **`ref.invalidate(chatControllerProvider(...))`** (SSOT как **10.6**).
  - **`AsyncLoading` при уже имеющихся данных** (например после **`invalidate`** при **`@Riverpod(keepAlive: true)`**): **не** заменять список полноэкранным спиннером, если есть предыдущее значение. Паттерн как в **10.4** (stale data):

    ```dart
    final asyncChat = ref.watch(chatControllerProvider(...));
    final stale = asyncChat.value;
    final isFirstLoad = asyncChat.isLoading && stale == null;
    final isReloadWithData = asyncChat.isLoading && stale != null;
    ```

    При **`isReloadWithData`** — оставить тело списка, опционально лёгкий индикатор (linear progress / overlay), без флика.

  - Отдельная ветка для **`ConversationNotFoundException`**: до экрана списка бесед — центр / SnackBar + **l10n**; навигация назад по кнопке.
- **Адаптивность:** колонка «список + input»; на широком экране при необходимости **`Center` + `ConstrainedBox(maxWidth: …)`** (паттерн **10.4**). Учесть **`MediaQuery.textScalerOf(context)`** — длинные сообщения и поле ввода не обрезаются без прокрутки; избегать фиксированных высот в **dp**, ломающихся при крупном шрифте.

## UX-инвариант: auto-scroll к «низу» чата

**Низ** = зона новейших сообщений (при **`reverse: true`** это **`pixels → 0`**).

**Скроллить к низу**, когда выполнены **все** условия:

1. **Изменились данные**, из-за которых ожидается новая строка внизу: например, **`messages.length`** увеличился, или исчез/обновился релевантный **pending** (точный триггер зафиксировать в PR — минимум **рост длины списка** после успешной отправки пользователя и появление нового **assistant**-сообщения).
2. **Пользователь уже «приклеен к низу»** — иначе **не** дёргать скролл (читает историю).

**Гард «у низа»** (для **`reverse: true`**): считать у низа, если **`pixels <= bottomThreshold`** (порог 24–48 logical px или доля viewport — зафиксировать константой в одном месте). Если пользователь отмотал вверх (**`pixels > threshold`**), auto-scroll **не** выполнять.

**Реализация:** после кадра с новыми данными — **`WidgetsBinding.instance.addPostFrameCallback`** → **`animateTo(0, ...)`** или **`jumpTo(0)`** для **`reverse: true`** (ноль = низ списка). При первом открытии чата — один раз проскроллить к низу, когда **`isLoadingInitial`** стал false и список не пуст.

**Гонка с анимацией:** если во время **`animateTo(0, …)`** пользователь начинает скроллить вручную, анимация может продолжить тянуть список к низу. Слушать **`ScrollPosition.userScrollDirection`** (или уведомления **`ScrollController`**, эквивалентные «жест пользователя»): при **ручном** скролле **прервать** анимацию, например **`scrollController.jumpTo(scrollController.offset)`** (зафиксировать на текущем offset), чтобы не перехватывать управление у пользователя.

**Тест (11.5 или 11.11):** сценарий «добавили сообщение + у низа → offset близок к 0»; «пользователь отмотал вверх + новое сообщение → offset не меняется принудительно»; «транзиентный сбой → **`lastError`** в pending → **Retry** → успех, pending снят, сообщение в ленте» (мок **`ChatController`** / репозитория по паттерну проекта).

## Клавиатура (IME) и отправка с клавиатуры

- **`Scaffold.resizeToAvoidBottomInset`:** по умолчанию **`true`** на мобильных; для **desktop/web** при необходимости согласовать с дизайном (избежать «прыгающего» списка). **`SafeArea`** вокруг поля ввода — по правилам безопасных зон устройства.
- Список + поле: **каноничный** приём — **`Column`** с **`Expanded(child: ListView.reverse: true)`** (или без reverse — по выбранному в PR) **+** зона ввода под списком, чтобы при появлении IME список **сжимался** (`Expanded`), а не оставался под перекрытием клавиатуры без прокрутки. **`SliverFillRemaining`** с **`reverse: true`** на **`ListView`** давать **не** рекомендуется — легко получить хрупкую вёрстку; в спецификации **11.5** не предлагать как альтернативу.
- **Enter vs новая строка:**
  - **Mobile (soft keyboard):** одна кнопка отправки; Enter в однострочном поле может отправлять (если **`maxLines: 1`**).
  - **Desktop / Web:** многострочное поле (**`TextField` / `TextFormField`** с **`minLines` / `maxLines`**): **Enter** — новая строка; **Ctrl+Enter** / **Meta+Enter** — отправка. Для детерминизма и тестируемости предпочтительно **`Shortcuts` + `Actions`** с **`SingleActivator(LogicalKeyboardKey.enter, control: true)`** / **`meta: true`**, либо **`Focus(onKeyEvent: …)`** — **не** опираться на **`HardwareKeyboard.instance`** как на основной сигнал (глобальный синглтон, хуже в widget-тестах). Сочетание клавиш — зафиксировать одинаково в PR и в **11.8**. До **11.8** допустима только кнопка «Отправить» без shortcuts — тогда в PR явно: «shortcuts — в 11.8».

## Доступность (a11y)

- Пузырь / строка сообщения: оборачивать в **`Semantics`**, передавать **`label`** (краткий текст или первые N символов), **`hint`** при необходимости; для роли пользователя/ассистента — префикс в **`label`** (**«Вы: …»** / **«Ассистент: …»**) из **l10n**, без захардкоженных рус/анг строк вне `.arb`. В **11.5** — **`Semantics` без `liveRegion`** для обычных сообщений (статичное наполнение списка).
- **System**-сообщения — отдельная **`Semantics`**-разметка; **`liveRegion: true`** — только если контент по смыслу **единичный анонс** (не каждое системное сообщение подряд).
- **`liveRegion` для появляющихся новых сообщений** (в том числе **стрим текста ассистента** по **11.9**) — проектируется в **11.9** вместе с потоком обновлений контроллера, чтобы не дублировать объявления screen reader и не спамить при каждом чанке; **11.5** не обязан включать **`liveRegion`** под стрим до готового UX в **11.9**.
- После успешной отправки (или по **`onSubmitted`**) — при необходимости **`FocusScope.of(context).requestFocus(focusNode)`** на поле ввода для клавиатурной навигации.
- Проверить пороговые **`textScaler`** (например до **1.3–1.5**): переносы, не обрезать кнопки.

**Рекомендуемый вариант (а):** под маршрутом ветки **`chat`** — дочерний **`GoRoute`** **`:conversationId`**, **`pageBuilder`** → **`ChatScreen(projectId: id, conversationId: conv)`**. **Вариант (б) — `extra` без сегмента в URL:** `extra` **не** попадает в адресную строку и **теряется при полной перезагрузке страницы** и при копировании URL — **несовместимо** с инвариантом deep link и с реалтаймом **11.9** как единственным способом восстановить контекст. Разрешать только как **краткий внутренний** переход (например, из другого экрана приложения) с обязательным планом миграции на **(а)** до конца Sprint 11; в PR явно предупредить.

Комментарий в **`project_dashboard_routes.dart`** («Под-маршруты веток — Sprint 11+») обновить после появления дочернего маршрута.

## Архитектурные решения

| Аспект | Решение |
|--------|---------|
| Данные | `ref.watch(chatControllerProvider(projectId: …, conversationId: …))` |
| Ошибки в UI | **`chatErrorTitle` / `chatErrorDetail`** (**11.4**) + **l10n** для экранных подписей |
| Retry (не 404) | **`ref.invalidate(chatControllerProvider(...))`** |
| Retry транзиентной отправки | **`ref.read(...notifier).retrySend(clientMessageId)`** |
| Уход со страницы | **`keepAlive` / dispose** контроллера (**11.4**); отдельный **`CancelToken`** на экране для HTTP **не** вводить |
| Декомпозиция | Приватные виджеты в том же файле или вынесение, когда **`build` не читается**; не дублировать **11.6** / **11.8** |

### `ConsumerStatefulWidget` vs `ConsumerWidget`

Предпочтительно **`ConsumerStatefulWidget`** при **`ScrollController`**, **`TextEditingController`**, **`FocusNode`**. Если ввод полностью уезжает в **11.8**, **`ScrollController`** для списка обычно остаётся на экране — чаще всего всё равно **`ConsumerStatefulWidget`**.
