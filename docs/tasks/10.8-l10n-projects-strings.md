# 10.8 Локализация / аудит l10n для Sprint 10 (ru, en)

**Sprint:** 10 — Frontend: Проекты и навигация  

**Основные файлы:**
- `frontend/lib/l10n/app_ru.arb` — **template** для `flutter gen-l10n` (см. `frontend/l10n.yaml` → `template-arb-file: app_ru.arb`): отсюда берутся **имена геттеров**, **плейсхолдеры** и основная масса метаданных **`@…`** (в т.ч. `description` для переводчиков).
- `frontend/lib/l10n/app_en.arb` — английские строки; **каждый ключ сообщения** совпадает с template по множеству ключей.

**Косвенно затрагивается:**
- `frontend/lib/l10n/app_localizations.dart`, `app_localizations_ru.dart`, `app_localizations_en.dart` — **генерируются** `flutter gen-l10n` и в этом репозитории **коммитятся в git** (в `frontend/.gitignore` исключён **`lib/generated/`**, а не `lib/l10n/`). После любых правок **`.arb`** обязательно **`make frontend-codegen`** (или `flutter gen-l10n` в правильном порядке с `build_runner`) и **коммит обновлённого триплета** — иначе CI и чужие клоны увидят устаревшие геттеры. Канон формулировки: **`docs/rules/frontend.md`** (раздел i18n).
- `frontend/l10n.yaml` — `arb-dir`, `template-arb-file`, `output-dir` (поля **`synthetic-package` нет** — см. предусловия ниже)

**Зависит от:**
- **10.4** — тексты списка проектов, фильтры, статусы, ошибки загрузки ([10.4-projects-list-screen.md](./10.4-projects-list-screen.md))
- **10.5** — форма создания проекта: поля, валидация, провайдер Git ([10.5-create-project-screen.md](./10.5-create-project-screen.md))
- **10.6** — дашборд проекта: вкладки shell, 404 ([10.6-project-dashboard-screen.md](./10.6-project-dashboard-screen.md))

Если мердж не добавил строк — объём = **verify** по чек-листу ниже.

## Предусловия перед началом

1. **Template ARB и порядок добавления ключей** — в `frontend/l10n.yaml` задано **`template-arb-file: app_ru.arb`**. Генератор **не создаёт** геттер для ключа, который есть **только** в `app_en.arb` и отсутствует в template. Порядок работы при появлении **новой** строки:
   - новый ключ и блок **`@messageKey`** (плейсхолдеры, при необходимости `description`) — **сначала в `app_ru.arb`**;
   - затем тот же ключ — в **`app_en.arb`** с английским текстом; для строк с **`{placeholder}`** — **зеркальный `@messageKey`** с теми же **`placeholders.<name>.type`** (см. чек-лист).
   - после генерации проверить геттер в `app_localizations*.dart`.

   Смена **`template-arb-file`** с `app_ru.arb` на **`app_en.arb`** — **вне scope 10.8**: потребует переноса **всех** связанных `description` из template в новый template, иначе метаданные теряются без предупреждения от анализатора; оформлять отдельной задачей/PR, если продукт решит мигрировать.

2. **Импорт `package:frontend/l10n/...`** — в `l10n.yaml` **нет** `synthetic-package`. В актуальном Flutter для `gen-l10n` по умолчанию **non-synthetic**, импорт — **`package:frontend/l10n/app_localizations.dart`**.

3. **`requireAppLocalizations` и роутер** — задача **10.8** ограничивается **ARB и генерацией**. Паттерны вызова в роутере — **10.7**. В 10.8: поправить **ключи/метаданные**, если аудит выявил пробелы.

4. **Порядок codegen vs gen-l10n** — из правил репозитория: **`build_runner` не запускать после `gen-l10n` с `--delete-conflicting-outputs`**, иначе можно снести сгенерированные l10n-файлы. Тот же порядок обязан соблюдать **`make frontend-test`** в корневом `Makefile` (**`dart run build_runner`** → **`flutter gen-l10n`** → тесты). Для локальной работы предпочтительно **`make frontend-codegen`** из корня (см. «Команды»).

## Контекст и цель

В Sprint 10 в UI появляются экраны проектов, форма создания, дашборд и навигация. Любой текст, видимый пользователю, **не должен** быть захардкожен в Dart — только через **`AppLocalizations`**.

**Факт на момент актуализации описания (сверка по репозиторию):**
- Множества **ключей сообщений** в **`app_en.arb`** и **`app_ru.arb`** **совпадают** (порядок величины — **~164 ключа** в каждом; расхождение **числа строк файла** из `wc -l` не показательно для ARB).
- Инвентарные ключи для списка проектов, статусов, Git-провайдеров, создания проекта, дашборда и общих ошибок роутера **уже присутствуют** в обоих ARB.
- В **`frontend/lib/features/projects/`** типичный grep по пользовательским литералам в UI (`Text("…")`, `AppBar(title: Text("…"))`, `hintText:` / `labelText:` / `tooltip:`) **не находит** захардкоженных строк — фича уже опирается на **`AppLocalizations`**.

**Отсюда реальный объём 10.8:**
1. **Аудит после мерджа 10.4–10.7:** дельта — любые **новые** литералы/ключи, появившиеся в этих PR; для каждого — цикл template (`ru`) → `en` → **`make frontend-codegen`**.
2. **Verify по чек-листу** (паритет ключей, плейсхолдеры, `@meta` в `en` там, где нужно типам), плюс точечная правка **зеркальных `@key` для плейсхолдеров** в `app_en.arb` (см. ниже).
3. Политика **`@meta`:** в **`app_ru.arb`** — блоки `@…` с **`description`** (канон контекста для переводчиков). В **`app_en.arb`** — **только** то, что нужно генератору: **`@@locale`**, **`@messageKey.placeholders`** для строк с **`{…}`**; **не** вводить в `en` отдельные англоязычные `description`, не совпадающие по смыслу с template (двойной SSOT). Если продукт когда-нибудь потребует `description` в обоих файлах — только как **перевод** строки из `app_ru.arb` (явная норма в `docs/rules/frontend.md`).

**Цель задачи 10.8 (итоговая формулировка):**
- Убедиться, что после завершения **10.4–10.7** в ARB отражена **вся** пользовательская текстовая дельта и что генерация не даёт «разъехавшихся» сигнатур для строк с **`{…}`**.
- **`make frontend-codegen`** + **`make frontend-analyze`** без ошибок.

## Правила именования и структуры ARB

| Правило | Описание |
|--------|----------|
| **Префиксы** | Ключи проектов группировать осмысленно: `projects…`, `project…`, `createProject…`, `projectDashboard…`, `gitProvider…`, `status…`, `error…`. |
| **Плейсхолдеры** | Для каждого `{name}` в строке — в **обоих** ARB блок `"@messageKey": { "placeholders": { "name": { "type": "…" } } }` с **одинаковыми именами и типами**. Иначе `gen-l10n` может сгенерировать параметры с типом по умолчанию (**`Object`**) и получить неочевидные ошибки форматирования в рантайме. |
| **Метаданные `description`** | Канон контекста для переводчиков — **только** в **template (`app_ru.arb`)**. В **`app_en.arb`** не добавлять произвольные `description`; для строк с **`{name}`** обязателен **`@messageKey`** с теми же **`placeholders.<name>.type`**, что и в `ru`. |
| **Контекст UI** | В `description` (в `ru`) указывать *где* строка: AppBar, snackbar, empty state, validation, 404 body. |

## Дубликаты семантики и «мёртвые» ключи

**Пример:** `errorAccessDenied` vs `errorForbidden` — оба относятся к отказу в доступу (часто 403), но **по текущему коду оба используются:**
- **`errorAccessDenied`** — экраны **`login` / `register`** (auth);
- **`errorForbidden`** — **`projects_list_screen`**, **`create_project_controller`** (контекст проектов, отдельная формулировка).

Унификация в один ключ — **не обязательна** в 10.8; это продуктовый/UX выбор. В рамках 10.8 достаточно **не считать один из ключей мёртвым без `grep` по `lib/`**.

**Правило:** при сомнении поискать вхождения имени ключа в **`frontend/lib`**; если ссылок на геттер нет вне **`lib/l10n/*.arb`** и сгенерированных **`app_localizations*.dart`** — кандидат на удаление в отдельном мини-рефакторинге (согласовать с ревьюером).

**Готовый пример (дубли по смыслу 403 / «нет доступа»):**

```bash
rg -n '\b(errorForbidden|errorAccessDenied)\b' frontend/lib --type dart
```

Пример вывода (фрагмент; пути могут меняться):

```
frontend/lib/features/projects/presentation/controllers/create_project_controller.dart:18:    ProjectForbiddenException _ => l10n.errorForbidden,
frontend/lib/features/projects/presentation/screens/projects_list_screen.dart:234:      return l10n.errorForbidden;
frontend/lib/features/auth/presentation/screens/login_screen.dart:59:              message = l10n.errorAccessDenied;
frontend/lib/features/auth/presentation/screens/register_screen.dart:62:              message = l10n.errorAccessDenied;
```

## Инвентаризация ключей (ориентир по Sprint 10)

Таблица ниже — **справочник по областям**; на момент актуализации описания перечисленные ключи **уже есть** в обоих ARB. После мерджа веток **10.4–10.7** перепроверить **дельту** (новые вызовы `l10n.*`).

| Область | Примеры ключей |
|--------|----------------|
| Список проектов | `projectsTitle`, `createProject`, `searchProjectsHint`, `filterAll`, `clearFilters`, `noProjectsYet`, `noProjectsMatchFilter`, `errorLoadingProjects`, `errorUnauthorized`, `errorForbidden` |
| Auth (вне фичи projects; см. дубликаты 403) | `errorAccessDenied` — `login_screen` / `register_screen` |
| Статусы проекта | `statusActive`, `statusPaused`, `statusArchived`, `statusIndexing`, `statusIndexingFailed`, `statusReady`, `statusUnknown` |
| Git на карточке / форме | `gitProviderGithub`, `gitProviderGitlab`, `gitProviderBitbucket`, `gitProviderLocal`, `gitProviderUnknown` |
| Создание проекта | `createProjectScreenTitle`, `projectNameFieldLabel`, … , `createProjectErrorConflict`, `createProjectErrorGeneric` |
| Дашборд | `projectDashboardFallbackTitle`, `projectDashboardNotFoundTitle`, `projectDashboardChat`, … , `projectDashboardNotFoundBackToList` |
| Роутер / ошибки | `routerNavigationError`, `retry`, `cancel` |

Полный перечень ключей, реально вызываемых из кода фичи проектов:

```bash
rg -oN --no-filename -r '$1' 'l10n\.(\w+)' \
  frontend/lib/features/projects --type dart | sort -u
```

(или заменить путь на **`frontend/lib`** для обзора всего приложения). Без **`--no-filename`** (`-I`) при **`rg -o`** в вывод попадает префикс пути (`path:l10n.key`), **`sed`** с **`^l10n\.`** перестаёт стрипать имя геттера — список ключей искажается (дубликаты по файлам, ложная «мёртвость» ключей).

**Дифф множеств ключей сообщений (без `@`-метаданных)** — вручную или через **`make frontend-l10n-check`** (скрипт `scripts/check_l10n_parity.sh`). Эквивалент в shell:

```bash
cd frontend/lib/l10n
diff <(jq -r 'keys[] | select(test("^@") | not)' app_en.arb | sort) \
     <(jq -r 'keys[] | select(test("^@") | not)' app_ru.arb | sort)
```

Ожидаемо при приёмке — **пустой вывод**.

**Авто-проверка плейсхолдеров (из каталога `frontend/lib/l10n`):** для каждого ключа сообщения со строкой, содержащей `{name}`, в **обоих** направлениях должен существовать зеркальный **`@<ключ>.placeholders`** в парном файле (`ru`→`en` и `en`→`ru`). Ожидаемо — **пустой stderr** от проверки; любая строка = пропуск зеркала (для CI: падать при ненулевом коде выхода).

```bash
cd frontend/lib/l10n
jq -r 'to_entries[] | select(.key | test("^@") | not) | select(.value | type == "string") | select(.value | test("\\{[a-zA-Z_]")) | .key' app_ru.arb \
  | while read -r k; do
      jq -e --arg key "$k" '.["@" + $key].placeholders' app_en.arb >/dev/null 2>&1 \
        || echo "missing @${k} placeholders in app_en.arb"
    done
jq -r 'to_entries[] | select(.key | test("^@") | not) | select(.value | type == "string") | select(.value | test("\\{[a-zA-Z_]")) | .key' app_en.arb \
  | while read -r k; do
      jq -e --arg key "$k" '.["@" + $key].placeholders' app_ru.arb >/dev/null 2>&1 \
        || echo "missing @${k} placeholders in app_ru.arb"
    done
```

**Автоматизация (рекомендуется в CI и перед PR):** `make frontend-l10n-check` — `./scripts/check_l10n_parity.sh` (паритет ключей + оба прохода плейсхолдеров).

### Backlog (зафиксировано аудитом, вне закрытия 10.8)

- Секция **`prompts*`** в `app_ru.arb` без **`@…`** с `description` (в отличие от соседних блоков `apiKeys`, `mcpConfig`, `projects`) — **техдолг**: при следующем касании промптов добавить метаданные по тому же шаблону, что у остальных модулей.

### Пограничные случаи (бренд / системный title)

- Ключ **`appTitle`** используется в UI (например `landing_screen`). Поле **`MaterialApp.router(title: …)`** раньше дублировало бренд строкой; для согласованности с локалью и будущими переводами заголовка задачи в ОС используйте **`onGenerateTitle: (context) => AppLocalizations.of(context)!.appTitle`** (см. `lib/main.dart`).

## Архитектурные решения

### 1. Template locale (факт конфигурации)

**`app_ru.arb`** — **`template-arb-file`**. Новые строки: **сначала `ru`**, затем **`en`**. Ключ только в **`en`** → геттер **не** появится.

### 2. Единый источник для тестов (10.9)

Предпочитать **`AppLocalizations`** из **`MaterialApp`** с **`AppLocalizations.delegate`** и **`supportedLocales`**, а не голые литералы — иначе ломается как сопоставление с продом, так и **`requireAppLocalizations`**.

### 3. Не смешивать продуктовый бренд и задачу

Точечные правки вне сценариев **10.4–10.7** — только если выпадают из аудита плейсхолдеров/мертвых ключей.

## Команды

Из **корня репозитория** (канон по `CLAUDE.md` / `docs/rules/frontend.md`):

```bash
make frontend-codegen   # pub get + dart run build_runner + gen-l10n (правильный порядок)
make frontend-l10n-check   # паритет ключей ru/en + зеркало плейсхолдеров (./scripts/check_l10n_parity.sh)
make frontend-analyze
```

Точечно только l10n без codegen (безопасно **только** если в том же PR **не** менялись аннотации **`@riverpod` / `@freezed`** и прочий кодоген — иначе **`make frontend-codegen`**):

```bash
cd frontend && flutter gen-l10n
```

Первичная настройка окружения — **`make frontend-setup`** (включает `build_runner` + `gen-l10n` в корректном порядке).
