## Что входит в state machine

### 1. Статусы задачи (10 штук)

| Статус | Тип | Описание |
|--------|-----|----------|
| `pending` | начальный | Задача создана, ожидает planning |
| `planning` | рабочий | Агент планирует реализацию |
| `in_progress` | рабочий | Агент выполняет задачу |
| `review` | рабочий | Код на ревью |
| `changes_requested` | рабочий | Требуются доработки |
| `testing` | рабочий | Тестирование |
| `paused` | управляющий | Задача приостановлена (пользователем) |
| `completed` | **терминальный** | Задача завершена успешно |
| `failed` | **полутерминальный** | Задача провалена — агент остановлен, пайплайн прерван. Допускает **только ручной** retry → pending (пользователь явно перезапускает). Оркестратор **ОБЯЗАН** трактовать `failed` как полную остановку и не пытаться продолжить выполнение. |
| `cancelled` | **терминальный** | Задача отменена |

**Классификация терминальности:**
- **Терминальный** (`completed`, `cancelled`) — выходов нет, задача закрыта навсегда
- **Полутерминальный** (`failed`) — агент остановлен, пайплайн прерван, но пользователь может вручную перезапустить задачу через Resume (→ `pending`). Оркестратор при `failed` **останавливает** пайплайн и ожидает ручного вмешательства

### 4. Реализация

```go
var allowedTransitions = map[models.TaskStatus][]models.TaskStatus{
    models.TaskStatusPending:          {models.TaskStatusPlanning, models.TaskStatusCancelled},
    models.TaskStatusPlanning:         {models.TaskStatusInProgress, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusInProgress:       {models.TaskStatusReview, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusReview:           {models.TaskStatusTesting, models.TaskStatusChangesRequested, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusChangesRequested: {models.TaskStatusInProgress, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusTesting:          {models.TaskStatusCompleted, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusPaused:           {models.TaskStatusPending},
    models.TaskStatusFailed:           {models.TaskStatusPending},
    // completed, cancelled — терминальные, нет допустимых переходов
}

func canTransition(from, to models.TaskStatus) bool {
    targets, ok := allowedTransitions[from]
    if !ok {
        return false
    }
    for _, t := range targets {
        if t == to {
            return true
        }
    }
    return false
}
```

### 4.1. Логика Resume из paused: осознанный полный рестарт

> **Дизайн-решение:** Resume из `paused` **всегда** переводит задачу в `pending` (полный рестарт пайплайна), а **не** в предыдущий статус.

**Почему полный рестарт, а не продолжение:**

1. **Контекст устаревает.** Пока задача на паузе, кодовая база может измениться (другие задачи влили PR, обновились зависимости). Агент, продолжающий с середины, работает с устаревшим контекстом — это приведёт к конфликтам и ошибкам.
2. **Агент stateless.** Оркестратор (Sprint 6) при каждом переходе собирает контекст заново (RAG из векторки, текущее состояние репозитория). Продолжение с середины требовало бы сохранения промежуточного состояния агента (промпты, tool calls, частичные результаты), что значительно усложняет архитектуру.
3. **Безопасность.** Полный рестарт гарантирует, что агент пройдёт все этапы валидации (planning → review → testing) заново, а не пропустит их.

**Будущая оптимизация (Sprint 8+):** При необходимости можно добавить «умный resume» — оркестратор при повторном planning анализирует `artifacts` (частичные результаты предыдущего запуска) и решает, какие шаги можно пропустить. Это не требует изменения state machine — pending → planning будет работать как раньше, но внутри planning агент получит обогащённый контекст.

**Аналогично для `failed` → `pending`:** Та же логика — полный рестарт. Агент при повторном planning увидит предыдущий `error_message` и `artifacts` через контекст задачи и учтёт их при новой попытке.

### 4.2. Конкурентный доступ (Race Conditions)

> **⚠️ КРИТИЧНО для распределённой системы.** State machine в среде, где задачу одновременно обновляют пользователь (через API) и агент (через Orchestrator), подвержена состоянию гонки.

**Сценарий проблемы:** Пользователь нажимает «Cancel» в UI, а агент в ту же миллисекунду через `Transition()` переводит задачу в `completed`. Оба читают текущий статус `testing`, оба проверяют `canTransition()` → оба получают `true`. Кто обновит последним — тот и «выиграет», второе обновление потеряется.

**Решение: Optimistic Locking через `updated_at`.**

При обновлении статуса Repository **ОБЯЗАН** использовать атомарное условие:

```sql
UPDATE tasks
SET status = $new_status, updated_at = NOW()
WHERE id = $task_id AND status = $expected_old_status AND updated_at = $expected_updated_at
```

Если `affected_rows == 0` — значит состояние изменилось между чтением и записью → вернуть `ErrTaskConcurrentUpdate`.

**Реализация в GORM:**

```go
result := db.Model(&task).
    Where("id = ? AND status = ? AND updated_at = ?", task.ID, expectedStatus, task.UpdatedAt).
    Updates(map[string]interface{}{
        "status":     newStatus,
        "updated_at": time.Now(),
    })
if result.RowsAffected == 0 {
    return ErrTaskConcurrentUpdate
}
```

**Где реализовать:**
- `TaskRepository.Update()` (задача 3.1) — добавить optimistic lock condition
- `TaskService` — при `ErrTaskConcurrentUpdate` можно retry (1 попытка) или вернуть ошибку клиенту
- `TaskHandler` (3.5) — маппинг `ErrTaskConcurrentUpdate` → `409 Conflict`

**Новая ошибка:**
```go
var ErrTaskConcurrentUpdate = errors.New("task was modified concurrently, please retry")
```

### 5. Автоматические timestamps при переходах

| Переход в статус | Поле | Значение |
|-----------------|------|----------|
| `in_progress` | `started_at` | `time.Now()` (только если `nil` — первый раз) |
| `completed` | `completed_at` | `time.Now()` |
| `failed` | `completed_at` | `time.Now()` |
| `cancelled` | `completed_at` | `time.Now()` |
| `pending` (resume) | `completed_at` | `nil` (сброс при retry/resume) |

### 6. Пользовательские действия vs программные переходы

| Действие | Метод | Кто вызывает | ABAC |
|----------|-------|-------------|------|
| Pause | `TaskService.Pause()` | User через API | Да (userID + userRole) |
| Cancel | `TaskService.Cancel()` | User через API | Да |
| Resume | `TaskService.Resume()` | User через API | Да |
| Любой переход | `TaskService.Transition()` | OrchestratorService (Sprint 6) | Нет (внутренний) |

### 7. Ошибки state machine

| Ошибка | Когда | HTTP (через handler) |
|--------|-------|---------------------|
| `ErrTaskInvalidTransition` | `canTransition()` вернул `false` | 409 Conflict |
| `ErrTaskTerminalStatus` | Попытка перехода из `completed`/`cancelled` | 409 Conflict |
| `ErrTaskInvalidStatus` | Невалидное значение статуса (не из enum) | 400 Bad Request |
| `ErrTaskConcurrentUpdate` | Optimistic lock: задача изменена между чтением и записью | 409 Conflict |

## Где используется

- `TaskRepository.Update()` (3.1) — optimistic lock (`WHERE status = ? AND updated_at = ?`) для атомарного обновления
- `TaskService.Update()` (3.3) — если клиент меняет `status` через PUT, проверяется `canTransition`
- `TaskService.Pause()` / `Cancel()` / `Resume()` — пользовательские action-эндпоинты
- `TaskService.Transition()` — программный переход (Orchestrator, Sprint 6). При `failed` оркестратор **ОБЯЗАН** остановить пайплайн и не пытаться продолжить выполнение
- `TaskHandler` (3.5) — маппинг ошибок state machine в HTTP 409 (включая `ErrTaskConcurrentUpdate`)
