## Описание

Создать репозиторий для CRUD-операций с таблицей `conversation_messages` (сообщения внутри чата). Слой только для БД: интерфейс + приватная реализация, `context.Context`, GORM, sentinel-ошибки, безопасная пагинация.

Репозиторий должен поддерживать транзакции для атомарной записи сообщений, порожденных оркестратором или пользователем.

---

## Интерфейс

```go
type ConversationMessageRepository interface {
    // WithTx возвращает копию репозитория, использующую переданную транзакцию
    WithTx(tx *gorm.DB) ConversationMessageRepository

    Create(ctx context.Context, msg *models.ConversationMessage) error
    
    // GetByID требует conversationID для защиты от IDOR
    GetByID(ctx context.Context, conversationID, id uuid.UUID) (*models.ConversationMessage, error)
    
    // ListByConversationID возвращает сообщения конкретного чата с пагинацией
    ListByConversationID(ctx context.Context, conversationID uuid.UUID, filter MessageFilter) ([]*models.ConversationMessage, int64, error)
    
    // Update используется для обновления статуса сообщения или его контента (редко)
    Update(ctx context.Context, conversationID, id uuid.UUID, updates map[string]interface{}) error
    
    Delete(ctx context.Context, conversationID, id uuid.UUID) error
}
```

---

## Методы и Краевые случаи

**Общее правило (Early Return):** Все методы должны возвращать ошибку `ErrInvalidInput`, если на вход передан `nil` указатель или `uuid.Nil`.

### Create

Вставляет строку в `conversation_messages`. Обязательные поля: `conversation_id`, `role`, `content`.
- **Ошибки БД:** Мапить `foreign_key_violation` на `conversation_id` в `ErrConversationNotFound`.
- Проверять валидность `role` (user/assistant/system) через констрейнты БД или доменную логику.

### GetByID

Возвращает сообщение по `id` и `conversation_id`.
- Возвращает `ErrMessageNotFound`, если записи нет.

### ListByConversationID

Список сообщений чата. Обычно сортируется по `created_at` ASC.
- **Пагинация:** `Count` + `Find` с `Limit`/`Offset`.
  - **Важно:** `Count` должен выполняться на том же наборе фильтров (scope), что и основной запрос, но **до** применения `Limit` и `Offset`, чтобы вернуть общее количество записей, а не размер страницы.
- **Фильтрация:** Возможность фильтрации по `role`.
- **Безопасность (DoS):** `Limit` должен быть ограничен сверху (например, max 100). Если `Limit <= 0`, использовать дефолт (например, 20). `Offset` не может быть отрицательным.

### Update

Частичное обновление через `Updates(map[string]interface{})`.
- **Early Return:** Если `updates` nil или пустая мапа, метод должен возвращать `nil` (или `ErrInvalidInput`), не выполняя запрос в БД.
- **Защита данных:** Репозиторий обязан удалять ключи `id` и `conversation_id` из мапы `updates` перед вызовом GORM, чтобы предотвратить случайное или злонамеренное изменение системных полей.
- Проверка `RowsAffected == 0` → `ErrMessageNotFound`.

## Фильтры и пагинация

```go
type MessageFilter struct {
    Role     *models.MessageRole
    Limit    int
    Offset   int
    OrderBy  string
    OrderDir string
}
```

### Валидация и Безопасность

- **OrderBy Whitelist:** Поля сортировки должны проверяться по белому списку (например, только `created_at`). Прямая подстановка строк в `.Order()` запрещена (защита от SQL Injection).
- **OrderDir Whitelist:** Только `ASC` или `DESC`.
- **Пагинация:** Применять нормализацию `Limit` и `Offset` (см. раздел "Методы").

---

## Конвенции реализации

1. **GORM Scopes:** Использовать скоупы для фильтрации по роли.
2. **Централизация ошибок:** Использовать `ErrMessageNotFound`, `ErrConversationNotFound` из общего файла ошибок репозитория.
3. Во всех запросах использовать `r.db.WithContext(ctx)`.

## Требования к БД (Миграция 015)

Миграция `015_create_conversation_messages.sql` должна содержать:
- **Составной индекс** `(conversation_id, created_at)` для быстрой выборки истории чата с сортировкой по времени без использования `filesort`.
- Индекс на `role` (опционально, если планируется частая фильтрация).
