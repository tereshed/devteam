## Go-специфика: `-race` и `t.Parallel()` (обязательно)

- **Детектор гонок:** любая проверка пакета `gitprovider` в рамках задачи должна проходить с **`go test -race`**. В командах ниже и в привычке разработчика **` -race` не опускать**. Если корневой `Makefile` цель `test-unit` пока не прокидывает `-race`, в PR по 4.8 явно зафиксировать успешный прогон `go test -race ./pkg/gitprovider/...` (и при необходимости предложить доработку Makefile отдельным коммитом).
- **`t.Parallel()`:** в **каждом** `Test*` (первым осмысленным оператором после проверок `t.Helper()`/настройки, если есть) вызывать **`t.Parallel()`**. Для **`t.Run`** — внутри каждого подтеста, который может выполняться независимо, также вызывать **`t.Parallel()`** (типовой паттерн: `for _, tc := range cases { tc := tc; t.Run(tc.name, func(t *testing.T) { t.Parallel(); ... }) }`).
- **Зачем:** ускорение прогона и **раннее выявление** общего мутабельного состояния (глобалы, один сервер на пакет без изоляции). Если параллельный запуск ломает тест — исправлять изоляцию (свой `httptest.Server` и свой провайдер на тест), а не убирать `t.Parallel()` без причины.
- **Исключения:** допускается **не** параллелить конкретный подтест только с **комментарием в коде**, почему он обязан быть последовательным (крайне редкие случаи); молчаливое отсутствие `t.Parallel()` не считается нормой.

## КРИТИЧНО: граница «юнит» vs «интеграция»

**Запрещено** вызывать реальный бинарник **`git`**, полагаться на `PATH`, плодить дочерние процессы и трогать реальную ФС в тестах, которые входят в **`make test-unit`**. Такие тесты по определению **интеграционные**: они медленные, хрупкие к окружению и версии git, дают flaky.

**Нужно выбрать ровно одну стратегию для локальной (CLI) части** (смешивать нельзя):

### Стратегия 1 — «Честные юнит-тесты» (предпочтительно для `test-unit`)

- Вынести запуск git за **абстракцию** (например интерфейс `GitCommandRunner` / `CommandExecutor` в пакете `gitprovider`): метод в духе «выполнить `git` с аргументами и `workDir`, вернуть stdout/stderr/код», реализация по умолчанию — текущий `exec.CommandContext`.
- Прод-код (`LocalGitCLI`, `GitHubProvider.Clone` / `Push`, `gitExec`, `GetLocalDiff` и т.д.) получает реализацию через **конструктор / поле структуры** (DI), без глобальных переменных и без `init()`.
- В **`github_test.go`** подменять mock’ом: проверять **точные срезы аргументов**, наличие разделителя **`--`** перед пользовательскими фрагментами (ветка, путь, ref), проброс `context.Context` до раннера (чтобы отмена контекста не игнорировалась).
- **Блокер:** без этой абстракции нельзя честно покрыть CLI-логику в юнит-слое; «опционально» не считается, если цель — `test-unit` без git.
- **Кто тестирует раннер (обязательно при стратегии 1):** моки покрывают **использование** интерфейса, но **прод-реализация** раннера (обёртка над `exec.CommandContext`) сама по себе не должна оставаться без дыма. Требование: **минимум один** интеграционный тест с **`//go:build integration`**, который вызывает **реальную** реализацию `GitCommandRunner` (например `git version` или `git rev-parse` в `t.TempDir()`), проверяет **ненулевой stdout**, корректный код выхода и отсутствие регресса при чтении stderr. Так доказывается, что бинарник из `PATH` реально вызывается и контракт раннера соблюдён.

### Стратегия 2 — «Реальный git только в integration»

- **Не** рефакторить под раннер: оставить прямые вызовы `exec.CommandContext`.
- Все сценарии с настоящим `git` и `t.TempDir()` перенести в отдельный файл с **первой строкой** `//go:build integration`.
- Запуск **только** через **`make test-integration`** (см. `.cursor/rules/backend.mdc`).
- В **`make test-unit`** для локальных методов остаются либо **отсутствие тестов** (допустимо только если явно зафиксировано в задаче/PR), либо тесты **чистой** логики без subprocess (валидации, сборка URL, если вынесены в отдельные функции).

**Итог:** в документе и в PR **недопустимо** утверждение «юнит-тесты» для вызовов реального git внутри `github_test.go` без тега `integration`.

## DRY и структура файлов (обязательно)

- **HTTP:** один переиспользуемый хелпер без «торчащих кишок»:  
  **`setupMockGitHubServer(t *testing.T, h http.Handler) (*GitHubProvider, func())`**  
  где `func()` — cleanup (закрытие сервера). Внутри хелпера создаётся `*github.Client` с `BaseURL` / `UploadURL` и **встраивается** в возвращаемый `*GitHubProvider`; **не** возвращать `*github.Client` наружу — иначе провоцируется изменение клиента в обход провайдера.
- **Маппинг ошибок GitHub → sentinel:** только **table-driven** тесты (`[]struct{ name, status, body, headers, wantErr }`), без копипасты пятнадцати отдельных `Test*` с одним и тем же бойлерплейтом.
- **`parseRepoURL`:** table-driven (уже требовалось ранее; без изменений по духу).
- Локальные юнит-тесты при **стратегии 1:** общий **mock runner** с записью последних `args` и опционально «задержка / отмена» для контекста.

Цель — удержать читаемый объём файлов, а не «свалку на 2000 строк».

## Часть A: Remote API — `httptest` + `go-github` (только `test-unit`)

- Провайдер получается из **`setupMockGitHubServer`** (или эквивалента), где под капотом собран `*github.Client` с `BaseURL` на мок-сервер.
- В каждом тесте и независимом подтесте — **`t.Parallel()`** (см. раздел выше).
- Сценарии по смыслу: `ValidateAccess`, `GetRepoInfo`, ветки (включая пагинацию и `maxBranchPages`), `GetDiff` + заголовок `Accept: application/vnd.github.v3.diff`, PR CRUD, файлы PR, комментарии (issue + review + сортировка), `SubmitPRReview` (невалидный event без HTTP), `MergePullRequest`, `GetFileContent`.
- **Маппинг `mapGitHubError` / `mapUnprocessableEntity` / rate limit:** обязательно **единая таблица** кейсов (401, 403 + `X-RateLimit-Remaining: 0`, 404 с разными `notFoundErr`, 409, 422 с сообщениями, 429, `RateLimitError` / `AbuseRateLimitError` из go-github, если воспроизводимы через ответ сервера). Актуальное поведение сверять с **`github_helpers.go`**, а не с устаревшими фрагментами спецификации 4.3.
- **Медленный Remote API и контекст:** обязательный тест на **`context.DeadlineExceeded`** (или эквивалентную отмену по таймауту): хендлер мок-сервера делает **`time.Sleep`** дольше, чем дедлайн контекста (например sleep **2s**, контекст **`100 * time.Millisecond`**), вызов любого remote-метода провайдера с этим контекстом должен завершиться ошибкой, связанной с истечением срока / отменой, **без** зависания теста. Это проверяет, что `go-github` + переданный `ctx` не игнорируются.

## Часть C: Безопасность (обязательные требования к тестам)

### C.1 Command / flag injection (git CLI)

- Тесты должны **документировать и проверять**, что пользовательские строки (**имя ветки**, **ref**, **части пути**, **имя remote**, элементы `opts.Files` при commit) **не** могут быть восприняты git как флаги.
- Обязательные негативные примеры (как минимум в таблице или подтестах): ветки в духе `-h`, `--upload-pack=evil`, сочетания с `--`.
- **Ассерт:** в переданных в раннер (mock) аргументах после фиксированных подкоманд везде, где это требуется протоколом git, присутствует разделитель **`--`** непосредственно перед подставляемыми значениями (как уже сделано для `checkout -b`, `branch -D`, `remote get-url` в текущем коде — тесты должны это **зафиксировать** и не допустить регрессии).
- Если в каком-то вызове **`--` отсутствует** (например `rev-parse --verify <ref>` без завершающего `--` перед ref), в рамках задачи **исправить** вызов до канонического вида (`--` перед пользовательским ref) **и** добавить тест.

### C.1.1 Git config injection (`git -c ...`)

- Разделитель **`--`** **не** защищает от инъекции через **ранние** глобальные флаги git, например **`git -c core.sshCommand=...`** subcommand — пользовательский ввод **не должен** оказываться в позиции, где git интерпретирует его как **ключ или значение `-c`** или как **имя подкоманды до фиксированной цепочки**.
- **Требование к тестам (mock раннера):** убедиться, что сформированный argv начинается с **`git`**, далее — **только фиксированные** литералы и заранее разрешённые флаги продукта; **любой** пользовательский фрагмент — **только после ` -- `** в конце цепочки (или после фиксированного набора позиционных аргументов, где это безопасно по контракту git). Регрессия: появление `fmt.Sprintf("git %s", userInput)` или склейки user input **до** `--` — недопустима.

### C.2 Path traversal и границы репозитория

- Для **`GetLocalFileContent`** и **`GetLocalDiff`**: сценарии с путями вида `../../../etc/passwd`, `..\\..\\`, `ref:../../../` и т.п.
- **Ожидание:** до вызова git выполняется **валидация**; опасные спецификации отклоняются **явной ошибкой** (sentinel или `fmt.Errorf` с стабильным смыслом), **без** запуска subprocess с этими строками.  
- Если в текущем коде валидации нет — **входит в объём 4.8** добавить её и покрыть тестами (юнит при стратегии 1, или integration + чистые функции валидации в unit — комбинация допустима).

### C.3 Symlink attacks (чтение содержимого через git)

- Риск: в рабочем дереве путь вроде `config.json` может быть **симлинком** на файл вне ожидаемого контекста; операции в духе «прочитать blob по пути» могут при определённой конфигурации и версии git вести себя неочевидно.
- **Требование:** либо **интеграционный** тест (или юнит над политикой путей + мок), который воспроизводит репозиторий, где целевой путь — **symlink**, и фиксирует **ожидаемую** политику (отказ / безопасное разрешение только внутри репо); либо **краткое обоснование в комментарии к тесту**, подкреплённое проверкой: например, что используется только **bare**-клон без произвольного worktree, а чтение идёт через `cat-file` с валидацией пути — и тест доказывает отсутствие обхода. Молчаливо игнорировать тему нельзя.

### C.4 Утечка токена (расширение относительно только «сырого» токена)

- Уже есть `sanitizeToken` + `userinfoEncodedPassword`. Тесты обязаны проверять, что в строках ошибок **нет**:
  - сырого токена;
  - **userinfo-encoded** формы (как при `url.UserPassword("x-access-token", token)` — см. `helpers_test.go`);
  - при появлении в stderr **других** канонических представлений токена (например фрагментов с `QueryEscape` / процентным кодированием, если такие кейсы реально воспроизводятся для выбранного сценария) — их тоже маскируют или тест фиксирует ожидаемое безопасное поведение.
- Минимум один тест на **`Clone`** / **`gitExec`** с падением команды и непустым stderr, где в URL/сообщении фигурирует токен.

## Часть D: Краевые случаи, контекст, ресурсы

### D.1 Пустые и невалидные входы

- Явно перечислить и покрыть **быстрый отказ** (без лишнего I/O): пустой `repoURL`, пустая ветка там, где это ошибка (в т.ч. `Push` / `validatePushBranch`), пустой `workDir` где применимо, пустой `path` для контента файла, пустой `remote` для push при необходимости, `MergePullRequest` с пустым `SHA` (уже есть — оставить в таблице).
- Для указателей в опциях: где `nil` допустим по контракту (partial update) — не требовать ошибки; где передача «обязательного» указателя критична — отдельные кейсы.
- **`context`:** передавать только неотменённые контексты в «успешных» путях; для отмены и таймаутов — отдельно (ниже и в части A для HTTP).

### D.2 Отмена контекста и зомби-процессы (локальный CLI / раннер)

- Тесты обязаны проверять, что при **`context.Canceled` / `context.DeadlineExceeded`**:
  - либо mock runner получает отменённый контекст и **не** имитирует успех;
  - либо (integration + реальный git) дочерний процесс завершается, **нет** зависаний и накопления зомби (по возможности CI: таймаут теста + проверка `Wait` / отсутствие «висящих» `git`).
- Для **`readCloserWithWait`** (`GetLocalDiff` и любые пути с `StdoutPipe` + `Wait`): тест на сценарий **закрытия `ReadCloser` без полного чтения stdout** — `Close()` должен корректно завершить ожидание процесса (порядок close/wait как в комментарии в `helpers.go` — без deadlock). При стратегии 1 допускается mock процесса / раннера, имитирующий блокировку и отмену.

### D.3 Стриминг больших тел (`GetDiff`, `GetFileContent`, локальный diff) — не читать всё в память

- Методы возвращают **`io.ReadCloser`**; тест обязан дать уверенность, что реализация **стримит**, а не буферизует гигантский ответ целиком в `[]byte` перед возвратом ридера.
- **Сценарий:** мок HTTP-сервера (для remote) или мок раннера/тела ответа отдаёт поток **неограниченного** или **очень большого** размера (например `io.Reader` без конца или Content-Length ≫ памяти тестового лимита). Клиент провайдера читает **не более первых 100 байт**, затем вызывает **`Close()`**.
- **Критерии успеха:** тест завершается без зависания; соединение/ридер освобождается (на стороне сервера фиксируется отмена записи или закрытие); опционально — **`testing.AllocsPerRun`** или сравнение аллокаций с наивным `io.ReadAll` на том же источнике, чтобы порядок величины не совпадал с «прочитать всё»; при появлении в модуле **`go.uber.org/goleak`** — `defer goleak.VerifyNone(t)` для подпакета/теста. Без обязательной зависимости: достаточно явной синхронизации (сервер сообщил о `Close` клиента) + `-race`.

## Критерии приёмки

- [x] Нет противоречия: **либо** стратегия 1 (раннер + моки, `test-unit` без git), **либо** стратегия 2 (реальный git **только** в `//go:build integration` + `make test-integration`).
- [x] При **стратегии 1:** минимум **один** integration-тест на **реальную** реализацию `GitCommandRunner` (см. выше).
- [x] **`-race`:** `go test -race ./pkg/gitprovider/...` проходит для юнит-слоя.
- [x] **`t.Parallel()`** во всех `Test*` и независимых подтестах (с обоснованными исключениями в комментариях).
- [x] Remote: `httptest`, хелпер **`setupMockGitHubServer(t, h) (*GitHubProvider, cleanup)`** без экспорта `*github.Client`.
- [x] Маппинг ошибок GitHub: **table-driven**; тест **таймаута контекста** на медленном HTTP-хендлере.
- [x] `parseRepoURL`: table-driven.
- [x] Безопасность: **flag injection** + **`--`**, **git `-c` injection**, **path traversal**, **symlink** (тест или доказанный безопасный кейс), **маскировка токена** (сырой + encoded).
- [x] **Стриминг:** частичное чтение + `Close()` без OOM/зависаний (часть D.3).
- [x] Краевые случаи: пустые строки; отмена контекста (local + remote); `readCloserWithWait`.
- [x] **Покрытие:** не ниже **80%** по пакету **`gitprovider`** (`go test -coverprofile=...`, см. команды ниже). Если отдельные файлы принципиально не покрываются в рамках 4.8 — явно перечислить в PR исключения и обосновать; по умолчанию порог относится ко всему пакету.
- [x] `make test-unit` проходит; при наличии integration-файла — `make test-integration` проходит в CI/локально.
