# 5.1 Dockerfile: образ `devteam/sandbox-claude`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)
**Файл:** `deployment/sandbox/claude/Dockerfile`
**Образ:** `devteam/sandbox-claude` (тег по политике CI, например `:latest` / semver)
**Статус:** ✅ Выполнена

---

## Описание

Подготовить **Dockerfile** для изолированного sandbox-контейнера, в котором Developer-агент выполняет задачи через **Claude Code CLI**. Базовый стек:

- **Node.js** (LTS, совместимый с официальным пакетом `@anthropic-ai/claude-code`)
- **Git** — клонирование репозитория, ветки, коммиты, сбор diff
- **Claude Code CLI** — глобальная установка через `npm` с **зафиксированной версией** пакета

Образ **не содержит** бизнес-логики пайплайна задачи: сценарий «clone → branch → агент → артефакты» выносится в **entrypoint** (задача **5.2**). В рамках **5.1** собирается воспроизводимый слой ОС + runtime + CLI; финальный `ENTRYPOINT` на `entrypoint.sh` — после **5.2**. До этого момента в Dockerfile **обязан** быть явный дефолт процесса (`CMD` или временный `ENTRYPOINT`), чтобы образ не был «пустым» для `docker run` (см. раздел «Разделение 5.1 и 5.2»).

## Архитектура и DRY (базовый образ и `sandbox-aider`)

В `main.mdc` запланирован отдельный образ **`devteam/sandbox-aider`**. С `devteam/sandbox-claude` у них пересекается фундамент: **`git`**, **`ca-certificates`**, при необходимости **`curl`**, каталог **`/workspace`**, непривилегированный пользователь, минимизация слоёв.

- **Если** вводится общий **`devteam/sandbox-base`** (multi-stage `FROM` / `COPY --from`): системные пакеты и создание пользователя **`sandbox`** выносятся в базовый образ; в `Dockerfile` Claude остаётся только Node + pinned `claude-code`.
- **Если** базового образа пока нет — дублирование между Claude и Aider допустимо на время MVP, но при добавлении второго sandbox-образа приоритет — вынести общий слой, чтобы не расходиться по версиям `ca-certificates`/user/`/workspace`.

### Разделение 5.1 и 5.2

**Не мержить** в `main` Dockerfile, после `docker run` который завершается с ошибкой из‑за отсутствия команды по умолчанию (неопределённое поведение по умолчанию = плохая практика).

Пока **нет** `entrypoint.sh` из **5.2**:

- задать **явно** **`CMD ["/bin/bash"]`** или **`CMD ["node"]`** (или эквивалент с понятным дефолтом), **либо** минимальный stub-`ENTRYPOINT`, согласованный с командой;
- после **5.2**: `COPY entrypoint.sh`, `chmod +x`, `ENTRYPOINT ["/entrypoint.sh"]` — **до** **`USER sandbox`** (или `COPY --chown` + `chmod` от root), затем при необходимости убрать/заменить временный `CMD`.

Так образ остаётся **консистентным**: сборка валидна, контейнер по умолчанию не «ломается молча».

## Требования к Dockerfile

### Базовый образ

- Предпочтительно официальный **`node:<major>-slim`** (например `bookworm-slim`), чтобы уменьшить размер и поверхность атаки.
- Явно зафиксировать мажорную версию Node (например `20`), а не `node:latest`.

### Системные пакеты

Обязательные пакеты (минимум):

| Пакет | Зачем |
|-------|--------|
| **`git`** | clone, ветки, diff, коммиты |
| **`ca-certificates`** | TLS для `git clone https://...`, `curl` к внешним API; без него типичны ошибки SSL |
| **`curl`** | по концепции sandbox в репозитории (проверки, скрипты); не добавлять пакеты «на всякий случай» |

Команда установки:

- **`apt-get install` с флагом `--no-install-recommends`**, иначе Debian-производные подтянут лишние рекомендуемые пакеты и **раздувают образ** на сотни мегабайт.
- Один слой вида:  
  `apt-get update && apt-get install -y --no-install-recommends git ca-certificates curl && rm -rf /var/lib/apt/lists/*`

### Непривилегированный пользователь (обязательно)

Sandbox выполняет **сгенерированный LLM-код**. Запуск основного процесса от **`root`** недопустим (лишние риски при эксплуатации уязвимостей, утечках и т.п.).

**Требования:**

1. Создать пользователя (имя по конвенции проекта, например **`sandbox`**). В Debian-based slim-образах **`useradd` без флагов не создаёт домашний каталог** — это **критичный краевой случай** для рантайма: **`git`**, **`npm`** и **Claude Code** пишут состояние в домашний каталог (`~/.gitconfig`, `~/.npm`, `~/.claude` и т.д.). Если каталога нет или **`HOME`** указывает на каталог без прав записи для **`sandbox`** (например на `/`), процесс упадёт с **`EACCES: permission denied`** в середине задачи.  
   **Обязательно:** создать пользователя **с домашней директорией**, например:  
   `useradd --create-home --home-dir /home/sandbox --shell /bin/bash sandbox`  
   (эквивалент часто пишут как `useradd -m -d /home/sandbox -s /bin/bash sandbox`). При использовании **`adduser`** — убедиться, что дом создаётся и путь совпадает с ожидаемым **`HOME`**.
2. Создать и выдать права на **`/workspace`**: на этапе сборки (ещё под `root`) выполнить `chown` (и при необходимости `chmod`) так, чтобы пользователь **`sandbox`** мог писать в рабочий каталог.

   **Внимание (volume mounts):** если оркестратор (**5.5**) при **`docker run`** монтирует каталог хоста в **`/workspace`** (например **`-v /host/path:/workspace`**), права и владелец, заданные в образе через **`chown` в Dockerfile**, **переопределяются** свойствами смонтированной директории на хосте (другой UID/GID, **`root`**, права только на чтение и т.д.). Тогда пользователь **`sandbox`** в контейнере может получить **`Permission denied`** при записи в **`/workspace`**, хотя образ собран «правильно». Полностью это **не** решается в **5.1** (это зона **5.2** — `entrypoint.sh`, и/или **5.5** — выбор UID/GID, user namespace, `chown` при старте, политика монтирования). В спецификациях **5.2** и **5.5** нужно явно предусмотреть согласование прав на смонтированный **`/workspace`** с UID пользователя **`sandbox`**.

3. После установки пакетов и глобального `npm`, после копирования и прав на **`entrypoint.sh`** (см. ниже) — директива **`USER sandbox`** (далее в спеке — **`sandbox`**). Docker подставляет **`HOME`** из `/etc/passwd` для **`USER`**; при сомнениях допустимо явно задать **`ENV HOME=/home/sandbox`** до **`USER`** для детерминизма.
4. Всё, что требует привилегий **root** (`apt`, глобальный **`npm -g`**, **`chown`**, **`chmod`** на файлы root), выполняется **до** **`USER sandbox`**. Инструкции только для непривилегированного режима — **после** **`USER sandbox`**.

Глобальная установка npm выполняется от **root**; затем **`USER sandbox`** — CLI остаётся в **`PATH`** и доступен для исполнения этим пользователем.

### Порядок слоёв и кэш Docker (CI/CD, локальные пересборки)

Чтобы при смене только версии **`claude-code`** не инвалидировать кэш слоя с тяжёлым **`apt-get`**, рекомендуется **жёсткий порядок**:

1. Слой(и) с **`apt-get update` / `apt-get install`** (системные пакеты меняются редко).
2. Затем создание пользователя **`sandbox`**, **`WORKDIR`**, **`chown`** на **`/workspace`** (если не перенесено в отдельный базовый образ).
3. Затем слой с **`npm install -g @anthropic-ai/claude-code@<version>`** и очисткой кэша/**`/tmp`** — этот слой пересобирается чаще всего при бампе версии CLI.

Так **Docker** и **BuildKit** чаще переиспользуют нижние слои в CI и на разных ОС при одинаковом контексте до изменения строки с версией npm-пакета.

### Claude Code CLI

- Установка: **`npm install -g @anthropic-ai/claude-code@<version>`** — версия **обязана быть зафиксирована** (например `0.2.37` или диапазон, согласованный с командой: `0.2.x` допустим только если в CI закреплён lock/конкретный минор). **Запрещено** `npm install -g @anthropic-ai/claude-code` без версии: мажорное обновление CLI может сломать флаги и **5.2** (`entrypoint.sh`) без изменений в репозитории.
- В **том же** `RUN`, сразу после установки: **`npm cache clean --force`**, затем очистка типичного мусора npm в **`/tmp`** (логи, временные файлы), чтобы слой оставался компактным.

  **`find /tmp ... -delete`** не рекомендуется как единственный вариант: на вложенных каталогах и в зависимости от версии **`find`** и порядка обхода возможны ошибки в духе **Directory not empty**. Надёжнее выполнить рекурсивное удаление содержимого через **`bash`**, чтобы корректно обработать **пустой** `/tmp`, скрытые имена и не ронять сборку из‑за неудаляемого сокета (на CI/разных хостах поведение может отличаться).

  Пример финала того же **`RUN`** ( **`|| true` здесь допустим** только для шага очистки мусора: цель — не провалить **`docker build`**, если отдельный файл в **`/tmp`** занят или не удаляется; это **не** относится к проверочным командам в разделе «Проверка локально»):

  `... && npm cache clean --force && bash -c 'rm -rf /tmp/* /tmp/.[!.]* /tmp/..?* 2>/dev/null || true'`

  Альтернатива проще: **`bash -c 'rm -rf /tmp/*'`** — если в образе **`/tmp`** при сборке не пуст и glob раскрывается; при полностью пустом **`/tmp`** возможна ошибка из‑за **dash**/glob (см. выше) — тогда предпочтительна расширенная форма с **`|| true`** на шаге очистки.
- Секреты и API-ключи **не** вшивать в образ; передача только через env при `docker run` (оркестратор, 5.5+).

### Рабочая директория

- `WORKDIR /workspace` — согласовано с Sandbox в `main.mdc`.

### Точка входа (после 5.2) и порядок относительно `USER`

**Краевой случай (права):** если **`USER sandbox`** указан **выше**, чем **`COPY` / `RUN chmod`** для `entrypoint.sh`, файл по умолчанию будет **`root:root`**, а **`RUN chmod +x`** от **`sandbox`** завершится **`permission denied`**.

**Обязательно один из вариантов:**

1. **`COPY` и `chmod +x` выполнять, пока активен `root`**, и только **после** этого — **`USER sandbox`**, **или**
2. **`COPY --chown=sandbox:sandbox entrypoint.sh /entrypoint.sh`** (и при необходимости **`chmod`** всё ещё от **`root`**, если бит исполняемости не сохраняется из контекста сборки — зависит от git/fs; надёжно: **`RUN chmod +x`** от **`root`** после **`COPY --chown`**).

Пример (фрагмент; порядок относительно **`USER`** соблюсти):

```dockerfile
COPY entrypoint.sh /entrypoint.sh
RUN chmod +x /entrypoint.sh
ENTRYPOINT ["/entrypoint.sh"]
```

Исполняемая форма **`ENTRYPOINT`** предпочтительнее shell-формы, если скрипт совместим с `exec`-стилем.

## Безопасность и эксплуатация

- **Обязательно:** пользователь **`sandbox`** с **домашней директорией** (см. выше) + **`USER sandbox`**; права на **`/workspace`** через **`chown`** на этапе сборки; **`entrypoint.sh`**: **`COPY`/`chmod`** от **`root`** или **`COPY --chown`**, не после **`USER sandbox`** без явного владельца.
- **Монтирование:** при **`-v …:/workspace`** права из образа не гарантируют запись **`sandbox`** — см. примечание выше; доработки в **5.2** / **5.5**.
- Не копировать в образ `.env`, токены, `docker.sock`.
- Образ для **изолированного** выполнения задач; сетевые политики задаёт хост/orchestrator (Sprint 5 в README).

## Проверка локально

Команды должны **завершаться с ненулевым кодом**, если бинарь отсутствует или сломан — **без** `|| true` и без маскировки ошибок (**исключение:** шаг очистки **`/tmp`** внутри **`Dockerfile`** — см. раздел «Claude Code CLI», там допустимо **`|| true`** только для этого шага).

После сборки (до **5.12** — вручную):

```bash
docker build -t devteam/sandbox-claude -f deployment/sandbox/claude/Dockerfile deployment/sandbox/claude
docker run --rm --entrypoint "" devteam/sandbox-claude node -v
docker run --rm --entrypoint "" devteam/sandbox-claude git --version
docker run --rm --entrypoint "" devteam/sandbox-claude claude --version
```

Команда **`claude --version`** уже проверяет наличие бинаря в **`PATH`** и возможность запуска; отдельный **`command -v claude`** не нужен.

`--entrypoint ""` отключает `ENTRYPOINT` из образа, чтобы проверять установленные бинари напрямую (после **5.2** entrypoint иначе перехватил бы команду).

После появления **5.12** — использовать цель `make sandbox-build` (или имя из принятого Makefile).
