# 9.6 Хук: индексация кода при создании проекта (после clone)

**Sprint:** 9 — Векторная индексация проекта (Backend)
**Файлы:**
- `backend/internal/service/project_service.go`
- `backend/internal/service/project_service_test.go`
**Статус:** ⬜ К выполнению

---

## Описание

Задача заключается в интеграции механизма индексации кода в процесс создания проекта. В данный момент метод `cloneForIndexing` в `ProjectService` только клонирует репозиторий проекта во временную директорию (`importDir`). Необходимо добавить вызов сервиса индексации сразу после успешного клонирования, чтобы код проекта автоматически попадал в векторную базу данных (Weaviate) и был доступен для контекстного поиска агентам.

---

## Зависимости

- `IndexerService` (или `CodeIndexer`) — сервис, отвечающий за парсинг, чанкинг и сохранение кода в векторную БД.
- `ProjectRepository` — для безопасного обновления статуса проекта в случае ошибки индексации.

---

## Функциональные требования

### 1. Инъекция зависимости и Архитектура (SRP)
- Добавить `IndexerService` (или интерфейс `CodeIndexer`) в структуру `projectService` и обновить конструктор `NewProjectService`.
- Обновить инициализацию `ProjectService` в `backend/cmd/api/main.go` (или DI контейнере), передав туда инстанс индексатора.
- **Декомпозиция пайплайна:** Чтобы не превращать `ProjectService` в God Object, логику оркестрации фоновой задачи (создание папки -> клонирование -> индексация -> очистка -> обработка паник) рекомендуется вынести в отдельный метод (например, `runIndexingPipeline`) или даже в отдельный компонент-воркер. Основной метод `Create` должен просто вызывать `go s.runIndexingPipeline(...)`.

### 2. Асинхронное выполнение, Контекст и Паники (КРИТИЧНО)
- Весь процесс (клонирование + индексация) **обязан** выполняться асинхронно (в горутине), чтобы не блокировать HTTP-ответ клиенту.
- **Предотвращение утечек горутин (Goroutine Leaks):** Использовать отвязанный от HTTP-запроса контекст, **обязательно обернутый в таймаут**. Например: `ctx, cancel := context.WithTimeout(context.WithoutCancel(reqCtx), 15*time.Minute)`. Обязательно вызывать `defer cancel()` внутри горутины для освобождения ресурсов таймера.
- **Перехват паник (Panic Recovery):** Внутри фоновой горутины на самом верхнем уровне обязан присутствовать `defer func() { if r := recover(); r != nil { ... } }()`. Внутри `recover` необходимо:
  1. Залогировать стек-трейс ошибки (через структурированный логгер).
  2. Безопасно перевести статус проекта в БД в `IndexingFailed` (см. пункт 3).

### 3. Обработка ошибок, Обновление статуса и Утечка секретов
- В случае ошибки индексации использовать **структурированное логирование** (`slog`, `zap` или `logrus`). Лог должен содержать: `projectID`, `workDir` и саму ошибку.
- **Маскировка токенов (Token Leakage):** Перед логированием ошибки клонирования (или любой другой ошибки, содержащей URL) **обязательно** убедиться, что из текста ошибки вырезан или замаскирован credential (токен/пароль), если он передавался в URL (например, `https://user:***@github.com/...`).
- **Безопасное обновление статуса (Race Conditions):** Обязательно обновлять статус проекта в БД при падении индексации (на `IndexingFailed` или `ReadyWithErrors`). Чтобы избежать состояния гонки (затирания чужих изменений или обновления удаленного проекта), обновление должно происходить с учетом текущего состояния: `UPDATE projects SET status = 'IndexingFailed' WHERE id = ? AND status = 'Indexing'`.

### 4. Безопасность и очистка (Security & Path Traversal)
- Для предотвращения уязвимостей типа Path Traversal при удалении директории, `workDir` должен создаваться через безопасный `os.MkdirTemp`, либо путь должен строго валидироваться.
- Независимо от результата индексации (успех, ошибка или паника) необходимо гарантированно удалить склонированную директорию (`workDir`). **Важно:** вызов `defer os.RemoveAll(workDir)` должен находиться **внутри самой асинхронной горутины**, а не в вызывающем методе (чтобы папка не удалилась до завершения индексации).

## Критерии приёмки

- [ ] `IndexerService` успешно внедрён в `ProjectService`.
- [ ] Логика оркестрации вынесена в отдельный метод/компонент (SRP).
- [ ] Процесс клонирования и индексации выполняется асинхронно и не блокирует HTTP-запрос.
- [ ] Используется безопасный, отвязанный от запроса контекст **с обязательным таймаутом**.
- [ ] Настроен перехват паник (`recover`) внутри горутины.
- [ ] Временная директория создаётся безопасно (через `os.MkdirTemp`) и гарантированно удаляется после завершения (вызов `defer` внутри горутины).
- [ ] Ошибки логируются структурно, **токены/пароли в URL маскируются**.
- [ ] При ошибке индексации статус проекта безопасно обновляется (`WHERE id = ? AND status = 'Indexing'`).
- [ ] Юнит-тесты обновлены и покрывают новые сценарии (включая обработку ошибок, паник, маскировку и безопасное обновление статуса).
