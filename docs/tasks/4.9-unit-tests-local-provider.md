## 0. Потокобезопасность моков (обязательно при `t.Parallel()` + `-race`)

**Проблема:** **stateful**-мок (в первую очередь **`recordingRunner`**) при записи в слайс / счётчик **без синхронизации** (`r.calls = append(r.calls, args)`) даёт **data race**, если один инстанс задействован из **`t.Parallel()`**-подтестов — CI с **`-race`** падает **нестабильно** или стабильно, но это всегда дефект тестовой обвязки.

**Правило:** любой тестовый double с **изменяемым состоянием** (журнал вызовов, последние `args`, счётчики, флаги) **ОБЯЗАН** быть потокобезопасным: **`sync.Mutex`** или **`sync.RWMutex`** вокруг **всех** чтений/записей полей состояния; либо **строгая изоляция**: **новый** мок на каждый `Test*` / подтест и **никогда** один и тот же незащищённый инстанс между параллельными goroutine.

**Запрещено:** один и тот же незащищённый мок в **`t.Parallel()`**-подтестах.

**Критерий:** `go test -race ./pkg/gitprovider/...` — **ноль** предупреждений о гонках в тестовом коде.

## 1. Безопасность и инъекции (КРИТИЧНО)

### 1.1 Command / flag injection — не только `-h` / `--help`

**Обязательная матрица (table-driven):**

| Вектор | Пример ввода | Ожидание |
|--------|----------------|----------|
| Upload-pack | **`--upload-pack=`** (и обходы, допустимые текущим парсером) | **до `RunGit`**: отказ; **нет** вызова раннера с опасным argv |
| Receive-pack | **`--receive-pack=`** | то же |

Охват: **`CreateBranch`**, **`DeleteLocalBranch`**, **`Push`** (ветка), **`Clone`** (`Branch`), **`GetLocalDiff`** / **`GetLocalFileContent`** (ref, path) — где строка попадает в argv.

### 1.2 Path traversal — `CloneOptions.DestPath`

**Table-driven** на **`Clone`**: `../../…`, `..\`, абсолютные пути вне политики. До **`RunGit`** — ошибка (**`ErrUnsafePath`** или согласованный sentinel); **нет** argv с токсичным `DestPath`. Отсутствие валидации в проде — **исправить в PR**.

**`filepath.Clean` — не панацея:** `filepath.Clean("/etc/passwd")` остаётся **`/etc/passwd`**; `filepath.Clean("../../../etc/passwd")` может остаться с **`..`** в цепочке, если исходный путь не был разрешён относительно доверенного корня. **Запрещено** считать, что один вызов **`filepath.Clean(path)`** устраняет path traversal. Нужна проверка, что итоговый путь **не выходит за пределы** разрешённой базы (например **`filepath.Abs`** для обоих путей и затем **`strings.HasPrefix`** с **границей сегмента** — `base + string(filepath.Separator)`, либо **`filepath.Rel(base, target)`** и отказ при `strings.HasPrefix(rel, "..")`), а не голый префикс без разделителя (иначе путаются `/safe/base` и `/safe/basement`).

Та же политика для **`Commit` (`Files`)**, **`GetLocalFileContent`**, пайпов **`readCloserWithWait`**.

### 1.3 Утечка токена — QueryEscape и PathEscape

Два **отдельных** кейса: **`+`** (QueryEscape) и **`%20`** (PathEscape). В **`err.Error()`** — нет секрета в любой из форм; маскировка через **`sanitizeToken`** (или единый слой).

## 2. Краевые случаи и валидация

### 2.1 Пустые значения и `nil`

- Пустые строки — **table-driven per method** (§3.2).
- **`ctx == nil`** — явная ошибка, **без panic** для всех покрываемых методов; иначе — исправление прод-кода в PR.
- Указатели внутри опций / будущие **`*CloneOptions`** — **`nil` → ошибка**, не panic.

### 2.2 Отмена контекста, таймауты, горутины и зомби-процессы

**Юнит (мок раннера):**

- Отмена **`ctx`** до/во время блокирующего вызова → метод **завершается**; после теста — **`goleak.VerifyNone(t)`** (или согласованная схема).
- **`GetLocalDiff` / `ReadCloser`:** отмена во время чтения + **`Close()`** — goleak, без deadlock (**`readCloserWithWait`**).

**Интеграция (реальный `git` в `local_integration_test.go`):**

- Обязателен сценарий **`context.WithTimeout`**: по истечении таймаута возвращается ошибка, **связанная с дедлайном** (`errors.Is(err, context.DeadlineExceeded)` там, где контракт это допускает), и при этом:
  - **освобождены** `StdoutPipe` / `StderrPipe` (нет утечки дескрипторов);
  - дочерний процесс git **не остаётся** висящим (ожидание через `Wait` / убийство по отмене контекста на уровне **`exec.CommandContext`** — поведение прод-кода **должно** соответствовать: отмена контекста **прерывает** ожидание и **завершает** процесс ОС; тест фиксирует отсутствие зомби в типичном CI-окружении).
- Для неинтерактивного git: окружение вроде **`GIT_TERMINAL_PROMPT=0`** (и при необходимости **`GIT_ASKPASS`**) — **обязательно** в integration-тестах, чтобы git **не зависал** на запросе пароля в stdin (иначе таймаут **маскирует** баг и даёт **flaky** «иногда 60s, иногда ок»).
- **Анти-flaky:** таймауты в integration выбирать **короткими и фиксированными** (достаточно, чтобы успешный сценарий укладывался стабильно в CI, но зависший процесс **гарантированно** рвал тест), без зависимости от порядка тестов в пакете.
- **Очистка ресурсов:** `t.TempDir()` удаляется автоматически **при нормальном завершении** теста; при **panic**, **`os.Exit`** в дочернем коде или на **Windows** (блокировки файлов живым **git**-процессом) мусор и утечки возможны. Любые ресурсы **вне** `t.TempDir()` (файлы в фиксированных путях, фоновые процессы, сокеты) — регистрировать через **`t.Cleanup(func() { ... })`**: остановка процесса, снятие блокировок, удаление каталогов. Цель — гарантированная очистка **даже при падении** теста (cleanup вызывается и после panic в самом тесте, пока не убит весь процесс).

### 2.3 Унификация маппинга stderr → классификация

Здесь задаётся **только** соответствие «текст stderr → классификация / sentinel». **Как вернуть ошибку вызывающему коду** (сохранить stderr для дебага и **`errors.Is`**) — **строго §2.4**; не смешивать в один `return` без `%w`.

- Один пакетный хелпер (**`parseGitStderr` / `mapGitCLIError`** — имя в PR).
- Провайдер/CLI **не** размножают несогласованный **`strings.Contains`**.
- Тесты хелпера — в **`helpers_test.go`**, table-driven; тесты методов проверяют **итоговую ошибку** и **§2.4**.
- Вспомогательные функции, собирающие таблицу/мок (**`newLocalProviderWithRunner`**, парсинг кейсов) — см. **§3.0** (`t.Helper()`).

### 2.4 Обёртка ошибок (`%w` / `errors.Join`)

**Запрещено:**

- `return ErrAuthFailed` **без** контекста — теряется stderr, невозможен отладочный лог.
- `return fmt.Errorf("… %v", ErrAuthFailed)` или строка без **`%w`** там, где вызывающий должен использовать **`errors.Is`** — ломается оркестратор/сервисный слой.

**Обязательно:**

- Sentinel-ошибки **в цепочке сохраняются** через **`fmt.Errorf("…: %w", ErrX)`** или **`errors.Join`** (несколько причин).
- Текст stderr (после **`sanitizeToken`**) попадает в сообщение **как детали**, **не** подменяя sentinel. **Канонические примеры** (точные префиксы — в PR, семантика обязательна):
  - `fmt.Errorf("git validate access: %w, details: %s", ErrAuthFailed, sanitizeToken(stderr, token))`
  - `fmt.Errorf("git auth failed: %w, details: %s", ErrAuthFailed, sanitizeToken(stderr, token))`

**Тесты ОБЯЗАНЫ:**

- для каждого сценария с **sentinel** — **`errors.Is(err, Err…)`** (например `require.ErrorIs(t, err, ErrAuthFailed)`): проверяем **семантику** «это наша ошибка домена» по цепочке `%w`.
- **`errors.As`** — когда нужен **конкретный тип** с полями, а не сравнение с глобальной переменной-sentinel. Пример: в цепочке после `git` остаётся **`*exec.ExitError`** — нужно проверить **код выхода** (`var ee *exec.ExitError; require.True(t, errors.As(err, &ee)); require.Equal(t, 128, ee.ExitCode())`) или кастомный тип обёртки с полем «сырой stderr» для инспекции в тесте. **`errors.Is` для этого не подходит**, если цель — доступ к структуре, а не равенство sentinel.
- отдельный кейс: в **`err.Error()`** есть маскированный фрагмент stderr **и** при этом **`errors.Is`** к нужному sentinel; **нет** сырого токена (**§1.3**).

## 3. Архитектура и антипаттерны в тестах

### 3.0 `t.Helper()` во вспомогательных функциях

Любая функция в `*_test.go`, вызываемая из `Test*` / `t.Run` (**`newLocalProviderWithRunner`**, сборка мока, кастомные **`assertXxx`**, парсинг табличного кейса), **ОБЯЗАНА** начинаться с **`t.Helper()`**. Иначе при падении `go test` указывает строку **внутри хелпера**, а не строку таблицы/теста — разбор **CI** заметно дольше. Исключение — функции **без** `*testing.T` (чистые утилиты), но тогда они **не** должны вызывать `t.Fatal`/`require` (передавать ошибку наверх).

### 3.1 Ошибки и `defer`

- **Запрещено:** `_ = err`, игнор **`Close()`**.
- **Обязательно:**

```go
defer func() { require.NoError(t, rc.Close()) }()
```

### 3.2 Table-driven — одна таблица на метод для валидации

Все кейсы одного класса (пустые строки, инъекции §1.1, traversal §1.2, невалидные ref/path) — **один** table-driven тест на публичный метод; общий конструктор мока **`newLocalProviderWithRunner`**, см. **§0** (mutex) и **§3.0** (`t.Helper()`).

## 4. Производительность и объём данных (стриминг без ложной аллокации в тесте)

**Запрещено** для проверки «стриминга» создавать **10+ MiB** в памяти теста через **`bytes.Repeat([]byte("a"), 10*1024*1024)`** или аналог — это **одна большая аллокация в самом тесте** и **не доказывает**, что прод стримит без удержания всего тела.

**Обязательно:**

- Источник для мока — **кастомный `io.Reader` без больших буферов** и **без лишней траты CPU** на мегабайты «зануления». Буфер `p` в `Read` при типичном использовании (`make([]byte, n)` / пул) уже **нулевой**; для проверки стриминга **содержимое байт не важно** — важен **объём** и отсутствие у прод-кода чтения всего тела в один слайс. Пример (**имя можно сократить до `infiniteReader`**, смысл тот же):

```go
type infiniteZeroReader struct{}

func (infiniteZeroReader) Read(p []byte) (int, error) {
	// Не тратим CPU на цикл по len(p); не аллоцируем 10 MiB в тесте
	return len(p), nil
}
```

- Обёртка: **`io.LimitReader(infiniteZeroReader{}, 10<<20)`** — логический объём 10 MiB **без** предварительного слайса / **`bytes.Repeat`** на 10 MiB в куче теста (**честная** проверка стриминга и нагрузки на аллокации со стороны **прод-пути**, а не подготовки данных в тесте).
- Для пути **`GetLocalDiff`** (pipe): тест читает **небольшой префикс** (например 64 KiB), затем **`Close()`**; **ассерт** на аллокации: **`testing.AllocsPerRun`** / **`runtime.MemStats`** / мем-профиль — порог и обоснование **в комментарии к тесту**; цель — порядок величины **не** как у «прочитать 10 MiB в `[]byte` одним куском» в **прод-коде** на этом пути.

**`GetLocalFileContent`:** либо стриминг + тот же паттерн **`LimitReader` + лёгкий `Reader`**, либо **жёсткий лимит размера** в проде + table-driven на **`ErrTooLarge`** без OOM. Молчаливое чтение гигабайта в память — **не принимается**.
