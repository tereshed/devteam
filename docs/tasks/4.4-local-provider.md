# 4.4 Реализация: LocalGitProvider (git CLI через exec)

**Sprint:** 4 — Git Provider Integration (Backend)
**Файл:** `backend/pkg/gitprovider/local.go`
**Статус:** ✅ Выполнена

---

## Описание

Реализовать `LocalGitProvider` — имплементацию интерфейса `GitProvider` (4.1) для работы с локальными git-репозиториями **без remote hosting API**. Провайдер работает исключительно через `exec.Command("git", ...)` и предназначен для:

- **Разработка и тестирование** — запуск агентов на локальной машине без GitHub/GitLab.
- **CI/CD** — операции с git без API-ключей.
- **Утилитарные сценарии** — автоматизация git-операций без sandbox.

**Ключевое отличие от GitHubProvider:** Нет зависимости от `go-github`, `oauth2` или любого REST API. Все **15 remote-only методов** возвращают `ErrNotImplemented`. Все **12 local-методов** реализованы через git CLI.

## Зависимости

**Требует:** 4.1 (интерфейс), 4.2 (типы)

**Внешние:** `git` в PATH (системный)

## Структура

```go
package gitprovider

// LocalGitProvider реализует GitProvider для локальных git-репозиториев.
// Все операции выполняются через системный git CLI (exec.Command).
// Remote API операции (PR, remote branches, repo info) возвращают ErrNotImplemented.
//
// Credentials используются только для Clone/Push (Token injection в URL).
// Если Token пуст — git использует системную аутентификацию (SSH agent, credential helper).
type LocalGitProvider struct {
    creds Credentials
}

// NewLocalGitProvider создаёт экземпляр LocalGitProvider.
// Вызывается из Factory (4.5), НЕ напрямую.
func NewLocalGitProvider(creds Credentials) *LocalGitProvider {
    return &LocalGitProvider{creds: creds}
}
```

**Минимальная структура:** Нет API-клиента, нет HTTP-транспорта. Только `Credentials` для опциональной Token injection при Clone/Push.

### Общие хелперы (helpers.go)

```go
// gitExec выполняет git-команду в указанной директории.
// ⚠️ Ошибки из stderr должны проходить через sanitizeToken, если команда
// может содержать URL с токеном (clone, push, ls-remote, fetch).
func gitExec(ctx context.Context, workDir string, args ...string) (string, error)

// readCloserWithWait — обёртка io.ReadCloser, предотвращающая zombie-процессы.
// ⚠️ КРИТИЧНО: Close() ОБЯЗАН закрыть pipe ДО вызова cmd.Wait(),
// иначе возможен deadlock (git блокируется на записи в переполненный pipe).
type readCloserWithWait struct {
    io.ReadCloser
    cmd    *exec.Cmd
    stderr *bytes.Buffer // захват stderr для понятных сообщений об ошибках
}

func (r *readCloserWithWait) Close() error {
    pipeErr := r.ReadCloser.Close() // 1. Закрываем pipe → git получает SIGPIPE и завершается
    waitErr := r.cmd.Wait()          // 2. Ждём завершения процесса (теперь безопасно)
    if waitErr != nil {
        // Включаем текст из stderr, чтобы ошибка была информативной
        // (вместо абстрактного "exit status 128" → "git command failed: ..., stderr: fatal: bad revision...")
        if r.stderr != nil && r.stderr.Len() > 0 {
            return fmt.Errorf("git command failed: %w, stderr: %s", waitErr, r.stderr.String())
        }
        return waitErr
    }
    return pipeErr
}

// injectTokenInURL заменяет https://... на https://x-access-token:{token}@...
// ⚠️ Только для HTTP/HTTPS URL. SSH URL (git@...) возвращаются без изменений.
// Если token пуст — возвращает URL без изменений.
// ⚠️ ОБЯЗАТЕЛЬНО использовать net/url для безопасного парсинга:
//
//   u, err := url.Parse(repoURL)
//   u.User = url.UserPassword("x-access-token", token)
//   return u.String()
//
// Простая замена строк (strings.Replace) ЗАПРЕЩЕНА:
// URL вида https://user:pass@github.com/... превратится в невалидный
// https://x-access-token:token@user:pass@github.com/...
func injectTokenInURL(repoURL string, token string) string

// isHTTPURL проверяет, является ли URL HTTP/HTTPS (для Token injection).
// Используется в injectTokenInURL и ValidateAccess.
func isHTTPURL(url string) bool

// sanitizeToken заменяет Token на *** в строке для безопасного логирования.
// ⚠️ КРИТИЧНО: ОБЯЗАНА вырезать как сырой токен, так и его URL-кодированную версию.
// Причина: net/url.UserPassword URL-кодирует спецсимволы в токене (abc@def → abc%40def).
// Git выводит URL-кодированный токен в stderr при ошибке, а простой ReplaceAll ищет
// только сырую строку и пропускает закодированную → утечка токена в логи.
//
//   func sanitizeToken(s string, token string) string {
//       if token == "" { return s }
//       s = strings.ReplaceAll(s, token, "***")
//       s = strings.ReplaceAll(s, url.QueryEscape(token), "***")
//       return s
//   }
func sanitizeToken(s string, token string) string
```

**⚠️ Рефакторинг 4.3:** При реализации 4.4 код `gitExec`, `readCloserWithWait`, `injectTokenInURL`, `isHTTPURL` и `sanitizeToken` из `github.go` / `github_helpers.go` переносится в общий `helpers.go`. GitHubProvider начинает использовать их оттуда.

**⚠️ КРИТИЧНО (deadlock-prevention):** `readCloserWithWait.Close()` **ОБЯЗАН** закрывать pipe **ДО** вызова `cmd.Wait()`. Если вызывающий код прочитал только часть stdout и вызвал `Close()`, git может быть заблокирован на записи в переполненный pipe. Закрытие pipe приводит к SIGPIPE → git завершается → `cmd.Wait()` возвращается.

**⚠️ БЕЗОПАСНОСТЬ (Flag Injection):** `exec.Command` защищает от shell-инъекций (нельзя передать `&& rm -rf /`), но **НЕ** защищает от флаговых инъекций. Если имя ветки = `"-f"`, git воспримет его как флаг. Два стратегии защиты:
```
# Стратегия 1: разделитель "--" (checkout, branch, add, push)
git checkout -b -- <branch>    # ← "всё после -- это аргументы, не флаги"
git branch -D -- <branch>
git push <remote> -- <branch>
git add -- <file1> <file2>

# Стратегия 2: rev-parse → SHA (diff)
# git diff НЕ поддерживает "--" для ref'ов (после "--" всё = пути к файлам!)
baseSHA=$(git rev-parse --verify <base>)    # SHA = hex, не начинается с "-"
headSHA=$(git rev-parse --verify <head>)
git diff $baseSHA..$headSHA
```

### Группа 1: Реализованные методы — 12 шт.

#### 1.1 ValidateAccess

```go
func (l *LocalGitProvider) ValidateAccess(ctx context.Context, repoURL string) error
```

| Шаг | Действие |
|-----|----------|
| 1 | Если `repoURL == ""` → `return ErrRepoNotFound` (быстрая проверка) |
| 2 | Если `isHTTPURL(repoURL)` и `l.creds.Token != ""` → `injectTokenInURL(repoURL, l.creds.Token)` |
| 3 | `gitExec(ctx, "", "ls-remote", checkURL)` (**без** `--exit-code`) |
| 4 | Exit code 0 → `nil` (доступ есть, в т.ч. пустой репозиторий без ref'ов) |
| 5 | "Authentication failed" / exit code 128 → `ErrAuthFailed` |
| 6 | "not found" / exit code 128 → `ErrRepoNotFound` |

**⚠️ Почему `git ls-remote` БЕЗ `--exit-code`:**
- Флаг `--exit-code` возвращает exit code **2**, если репозиторий валиден, но не содержит ни одного ref'а (веток/тегов). Новый пустой репозиторий на GitHub = exit code 2 = наш код ошибочно вернёт `ErrRepoNotFound`.
- Без `--exit-code`: exit code 0 для **любого** доступного репозитория (включая пустые), exit code 128 при ошибке доступа/отсутствия.

**⚠️ Почему `git ls-remote` для ВСЕХ URL (включая локальные пути):**
- Корректно работает и с обычными, и с **bare-репозиториями** (у bare нет `.git/` — сам каталог является репо).
- Не требует эвристик для определения типа URL (локальный путь, SSH, HTTPS).
- Проверяет реальную доступность (не только наличие файлов на диске).
- Единственный минус: создаёт процесс git вместо `os.Stat` — для MVP это несущественно.

**Отличие от GitHubProvider:** Нет API-вызова. `git ls-remote` работает для любых URL и локальных путей.

#### 1.6 Commit

```go
func (l *LocalGitProvider) Commit(ctx context.Context, workDir string, opts CommitOptions) (string, bool, error)
```

| Шаг | Действие |
|-----|----------|
| 1 | Если `opts.Files` пуст → `gitExec(ctx, workDir, "add", "-A")` (**⚠️ см. примечание ниже**) |
| 2 | Иначе → `gitExec(ctx, workDir, "add", "--", opts.Files...)` |
| 3 | Проверить наличие HEAD: `gitExec(ctx, workDir, "rev-parse", "--verify", "HEAD")` |
| 3a | Если HEAD **НЕ** существует (первый коммит) → если `gitExec(ctx, workDir, "status", "--porcelain")` пуст → `return "", false, nil`; иначе — есть изменения, переходим к шагу 4 |
| 3b | Если HEAD существует → `exec.CommandContext(ctx, "git", "diff", "--cached", "--quiet")` в `workDir` — exit code 1 → есть staged-изменения, exit code 0 → `return "", false, nil` |
| 4 | args: `["commit", "-m", opts.Message]` |
| 5 | Если Author задан → `args = append(args, "--author", opts.Author.String())` |
| 6 | `gitExec(ctx, workDir, args...)` |
| 7 | SHA: `gitExec(ctx, workDir, "rev-parse", "HEAD")` |
| 8 | `return strings.TrimSpace(sha), true, nil` |

**⚠️ Проверка staged-изменений:** Используется `git diff --cached --quiet`, а **НЕ** `git status --porcelain`. Причина: `git status --porcelain` показывает ВСЕ изменения (staged + unstaged + untracked). Если агент вызвал `git add A.go`, а файл `B.go` тоже изменён, `git status --porcelain` вернёт непустой вывод из-за `B.go`, но `git commit` упадёт с "nothing to commit", т.к. в индексе (staged) изменений `B.go` нет. `git diff --cached --quiet` проверяет **только индекс** — ровно то, что попадёт в коммит.

**⚠️ Парадокс первого коммита:** `git diff --cached` падает с `fatal: bad revision 'HEAD'` в пустом репозитории без единого коммита. Поэтому перед проверкой staged-изменений **ОБЯЗАТЕЛЬНО** проверяется существование HEAD через `git rev-parse --verify HEAD`. Если HEAD нет (первый коммит), используется fallback через `git status --porcelain` — это единственный безопасный случай для porcelain-проверки.

**⚠️ `git add -A` (MVP-решение):** Если `opts.Files` пуст, `git add -A` добавляет ВСЕ untracked файлы (включая временные файлы агента, логи, артефакты сборки, не попавшие в `.gitignore`). Для MVP это допустимо: Sandbox гарантирует чистоту workDir, а `.gitignore` проекта исключает мусор. **Post-MVP:** Агент должен явно передавать список изменённых файлов через `opts.Files`.

**⚠️ Валидация Author:** Если `opts.Author` задан, перед формированием `--author` **ОБЯЗАТЕЛЬНА** проверка:
1. `Name` и `Email` **НЕ ПУСТЫЕ** (и не состоят только из пробелов) — иначе `git commit --author=" <email>"` или `--author="Name <>"` упадёт с фатальной ошибкой.
2. `Name` и `Email` **НЕ СОДЕРЖАТ** `\n`, `<`, `>` — это сломает формат `"Name <Email>"`.
3. При нарушении — вернуть ошибку валидации **до** вызова `exec.Command`.

Валидация реализуется в `Author.Validate() error` или в самом `Commit` перед шагом 5.

#### 1.7 Push

```go
func (l *LocalGitProvider) Push(ctx context.Context, workDir string, opts PushOptions) error
```

| Шаг | Действие |
|-----|----------|
| 1 | `remote` = opts.Remote, если пусто → `"origin"` |
| 2 | Если `l.creds.Token != ""` и remote URL — HTTP/HTTPS → собрать authenticated push URL |
| 3 | `pushTarget` = authenticated URL (если Token) или remote name (без Token) |
| 4 | args: `["push", pushTarget, "--", opts.Branch]` |
| 5 | Если `opts.Force` → `args = append(args, "--force")` |
| 6 | `gitExec(ctx, workDir, args...)` |
| 7 | "rejected" в stderr → `ErrConflict` |
| 8 | "Permission denied" → `ErrPermissionDenied` |
| 9 | **Все ошибки (stderr)** оборачиваются через `sanitizeToken` для предотвращения утечки Token в логах |

**⚠️ Token injection:** Как и в GitHubProvider, Token передаётся через URL в команде `git push <URL> <branch>`, **НЕ** через `git remote set-url`. Если Token пуст — используется remote name и системная аутентификация.

**⚠️ Безопасность:** При ошибке `git push` stderr содержит URL (например, `fatal: unable to access 'https://x-access-token:secret@...'`). Ошибка **ОБЯЗАНА** проходить через `sanitizeToken` перед возвратом или логированием.

#### 1.8 CommitAndPush

```go
func (l *LocalGitProvider) CommitAndPush(ctx context.Context, workDir string, commitOpts CommitOptions, pushOpts PushOptions) (string, bool, error)
```

| Шаг | Действие |
|-----|----------|
| 1 | `sha, hasChanges, err := l.Commit(ctx, workDir, commitOpts)` |
| 2 | Если `err != nil` → `return "", false, err` |
| 3 | `err = l.Push(ctx, workDir, pushOpts)` — **Push выполняется ВСЕГДА**, даже если `!hasChanges` |
| 4 | `return sha, hasChanges, err` |

**⚠️ КРИТИЧНО: Push выполняется даже при `hasChanges=false`.**
Сценарий: Агент делает промежуточный `Commit` (чекпоинт), затем в конце вызывает `CommitAndPush`. Новых изменений нет (`hasChanges=false`), но в локальной ветке есть непушнутые коммиты. Если пропустить Push — код останется в контейнере, а Оркестратор решит, что задача завершена без результатов.

#### 1.9 GetLocalDiff

```go
func (l *LocalGitProvider) GetLocalDiff(ctx context.Context, workDir string, base, head string) (io.ReadCloser, error)
```

| Шаг | Действие |
|-----|----------|
| 1 | `baseSHA, err := gitExec(ctx, workDir, "rev-parse", "--verify", base)` — если ошибка → `return nil, fmt.Errorf("invalid base ref %q: %w", base, err)` |
| 2 | `headSHA, err := gitExec(ctx, workDir, "rev-parse", "--verify", head)` — если ошибка → `return nil, fmt.Errorf("invalid head ref %q: %w", head, err)` |
| 3 | `cmd := exec.CommandContext(ctx, "git", "diff", strings.TrimSpace(baseSHA)+".."+strings.TrimSpace(headSHA))` |
| 4 | `cmd.Dir = workDir` |
| 5 | `var stderr bytes.Buffer` → `cmd.Stderr = &stderr` |
| 6 | `stdout, _ := cmd.StdoutPipe()` |
| 7 | `cmd.Start()` |
| 8 | Вернуть `&readCloserWithWait{ReadCloser: stdout, cmd: cmd, stderr: &stderr}` |

**⚠️ БЕЗОПАСНОСТЬ (Flag Injection в diff):** `git diff` НЕ поддерживает `--` для разделения ref'ов и путей так, как другие команды. `git diff -- base head` интерпретирует `base` и `head` как **пути к файлам**, а не как ветки. Единственный безопасный способ передать недоверенные имена веток — предварительно разрезолвить их в SHA-хэши через `git rev-parse --verify`. SHA состоит из hex-символов и **никогда** не начинается с `-`.

**⚠️ Ранняя валидация ref'ов:** В отличие от предыдущей версии, ошибки невалидных ref'ов обнаруживаются **сразу** (на шагах 1-2), а не через отложенный `Close()`. Это значительно упрощает обработку ошибок вызывающим кодом.

**⚠️ КРИТИЧНО:** `Close()` сначала закрывает pipe (SIGPIPE → git завершается), затем вызывает `cmd.Wait()`.

**⚠️ Захват stderr:** При ошибке `cmd.Wait()` возвращает абстрактный `exit status 128`. `readCloserWithWait` включает текст из stderr в сообщение об ошибке.

#### 1.10 GetLocalFileContent

```go
func (l *LocalGitProvider) GetLocalFileContent(ctx context.Context, workDir string, ref string, path string) (io.ReadCloser, error)
```

| Шаг | Действие |
|-----|----------|
| 1 | `output, err := gitExec(ctx, workDir, "cat-file", "blob", ref+":"+path)` |
| 2 | Если err и stderr содержит "does not exist" / "not a valid object" → `return nil, ErrFileNotFound` |
| 3 | Если err → `return nil, err` |
| 4 | `return io.NopCloser(bytes.NewReader([]byte(output))), nil` |

**MVP-подход:** Используется `gitExec` (не streaming). Ошибка `ErrFileNotFound` возвращается **сразу**, а не через отложенный `Close()`. Файлы кода редко превышают 1MB — загрузка в память допустима. (см. дизайн-решение §4).

**⚠️ Почему `git cat-file blob`, а НЕ `git show`:** `git show` — porcelain (человекочитаемая) команда, подверженная влиянию пользовательских настроек (`core.autocrlf`, пейджеры, фильтры). `git cat-file blob` — plumbing (низкоуровневая) команда, возвращающая сырые байты файла без трансформаций. Для программного извлечения контента это единственный правильный выбор.

### Группа 2: ErrNotImplemented — 15 методов

## Дизайн-решения

### 1. Код-дедупликация с GitHubProvider

9 из 12 реализованных методов LocalGitProvider **идентичны** соответствующим local-методам GitHubProvider (CreateBranch, ListLocalBranches, DeleteLocalBranch, Commit, CommitAndPush, GetLocalDiff, GetLocalFileContent, ProviderType, SupportsPullRequests отличается лишь возвращаемым значением).

**Подход MVP:** Дублирование кода допустимо. Каждый провайдер — самостоятельная реализация.

**Post-MVP:** Вынести общую логику в `baseGitCLI` (embedded struct):

```go
type baseGitCLI struct{}
func (b *baseGitCLI) CreateBranch(ctx context.Context, workDir string, opts BranchOptions) error { ... }
func (b *baseGitCLI) Commit(ctx context.Context, workDir string, opts CommitOptions) (string, bool, error) { ... }
// ...

type LocalGitProvider struct {
    baseGitCLI
    creds Credentials
}

type GitHubProvider struct {
    baseGitCLI
    client *github.Client
    creds  Credentials
}
```

### 2. ValidateAccess: единый `git ls-remote` для всех URL

**Выбор:** `git ls-remote` для **всех** типов URL (локальные пути, HTTP/HTTPS, SSH).

**Альтернатива (отклонена):** `os.Stat` + `.git` для локальных путей.

**Почему отклонена:**
- `os.Stat(filepath.Join(path, ".git"))` **ломается** на bare-репозиториях (`git init --bare`) — у них нет `.git/` подпапки.
- Эвристика `strings.Contains(repoURL, "@")` для определения remote URL ошибочна: локальный путь `/Users/user/my@project` содержит `@`.
- `git ls-remote` корректно проверяет **любой** тип репозитория: обычный, bare, remote HTTP, remote SSH.

**Token injection:** Применяется только если `isHTTPURL(repoURL)` и Token задан. Для SSH и локальных путей — без изменений.

### 3. Аутентификация — опциональна

В отличие от GitHubProvider (Token обязателен для API), LocalGitProvider работает **без Token**:

| Credentials | Поведение |
|---|---|
| Token задан | Token injection в URL при Clone/Push |
| Token пуст | Git использует системную аутентификацию (SSH agent, credential helper, `~/.gitconfig`) |
| Все поля пусты | Работает только с локальными репо (без remote) или public repos |

### 4. GetLocalFileContent — plumbing-команда `cat-file` + MVP через `cmd.Output()`

**Выбор команды:** `git cat-file blob <ref>:<path>` (plumbing), а **НЕ** `git show <ref>:<path>` (porcelain).

**Почему не `git show`:**
- `git show` — porcelain-команда, подверженная влиянию `core.autocrlf`, пейджеров, фильтров и пользовательских настроек.
- `git cat-file blob` — plumbing-команда, возвращающая **сырые байты** файла без трансформаций. Быстрее, стабильнее, не зависит от локальных конфигов.

**Выбор I/O (MVP):** `gitExec` + `io.NopCloser(bytes.NewReader([]byte(output)))` — весь файл в память.

**Альтернатива (post-MVP):** `StdoutPipe()` + `readCloserWithWait` — streaming.

**Обоснование:** Подход через `cmd.Output()` **полностью снимает** проблемы с:
- Deadlock при частичном чтении pipe.
- Отложенными ошибками (ошибка `ErrFileNotFound` возвращается **сразу** из `gitExec`, а не из `Close()`).
- Zombie-процессами (нет pipe → нет cmd.Wait).

Файлы исходного кода редко превышают 1MB, для MVP это оптимальный компромисс.

```go
func (l *LocalGitProvider) GetLocalFileContent(ctx context.Context, workDir string, ref string, path string) (io.ReadCloser, error) {
    output, err := gitExec(ctx, workDir, "cat-file", "blob", ref+":"+path)
    if err != nil {
        if strings.Contains(err.Error(), "does not exist") || strings.Contains(err.Error(), "not a valid object") {
            return nil, ErrFileNotFound
        }
        return nil, err
    }
    return io.NopCloser(bytes.NewReader([]byte(output))), nil
}
```

**Post-MVP:** Переключиться на streaming (`StdoutPipe` + `readCloserWithWait`) для обработки больших файлов (бинарные ассеты, сгенерированный код).

### 5. GetLocalDiff — rev-parse + streaming через `readCloserWithWait`

В отличие от `GetLocalFileContent`, `GetLocalDiff` **остаётся streaming** (через `StdoutPipe`), т.к. диффы при рефакторинге могут быть огромными (десятки MB). Для них загрузка в память неприемлема.

**Ранняя валидация ref'ов:** Перед запуском `git diff` имена веток разрезолвиваются в SHA через `git rev-parse --verify`. Это даёт два преимущества:
1. **Безопасность:** SHA — hex-строка, не может быть интерпретирована как флаг. `git diff` не поддерживает `--` для разделения ref'ов и путей (после `--` всё трактуется как пути к файлам).
2. **Быстрый фейл:** Ошибки невалидных ref'ов обнаруживаются сразу, а не через отложенный `Close()`.

**Захват stderr:** `readCloserWithWait` хранит `*bytes.Buffer` stderr для информативных ошибок.

### 6. CommitAndPush — Push всегда выполняется

`CommitAndPush` **НЕ** пропускает Push при `hasChanges=false`. В локальной ветке могут быть предыдущие непушнутые коммиты (от промежуточных `Commit`). Пропуск Push приведёт к потере кода агента в эфемерном контейнере.

### 7. Безопасность: Flag Injection

`exec.Command` защищает от shell-инъекций, но **не** от флаговых инъекций. Если имя ветки = `"-f"`, `git checkout -b -f` будет воспринят как `--force`.

**Стратегия защиты (два подхода):**

| Подход | Команды | Описание |
|---|---|---|
| `--` разделитель | `checkout`, `branch`, `add`, `push` | `--` говорит git: "дальше только аргументы, не флаги" |
| `rev-parse` → SHA | `diff` | `--` НЕ работает для ref'ов в `git diff` (трактуются как пути). Резолвим в SHA. |

**Команды с `--`:**
- `git checkout -b -- <branch>`
- `git branch -D -- <branch>`
- `git add -- <file1> <file2>`
- `git push <remote> -- <branch>`

**Команды с SHA-резолвингом:**
- `git diff <baseSHA>..<headSHA>` — SHA = hex-строка, не может начинаться с `-`

Это применяется в обоих провайдерах (LocalGitProvider и GitHubProvider).

### 8. Безопасность: `injectTokenInURL` через `net/url`

Простая замена строк (`strings.Replace(url, "https://", "https://x-access-token:token@", 1)`) **ЗАПРЕЩЕНА**: URL вида `https://user:pass@github.com/repo.git` превратится в невалидный `https://x-access-token:token@user:pass@github.com/...`.

**ОБЯЗАТЕЛЬНО** использовать стандартный `net/url.Parse` + `url.UserPassword`:

```go
u, _ := url.Parse(repoURL)
u.User = url.UserPassword("x-access-token", token)
return u.String()
```

### 9. `sanitizeToken` и URL-Encoding

`sanitizeToken` **ОБЯЗАНА** вырезать токен в двух формах:
1. **Сырой токен** — `strings.ReplaceAll(s, token, "***")`
2. **URL-кодированный токен** — `strings.ReplaceAll(s, url.QueryEscape(token), "***")`

**Причина:** `net/url.UserPassword` автоматически URL-кодирует спецсимволы в токене (`abc@def` → `abc%40def`). Git выводит закодированную форму в stderr при ошибке. Если `sanitizeToken` ищет только сырой токен, закодированная версия **утекает в логи** в открытом виде.

### 10. `gitExec` и `sanitizeToken`

`gitExec` — центральная точка для выполнения git-команд. Для команд, работающих с URL (clone, push, ls-remote, fetch), stderr **ДОЛЖЕН** проходить через `sanitizeToken` перед возвратом ошибки. Для команд без URL (status, diff, cat-file, branch) это не требуется, но не вредит.
