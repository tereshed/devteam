# 6.3 Реализация: `SandboxAgentExecutor`

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/agent/sandbox_executor.go`
**Статус:** ⬜ В планах

---

## Описание

Реализовать **`SandboxAgentExecutor`** — конкретную реализацию интерфейса **`AgentExecutor`** (задача **6.1**), которая выполняет шаги агента (преимущественно роли **Developer** и **Tester**) внутри изолированных Docker-контейнеров с использованием **`SandboxRunner`** (Sprint 5).

В отличие от **`LLMAgentExecutor`** (6.2), который делает один запрос к LLM, **`SandboxAgentExecutor`** запускает полноценный жизненный цикл задачи в контейнере: клонирование репозитория, создание ветки, запуск CLI-агента (Claude Code или Aider) и сбор артефактов (diff, логи, результат).

## Функциональные требования

### 1. Архитектура и маппинг (🏗️ Stateless & Performance)
- **Stateless-компонент:** Экзекутор **НЕ ДОЛЖЕН** самостоятельно ходить в БД за секретами, конфигурациями или моделями. Все данные приходят исключительно через `agent.ExecutionInput`.
- **Оптимизация аллокаций:** При объединении контекстов использовать **`strings.Builder`** с предварительным вызовом **`Grow()`**. Запрещена конкатенация через `+`.

### 2. Управление жизненным циклом (🚨 Race & Leaks)
- **Early Return Validation:** Проверка обязательных полей (`TaskID`, `ProjectID`, `GitURL`, `BranchName`) в первой строке метода.
- **Гарантированная очистка (Docker Edge Cases):** 
  - Если `RunTask` вернул ошибку, но при этом вернул непустой `sandboxID`, экзекутор **ОБЯЗАН** вызвать `Cleanup` для этого ID.
  - При успешном получении `sandboxID` — немедленный `defer SandboxRunner.Cleanup`.
  - **КРИТИЧНО (Context in Defer):** При вызове `Cleanup` в `defer` **ЗАПРЕЩЕНО** использовать родительский (возможно отмененный) `context.Context`. Необходимо использовать `context.WithoutCancel(ctx)` (Go 1.21+) или `context.WithTimeout(context.Background(), 10*time.Second)`, чтобы гарантировать выполнение очистки даже при отмене задачи.
- **Таймауты (Fallback Timeout):** Если входящий `context.Context` не имеет дедлайна, экзекутор **ОБЯЗАН** обернуть его в `context.WithTimeout` (например, 30-60 минут).
- **Безопасная отмена:** Паттерн `select` с `ctx.Done()` для прерывания `Wait()`. Гарантировать отсутствие утечек горутин и состояний гонки.

### 3. Обработка результата и лимиты (🧠 Memory Safety)
- **Защита от OOM:** Экзекутор обязан ограничивать размер считываемых артефактов (`Diff`, `Output`). Если размер превышает лимит (например, 5-10 МБ), артефакт должен быть усечен с добавлением пометки `...[TRUNCATED]`.
- **Защита от Nil Pointer:** Обработка `nil` результата `CodeResult` (например, при OOMKilled контейнера) с возвратом инфраструктурной ошибки.
- **Оборачивание ошибок:** Все ошибки инфраструктуры оборачивать через `fmt.Errorf("...: %w", err)` с контекстом задачи.

### 4. Безопасность и Наблюдаемость (🔐 Security & Observability)
- **Path Traversal:** Строгая валидация схем `GitURL` (`http://`, `https://`, `git://`).
- **Валидация BranchName:** Запрещены пробелы, спецсимволы и имена, начинающиеся с дефиса (`-`), для защиты от внедрения флагов в `git checkout`.
- **Маскирование секретов:** Запрет на логирование `SandboxOptions` целиком. Использовать `ExecutionInput.String()`.
- **Структурное логирование:** Использовать `slog` (или `zap`). Обязательно логировать ключевые этапы с полем `task_id`.

---

## Контракт ошибок (согласно 6.1)

- **`error != nil`**: Ошибки Docker API, таймаут инфраструктуры, невалидный вход, OOM/Crash контейнера без результата.
- **`error == nil && Success == false`**: Контейнер завершился штатно, но бизнес-цель не достигнута.
