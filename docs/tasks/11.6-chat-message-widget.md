## Выбор пакета markdown (блокер)

Пакет **`flutter_markdown`** официально не рекомендуется к новым зависимостям (discontinuation / перенос сопровождения). В PR **нельзя** добавлять именно `flutter_markdown` без исключения; допустимые стартовые точки:

| Кандидат | Заметка |
|----------|---------|
| `flutter_markdown_plus` | Преемник API `flutter_markdown`, активное сопровождение сообщества. |
| `markdown_widget` | Гибкие кастом-билдеры, удобно для code blocks / запретов. |
| `gpt_markdown` | Ориентир на чат-LLM; проверить selectable, кастом блоков и лицензию. |

**Критерии выбора (обязательная строка в описании PR):** активный maintenance на pub.dev, поддержка GFM-совместимого подмножества, возможность отключить опасные режимы (HTML, произвольные картинки), кастомизация блока кода, **лицензия:** допустимы **MIT, BSD, Apache-2.0** (и совместимые пермиссивные); **GPL / AGPL — нет** (транзитивно тоже), приблизительный размер транзитивных зависимостей, поддержка `selectable` или обход через `SelectionArea`.

## Sanitization и безопасность URL

Требования **обязательны**, не «на усмотрение ревьюера».

1. **Схемы ссылок.** Любой интерактив или будущий tap (вне 11.6 запрещён) должен опираться на whitelist: разрешены только `http`, `https`, `mailto`. Проверка: `final u = Uri.tryParse(href);` затем `u != null && allowedSchemes.contains(u.scheme.toLowerCase())`. Схемы `javascript:`, `data:`, `file:`, пустая/невалидная схема — **не** обрабатывать как ссылку; рендерить как обычный текст (или экранированный вид без навигации).
2. **HTML в markdown.** Встроенный HTML в ответах LLM — вектор XSS и нежелательной вёрстки. Конфигурация парсера: **без** raw HTML (для `package:markdown` — не включать расширения с `inlineHtml`; для форков `flutter_markdown` — отключить HtmlExtension / эквивалент). В PR явно указать, какие extension sets включены.
3. **Картинки `![alt](url)`.** Загрузка внешних URL по сети **запрещена** (трекинг, неожиданный трафик). Варианты реализации: не регистрировать image builder; заменить на текст `alt` в скобках; или плейсхолдер с l10n. Сетевой `Image.network` для произвольного URL из чата — **нет**.
4. **Копирование кода.** Кнопка «Копировать» (если есть) вызывает `Clipboard.setData(ClipboardData(text: …))` с **исходным текстом тела fenced block** из markdown (сырой подстрочник из `content`), а не HTML и не собранный из виджетов фрагмент — чтобы в буфер не попали артефакты разметки.
5. **Ссылки без открытия.** Даже при валидном `https://` в 11.6 навигация не выполняется (см. таблицу решений).

**Связка п.1 и п.5:** whitelist схем в 11.6 нужен как **контракт на будущее**: визуально различать «похоже на допустимую ссылку» (accent-стиль) и «подозрительный href» (обычный текст), без навигации. Отдельная задача на открытие в браузере / in-app обязана **переиспользовать** ту же функцию проверки (условное имя `isAllowedHref(String href) → bool`), чтобы не дублировать правила.

## Функциональные требования

### Роли

| Значение `role` | Поведение внутри `ChatMessage` |
|-----------------|--------------------------------|
| `user` | Markdown + код; стиль пузыря задаёт родитель. |
| `assistant` | То же; основной сценарий длинных ответов. |
| `system` | То же; нейтральная типографика при необходимости. |

**Значение вне whitelist** (например будущий `tool`): в release рендер как для `system` (внутренние стили виджета); в `kDebugMode` — `assert` с понятным сообщением, чтобы рано ловить контракт. Родитель (`ChatScreen`) по-прежнему может маппить неизвестную роль на выравнивание/цвет пузыря — это отдельно от внутреннего fallback.

### Markdown (минимум)

- Параграфы, переносы, inline code `` `...` ``.
- Жирный и курсив — если поддерживает выбранный пакет из коробки.
- Списки — если есть; иначе ограничение перечислить в PR одной строкой.
- Ссылки: только безопасное отображение + sanitization по схемам; без открытия.

### Fenced code

- Тройные backticks, опциональный идентификатор языка.
- Моноширинный шрифт, фон из `ColorScheme`, скругление, горизонтальный скролл длинных строк: `SingleChildScrollView(scrollDirection: Axis.horizontal, primary: false, …)` (или эквивалент с `primary: false`), чтобы горизонтальная ось **не** забирала роль primary и не ломала вертикальный список чата.
- Подсветка синтаксиса — по возможности; если нет — явный визуальный отличитель от обычного текста (фон, padding).

### Edge cases

- **Незакрытый fence при стриме.** Политика: после первой открывающей последовательности ``` (с опциональным языком) весь **хвост** `content` от следующей за fence строки до **конца всего `content`** либо до появления **закрывающей** последовательности ``` считается частью **одного** многострочного блока кода в состоянии «ещё дописывается»; не переключать остаток в обычный текст до закрывающих backticks. Если пакет не позволяет — кастомный препроцессор или документированный shim перед парсером в том же PR.
- **`content.isEmpty` и `isStreaming: true`.** Показывать компактный индикатор набора текста (l10n, см. § Локализация) или курсор; не оставлять нулевую высоту без объяснения.
- **`content.isEmpty` и `isStreaming: false`.** Пустой виджет или минимальный spacer — согласовать с экраном; не падать.
- **Очень длинные токены** (URL, base64) в **теле** сообщения вне fenced block: **предпочтительно** `softWrap` и переносы (в т.ч. по символам для длинных URL), чтобы конец строки оставался доступен; горизонтальный скролл для всего тела — допустим только если перенос технически невозможен. **`ellipsis` для многострочного тела сообщения не использовать** (теряется хвост URL и контекст); `ellipsis` допустим только для **однострочных метаданных** (не для основного markdown-тела).
- **Кнопка «Копировать» и стрим.** По умолчанию кнопка **неактивна** при `isStreaming: true` (нет гонки с частичным fence). После `isStreaming: false` — копирование полного сырого тела блока из `content`. Отступление (копирование черновика) только с явной мотивацией в PR и отдельным тестом.

## Производительность и стрим

Формулировка «достаточно `StatelessWidget`» без кэша **противоречит** частым ребилдам при стриме: парсинг markdown на каждый кадр даёт jank.

**Обязательный минимум в связке 11.6 + 11.9:**

1. **Throttle на стороне 11.9:** обновления `content` одного и того же сообщения от потока — **не чаще 10 Гц** (интервал **≥ 100 ms** между применениями новой строки), если отдельная задача/PR 11.9 не вводит иной явный бюджет (тогда сослаться на него).
2. **Мемоизация разбора:** кэшировать по паре **(стабильный ключ сообщения, `content`)**. Для **подтверждённых** сообщений ключ — **`id`** из `ConversationMessageModel` (11.1). Для **pending** серверного `id` ещё нет — использовать **клиентский идентификатор** из потока **11.5** / **11.4** (например `clientMessageId` в `PendingMessage`); если кэш для pending не вводится в PR, явно обосновать: короткий ввод, редкие ребилды — **допустимо** без кэша для pending при сохранении throttle **11.9** для ассистента. Инструменты: `Provider`/`Notifier` с `select`, `StatefulWidget` с полем «последний распарсенный `content`» и инвалидацией при изменении. **Не** использовать `compute()` для AST/builder-дерева `flutter_markdown_plus` / аналогов, если объекты привязаны к `BuildContext` или не `Send`; изолят — только если пакет официально поддерживает чистый AST без контекста.
3. **`RepaintBoundary`** вокруг тела `ChatMessage` (внутри или снаружи — зафиксировать), чтобы стрим одного пузыря не перерисовывал всю ленту без необходимости.
4. **`const`** для неизменяемых дочерних виджетов и стилей, где позволяет язык.
