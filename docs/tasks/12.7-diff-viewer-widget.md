## Канон (SSOT)

| Тема | Решение |
|------|---------|
| **`diff` (проп)** | **`String`**, **non-null**. Пустую строку и строку из одних пробелов/переводов строк **call-site не передаёт** (остаётся **`taskDetailNoDiff`** в **12.5**). Внутри **`DiffViewer` повторный `trim` не выполняется** — упрощает контракт и тесты; переданная строка ровно та, что пришла с API после отбора ветки «есть diff». |
| Формат входа | Текст **unified diff** (как в **`git diff`**). Если признаки diff слабые — **§ «Нераспознанный ввод»**. Парсер обязан пережить **§ «Edge-cases unified diff»** без падения **`build`**. |
| Источник данных | Виджет **не** читает **`TaskModel`** / **`artifacts`**. Интеграция — только передача **`String`**. |
| Подсветка «строка добавлена» (**`+`**, не **`+++`**) | Фон **`colorScheme.primaryContainer`**, текст **`colorScheme.onPrimaryContainer`**. **Собственные RGB-пары не вводим** — пары из стандартного **Material 3 `ColorScheme`** в **`ThemeData(useMaterial3: true)`**; **§ «Контраст и WCAG»**. |
| Подсветка «строка удалена» (**`-`**, не **`---`**) | Фон **`colorScheme.surfaceContainerHighest`**, текст **`colorScheme.onSurface`**. **Не** семантизировать как ошибку (**не** **`errorContainer`**). **§ «Контраст и WCAG»**. |
| Тема (**Material 3**) | Виджет рассчитан на **`ThemeData(useMaterial3: true)`** (как в тестах и в приложении). Слоты вроде **`surfaceContainerHighest`** в **M2**-теме или устаревших схемах **могут отсутствовать или отличаться** — на **M2** корректные цвета **не гарантируются**; call-site обязан подавать **M3**-тему. |
| Прочие классы строк | **§ «Стили строк»**, **§ «Состояние парсера»**. Любые цвета — только **`Theme.of(context).colorScheme`** / **`textTheme`** — **без** **`Colors.*`** для семантики +/- . |
| Шрифт | Строго как в текущем **`_diffBody`**: **`Theme.of(context).textTheme.bodySmall`** + **`copyWith(fontFamily: 'monospace')`**. **`fontFamilyFallback`** и другие варианты **не** вводить без отдельной задачи/ADR. |
| Выделение и копирование | **Одно `SelectionArea`** вокруг всего блока содержимого diff — **канон**, без «эквивалентов» (**`SelectableRegion`** «голым» **не** подменяет **`SelectionArea`** — иной API выделения, чаще хуже для UX копирования; отдельная задача, если понадобится). Построчные **`SelectableText`** как основной режим **запрещены** (**§ Производительность**). Смена подхода только с письменным обоснованием и perf-замерами. |
| Прокрутка и размер | Высота блока: **`maxHeight`** по умолчанию = **`(MediaQuery.sizeOf(context).height * 0.4).clamp(180.0, 480.0)`** (логические пиксели). Явный проп **`maxHeight`** если задан — верхняя граница виджета; с **`null`** используется формула с **`clamp(180, 480)`**. Нижняя граница **180** предотвращает микроскопический блок на узких окнах; **480** — потолок без «неба» в колонке. Наружный **`CustomScrollView`** **12.5** уже скроллит: вложенный вертикальный скролл **`DiffViewer`** — с **фиксированной высотой** родителя / **`shrinkWrap`** только если иначе невозможно; цель — избежать неограниченного nested scroll без контракта по высоте. |
| Локализация | Строки для пользователя — **`.arb`**. Содержимое diff и префиксы **`+`/`-`** не переводятся. Пустое «нет diff» — только на экране (**`taskDetailNoDiff`**). Если нужна подпись «обрезано» — отдельный ключ; по умолчанию контент **не** обрезается (**§ «Производительность»**). |
| Импорты | Только **`package:frontend/...`**. |
| Зависимости pub | Новый пакет — только с обоснованием. По умолчанию — свой парсер + **`Theme`**. **В рамках 12.7** **`pubspec.yaml` не меняется** (ни хуки, ни подсветка, ни мемо-пакеты) — **§ Критерии приёмки**. |
| Доступность (a11y) | Текст diff читается скринридером как текст (**не** «декоративный» блок без семантики). Стили фона/цвета не должны удалять/перекрывать осмысленную озвучку: избегать лишних **`excludeSemantics`** на строках с содержимым; декоративные маркеры — по необходимости **`excludeSemantics: true`**, если дублируют уже озвученный **`Text`**. **`SelectionArea`** — проверить, что фокус/selection не ломают **`Semantics`** в типичном сценарии (см. тесты). |

### Производительность (не блокировать UI)

| Тема | Решение |
|------|---------|
| Рендер больших diff | Список распарсенных строк рендерить через **`ListView.builder`** (или **`SliverList`** в составе будущего рефактора — вне **12.7** если не нужно). **`Column` со всеми `Text`** для всего diff **запрещён** как основной режим для вывода тысяч строк. |
| Парсинг | **`DiffViewer`** — **`StatefulWidget`**: кэш результата парсинга в **`State`** (поле) + в **`didUpdateWidget`** сравнивать предыдущий **`widget.diff`** с новым; при равенстве — **не** перепарсивать. **`StatelessWidget` + `flutter_hooks` / `useMemoized`**, **riverpod**-хуки и **сторонние** пакеты мемоизации **запрещены** (новые зависимости и обход **§ pub**). |
| Обрезка контента | **По умолчанию не обрезаем:** полагаемся на виртуализацию + ограничение высоты блока. Сообщение «показано N из M строк» **не** вводится в **12.7**; если в проде появятся diff на порядок больше ожидаемого — отдельная задача. |
| Дымовой порог | На diff **≥ ~1000 строк** (или сгенерированная строка заданной длины) — **`pumpAndSettle`** без **`FlutterError`**, **`tester.takeException()`** **`null`** (**§ тестов**). |

### Контраст и WCAG (процедура, не «на глаз»)

- **Канон по умолчанию:** цвета — **только** стандартные пары **Material 3** из **`ColorScheme`** при **`useMaterial3: true`**. Собственные пары **не** вводим → полагаемся на WCAG-совместимость палитры **M3** у Flutter для типичных тем; **отдельный численный расчёт контраста в CI для задачи 12.7 не обязателен**.
- **Если в будущем** появятся **нестандартные** пары фон/текст в самом виджете (не слоты **M3**) — тогда вводится **unit/widget-хелпер** контраста **≥ 4.5:1**; до тех пор достаточно канона «стандартный **M3 ColorScheme**» из первого пункта этого раздела.

### Состояние парсера: вне hunk / внутри hunk (**---** / **+++** vs содержимое)

Парсер ведёт **явное состояние** по каждому текущему файлу в потоке unified diff:

1. **Вне hunk** (до первого **`@@`** после заголовков текущего файла): строки **`--- a/…`**, **`+++ b/…`** (в т.ч. **`/dev/null`**) — **заголовки файлов**, не строки удаления/добавления. Прочие строки метаданных — **§ «Стили строк»**.
2. После строки **`@@ … @@`** парсер переходит в режим **«внутри hunk»** до следующего **`@@`** или до следующего блока заголовков файла (**`diff --git`** / новый файл в том же тексте — по правилам unified diff для вашего парсера).
3. **Внутри hunk** классификация **только по первому значимому символу строки** после принятого unified-префикса: **` `** (пробел) → контекст, **`+`** → добавление, **`-`** → удаление. Любая строка, где после префикса hunk идёт **`+`** или **`-`**, трактуется так же — в том числе если текст содержимого начинается с **`---`** или **`+++`** (например удалённая строка документации **`---`**): это **не** заголовки **`--- a/`**, а **строка содержимого**; **не** переключать состояние «заголовок файла».
4. При выходе из hunk (новый **`@@`** или конец текущего файлового блока) состояние сбрасывается по контексту следующих строк.

Иначе два исполнителя получат **разные** парсеры на кейсе «внутри diff текст начинается с **`---`**».

### Стили строк (минимум)

Распознавание классов для строк **вне hunk** — по таблице ниже; для строк **внутри hunk** — **§ «Состояние парсера»** (первый символ **`+`/`-`/` `**).

| Префикс / паттерн | Смысл | Визуал |
|-------------------|-------|--------|
| **`diff --git`**, **`index `**, **`similarity index`**, **`rename from`**, **`rename to`**, **`Binary files … differ`**, **`Submodule …`**, **`Subproject commit`** | Метаданные | Текст **`colorScheme.onSurfaceVariant`** (без **`outline`** — один канон для всего блока метаданных; **§ тесты**: фикстуры rename/binary/submodule). |
| **`--- a/…`**, **`+++ b/…`** (в т.ч. **`/dev/null`**) | Заголовки файлов (**только в режиме «вне hunk»**, до первого **`@@`** для текущего файла) | **`onSurfaceVariant`** |
| **`@@ … @@`** | Начало hunk (переход в режим «внутри hunk») | Текст **`colorScheme.primary`** на фоне по умолчанию (прозрачный / surface родителя); **без** отдельного **`Container`** с заливкой под заголовок hunk — контраст через **`primary` vs surface**, без «или контейнер». |
| Строка добавления (**внутри hunk**) | Один префикс **`+`** (**§ «Состояние парсера»**) | **`primaryContainer` / `onPrimaryContainer`** |
| Строка удаления (**внутри hunk**) | Один префикс **`-`** (**§ «Состояние парсера»**) | **`surfaceContainerHighest` / `onSurface`** |
| Контекст (**внутри hunk**) | Префикс **` `** (пробел) | **`surface` / `onSurface`** (или фон по умолчанию карточки / прозрачный + **`onSurface`**) |
| **`\ No newline at end of file`** | Служебная git-строка | Как строка метаданных: **`onSurfaceVariant`** (**§ Стили строк**, строка метаданных); **не** как контекст кода; **не** падать. |
| **`Binary files … differ`** | Нет текстовых +/- | Показать строку(и) как метаданные; **не** пытаться разбирать как hunk. |

### Edge-cases unified diff (парсер и тесты)

Парсер **не бросает** в **`build`**; при сомнении классифицирует строку максимально консервативно (например как контекст/метаданные).

- **`--- /dev/null`** / **`+++ b/path`** / симметрично для нового файла — заголовки файлов, дальше **`+`** строки.
- **`--- a/path`** / **`+++ /dev/null`** — удалённый файл.
- **`Binary files a/foo and b/foo differ`** — только метаданные, без генерации фальшивых **`+`/`-`**.
- **`\ No newline at end of file`** — отдельная классификация (см. таблицу).
- **`Submodule …` / `Subproject commit <hash>`** — метаданные, как **`rename`** / **`similarity`**.
- **`similarity index`**, **`rename from`**, **`rename to`** — метаданные (**§ Стили строк**).
- **Внутри hunk строка, начинающаяся с `---` или `+++` как текст содержимого** (после префикса **`-`** / **`+`**) — **не** заголовок файла; ожидаем классификацию по **`§ Состояние парсера`** п. **3** — отдельная фикстура в тестах.

### Нераспознанный ввод

Если нет признаков unified diff (**`@@`**, **`diff --git`**, типичные пары **`---`/`+++`**):

- Рендер всего текста одним потоком моноширинным стилем (**без** подмены первых символов строк на **`+`/`-`**).
- **Не** пустой виджет, **не** исключение из **`build`**.

## Сценарии widget-тестов (`diff_viewer_test.dart`)

Окружение: **`MaterialApp`**, **`ThemeData(useMaterial3: true)`**, при необходимости локаль — если появятся l10n-ключи внутри виджета.

**Базовые:**
- Типичный unified diff — hunk и строки **`+`/`-`** находятся на дереве.
- Длинная строка без переносов — нет overflow; горизонтальный скролл при необходимости.
- Plain text без признаков diff — рендер без исключения.
- **`ThemeData(brightness: Brightness.dark)`** — без **`FlutterError`** при том же diff.

**Контракт и layout:**
- Явный **`maxHeight`**: **`tester.getSize(find.byType(DiffViewer)).height ≤ maxHeight + 2.0`** (логические px). Допуск **не более 2 px** только на округление субпикселей; **без** иных «допусков по согласованию с реализацией».
- **`maxHeight == null`**: высота блока **≤** **`min(480, max(180, screenHeight * 0.4)) + 2.0`** при том же допуске **2 px** (фиксированный **`MediaQuery`** в тесте).

**Копирование / selection:**
- После **`pump`** на дереве есть **`SelectionArea`** ( **`find.byType(SelectionArea)`** ); исключений при базовом взаимодействии нет.

**Цвета из темы (один способ, детерминированно):**
- Для строк **`+`/`-`** найти обёртку с фоном (**`Container`**, **`DecoratedBox`**, **`Material`** и т.д. — по фактической разметке PR) и сравнить **`color`** с **`Theme.of(context).colorScheme.primaryContainer`** / **`surfaceContainerHighest`** соответственно (через **`tester.element(find…)`** и контекст темы).
- Два прогона с разными **`ThemeData`** / **`ColorScheme`**: после **`pump`** цвет фона строк **`+`/`-`** меняется согласно активной схеме (регресс «залипания» одного RGB).
- Минимальный барьер от регресса: в **`diff_viewer.dart`** **нет** литералов **`Color(0xFF…)`** и произвольных RGB для подсветки (**`grep` / ревью**).

**Парсер / hunk:**
- Фикстура: **внутри hunk** строка удаления, где содержимое начинается с **`---`** (или добавления с **`+++`**) — классификация по **`§ Состояние парсера`**, **не** как **`--- a/`** заголовок.

**Большой diff:**
- Сгенерировать **≥ 1000** строк (или согласованный порог из **§ Производительность**) — **`pumpAndSettle`**, **`tester.takeException()`** **`null`**.

**Edge-cases (отдельные мини-фикстуры):**
- **`Binary files … differ`**
- **`--- a/x` / `+++ /dev/null`** и обратный кейс
- **`rename from` / `rename to`**, **`similarity index`**
- **`Subproject commit` / submodule**
- **`\ No newline at end of file`**
- **Строка в hunk с текстом, начинающимся с `---` / `+++`** (**§ Edge-cases**, пункт про содержимое)

## Частые ошибки

1. **`trim` внутри `DiffViewer`** при контракте «call-site фильтрует пустое».
2. **`Colors.green` / `Colors.red`**.
3. **`Column` со всеми строками** на больших diff.
4. Парсинг при каждом **`build`** без мемоизации.
5. **`SelectableText` построчно** по умолчанию.
6. Классификация **`---`** внутри hunk как заголовок файла (**§ Состояние парсера**).
7. **`StatelessWidget` + hooks** вместо **`StatefulWidget` + State**.
8. Относительные импорты **`../../`**.
9. **`errorContainer`** для удалённых строк.
10. Изменение **`pubspec.yaml`** без смены статуса задачи / ADR.
11. Зависимость pub без обоснования.
