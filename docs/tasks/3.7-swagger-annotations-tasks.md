## Описание

> **⚠️ Эта задача поглощена задачей 3.5 (`TaskHandler`).** Swagger-аннотации пишутся непосредственно в handler при его создании, а не как отдельный шаг. Данный документ служит **чеклистом ревизии** — после реализации 3.5 проверить, что аннотации соответствуют стандарту из [2.9-swagger-annotations.md](2.9-swagger-annotations.md).

## Стандарт авторизации (из задачи 2.9)

Каждый защищённый эндпоинт **ОБЯЗАН** иметь три `@Security` на **отдельных строках** (OR-семантика в swaggo):

```go
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
```

**Почему OR, а не AND:** Три строки `@Security` в swaggo означают «клиент может использовать **любой** из трёх способов». Если записать в одну строку через запятую — это AND (нужны все три одновременно), что неправильно.

**Swagger UI при этом показывает:**
1. **BearerAuth** — ручной ввод `Bearer <token>`
2. **ApiKeyAuth** — ввод API-ключа (header `X-API-Key`)
3. **OAuth2Password** — форма логин/пароль → автоматическое получение JWT

## Полный каталог аннотаций для TaskHandler (10 эндпоинтов)

### 1. Create (`POST /projects/{id}/tasks`)

```go
// Create создаёт задачу в проекте
// @Summary Создание задачи
// @Description Создаёт задачу в указанном проекте. Статус устанавливается pending.
// @Tags tasks
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
// @Accept json
// @Produce json
// @Param id path string true "Project ID"
// @Param request body dto.CreateTaskRequest true "Данные задачи"
// @Success 201 {object} dto.TaskResponse
// @Failure 400 {object} apierror.ErrorResponse "Невалидный JSON, пустой title, невалидный priority"
// @Failure 401 {object} apierror.ErrorResponse "Не авторизован"
// @Failure 403 {object} apierror.ErrorResponse "Нет доступа к проекту"
// @Failure 404 {object} apierror.ErrorResponse "Проект не найден"
// @Failure 422 {object} apierror.ErrorResponse "Агент не в команде / parent task не найден"
// @Failure 500 {object} apierror.ErrorResponse "Внутренняя ошибка"
// @Router /projects/{id}/tasks [post]
```

### 2. List (`GET /projects/{id}/tasks`)

```go
// List возвращает список задач проекта
// @Summary Список задач проекта
// @Description Возвращает задачи проекта с фильтрацией и пагинацией
// @Tags tasks
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
// @Produce json
// @Param id path string true "Project ID"
// @Param status query string false "Фильтр по статусу"
// @Param statuses query []string false "Фильтр по нескольким статусам" collectionFormat(multi)
// @Param priority query string false "Фильтр по приоритету"
// @Param assigned_agent_id query string false "Фильтр по агенту"
// @Param created_by_type query string false "Фильтр по типу создателя (user/agent)"
// @Param created_by_id query string false "Фильтр по ID создателя"
// @Param parent_task_id query string false "Фильтр по parent задаче"
// @Param root_only query bool false "Только корневые задачи (без parent)"
// @Param branch_name query string false "Фильтр по git-ветке"
// @Param search query string false "Поиск по title/description"
// @Param limit query int false "Лимит (1–200, по умолчанию 50)"
// @Param offset query int false "Смещение"
// @Param order_by query string false "Поле сортировки"
// @Param order_dir query string false "Направление сортировки (asc/desc)"
// @Success 200 {object} dto.TaskListResponse
// @Failure 400 {object} apierror.ErrorResponse "Невалидные query params"
// @Failure 401 {object} apierror.ErrorResponse "Не авторизован"
// @Failure 403 {object} apierror.ErrorResponse "Нет доступа к проекту"
// @Failure 404 {object} apierror.ErrorResponse "Проект не найден"
// @Failure 500 {object} apierror.ErrorResponse "Внутренняя ошибка"
// @Router /projects/{id}/tasks [get]
```

> **Примечание по `statuses []string`:** Gin парсит массив из query string в формате `?statuses=in_progress&statuses=review` (повторение ключа). В DTO используется тег `form:"statuses"` — Gin `ShouldBindQuery` корректно заполнит `[]string`. Директива `collectionFormat(multi)` в Swagger-аннотации указывает swaggo генерировать UI с поддержкой этого формата (множественные отдельные параметры, а не через запятую). При реализации стоит проверить, что Swagger UI отображает поле ввода с возможностью добавить несколько значений.

### 5. Delete (`DELETE /tasks/{id}`)

```go
// Delete удаляет задачу
// @Summary Удаление задачи
// @Description Удаляет задачу (каскадно удаляет messages, SET NULL для подзадач)
// @Tags tasks
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
// @Produce json
// @Param id path string true "Task ID"
// @Success 204 "Задача успешно удалена"
// @Failure 400 {object} apierror.ErrorResponse "Невалидный UUID"
// @Failure 401 {object} apierror.ErrorResponse "Не авторизован"
// @Failure 403 {object} apierror.ErrorResponse "Нет доступа к проекту задачи"
// @Failure 404 {object} apierror.ErrorResponse "Задача не найдена"
// @Failure 500 {object} apierror.ErrorResponse "Внутренняя ошибка"
// @Router /tasks/{id} [delete]
```

> **Примечание по 204 vs 200:** В проекте `ProjectHandler.Delete` возвращает `200 OK` + JSON-сообщение (сложившаяся конвенция). Для tasks мы переходим на строгий REST — `204 No Content` без тела ответа. В handler'е: `c.Status(http.StatusNoContent)` вместо `c.JSON(http.StatusOK, ...)`. При желании можно в будущем рефакторить `ProjectHandler.Delete` на 204 для консистентности.

## Нюансы @Router путей

**КРИТИЧНО:** В `@Router` **НЕ УКАЗЫВАТЬ** префикс `/api/v1` — он определён в `@BasePath` в `main.go`. Swagger автоматически собирает: `host` + `basePath` + `path`.

| Эндпоинт | Полный URL | @Router |
|----------|-----------|---------|
| Create | `/api/v1/projects/{id}/tasks` | `/projects/{id}/tasks [post]` |
| List | `/api/v1/projects/{id}/tasks` | `/projects/{id}/tasks [get]` |
| GetByID | `/api/v1/tasks/{id}` | `/tasks/{id} [get]` |
| Update | `/api/v1/tasks/{id}` | `/tasks/{id} [put]` |
| Delete | `/api/v1/tasks/{id}` | `/tasks/{id} [delete]` |
| Pause | `/api/v1/tasks/{id}/pause` | `/tasks/{id}/pause [post]` |
| Cancel | `/api/v1/tasks/{id}/cancel` | `/tasks/{id}/cancel [post]` |
| Resume | `/api/v1/tasks/{id}/resume` | `/tasks/{id}/resume [post]` |
| ListMessages | `/api/v1/tasks/{id}/messages` | `/tasks/{id}/messages [get]` |
| AddMessage | `/api/v1/tasks/{id}/messages` | `/tasks/{id}/messages [post]` |

## @Tags: группировка в Swagger UI

| Tag | Эндпоинты | Описание |
|-----|-----------|----------|
| `tasks` | Create, List, GetByID, Update, Delete, Pause, Cancel, Resume | CRUD + action'ы |
| `task-messages` | ListMessages, AddMessage | Вложенный ресурс сообщений |

**Почему два тега:** В Swagger UI эндпоинты группируются по тегам. Если все 10 будут под `tasks`, список станет громоздким. Сообщения — логически отдельная сущность, заслуживают свою секцию.

## Описания @Failure (человекочитаемые)

Каждый `@Failure` **ОБЯЗАН** иметь описание в кавычках. Это отображается в Swagger UI как подсказка для разработчика.

| HTTP | Типичное описание |
|------|-------------------|
| 400 | Конкретная причина: "Невалидный JSON", "Пустой title", "Невалидный UUID" |
| 401 | "Не авторизован" |
| 403 | "Нет доступа к проекту задачи" |
| 404 | "Задача не найдена" или "Проект не найден" |
| 409 | Конкретная причина: "Недопустимый переход статуса", "Задача в терминальном статусе" |
| 422 | "Агент не в команде проекта" или "Parent task не найден" |
| 500 | "Внутренняя ошибка" |

## Проверка после реализации

### Команды

```bash
make swagger
# Открыть http://localhost:8080/swagger/index.html
```

### Чеклист ревизии

1. **Authorize работает для tasks:**
   - [ ] OAuth2Password: логин → `GET /projects/{id}/tasks` → 200
   - [ ] BearerAuth: вставить токен → `GET /tasks/{id}` → 200 (или 404 если задачи нет)
   - [ ] ApiKeyAuth: вставить ключ → `POST /tasks/{id}/pause` → 200/409

2. **Группировка в UI:**
   - [ ] Секция "tasks" содержит 8 эндпоинтов (CRUD + actions)
   - [ ] Секция "task-messages" содержит 2 эндпоинта (ListMessages, AddMessage)

3. **@Param корректны:**
   - [ ] `id` в `/projects/{id}/tasks` описан как "Project ID"
   - [ ] `id` в `/tasks/{id}` описан как "Task ID"
   - [ ] Query params для List содержат все 13 фильтров из `dto.ListTasksRequest`
   - [ ] Query params для ListMessages содержат все 4 фильтра из `dto.ListTaskMessagesRequest`

4. **@Failure описания:**
   - [ ] Каждый @Failure имеет человекочитаемое описание в кавычках
   - [ ] 409 присутствует только у Update, Pause, Cancel, Resume
   - [ ] 422 присутствует только у Create и Update

5. **Отсутствие /api/v1 в @Router:**
   - [ ] Ни один @Router не начинается с `/api/v1/`

## Критерии приёмки

Задача считается выполненной **автоматически** при приёмке задачи 3.5 (`TaskHandler`), если:

- [ ] Все 10 эндпоинтов имеют `@Security BearerAuth`, `@Security ApiKeyAuth`, `@Security OAuth2Password` (каждый на **отдельной строке**)
- [ ] `@Tags tasks` для CRUD + actions, `@Tags task-messages` для сообщений
- [ ] `@Router` пути **без** префикса `/api/v1`
- [ ] `@Param id` — "Project ID" для `/projects/{id}/tasks`, "Task ID" для `/tasks/{id}`
- [ ] `@Failure` описания человекочитаемые, 409 и 422 только у соответствующих эндпоинтов
- [ ] `@Success 201` для Create и AddMessage, `@Success 204` для Delete, `@Success 200` для остальных
- [ ] `@Accept json` только у Create, Update, AddMessage (остальные не принимают body)
- [ ] `make swagger` проходит без ошибок
- [ ] Swagger UI: секция "Authorize" показывает 3 способа
- [ ] Swagger UI: после OAuth2Password-логина task-эндпоинты работают
