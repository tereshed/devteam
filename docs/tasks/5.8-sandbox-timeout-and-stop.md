# 5.8 Таймаут и принудительная остановка (бизнес-таймер + `Stop`)

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/docker_runner.go` — **тонкий** слой: **`RunTask`/`Stop`/`Wait`**, сборка **`lifecycleManager`** с **реализацией** узкого интерфейса поверх **`*client.Client`** (ориентир **`docker_container_stopper.go`** или методы на **`DockerSandboxRunner`**); **`lifecycle_manager.go`** — таймер (**`time.Timer`**), стейт-машина, **без прямой зависимости от Docker SDK**; **`docker_runner_state.go`** — **`instanceState`** (см. ниже); поле и хелпер grace — **`types.go`**, **`options_validate.go`** (**5.4**).  
**Статус:** ⬜ Не выполнена

---

## Описание

Сформулировать и реализовать полный контракт **жёсткого бизнес-таймаута** жизни задачи в изоляции после успешного запуска контейнера: по истечении **`EffectiveTimeout()`** раннер **обязан** перевести инстанс в **`SandboxStatusTimedOut`** и **принудительно** остановить контейнер (без «тихого» зависания агента). Отдельно — **`SandboxRunner.Stop`**: остановка по запросу пользователя/оркестратора с итогом **`SandboxStatusStopped`**, цепочка **SIGTERM** (grace) → **SIGKILL** при необходимости, **согласованность с таймером** (нет ложного **`timed_out`** при ручном стопе и гонках с колбэком дедлайна).

**Границы задачи:** политика таймера, **единое** управляемое **мьютексом** состояние жизненного цикла инстанса (не несколько независимых атомиков — см. раздел «Состояние и гонки»), **единый DRY-хелпер** остановки контейнера, отмена бизнес-таймера при **`Stop`** и в пути **`containerWaitLoop`**, **отмена контекста `ContainerWait`**, если инстанс переведён в терминальное состояние при «зависшем» демоне (см. раздел «Горутина ожидания»). Лимиты cgroup — **5.9**; сбор артефактов — **5.7**; стрим логов — **5.6**.

## Архитектура: декомпозиция (God Object, `review.mdc`)

**Проблема:** **`DockerSandboxRunner`** не должен разрастаться логикой старта, стрима (**5.6**), артефактов (**5.7**) **и** сложного lifecycle (таймеры, сигналы, гонки) в одном типе.

**Требование:** логика **таймера бизнес-дедлайна**, **принудительной остановки** (через **инъецированный** низкоуровневый контракт) и **инвариантов состояния** инкапсулируется в **отдельной сущности** (ориентир: **`lifecycleManager`** в отдельном файле пакета **`sandbox`**). **`DockerSandboxRunner`** передаёт **ссылку на `instanceState`**, идентификаторы и реализацию **узкого интерфейса** (см. ниже), **не** **`*client.Client`** напрямую в **`lifecycleManager`**.

### Инверсия зависимостей: интерфейс вместо SDK (**тестируемость**, Rule 4 **`review.mdc`**)

**Запрещено:** тип **`lifecycleManager`** (или эквивалент) **напрямую** импортировать/хранить **`*client.Client`** из **`github.com/docker/docker/client`** — иначе юнит-тесты стейт-машины (гонки таймера, **`Stop` до старта**, ручной стоп, отмена **`ContainerWait`**) потребуют **живой Docker-демон**.

**Требование:** **`lifecycleManager`** зависит только от **узкого интерфейса**, например:

```go
// Имена методов — на усмотрение PR; смысл: Stop с grace + Kill SIGKILL с ctx.
type ContainerStopper interface {
	StopContainer(ctx context.Context, containerID string, graceSeconds int) error
	KillContainer(ctx context.Context, containerID, signal string) error
}
```

**`*client.Client`** остаётся в **`DockerSandboxRunner`** (или в малом адаптере **`type dockerStopper struct { cli *client.Client }`**) и **реализует** этот интерфейс вызовами SDK. **5.13**: мок **`testify/mock`** на **`ContainerStopper`** покрывает всю логику **`lifecycleManager`** в памяти, **без** демона.

### DRY: один путь к Docker для остановки (**`review.mdc`**, Rule 2)

**Запрещено** дублировать цепочку **`ContainerStop`** (grace / нулевой grace) → фоллбэк **`ContainerKill`**, отдельные контексты **`WithTimeout`/`detachTimeout`**, разбор ошибок демона **в двух местах** (колбэк бизнес-таймера и тело **`Stop`**).

**Требование:** одна реализация цепочки **Stop → Kill** на адаптере **`ContainerStopper`** (ориентир: **`(*dockerStopper) ForceStop(ctx, id, graceSec, reason)`** или пакетная **`forceStop(...)`**), вызываемая из **`DockerSandboxRunner.Stop`** и из пути таймаута; **`lifecycleManager`** вызывает **только** **`ContainerStopper`**, без дублирования SDK. Параметры: **grace** для ручного стопа vs **0** для бизнес-deadline, **`reason`** для логов — **без** полного **`SandboxOptions`** (см. раздел «Логи и секреты»).

### Читаемость: комментарии «ПОЧЕМУ», не «ЧТО» (Rule 7 **`review.mdc`**)

Код **`lifecycle_manager.go`** (порядок захвата **`mu`**, **`timer.Stop`**, колбэк **`time.Timer`**, **`cancelWait`**, откат при **`creating`**) **обязан** сопровождаться **комментариями с rationale**: зачем лок взят **до** чтения поля, почему порядок с **`streamMu`** именно такой, почему в колбэке сначала **`mu`** (например: согласование с **`Stop`** и исключение **TOCTOU**).

**Запрещён** шум без смысла вроде «`// Блокируем мьютекс`» или «`// Проверяем флаг`».

## Состояние и гонки: мьютекс vs разрозненные атомики (**КРИТИЧНО**)

**Запрещено** для финального автомата инстанса опираться на **несколько независимых атомиков** (**`timedOut`**, **`stoppedByRunner`**, **`cleaned`** и т.д.) без **единой** точки согласования: это классический **TOCTOU** (проверили один атомик — другой изменился до использования) и ломает инварианты между **`composeFinalStatus`**, колбэком **`time.Timer`** и **`Stop`**.

**Требование:** поля, описывающие **намерение и фазу** остановки (**ручной стоп**, **бизнес-таймаут**, **уборка завершена**, при необходимости — **«kill уже отправлен»**), хранить под **`sync.Mutex`** (или **`RWMutex`**, если докажете безопасность читателей) **внутри** **`instanceState`** (или внутри выделенного **`lifecycleManager`**, если стейт переносится туда — главное: **один** мьютекс на связный набор полей).

Ориентир структуры (имена полей — на усмотрение PR, смысл — обязателен):

```go
type instanceState struct {
	mu sync.Mutex
	// lifecycle: running → stopping / timed_out intent / stopped intent → …
	// timedOut, stoppedByUser, cleaned, killSent — только под mu
	// ...
}
```

**`composeFinalStatus`**, колбэк таймера и **`Stop`** при чтении/записи этих полей **обязаны** соблюдать один и тот же порядок захвата **`mu`** (без взаимных локов с **`streamMu`** — зафиксировать иерархию локов в комментарии к коду: например **«сначала lifecycle `mu`, потом streamMu»** или наоборот, **никогда** в обратном порядке).

Публичные **`SandboxStatusType`** для API по-прежнему выводятся из **согласованного** снимка: **`ContainerWait`/`Inspect`** + защищённое мьютексом **намерение** (см. раздел про гонку с нормальным **exit**).

## Бизнес-таймаут (длительность и принудительный kill)

| Требование | Поведение |
|------------|-----------|
| Длительность | **`opts.EffectiveTimeout()`**; при **`Timeout <= 0`** — **`DefaultSandboxTimeout`** (**5.4**). |
| Старт | После **`ContainerStart`** + sanity (**например `postStartSanity`**). Планирование — через **`lifecycleManager`** (или эквивалент). |
| Истечение | Под **`mu`**: зафиксировать **намерение бизнес-таймаута**, затем вызвать остановку через **`ContainerStopper`** (единая DRY-цепочка, grace **0** / kill-path по политике **5.5**), **не** копировать вызовы SDK в колбэке. |
| Нормальный exit | Отмена таймера + сброс полей под **`mu`** в пути **`containerWaitLoop`**; плюс проверки в колбэке (**см. ниже**). |

## Семантика `time.Timer` в Go (обязательно к реализации)

**`timer.Stop()` недостаточно:** колбэк может выполниться после внешней отмены таймера.

**Требование:** в **начале** колбэка — **захват `mu`**, чтение связного стейта (**уже очищен**, **ручной стоп зафиксирован**, **контейнер финализирован wait-ом** — по политике полей) и **ранний `return`** без Docker и без выставления **намерения таймаута**, если сценарий не применим. Внешняя **`stopBusinessTimer`** остаётся обязательной, но **не заменяет** проверку под **`mu`**.

## Контракт: `Stop(ctx, sandboxID string) error`

### Ранняя валидация (early return, без Docker)

| Условие | Поведение |
|---------|-----------|
| **`sandboxID == ""`**, пробелы/мусор, нарушение формата (**`ValidateSandboxID`**, **`validate.go`**) | **`ErrInvalidSandboxID`**, **до** мапы и Docker. **КРИТИЧНО (DoS):** первая проверка в **`ValidateSandboxID`** — **константное по времени ограничение длины** (**`len(id) == 64`** для текущего контракта **5.5** **или** **`len(id) > max` → fail**), **строго до** любых **регулярок** и **до** побайтового прохода по потенциально огромной строке (ReDoS / линейные аллокации от злоумышленного ввода). |
| Валидный ID, но раннер **не** может сопоставить инстанс (нет в реестре активных / политика **5.5** без adopt) | **`ErrSandboxNotFound`** (или **`nil`** для идемпотентности **после полной уборки** — **один** вариант на весь пакет, задокументировать в **5.3** и здесь). |
| Повторный **`Stop`** при уже завершённом контейнере | Идемпотентность (см. ниже); **не** дергать Docker с пустым/невалидным ID. |

### Остальное поведение `Stop`

| Требование | Поведение |
|------------|-----------|
| Стейт | Под **`mu`**: зафиксировать **ручной стоп** (**без** флага бизнес-таймаута). |
| Таймер | Отменить бизнес-таймер (**`timer.Stop`** + сброс ссылки под тем же **`mu`**, если таймер принадлежит этому инстансу). |
| Docker | Только **после** валидации ID и разрешения инстанса — **та же** DRY-цепочка через **`ContainerStopper`** с **`EffectiveStopGrace()`**; при отмене **`ctx`** вызова **`Stop`** — best-effort **`Kill`** внутри неё (**`defer cancel()`** — см. «Контексты»). |
| **Идемпотентность** | Повторный **`Stop`**, **`Stop`** после таймаута — **без panic**; ожидаемые ошибки движка (**not found**, уже остановлен) — **без спама WARN** на каждый вызов; возврат **`nil`** vs sentinel — согласовать с таблицей выше и **5.3**. |

### Grace period

Как в первой итерации ревью: **`SandboxOptions.StopGracePeriod`**, **`EffectiveStopGrace()`**, дефолт **`DefaultSandboxStopGrace`** в **5.4**, **`Clone`/`Validate`**.

### Гонка «таймер vs `Stop`»

Под **`mu`**: если **ручной стоп** записан раньше **намерения таймаута**, колбэк при входе **не** эскалирует сценарий в **timed_out**.

### Гонка «таймер / `Stop` vs нормальное завершение контейнера»

Без изменения смысла первой итерации: **первичны** **`ContainerWait`** / post-wait **`Inspect`** для **`exited`** и **`ExitCode`**, чтобы **не** классифицировать успешный **`exit 0`** как **`timed_out`/`stopped`** только из-за гонки записи намерений; **OOM** — исключение, остаётся **строже**. Таблица кодов **137** и т.д. — **5.13**.

### `GetStatus` / `Wait`

Согласовать приоритет **снимка движка** vs **намерений под `mu`**; одинаковая семантика для одного инстанса после финала.

## `Stop` во время инициализации (**`Creating`**, pull, `ContainerCreate`) — anti-zombie (**`review.mdc`**, Rule 3)

**Проблема:** таймер в ТЗ стартует после **`ContainerStart`**, но **`Stop(sandboxID)`** может прийти **раньше**: долгий **`ImagePull`**, **`NetworkCreate`**, **`ContainerCreate`**, **`CopyToContainer`**. Если **`Stop`** вернёт **`nil`/`ErrSandboxNotFound`** (инстанса ещё нет в **`instances`**), а **`RunTask`** потом успешно стартует контейнер — оркестратор считает задачу остановлённой, а контейнер **жив** (**«зомби»**).

**Требование:**

1. Ввести (или явно использовать) фазу **`creating` / initializing`** для записи в **`creating`**-мапе **5.5** **до** долгих операций: **`Stop`** по **`TaskID`** или по **детерминированному имени**/связке (**политика одна на PR**) должен иметь возможность пометить **«отмена инициализации»** под **`mu`**.
2. **`RunTask`** на **каждом** долгом шаге (**после** точки регистрации в **`creating`**) **обязан** проверять флаг отмены; при **`Stop` до старта** — **early return** с ошибкой (**новый sentinel** или **`ErrSandboxStopped`** / контекстная ошибка — зафиксировать в **`errors.go`**), **синхронная уборка**: удалить частично созданные сеть/контейнер/временные пути, **не** выполнять **`ContainerStart`**.
3. **Ошибки отката (антипаттерн «`_ =`»**, Rule 5 **`review.mdc`):** при уборке «недосозданного» инстанса **запрещено** безмолвно игнорировать ошибки (**`_ = removeNetwork()`** и т.п.). Любая ошибка **`NetworkRemove`/`ContainerRemove`/`RemoveAll`** на откате **обязательно** логируется (**`slog`** **WARN** или **ERROR** в зависимости от критичности), с **безопасными** полями (**`TaskID`**, имена ресурсов), иначе на хосте накапливаются **невидимые** сети/volume/мусор в **`/tmp`**, которые невозможно отследить в проде.
4. Если **`sandboxID`** известен только после **`ContainerCreate`**, до этого момента отмена идёт по **`TaskID`** (или эквивалент, согласованный с **5.5**); после появления ID — **`Stop`** по **`sandboxID`** продолжает ту же цепочку отмены.

**Критерий успеха:** нет сценария «**`Stop`** вернул успех/не найден, а контейнер для той же задачи позже оказался **running** без явного нового **`RunTask`**.

## Контексты: отсутствие утечек (**Resource Leaks**, `review.mdc`)

Любой **оторванный** от **`ctx`** вызывающего контекст для **`ContainerStop`/`ContainerKill`** (**`context.WithTimeout(context.Background(), d)`**, **`detachTimeout`**) **обязан** иметь пару **`defer cancel()``** сразу после успешного создания, **на том же уровне стека**, где создаётся **`cancel`** (не «потерять» **`cancel`** на ветке **`return`**). Иначе — **утечка таймера дедлайна контекста** до истечения **`d`**.

Шаблон (обязателен по смыслу в реализации):

```go
stopCtx, cancel := context.WithTimeout(context.Background(), d)
defer cancel()
// … ContainerStop(stopCtx, …)
```

## Горутина `containerWaitLoop` и отмена `ContainerWait` (**`review.mdc`**, Rule 3)

**Проблема:** при зависании **dockerd** **`ContainerKill`** может завершиться по **`detachTimeout`** с ошибкой, а горутина, заблокированная на **`client.ContainerWait`**, **никогда** не получит ответ и **утечёт** (или **`Wait`** пользователя повиснет).

**Требование:**

1. Контекст, переданный в **`ContainerWait`** (или обёртку SDK), **не** должен быть «вечным **`Background`** без отмены»: связать с **жизненным циклом инстанса** — **`context.WithCancel`**, хранить **`cancelWait`** под **`mu`** (или внутри **`lifecycleManager`**).
2. При переходе в **терминальное** состояние по политике раннера (**`Stopped`**, **`TimedOut`**, **`Cleanup`**, ошибка kill после таймаута демона — перечень зафиксировать в коде) **обязательно** вызывать **`cancelWait()`**, чтобы **`ContainerWait`** отпустил горутину (ошибка отмены контекста обрабатывается как завершение ожидания + уборка без двойного **`closeDone`**).
3. **`drainDockerWait`** (если остаётся) **не** заменяет отмену **`Wait`** при «зомби»-ожидании: отмена **первична** для освобождения горутины.

## Безопасность: логи без утечки секретов (**КРИТИЧНО**, Rule 1 **`review.mdc`**)

В **`SandboxOptions`** (**5.4**) лежат **`EnvVars`** (API-ключи Anthropic/OpenAI, GitHub и т.д.). При таймауте, **`Stop`**, ошибках kill и **`ERROR`** инфраструктуры разработчики часто пишут **`log.Printf("… %+v", opts)`** или **`slog`** с полем **`opts`**.

**Запрещено:** логировать **`SandboxOptions` целиком** (**`%+v`**, **`%#v`**, **`slog.Any("opts", opts)`** без **`LogValuer`**), а также сырые **`EnvVars`**, **`Instruction`/`Context`**, полный **`RepoURL`** с кредами.

**Разрешено:** **`TaskID`**, **`ProjectID`** (если есть), **`sandboxID`/`containerID`**, **`reason`** (строка-enum: **`timeout`**, **`user_stop`**, **`init_cancel`**), длительности/факты события. Для структурированного лога опций — только **`opts.LogSafe()`** / **`String()`** из **`options_log.go`** (**5.4**), без расширения полей секретами.

**Изоляция контекстов:** операции к демону, которые не должны обрываться отменой **`RunTask`**, выполнять с отдельным **`WithTimeout`/`detachTimeout`** и **`defer cancel()`** (см. выше). **Отмена `ContainerWait`** — см. раздел «Горутина ожидания».

## Унификация ошибок и инфраструктура vs бизнес-исход (`review.mdc`)

| Ситуация | Поведение |
|----------|-----------|
| Ожидаемые ответы Docker при **корректной** остановке (**уже остановлен**, **not found**) | Идемпотентность: **не** считать **`ErrSandboxDocker`** для пользовательского пути; **не** засорять **WARN**. |
| **`ContainerStop`/`Kill`** вернули ошибку из-за **истечения нашего внутреннего** **`stopCtx`** (демон «повис», не ответил до дедлайна) | Это **не** то же самое, что «бизнес-таймаут задачи истёк». Логировать минимум на **`ERROR`** (**`slog`**) как **критический сбой инфраструктуры** (зависание/недоступность движка при kill) **только с безопасными полями** (см. «Логи без утечки секретов»); цепочку **`error`** — **server-side**; для вызывающего API — по политике **5.3** (**обёртка `ErrSandboxDocker`** или отдельный sentinel на усмотрение PR, но **не** молчаливое проглатывание без лога). После этого — **отменить `ContainerWait`**, чтобы не оставить висящую горутину. |
| Прочие ошибки SDK | Как в **5.5**/**5.6**: **`errors.Join(ErrSandboxDocker, …)`** где уместно. |

## Критерии приёмки

- [ ] Нет **разрозненных атомиков** для связного lifecycle-стейта: **один `mu`**, согласованные поля, документированный порядок с **`streamMu`**.
- [ ] Логика таймера/kill/stop вынесена из раздутого **`docker_runner.go`** в **отдельный** тип/файл (**`lifecycle_manager.go`** или эквивалент); раннер **делегирует**.
- [ ] **`lifecycleManager`** зависит от **`ContainerStopper`** (или эквивалента), **не** от **`*client.Client`**; адаптер + **5.13**: моки **`testify/mock`**, полное покрытие стейт-машины **без** Docker.
- [ ] **DRY:** одна реализация цепочки **Stop/kill** на адаптере **`ContainerStopper`**; колбэк таймера и **`Stop`** не дублируют SDK.
- [ ] **`lifecycle_manager.go`**: комментарии **«ПОЧЕМУ»** к порядку локов, колбэку таймера и **`cancelWait`** (Rule 7).
- [ ] **`Stop` во время инициализации:** отмена под **`mu`**, **`RunTask`** проверяет флаг между долгими шагами, уборка без **«зомби»**-контейнера; ошибки отката — **WARN/ERROR** в лог, **без** молчаливого **`_ =`**.
- [ ] **`ContainerWait`:** отменяемый контекст + **`cancelWait`** при терминальном состоянии / после неудачного kill при зависании демона; горутина ожидания **не** течёт бесконечно.
- [ ] **`ValidateSandboxID`**: **O(1)** по длине (**`len` / верхняя граница до** тяжёлого парсинга); поведение «инстанс неизвестен» — по таблице; **нет** RPC с пустым/невалидным ID.
- [ ] **Логи:** нет **`%+v`/`%#v`/`opts`** с секретами; только безопасные идентификаторы + **`LogSafe()`** при необходимости.
- [ ] Каждый **`WithTimeout`/`detachTimeout`** для оторванных вызовов — с **`defer cancel()`**.
- [ ] Разделение: **бизнес-таймаут** vs **таймаут нашего контекста при kill зависшего демона** — разный лог и семантика (**ERROR** инфраструктуры для второго).
- [ ] Семантика **`time.Timer`**: проверки **под `mu`** в начале колбэка; **`stopBusinessTimer`** при **`Stop`** и в **`defer`** wait-пути.
- [ ] Идемпотентность **`Stop`** и колбэка; политика **нормальный exit vs намерения** + тесты **5.13**.
- [ ] **`EffectiveStopGrace`** из **5.4**; ошибки Docker — по разделу «Унификация».
- [ ] В README у **5.8** — ссылка «детали» после принятия задачи.
