## Канон UX-решений (SSOT, не откладывать в PR)

| Тема | Канон |
|------|--------|
| Порядок сообщений | **Сверху вниз по времени:** первый элемент — самое раннее **`created_at`** (**12.3**: merge **`created_at ASC`**, **без** **`ListView(reverse: true)`**). |
| Первичный скролл к «хвосту» | После первой успешной загрузки с **`messages.isNotEmpty`** — **однократно** **`_scrollController.jumpTo(_scrollController.position.maxScrollExtent)`** (тот же **`ScrollController`**, что у **`CustomScrollView`**). **Guard’ы:** **`_scrollController.hasClients`**, **`messages.isNotEmpty`**, флаг **`_didInitialScrollToBottom`** для пары **`(projectId, taskId)`**; **не** вызывать **`jumpTo`**, если **`taskDeleted`**. **Сброс флага** — в **`didUpdateWidget`** при смене **`taskId`/`projectId`**; **не** сбрасывать при **`refresh()`** / смене фильтров. **`addPostFrameCallback`** после отстройки слайвера. |
| Семантика **`loadMoreMessages`** (**согласование с 12.3 и кодом**) | **`listTaskMessages`** с **`offset: 0`** даёт **раннюю** хронологическую страницу; **`loadMoreMessages`** запрашивает **`offset = messagesOffset`** — **следующую** страницу по времени (**более новые** сообщения относительно уже загруженных), результат **merge** в конец списка (**вперёд по времени**, как **12.3**). **Запрещено** документировать или реализовывать «догрузку более старых в начало» через **`loadMoreMessages`** без отдельной задачи на смену контракта API/контроллера. |
| Триггер **`loadMoreMessages`** | **Только индекс у нижнего края ленты** в **`SliverChildBuilderDelegate`** для **слайвера сообщений** (индексы **только** по **`messages.length`**): при **`index >= max(0, messages.length - kTaskDetailMessageLoadMoreTrailingThreshold)`**, где **`kTaskDetailMessageLoadMoreTrailingThreshold == 3`**, и при **`hasMoreMessages && !isLoadingMessages`**, **не** вызывать нотифайер **синхронно из `build`**: обернуть в **`WidgetsBinding.instance.addPostFrameCallback((_) { if (!mounted) return; ref.read(taskDetailControllerProvider(…).notifier).loadMoreMessages(); })`** — без побочных эффектов внутри фазы layout. **Запрещено:** **`NotificationListener` + `extentAfter` / `pixels`** на общем **`CustomScrollView`** с шапкой / описанием / diff. **Запрещено:** порог «первые N элементов» (это семантика догрузки **к началу**). |
| Идемпотентность **`loadMoreMessages`** | При **scroll-spam** несколько **`addPostFrameCallback`** с одного кадра могут вызвать **`loadMoreMessages()`** подряд. **Предусловие 12.3:** повторный вызов, пока выполняется предыдущий запрос страницы сообщений (**`isLoadingMessages == true`** и/или **`_loadMoreMessagesInFlight`**), **не** инициирует второй HTTP (**no-op** / тот же **`Future`**). **Экран 12.5** **не** добавляет собственный debounce/throttle. Если в **12.3** этой гарантии **нет** — **блокер на 12.3**, **12.5** не стартует без исправления. |

Если после реализации зафиксирована **визуальная регрессия** — отдельная задача с минимальным воспроизведением и одним выбранным патчем. |

| Навигация подзадача | Со **списка** / **чата** — **`context.push`**. ## Пагинация сообщений (связка UX ↔ **12.3**)

1. **Первая страница** — **`offset: 0`** (**ранние** сообщения в ленте сверху).  
2. **`loadMoreMessages`** — **`offset = state.messagesOffset`**, на сервер уходит **следующий** срез; после merge сообщения **дополняются снизу** (более новые).  
3. **Триггер** — только **§ «Канон UX»**, строка «Триггер **`loadMoreMessages`**» (индекс у **хвоста** ленты + **`addPostFrameCallback`**).  
4. **Новые сообщения в реальном времени** (**12.9**) приходят через контроллер, **не** через polling **`loadMoreMessages`**.  
5. **`setMessageFilters`** (**12.3**): контроллер сбрасывает **`messagesOffset`** в **0**, очищает ленту и **сам** перезагружает первую страницу под новый фильтр. **Экран 12.5** **не** инициирует **`loadMoreMessages`** параллельно этому пути; следующий триггер по индексу сработает **естественно**, когда пользователь снова дойдёт до хвоста **уже отфильтрованной** ленты и выполнятся **`hasMoreMessages && !isLoadingMessages`**.  
6. **Дрейф `messagesOffset` vs WS (предусловие 12.3):** (см. также **§ «Предусловия»** п. **5**.) Сценарий: первая страница **1…50**, **`messagesOffset = 50`**; затем по WS в state попали **51…55**; **`loadMoreMessages(offset=50)`** может вернуть те же id с сервера. **Предусловие:** **`mergeTaskMessagesCanonical`** (**`frontend/lib/features/tasks/domain/task_messages_merge.dart`**) **гарантирует дедупликацию по `TaskMessageModel.id`** (последняя версия по map **выигрывает**), сортировка **`created_at ASC`**, **`id ASC`**. **Экран 12.5** **не** дедуплицирует сам и **не** корректирует **`messagesOffset`** после WS-вставок. Если в **12.3** merge при **`loadMoreMessages`** / WS **не** использует эту функцию или дедуп нарушен — **блокер на 12.3**, **12.5** без этого даст дубликаты или рассинхрон в проде.

### Шов с чатом (**11.7**)

Реализация (**не** голый `context.push` из контекста чата — иначе drift вкладки shell и неверный стек, см. review 12.5 ит.4):

1. **`StatefulNavigationShell.maybeOf(context)`** — если shell есть (дефолтный дашборд проекта):
   - **`goBranch(tasksIdx, initialLocation: true)`** — переключить вкладку на **tasks** и **сбросить стек ветки tasks** к **`/projects/:id/tasks`** перед открытием детали.
   - Затем **`GoRouter.of(projectDashboardShellTasksNavigatorKey.currentContext!).push('/projects/$projectId/tasks/$taskId')`** — push выполняется из контекста навигатора ветки **tasks** (`frontend/lib/core/routing/project_dashboard_routes.dart` — **`projectDashboardShellTasksNavigatorKey`**).
   - До появления контекста tasks-навигатора после **`goBranch`** выполняется до **`kTasksCrossBranchPushMaxRetries`** ожиданий **`SchedulerBinding.instance.endOfFrame`** (`frontend/lib/features/chat/presentation/screens/chat_screen.dart`). Если контекст так и не появился — fallback **`context.push`** (редкий race; в лог пишется **`dart:developer` `log`** с именем **`devteam.chat.cross_branch`**, уровень **900** — см. review ит.6).
2. Если shell **нет** (например тестовый плоский роутер) — один **`context.push`** на полный путь.

**`projectId`:** поле **`ChatScreen.projectId`** (тот же UUID, что в URL дашборда из **`pageBuilder`**), **не** из снимка задачи в сообщении.

**Контракт UX:**

- После открытия задачи **B** из чата вкладка **tasks** активна; **Back** с детали ведёт на **список** **`/projects/:id/tasks`**, а не на предыдущую задачу в стеке этой вкладки (**review ит.5 п.2**).
- **Trade-off (review ит.6):** **`initialLocation: true`** сбрасывает стек ветки tasks; если пользователь был на **`/tasks/A`**, ушёл в чат и открыл карточку **B**, возврат к **A** через Back **не** сохраняется — после Back с **B** пользователь на списке. Сохранение **A** потребовало бы **`initialLocation: false`** и вернуло бы «Back → **A** вместо списка».

**Тесты:** виджет-тесты в **`chat_screen_test.dart`** (shell + URL + Back → список; без shell; race guard по числу **`kTasksCrossBranchPushMaxRetries`**).

## Безопасность `metadata` (**`docs/rules/review.md`**)

- **Не** монтировать сырой **`metadata`** в дереве виджетов в **12.5**.
- **`task_message_metadata_redaction.dart`:**  
  - **Deny-list по имени ключа** (регистронезависимый подстрочный матч): **`token`**, **`secret`**, **`auth`**, **`password`**, **`cookie`**, **`api_key`**, **`apikey`**, **`authorization`**, **`bearer`**, … — **без** ключей-«провайдеров» вроде **`openai`** (поля вида **`openai_api_version`**, **`provider: openai`** — не секреты; маскирование по имени **`openai`** ломает диагностику).  
  - **Дополнительно по значению:** префиксы **`sk-`**, **`Bearer `**; **JWT-эвристика** — только regex  
    **`^[A-Za-z0-9_-]{16,}\.[A-Za-z0-9_-]{16,}\.[A-Za-z0-9_-]{8,}$`**  
    (**не** «ровно три сегмента по `.`» — иначе ложные срабатывания вроде **`1.2.3`**). Совпадение → заменить значение на **`***`**.  
- Unit-тест: **`api_key` / `sk-…`**; кейс безопасного ключа **`openai_api_version`** → значение **не** маскируется из-за имени «openai» (если значение не попадает под эвристику секрета).
