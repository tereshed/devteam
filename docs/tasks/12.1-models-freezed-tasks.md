## Описание

**Решение по скоупу (явно): вариант (A).** В roadmap названы две сущностные модели — **`TaskModel`** и **`TaskMessageModel`**. Для парсинга ответов **`GET /api/v1/projects/{id}/tasks`** и **`GET /api/v1/tasks/{id}/messages`** бэкенд отдаёт **не** голые массивы, а объекты **`TaskListResponse`** / **`TaskMessageListResponse`** с пагинацией; элемент списка задач — **`TaskListItem`**, а не полный **`TaskResponse`**. Поэтому в **12.1** добавляются также **`TaskListItemModel`**, **`TaskListResponse`**, **`TaskMessageListResponse`**. Иначе задача **12.2** (`TaskRepository`) не сможет типизировать ответы Dio без ручного `Map`/кастов.

**Вариант (B)** «оставить только две Freezed-сущности и перенести list-DTO в 12.2» в этом документе **не** выбран.

Контракт полей JSON — **`backend/internal/handler/dto/task_dto.go`**; допустимые строки enum-подобных полей — см. **Источники контракта (SSOT)** в шапке (`models/task.go`, `task_message.go`, `workflow.go`).

| DTO (Go) | Dart (класс) | Назначение |
|----------|--------------|------------|
| `AgentSummary` | `AgentSummaryModel` | Кратко об агенте (внутри задачи и списка) |
| `TaskSummary` | `TaskSummaryModel` | Подзадача в `TaskResponse.sub_tasks` |
| `TaskResponse` | `TaskModel` | Полная карточка задачи (`GET /tasks/:id`) |
| `TaskListItem` | `TaskListItemModel` | Строка списка (`GET /projects/:id/tasks`) |
| `TaskListResponse` | `TaskListResponse` | Страница списка задач |
| `TaskMessageResponse` | `TaskMessageModel` | Одно сообщение задачи |
| `TaskMessageListResponse` | `TaskMessageListResponse` | Страница сообщений задачи |

**Именование:** ответы-обёртки списков в **`requests.dart`** без суффикса **`Model`** (`TaskListResponse`, как **`ProjectListResponse`** в Sprint 10). Сущности предметной области — с суффиксом **`TaskModel`**, **`TaskMessageModel`**, **`TaskListItemModel`**.

**Отличие от чата (11.1):** у **`TaskListResponse`** и **`TaskMessageListResponse`** в Go **нет** поля `has_next` — только `tasks`/`messages`, `total`, `limit`, `offset`. **Не** переносить `hasNext` с `ConversationListResponse`.

**Вне скоупа 12.1:** request-тела (`CreateTaskRequest`, `UpdateTaskRequest`, `CreateTaskMessageRequest`, …) — отдельные задачи, когда появится строгий контракт в репозитории/формах.

**Вне скоупа 12.1 (ошибки HTTP):** разбор тел **4xx/5xx** и общий контракт ошибок — **`frontend/lib/core/api/dio_api_error.dart`** (и перехватчики Dio), не фича-модели. **`TaskModel` / `TaskMessageModel` парсят только успешные JSON-тела** (2xx с телом). Не путать поле **`error_message`** в **`TaskResponse`** (бизнес-текст задачи при `failed` и т.д.) с телом HTTP-ошибки — иначе возможен концептуальный и именованный конфликт в UI.

**Критичные требования:**

- ✅ Freezed v3: `abstract class X with _$X` (не `implements`); готовые **`==`**, **`hashCode`**, **`copyWith`** — отдельно **Equatable** не подключать.
- ✅ `snake_case` в JSON — `@JsonKey(name: '...')` где имя в Dart в `camelCase`
- ✅ UUID из API — `String`
- ✅ **JSONB-поля** `context`, `artifacts` в **`TaskModel`** и **`metadata`** в **`TaskMessageModel`**: **`@Default(<String, dynamic>{}) Map<String, dynamic> ...`** (как **`techStack` / `settings`** с **`@Default(<String, dynamic>{})`** в `project_model.dart`). Бэкенд гарантирует **not null** в БД и ключ в JSON, но **`@Default`** закрывает моки, тесты и отсутствие ключа в ручных фикстурах; **не** помечать как единственный `required Map` без дефолта.
- ✅ **`context` / `artifacts` / `metadata` в JSON** приходят как **готовые JSON-объекты** (массивы/объекты), **не** как строки; кастомные **`@JsonKey(fromJson: …, toJson: …)`** для «строка → `Map`» **не** нужны (см. `datatypes.JSON` + `MarshalJSON` на бэкенде).
- ✅ **DateTime**: парсинг через **`json_serializable`** (отдельного UTC-конвертера в **`frontend/lib`** нет). **Сейчас** ответы — RFC 3339 с суффиксом **`Z`** → Dart **`isUtc == true`**. **Защита на всякий случай**, если формат когда-либо изменится: сравнивать даты через **`Duration`** и при необходимости приводить к UTC — **не** через **`==` по календарным компонентам**.
- ✅ Поля счётчиков из Go **`int64`** (`total`, `message_count` и т.д.) — в Dart тип **`int`**. На **Flutter Web** `int` представим без потери только до **2⁵³**; для лимитов/offset/total и счётчика сообщений этого достаточно. Если когда-нибудь появятся гигантские агрегаты (токены, глобальные счётчики), обсудить **`BigInt`** отдельно.
- ✅ **Публичные поля** снабжать краткими **`///` dartdoc** по прецеденту **`project_model.dart`** (подсказки в IDE) — **исключение** к минимализму комментариев в остальном коде.
- ✅ **Non-nullable** поля в **`const factory`**: либо **`required`**, либо с **`@Default(...)`** — иначе **build_runner** / Freezed v3 падает с ошибкой генерации. **Nullable** (`?`) — без `required`, без лишнего `@Default`, если в таблице ниже не сказано иное.
- ✅ **Без выдуманных полей** — только то, что в соответствующих struct в `task_dto.go`
- ✅ Зависимости: те же `freezed`, `json_serializable`, что в Sprint 10 (`frontend/pubspec.yaml`)
- ✅ Импорты только `package:frontend/...`
- ✅ Обёртки списков в `requests.dart` — имена как в Go (**без** суффикса `Model`), см. таблицу выше.
- ✅ **`task_list_item_model.dart`** импортирует **`AgentSummaryModel`** только из **`task_model.dart`**, не через **`models.dart`** (избежание циркулярных **`part`** при codegen).
- ✅ **`AgentSummaryModel`** и **`TaskSummaryModel`** — полноценные **`@freezed` `abstract class … with _$…`** с **`factory … fromJson`**, отдельные **`part`** для codegen. **Запрещено:** `typedef`, Dart **records** и прочие обходы без codegen — иначе вложенный **`assigned_agent`** / **`sub_tasks`** в **`TaskModel.fromJson`** не соберётся.

### Поля (Dart)

**Стратегия `const factory` (Freezed v3 + json_serializable):** каждое non-nullable поле — либо **`required`**, либо с **`@Default`**. Сводка по прецеденту Sprint 10:

| Категория | Как писать в Dart |
|-----------|-------------------|
| Строки, `DateTime` (обязательные в DTO) | `required` |
| Указатели Go `*T` / omitempty | тип **`T?`**, **без** `required`, **без** `@Default`, если отдельно не указано |
| JSONB (`context`, `artifacts`, `metadata`) | **`@Default(<String, dynamic>{})`** — не единственный `required Map` |
| Списки (`sub_tasks`) | **`@Default(<TaskSummaryModel>[])`** |
| В **`TaskListResponse` / `TaskMessageListResponse`**: `total`, `limit`, `offset` | **`@Default(0)`** (как в **11.1** для числовых полей пагинации) |

| Поле | JSON-ключ | Тип Dart | Стратегия | Примечание |

| `id` | `id` | `String` | `required` | |
| `projectId` | `project_id` | `String` | `required` + `@JsonKey` | |
| `parentTaskId` | `parent_task_id` | `String?` | nullable | omitempty |
| `title` | `title` | `String` | `required` | |
| `description` | `description` | `String` | `required` | |
| `status` | `status` | `String` | `required` | см. статусы ниже |
| `priority` | `priority` | `String` | `required` | см. приоритеты ниже |
| `assignedAgent` | `assigned_agent` | `AgentSummaryModel?` | nullable | omitempty |
| `createdByType` | `created_by_type` | `String` | `required` | |
| `createdById` | `created_by_id` | `String` | `required` | |

| `result` | `result` | `String?` | nullable | см. ниже про **`String?`** и UI |

**`result`:** в Go `*string` + omitempty; **`String?`** в Dart не разводит «нет ключа» и пустую строку как разные UX-состояния — см. предыдущие разделы.

**`message_count`:** поле **зарезервировано** в **`TaskResponse`** (`*int64`, omitempty), но в текущем коде **`task_handler.go`** присваиваний **`MessageCount`** нет (**`grep MessageCount` → пусто**); **`ToTaskResponse`** его **не выставляет** → в реальных ответах ключ **`message_count` всегда отсутствует**. Поле остаётся в **`TaskModel`** на случай будущего заполнения в хендлере. **Тесты:** обязателен кейс **без ключа** → **`null`**; кейс с числом — **опционально** (страховка / будущее). Кейс **`"message_count": null`** в JSON — по желанию (эквивалент **`null`** для **`int?`**).

### `sub_tasks`: порядок и `@Default`

**Порядок:** `task_repository.go` — **`GetByID`**, цепочка **`Preload("SubTasks", … Order("created_at ASC"))`**. Итог: если в JSON есть ключ **`sub_tasks`**, элементы приходят в порядке **`created_at ASC`**. UI **12.4** может **не** сортировать повторно; при своей сортировке — осознанно, зная контракт.

**`@Default` / omitempty:** `task_dto.go` — **`ToTaskResponse`**, slice подзадач через **`make([]TaskSummary, 0, len(t.SubTasks))`** — **не-nil**, но при **`len == 0`** это **пустой slice**. В **`encoding/json`** для тега **`omitempty`** пустой slice считается *empty value* и поле **не сериализуется** — ключа **`sub_tasks` в JSON нет** (аналогично пустой `map`, `""`, `0` и т.д. в документации пакета). То есть «типичный» ответ **`GET /tasks/:id`** без подзадач — **без** ключа **`sub_tasks`**, а не обязательно **`"sub_tasks": []`** (ключ **`[]`** возможен в фикстурах и иных путях). **`@Default(<TaskSummaryModel>[])`** в Dart покрывает **оба** варианта. Тесты: **`[]`** и **без ключа** (оба → пустой список).

### Полиморфные `created_by_id` и `sender_id`

Поля **`created_by_type`** / **`sender_type`** задают семантику UUID в **`created_by_id`** / **`sender_id`**: при **`user`** это пользователь, при **`agent`** — агент (разные таблицы на бэкенде). В моделях остаётся **`String`**; **разрешение имени для UI** — задача репозитория и виджетов (**12.2+**), не этой задачи.

## `TaskListItemModel`

**Импорт `AgentSummaryModel`:** в **`task_list_item_model.dart`** подключать **`AgentSummaryModel`** **прямым** импортом файла **`package:frontend/features/tasks/domain/models/task_model.dart`**, **не** через barrel **`models.dart`**. Иначе при совместном использовании с **`part`** / Freezed codegen возможен **циркулярный** граф импортов и **непрозрачная** ошибка **`build_runner`**. Константы **`kCreatedByType…`** / **`createdByTypes`** брать из того же **`task_model.dart`** (или дублировать строки **не** рекомендуется). Остальной код приложения может по-прежнему использовать **`import 'package:frontend/features/tasks/domain/models.dart'`**.

**1:1 с** `TaskListItem` (облегчённая строка списка):

```go
type TaskListItem struct {
	ID              string        `json:"id"`
	ProjectID       string        `json:"project_id"`
	ParentTaskID    *string       `json:"parent_task_id,omitempty"`
	Title           string        `json:"title"`
	Status          string        `json:"status"`
	Priority        string        `json:"priority"`
	AssignedAgent   *AgentSummary `json:"assigned_agent,omitempty"`
	CreatedByType   string        `json:"created_by_type"`
	CreatedByID     string        `json:"created_by_id"`
	BranchName      *string       `json:"branch_name,omitempty"`
	StartedAt       *time.Time    `json:"started_at,omitempty"`
	CompletedAt     *time.Time    `json:"completed_at,omitempty"`
	CreatedAt       time.Time     `json:"created_at"`
	UpdatedAt       time.Time     `json:"updated_at"`
}
```

**Не** включать в **`TaskListItemModel`** поля, которых **нет** в DTO **`TaskListItem`**, но есть в полной **`TaskResponse`** — ровно семь: `description`, `context`, `result`, `artifacts`, `error_message`, `sub_tasks`, `message_count`.

**Список задач без агента в JSON:** если в **`models.Task`** у задачи **`AssignedAgentID == nil`** (агент не назначен), **`ToTaskListItem`** пробрасывает **`nil`** в **`AssignedAgent`** → **`omitempty`** убирает ключ **`assigned_agent`**. В **`task_repository.List`** **`Preload("AssignedAgent")`** выполняется, но грузить нечего — ключ всё равно отсутствует. Это ожидаемый **кейс** для **`GET /projects/:id/tasks`**. В тестах **`TaskListItemModel`** нужен вариант **без** `assigned_agent` → **`assignedAgent == null`** (не путать с «забыли Preload» — другая диагностика).

**Нет подзадач в списке:** в DTO **`TaskListItem`** поля **`sub_tasks` нет**. В **`task_repository.List`** (типичный сценарий) **`Preload("SubTasks")`** нет — плоский **`tasks[]`** + опциональный **`parent_task_id`**. Дерево для UI **12.4** строится с **плоского списка** по **`parent_task_id`**, либо детали — через **`GET /tasks/:id`** (полный **`TaskModel`** с **`sub_tasks`**).

## Тесты (unit)

Каталог: `frontend/test/features/tasks/domain/`, файл **`requests_test.dart`** для обёрток.

**Шапка** новых `*_test.dart` — канон в **`frontend/test/README.md`** (блок «Шапка unit-файла (VM)»): `// @dart=2.19`, `// @TestOn('vm')`, `// @Tags(['unit'])`. Полный пример — **`frontend/test/core/utils/uuid_test.dart`**.

Минимум:

1. **`TaskModel`** — полный JSON от **`GET /tasks/:id`**: вложенный `assigned_agent`, `context`/`artifacts` как объекты, опциональные `branch_name`, даты.
2. **`message_count`:** в текущем API ключ **практически всегда отсутствует** (хендлер не выставляет; см. таблицу полей). Обязательный тест: JSON **без ключа** → **`null`**. Опционально: **`"message_count": null`**, фикстура с числом (**будущее / страховка**). См. **`task_dto_test.go`** (**`MessageCount == nil`** после **`ToTaskResponse`**).
3. **Все omitempty-поля TaskResponse одним JSON:** один компактный фикстурный объект, где одновременно отсутствуют (или заданы как `null`, где уместно) `parent_task_id`, `result`, `branch_name`, `error_message`, `assigned_agent`, `started_at`, `completed_at`, `message_count` — `TaskModel.fromJson` не падает, nullable поля `null`.
4. **`sub_tasks`** — **`"sub_tasks": []`** и сценарий **без ключа** `sub_tasks` (оба → пустой список при **`@Default`**); у Go с **`omitempty`** пустой slice чаще **опускает** ключ (см. раздел **`sub_tasks`**).
5. **`TaskListItemModel`** — минимальный JSON **без** **`assigned_agent`** (**`assignedAgent == null`**); отдельно — с вложенным агентом.
6. **`TaskListResponse`** / **`TaskMessageListResponse`** — один полный JSON каждого типа; корректная вложенность `tasks` / `messages`; поля **`total`** (int64 с API) как **`int`**; пустые **`tasks`[]** / **`messages`[]**; сценарий **без ключа** **`tasks`** или **`messages`** → **`@Default`** (пустой список).
7. **`TaskMessageModel`** — `metadata` как `{}`, с произвольными ключами и **без ключа** `metadata` → **`@Default`** (пустая карта).
8. **Roundtrip и `toJson`:** для **`TaskModel`** и **`TaskMessageModel`** — цикл **`fromJson` → `toJson` → `fromJson`** (или сравнение карт); проверить **snake_case** в **`toJson`**.

Пункты **2–5** критичны для **12.2** (`TaskRepository`): те же модели разбирают ответы **всех** успешных мутаций (см. таблицу API ниже), без дублирующих DTO.

## Связь с Backend API

Префикс **`/api/v1`** (`@BasePath` в `backend/cmd/api/main.go`). Регистрация: **`backend/internal/server/server.go`** — группы **`api.Group("/tasks")`** и вложенные маршруты проектов (**`…/:id/tasks`**). Swagger / статусы — **`task_handler.go`**.

**Один набор response-моделей для успешных тел:** **`TaskModel`** (**`TaskResponse`**) и **`TaskMessageModel`** (**`TaskMessageResponse`**) используются **не только для GET**, но и для тел ответов **POST/PUT** и действий **pause / cancel / resume / correct** и **POST сообщения**. Request-body для этих ручек — **вне скоупа 12.1**; отдельные «mutation DTO» на клиенте **не** вводить — **12.2** будет парсить те же классы.

**`toJson()` и мутации (12.2+):** сгенерированный **`json_serializable`** для nullable-полей обычно кладёт в карту **`null`** (например, **`parent_task_id`: null**). Для **чтения** ответов API это не мешает. Если в **12.2**/`12.4` решат слать **`TaskModel.toJson()`** в **PUT** и нужна семантика «не трогать поле», а не «очистить», понадобятся отдельные request-DTO или **`@JsonKey(includeIfNull: false)`** (или ручная сборка тела) — не часть **12.1**.

Контракт **списка полей** у **`TaskListItem`** на бэкенде дополнительно защищён **`task_dto_test.go`** (отсутствие лишних ключей в JSON); на клиенте тот же набор дублировать не обязательно.

| Метод и путь | Успешный ответ (тело) | Модель(и) Flutter |
|--------------|------------------------|-------------------|
| `POST /api/v1/projects/{id}/tasks` | **`TaskResponse`** (201) | `TaskModel` |
| `GET /api/v1/projects/{id}/tasks` | **`TaskListResponse`** (200) | `TaskListResponse` |
| `GET /api/v1/tasks/{id}` | **`TaskResponse`** (200) | `TaskModel` |
| `PUT /api/v1/tasks/{id}` | **`TaskResponse`** (200) | `TaskModel` |
| `DELETE /api/v1/tasks/{id}` | **нет тела** (204) | — |
| `POST /api/v1/tasks/{id}/pause` | **`TaskResponse`** (200) | `TaskModel` |
| `POST /api/v1/tasks/{id}/cancel` | **`TaskResponse`** (200) | `TaskModel` |
| `POST /api/v1/tasks/{id}/resume` | **`TaskResponse`** (200) | `TaskModel` |
| `POST /api/v1/tasks/{id}/correct` | **`TaskResponse`** (200) | `TaskModel` |
| `GET /api/v1/tasks/{id}/messages` | **`TaskMessageListResponse`** (200) | `TaskMessageListResponse` |
| `POST /api/v1/tasks/{id}/messages` | **`TaskMessageResponse`** (201) | `TaskMessageModel` |

Ошибки **4xx/5xx** не парсятся этими моделями; разбор — в **`frontend/lib/core/api/dio_api_error.dart`**. Поле **`message_count`** описано в таблице полей **`TaskModel`**; в текущем **`task_handler`** оно нигде не выставляется, поэтому ключа в JSON нет.

## Частые ошибки

1. **`implements _$X` вместо `with _$X`** — ломает Freezed v3.  
2. **Добавить `hasNext` «как в чате»** — в DTO задач этого поля нет.  
3. **Путать `TaskModel` (`TaskResponse`) и `TaskListItemModel` (`TaskListItem`)** — разный набор полей.  
4. **Хардкод относительных импортов** — только `package:frontend/...`.  
5. **Пропуск `build_runner`** — отсутствуют `*.freezed.dart` / `*.g.dart`.  
6. **Вложенные `AgentSummary` / `TaskSummary` без полного `@freezed`** — ломает вложенный парсинг в `TaskModel`.  
7. **Интерпретировать `result != null` как «есть блок результата»** — при появлении `""` в JSON семантика ломается; для UI безопаснее проверять ещё **непустоту** строки (или договор о семантике с API).  
8. **Путать тело HTTP-ошибки с полем `error_message` задачи** — ошибки Dio: **`dio_api_error.dart`**; **`errorMessage` в `TaskModel`** — доменное поле.  
9. **Кастомный `fromJson` для JSONB «строка → Map»** — не нужен; объекты уже в JSON.  
10. **`required Map` без `@Default` для `context` / `artifacts` / `metadata`** — ломает фикстуры и **build_runner**-ожидания Freezed.
