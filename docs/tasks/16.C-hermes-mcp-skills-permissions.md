## Описание

Включить per-agent кастомизацию Hermes Agent так же, как это сделано для Claude Code в Sprint 15.D: MCP-серверы, Skills, permissions / toolsets. После завершения C Hermes-агент перестаёт быть «всегда дефолтный» — пользователь выставляет:

- какие MCP-серверы доступны (с env-переменными для аутентификации);
- какие Hermes Skills предзагружены в сессию;
- какие toolsets разрешены (file ops / shell / web fetch / …) и какой permission mode (**в продукте для sandbox допускаются только `yolo` и `accept`**; `plan`/`default` отклоняются валидацией — см. контракт);
- какие пер-агентные настройки `temperature` / `max_turns`.

Hermes конфиг живёт в `~/.hermes/config.yaml` + `~/.hermes/.env`, плюс есть `~/.hermes/skills/` и `~/.hermes/mcp.json`. Бэкенд должен сериализовать `agent.code_backend_settings` JSONB в эти файлы, runner — смонтировать их в контейнер до старта `hermes chat`. Резолвер уже выдаёт нужные env (фаза A); фаза C добавляет файловую часть.

## Контекст

После A/B/D:

- `agent.code_backend = 'hermes'` запускает `devteam/sandbox-hermes:local`.
- `agent.provider_kind` маршрутизирует на `OPENROUTER_API_KEY` / `ANTHROPIC_API_KEY` / etc.
- `agent.model` напрямую попадает в `hermes chat -m`.
- `agent.code_backend_settings` (JSONB) **игнорируется** — Hermes-агент работает только с дефолтными toolsets и без MCP/Skills.

Это блокер для production: Hermes мощен именно своей расширяемостью (40+ toolsets, agentskills.io-совместимые skills, любые MCP-серверы из ~/.hermes/mcp.json). Без C он эквивалентен «голому» `hermes chat -q` и не имеет преимущества над Claude Code.

## Архитектура и DRY

Hermes-конфиг **сильно** отличается от Claude Code'овского:

| Артефакт | Claude Code (15.22) | Hermes (16.C) |
|---|---|---|
| Settings | `~/.claude/settings.json` (`permissions.allow/deny/defaultMode`) | `~/.hermes/config.yaml` (`model`, `tools.toolsets`, `tools.allowlist`, `display.*`, …) |
| MCP | `.mcp.json` в корне репо или `~/.claude/mcp.json` | `~/.hermes/mcp.json` (другая схема) или `hermes mcp add ...` |
| Skills | `~/.claude/skills/<name>/` (директория) | `~/.hermes/skills/<name>/` (директория) — формат **agentskills.io** совместим |
| Permission flag | `--permission-mode acceptEdits` или `--dangerously-skip-permissions` | `--yolo` либо отсутствие = interactive prompt (в headless контейнере = ошибка) |
| Toolsets | `--allowedTools "Bash,Edit,..."` (просто строка) | `-t toolsets1,toolsets2` (Hermes toolsets — это профили из их каталога) |

DRY и **OCP (без God Object в `AgentSettingsService`)**:

- Сборка файловых артефактов для sandbox **не** раздувает `AgentSettingsService` отдельными методами `BuildArtifacts` / `BuildHermesArtifacts` / будущий `BuildAiderArtifacts`.
- Вводим общий контракт, например `ArtifactBuilder` (или `CodeBackendArtifactBuilder`) с методом в духе `Build(ctx, agent) (*BackendArtifacts, error)` / доменно-специфичный результат.
- Реализации: **`ClaudeArtifactBuilder`** (текущая логика из Sprint 15.22), **`HermesArtifactBuilder`** (новая логика 16.C). При появлении Aider / др. — новая реализация + регистрация, без правки существующих билдеров.
- `AgentSettingsService` держит **фабрику или registry** `map[code_backend]ArtifactBuilder` (или фабрику по `agent.CodeBackend`) и делегирует сборку выбранному билдеру. Валидация JSON Schema и дефолты — по-прежнему в сервисе или в shared-хелпере, вызываемом до `Build`.
- Skills, совместимые с agentskills.io, переносятся как есть — копируется директория `~/.claude/skills/foo/` → `~/.hermes/skills/foo/` без изменений. Если пользователь подписан на Skill один раз, он работает в обоих бэкендах.

## Контракт `code_backend_settings` для Hermes

**Жёсткое правило (headless sandbox):** для `code_backend = hermes` значения `permission_mode` **`plan` и `default` запрещены** — при сохранении агента / PATCH настроек возвращаем **400 Bad Request** с понятным телом ошибки (никаких тихих подмен на `yolo`, никакого «предупреждения в логе вместо отказа»). Это защита от эскалации привилегий и нарушения ожиданий пользователя. В UI эти значения либо не показываем, либо показываем disabled с подсказкой.

Расширяем JSON Schema (валидация в `AgentSettingsService.Validate`):

```json
{
  "code_backend": "hermes",
  "hermes": {
    "toolsets": ["file_ops", "shell", "web_fetch"],
    "permission_mode": "yolo",
    "max_turns": 12,
    "temperature": 0.1,
    "skills": [
      { "name": "code-review-checklist", "source": "builtin" },
      { "name": "company-style", "source": "agentskills:org/style@v1.2" }
    ],
    "mcp_servers": [
      {
        "name": "github",
        "transport": "stdio",
        "command": "npx",
        "args": ["-y", "@modelcontextprotocol/server-github"],
        "env": {
          "GITHUB_PERSONAL_ACCESS_TOKEN": "${secret:github_pat}"
        }
      }
    ]
  }
}
```

Сегрегация полей:

- `toolsets` — белый список Hermes toolset-имен (валидируется по их каталогу). Дефолт: `["file_ops","shell"]`.
- `permission_mode` — в схеме перечислены `yolo|accept|plan|default`, но для **Hermes в DevTeam** допустимы только **`yolo`** и **`accept`** при сохранении агента с `code_backend=hermes`; `plan`/`default` → **400** (см. выше). Дефолт для новых / пустых настроек: `yolo` (изолированный контейнер).
- `max_turns` — int 1..200. Дефолт 12 (соответствует Hermes-дефолту).
- `temperature` — float 0..2. Дефолт null (Hermes сам решит).
- `skills` — список объектов `{ name, source }`. `source` ∈ `builtin|agentskills|path`.
- `mcp_servers` — список объектов в схеме `~/.hermes/mcp.json`. Поддерживается substitution `${secret:NAME}` для секретов из `user_llm_credentials` или `agent_secrets`.

## Backend

### 1. `HermesArtifactBuilder`

Пакет/файл: например `backend/internal/service/hermes_artifact_builder.go` (или рядом с `claude_artifact_builder`). Тип реализует интерфейс **`ArtifactBuilder`** (единое имя в коде), регистрируется в фабрике/registry внутри `AgentSettingsService` (см. раздел «Архитектура и DRY»).

```go
type HermesArtifacts struct {
    ConfigYAML []byte             // ~/.hermes/config.yaml
    MCPJSON    []byte             // ~/.hermes/mcp.json (опц.)
    Skills     map[string][]byte  // map[skillName/file] → содержимое; ключи — только относительные сегменты после валидации (см. безопасность ниже)
    Env        map[string]string  // секреты для MCP serve env (HERMES_MCP_*)
}
```

Реализация:

1. Лукап `agent.code_backend_settings` JSONB; при **nil или пустом JSON** — сначала **нормализация к дефолтной in-memory структуре** (те же дефолты, что и на фронте: `toolsets`, `permission_mode`, `max_turns`, пустые `skills` / `mcp_servers`), чтобы билдер и валидатор **никогда** не разыменовывали nil.
2. Валидация против JSON Schema.
3. Резолв `${secret:NAME}` → `user_llm_credentials.GetPlaintext(...)` или `agent_secrets`.
4. Сериализация:
   - `config.yaml` — YAML с полями `model`, `provider`, `tools.toolsets`, `display`, `terminal.timeout`.
   - `mcp.json` — массив MCP-серверов в их схеме.
   - `skills/` — копируется из реестра Skills (схема Sprint 15.23 + agentskills.io).
5. Возврат структуры. Caller (sandbox runner) сам решает, какие файлы куда положить.

#### Безопасность: Path Traversal при записи Skills

При распаковке `HermesArtifacts.Skills` в `/home/sandbox/.hermes/skills/...` **обязательно**:

- для каждого ключа карты: `filepath.Clean`, сборка целевого пути только через `filepath.Join(baseDir, relativeKey)` от фиксированного `baseDir = /home/sandbox/.hermes/skills` (или эквивалент в коде раннера);
- проверка, что итоговый путь остаётся **внутри** `baseDir` (например `filepath.Rel(baseDir, dest)` не начинается с `..`; либо канонический префикс после `filepath.Abs` / сравнение с `filepath.Clean(baseDir+sep)` — избегать наивного `strings.HasPrefix` без завершающего разделителя и без нормализации);
- отклонять ключи с абсолютными путями, `..`, домашними шорткатами (`~`), нулевыми байтами и т.п. — **400 на валидации настроек агента** или явная ошибка при копировании до старта контейнера (предпочтительно раньше).

Иначе вредоносный или битый JSON может перезаписать файлы вне `~/.hermes/skills/` внутри контейнера.

### 2. Расширение `DockerSandboxRunner.RunTask` для Hermes

`backend/internal/sandbox/docker_runner.go`: после создания контейнера и до `ContainerStart` копируем артефакты через `CopyToContainer`. Сейчас аналогичная логика есть для Claude Code (`/workspace/.claude/settings.json`, `/workspace/.mcp.json`) — расширяем dispatch на Hermes. Логика выше по **path traversal** реализуется здесь (или в общем хелпере `safeWriteSkillFiles`).

| Артефакт | Целевой путь в контейнере |
|---|---|
| `ConfigYAML` | `/home/sandbox/.hermes/config.yaml` (chmod 0600) |
| `MCPJSON` | `/home/sandbox/.hermes/mcp.json` (chmod 0600) |
| `Skills` (директория) | `/home/sandbox/.hermes/skills/<name>/<file>` (chmod 0644 файлы, 0700 директории) |
| `Env` (MCP secrets) | в `SandboxOptions.EnvVars` (см. ValidateEnvKeys whitelist) |

**Утечка секретов в логах:** при отладочном логировании старта контейнера **запрещено** логировать целиком `SandboxOptions`, `EnvVars` или сырые map'ы, содержащие ключи с префиксом `HERMES_MCP_*` (и любые другие секретные env). Использовать редикцию значений, список имён без значений или отдельный `String()` без секретов. Проверить все ветки `DockerSandboxRunner` при добавлении Hermes.

Источник опций: `SandboxOptions.AgentArtifacts` (новое поле, опц.). Если nil — текущий путь.

### 3. Резолвер env для permission_mode и toolsets

`backend/internal/service/sandbox_auth_resolver.go` остаётся как есть. Permission/toolsets — это **не auth**, а агент-настройки, идут через `code_backend_settings`. Резолвер пробрасывает только провайдер-ключи (фаза A). Каскад:

- `code_backend_settings.hermes.permission_mode` → env `DEVTEAM_HERMES_PERMISSION_MODE` → entrypoint подставляет `--yolo` / `--accept-hooks` / пусто. Значения **`plan`/`default` не должны доходить до runner'а** — их отсекает `Validate` (400).
- `code_backend_settings.hermes.toolsets` → env `DEVTEAM_HERMES_TOOLSETS` (comma-separated) → entrypoint подставляет `-t "$DEVTEAM_HERMES_TOOLSETS"`.
- `code_backend_settings.hermes.skills` → entrypoint подставляет `-s "name1,name2"` ИЛИ `--skills` (см. `hermes chat --help`), плюс файлы в `~/.hermes/skills/`.
- `code_backend_settings.hermes.max_turns` → env `DEVTEAM_HERMES_MAX_TURNS` → `--max-turns N`.

ContextBuilder Sprint 15 уже формирует `EnvSecrets` map для `EnvVars`; добавляем новые ключи (нужно расширить whitelist в `ValidateEnvKeys`).

### 4. Entrypoint

`deployment/sandbox/hermes/entrypoint.sh`:

- Новая фаза `prepare_hermes_dotdir` (между clone и agent): проверяет наличие `/home/sandbox/.hermes/`, выставляет mode 0700, валидирует MCP-конфиг через `hermes mcp list` (быстрый sanity-check).
- Фаза `agent` обогащает аргументы:
  ```bash
  HERMES_TOOLSETS_ARGS=()
  [[ -n "${DEVTEAM_HERMES_TOOLSETS:-}" ]] && HERMES_TOOLSETS_ARGS+=("-t" "${DEVTEAM_HERMES_TOOLSETS}")

  HERMES_SKILLS_ARGS=()
  [[ -n "${DEVTEAM_HERMES_SKILLS:-}" ]] && HERMES_SKILLS_ARGS+=("-s" "${DEVTEAM_HERMES_SKILLS}")

  HERMES_PERM_ARGS=()
  case "${DEVTEAM_HERMES_PERMISSION_MODE:-yolo}" in
    yolo)   HERMES_PERM_ARGS+=("--yolo") ;;
    accept) HERMES_PERM_ARGS+=("--accept-hooks") ;;
    plan|default) : ;;   # только если бэкенд уже отклонил эти режимы при сохранении; в entrypoint не подменять на yolo
    *)      echo "invalid DEVTEAM_HERMES_PERMISSION_MODE" >&2; exit 1 ;;
  esac

  HERMES_MAX_TURNS_ARGS=()
  [[ -n "${DEVTEAM_HERMES_MAX_TURNS:-}" ]] && HERMES_MAX_TURNS_ARGS+=("--max-turns" "${DEVTEAM_HERMES_MAX_TURNS}")
  ```

## Pitfalls и edge cases

1. **Hermes config.yaml vs CLI args**: некоторые опции (toolsets, model, provider) дублируются между config.yaml и `hermes chat` флагами. Контракт — **CLI args wins**, конфиг для defaults. MVP передаёт всё через флаги, конфиг минимален.
2. **Permission mode `plan`/`default`**: интерактивный prompt в headless-контейнере привёл бы к зависанию/таймауту. **Только жёсткий отказ на валидации** (`AgentSettingsService.Validate` / JSON Schema): **400 Bad Request**, понятное сообщение. **Запрещено:** тихая подмена на `yolo`, «исправление в entrypoint», предупреждение в логе вместо ошибки пользователю.
3. **MCP secrets и env в логах**: `${secret:NAME}` — шаблон; резолв до сериализации; `mcp.json` не светим в логах. Дополнительно: переменные **`HERMES_MCP_*`** в `SandboxOptions.EnvVars` — **не логировать значения** (см. Backend §2).
4. **Skill collision**: если skill_name из реестра совпадает с встроенным Hermes skill — приоритет user-supplied (наш файл перетирает). Документируем в JSON Schema.
5. **agentskills.io-источники**: при `source: "agentskills:org/style@v1.2"` нужен download-механизм с кэшем. MVP — не реализуем, валидатор отказывает; делается отдельной задачей 16.C.2.
6. **Hermes update**: запиненный `HERMES_REF` в Dockerfile — `v2026.5.7`. CLI флаги (`-t`, `-s`, `--yolo`, `--max-turns`) могут переименоваться в новых релизах. Перед бампом ARG — прогон e2e обязателен.
7. **Размер образа**: каждый skill — это файлы в контейнере. Если skill 50MB и таких 10 — образ распухает. Skills копируются **per-task** через `CopyToContainer`, не зашиваются в image — что и происходит.
8. **`code_backend_settings` = NULL (старые агенты):** до сборки артефактов бэкенд обязан подставить **ту же дефолтную структуру**, что ожидает фронт (см. Backend п.1), иначе риск panic при обращении к вложенным полям. Миграция данных опциональна; безопасная инициализация в коде — обязательна.
