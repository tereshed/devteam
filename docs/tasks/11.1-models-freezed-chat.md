## Описание

**Решение по скоупу (явно): вариант (A).** Кроме двух сущностных моделей, в **11.1** добавляются Freezed-обёртки пагинированных ответов — иначе исполнитель **11.3** (`ConversationRepository`) не сможет распарсить тело `GET` списков (это **не** голый массив, а объект с `total`, `limit`, `offset`, `has_next`). **Вариант (B)** «вынести list-обёртки целиком в 11.3» в этом документе **не** выбран.

Модели **1:1** соответствуют типам в `backend/internal/handler/dto/conversation_dto.go` (ориентир — **имена структур**, без номеров строк):

| DTO (Go) | Dart (имя класса) | Назначение |
|----------|-------------------|------------|
| `ConversationResponse` | `ConversationModel` | один чат |
| `MessageResponse` | `ConversationMessageModel` | одно сообщение |
| `ConversationListResponse` | `ConversationListResponse` | список чатов проекта |
| `MessageListResponse` | `MessageListResponse` | история сообщений |

Имена list-классов на Dart совпадают с Go, по аналогии с `ProjectListResponse` в проектах.

**Расположение по фиче-паттерну:** сущности — в `domain/models/` + `models.dart`; **list-обёртки** — в **`domain/requests.dart`** рядом с будущими request-моделями (как `ProjectListResponse` в `frontend/lib/features/projects/domain/requests.dart`). В одном файле `requests.dart` позже (отдельная задача) появятся `CreateConversationRequest` / `SendMessageRequest`.

**Технический долг эталона projects:** `ProjectListResponse` **не** содержит `has_next`, хотя backend отдаёт поле — **не** копировать эту дыру: у чат-обёрток **обязательно** поле **`hasNext`** с `@JsonKey(name: 'has_next')`.

**Вне скоупа 11.1:** только **request-тела** **`CreateConversationRequest`**, **`SendMessageRequest`** — Freezed-модели для них добавляет **отдельная задача** (в тот же `requests.dart`, после появления контракта в репозитории).

**Критичные требования:**

- ✅ Синтаксис Freezed v3: `abstract class X with _$X` (не `implements`)
- ✅ Все JSON-ключи в **snake_case** — через `@JsonKey(name: '...')` там, где имя поля в Dart отличается от ключа JSON (`project_id`, `conversation_id`, `linked_task_ids`, `created_at`, `updated_at`, **`has_next`**)
- ✅ UUID с бэкенда приходят как **строки** в JSON → тип `String` в Dart
- ✅ Даты — `DateTime` (RFC 3339), парсинг через `json_serializable`
- ✅ `const factory` + `fromJson` / `toJson`
- ✅ **Без выдуманных полей** — только то, что реально отдают перечисленные DTO (в `ConversationResponse` **нет** `user_id`, хотя колонка есть в БД)
- ✅ Зависимости: те же `freezed`, `freezed_annotation`, `json_serializable`, что и в Sprint 10 (`frontend/pubspec.yaml`)

## ConversationModel

**1:1 с** `ConversationResponse`:

```go
type ConversationResponse struct {
	ID        uuid.UUID `json:"id"`
	ProjectID uuid.UUID `json:"project_id"`
	Title     string    `json:"title"`
	Status    string    `json:"status"`
	CreatedAt time.Time `json:"created_at"`
	UpdatedAt time.Time `json:"updated_at"`
}
```

### Пример Dart

```dart
import 'package:freezed_annotation/freezed_annotation.dart';

part 'conversation_model.freezed.dart';
part 'conversation_model.g.dart';

@freezed
abstract class ConversationModel with _$ConversationModel {
  const factory ConversationModel({
    /// UUID чата
    required String id,

    /// UUID проекта
    @JsonKey(name: 'project_id')
    required String projectId,

    /// Заголовок чата (автогенерация / пользовательский)
    required String title,

    /// Статус: 'active' | 'completed' | 'archived' (см. models.ConversationStatus)
    required String status,

    @JsonKey(name: 'created_at')
    required DateTime createdAt,

    @JsonKey(name: 'updated_at')
    required DateTime updatedAt,
  }) = _ConversationModel;

  const ConversationModel._();

  factory ConversationModel.fromJson(Map<String, dynamic> json) =>
      _$ConversationModelFromJson(json);
}
```

### Поля

| Поле | JSON-ключ | Тип | Nullable | Описание |
|------|-----------|-----|----------|----------|
| `id` | `id` | String | NO | UUID чата |
| `projectId` | `project_id` | String | NO | UUID проекта |
| `title` | `title` | String | NO | Заголовок |
| `status` | `status` | String | NO | `active` / `completed` / `archived` |
| `createdAt` | `created_at` | DateTime | NO | Создан |
| `updatedAt` | `updated_at` | DateTime | NO | Обновлён |

## ConversationMessageModel

**1:1 с** `MessageResponse`:

```go
type MessageResponse struct {
	ID             uuid.UUID      `json:"id"`
	ConversationID uuid.UUID      `json:"conversation_id"`
	Role           string         `json:"role"`
	Content        string         `json:"content"`
	LinkedTaskIDs  []uuid.UUID    `json:"linked_task_ids,omitempty"`
	Metadata       datatypes.JSON `json:"metadata,omitempty" swaggertype:"object"`
	CreatedAt      time.Time      `json:"created_at"`
}
```

### Особенности JSON

**Асимметрия полей:** `linked_task_ids` и `metadata` в Go оба с `omitempty`, но типы и жизненный цикл значения разные — **не** копировать одну логику «ключ всегда есть / всегда нет» для обоих и **не** закладывать в Repository инвариант «`metadata` ведёт себя одинаково на всех ответах» (см. ниже POST vs GET).

| Поле Go | Поведение | Dart |
|---------|-----------|------|
| `linked_task_ids` | `[]uuid.UUID` + `omitempty`: пустой слайс → ключ **нет** в JSON (типично и после `Create`, и после чтения из БД: пустой `pq.Array` → slice длины 0). Явное `"linked_task_ids": []` — реже, покрыть тестом. | **`@JsonKey(name: 'linked_task_ids')` + `@Default(<String>[])`**, как `tech_stack` / `settings` в `frontend/lib/features/projects/domain/models/project_model.dart` |
| `metadata` | См. отдельный подпункт **«POST vs GET»** — форма JSON **различается** по эндпоинту; `omitempty` на `[]byte` не «сломан», а срабатывает или нет в зависимости от того, **nil или непустой** `Metadata` в структуре перед `Marshal`. | **`Map<String, dynamic>?`**; в тестах — **все три** формы ниже |

#### `metadata`: POST vs GET (цепочка сервиса)

При создании сообщения в **`internal/service/conversation_service.go`** (`SendMessage`) в структуру кладут только `ConversationID`, `Role`, `Content` — **`Metadata` не задаётся** → в памяти остаётся **`nil`**. GORM на INSERT подставит в SQL `default '{}'`, но **не перезапишет** поле в Go-структуре (в отличие от PK / RETURNING). После `json.Marshal` для `MessageResponse`: `nil` `[]byte` + `omitempty` → **ключа `metadata` в JSON нет** (типичный ответ **`POST /api/v1/conversations/{id}/messages`**).

Сообщения, прочитанные из БД, попадают в DTO с **`Metadata` = сырые байты `{}`** (len ≥ 2) → `omitempty` **не** срабатывает → в JSON есть **`"metadata": {}`** минимум (типичный элемент **`GET /api/v1/conversations/{id}/messages`**).

Итого для **11.1** и **`ConversationRepository` (11.3 в README)** один и тот же `ConversationMessageModel` должен корректно разбирать **обе** формы; в тестах не ограничиваться только «GET-формой».

### Пример Dart

```dart
import 'package:freezed_annotation/freezed_annotation.dart';

part 'conversation_message_model.freezed.dart';
part 'conversation_message_model.g.dart';

@freezed
abstract class ConversationMessageModel with _$ConversationMessageModel {
  const factory ConversationMessageModel({
    required String id,

    @JsonKey(name: 'conversation_id')
    required String conversationId,

    /// 'user' | 'assistant' | 'system'
    required String role,

    required String content,

    /// Как `tech_stack` / `settings` в `project_model.dart`: @Default задаёт значение и для fromJson (нет ключа), и для ручного конструктора.
    @JsonKey(name: 'linked_task_ids')
    @Default(<String>[])
    List<String> linkedTaskIds,

    Map<String, dynamic>? metadata,

    @JsonKey(name: 'created_at')
    required DateTime createdAt,
  }) = _ConversationMessageModel;

  const ConversationMessageModel._();

  factory ConversationMessageModel.fromJson(Map<String, dynamic> json) =>
      _$ConversationMessageModelFromJson(json);
}
```

Использовать **`@JsonKey` + `@Default(<String>[])`** для списка; не полагаться только на `@JsonKey(defaultValue: …)` без `@Default` — иначе у Freezed-конструктора не будет дефолта для ручного создания модели (см. «Частые ошибки», п. 6).

| Поле | JSON-ключ | Тип | Nullable | Описание |
|------|-----------|-----|----------|----------|
| `id` | `id` | String | NO | UUID сообщения |
| `conversationId` | `conversation_id` | String | NO | UUID чата |
| `role` | `role` | String | NO | `user` / `assistant` / `system` |
| `content` | `content` | String | NO | Текст сообщения |
| `linkedTaskIds` | `linked_task_ids` | List\<String\> | NO (дефолт `[]`) | ID задач; см. `@Default(<String>[])` |
| `metadata` | `metadata` | Map\<String, dynamic\>? | YES | Зависит от ответа: см. **POST vs GET** выше (`null` / `{}` / объект) |
| `createdAt` | `created_at` | DateTime | NO | Время создания |

### Допустимые значения `role`

Совпадают с `ConversationRole` в `backend/internal/models/conversation.go`:

```dart

  'user',
  'assistant',

import 'package:freezed_annotation/freezed_annotation.dart';
import 'package:frontend/features/chat/domain/models.dart';

part 'requests.freezed.dart';
part 'requests.g.dart';

@freezed
abstract class ConversationListResponse with _$ConversationListResponse {
  const factory ConversationListResponse({
    @Default(<ConversationModel>[]) List<ConversationModel> conversations,
    @Default(0) int total,
    @Default(0) int limit,
    @Default(0) int offset,
    @JsonKey(name: 'has_next')
    @Default(false)
    bool hasNext,

  factory ConversationListResponse.fromJson(Map<String, dynamic> json) =>
      _$ConversationListResponseFromJson(json);

@freezed
abstract class MessageListResponse with _$MessageListResponse {
  const factory MessageListResponse({
    @Default(<ConversationMessageModel>[])
    List<ConversationMessageModel> messages,
    @Default(0) int total,
    @Default(0) int limit,
    @Default(0) int offset,
    @JsonKey(name: 'has_next')
    @Default(false)
    bool hasNext,
  }) = _MessageListResponse;

  const MessageListResponse._();

  factory MessageListResponse.fromJson(Map<String, dynamic> json) =>
      _$MessageListResponseFromJson(json);
}
```

## Тесты (unit)

Каталоги: `frontend/test/features/chat/domain/models/`, файл **`frontend/test/features/chat/domain/requests_test.dart`**.

Минимум:

1. **ConversationModel** — `fromJson` / `toJson` по полному JSON; парсинг дат; `copyWith`.
2. **`metadata` в `ConversationMessageModel`** — **три** обязательных сценария (реальная асимметрия **POST** vs **GET**, см. подпункт выше):
   - ключа **`metadata` нет** в JSON (форма ответа **POST** `SendMessage` после создания без re-SELECT) → после `fromJson` ожидается **`metadata == null`**;
   - **`"metadata": {}`** (форма типичного элемента **GET** истории сообщений) → пустая `Map` / ожидаемое поведение парсера;
   - **`"metadata": {"tokens": 42}`** (или другое непустое поле) → `metadata!["tokens"] == 42` (или эквивалентная проверка).
3. **`linked_task_ids`** — три явных сценария (после `@Default(<String>[])` там, где итог пустой список, результат одинаковый):
   - ключ **`linked_task_ids` отсутствует**;
   - **`"linked_task_ids": []`**;
   - **`"linked_task_ids": ["<uuid-a>", "<uuid-b>"]`**.
4. **`ConversationListResponse` / `MessageListResponse`** — `fromJson` минимум на одном полном JSON каждого типа: вложенные списки десериализуются, **`hasNext`** / **`has_next`** совпадает с числовыми полями пагинации.

## Связь с Backend API

Регистрация путей: префикс **`/api/v1`** (`@BasePath` в `backend/cmd/api/main.go`); точные `@Router` — в `backend/internal/handler/conversation_handler.go`. **Полные URL и форму JSON-ответов** (в т.ч. для ручных проверок) удобно сверять с **unit-тестами хендлера** в `backend/internal/handler/conversation_handler_test.go` (там свой `gin.Engine` — это **не** production-роутер; живое монтирование в приложении — см. сноску в начале документа).

| Метод и путь | Тело ответа | Модель(и) Flutter | DTO (Go) |
|--------------|-------------|-------------------|----------|
| `POST /api/v1/projects/{project_id}/conversations` | один объект | `ConversationModel` | `ConversationResponse` |
| `GET /api/v1/projects/{project_id}/conversations` | объект-обёртка | `ConversationListResponse` | `ConversationListResponse` |
| `GET /api/v1/conversations/{id}` | один объект | `ConversationModel` | `ConversationResponse` |
| `DELETE /api/v1/conversations/{id}` | **нет тела** (`204 No Content`) | — | — |
| `POST /api/v1/conversations/{id}/messages` | один объект | `ConversationMessageModel` | `MessageResponse` |
| `GET /api/v1/conversations/{id}/messages` | объект-обёртка | `MessageListResponse` | `MessageListResponse` |

Строка **DELETE**: отдельной Freezed-модели не требуется; **11.3** (`ConversationRepository`) должен уметь вызвать удаление/архивацию и обработать `204`.

**Сноска для 11.3 (Repository), не для полей сущностей:** `POST /api/v1/conversations/{id}/messages` (**SendMessage**) требует заголовок **`X-Client-Message-ID`** (валидный **UUID**, в Swagger — UUIDv4; см. валидацию в `conversation_handler.go`). Идемпотентный повтор с тем же id → **`200 OK`** и тот же `MessageResponse`; первое создание → **`201 Created`**; тело в обоих случаях десериализуется как `ConversationMessageModel` (`dto.ToMessageResponse`).

Контракт полей JSON — **`backend/internal/handler/dto/conversation_dto.go`** (все четыре response-структуры выше).

## Частые ошибки (кратко)

1. **`implements _$X` вместо `with _$X`** — ломает Freezed v3.  
2. **Пропуск `@JsonKey` для snake_case** — `NoSuchMethodError` при `fromJson`.  
3. **`user_id` в модели чата** — в REST-ответе `ConversationResponse` этого поля нет; не добавлять «для полноты».  
4. **Путаница имён:** Go `MessageResponse` ↔ Dart `ConversationMessageModel` — ок, главное — совпадение JSON.  
5. **Забытый `build_runner`** — не компилируются `*.freezed.dart` / `*.g.dart`.  
6. **Только `@JsonKey(defaultValue: [])` для списка без `@Default`** — конструктор Freezed остаётся без дефолта для поля; ручное создание модели ломается или ловит `null` там, где ждёте `List`.  
7. **Неверный базовый путь для сообщений** — `GET`/`POST` истории и отправки сообщений идут на **`/api/v1/conversations/{id}/messages`**, а не под `/api/v1/projects/{project_id}/conversations/...`. Префикс проекта — только для списка/создания чатов (`conversation_handler.go`, `conversation_handler_test.go`).  
8. **`@Tags('unit')` вместо `@Tags(['unit'])`** — тест не попадёт в выборку `flutter test --tags=unit` (канон — `frontend/test/README.md`, `auth_repository_test.dart`).  
9. **Инвариант «`metadata` в JSON всегда как в GET»** — после **POST** `SendMessage` тело часто **без** ключа `metadata`; после выборки из БД — с `{}`. Парсер и тесты должны учитывать **обе** формы (см. подпункт POST vs GET).  
10. **Скопировать `ProjectListResponse` без `hasNext`** для чат-обёрток — backend отдаёт `has_next`; поле обязательно в `ConversationListResponse` / `MessageListResponse`.

## Поддержка и отладка

При расхождении с API:

1. Снять фактический JSON: `curl` / DevTools к ответу эндпоинта.  
2. Сверить поля с `conversation_dto.go`, пути и Swagger — с `conversation_handler.go` / `conversation_handler_test.go`; для живого HTTP убедиться, что conversation-роуты смонтированы в `internal/server` (см. сноску у «Зависит от»).  
3. При необходимости `dart run build_runner clean` и снова `make frontend-codegen`.
