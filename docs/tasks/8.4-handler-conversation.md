# 8.4 Handler: ConversationHandler

**Sprint:** 8 — Conversation API (Backend)
**Файл:** `backend/internal/handler/conversation_handler.go`
**Тест:** `backend/internal/handler/conversation_handler_test.go` (unit/integration)
**Статус:** ✅ Завершено

---

## Описание

Реализовать HTTP-обработчик `ConversationHandler` для управления чатами (Conversations) и сообщениями. Хендлер должен использовать `ConversationService` для выполнения бизнес-логики и обеспечивать RESTful API согласно спецификации.

---

## Эндпоинты

| Метод | Путь | Успех | Описание |
|-------|------|-------|----------|
| POST | `/api/v1/projects/:project_id/conversations` | 201 | Создать новый чат в проекте |
| GET | `/api/v1/projects/:project_id/conversations` | 200 | Список чатов проекта (пагинация) |
| GET | `/api/v1/conversations/:conversation_id` | 200 | Детали чата |
| POST | `/api/v1/conversations/:conversation_id/messages` | 201 | Отправить сообщение в чат |
| GET | `/api/v1/conversations/:conversation_id/messages` | 200 | История сообщений (пагинация) |
| DELETE | `/api/v1/conversations/:conversation_id` | 204 | Удалить чат |

---

## Технические требования

### 1. Аутентификация и Контекст
- Все эндпоинты должны быть защищены JWT-мидлваром.
- `userID` должен извлекаться из контекста запроса.
- **КРИТИЧНО:** Во все методы `ConversationService` ОБЯЗАТЕЛЬНО передавать `r.Context()` для поддержки отмены запросов (context cancellation) и предотвращения утечек ресурсов.
- **Tenant Isolation (IDOR):** Сервис обязан проверять права доступа. Пользователь может обращаться только к тем чатам и проектам, к которым у него есть доступ. При попытке доступа к чужому ресурсу возвращать `403 Forbidden` (или `404 Not Found` для скрытия факта существования).

### 2. Специфика парсинга и HTTP (КРИТИЧНО)
- **Streaming JSON:** Для парсинга Body использовать `json.NewDecoder(r.Body).Decode(&dto)` вместо `io.ReadAll`. Это минимизирует аллокации памяти.
- **Resource Management:** ОБЯЗАТЕЛЬНО закрывать тело запроса (`defer r.Body.Close()`) после чтения/парсинга JSON.
- **Body Limit:** Использовать `http.MaxBytesReader` (лимит 1 МБ) перед декодированием JSON.
- **Обработка ошибок парсинга:**
    - Ошибки `json.Unmarshal` / `Decode` (кривой JSON, несовпадение типов) — `400 Bad Request`.
    - Ошибки парсинга Query-параметров (например, `strconv.Atoi` для `limit=abc`) — `400 Bad Request`.
    - Ответ должен содержать понятное описание, какое именно поле содержит ошибку.
- **Идемпотентность:**
    - Ключ идемпотентности передается в заголовке `X-Client-Message-ID`.
    - **Валидация:** Ключ ОБЯЗАТЕЛЬНО должен быть валидным UUIDv4. Если заголовок пустой или невалидный — `400 Bad Request`.
    - **Race Condition:** Хендлер должен корректно обрабатывать ошибку уникальности от сервиса (например, `ErrConflict`) и превращать её в `200 OK` с существующим сообщением.

### 3. Безопасность и Валидация
- **Rate Limiting:** Использовать Rate Limiter middleware для эндпоинтов создания сущностей (особенно сообщений), так как они инициируют тяжелую бизнес-логику (LLM, БД).
- **Валидация UUID:** Проверять формат всех ID из пути (`project_id`, `conversation_id`) перед вызовом сервиса. При ошибке — `400 Bad Request`.
- **Строковые поля:** 
    - Выполнять `TrimSpace` для `title` и `content`.
    - Лимиты длины: `title` (max 255), `content` (max 4096).
    - Если после трима строка пустая или превышает лимит — `400 Bad Request`.
- **XSS Protection:** API возвращает сырой текст (Markdown). Защита от XSS лежит на стороне клиента.
- **Пагинация:** 
    - Проверять `limit > 0` и `offset >= 0`.
    - **Дефолтные значения:** Если `limit` не передан, использовать `limit = 20`.
    - Явное ограничение: Если `limit > 100`, возвращать `400 Bad Request`.
- **PII & Logging:** При логировании ошибок маскировать или исключать поле `content` из логов.

### 4. Архитектура и Производительность
- **Pagination Metadata:** Списочные методы (GET) должны возвращать данные, обернутые в структуру с метаданными пагинации (`total`, `limit`, `offset`, `has_next`).
- **Сортировка:** Список сообщений должен возвращаться отсортированным по `created_at DESC` (от новых к старым).
- **DTO:** Использовать структуры из `backend/internal/handler/dto/conversation_dto.go`.
- **Слои:** Хендлер не должен знать о деталях реализации БД (Soft Delete, JOIN и т.д.). Он только вызывает методы сервиса.

### 5. Обработка ошибок
- **Унификация:** ЗАПРЕЩЕНО вручную маппить ошибки сервиса в HTTP-статусы.
- Использовать общий хелпер `httputil.RespondError(w, err)`, который централизованно содержит маппинг ошибок сервиса в HTTP-коды.
- Ответы должны соответствовать стандартному формату ошибок API.

### 6. Swagger-аннотации
- Каждый метод должен содержать полные аннотации `swag`.
- **Security:** Обязательно указывать `// @Security BearerAuth`.
- Описать заголовок `X-Client-Message-ID` для эндпоинта создания сообщений.
- Описать все возможные ответы (200, 201, 204, 400, 401, 403, 404, 500).

## Тестирование

**Файл:** `conversation_handler_test.go`

### Тест-кейсы
1. **Success:** Проверка статус-кодов 201 (POST), 200 (GET), 204 (DELETE).
2. **Parsing Errors:**
    - Невалидный JSON в Body (400).
    - Невалидные типы в Query-параметрах (limit=abc) (400).
    - Невалидный UUID в заголовке `X-Client-Message-ID` (400).
3. **Empty State:** Возврат `200 OK` и `[]` для пустых списков.
4. **Pagination:** Дефолтные значения и ошибка при `limit > 100`.
5. **Security:**
    - Превышение Max Body Size (413).
    - Невалидный UUID в пути (400).
    - Пустые строки или превышение лимитов длины (400).
6. **Idempotency & Race Condition:** Параллельные запросы с одним `X-Client-Message-ID` возвращают корректные ответы без дублирования в БД.
7. **Auth & Access (IDOR):** 401 (No Token), 403/404 (Access to other user's resource).

- [ ] Реализованы все эндпоинты с использованием именованных параметров `:project_id` и `:conversation_id`.

- [ ] Ошибки парсинга (JSON, Query, UUID) возвращают `400 Bad Request`.
- [ ] Идемпотентность вынесена в заголовок `X-Client-Message-ID` с валидацией UUIDv4.
- [ ] Реализовано ограничение размера Body и закрытие `r.Body`.
- [ ] Ошибки обрабатываются через `httputil.RespondError`.
- [ ] Хендлер полностью очищен от логики БД (Soft Delete и т.д.).
- [ ] Тесты покрывают ошибки парсинга, идемпотентность и конкурентные запросы.
