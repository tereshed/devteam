# 9.5 Service: `IndexerService` (полная индексация проекта, инкрементальное обновление)

**Sprint:** 9 — Vector DB & Knowledge Base
**Файл:** `backend/internal/service/indexer_service.go`
**Статус:** ⬜ В ожидании

---

## Описание

Задача заключается в создании `IndexerService`, который выступает в роли координатора всех процессов индексации данных проекта в векторную БД Weaviate. Сервис должен объединять работу специализированных индексаторов (кода, задач и переписок), обеспечивая как полную переиндексация проекта «с нуля», так и точечное обновление данных при их изменении (инкрементальная индексация).

Это необходимо для того, чтобы AI-агенты имели актуальный контекст проекта, не переиндексируя весь объем данных при каждом мелком изменении.

---

## Функциональные требования

### 1. Полная индексация проекта (Full Reindex)
- Метод `FullIndex(ctx, projectID)` должен:
    1. Проверить наличие «зависших» предыдущих сессий индексации. Если сессия была прервана, восстановить её или очистить её данные.
    2. Создать новый `index_run_id` для текущей сессии индексации.
    3. Запустить потоковую индексацию (streaming):
        - Всего исходного кода проекта.
        - Всех завершенных и активных задач.
        - Всей истории переписок.
    4. После успешного завершения всех потоков — обновить активный `index_run_id` проекта в БД. Старые данные удаляются фоновым процессом.
- **Распределенный лок:** Использовать distributed lock с обязательным **TTL и механизмом продления (watchdog/renewal)**.
- **Потоковая обработка и Batching:** Запрещено загружать все данные в память. Индексаторы возвращают данные через каналы. Запись в Weaviate должна идти **строго батчами (Bulk API)** с заданным размером и таймаутом для обеспечения производительности.
- **Разрешение конфликтов (Race Condition):** Если во время работы `FullIndex` поступает запрос на инкрементальное обновление (`IndexCode` и т.д.), такие обновления должны записываться сразу в новый `index_run_id`, чтобы не быть удаленными при переключении активного индекса.

### 2. Инкрементальное обновление и синхронизация
- **Безопасность и Санитаризация:** 
    - Любой `filePath` должен проходить через `filepath.Clean()` и проверку на то, что он не выходит за пределы корневой директории проекта (защита от Path Traversal).
    - Слой Handler обязан проверить права доступа пользователя к `projectID` перед вызовом методов сервиса (защита от IDOR).
- **Обновление (Upsert):**
    - `IndexCode(ctx, projectID, filePath)` / `IndexCodes(ctx, projectID, filePaths)`
    - `IndexTask(ctx, projectID, taskID)` / `IndexTasks(ctx, projectID, taskIDs)`
    - `IndexMessage(ctx, projectID, messageID)` / `IndexMessages(ctx, projectID, messageIDs)`
- **Удаление (Anti-Zombie):**
    - `DeleteCode(ctx, projectID, filePath)` / `DeleteCodes(ctx, projectID, filePaths)`
    - `DeleteTask(ctx, projectID, taskID)`
    - `DeleteMessage(ctx, projectID, messageID)`
- **Надежность (Reliability):** 
    - Реализовать механизм **Retry** для всех операций записи/удаления в Weaviate.
    - **Персистентность отказов:** При неудачном удалении/обновлении операция должна записываться в персистентную очередь (например, через `SyncStateRepository`), чтобы фоновый воркер мог завершить её после рестарта сервиса.
    - `MoveCode(ctx, projectID, oldPath, newPath)`: стратегия «Запись нового $\rightarrow$ Удаление старого». При частичном отказе запись в очередь неудачных операций обязательна.

### 3. Управление очередью и ресурсами
- **Обработка Rate Limits (429):** При получении ошибки 429 от LLM API реализовать **Exponential Backoff с Jitter**. Рекомендуется использовать token bucket / rate limiter перед отправкой запросов.
- **Жизненный цикл горутин:** При отмене `context.Context` сервис должен дождаться завершения текущих батчевых операций (используя `sync.WaitGroup` или `errgroup`), чтобы избежать утечек горутин и частичных записей.

### 4. Обработка ошибок, мониторинг и Observability
- **Безопасное логирование:** Запрещено писать `filePath` или ответы от API в логи без санитаризации. Все строки должны быть очищены от символов переноса строки ($\n, \r$) для предотвращения Log Forging.
- **Метрики (Prometheus/OpenTelemetry):**
    - Размер очереди на индексацию.
    - Средний latency получения эмбеддингов.
    - Количество ошибок 429 (Too Many Requests).
    - Количество проиндексированных объектов в секунду.
- Реализовать детальное логирование: прогресс, время выполнения, ошибки по конкретным сущностям.

## Технические подробности

### Схема ID (Deterministic IDs)
ID объектов в Weaviate формируются для обеспечения Upsert и предсказуемой длины ключа:
- **Код:** `code_{projectID}_{sha256_16(filePath)}` (использование усеченного SHA-256 до 16 символов для обеспечения фиксированной длины и читаемости).
- **Задачи:** `task_{projectID}_{taskID}`
- **Сообщения:** `msg_{projectID}_{messageID}`

### Интерфейс сервиса
```go
type IndexStatus struct {
    State      string    // idle | indexing | failed
    Progress   float64   // 0.0 to 1.0
    RunID      string    // current index_run_id
    StartTime  time.Time
    LastError  string
}

type IndexerService interface {
    FullIndex(ctx context.Context, projectID string) error
    
    // Single updates
    IndexCode(ctx context.Context, projectID string, filePath string) error
    IndexTask(ctx context.Context, projectID string, taskID string) error
    IndexMessage(ctx context.Context, projectID string, messageID string) error
    
    // Bulk updates for performance
    IndexCodes(ctx context.Context, projectID string, filePaths []string) error
    IndexTasks(ctx context.Context, projectID string, taskIDs []string) error
    IndexMessages(ctx context.Context, projectID string, messageIDs []string) error
    
    // Deletions
    DeleteCode(ctx context.Context, projectID string, filePath string) error
    DeleteCodes(ctx context.Context, projectID string, filePaths []string) error
    DeleteTask(ctx context.Context, projectID string, taskID string) error
    DeleteMessage(ctx context.Context, projectID string, messageID string) error
    
    // Utilities
    MoveCode(ctx context.Context, projectID string, oldPath, newPath string) error
    GetIndexStatus(ctx context.Context, projectID string) (IndexStatus, error)
}
```

### Стратегия консистентности и отката (Blue-Green + GC)
1. **Индексация:** Новые данные пишутся с текущим `index_run_id`.
2. **Атомарное переключение:** Только после успешного завершения всего `FullIndex` в БД обновляется «активный» `index_run_id` проекта.
3. **Очистка (Garbage Collection):** Фоновый воркер удаляет объекты, если их `index_run_id` **не является** активным и **не является** в данный момент запущенным (in-progress с учетом таймаута на зависшие сессии).
