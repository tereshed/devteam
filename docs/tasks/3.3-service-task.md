# 3.3 Service: TaskService

**Sprint:** 3 — CRUD API для Tasks
**Файл:** `backend/internal/service/task_service.go`
**Тест:** `backend/internal/service/task_service_test.go` (unit)
**Статус:** ⬜ Не начата

---

## Описание

Создать сервис бизнес-логики для задач. Это **ключевой сервис Sprint 3** и самый сложный в приложении — он управляет жизненным циклом задачи: создание, назначение агенту, переходы статусов (state machine), валидация переходов, пользовательские действия (pause/cancel/resume), добавление сообщений. Service оркестрирует два репозитория (`TaskRepository`, `TaskMessageRepository`), проверяет права доступа (ABAC через ProjectService) и устанавливает временные метки (`started_at`, `completed_at`).

## Интерфейс

```go
type TaskService interface {
    // CRUD
    Create(ctx context.Context, userID uuid.UUID, userRole models.UserRole, projectID uuid.UUID, req dto.CreateTaskRequest) (*models.Task, error)
    GetByID(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID) (*models.Task, error)
    List(ctx context.Context, userID uuid.UUID, userRole models.UserRole, projectID uuid.UUID, req dto.ListTasksRequest) ([]models.Task, int64, error)
    Update(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID, req dto.UpdateTaskRequest) (*models.Task, error)
    Delete(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID) error

    // State transitions (пользовательские действия)
    Pause(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID) (*models.Task, error)
    Cancel(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID) (*models.Task, error)
    Resume(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID) (*models.Task, error)

    // Transition (программный — для Orchestrator)
    Transition(ctx context.Context, taskID uuid.UUID, newStatus models.TaskStatus, opts TransitionOpts) (*models.Task, error)

    // Messages
    AddMessage(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID, req dto.CreateTaskMessageRequest) (*models.TaskMessage, error)
    ListMessages(ctx context.Context, userID uuid.UUID, userRole models.UserRole, taskID uuid.UUID, req dto.ListTaskMessagesRequest) ([]models.TaskMessage, int64, error)
}
```

**Все методы, кроме `Transition`, принимают `userID` + `userRole`** — для ABAC (проверка владения проектом).

**`Transition`** — внутренний метод для оркестратора (не требует userID, агент уже авторизован через task pipeline). Доступен только из `OrchestratorService` (Sprint 6).

## Зависимости (DI)

```go
type taskService struct {
    taskRepo       repository.TaskRepository
    taskMsgRepo    repository.TaskMessageRepository
    projectService ProjectService
    teamService    TeamService
}

func NewTaskService(
    taskRepo    repository.TaskRepository,
    taskMsgRepo repository.TaskMessageRepository,
    projectSvc  ProjectService,
    teamSvc     TeamService,
) TaskService
```

**Зависимость от `ProjectService`** — для ABAC: `projectService.GetByID(ctx, userID, userRole, task.ProjectID)` одновременно загружает проект и проверяет права.

**Зависимость от `TeamService`** — для валидации `assigned_agent_id`. При назначении агента на задачу необходимо проверить, что агент принадлежит команде проекта. Метод `teamService.GetByProjectID(ctx, projectID)` возвращает команду с Preload `Agents`, после чего проверяем вхождение:

```go
func (s *taskService) checkAgentInTeam(ctx context.Context, projectID uuid.UUID, agentID uuid.UUID) error {
    team, err := s.teamService.GetByProjectID(ctx, projectID)
    if err != nil {
        return fmt.Errorf("failed to get team: %w", err)
    }
    for _, agent := range team.Agents {
        if agent.ID == agentID {
            return nil
        }
    }
    return ErrAgentNotInTeam
}
```

**Почему `TeamService`, а не `TeamRepository` напрямую:** Service не должен зависеть от чужих репозиториев напрямую (нарушение слоистости). `TeamService` уже оборачивает `TeamRepository` и маппит ошибки.

## State Machine (задача 3.6 включена сюда)

### Допустимые переходы

```
pending          → planning, cancelled
planning         → in_progress, failed, cancelled, paused
in_progress      → review, failed, cancelled, paused
review           → testing, changes_requested, failed, cancelled, paused
changes_requested → in_progress, cancelled, paused
testing          → completed, failed, cancelled, paused
paused           → pending (resume — возврат в предыдущий статус или в pending)
completed        → (терминальный)
failed           → pending (retry — только ручной)
cancelled        → (терминальный)
```

### Реализация

```go
var allowedTransitions = map[models.TaskStatus][]models.TaskStatus{
    models.TaskStatusPending:          {models.TaskStatusPlanning, models.TaskStatusCancelled},
    models.TaskStatusPlanning:         {models.TaskStatusInProgress, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusInProgress:       {models.TaskStatusReview, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusReview:           {models.TaskStatusTesting, models.TaskStatusChangesRequested, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusChangesRequested: {models.TaskStatusInProgress, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusTesting:          {models.TaskStatusCompleted, models.TaskStatusFailed, models.TaskStatusCancelled, models.TaskStatusPaused},
    models.TaskStatusPaused:           {models.TaskStatusPending},
    models.TaskStatusFailed:           {models.TaskStatusPending},
    // completed, cancelled — терминальные, нет допустимых переходов
}

func canTransition(from, to models.TaskStatus) bool {
    targets, ok := allowedTransitions[from]
    if !ok {
        return false
    }
    for _, t := range targets {
        if t == to {
            return true
        }
    }
    return false
}
```

### TransitionOpts

```go
type TransitionOpts struct {
    AssignedAgentID *uuid.UUID  // Назначить/переназначить агента при переходе
    Result          *string     // Результат (при completed)
    ErrorMessage    *string     // Описание ошибки (при failed)
    Artifacts       *datatypes.JSON // Артефакты (diff, PR URL)
    BranchName      *string     // Git-ветка
}
```

## Методы

### Create

1. `projectService.GetByID(ctx, userID, userRole, projectID)` — ABAC
2. Валидация DTO: `title` обязателен, `priority.IsValid()`, `parent_task_id` если задан — проверить существование
3. Создать `Task` с `Status = pending`, `CreatedByType = user`, `CreatedByID = userID`
4. Если `assigned_agent_id` задан — проверить, что агент принадлежит команде проекта
5. `taskRepo.Create`

**Ошибки:**
- `ErrProjectNotFound`, `ErrProjectForbidden` — из projectService
- `ErrTaskInvalidTitle` — пустой title
- `ErrTaskInvalidPriority` — невалидный priority
- `ErrTaskParentNotFound` — parent_task_id не существует
- `ErrAgentNotInTeam` — агент не принадлежит команде проекта

### GetByID

1. `taskRepo.GetByID(ctx, taskID)` — загрузить задачу с Preload
2. `projectService.GetByID(ctx, userID, userRole, task.ProjectID)` — ABAC через проект
3. Вернуть задачу

### List

1. `projectService.GetByID(ctx, userID, userRole, projectID)` — ABAC
2. Нормализация пагинации: `Limit=0 → 50`, clamp `Limit ≤ 200`
3. Маппинг DTO → `TaskFilter` (с `ProjectID = projectID`)
4. `taskRepo.List(ctx, filter)`

### Update

1. `taskRepo.GetByID` — загрузить текущее состояние
2. ABAC через projectService
3. Partial update: применить поля из DTO к модели
4. **Если меняется `status`** → валидация через `canTransition` + установка timestamps
5. **Если меняется `assigned_agent_id`** → проверить принадлежность агента к команде
6. `taskRepo.Update`

**Важно:** `Update` позволяет менять `title`, `description`, `priority`, `assigned_agent_id`, `status`, `branch_name`. Поля `result`, `artifacts`, `error_message` обновляются через `Transition` (программный API).

### Delete

1. `taskRepo.GetByID` — загрузить задачу
2. ABAC через projectService
3. `taskRepo.Delete` — каскадно удаляет messages, SET NULL для subtasks

### Pause

1. `taskRepo.GetByID` + ABAC
2. `canTransition(current, paused)` — валидация
3. Обновить status → `paused`
4. `taskRepo.Update`

**Допустимые исходные статусы:** planning, in_progress, review, changes_requested, testing

### Cancel

1. `taskRepo.GetByID` + ABAC
2. `canTransition(current, cancelled)` — валидация
3. Обновить status → `cancelled`, `completed_at = now`
4. `taskRepo.Update`

### Resume

1. `taskRepo.GetByID` + ABAC
2. Проверить `current == paused` или `current == failed`
3. Обновить status → `pending` (рестарт pipeline)
4. `taskRepo.Update`

### Transition (программный — для Orchestrator)

Внутренний метод, не требует ABAC (вызывается из `OrchestratorService` после авторизации).

1. `taskRepo.GetByID`
2. `canTransition(current, newStatus)` — валидация
3. Применить `TransitionOpts` (agent, result, error, artifacts, branch)
4. Установить timestamps:
   - `in_progress` → `started_at = now` (если nil)
   - `completed` → `completed_at = now`
   - `failed` → `completed_at = now`
   - `cancelled` → `completed_at = now`
5. `taskRepo.Update`

### AddMessage

1. `taskRepo.GetByID` + ABAC через projectService
2. Создать `TaskMessage` с `SenderType = user`, `SenderID = userID`
3. `taskMsgRepo.Create`

### ListMessages

1. `taskRepo.GetByID` + ABAC через projectService
2. Нормализация пагинации: `Limit=0 → 50`, clamp `Limit ≤ 200`
3. `taskMsgRepo.ListByTaskID(ctx, taskID, filter)`

## ABAC

Проверка доступа к задаче **через проект**: пользователь имеет доступ к задаче, если имеет доступ к проекту.

```go
func (s *taskService) checkTaskAccess(ctx context.Context, userID uuid.UUID, userRole models.UserRole, task *models.Task) error {
    _, err := s.projectService.GetByID(ctx, userID, userRole, task.ProjectID)
    return err // ErrProjectNotFound или ErrProjectForbidden
}
```

Admin (`userRole == admin`) имеет доступ ко всем задачам (через ProjectService).

## Валидация и дефолты

| Поле | Валидация (Service) | Дефолт |
|------|---------------------|--------|
| `title` | Обязательно, непустое, max 500 | — |
| `priority` | `IsValid()` | `medium` |
| `status` (Create) | Не передаётся — всегда `pending` | `pending` |
| `status` (Update) | `canTransition()` | — |
| `assigned_agent_id` | Если задан — агент в команде проекта | `nil` |
| `parent_task_id` | Если задан — задача существует в том же проекте | `nil` |
| `created_by_type` (Create) | Не передаётся — всегда `user` (из контекста) | `user` |
| `Limit` (List/ListMessages) | 0 → 50, clamp ≤ 200 | 50 |
| `Offset` | ≥ 0 | 0 |

## Timestamps: автоматическая установка

| Переход в статус | Поле | Значение |
|-----------------|------|----------|
| `in_progress` | `started_at` | `time.Now()` (если nil — первый раз взята в работу) |
| `completed` | `completed_at` | `time.Now()` |
| `failed` | `completed_at` | `time.Now()` |
| `cancelled` | `completed_at` | `time.Now()` |
| `pending` (resume) | `completed_at` | `nil` (сброс, задача перезапускается) |

## Конвенции

1. **Интерфейс + приватная struct** — как `ProjectService`
2. **DI через конструктор** — все зависимости инжектируются (repos + projectService)
3. **ABAC через ProjectService** — не дублировать логику проверки владения
4. **Кастомные sentinel errors** — маппинг из repository-ошибок
5. **Partial update** — GetByID → изменить поля → Save
6. **DTO на входе** — Service принимает DTO, не сырые модели
7. **`context.Context`** — во всех методах
8. **State machine** — `canTransition()` + `allowedTransitions` map
