## Описание

Реализовать **`LLMAgentExecutor`** — конкретную реализацию интерфейса **`AgentExecutor`** (задача **6.1**), предназначенную для выполнения шага агента через прямой вызов LLM (Large Language Model) с использованием системного промпта, пользовательского промпта и (опционально) инструментов (tools/functions).

Этот исполнитель используется для ролей, не требующих изолированного выполнения кода в sandbox (например, **Orchestrator**, **Planner**, **Reviewer**), либо для предварительного анализа перед запуском разработчика.

**Ключевой функционал:**
1. Формирование запроса к LLM API на основе `ExecutionInput`.
2. Поддержка системного и пользовательского промптов.
3. Обработка структурированного контекста (`StructuredContext`).
4. Парсинг ответа LLM и маппинг в `ExecutionResult`.
5. Сбор метрик использования токенов (`PromptTokens`, `CompletionTokens`).

## Требования к реализации

### 1. Безопасность и Prompt Injection (КРИТИЧНО)
- **Изоляция ввода:** Пользовательский ввод (`PromptUser`) и контекст (`ContextJSON`, `StructuredContext`) должны быть четко отделены от системных инструкций.
- **XML-теги (для Claude):** Обязательно оборачивать сырые данные в XML-теги (например, `<context>...</context>`, `<user_input>...</user_input>`) для предотвращения перехвата управления через данные.
- **Маскирование:** При логировании входа использовать `in.String()` для исключения утечки секретов из `EnvSecrets`.

### 2. Архитектура (DRY & SRP)
- **Инъекция клиента:** `LLMAgentExecutor` не должен содержать логику HTTP-запросов. Он принимает через конструктор интерфейс `LLMClient` (или аналогичный), что позволяет легко менять провайдера и писать юнит-тесты с моками.
- **Sentinel Errors:** Использовать кастомные ошибки для типичных сбоев LLM: `ErrRateLimit`, `ErrContextTooLarge`, `ErrInvalidResponse`.

### 3. Обработка краевых случаев (Edge Cases)
- **Rate Limits (429/529):** Стратегия ретраев (exponential backoff + jitter) должна быть реализована на уровне `LLMClient`. Экзекутор должен корректно обрабатывать финальный отказ.
- **Context Length:** Если суммарный объем промпта и контекста превышает лимит модели, экзекутор должен возвращать `ErrContextTooLarge` (early return), не дожидаясь ошибки от API, если это возможно рассчитать.
- **Отмена контекста:** Метод `Execute(ctx context.Context, ...)` обязан прокидывать `ctx` во все вызовы `LLMClient`. При отмене контекста (например, пользователь нажал "Стоп") выполнение должно немедленно прерываться с возвратом `ctx.Err()`, чтобы не тратить токены и ресурсы.

### 4. Контракт Error vs Success
- **`error != nil`**: Инфраструктурные сбои, таймауты, отмена контекста, `ErrRateLimit` (после всех попыток), `ErrContextTooLarge`, невалидный конфиг.
- **Оборачивание ошибок:** Все ошибки от `LLMClient` или парсинга должны оборачиваться через `fmt.Errorf("...: %w", err)` для сохранения трассировки.
- **`error == nil && Success == false`**: LLM вернула ответ, но он не прошел валидацию формата или содержит явный отказ (refusal).
- **`error == nil && Success == true`**: Успешная генерация ответа.

## Технические детали

### Провайдеры (MVP)
Основной провайдер — **Anthropic Claude** (модели `claude-3-5-sonnet-*`).

### Стриминг (Streaming)
Для MVP реализуется **синхронный** вызов (блокирующий до получения полного ответа). Стриминг токенов через WebSocket фиксируется как **Technical Debt** и будет реализован в рамках Sprint 7.

### Структура ответа и "защитный" парсинг
Исполнитель должен уметь извлекать:
- Текстовое резюме (`Summary`).
- Полный вывод (`Output`).
- **ArtifactsJSON:** Если LLM вернула структурированные данные (например, внутри блоков ```json ... ```), экзекутор обязан очистить их от Markdown-разметки перед парсингом. Ошибки парсинга JSON не должны приводить к панике; в случае невалидного JSON шаг помечается как `Success == false` с описанием ошибки формата в `Summary`.
