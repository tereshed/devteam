# 9.4 Индексатор чатов: сообщения пользователя и ассистента

**Статус:** ⬜ К выполнению

## Описание

Задача заключается в реализации компонента `ConversationIndexer`, который отвечает за индексацию сообщений в чатах (диалогах) между пользователем и ассистентом в Weaviate. Это позволит реализовать семантический поиск по истории переписки (например: "что мы обсуждали по поводу архитектуры базы данных в прошлом чате?").

В отличие от `TaskIndexer`, который индексирует задачу как единый объект с результатом, `ConversationIndexer` должен фокусироваться на потоке сообщений, сохраняя контекст диалога и роли участников.

---

## Зависимости

**Базовые компоненты:**
- `vectordb.Client` (для сохранения векторов и метаданных в Weaviate).
- `models.Conversation` и `models.ConversationMessage` (доменные модели).
- `internal/eventbus` (для подписки на события создания/обновления/удаления сообщений, проектов и пользователей).

**Интерфейсы сборки контекста:**
- Репозитории `ConversationRepository` и `ConversationMessageRepository` для получения данных.

## Функциональные требования

### 1. Формирование документа (Document Construction)
- **Контекстный чанкинг (Contextual Chunking):** При разбиении длинных ответов ассистента на чанки, **каждый чанк обязан включать в себя исходный вопрос пользователя** (или его summary) и маркеры ролей.
- **Лимиты контекста (Token Truncation):** Если исходный вопрос пользователя слишком велик (например, > 500 токенов), для контекста чанка использовать его обрезанную версию или summary. Итоговый чанк с учетом вопроса и части ответа **гарантированно** должен влезать в лимиты модели векторизации (например, 8191 токен для OpenAI).
- **Разметка структуры:** Использовать `strings.Builder` с предварительным выделением памяти через `Grow()`.
- **Валидация и очистка (Early Return):** Применять `strings.TrimSpace()`. Если текст пуст, выполнять early return и помечать `is_indexed = true`.
- **Маскирование секретов (КРИТИЧНО):** Обязательная санитаризация с предварительным **URL-decoding**.
- **Контроль размера (Chunking):** Использовать `tiktoken` (синглтон/пул).

### 2. Метаданные и Схема Weaviate
- **Метаданные (RBAC/Isolation):** `conversation_id`, `message_id`, `project_id`, **`user_id`** (владелец чата), `role`, `created_at`.
- **Изоляция данных (Security):** При выполнении поиска фильтрация **ОБЯЗАНА** идти по составному ключу: `project_id == X AND user_id == Y`. Пользователь не должен иметь доступа к чанкам чужих приватных переписок внутри одного проекта.
- **Схема Weaviate (Schema Management):** 
    - **ЗАПРЕЩЕНО** автоматическое создание/изменение схемы из кода приложения. Схема управляется внешними миграциями.
    - При старте индексатор выполняет **Healthcheck** схемы.
    - Для полей UUID использовать тип `uuid` или `text` с настройкой **`tokenization: "field"`**.
- **Детерминированные UUID:** Использовать `uuid.NewMD5` для обеспечения идемпотентности (**Upsert**).

### 3. Инкрементальная индексация и Архитектура
- **Decoupling (Event-Driven):** Подписка на события (`MessageCreatedEvent`, `MessageUpdatedEvent`, `MessageDeletedEvent`, `ProjectDeletedEvent`, `UserDeletedEvent`) через Event Bus.
- **Обработка Удалений (GDPR Compliance):**
    - Реализовать методы `DeleteMessage`, `DeleteConversation`, `DeleteProjectConversations`, `DeleteUserConversations`.
    - При батч-индексации и поиске строго фильтровать сообщения: `deleted_at IS NULL`.
- **Асинхронность и Observability:**
    - Использовать `context.WithoutCancel(ctx)` с таймаутом (10s).
    - **TraceID Propagation:** Прокидывать **TraceID/RequestID** в фоновые воркеры для сквозного логирования.
    - **Graceful Shutdown:** Использовать `sync.WaitGroup` для дожидания завершения текущих задач при SIGTERM.
- **Data Consistency (Replication Lag):** Чтение актуальных данных строго из Master-узла БД или использование данных из самого события.

### 4. Производительность и отказоустойчивость
- **Batching (Keyset Pagination):** При массовой индексации **ЗАПРЕЩЕНО** использовать `OFFSET`. Использовать только **Cursor-based (Keyset) pagination**.
- **N+1 Prevention:** Использовать **Eager Loading** (Preload) при массовой индексации.
- **Защита Weaviate (Stability):**
    - Использовать **Semaphore** для ограничения одновременных запросов (лимит: 50-100).
    - Реализовать паттерн **Circuit Breaker**.
- **Обработка ошибок (Monitoring):**
    - **Никаких тихих ошибок:** Ошибки в фоне отправляются в **Sentry** и метрики.
    - **Унификация:** Использовать хелперы `vectordb` (`IsTransientError`, `IsNotFoundError`).

## Критерии приёмки

- [ ] Индексатор работает через Event Bus, поддерживает Graceful Shutdown, Semaphore и TraceID.
- [ ] **Безопасность:** В метаданных есть `user_id`, поиск фильтрует по `project_id` + `user_id`.
- [ ] **GDPR:** Реализовано каскадное удаление векторов при удалении сообщений, чатов, проектов или пользователей.
- [ ] **Стабильность:** Чанки гарантированно влезают в лимиты модели векторизации (Truncation контекста).
- [ ] Схема Weaviate не создается автоматически (только Healthcheck).
- [ ] Массовая индексация использует Keyset Pagination (без OFFSET) и Preload.
- [ ] Реализована защита от Replication Lag при чтении контекста.
- [ ] Интеграционные тесты используют **Testcontainers**.
- [ ] Поиск защищен от IDOR.
