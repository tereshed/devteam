# 8.5 Интеграция: `ConversationService` → `OrchestratorService`

**Sprint:** 8 — Conversation API (Backend)
**Файлы:** 
- `backend/internal/service/conversation_service.go`
- `backend/internal/service/orchestrator_service.go`
**Тест:** `backend/internal/service/conversation_service_test.go` (интеграционный сценарий)
**Статус:** ⬜ В планах

---

## Описание

Реализовать связку между сервисом чатов (`ConversationService`) и оркестратором (`OrchestratorService`). Основная цель — обеспечить автоматический запуск процесса обработки (планирование, выполнение, ревью) при получении нового сообщения от пользователя в чате.

---

## Зависимости

**Требует:**
- **8.3** — `ConversationService` (базовая реализация).
- **6.4** — `OrchestratorService` (метод `ProcessTask`).
- **3.3** — `TaskService` (создание задач).

---

## Функциональные требования

### 1. Триггер оркестрации
- При вызове `ConversationService.SendMessage` после успешного сохранения сообщения в БД должен инициироваться процесс оркестрации.
- **Логика связки:**
    1. Создание системной задачи (`models.Task`) через `TaskService.Create`.
    2. Заголовок задачи формируется на основе текста сообщения (с обрезкой до 50 символов).
    3. Описание задачи — полный текст сообщения пользователя.
    4. Вызов `OrchestratorService.ProcessTask(ctx, taskID)`.

### 2. Асинхронность и надежность
- Вызов оркестратора должен происходить в **отдельной горутине**, чтобы не блокировать HTTP-ответ пользователю.
- **Context Management:** Использовать `context.WithoutCancel(ctx)` (Go 1.21+) при запуске горутины. Это отвяжет выполнение от жизненного цикла HTTP-запроса (который отменяется сразу после ответа), но сохранит метаданные (traceID и др.).
- **Graceful Shutdown:** Использовать `s.wg.Add(1)` перед запуском и `defer s.wg.Done()` внутри горутины, чтобы сервис дождался завершения активных процессов при выключении (метод `Shutdown`).
- **Panic Recovery:** Обязательный `recover()` с логированием **стектрейса** (`debug.Stack()`), чтобы не потерять место возникновения ошибки.

### 3. Безопасность и Архитектура
- **Unicode Safety:** При формировании заголовка задачи использовать обрезку по **рунам** (`[]rune`), а не по байтам, чтобы избежать повреждения UTF-8 символов (эмодзи, кириллица).
- **Tenant Isolation:** Оркестратор должен получать `taskID`, по которому он сам восстановит контекст проекта и проверит права (уже реализовано в 6.4).
- **Error Handling (Background):** Гарантировать обновление статуса при любом сбое (паника, ошибка создания задачи, ошибка оркестратора). Использовать паттерн с `defer` и флагом успеха, чтобы избежать "зависших" состояний в UI.

## Технические детали реализации

### Метод `runOrchestrator` в `ConversationService`:
```go
// Вызов в SendMessage:
// s.wg.Add(1)
// go s.runOrchestrator(context.WithoutCancel(ctx), userID, projectID, conversationID, content)

func (s *conversationService) runOrchestrator(ctx context.Context, userID, projectID, conversationID uuid.UUID, content string) {
    defer s.wg.Done()
    
    var success bool
    defer func() {
        if r := recover(); r != nil {
            slog.Error("Panic in orchestration flow", 
                "error", r, 
                "conversationID", conversationID,
                "stack", string(debug.Stack()),
            )
            success = false
        }

        if !success {
            // TODO: Обновить статус сообщения/разговора на Failed в БД,
            // чтобы UI не оставался в состоянии вечного ожидания.
            // s.markMessageFailed(ctx, conversationID, ...)
        }
    }()

    // 1. Создание задачи
    // ВНИМАНИЕ: truncate должен корректно работать с рунами (unicode)!
    taskReq := dto.CreateTaskRequest{
        Title:       fmt.Sprintf("Chat: %s", truncateRunes(content, 50)),
        Description: content,
    }
    task, err := s.taskSvc.Create(ctx, userID, models.RoleUser, projectID, taskReq)
    if err != nil {
        slog.Error("Failed to create task for conversation", 
            "error", err, 
            "conversationID", conversationID,
        )
        return // success остается false
    }

    // 2. Запуск процесса
    if err := s.orchestratorSvc.ProcessTask(ctx, task.ID); err != nil {
        slog.Error("Orchestration failed", 
            "taskID", task.ID, 
            "conversationID", conversationID, 
            "error", err,
        )
        return // success остается false
    }

    success = true
}
```

---

## Критерии приёмки

- [ ] Метод `SendMessage` выполняет валидацию `content` (не пустой, trim) до запуска горутины.
- [ ] Метод `SendMessage` запускает `runOrchestrator` в горутине с использованием `context.WithoutCancel`.
- [ ] Реализован `recover()` с логированием стектрейса и `conversationID`.
- [ ] Метод `Shutdown` корректно ожидает завершения горутин через `wg.Done()`.
- [ ] Создаваемая задача корректно привязана к `projectID` и `userID`.
- [ ] Логирование ошибок оркестрации содержит `taskID` и `conversationID`, но не содержит PII (текст сообщения).
- [ ] Реализована гарантированная обработка ошибок через `defer` (защита от зависания UI при панике или ошибке создания задачи).
- [ ] Ошибки оркестратора в фоне логируются с `taskID` и `conversationID`.
- [ ] (Опционально) Добавлен TODO или механизм для обновления статуса сообщения на Failed при любом сбое в фоне.
