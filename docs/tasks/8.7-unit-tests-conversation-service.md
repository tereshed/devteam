## Описание

Unit-тесты для `ConversationService` — сервиса, отвечающего за управление чатами (создание, получение списка, удаление), отправку сообщений, идемпотентность запросов и асинхронный запуск оркестратора. Тесты покрывают бизнес-логику, валидацию, права доступа, идемпотентность, фоновую очистку и обработку паник.

> **Критично:** Сервис активно использует горутины (`SendMessage`, `cleanupProcessedMessagesLoop`, `Shutdown`). Проверка на утечки горутин (goleak) строго обязательна.
> **Критично:** Все тесты должны проходить с флагом `-race` (`go test -race ./...`).

## Требования к качеству тестов

### Table-Driven Tests — ОБЯЗАТЕЛЬНО

Для всех методов с валидацией и проверкой прав (`CreateConversation`, `SendMessage`, `GetConversation` и т.д.) **обязательно** использовать table-driven подход.

**Структура хелперов:**
```go
// Фабрика моков — один раз, переиспользуется везде
func newTestConversationHarness(t *testing.T) (*conversationService, *mockDeps)

// Табличные тесты для валидации
var createValidationTests = []struct {
    name        string
    title       string
    setupMocks  func(m *mockDeps)
    expectedErr error
}{
    {"empty title", "", nil, ErrInvalidConversationTitle},
    {"too long title", strings.Repeat("a", 256), nil, ErrInvalidConversationTitle},
    // ...
}
```

### Проверка на утечки горутин — через t.Cleanup()

Для всех тестов, затрагивающих асинхронную логику (`SendMessage`, `Shutdown`, инициализация сервиса с фоновым лупом), **обязательно** использовать `goleak` (uber-go/goleak) через `t.Cleanup()`.

```go
import "go.uber.org/goleak"

func TestSendMessage_Success(t *testing.T) {
    t.Cleanup(func() { goleak.VerifyNone(t) })
    // ... тело теста
}
```

### Race Conditions (Состояние гонки)

Сервис использует `sync.RWMutex` для защиты мапы `processedMessages`. Необходимо явно проверить потокобезопасность при одновременной записи (через `SendMessage`) и чтении/удалении (через фоновый луп). Обязательно наличие теста, где 10-50 горутин одновременно вызывают `SendMessage` с одинаковыми и разными `clientMsgID`.

## Функциональные требования

### 1. Управление чатами (Create, Get, List, Delete)

- **CreateConversation**: валидация `title` (пустой, >255 символов), проверка прав на проект (через `ProjectService`), сохранение в БД.
- **GetConversation**: получение из БД, проверка, что `conv.UserID == userID` (иначе `ErrConversationForbidden`). Обязательная проверка на `userID == uuid.Nil` (предотвращение IDOR).
- **ListConversations**: проверка прав на проект, нормализация пагинации (limit 1-100, offset >= 0, **отрицательные значения должны сбрасываться в дефолтные**), вызов репозитория.
- **DeleteConversation**: проверка прав на чат, удаление через `TransactionManager`.

### 2. Отправка сообщений (SendMessage)

- Валидация `content` (пустой, >4096 символов, **строка из одних пробелов/переносов строк должна отсекаться через `strings.TrimSpace`**).
- Проверка прав на чат.
- **Идемпотентность**: если `clientMsgID` уже есть в мапе `processedMessages`, возвращать `ErrDuplicateMessage`. Защита мапы через `sync.RWMutex`.
- Сохранение сообщения в БД.
- **Асинхронность**: запуск `runOrchestrator` в отдельной горутине. Увеличение `sync.WaitGroup`.

### 3. История сообщений (GetHistory)

- Проверка прав на чат.
- Нормализация пагинации (сброс отрицательных значений).
- Получение списка сообщений из репозитория.

### 4. Асинхронный запуск оркестратора (runOrchestrator)

- Создание задачи через `TaskService` с обрезанным до 50 символов заголовком.
- Запуск `OrchestratorService.ProcessTask`.
- **Panic Recovery**: перехват паник внутри горутины, чтобы не уронить весь сервер.
- Корректный вызов `wg.Done()`.

### 5. Фоновая очистка (cleanupProcessedMessagesLoop)

- Очистка старых сообщений (старше 10 минут) из `processedMessages` для предотвращения утечки памяти.
- Корректная остановка горутины при получении сигнала из `stopChan`.

### 6. Graceful Shutdown

- Закрытие `stopChan` (только один раз через `sync.Once`).
- Ожидание завершения всех горутин (`wg.Wait()`).
- Поддержка таймаута через `context.Context` (если `OrchestratorService.ProcessTask` завис и не реагирует на отмену, `Shutdown` должен вернуть ошибку контекста, а не ждать вечно).
