# 5.4 Типы: `SandboxOptions`, `SandboxStatus`, `CodeResult`, `ResourceLimit`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/types.go` (структуры и константы); при необходимости контракта — соседние файлы **того же пакета** `sandbox` (например `options_*.go` для методов на `SandboxOptions`), без выноса логики в `handler`/`service`.  
**Статус:** ✅ Выполнена (структуры и константы); ниже зафиксированы **обязательные требования к контракту** — их закрытие при ревью может потребовать доработок в `options_*.go`, **5.2** (entrypoint) и **5.5** (раннер).

---

## Описание

Зафиксировать в пакете `sandbox` **контракт данных** для интерфейса **`SandboxRunner`** (задача **5.3**): входные опции запуска, снимок/финальное состояние инстанса, результат работы агента в контейнере и лимиты ресурсов. Дополнительно в `types.go` (или рядом в пакете) — **смежные типы и константы**, без которых сигнатуры раннера не компилируются и не согласуются с **`deployment/sandbox/claude/entrypoint.sh`** (задача **5.2**).

**Правило:** типы в `types.go` **не импортируют** `internal/models`, `gin`, GORM. Идентификаторы задачи и проекта — **`string`**; маппинг из доменных UUID — на стороне вызывающего сервиса/оркестратора (**6.3**).

В README в строке таблицы перечислены четыре ключевые структуры; фактический объём модуля шире — см. раздел «Полный перечень».

## Согласование с entrypoint (5.2)

- Поля **`SandboxOptions`**, которые мапятся на **env**, должны использовать **те же имена**, что ожидает `entrypoint.sh` — константы `EnvRepoURL`, `EnvBranchName`, `EnvBaseRef`, `EnvGitDefaultBranch`, `EnvBackend`, `EnvAnthropicAPIKey`, `EnvMaxTurns` и т.д.
- **`Instruction`** и **`Context`** — большие тексты: в контейнер передаются **через файлы** (`PromptFilePath`, `ContextFilePath`), а не через большие ENV (см. **5.5**, `CopyToContainer`).
- Пути **`StatusJSONPath`**, **`FullDiffPath`**, **`AgentLogPath`** и др. — единая точка правды для парсера артефактов (**5.7**).

### Утечка секретов (Token Leakage) — КРИТИЧНО

**Риск:** в `EnvVars` попадают `ANTHROPIC_API_KEY`, токены Git и т.п. Если оркестратор или раннер логирует **`SandboxOptions`** «как есть» (`%+v`, рефлексия, сырой JSON), ключи окажутся в логах **plaintext**.

**Требования к задаче (контракт на `SandboxOptions`):**

1. Реализовать **безопасное строковое представление** для логов: например `fmt.Stringer` (`String()`), отдельный явный метод вроде `LogSafe()`, или оба — но **семантика одна**: не выводить значения секретных env, не дублировать полные `Instruction`/`Context` (достаточно длины или `<N bytes>`).
2. Реализовать **`MarshalJSON()`** (или эквивалент для сериализации в очереди/отладочные API) с **той же политикой маскирования**, что и для логов: значения по ключам, чьё имя указывает на секрет, заменять на маску (например `***`); эвристика имени — как минимум подстроки в верхнем регистре: **`TOKEN`**, **`SECRET`**, **`KEY`**, **`PASSWORD`**, **`API_KEY`**, а также известные имена вроде `ANTHROPIC_API_KEY` из констант пакета.
3. В спецификации/комментарии к типу **явно предупредить:** `fmt.Printf("%+v", opts)` **обходит** `String()` и может напечатать сырые поля — для отладки использовать только безопасные методы/сериализацию выше.

*Зона ответственности:* реализация может лежать в `options_log.go` / `options_json.go` рядом с `types.go`, пакет тот же.

### Command / Flag Injection (git, shell)

**Риск:** `RepoURL`, имя ветки (`Branch` / `BRANCH_NAME`), база для diff (`BASE_REF` и т.п.) — внешний ввод. При подстановке в **git** или shell без разделителя аргументов возможны `-h`, `--upload-pack=...` и аналоги (см. правила ревью `.cursor/rules/review.mdc`).

**Требования:**

1. В **`deployment/sandbox/claude/entrypoint.sh`** (задача **5.2**, согласование с **5.4**): для любых вызовов `git`, где после флагов идёт пользовательская строка (URL, ref, имя ветки), **обязательно** использовать конец опций **`--`** перед этими аргументами, например:  
   `git clone … -- "$REPO_URL" "$REPO_DIR"`,  
   `git checkout -b -- "$BRANCH_NAME" "origin/${BASE_REF_RESOLVED}"`  
   (точный набор команд — по факту скрипта; правило: **не передавать сырой ввод туда, где git может истолковать его как опцию**; ветка вида `-h` без **`--`** была бы опасна — с **`--`** git воспринимает её как имя ref, а не флаг.)
2. На стороне Go: **defense in depth** — валидация URL и имени ветки до Docker/entrypoint (отдельные функции в пакете `sandbox`, см. задачу **5.5** / `RunTask`): пустые/опасные символы, префикс `-`, и т.д. по политике репозитория.

## Валидация и быстрый фейл (Edge Cases)

**Риск:** пустые `RepoURL`, `Instruction`, `TaskID` приводят к старту контейнера и немедленному падению внутри — лишняя нагрузка и время.

**Требования:**

1. Контракт **`SandboxOptions`**: метод **`Validate(ctx context.Context) error`** (или эквивалент с той же семантикой), вызываемый **в начале `RunTask`** (задача **5.5**) **до любых** обращений к Docker API (ctx — для DNS при проверке `RepoURL`).
2. **Early return** с оборачиваемой ошибкой и **`errors.Is(..., ErrInvalidOptions)`** (или согласованный sentinel пакета): минимум непустые (после `strings.TrimSpace`) **`TaskID`**, **`RepoURL`**, **`Instruction`**, а также обязательные для запуска поля по политике продукта (**`Image`**, **`Backend`**, непустая **`Branch`** с учётом уже существующих `ValidateRepoURL` / `ValidateBranchName` / `ValidateEnvKeys`, если они применяются к полям опций). Для **`TaskID`** и при необходимости **`ProjectID`** — дополнительно проверка **формата**, см. подраздел ниже.
3. **`Context`**: по умолчанию допускается пустым, если бизнес-сценарий не требует обязательного контекста (иначе зафиксировать в спеке отдельно).

### Формат `TaskID` и `ProjectID` (имя контейнера, хостовые пути)

**Риск (Container Name Injection / path traversal):** проверка только на «не пусто» недостаточна. **`TaskID`** участвует в детерминированном имени контейнера (префикс **`TaskContainerNamePrefix`** + идентификатор, задача **5.5**). Строки со **пробелами**, **`/`**, **`..`** и прочим вне допустимого набора для Docker reference дают **`invalid reference format`** или хуже — если тот же идентификатор когда‑либо попадёт в путь **volume / временной директории на хосте**, возможен **path traversal** и непредсказуемое поведение API ОС.

**Требования к `Validate()` (или отдельным `ValidateTaskID` / `ValidateProjectID`):**

1. **`TaskID`**: после trim — не только непустой, но и **строгий безопасный формат** на выбор политики продукта, например:
   - **UUID** в каноническом виде (`uuid` из БД), **или**
   - регулярное выражение в духе **`^[a-zA-Z0-9_-]+$`** с разумной верхней границей длины (чтобы имя контейнера и пути оставались предсказуемыми).
2. **`ProjectID`**: если поле **участвует** в имени контейнера, bind-mount, пути временной директории на хосте или в любом пути, передаваемом в Docker/ФС — **тот же контракт**, что и для **`TaskID`**. Если в реализации **5.5** `ProjectID` в такие строки **не** попадает, допускается более мягкая политика, но она должна быть **явно задокументирована** в коде (чтобы при следующем рефакторинге не «протащить» ID в имя контейнера без валидации).

### Таймаут `Timeout == 0` (утечка ресурсов)

**Риск:** при **`Timeout: 0`** контейнер теоретически может исполняться неограниченно долго.

**Требования:**

1. Ввести **жёсткий дефолт** на уровне контракта, например константа **`DefaultSandboxTimeout = 30 * time.Minute`** (значение можно уточнить в конфиге **5.10**, но ноль как «бесконечность» **запрещён**).
2. Раннер (**5.5**) и логика **5.8** обязаны для таймеров использовать **эффективный** таймаут: если в опциях `Timeout <= 0`, подставляется **`DefaultSandboxTimeout`**. Поле структуры при этом можно **не мутировать** в `Validate()` — достаточно вспомогательного метода вроде **`EffectiveTimeout() time.Duration`** или единой функции в раннере; главное — **нигде** не планировать бесконечное выполнение контейнера по умолчанию.

## Архитектура пакета (DRY, «God Object»)

Для MVP **допустимо** держать опции, статусы, `LogEntry` и константы лимитов в одном `types.go`, пока пакет `sandbox` небольшой.

**Рекомендация на будущее (зафиксировать в задаче):** при росте логики стриминга (**5.6**) и сериализации опций имеет смысл вынести **`LogEntry`**, константы буферов логов и код, связанный только со стримом, в отдельный файл (`logs.go` / `stream.go`) **в том же пакете**, без изменения публичных имён типов для потребителей.

## Ключевые дизайн-решения

1. **Два уровня таймаутов:** поле **`SandboxOptions.Timeout`** — желаемый бизнес-таймаут (после успешного `start`, **5.5** / **5.8**); при нуле — см. раздел «Таймаут». Не заменяет `context.Context` в методах раннера (**5.3**).

2. **`SandboxOptions.EnvVars`:** shallow copy при передаче структуры по значению; **`RunTask`** обязан сделать **глубокую копию** мапы перед асинхронным использованием.

3. **`CodeBackendType` и `SandboxStatusType`:** строковые типы с константами — логи, JSON, entrypoint.

4. **`SandboxStatus.Result`:** указатель; до **5.7** может быть `nil`; безопасный доступ — **`HasResult()`**.

5. **`CodeResult`:** усечение **`Diff`** / **`Output`**; **`FilesChanged`** — предпочтительно из `git diff --name-only -z`; санитизация путей для UI.

6. **`ResourceLimit`:** нули для CPU/RAM/disk/PIDs — «не задавать в Docker / дефолт движка» (**5.9**). Это **не** отменяет обязательного **дефолта по времени** для `Timeout` (см. выше).

7. **`LogEntry`:** чанкирование по **`LogEntryMaxLineBytes`**; терминальное **`Error`**.

8. **`DisableNetwork`:** политика сети — **5.5** / compose.
