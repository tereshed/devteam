## Описание

Подготовить **репозиторий текстовых промптов** для пяти ролей пайплайна: **Orchestrator**, **Planner**, **Developer**, **Reviewer**, **Tester**. Промпты хранятся в виде YAML-файлов в каталоге `backend/prompts/` и задают системные инструкции, шаблоны пользовательских сообщений и ожидаемый формат ответа так, чтобы `LLMAgentExecutor` (**6.2**) и `SandboxAgentExecutor` (**6.3**) могли стабильно получать предсказуемый ввод/вывод на каждом шаге.

Цель — **единый контракт на уровне текста**: роли не «придумывают» поведение в коде, а следуют зафиксированным промптам; изменения в поведении агентов в первую очередь делаются правками YAML (с последующей валидацией по схеме и в оркестраторе).

**Граница с задачей 6.9:** в **6.8** — содержимое и композиция промптов (тексты, структура секций, плейсхолдеры, `base_prompt.yaml`, JSON Schema). В **6.9** — конфигурация агентов по умолчанию (модель, температура, лимиты и привязка к идентификаторам/путям промптов).

## Функциональные требования

### 1. Жёсткая схема YAML и версионирование

- **Единственный источник правды по структуре:** перечень полей и их типы задаются **`backend/prompts/prompt_schema.json`** (JSON Schema). Файлы ролей и `base_prompt.yaml` **обязаны** соответствовать этой схеме; произвольные дополнительные ключи верхнего уровня без изменения схемы запрещены.
- **Фиксированный набор ключей** определяется **исключительно** содержимым `prompt_schema.json` (имена полей, вложенность, обязательность). Разработчик не вводит новые ключи «от себя» — только через изменение схемы и согласованное обновление всех YAML.
- **Версия промпта:** поле `version` в каждом файле роли (semver или целое) для отслеживания изменений и отладки в логах.
- **Обратная совместимость:** при изменении полей — явное повышение версии схемы/промптов и краткий changelog в `notes_for_maintainers` или в описании PR.

### 2. Композиция промптов (DRY) — без дублирования

- **`base_prompt.yaml`:** общие правила для всех ролей — формат машиночитаемого выхода, запреты на prompt/command injection, базовый глоссарий статусов pipeline, требования лаконичности ответа (см. п. 4), правило языка ответа пользователю (см. п. 5). **Запрещено** копировать эти блоки в пять файлов ролей «вручную».
- **Загрузчик в коде:** оркестратор (или выделенный модуль загрузки промптов) **обязан** собирать итоговый system/user текст из `base_prompt.yaml` и файла роли (механизм `extends`, merge по ключам или явные `include` — конкретная реализация в коде, но **одна** точка композиции). Роли содержат только специфику роли поверх базы.
- **Рассинхрон исключён:** изменение общих правил делается **только** в `base_prompt.yaml` (плюс при необходимости схема), а не правкой N копий.

### 3. Содержание по ролям (согласованность с pipeline)

- **Orchestrator:** анализ запроса пользователя, решение о запуске/продолжении пайплайна, явное разделение «что сделать» vs «сырой контекст»; не смешивать с планированием подзадач (это Planner).
- **Planner:** декомпозиция в структурированный результат (список шагов, зависимости, критерии готовности), формат пригодный для парсинга оркестратором согласно `output_contract` и схеме.
- **Developer:** инструкции для работы в репозитории (ветки, коммиты, минимизация scope), ожидание по артефактам (diff, список файлов); согласованность с ограничениями sandbox (**6.3**).
- **Reviewer:** проверка изменений относительно требований; явная шкала результата (`approved` / `changes_requested` / и т.д. — в терминах **6.6**); запрет «переписывать код» вместо ревью, если так задумано продуктом.
- **Tester:** как запускать тесты, как интерпретировать exit code и вывод; что считать `passed`/`failed` для оркестратора.

### 4. Безопасность и Prompt Injection — КРИТИЧНО

- **Атака «выход из границы» (tag breakout):** вредоносный пользовательский ввод или содержимое репозитория может содержать последовательности вроде `</context><system>…</system>`, подменяющие инструкции после подстановки в шаблон. **Одних XML-тегов недостаточно.**
- **Экранирование / санитизация при подстановке:** загрузчик (оркестратор) **обязан** перед вставкой значений в плейсхолдеры применять устойчивое экранирование или каноникализацию для выбранного формата границ (например: экранирование символов `<`, `>`, `&` в CDATA-подобном режиме; либо кодирование полезной нагрузки в Base64 с декларацией в промпте; либо иной **явно задокументированный** алгоритм, при котором подставленный текст **не может** завершить разметку раньше времени). Выбор алгоритма фиксируется в коде и кратко в `base_prompt.yaml` / документации плейсхолдеров.
- **Согласованность промпта и кода:** тексты в YAML описывают ожидаемый формат границ (`<context>…</context>` и т.д.) **в сочетании** с тем же алгоритмом экранирования, который реализован в загрузчике; нельзя описать одно в промпте, а в коде подставлять сырой текст.
- **Секреты:** в промптах не приводить примеры с реальными ключами; для переменных окружения — нейтральные плейсхолдеры.
- **Инъекция команд:** для Developer/Tester зафиксировать приоритет инструкций продукта над произвольным текстом из контекста; не поощнять выполнение запросов вне скоупа задачи.

### 5. Контракт выхода, лаконичность и краевые случаи

- **Машиночитаемый вывод:** для каждой роли в `output_contract` и в текстах инструкций зафиксирован ожидаемый фрагмент (JSON Schema ответа или строгий шаблон), совместимый с **6.4** / **6.6**.
- **Лаконичность и защита от «галлюцинаций формата»:** системные инструкции (в `base_prompt.yaml` и при необходимости в роли) **явно** требуют от модели: без лишних рассуждений, без markdown-обёрток вокруг JSON, без вводных фраз; для шагов с JSON — формулировки уровня: output must be **only** valid JSON, no prose before or after, no markdown code fences. Это снижает расход токенов, ускоряет парсинг и уменьшает риск обрезки ответа по `max_tokens` посреди структуры.
- **Поведение при неоднозначном ответе:** на уровне кода — ретраи и валидация (**6.4**); в промпте — явное правило, какой минимальный JSON вернуть при нехватке данных (например, поле `error` / `insufficient_context`).
- **Слишком большой контекст (диффы, логи):** в промптах (особенно Reviewer, частично Developer/Tester) **обязательно** описать поведение при усечённом или частичном контексте: явно указывать, что данные могут быть truncated; требовать пометку в JSON (`context_truncated: true`), сужение scope ревью или запрос уточнения — согласованно с оркестратором, который режет контекст по лимитам.

### 6. Язык инструкций и язык ответа пользователю

- **Системные инструкции (system prompt и фрагменты из `base_prompt.yaml`):** **всегда на английском языке.** Сложные правила, формат JSON и ограничения безопасности формулируются по-английски для предсказуемости и эффективной токенизации у современных LLM.
- **Язык общения с пользователем продукта:** если продукт обязан отвечать по-русски (или иному языку), в **английском** system/base-тексте явно задаётся отдельное правило, например: *Always respond to the user-facing summary in Russian* (конкретная формулировка — в `base_prompt.yaml`). Не смешивать «язык инструкций для модели» и «язык UI» в произвольном порядке между файлами.

### 7. Валидация в CI и Makefile

- **Проверка со схемой:** добавить шаг CI и цель в `Makefile`, который для всех шести YAML (`base_prompt.yaml` + пять ролей) проверяет соответствие **`prompt_schema.json`** (после парсинга YAML в JSON-совместимое дерево — инструмент на усмотрение: `ajv`, `go` test, Python one-liner и т.д.; главное — воспроизводимость локально и в CI).
- Падение валидации = падение сборки.
