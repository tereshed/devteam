# 7.6 Интеграция: SandboxRunner → EventBus для стриминга логов

**Sprint:** 7 — WebSocket и Реалтайм (Backend)
**Файл:** `backend/internal/sandbox/docker_runner.go`
**Статус:** ⬜

---

## Описание

Интегрировать **`SandboxRunner` (5.x)** с **`EventBus` (7.4)** так, чтобы **строки логов агента** из контейнера (stdout/stderr, см. `LogEntry` в 5.4/5.6) **в реальном времени долетали до фронта** через WS-канал проекта (7.1, 7.2).

Цель — пользователь, открывший экран задачи в Flutter, **видит вывод Claude Code / Aider** живьём, как `docker logs -f`, без поллинга и без чтения REST.

В этой задаче — **продюсер событий из sandbox**. **Sanitize/маскирование/whitelist** живут в **HubBridge (7.4)**, sandbox **сырые строки** просто публикует.

---

## Контекст использования

```
DockerSandboxRunner.RunTask(ctx, opts)         (опц. с EventBus)
        │
        ├─ ContainerCreate / Start
        │
        └─ go r.streamLogsToBus(ctx, instance)
                ├─ открывает StreamLogs (5.6) → <-chan LogEntry
                ├─ для каждого LogEntry:
                │     bus.Publish(ctx, ws.SandboxLogEmitted{...})
                │     seq++ (монотонный per-sandbox)
                ├─ при truncated/drop из 5.6 → флаг Truncated в Event
                └─ закрытие канала → return (без утечек горутин)
```

EventBus (7.4) → HubBridge (7.4) → `ws.Hub.SendToProject` (7.1) → WS-клиенты.

## Функциональные требования

### 1. Интерфейс `LogPublisher` (Clean Architecture, DIP)

Sandbox-пакет **не должен импортировать `ws`** — это нарушает Clean Architecture: домен зависит от инфраструктуры. Применяем **инверсию зависимостей**:

**Пакет `sandbox`** определяет только интерфейс с доменными типами:

```go
// backend/internal/sandbox/publisher.go
// LogPublisher публикует логи песочницы.
// Реализация находится вне sandbox — в ws-пакете или main.go.
// Sandbox не знает про ws.EventBus и ws.SandboxLogEmitted.
type LogPublisher interface {
    Publish(ctx context.Context, projectID, taskID, sandboxID uuid.UUID, seq int64, entry LogEntry) error
}
```

**Параметры ID** передаются через сигнатуру, а не через поле структуры. Это позволяет использовать один интерфейс для разных инстансов.

Опция для подключения реализации:

```go
// WithLogPublisher подключает произвольный LogPublisher.
// Если не задан — стрим логов наружу НЕ запускается; StreamLogs остаётся
// доступным напрямую через метод (для интеграционных тестов и MCP).
func WithLogPublisher(p LogPublisher) RunnerOption {
    return func(r *DockerSandboxRunner) {
        r.publisher = p
    }
}
```

**Пакет `ws`** (или `cmd/api/main.go`) содержит адаптер, который реализует `sandbox.LogPublisher` и перекладывает данные в `ws.EventBus`:

```go
// backend/internal/ws/sandbox_log_adapter.go
// SandboxLogAdapter реализует sandbox.LogPublisher и публикует в ws.EventBus.
// Маскирование секретов — здесь, до попадания в шину.
type SandboxLogAdapter struct {
    bus ws.EventBus
}

func (a *SandboxLogAdapter) Publish(ctx context.Context, projectID, taskID, sandboxID uuid.UUID, seq int64, entry sandbox.LogEntry) error {
    // Маскируем секреты ДО публикации в шину
    masked := sanitizeLine(entry.Line)

    ev := ws.SandboxLogEmitted{
        ProjectID:  projectID,
        TaskID:     taskID,
        SandboxID:  sandboxID,
        Stderr:     entry.Stderr,
        Line:       masked,
        Seq:        seq,
        Truncated:  entry.Truncated,
        OccurredAt: entry.Timestamp,
    }
    a.bus.Publish(context.WithoutCancel(ctx), ev)
    return nil
}

func sanitizeLine(line string) string {
    // Единый реплейсер на основе env SANDBOX_MASK_PATTERNS
    // ...
}
```

**Сборка в `main.go`:**

```go
bus := ws.NewEventBus()
adapter := &ws.SandboxLogAdapter{bus: bus}
runner := sandbox.NewDockerRunner(
    sandbox.WithLogPublisher(adapter),
    // остальные опции 5.x...
)
```

**Важно:** Bus **опционален**. Это сохраняет backwards-compat с существующими 5.x тестами и позволяет использовать раннер в окружениях без WS-стека.

### 2. Где вешать публикацию (паттерн «log pump») и механизм Fan-Out

После успешного старта контейнера **в `RunTask`** запустить отдельную горутину `streamLogsToBus`. Раннер сам открывает **один** `StreamLogs` и раздаёт копии читателям через **tee-канал** (fan-out):

```
StreamLogs (один Docker API вызов)
        │
        ├── tee-канал (cap N, не буферизировать default)
        │       │
        │       ├── читатель 1: r.logChan → внешний StreamLogs (для пользователя)
        │       │
        │       └── читатель 2: pump-горутина → publisher.Publish
        │
        └── при закрытии оригинального канала оба читателя получают EOF
```

**Реализация tee (обязательно):**

```go
// tee создаёт n каналов-дублёров из одного источника.
// Все получатели читают из своих каналов; закрытие src закрывает все.
// При ошибке чтения из src — все каналы закрываются с ошибкой.
func tee(src <-chan LogEntry, n int) []<-chan LogEntry {
    chs := make([]<-chan LogEntry, n)
    out := make([]chan LogEntry, n)
    for i := 0; i < n; i++ {
        chs[i] = out[i]
    }
    go func() {
        defer func() {
            for i := 0; i < n; i++ {
                close(out[i])
            }
        }()
        for entry := range src {
            for i := 0; i < n; i++ {
                select {
                case out[i] <- entry:
                default:
                    // получатель не успевает — пропускаем (slow consumer)
                }
            }
        }
    }()
    return chs
}
```

Если в 5.6 уже есть механизм мультиплексирования и второй `StreamLogs` разрешён — используем его. Если нет — **tee обязателен**.

**Почему не читать один канал в двух местах:** горутины будут «воровать» записи друг у друга (race на select). Tee решает это: src читается в одной горутине, рассылает всем получателям атомарно.

### 3. Маппинг `LogEntry → ws.SandboxLogEmitted` и маскирование секретов

Маппинг находится **в адаптере `ws.SandboxLogAdapter`** (см. раздел 1), не в sandbox-пакете. Sandbox вызывает `publisher.Publish(ctx, projectID, taskID, sandboxID, seq, entry)`, передавая доменный `LogEntry`. Адаптер в ws-пакете делает:

1. **Маскирование секретов** — `sanitizeLine(entry.Line)` до попадания в шину.
2. **Формирование `ws.SandboxLogEmitted`** — маппинг полей.
3. **Публикация в `ws.EventBus`** — `context.WithoutCancel(ctx)`.

Это гарантирует, что `ws.SandboxLogEmitted` никогда не создаётся внутри sandbox, и HubBridge получает уже маскированные строки.

### 4. Признак `Truncated` (slow-consumer drop marker)

`5.6` уже:

- Чанкует строки длиннее `LogEntryMaxLineBytes`.
- Помечает chunk-маркер при slow consumer (см. `TestStreamLogs_SlowConsumerSeesDropMarker`).

Источник истины — поле(я) в `LogEntry`. Этот тикет **не** должен переизобретать чанкование, только корректно прокинуть `Truncated` во внешний event:

- `Truncated: true` для не-последнего чанка строки.
- `Truncated: true` для специального drop-marker entry (slow consumer).
- `Truncated: false` для обычных и последних чанков.

Если в 5.6 на момент реализации этих полей **нет** — открыть багу/расширение в 5.6 как блокер.

### 5. Монотонный `Seq` per-sandbox

- Счётчик инициализируется в **0** при старте `streamLogsToBus`.
- Атомарно инкрементируется на **каждый** опубликованный event.
- Уникален в рамках **одного sandbox-инстанса**. После `Cleanup` инстанса нумерация сбрасывается (для нового запуска).
- Назначение — фронт переупорядочивает события, пришедшие через несколько reconnect'ов; пробелы в `Seq` ⇒ был дроп.

### 6. Lifecycle горутины (КРИТИЧНО — нет утечек)

Контекст **не должен храниться в структуре раннера**. Его жизненный цикл привязан к конкретному `SandboxInstance` — при `Cleanup` инстанса контекст отменяется.

```go
// RunTask получает rootCtx для pump-горутины
func (r *DockerSandboxRunner) RunTask(rootCtx context.Context, opts SandboxOptions) (*SandboxInstance, error) {
    inst, err := r.startContainer(rootCtx, opts)
    if err != nil { return nil, err }

    if r.publisher != nil && opts.ProjectID != uuid.Nil && opts.TaskID != uuid.Nil {
        // Создаём канал для сигнала shutdown при Cleanup/Stop
        stopCh := make(chan struct{})
        inst.onCleanup = func() { close(stopCh) }

        // Контекст для pump живёт пока не закроется stopCh или rootCtx
        pumpCtx, pumpCancel := context.WithCancel(rootCtx)
        go r.streamLogsToBus(pumpCtx, pumpCancel, stopCh, inst, opts)
    }

    return inst, nil
}
```

| Сигнал | Что делать |
|--------|-----------|
| `stopCh` closed (Cleanup/Stop) | `pumpCancel()` → pump exit |
| `rootCtx` cancelled (SIGTERM) | pump exit через pumpCtx |
| Канал `StreamLogs` закрылся | normal exit (контейнер завершился) |
| Падение Docker daemon | канал закрывается с ошибкой → exit + WARN-лог |

**Запрещено:**
- Привязывать lifecycle к `ctx` HTTP-запроса (запрос отвалится — пропустим логи).
- Хранить контекст в полях структуры `DockerSandboxRunner` — контекст живёт только внутри `RunTask`/`streamLogsToBus`.
- Использовать `context.Background()` без shutdown-сигнала (горутина переживёт graceful shutdown сервиса).

Тест на отсутствие утечек — обязателен через `goleak`.

### 7. Backpressure и взаимодействие с Bus

- `ws.SandboxLogAdapter.Publish` internally calls `bus.Publish` (неблокирующий, см. 7.4); slow-subscriber дропается на стороне Bus.
- Между sandbox и Bus **дополнительной буферизации не нужно** — `StreamLogs` уже буферизован (`StreamLogsDefaultBuffer`).
- Если `bus.Publish` дропнул event — счётчик `Seq` всё равно инкрементирован (это правильно: фронт по дыре в Seq поймёт, что был дроп).

### 8. Безопасность

- **Маскирование секретов — в `ws.SandboxLogAdapter.Publish`** (до попадания в шину). Секреты (API-клюи LLM, токены) не должны лететь по внутренней шине EventBus в открытом виде. HubBridge делает whitelist/форматирование, но не маскирование.
- Единый `sanitizeLine(line)` реплейсер на основе env `SANDBOX_MASK_PATTERNS`. HubBridge получает уже маскированные строки — дублирования нет.
- **Контейнер не имеет доступа к секретам бэка** (см. `main.mdc` §4.8: env-only, no host secrets) — sandbox не сможет «выплюнуть» наш `SECRET_KEY`.
- **Не публиковать `opts.EnvVars`** или иные приватные поля `SandboxOptions` в события — только то, что в `ws.SandboxLogEmitted` по схеме 7.3.

### 9. Метрики

| Метрика | Тип | Назначение |
|---------|-----|------------|
| `sandbox_log_events_published_total` | Counter (label: `task_id` — **нет**, кардинальность; label: `stream`) | Сколько событий ушло в Bus |
| `sandbox_log_pump_active` | Gauge | Сколько активных pump-горутин (для лика) |
| `sandbox_log_pump_errors_total` | Counter | Ошибки чтения из StreamLogs |

**ВАЖНО:** не использовать `task_id`/`sandbox_id` как лейблы Prometheus — кардинальность взорвётся.

### 10. Совместимость с существующими тестами 5.x

- `WithLogPublisher` — **новая** опция, без неё поведение раннера **идентично** текущему.
- Ни один существующий unit/integration тест 5.x **не должен сломаться**.
- Перед коммитом — `make test-unit` + `make test-integration` зелёные.

## Дизайн-решения

1. **Интерфейс `LogPublisher` в sandbox, адаптер в ws:** Clean Architecture + DIP. Пакет sandbox определяет только интерфейс с доменными типами, не знает про `ws.EventBus` и `ws.SandboxLogEmitted`. Адаптер `ws.SandboxLogAdapter` находится в ws-пакете.
2. **Bus как опция, не обязательная зависимость:** не ломаем 5.x тесты и контракты, MCP-инструменты для `sandbox_*` (если будут) работают без WS-стека.
3. **Отдельная горутина-pump в `RunTask`:** изоляция lifecycle стрима логов от пользовательских вызовов `StreamLogs`. Внешние потребители (например, integration tests) могут продолжать пользоваться `StreamLogs` напрямую.
4. **Контекст живёт в `RunTask`, не в структуре:** lifecycle pump привязан к `SandboxInstance` через `stopCh`, не через stored ctx.
5. **Маскирование секретов в адаптере, до публикации:** единый реплейсер в `ws.SandboxLogAdapter.Publish`; HubBridge работает с уже маскированными строками.
6. **Tee для fan-out:** единый `StreamLogs` читается в одной горутине, рассылается всем получателям атомарно — не race condition.
7. **`Seq` инкрементируется и при дропе в Bus:** даёт фронту достоверный признак потери — пробел в Seq.
8. **Без лейбла `task_id` в метриках:** prevented Prometheus cardinality bomb.

## Анти-паттерны (что ЗАПРЕЩЕНО)

| Анти-паттерн | Почему |
|--------------|--------|
| `panic` в pump-горутине | `backend.mdc` §2.1 — никаких паник в обычном коде |
| Импорт `ws` в sandbox-пакете | Нарушение Clean Architecture: домен не зависит от WS |
| Маскирование секретов в HubBridge, а не в адаптере | Секреты летят по шине в открытом виде |
| Определение `ws.SandboxLogEmitted` в sandbox-пакете | Маппинг находится в `ws/sandbox_log_adapter.go` |
| Хранить контекст в полях `DockerSandboxRunner` | Контекст должен жить в scope `RunTask`/`streamLogsToBus` |
| `r.detachedCtx()` как поле структуры | Контекст отвязан от lifecycle инстанса |
| Читать один канал в двух горутинах без tee | Race: горутины воруют записи друг у друга |
| time.Sleep для решения «слишком быстро» | Backpressure уже есть в 5.6 и Bus |
| Глобальная переменная Bus | `backend.mdc` §2.2 |
| Передача `bus` в каждый метод раннера | Вешаем на структуру через опцию (как и остальные опции 5.x) |
| Дополнительный `StreamLogs` под капотом без tee, нарушающий `ErrStreamAlreadyActive` | Сломает существующие тесты 5.6 |

## Критерии приёмки

- [ ] Интерфейс `sandbox.LogPublisher` определён в `sandbox/publisher.go`; пакет sandbox **не импортирует `ws`**.
- [ ] `WithLogPublisher(p LogPublisher)` подключает реализацию; без опции поведение раннера не меняется.
- [ ] Адаптер `ws.SandboxLogAdapter` реализует `sandbox.LogPublisher` и находится в `ws/sandbox_log_adapter.go`; маппинг `LogEntry → ws.SandboxLogEmitted` — там же.
- [ ] `RunTask` после успешного старта контейнера запускает pump-горутину **только если** `publisher != nil` и `opts.ProjectID`/`opts.TaskID` валидны (не `uuid.Nil`).
- [ ] Контекст для pump создаётся внутри `RunTask`, передаётся параметром, **не хранится** в полях `DockerSandboxRunner`.
- [ ] При `Cleanup`/`Stop` закрывается `stopCh` → `pumpCancel()` → горутина завершается; `goleak` подтверждает отсутствие утечек.
- [ ] `streamLogsToBus` использует **tee** для fan-out — единый апстрим из `StreamLogs` рассылается всем получателям атомарно. `ErrStreamAlreadyActive`-контракт не нарушается.
- [ ] На каждый `LogEntry` вызывается `publisher.Publish(...)` ровно один раз; маппинг в `ws.SandboxLogEmitted` — в адаптере.
- [ ] **Маскирование секретов выполняется в `ws.SandboxLogAdapter.Publish` ДО публикации в шину**. HubBridge получает уже маскированные строки.
- [ ] `Seq` передаётся как параметр в `Publish`, инкрементируется атомарно, **не сбрасывается** при дропах в Bus.
- [ ] `Truncated: true` корректно прокидывается из chunk-маркеров и slow-consumer drop-marker'ов из 5.6. Если требуемых полей в 5.6 ещё нет — открыта блокирующая бага на расширение `LogEntry`/маркеров.
- [ ] `opts.EnvVars` и иные приватные поля `SandboxOptions` **не попадают** в события.
- [ ] Метрики экспонированы: `sandbox_log_events_published_total`, `sandbox_log_pump_active`, `sandbox_log_pump_errors_total`. **Без** лейблов высокой кардинальности.
- [ ] Существующие unit/integration тесты `5.x` зелёные; новые юнит-тесты `eventbus_pump_test.go`:
  - Без `WithEventBus` — pump не запускается.
  - С `WithEventBus` и `RunTask` — публикация ровно N событий на N `LogEntry`.
  - `Cleanup` → горутина завершается, `goleak` чист.
  - При `bus.Publish` дропе — `Seq` всё равно инкрементируется.
  - Не публикуем при `opts.ProjectID == uuid.Nil` или `opts.TaskID == uuid.Nil`.
- [ ] DI обновлён в `cmd/api/main.go`: при создании раннера передаётся `bus`.
- [ ] `make test-unit` и `make test-integration` зелёные.
- [ ] Никаких прямых обращений из sandbox-пакета к `ws.Hub` (подтверждено grep'ом). Только через `LogPublisher` → `ws.EventBus`.
