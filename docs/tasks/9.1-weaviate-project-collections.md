# 9.1 Коллекция Weaviate per project: `DevTeam_Project_{id}`

**Sprint:** 9 — Vector DB & Knowledge Base
**Файл:** `backend/internal/vectordb/weaviate_client.go`
**Статус:** ✅ Завершено

---

## Описание

Задача заключается в реализации динамического создания коллекций (классов) в Weaviate для каждого отдельного проекта. Вместо использования одной общей коллекции, каждый проект должен иметь свою изолированную коллекцию с именем формата `DevTeam_Project_{project_id}`. Это обеспечит лучшую производительность при поиске, упростит удаление данных проекта и повысит безопасность за счет изоляции данных.

---

## Зависимости

**Требует реализации:**
- `vectordb.Client` (интерфейс для работы с векторной БД)
- `service.ProjectService` (для получения ID проекта и проверки прав доступа)

**Библиотеки:**
- `github.com/weaviate/weaviate-go-client/v4`
- `golang.org/x/sync/singleflight` (для предотвращения Race Conditions при создании схем)

---

## Функциональные требования

### 1. Форматирование и валидация имени коллекции
- Имя коллекции должно строго соответствовать шаблону `DevTeam_Project_{id}`, где `{id}` — это UUID проекта без дефисов.
- **Критично:** Метод `GetClassName(projectID)` должен выполнять строгую валидацию `projectID`. Если ID пустой или не является валидным UUID, возвращать ошибку валидации немедленно (Early Return).
- Weaviate требует, чтобы имена классов начинались с заглавной буквы и содержали только алфавитно-цифровые символы.

### 2. Алгоритм работы с коллекциями (Cache-Aside)
Для исключения лишних сетевых запросов и обеспечения производительности используется следующий флоу:
1. **In-memory Cache:** Проверка наличия `projectID` в локальном кэше клиента (`sync.Map`).
2. **Hit:** Если коллекция помечена как существующая — сразу переходим к операции (Upsert/Search).
3. **Miss:** Если в кэше пусто — вызываем `EnsureCollection(ctx, projectID)`.
4. **Singleflight + Context:** Внутри `EnsureCollection` используем `singleflight.Group` для атомарной проверки/создания класса в Weaviate.

### 3. Предотвращение Race Conditions и утечек горутин
- **Критично:** Использование `singleflight.DoChan` вместо `Do`. Это необходимо для корректной обработки отмены контекста (`ctx.Done()`). Если Weaviate зависнет, вызывающая горутина должна освободиться по таймауту, не дожидаясь завершения функции внутри `singleflight`.
- Схема должна включать поля: `content` (text), `metadata` (object), `source` (string).

### 4. Безопасность и Изоляция
- `vectordb.Client` работает как низкоуровневый компонент, принимающий `projectID`.
- **Критично:** Проверка прав доступа (RBAC/ACL) должна выполняться на уровне `service` или `handler`. Пользователь должен иметь доступ к `projectID` прежде, чем запрос попадет в `vectordb.Client`.

### 5. Удаление данных и Транзакционность (Outbox Pattern)
- При удалении проекта соответствующая коллекция в Weaviate должна удаляться.
- **Гарантия удаления:** Для предотвращения "сиротских" коллекций при сбоях сети использовать паттерн **Transactional Outbox** или **Soft Delete**:
    1. В транзакции Postgres проект помечается как удаленный (или создается запись в таблице `outbox_events`).
    2. Фоновый воркер пытается удалить коллекцию в Weaviate.
    3. Только после успешного подтверждения от Weaviate задача в Outbox помечается как выполненная.
- Логирование `ERROR` с деталями для алертинга обязательно.

## Технические подробности

### Безопасное использование singleflight с контекстом
```go
func (c *Client) EnsureCollection(ctx context.Context, projectID string) error {
    className := c.GetClassName(projectID)
    
    // Используем DoChan для поддержки context cancellation
    ch := c.sfGroup.DoChan(className, func() (interface{}, error) {
        // 1. Повторная проверка Weaviate (на случай если кэш был пуст, но класс уже создан)
        // 2. Создание класса, если его нет
        return nil, c.createClass(ctx, className)
    })

    select {
    case <-ctx.Done():
        return ctx.Err()
    case res := <-ch:
        if res.Err == nil {
            c.cache.Store(projectID, true) // Обновляем кэш только при успехе
        }
        return res.Err
    }
}
```

## План реализации

| # | Шаг | Описание |
|---|-----|----------|
| 1 | `GetClassName(projectID)` | Реализовать хелпер со строгой валидацией UUID. |

| 3 | `EnsureCollection` | Реализовать `singleflight.DoChan` с обработкой `select { case <-ctx.Done() }`. |

| 5 | `Integration Tests` | Тесты на Race Condition при создании и проверку утечек горутин при таймаутах. |
