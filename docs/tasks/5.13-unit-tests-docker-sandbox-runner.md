## Описание

Добавить **юнит-тесты** для **`DockerSandboxRunner`** (реализация **5.5**), проверяющие контракт **`SandboxRunner`** (**5.3**), корректность работы с **`context.Context`**, идемпотентность **`Cleanup`**, обработку ошибок Docker Engine API и **потокобезопасность** критических участков — **без** поднятия реального Docker daemon и **без** запуска контейнеров на хосте CI.

**«Мок Docker Client»** в этой задаче означает: тестовый **HTTP transport** к **подменённому** Docker API (как минимум **`net/http/httptest.Server`** + **`github.com/docker/docker/client.NewClientWithOpts`**, см. уже существующий паттерн в **`stream_logs_test.go`**: **`newTestDockerClient`**, **`testSandboxID`**). Достаточно реализовать **минимальный подмножество** эндпоинтов и тел ответов, которые реально вызывает текущий **`docker_runner.go`** (mux с диспетчеризацией по `URL.Path` + нужные коды/JSON). Альтернатива — узкий интерфейс-адаптер над вызовами SDK с **`testify/mock`** — **допустима только** если она **не раздувает** прод-абстракции; по умолчанию предпочтителен **HTTP double**, чтобы не дублировать сигнатуры всего SDK.

**Граница с 5.14:** всё, что требует **`docker.sock`**, реального образа и exit-кода entrypoint — **строго** в **`//go:build integration`** и **`make test-integration`** (отдельный файл, например `integration_test.go`). В **`make test-unit`** таких зависимостей **нет**.

## Go-специфика: `-race` и `t.Parallel()` (обязательно)

- **`go test -race`:** весь пакет `sandbox` в рамках PR по **5.13** должен проходить с детектором гонок. Если корневой **`make test-unit`** пока без `-race`, в PR явно зафиксировать успешный прогон `cd backend && go test -race ./internal/sandbox/... -count=1` (и при необходимости — отдельная задача на Makefile).
- **`t.Parallel()`:** в каждом независимом `Test*` и в подтестах `t.Run` — **`t.Parallel()`** первым осмысленным шагом (как в **4.8**). Исключения — только с **комментарием**, почему тест обязан быть последовательным (например, глобальный счётчик в mux, который намеренно проверяет последовательность — крайне редко).

## DRY и структура

- **Один** хелпер уровня **`newDockerRunnerTestMux(t, …)`** или **`setupDockerRunnerWithHandler(t, h http.Handler) (*DockerSandboxRunner, *client.Client, cleanup)`**, возвращающий раннер с **`NewDockerSandboxRunner(newTestDockerClient(t, mux), allowedImages, …opts)`**. **Не** копировать десятки строк регистрации `httptest` в каждом тесте.
- **Закрытие `httptest.Server` и FD (критично при `t.Parallel()`):** любой хелпер, поднимающий **`httptest.Server`**, **обязан** гарантировать **`Close()`** после теста:
  - либо **`t.Cleanup(srv.Close)`** внутри хелпера (предпочтительно — как уже сделано в **`newTestDockerClient`** через **`t.Cleanup(srv.Close)`**);
  - либо возвращаемая **`cleanup func()`**, которую вызывают через **`t.Cleanup(cleanup)`** у вызывающего.
  **Запрещено** оставлять сервер без cleanup: при сотнях параллельных тестов CI быстро упирается в **исчерпание файловых дескрипторов** и нестабильные флейки.
- **Изоляция mux на тест:** при **`t.Parallel()`** у каждого теста — **свой** экземпляр сервера/хендлера (не общий глобальный mux на пакет без синхронизации), иначе гонки по счётчикам вызовов и общая мутация состояния.
- Табличные тесты для: невалидный **`sandboxID`**, маппинг **`ContainerInspect`** → **`SandboxStatus`**, ошибки allowlist образа, и т.п.
- **`newTestDockerClient`** уже есть в **`stream_logs_test.go`** — при росте дублирования **вынести** в общий `*_test.go` того же пакета (см. «Файлы (план)» выше); при выносе **сохранить** контракт **`t.Cleanup`** на закрытие сервера.

## Область покрытия (минимально ожидаемые сценарии)

### A. `RunTask`

- **До Docker API:** невалидные **`SandboxOptions`** → ожидаемые ошибки из **`opts.Validate`** / **`ErrInvalidOptions`**; образ **вне** `allowed` → отказ **до** `ContainerCreate` (контракт **5.5**).
- **Быстрый фейл: пустые обязательные поля (до любого вызова mux / Docker HTTP):** table-driven кейсы с **`errors.Is(..., ErrInvalidOptions)`** (и при необходимости **`ErrInvalidTaskID`**) и проверкой, что **mux не получил** ни одного запроса (счётчик хендлера = 0), в частности:
  - **`TaskID == ""`** → отказ на **`ValidateTaskID`** / **`validateWithoutResourceLimits`**;
  - **`Image == ""`** → **`ErrInvalidOptions`** (сообщение про пустой image — см. **`options_validate.go`**);
  - другие обязательные поля по контракту **`SandboxOptions.Validate`** (пустой **`Backend`**, пустой **`Instruction`**, и т.д.) — аналогично «ошибка + ноль HTTP-вызовов».
- **Безопасность входных данных (см. `.cursor/rules/review.mdc`, Security):**
  - **Command / flag injection:** **`Instruction`** и **`Context`** в текущей реализации **не** должны собираться в **`exec.Command`** на хосте и **не** должны подставляться в один **`sh -c "..."`** на стороне Go; контейнер получает payload через **tar → `CopyToContainer`** и фиксированный **`WorkspacePath`**. Тесты обязаны **зафиксировать регрессии**: payload’ы с shell-метасимволами и типичными векторами (`; rm -rf /`, подстроки в духе **`--upload-pack`**) в **`Instruction`/`Context`** **не** ломают пайплайн там, где это допустимо по продукту, а опасные значения в **`Branch`**, **`RepoURL`** и других «командных» полях — **отсекаются валидаторами** (**`ValidateBranchName`**, **`ValidateRepoURL`**, …) **до** Docker API. Минимум одна **таблица негативных** значений для полей, которые участвуют в git/remote-логике entrypoint, с **быстрым фейлом** и **нулём** вызовов create/start в mux.
  - **Path traversal и `CopyToContainer`:** в проде имена файлов в tar для prompt/context — **литералы** **`prompt.txt`** / **`context.txt`** (**`buildPromptContextTar`**); путь назначения копирования — **константа** пакета, не из пользовательского ввода. Тесты должны **зафиксировать инвариант**: пользовательский текст **никогда** не попадает в **`tar.Header.Name`** как **`../../../etc/passwd`** или абсолютный путь (например, рефлексия/white-box на результат **`buildPromptContextTar`** или чтение первых байт tar в тесте того же пакета). Если в будущем появится пользовательское имя пути — **обязательны** валидация + негативные кейсы traversal с **early return до** `CopyToContainer`.
- **Успешный путь (упрощённый mux):** последовательность вызовов, которую делает раннер (как минимум: **`ImageInspect`** или **`ImagePull`** по политике кода, **`NetworkCreate`**, **`ContainerCreate`**, **`CopyToContainer`** при наличии instruction/context, **`ContainerStart`**, **`ContainerInspect`** для post-start sanity — **сверить с актуальным** `RunTask` в **`docker_runner.go`**, не с устаревшей выжимкой из спеки 5.5). Проверить, что возвращается **`SandboxInstance`** с **полным** 64 hex ID и ожидаемым начальным статусом.
- **Ошибка на середине цепочки:** после **`ContainerCreate`** — ошибка **`Start`** / **`CopyToContainer`** → best-effort **`ContainerRemove`** (mux фиксирует факт вызова remove); **нет** утечки «зарегистрированного» инстанса в карте (или документированная политика стейта — сверить с кодом).
- **`CreateResponse.Warnings`:** если прод логирует предупреждения — тест может подставить непустые `Warnings` и проверить отсутствие паники / ожидаемое поведение (без хрупкого assert на текст лога, если логгер не подменяется; тогда ограничиться «ветка выполнена»).

### B. `Wait` / завершение ожидания

- **`ValidateSandboxID`:** короткий ID → **`ErrInvalidSandboxID`** без вызова сети.
- **Пустой `sandboxID`:** **`""`** → **`ErrInvalidSandboxID`** (или узкий sentinel по коду **`ValidateSandboxID`**), **без** HTTP к mux — в одной таблице с другими невалидными ID.
- **Успех:** инстанс в стейте с известным `containerID`, mux отдаёт **`ContainerWait`** / финальный inspect в соответствии с реализацией **`containerWaitLoop`** — ожидаемый **`SandboxStatus`** (running / completed / failed — по сценарию).
- **Отмена `context`:** вызов **`Wait`** с отменённым ctx → **`context.Canceled`** (или `DeadlineExceeded`); mux **не** должен зависать (таймаут теста — страховка).
- **Два канала `ContainerWait`:** если тест воспроизводит отмену ctx во время wait — убедиться, что нет зависания горутин (в связке с **`goleak`**; при необходимости см. комментарии в **5.5** про drain каналов — тест должен соответствовать принятому в коде паттерну).

### C. `GetStatus`

- Валидный ID + JSON inspect от mux → корректный маппинг в **`SandboxStatus`** (хотя бы **running** и **exited**).
- **`errdefs.NotFound`** / 404 от API → **`ErrSandboxNotFound`** (или актуальный sentinel из **`errors.go`** — сверить с кодом).

### D. `Stop` / `StopTask`

- Успешный вызов **`ContainerStop`** (или политика **`stopper`** — мок **`ContainerStopper`** уже используется в **`lifecycle_manager_test.go`**; для **`DockerSandboxRunner.Stop`** предпочтительно проверить через mux вызовы к Engine, если они идут напрямую в клиент).
- Идемпотентность / повторный вызов — по контракту кода.

### E. `Cleanup`

- Удаление контейнера + сеть + temp dir: mux фиксирует **`ContainerRemove`**, **`NetworkRemove`** (если применимо); повторный **`Cleanup`** — **с точки зрения вызывающего** возвращает **`nil`** (идемпотентность).
- **`errdefs.IsNotFound` при `ContainerRemove`:** для бизнес-логики это **ожидаемое** состояние («уже убрали»), метод **не** должен возвращать ошибку наверх. **Антипаттерн (запрещено):** молчаливое **`_ = err`** или отсутствие явной ветки — ошибка **должна классифицироваться** (`if err != nil { if errdefs.IsNotFound(err) { … } else { … } }`).
- **Логирование при `NotFound`:** в ветке **`IsNotFound`** допустимо не считать ситуацию сбоем, но **нужно** явно зафиксировать в логе (ориентир **`slog.Debug`** или однократный **`slog.Warn`** с атрибутами вроде **`sandbox_id`**, **`op`="container_remove"`**), чтобы при отладке было видно, что движок ответил «нет such container», а не «мы проглотили неизвестную ошибку». Если в текущем **`Cleanup`** такого лога **нет** — **входит в объём PR по 5.13** добавить ветку + лог + тест.
- **Тест на лог (обязательно, если добавляется/уже есть ветка лога):** подменить default-логгер через **`slog.SetDefault`** + **`t.Cleanup`** восстановления прежнего логгера; использовать **`slog.Handler`**, накапливающий записи (**`slog.NewJSONHandler` / `TextHandler`** в **`bytes.Buffer`**). Вызвать **`Cleanup`** для несуществующего, но **валидного** ID (mux возвращает **404/NotFound**). **Ассерт:** есть запись с ожидаемым уровнем/атрибутами **или** стабильная подстрока в сообщении; при этом **`Cleanup` == nil`**.
- **`context` отменён:** уборка с **`detachTimeout`/`WithoutCancel`** — не должна полностью проваливаться из‑за отмены пользовательского ctx (сверить с **`removeContainerForceLogged`** и аналогами).

### F. Конкурентность

- Краткий тест: **`Cleanup`** / **`StopTask`** во время «медленного» **`RunTask`** (mux с задержкой на create или start) — инвариант **нет сироты** контейнера и **нет** deadlock под **`-race`** (см. **5.5**, раздел про регистрацию в `creating` до create).

### G. Что уже покрыто соседними файлами

- **`StreamLogs`** — **`stream_logs_test.go`**; в **5.13** **не** дублировать те же сценарии, кроме случая **интеграции** «`RunTask` → сразу `StreamLogs`» если это не покрыто и ловит регресс (опционально один тест).
