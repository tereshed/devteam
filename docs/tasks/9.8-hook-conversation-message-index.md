## Описание

Задача заключается в интеграции механизма индексации сообщений чата в `ConversationService`. Необходимо обеспечить автоматическую индексацию каждого нового сообщения пользователя и системы в векторной базе данных (Weaviate). Это позволит агентам и пользователям выполнять семантический поиск по истории обсуждений в рамках проекта.

---

## Зависимости

- `ConversationIndexer` — сервис, отвечающий за формирование документа сообщения и его сохранение в Weaviate.
- `pkg/async` — общий пакет для безопасного запуска фоновых задач.
- `pkg/sanitizer` — общий пакет для маскирования секретов.

---

## Функциональные требования

### 1. Синхронизация и Идемпотентность (Architecture) — КРИТИЧНО
- **Детерминированные UUID**: UUID объекта в Weaviate **должен** генерироваться на основе ID сообщения из основной БД (например, через `uuid.NewMD5(uuid.NameSpaceOID, []byte(messageID))`). Это обеспечивает автоматический **Upsert** и позволяет точечно обновлять/удалять векторы.
- **Race Conditions**: Использовать Upsert по детерминированному UUID и сохранять `updated_at` в метаданных для минимизации последствий гонок.

### 2. Транзакционность и Жизненный цикл (Consistency)
- **После коммита**: Все вызовы индексации/удаления в Weaviate запускаются асинхронно **строго после** успешного `Commit` транзакции в основной БД.
- **DeleteMessage**: Добавить хук на удаление одиночного сообщения. Удаление из Weaviate выполняется по детерминированному UUID.
- **DeleteConversation (Batch Delete)**: При удалении всей беседы использовать пакетное удаление (Batch) в Weaviate по `conversation_id`.
- **Призраки (INDEX_ORPHAN)**: Логирование критических сбоев удаления для ручной очистки.

### 3. Надежность и Graceful Shutdown (Reliability) — КРИТИЧНО
- **Ожидание завершения**: Пакет `pkg/async` (или сервис через него) обязан отслеживать активные задачи индексации (через `sync.WaitGroup`).
- **Shutdown**: При получении сигнала остановки (SIGTERM) приложение должно давать фоновым задачам разумное время (например, 5-10 секунд) на завершение перед выходом, чтобы избежать потери данных, уже закоммиченных в БД, но не дошедших до индекса.

### 4. Безопасность и Изоляция (Security)
- **Метаданные**: Сохранять `project_id`, `user_id`, `conversation_id` и `role` (автор: `user`, `assistant`, `system`).
- **Native Multi-Tenancy (Weaviate)**: Если коллекция в Weaviate использует Native MT (где tenant = `project_id`), при всех операциях (Upsert, Delete) необходимо явно передавать параметр `Tenant`.
- **Маскирование**: Использовать `pkg/sanitizer` перед отправкой текста в Weaviate.

### 5. Производительность и Качество данных
- **N+1 Protection**: Передавать все необходимые ID в памяти, исключая запросы к БД из горутины.
- **Early Return**: Игнорировать пустые сообщения и технические пинги.
- **Smart Truncate**: Обрезка текста до лимита (`cfg.Indexer.MaxMessageLength`) должна быть **UTF-8 safe** и по возможности по границам слов.

### 6. Наблюдаемость (Observability)
- **Метрики**: Счетчики (Prometheus) для успеха/ошибок фоновых задач.
- **Логирование**: Теги `[INDEX_DROP]` и `[INDEX_ORPHAN]` для алертинга.

## Критерии приёмки

- [ ] Используются детерминированные UUID для Weaviate (Upsert работает корректно).
- [ ] Индексация/удаление запускаются только после коммита транзакции БД.
- [ ] Реализовано удаление одиночного сообщения и пакетное удаление бесед.
- [ ] Система дожидается завершения фоновых задач при выключении (Graceful Shutdown).
- [ ] Учтен Native Multi-Tenancy Weaviate (если применимо).
- [ ] Пустые сообщения игнорируются, длинные — обрезаются безопасно для UTF-8.
- [ ] Внедрены метрики и логирование ошибок.
- [ ] Тесты покрывают все CRUD операции и сценарии надежности.
