# 9.3 Индексатор задач: описание + результат + сообщения

**Sprint:** 9 — Vector DB & Knowledge Base

**Статус:** ⬜ К выполнению

## Описание

Задача заключается в создании компонента `TaskIndexer`, который будет отвечать за индексацию задач (описание, результат выполнения, связанные сообщения) в Weaviate. Это необходимо для того, чтобы в будущем агенты (или пользователи) могли осуществлять семантический поиск по ранее выполненным задачам (например: "как мы реализовывали авторизацию в прошлом месяце?").

В отличие от исходного кода, задача представляет собой структурированный контекст. Индексировать нужно:
1. Заголовок и промпт задачи.
2. Итоговый статус и текстовый результат (CodeResult/Artifacts).
3. Важные сообщения из обсуждения задачи (опционально, если они несут смысловую нагрузку).

---

## Зависимости

**Базовые компоненты:**
- `vectordb.Client` (для сохранения сущностей в Weaviate).
- `models.Task` и `models.TaskMessage` (доменные модели).

**Интерфейсы сборки контекста:**
- Репозитарии `TaskRepository` и `TaskMessageRepository` для вытягивания полных данных перед индексацией.

## Функциональные требования

### 1. Формирование документа (Document Construction)
- Собирать единый текстовый документ (чанк) для задачи, используя `strings.Builder` (или `bytes.Buffer`) для минимизации аллокаций памяти при конкатенации больших объемов текста.
- **Разметка структуры (Prompt Injection Protection):** Для предотвращения слияния контекста и защиты от атак типа Prompt Injection при последующем использовании в RAG, документ должен иметь четкую разметку. Использовать Markdown-заголовки или специальные разделители:
  ```text
  --- TASK TITLE ---
  [Название]
  --- TASK PROMPT ---
  [Описание/Промпт]
  --- AGENT RESULT ---
  [Результат/Артефакты]
  --- DISCUSSION ---
  [Ключевые сообщения]
  ```
- **Маскирование секретов (КРИТИЧНО):** Перед отправкой текста в Weaviate ОБЯЗАТЕЛЬНО проводить санитаризацию. Маскировать API-ключи, токены, пароли БД и другие чувствительные данные. Векторная база не должна хранить сырые учетные данные.
- Включать в документ:
  - Название/цель задачи.
  - Описание (prompt).
  - Статус (completed, failed и т.д.).
  - Вывод агентов (summary/результат).
- Если у задачи длинная история переписки (TaskMessages), объединять ключевые сообщения (или лимитировать по токенам).
- Использовать `tiktoken` (или эвристику длины) для контроля размера: если общий контекст задачи превышает лимит (например, 8k токенов), обрезать или разбивать на логические части (Task Chunk 1, Task Chunk 2).
- **Пустые значения:** Если описание задачи и сообщения отсутствуют, выполнять early return без создания мусорных векторов.

### 2. Метаданные (Metadata)
Каждый объект в векторной БД должен иметь следующие метаданные для фильтрации:
- `task_id` (UUID).
- `project_id` (UUID). **КРИТИЧНО:** При поиске фильтрация по `project_id` должна быть обязательной на уровне запроса.
- `status` (string).
- `agent_role` (string - кто выполнял).
- `created_at` / `completed_at` (timestamps).

### 3. Инкрементальная индексация
- Поддерживать метод `IndexTask(ctx context.Context, taskID uuid.UUID) error`, который можно будет вызывать синхронно или асинхронно при изменении статуса задачи (например, при переходе в `Completed`).
- **Асинхронность и контекст:** При асинхронном вызове (в горутине) необходимо использовать отвязанный от жизненного цикла HTTP-запроса контекст (например, `context.WithoutCancel(ctx)` для Go 1.21+), чтобы избежать ошибки `context canceled`. Горутины должны корректно отслеживаться для обеспечения graceful shutdown.
- Поддерживать метод `DeleteTask(ctx context.Context, taskID uuid.UUID) error` для очистки индекса при удалении задачи пользователем.
- **Удаление проекта:** Реализовать метод `DeleteProjectTasks(ctx context.Context, projectID uuid.UUID) error` для каскадного удаления всех векторов задач при удалении проекта, чтобы избежать появления "сирот" в Weaviate.
- **Консистентность и детерминизм (Upsert):** При повторной индексации использовать детерминированные ID для векторов (например, `UUIDv5(task_id + chunk_index)`) или гарантировать удаление всех старых чанков по `task_id` перед вставкой новых, чтобы избежать дублирования контекста.

### 4. Производительность и Ошибки
- **Fail Gracefully & Sync:** Если Weaviate недоступен, ошибка должна логироваться. Для предотвращения рассинхрона индекса с БД необходимо предусмотреть механизм отложенной обработки (например, флаг `is_indexed = false` в БД или очередь Dead Letter Queue) для последующей доиндексации фоновым джобом.
- **Архитектура слоев (Layers):** `TaskIndexer` не должен содержать GORM-кода или прямых вызовов БД. Вся логика извлечения данных (включая батчинг и Preload) должна быть инкапсулирована в репозиториях (`TaskRepository`, `TaskMessageRepository`).
- **Батчинг (N+1 Prevention):** Метод `IndexProjectTasks` (массовая индексация) ОБЯЗАН использовать пагинацию (например, `FindInBatches` по 100-500 записей) для предотвращения Out Of Memory (OOM) при работе с большими проектами.

## План реализации

| # | Шаг | Описание |
|---|-----|----------|
| 1 | `TaskIndexer Interface` | Создать интерфейс с методами `IndexTask`, `DeleteTask`, `DeleteProjectTasks`, `IndexProjectTasks`. |
| 2 | `Document Builder` | Написать логику сборки текстового представления задачи с использованием `strings.Builder`, разметкой структуры и санитаризацией (маскирование секретов). Реализовать проверку лимита токенов. |
| 3 | `Weaviate Integration` | Интегрировать маппинг полей и детерминированную генерацию ID чанков. *Уточнить схему с существующим `vectordb.Client`.* |
| 4 | `Error Handling` | Реализовать безопасную обработку ошибок, механизм `is_indexed`/DLQ для доиндексации и корректную работу с контекстом в горутинах. |
| 5 | `Unit Testing` | Написать юнит-тесты с моками. Проверить сборку текста, маскирование, разметку и батчинг (пагинацию). |

## Критерии приёмки

- [ ] Интерфейс `TaskIndexer` спроектирован и реализован.
- [ ] Логика объединения задачи и ее сообщений в единый текст корректно обрабатывает длинные тексты (превышение лимита токенов не приводит к панике).
- [ ] Метаданные (`task_id`, `project_id`, `status`) корректно передаются в `vectordb.Client`, реализована обязательная фильтрация по `project_id` при поиске.
- [ ] Реализованы методы для добавления/обновления задачи (с детерминированными ID чанков) и её удаления из индекса.
- [ ] Ошибки Weaviate логируются, предусмотрен механизм (флаг/очередь) для обработки рассинхрона данных.
- [ ] Массовая индексация (`IndexProjectTasks`) использует пагинацию и не вызывает OOM.
- [ ] Логика работы с БД полностью вынесена в репозитории, в индексаторе отсутствует GORM-код.
- [ ] Реализовано маскирование секретов перед индексацией.
- [ ] Написаны юнит-тесты на `Document Builder` (включая проверку `strings.Builder` и маскирования) и саму интеграцию.
