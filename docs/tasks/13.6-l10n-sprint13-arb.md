## `features/settings`: presentation и domain

Grep бьёт по **`frontend/lib/features/settings/`** целиком, включая **`domain/`**. Тексты в **Failures** / доменных исключениях могут оставаться **техническими** (в т.ч. англ.) для логов и отладки. Пользовательские фразы для UI — **только** в **presentation**, через маппинг на `AppLocalizations`; **не** переносить в ARB служебные сообщения исключений «на всякий случай» только потому что файл попал под grep.

## Пример (обязательный паттерн)

```dart
import 'package:frontend/core/l10n/require.dart';
import 'package:frontend/l10n/app_localizations.dart';

// BAD — пользовательская строка в UI без ARB
Text('Настройки команды')

// GOOD — эталон 10.8 / 10.7: where для диагностики отсутствия delegate
final AppLocalizations l10n = requireAppLocalizations(context, where: 'teamScreen');
Text(l10n.teamEmptyAgents)
```

`AppLocalizations.of(context)!` в легаси допустим до миграции в том же PR.

## Предусловия перед началом

1. Template и порядок ключей — как **10.8** §предусловие 1. **`make frontend-l10n-check`:** паритет ключей; для строк с `{…}` — зеркальные `@*.placeholders` в ru и en; скрипт дополнительно сравнивает **набор имён и типов** placeholders (см. `scripts/check_l10n_parity.sh`). Поле **`description`** (в ru и в en) скрипт **не** проверяет — ручная дисциплина и ревью; см. §«Политика `@…` в `app_en.arb`» ниже.

2. Импорт: `l10n.yaml` → **`output-dir: lib/l10n`** → `package:frontend/l10n/app_localizations.dart`.

3. Порядок codegen: **`make frontend-codegen`** = `dart run build_runner build --delete-conflicting-outputs`, затем **`flutter gen-l10n`**. **Нельзя** запускать **`build_runner … --delete-conflicting-outputs` после `flutter gen-l10n`** — этот флаг сносит сгенерированные l10n-файлы ([docs/rules/frontend.md](../rules/frontend.md) §2.3). Обычный `build_runner` **без** `--delete-conflicting-outputs` после `gen-l10n` сам по себе каноном не запрещён (например watch-режимы); риск — именно сочетание с флагом.

4. Импорты Dart — только `package:frontend/...`.

5. `Intl.message` для пользовательского UI не вводить.

## Известная проблема (канон из правил)

**`flutter run`** (в т.ч. через **`make frontend-run-web`**) может запускать **`build_runner`**, который удаляет сгенерированные **`app_localizations*.dart`**, даже при исключениях в `build.yaml`. **Решение:** после такого запуска выполнить **`cd frontend && flutter gen-l10n`** в корректном порядке с codegen (см. [docs/rules/frontend.md](../rules/frontend.md) §2.3 «КРИТИЧНО: Порядок генерации»).

## Политика `@…` в `app_en.arb` (согласовано с 10.8)

В **`app_ru.arb`** — блоки `@…` с **`description`** там, где нужен контекст для переводчиков.

В **`app_en.arb`** обязательно: **`@@locale`**; для строк с **`{…}`** — **`@messageKey.placeholders`** с теми же **`placeholders.<name>.type`**, что в ru.

**Для новых ключей** в скоупе **13.6** (и далее по духу **10.8**): **не вводить** в `app_en` новые произвольные англоязычные **`description`** в `@…` (канон контекста — только в template; иначе двойной SSOT). Если продукт когда-нибудь потребует `description` и в en — только как **перевод** смысла из `app_ru.arb`, не независимый текст.

**Факт репозитория:** уже сейчас во многих `@…` в **`app_en.arb`** есть поле **`description`** (например **`@teamAgentEditFieldTools`** и десятки аналогов в ветках **`team*`** / **`projectSettings*`** / **`globalSettings*`** после **13.3** / **13.3.1** и др.) — это **легаси**. **Не вычищать** их массово в рамках **13.6** (неожиданный объём диффа и шум в ревью). Жёсткое «в en только `@@locale` + placeholders» — отдельный мелкий тикет: либо чек-лист + `jq`-аудит, либо расширение **`check_l10n_parity.sh`** (сейчас скрипт **`description` в en не проверяет** — только паритет ключей и зеркало placeholders).

### Плейсхолдеры и рантайм

| Правило | Описание |
|--------|----------|
| Имена и `type` | Для каждого `{name}` в строке — в **обоих** ARB блок `@messageKey.placeholders` с **одинаковыми** именами и **`type`**. Иначе `gen-l10n` может выставить тип по умолчанию (**`Object`**) и получить **неочевидные ошибки форматирования в рантайме**. **Общие ключи:** `error*`, `retry`, … — переиспользовать; новый ключ только при ином смысле.

**Порядок ключей в ARB:** новые — в конец логической группы.

**`description` в template:** существующие ключи без `description` не обязательно бэкфиллить в 13.6; для **новых и изменённых** — добавлять в `app_ru.arb` при плейсхолдерах или неочевидном UI-контексте.

## Инвентаризация и grep

**Литералы (канон аудита; ловит в т.ч. `const Text('…')` в `dashboard_screen` и т.п.):** в **ripgrep** не использовать флаг **`-E`** (у `rg` это `--encoding`, не extended regexp). Достаточно:

```bash
rg -n "Text\(['\"]|labelText:\s*['\"]|hintText:\s*['\"]|tooltip:\s*['\"]|errorText:\s*['\"]|semanticLabel:\s*['\"]" \
  frontend/lib/features/team \
  frontend/lib/features/settings \
  frontend/lib/features/projects/presentation/screens/project_settings_screen.dart \
  frontend/lib/features/auth/presentation/screens/dashboard_screen.dart \
  frontend/lib/features/auth/presentation/screens/profile_screen.dart \
  frontend/lib/core/routing/app_router.dart
```

Дополнительно (подписи `Semantics`, `Tab`, `helperText` / `prefixText` / `suffixText`, если основная команда не покрыла кейс):

```bash
rg -n '(\b(labelText|hintText|tooltip|errorText|helperText|prefixText|suffixText|semanticLabel)\s*:\s*["\x27])|(Semantics\([^)]*label:\s*["\x27])|(Tab\(\s*text:\s*["\x27])' \
  frontend/lib/features/team \
  frontend/lib/features/settings \
  frontend/lib/features/projects/presentation/screens/project_settings_screen.dart \
  frontend/lib/features/auth/presentation/screens/dashboard_screen.dart \
  frontend/lib/features/auth/presentation/screens/profile_screen.dart \
  frontend/lib/core/routing/app_router.dart
```

```bash
SPRINT13_PATHS=(
  frontend/lib/features/team
  frontend/lib/features/settings
  frontend/lib/features/projects/presentation/screens/project_settings_screen.dart
  frontend/lib/features/auth/presentation/screens/dashboard_screen.dart
  frontend/lib/features/auth/presentation/screens/profile_screen.dart
  frontend/lib/core/routing/app_router.dart
)
rg -n "l10n\\.\\w+" "${SPRINT13_PATHS[@]}" --type dart | sort -u
rg -oN --no-filename -r '$1' '(?:l10n|AppLocalizations\.of\([^)]+\)!?|requireAppLocalizations\([^)]*\))\s*\.\s*(\w+)' "${SPRINT13_PATHS[@]}" --type dart | sort -u
```

В корневом **Makefile** цели вида **`frontend-l10n-debug`** нет — достаточно **`make frontend-l10n-check`**.

**Факт по литералам в скоупе (основной `rg -n`):** (1) **`dashboard_screen.dart`** — три `const Text('…')` (admin); **`profile_screen.dart`** и **`app_router.dart`** — без совпадений. (2) **`agent_edit_dialog.dart`** (~строка **686**) — `FilterChip` с **`Text('${t.name} (${t.category})')`** (интерполяция; основной grep ловит **`Text('`).** Пути в командах grep перечислять **полностью** — дельта после мерджей может расшириться.
