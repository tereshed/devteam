# 2.9 Swagger-аннотации для всех новых эндпоинтов

**Sprint:** 2 — CRUD API для Project и Team
**Файлы:** Все handler'ы в `backend/internal/handler/`
**Статус:** ✅ Выполнена

---

## Описание

Ревизия и стандартизация Swagger-аннотаций **во всех** handler'ах. Текущая проблема: разные handler'ы используют разные комбинации `@Security`, что ломает UX в Swagger UI. Задача приводит все аннотации к единому стандарту и гарантирует, что Swagger UI поддерживает **три способа авторизации**: логин/пароль (OAuth2), Bearer-токен, API-ключ.

---

## Проблема: текущее состояние

### Определения безопасности (main.go) — ✅ корректны

```go
// @securityDefinitions.apikey BearerAuth      // JWT в Header Authorization
// @securityDefinitions.apikey ApiKeyAuth       // API-ключ в Header X-API-Key
// @securityDefinitions.oauth2.password OAuth2Password  // Логин/пароль → токен
```

Swagger UI отображает три кнопки "Authorize":
1. **BearerAuth** — вручную вставить `Bearer <token>`
2. **ApiKeyAuth** — вставить API-ключ
3. **OAuth2Password** — форма логин/пароль → автоматически получает токен

### Проблема: несогласованные `@Security` в handler'ах

| Handler | Текущие @Security | Проблема |
|---------|-------------------|----------|
| `auth_handler.go` (me, logout) | `BearerAuth` + `OAuth2Password` | ❌ Нет `ApiKeyAuth` |
| `api_key_handler.go` | Только `BearerAuth` | ❌ Нет `OAuth2Password`, нет `ApiKeyAuth` |
| `prompt_handler.go` | `BearerAuth` + `OAuth2Password` | ❌ Нет `ApiKeyAuth` |
| `llm_handler.go` | `BearerAuth` + `OAuth2Password` | ❌ Нет `ApiKeyAuth` |
| `workflow_handler.go` | `BearerAuth` + `OAuth2Password` | ❌ Нет `ApiKeyAuth` |
| `webhook_handler.go` (protected) | Только `BearerAuth` | ❌ Нет `OAuth2Password`, нет `ApiKeyAuth` |
| `project_handler.go` | `BearerAuth` + `ApiKeyAuth` | ❌ Нет `OAuth2Password` |
| `team_handler.go` | `BearerAuth` + `ApiKeyAuth` | ❌ Нет `OAuth2Password` |

**Результат:** Swagger UI работает по-разному для разных эндпоинтов. Например:
- Залогинился через OAuth2 (логин/пароль) → работают prompts, но **не работают** projects (нет `OAuth2Password` в аннотации)
- Вставил API-ключ → работают projects, но **не работают** prompts (нет `ApiKeyAuth`)

---

## Решение: единый стандарт

### Правило для ВСЕХ защищённых эндпоинтов

Каждый endpoint, требующий авторизации, **ОБЯЗАН** иметь все три `@Security`:

```go
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
```

**Исключения:**
- Публичные эндпоинты (`/auth/register`, `/auth/login`, `/auth/refresh`, `/hooks/:name`) — **без** `@Security`
- API-key management (`/auth/api-keys/*`) — только `BearerAuth` + `OAuth2Password` (нельзя управлять ключами через ключ, только через JWT)

### Как работает множественный @Security в swag

В `swaggo/swag` несколько `@Security` на одной строке — это **AND** (оба нужны).
Каждый `@Security` на **отдельной строке** — это **OR** (любой из них).

```go
// ✅ OR (любой из трёх способов — правильно!)
// @Security BearerAuth
// @Security ApiKeyAuth
// @Security OAuth2Password
```

```go
// ❌ AND (нужны все три одновременно — неправильно!)
// @Security BearerAuth,ApiKeyAuth,OAuth2Password
```

## Проверка OAuth2Password в Swagger UI

После `make swagger`, открыть `http://localhost:8080/swagger/index.html`:

1. Нажать **"Authorize"** → появятся 3 секции:
   - **BearerAuth** — поле для ручного ввода `Bearer <token>`
   - **ApiKeyAuth** — поле для ввода API-ключа
   - **OAuth2Password** — поля `username` + `password` + кнопка "Authorize"

2. Вввести в OAuth2Password:
   - username: `admin@example.com`
   - password: `admin123`
   - Нажать "Authorize"

3. Swagger UI автоматически вызовет `POST /api/v1/auth/login` и подставит полученный `access_token` в header `Authorization: Bearer <token>` для всех последующих запросов.

4. Проверить, что **все** защищённые эндпоинты (включая `/projects`, `/projects/:id/team`) работают с полученным токеном.
