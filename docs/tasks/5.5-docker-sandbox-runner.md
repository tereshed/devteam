# 5.5 Реализация: `DockerSandboxRunner` (Docker SDK для Go)

**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (ориентир):** `backend/internal/sandbox/docker_runner.go` (основной тип и методы); при необходимости вспомогательные файлы **того же пакета** `sandbox` (например `docker_runner_state.go` для mutex/карты инстансов) — **без** импорта `gin`, `handler`, GORM.  
**Статус:** ✅ Выполнена

## Описание

Реализовать структуру **`DockerSandboxRunner`**, удовлетворяющую интерфейсу **`SandboxRunner`** (задача **5.3**), используя официальный **Docker Engine API для Go** (`github.com/docker/docker/client` и связанные типы из `github.com/docker/docker/api/types/...`). Раннер — единственное место в пакете `sandbox`, где допускаются прямые вызовы Docker SDK.

**Цель этой задачи (границы 5.5):** корректный жизненный цикл инстанса на уровне API движка: **`RunTask`** (create → конфигурация → copy/mount больших текстов → start), **`Wait`** (блокирующее ожидание без busy-poll **`GetStatus`**), **`GetStatus`**, заглушка или минимально корректная реализация **`StreamLogs`** / **`Stop`** / **`Cleanup`**, если их полная семантика вынесена в **5.6–5.9**. Допустимый вариант для merge **5.5**: все методы компилируются и проходят базовые сценарии, а детальная доработка стримов, копирования артефактов, таймеров и cgroups отмечена явными ссылками на последующие задачи — **но** нельзя оставлять контракт **5.3** нарушенным (например «`Wait` не работает» или «`Cleanup` оставляет контейнеры после ошибки create»).

**Рекомендуемая политика для одного PR:** закрыть **5.5** полностью по **MVP-скелету**: `RunTask`/`Wait`/`GetStatus`/`Stop`/`Cleanup` работают согласованно с Docker; **`StreamLogs`** — либо полноценный минимальный стрим (stdout+stderr, один attach), либо возврат **`ErrStreamAlreadyActive`** по контракту и реальное чтение логов — в **5.6**, если в **5.5** оставляется явный `TODO` и тесты не обещают стрим (согласовать с **5.13**).

## Архитектура и DI

1. **`DockerSandboxRunner`** принимает зависимости через конструктор, например `NewDockerSandboxRunner(client *client.Client, …)` или обёртку над интерфейсом с узким набором методов для **моков в 5.13**. Глобальный `docker.Client` **запрещён**.
2. Создание клиента из env (`DOCKER_HOST`, TLS) — в **`cmd/api/main.go`** или фабрике рядом с wire/DI; пакет `sandbox` получает уже сконфигурированный клиент (как `GitProvider` в sprint 4).
3. **Потокобезопасность и гонка `RunTask` / `Cleanup`:** все методы **5.3** обязаны быть безопасны при конкурентных вызовах; внутреннее состояние (активный стрим, ожидания `Wait`, таймер бизнес-таймаута) — под мьютексами или через `sync.Map` + отдельные примитивы на инстанс.
   - **Критично:** запись инстанса во внутреннюю карту/стейт раннера (по **`TaskID`** и/или зарезервированному **`sandboxID`**, путям хоста и т.д.) в статусе **`creating`** должна выполняться **до** первого вызова **`ContainerCreate`** (или сразу после него, но **атомарно** под тем же мьютексом до `start` — главное: к моменту, когда другая горутина может вызвать **`Cleanup`/`Stop`** по этому **`TaskID`**, инстанс **уже** находится в стейте). Иначе сценарий «пользователь отменил, пока демон тормозит на create» приводит к **`Cleanup`**, который не находит инстанс, и контейнер позже стартует **сиротой**.
   - **`Cleanup`** для ещё инициализирующегося инстанса должен либо дождаться завершения критической секции **`RunTask`**, либо кооперировать отмену через общий **`context`**/флаги «abort create» — конкретика в реализации, но итог: **нет** бессрочных контейнеров из-за гонки отмены с регистрацией.

## Объём работ по методам (связь с 5.3)

### `RunTask(ctx, opts)`

1. **Первая строка тела:** `opts = opts.Clone()` (уже зафиксировано в комментарии к **5.3**).
2. Вызов **`opts.Validate(ctx)`** и отдельных валидаторов из **`validate.go`** (`ValidateRepoURL`, `ValidateBranchName`, `ValidateEnvKeys` и т.д.). Ошибки — **`ErrInvalidOptions`** / узкие sentinel’ы с **`fmt.Errorf`** и `%w`, как в **5.3**. Раннер **не** имеет права «добавить в контейнер» пары из **`opts.EnvVars`**, минуя **`ValidateEnvKeys`** (см. раздел «Переменные окружения: command/flag injection»).
3. **Регистрация инстанса в стейте** (ориентир **`docker_runner_state.go`**): после успешной валидации и выделения записи (в т.ч. пути **`MkdirTemp`** на хосте, если нужен) — **зарегистрировать** инстанс под мьютексом в статусе **`creating`** **до** **`ContainerCreate`** — см. раздел «Потокобезопасность» выше.
4. **Конфигурация контейнера и образ:**
   - образ **`opts.Image`** — **только после жёсткой проверки по allowlist** (см. раздел «Образ: allowlist» ниже). Произвольный тег из внешнего слоя без проверки **запрещён** (риск произвольного кода на хосте: майнинг, escape, DoS).
   - **Политика `ImagePull` (зафиксировать одну в коде и в README/5.10):**
     - **Вариант А — раннер тянет образ:** если локально образа нет, перед **`ContainerCreate`** вызвать **`ImagePull(ctx, …)`**. Ответ часто содержит **`io.ReadCloser`** (стрим JSON-прогресса). Его **обязательно** **дочитать до EOF** (например **`io.Copy(io.Discard, pullRC)`**) и затем **`Close()`** — иначе pull может **зависнуть** или завершиться некорректно; при отмене **`ctx`** — всё равно закрыть ридер в **`defer`**. Таймаут на pull — через **`ctx`** или вложенный **`WithTimeout`**.
     - **Вариант Б — только предзагрузка:** раннер **не** вызывает **`ImagePull`**; отсутствие образа на хосте → **фатальная ошибка** **`RunTask`** с понятным текстом («образ не найден, предзагрузите через CI/Ansible/`make sandbox-build`» — **5.12**). Тогда операционная команда явно гарантирует наличие allowlist-образов на ноде.
   - env для entrypoint: значения из **`opts.EnvVars`** (только после **`ValidateEnvKeys`**) плюс обязательные **`EnvRepoURL`**, **`EnvBranchName`**, **`EnvBackend`** и др. по контракту **5.2** (не дублировать большие тексты **`Instruction`/`Context`** в ENV — только файлы **`PromptFilePath`** / **`ContextFilePath`** через **`CopyToContainer`** или bind-mount; см. раздел «Хостовые пути и bind-mount»).
   - **`CopyToContainer` и tar-поток (утечки FD / дедлоки):** API принимает **tar-поток** (**`io.Reader`**). Типичная реализация: **`io.Pipe`** + горутина, пишущая tar в **`PipeWriter`**, или временный tar-файл на диске. **Обязательно:** на **каждой** ветке выхода (успех, ошибка **`ContainerCreate`/`Start`**, отмена **`ctx`**) гарантировать **`Close()`** для **`PipeWriter`**, **`PipeReader`**, временных **`*os.File`** и любых **`ReadCloser`/`WriteCloser`** через **`defer`** и/или явные **`Close`** в обработчиках ошибок — иначе **дедлок** архиватора на записи и **утечка FD**. То же для промежуточных буферов при стриминге tar «на лету».
   - политика сети: **`DisableNetwork`** → режим **`none`** (без исходящего интернета); иначе — **изолированная** сеть (детали compose — **5.11**; в коде раннера — **отдельная bridge-сеть на задачу** предпочтительнее «общей»).
   - **ICC (межконтейнерный трафик):** на **дефолтном** **`bridge`** с **`ICC=true`** контейнеры разных задач могут достигать друг друга по **внутренним IP** — скомпрометированный агент атакует соседа. **Требования для MVP 5.5:** (1) **не** подключать sandbox к **дефолтной** сети **`bridge`**; (2) если используется **общая** пользовательская сеть из конфига (**5.10**), она должна быть создана с **`com.docker.network.bridge.enable_icc=false`** (или эквивалентная изоляция по документации движка); (3) **проще и безопаснее** — **одна изолированная сеть на `TaskID`/инстанс** без ICC к чужим контейнерам. Минимум при сомнениях: при **`NetworkConnect`** логировать **WARN**, если нельзя доказать, что **ICC отключён** / контейнер единственный в сегменте.
   - **лимиты ресурсов (HostConfig), минимум для MVP:** даже до полной задачи **5.9** **`RunTask` обязан** задавать в **`HostConfig`** не ниже **безопасного пола** (защита хоста от fork-бомбы / OOM из кода в контейнере): как минимум **лимит памяти** (ориентир **1 GiB**) и **`PidsLimit`** (ориентир **100**). Поля **`opts.ResourceLimit`** (**5.4**) учитывать так: итог **не ниже** этого пола; если в **`opts`** или конфиге (**5.10**) заданы **большие** значения — применять с **верхним потолком** из конфига; если в **`opts`** нули / не задано — брать минимум из пола. **Никогда** не запускать контейнер с безлимитной памятью и без **`PidsLimit`** по умолчанию. CPU / **`NanoCPUs`** и тонкая настройка — **5.9** / **5.10**.
   - **PID 1, зомби и SIGTERM (`Init`):** в **`HostConfig`** **обязательно** **`Init: pointer(true)`** (Moby: внедрение лёгкого **init**, фактически **`tini`**, как PID **1**). Если **`entrypoint`** (bash/скрипт) остаётся PID 1, дочерние процессы агента (git, компиляторы, тесты) порождают **зомби**, а **SIGTERM** от **`docker stop`** часто **обрабатывается некорректно** — **`Stop`** (**5.3**/**5.8**) и чистое зависание контейнера ломаются. **`Init: true`** — **критично** для корректного завершения и отсутствия «висящих» процессов в namespace.
   - **Логи контейнера и диск хоста (`LogConfig`):** по умолчанию движок неограниченно пишет **`json.log`** под **`/var/lib/docker/containers/...`**; бесконечный **`stdout`** (баг или злоумышленник) — **исчерпание диска** на ноде (**DoS**). В **`HostConfig.LogConfig`** задать драйвер (например **`json-file`**) с ротацией: ориентир **`max-size=10m`**, **`max-file=3`** (**≤ ~30 MiB** на контейнер на диске); точные значения — из **5.10** с безопасными дефолтами в коде.
5. **Имя контейнера:** детерминированно от **`opts.TaskID`** **только после** **`ValidateTaskID`** (уже вызывается из **`opts.Validate`** — см. `options_validate.go`). **Запрещено** склеивать префикс **`TaskContainerNamePrefix`** с сырой строкой до валидации. Дополнительно имя должно удовлетворять ограничениям Docker на имя контейнера (длина, допустимые символы — при несовместимости нормализовать **детерминированно**, например хэш от проверенного `TaskID`, без обхода валидации). При коллизии — политика из комментария к **5.3** (вернуть существующий инстанс или **`ErrSandboxRunConflict`**). *Продуктовая рекомендация:* в API передавать в раннер **`TaskID`** в виде канонического UUID задачи — меньше поверхность для неожиданных имён.
6. **`ContainerCreate` → ID и предупреждения демона:** в **`SandboxInstance.ID`** возвращать только **доверенный** ID от Engine — **64 hex** (как ожидает **`ValidateSandboxID`**). Поле **`Warnings []string`** в **`container.CreateResponse` обязано** обрабатываться: при **`len(Warnings) > 0`** логировать **каждую** строку уровнем **WARN** (не только DEBUG): демон мог **частично проигнорировать** запрошенные лимиты (**`PidsLimit`**, swap, профили безопасности и т.д.) без фатальной ошибки — без лога админы будут думать, что «всё применилось».
7. **`ContainerStart` и ложный успех:** **`ContainerStart(ctx, id)`** возвращает **`nil`**, когда демон **принял команду запуска**, а **не** когда **`entrypoint`** успешно выполняется. Контейнер может **сразу** завершиться (**exec format error**, нет **`+x`** на скрипте, пустой образ и т.д.). **После** успешного **`ContainerStart`** выполнить **короткий sanity-check** (например **`ContainerInspect`** или один цикл ожидания «не exited» с малым таймаутом, согласованным с **`ctx`**): если контейнер уже **в статусе exited** с **ненулевым** кодом / **`OOMKilled`** / явной ошибкой старта — вернуть из **`RunTask`** ошибку **инициализации** (обёртка + sentinel), **не** возвращать «успешный» инстанс «как работающий» и **не** запускать бизнес-таймер (**п. 9**). Граничный случай «ещё в `created`/`running` на миллисекунду» — допустимо зафиксировать политику «достаточно inspect один раз» vs «короткое ожидание» в коде.
8. **Сироты после частичного create / copy / start:** если create прошёл, а copy/start упал или **`ctx`** отменён — **best-effort** удалить контейнер до возврата ошибки (см. комментарий к **5.3**). **Важно:** если **`ctx`** вызова **`RunTask`** уже отменён (**`ctx.Err() != nil`**), вызов **`ContainerRemove(ctx, …)`** с тем же **`ctx`** немедленно вернёт **`context.Canceled`** и **не** удалит контейнер. Для удаления сироты использовать **отдельный контекст**, не отменённый вместе с родительским: например **`context.WithoutCancel(ctx)`** + короткий **`WithTimeout`**, либо **`context.Background()`** + **`WithTimeout`** (например 30–60 с) — политика зафиксировать в коде. То же относится к **`ImagePull`** / другим RPC уборки при отменённом пользовательском **`ctx`**.
9. **Старт бизнес-таймера:** фиксировать момент для **`opts.Timeout`** / **`EffectiveTimeout()`** только **после** пункта **7** (sanity-check пройден, контейнер не «мёртвый сразу после старта») — согласование с **5.3** «Два уровня таймаутов»; детали kill — **5.8**.

### `Wait(ctx, sandboxID)`

- В начале **`ValidateSandboxID(sandboxID)`**; при ошибке — **`ErrInvalidSandboxID`** без вызова API.
- Ожидание завершения контейнера через **`ContainerWait`** (или эквивалент), **без** цикла **`ContainerInspect`**. Вызов **`ContainerWait`** и любые вспомогательные RPC — **только с пробросом `ctx`** (см. раздел «Docker SDK и `context`»).
- **`ContainerWait` и два канала (утечка горутин в SDK):** сигнатура **`client.ContainerWait(ctx, …)`** возвращает **`<-chan` `container.WaitResponse`** и **`<-chan error`**. Внутри SDK в горутину пишут **оба** канала. Если читать только **`WaitResponse`**, а по **`ctx.Done()`** выйти раньше **не** прочитав из **`err`**, горутина SDK может **заблокироваться навсегда** при попытке отправить во второй канал (**известный класс проблем**). **Обязательный паттерн:** ожидание через **`select`**, учитывающий **оба** канала **и** **`ctx.Done()`**; при отмене **`ctx`** — **дочитать** (drain) канал **`error`** до закрытия **или** иным образом гарантировать, что отправитель не зависнет (конкретика по версии SDK — зафиксировать в коде/комментарии; минимум — **`select`** с приоритетом чтения **`err`** при выходе по контексту).
- Учёт **`SandboxOptions.Timeout`**, привязанного к инстансу при **`RunTask`** (внутренняя структура состояния): по истечении — принудительная остановка, в **`SandboxStatus.Status`** — **`SandboxStatusTimedOut`** (см. **5.3**, **5.4**).
- **Несколько горутин `Wait` на один `sandboxID`:** не опираться на самодельный refcount/broadcast на мьютексах как на единственный вариант — высокий риск дедлоков и гонок. **Рекомендуемый паттерн:** в состоянии инстанса хранить **`done chan struct{}`**, который **закрывается ровно один раз** (`close(done)`) в единственной точке, когда финальный исход известен (контейнер exited/stopped, таймаут, ошибка ожидания после фиксации статуса). Все вызовы **`Wait`** делают **`select { case <-ctx.Done(): …; case <-done: … }`** (при необходимости читают уже записанный финальный **`SandboxStatus`** под мьютексом после `<-done`). Так достигается потокобезопасное «много подписчиков — одно событие завершения» без refcount-магии.
- Отмена только **`ctx`** у **`Wait`**: возврат **`context.Canceled`/`DeadlineExceeded`**; контейнер по-прежнему ограничен бизнес-таймаутом (**5.3**).

### `GetStatus(ctx, sandboxID)`

- Валидация ID → **`ContainerInspect`** + маппинг состояния движка на **`SandboxStatus`** / **`SandboxStatusType`**.
- **Не** использовать как основу ожидания завершения в оркестраторе.

### `StreamLogs(ctx, sandboxID)`

- Минимум для **5.5:** соблюдение **варианта А** (**`ErrStreamAlreadyActive`**); закрытие канала при отмене **`ctx`** и при **`Cleanup`**.
- **Утечка дескрипторов:** ответ Docker SDK на чтение логов (**`ContainerLogs`** и аналоги) содержит **`io.ReadCloser`**. При **любом** завершении стрима (EOF, ошибка чтения, отмена **`ctx`**, **`Cleanup`**) этот **`ReadCloser` обязан быть закрыт (`Close()`)** — иначе утечка FD и зависшие горутины внутри клиента. Использовать **`defer`**/`finally`-паттерн в горутине-ридере с гарантией вызова **`Close`** на всех ветках.
- Полная реализация attach, чанкирования, буфера **`StreamLogsDefaultBuffer`**, **`LogEntryMaxLineBytes`** — **5.6** (если выносится — в **5.5** оставить короткий стаб и ссылку в комментарии).

### `Stop(ctx, sandboxID)`

- Graceful stop с таймаутом, затем kill (**5.3** комментарий; детализация — **5.8**). Типично: **`ContainerStop`** с **`Timeout`** (например 10 с) для SIGTERM, затем при необходимости принудительное завершение.
- **`ctx`** ограничивает только RPC к движку — **но:** если **`ContainerStop`** прерывается из‑за отмены **`ctx`** **раньше**, чем истёк внутренний graceful-таймаут движка, контейнер может **остаться работать**, а фаза «kill» внутри одного вызова **`Stop`** не выполнится.
- **Требование:** при выходе из **`Stop`** по **`ctx.Done()`** / **`context.Canceled`** метод **возвращает ошибку** (по правилам Go), но **инвариант безопасности:** контейнер не должен остаться бесконтрольно живым из‑за этой гонки. Варианты (зафиксировать один в реализации): (а) **`Stop`** после отмены **`ctx`** запускает **best-effort** догоняющий **`ContainerRemove(Force: true)`** с отдельным контекстом (**`WithoutCancel`/`Background` + короткий таймаут**), либо (б) контракт перекладывается на вызывающего: оркестратор **обязан** вызвать **`Cleanup`** (который удалит/остановит контейнер с force по политике) — и в спецификации **`Cleanup`** явно заложить принудительное удаление «живого» контейнера. **5.8** может унифицировать таймауты SIGTERM/SIGKILL; **5.5** не должен оставлять дыру «отменили HTTP — контейнер крутится вечно».

### `Cleanup(ctx, sandboxID)`

- Удаление контейнера, освобождение сетей/volumes, зарегистрированных для инстанса.
- **Идемпотентность и контейнер:** повторный **`Cleanup`** (двойной **`defer`**, гонка с **`RunTask`**, таймаут оркестратора) вызовет **`ContainerRemove`** для уже удалённого ID. Ошибка **`No such container`** от Engine **не** является сбоем метода: обрабатывать через **`errdefs.IsNotFound(err)`** (или эквивалент в версии SDK) — **игнорировать**, не пачкать логи «ложной» ошибкой, не возвращать наверх как фатальную (метод остаётся **полностью** идемпотентным). Аналогично для сетей/volumes, если API возвращает not found.
- **Хостовые временные каталоги:** если для инстанса был создан каталог через **`os.MkdirTemp`** (файлы для **`CopyToContainer`** / bind-mount), путь к нему **обязан** храниться в стейте инстанса (ориентир **`docker_runner_state.go`**). **`Cleanup` обязан** выполнить **`os.RemoveAll(hostTempDir)`** (или эквивалент) **после** остановки/удаления контейнера или при отсутствии контейнера — иначе **утечка диска** на хосте (`/tmp`, inode exhaustion). Повторный **`Cleanup`** — идемпотентно (нет директории — no-op).
- Отмена активных стримов, разблокировка **`Wait`** (**5.3**).
- **Важно:** для **`defer Cleanup`** не использовать уже отменённый HTTP-**`ctx`** — см. комментарий в **5.3** (`context.WithoutCancel` / отдельный timeout на уборку).

## Безопасность и инъекции

### Образ: allowlist (**критично**)

Поле **`opts.Image`** **не** может трактоваться как «любая строка для Docker». Раннер **обязан** отклонять образы вне **жёсткого allowlist** (набор строк полного имени образа, например `devteam/sandbox-claude:latest`, `devteam/sandbox-aider:latest`).

- Список разрешённых образов — из **конфигурации приложения** (**5.10**: например `SANDBOX_ALLOWED_IMAGES` или структура в `SandboxConfig`), с **безопасными дефолтами** в коде (только известные sandbox-образы репозитория).
- Проверка выполняется в **`RunTask`** после **`opts.Validate`**, либо расширяет **`SandboxOptions.Validate`** отдельной функцией вида **`ValidateAllowedImage(image string, allowed []string) error`** с возвратом **`ErrInvalidOptions`** / отдельного sentinel’а по выбору пакета.
- Сравнение **строгое** (полное совпадение ref’а или нормализованное сравнение по правилам, зафиксированным в коде — без «подстрока совпала»).

Цель: исключить сценарий, когда LLM/злоумышленник подставляет `ubuntu:latest`, публичный вредоносный образ или образ с escape с хоста.

### Хостовые пути и bind-mount (**path traversal**)

Если **`Instruction`/`Context`** попадают на хост во временные файлы для **`CopyToContainer`** или bind-mount:

- Каталог на хосте создавать **только** через **`os.MkdirTemp("", "devteam-sandbox-*")`** (или эквивалент с тем же уровнем безопасности), без ручной конкатенации вида **`/tmp/sandbox/` + opts.TaskID**.
- **Запрещено** строить путь к хостовому каталогу конкатенацией пользовательских полей без предшествующей валидации; **`TaskID`** до использования в любых путях — **`ValidateTaskID`** (уже исключает `../`, пробелы и произвольные символы вне допустимого набора).
- **Жизненный цикл:** абсолютный путь к созданному каталогу сохраняется в стейте инстанса; **удаление только в `Cleanup`** — см. раздел **`Cleanup`** выше.

### Переменные окружения: command / flag injection (**критично**)

Ключи в **`opts.EnvVars`**, если передать их в контейнер без фильтрации, могут переопределить поведение утилит (**`PATH`**, **`LD_PRELOAD`**, **`LD_LIBRARY_PATH`**, **`GIT_SSH_COMMAND`**, **`GIT_EXEC_PATH`** и т.д.) и дать произвольное исполнение в обход **`entrypoint`**.

- **Источник правды по ключам:** **`ValidateEnvKeys`** и белый список в **`validate.go`** (задача **5.4**). Раннер **не** сливает в конфиг контейнера произвольную мапу: только результат валидации + явно заданные системные пары для entrypoint (**`EnvRepoURL`**, …).
- При расширении белого списка под новые бизнес-переменные — обновлять **5.4** и ревью на предмет «не системная ли это переменная».

### Имя контейнера

См. пункт 5 в **`RunTask`**: только **`ValidateTaskID(opts.TaskID)`** → затем формирование имени с **`TaskContainerNamePrefix`**.

### Лимиты ресурсов и DoS на хосте (**критично для MVP**)

Даже без задачи **5.9** один контейнер с «безлимитной» памятью и PID может положить хост (**OOM**, fork-бомба). Отдельный вектор — **неограниченные логи** на диск движка и **зомби/SIGTERM** без **`Init`**. Минимальные требования — в подпунктах **`HostConfig`** внутри **`RunTask`** (память, **`PidsLimit`**, **`Init`**, **`LogConfig`**); **полная** настройка и **`opts.ResourceLimit`** — **5.9** / **5.10**, но **ни один** успешный **`ContainerStart`** в **5.5** без пола по памяти, **`PidsLimit`**, **`Init: true`** и потолка на **`json-file`** не допускается.

### Сеть: ICC между задачами (**безопасность**)

Сводка политики — в подпункте **«ICC»** в **`RunTask`** выше. Цель: контейнер задачи **A** не должен открывать TCP/UDP к контейнеру задачи **B** только потому, что они на одном хосте.

## Docker SDK и `context` (**критично**)

**Правило:** **ни одного** вызова к **`client.Client`** / низкоуровневым API Docker SDK **без** передачи **`context.Context`** первым аргументом там, где API это поддерживает (в т.ч. **`ContainerCreate`**, **`ContainerStart`**, **`ContainerWait`**, **`ContainerLogs`**, **`CopyToContainer`**, **`ContainerRemove`**, **`ImagePull`** и т.д.).

Если демон Docker зависнет, вызов **без** **`ctx`** может заблокировать горутину навсегда. Для операций без встроенного контекста в старых версиях SDK — оборачивать в горутину + **`select`** с **`ctx.Done()`** и явной политикой таймаута (зафиксировать в коде/комментарии).

**Исключение для уборки при отменённом `ctx`:** для **`ContainerRemove`** / финальной уборки в **`RunTask`** после ошибки, когда пользовательский **`ctx` уже отменён**, использовать отдельный контекст (см. пункт **8** в **`RunTask`**) — иначе best-effort не сработает.

## Ошибки Docker SDK и внешний контракт

Сырой **`error`**, возвращённый Docker Engine (длинные сообщения, внутренние пути демона, стектрейсы плагинов), **нельзя** без обработки отдавать наверх как «сообщение для пользователя» / в JSON API.

**Требование:** оборачивать ошибки SDK в **узкие sentinel’ы пакета `sandbox`** и/или **`fmt.Errorf("…: %w", Err…)`** с **кратким, стабильным** текстом для логов (например `failed to start sandbox: %w`, `docker create: %w` с внутренним sentinel **`ErrSandboxStart`** / **`ErrDocker`** — конкретные имена выбрать в реализации, задокументировать в `errors.go`). Полный текст демона допускается **только в логах сервера** на уровне debug/trace при явном флаге, не в ответе клиенту.

Секреты из env в тексте ошибок Docker — по-прежнему маскировать; см. **5.4**.

## Безопасность (прочее)

- Не логировать **`SandboxOptions`** через **`%+v`** — использовать **`LogSafe`/`String`/`MarshalJSON`** (**5.4**).
- **`sandboxID`** — только после **`ValidateSandboxID`** перед любым RPC.

## Критерии приёмки

- [ ] Файл **`backend/internal/sandbox/docker_runner.go`** добавлен; тип **`DockerSandboxRunner`** реализует **`SandboxRunner`**.
- [ ] Зависимость Docker SDK внедряется через конструктор; **нет** глобального клиента и **`init()`**-конфигурации.
- [ ] **`opts.Image`:** только образы из **конфигурируемого allowlist**; произвольный образ отклоняется до **`ContainerCreate`**.
- [ ] **Хостовые временные каталоги** (если есть): только **`os.MkdirTemp`**, путь в стейте инстанса; **`Cleanup`** — **`os.RemoveAll`** для этого пути (идемпотентно).
- [ ] **Гонка `RunTask`/`Cleanup`:** инстанс в стейте (**`creating`**) **до** **`ContainerCreate`**; отмена не оставляет сирот.
- [ ] **Best-effort удаление контейнера** при отменённом **`ctx`:** **`ContainerRemove`** (и аналоги) с **`context.WithoutCancel`/`Background` + короткий таймаут**, не с отменённым родительским **`ctx`**.
- [ ] **`opts.EnvVars`:** только после **`ValidateEnvKeys`**; без слепой передачи системных ключей.
- [ ] **`TaskID`:** перед именем контейнера — **`ValidateTaskID`** (через **`opts.Validate`**); имя контейнера совместимо с правилами Docker.
- [ ] **Все вызовы Docker SDK** с поддержкой **`ctx`** — с пробросом **`ctx`**; нет вечной блокировки при зависшем демоне из-за вызова без отмены.
- [ ] **`RunTask`:** `Clone` → валидация → create → лог **`Warnings`** из **`CreateResponse`** (WARN) → copy/start → **`SandboxInstance.ID`** = полный Docker ID (64 hex); **`Instruction`/`Context`** через файлы / tar, не через огромные ENV.
- [ ] **`CopyToContainer` / tar:** **`defer`/`Close`** на **`io.Pipe`**, файлах и стримах без дедлоков и утечек FD.
- [ ] **После `ContainerStart`:** sanity-check (inspect / короткое ожидание); мгновенный crash entrypoint → ошибка из **`RunTask`**, бизнес-таймер **не** стартует до успешной проверки (**п. 9**).
- [ ] **Политика `ImagePull`:** задокументирован вариант А (pull + **`io.Copy(io.Discard, …)`** + **`Close`**) или Б (без pull, фатальная ошибка при отсутствии образа).
- [ ] **`Wait`:** **`ValidateSandboxID`**; **`ContainerWait`** **без** busy-poll; **два канала** от **`ContainerWait`:** **`select`** по **`WaitResponse`**, **`error`** и **`ctx.Done()`**; при отмене **`ctx`** не оставлять горутину SDK зависшей на отправке в **`err`**.
- [ ] **`Wait`:** **несколько `Wait`:** паттерн **`done chan struct{}`** + **`close(done)`** один раз (или эквивалент с той же семантикой); согласованность с бизнес-таймаутом (**5.8** может уточнить kill, но **`SandboxStatusTimedOut`** уже различим).
- [ ] **`RunTask` / `HostConfig`:** минимальный пол по **памяти** (~1 GiB) и **`PidsLimit`** (~100); слияние с **`opts.ResourceLimit`**; без запуска «без лимитов».
- [ ] **`HostConfig.Init`:** **`true`** (указатель), чтобы PID 1 корректно обрабатывал сигналы и зомби.
- [ ] **`HostConfig.LogConfig`:** ротация (**`max-size`/`max-file`**) — защита от лог-бомбы на диск движка.
- [ ] **Сеть:** нет дефолтного **`bridge`** с ICC между задачами; либо **`none`**, либо сеть с **`enable_icc=false`**, либо per-task сеть; при риске — **WARN** в лог.
- [ ] **`Stop`:** при прерывании по **`ctx`** документированная гарантия финальной остановки (**best-effort в `Stop`** или обязательный **`Cleanup`** с force-remove).
- [ ] **`Cleanup`:** **`errdefs.IsNotFound`** от **`ContainerRemove`** (и аналоги) — успех идемпотентного вызова; без шума в логах.
- [ ] **`GetStatus`:** снимок без нарушения контракта **5.3**.
- [ ] **`StreamLogs`:** при использовании **`ContainerLogs`** — гарантированный **`Close()`** на **`io.ReadCloser`** на всех путях завершения; не противоречит **5.3** / **5.6**.
- [ ] **Ошибки Docker:** не пробрасываются «как есть» на HTTP-границу; обёртки + sentinel’ы в **`errors.go`**; длинные внутренние тексты — только осознанный server-side debug.
- [ ] **`StreamLogs`/`Stop`/`Cleanup`:** не противоречат **5.3**; либо полный MVP, либо явное разделение с **5.6–5.8** в комментариях + открытые задачи в трекере.
- [ ] Пакет **`sandbox`** не тянет **`gin`** / **`internal/handler`** / GORM.
- [ ] `go build ./backend/internal/sandbox/...` проходит.
- [ ] В README у строки **5.5** указана ссылка «детали» на этот файл.
