**Sprint:** 1 — Новые модели данных и миграции
**Файл миграции:** `backend/db/migrations/014_create_projects.sql`
**Go-модели:** `backend/internal/models/project.go`, `backend/internal/models/git_credential.go`
**Статус:** ✅ Выполнена

---

## Описание

Создать две связанные таблицы в одной миграции:

1. **`git_credentials`** — зашифрованные учётные данные для доступа к Git-репозиториям (токены, SSH-ключи, OAuth)
2. **`projects`** — центральная сущность системы, связывающая репозиторий, команду агентов и весь контекст

Порядок важен: `git_credentials` создаётся первой, потому что `projects.git_credentials_id` ссылается на неё через FK.

---

## Связь между таблицами

```
users (001)
  │
  ├──< git_credentials (FK: user_id)
  │         │
  └──< projects (FK: user_id, FK: git_credentials_id)
              │
              └──< teams (016, FK: project_id)
                     │
                     └──< agents (017, FK: team_id)
```

Один пользователь может иметь:
- Несколько git_credentials (для разных провайдеров/аккаунтов)
- Несколько projects
- Один project может использовать одну git_credential (или NULL для публичных/локальных)

## Таблица `git_credentials`

### Поля

| Поле | Тип | Nullable | Default | Описание |
|------|-----|----------|---------|----------|
| `id` | UUID | NO | `gen_random_uuid()` | PK |
| `user_id` | UUID | NO | — | FK → users, владелец |
| `provider` | VARCHAR(50) | NO | — | `github` / `gitlab` / `bitbucket` |
| `auth_type` | VARCHAR(50) | NO | `'token'` | `token` / `ssh_key` / `oauth` |
| `encrypted_value` | BYTEA | NO | — | Зашифрованные данные (AES-256-GCM) |
| `label` | VARCHAR(255) | NO | — | Метка для UI ("Мой GitHub рабочий") |
| `created_at` | TIMESTAMPTZ | NO | `NOW()` | |
| `updated_at` | TIMESTAMPTZ | NO | `NOW()` | |

### Детали полей

**`provider`** — к какой системе относятся credentials:
- `github` — Personal Access Token / OAuth App / GitHub App
- `gitlab` — Project Access Token / Personal Token
- `bitbucket` — App Password / OAuth Consumer

**`auth_type`** — тип аутентификации:
- `token` — PAT / API token (самый частый)
- `ssh_key` — SSH private key
- `oauth` — OAuth token (access + refresh)

**`encrypted_value`** — зашифрованные данные:
- Шифруется на уровне приложения (AES-256-GCM с ключом из ENV)
- Для `token`: зашифрованная строка токена
- Для `ssh_key`: зашифрованный private key
- Для `oauth`: зашифрованный JSON `{"access_token": "...", "refresh_token": "..."}`
- **НИКОГДА** не хранится в открытом виде

**`label`** — человекочитаемая метка:
- Обязательна, помогает пользователю отличить credentials между собой
- Примеры: "GitHub Personal", "GitLab CI/CD", "Bitbucket Work"

### Индексы

| Индекс | Поля | Назначение |
|--------|------|------------|
| `idx_git_credentials_user_id` | `user_id` | Список credentials пользователя |
| `idx_git_credentials_provider` | `user_id, provider` | Фильтрация по провайдеру |

## Таблица `projects`

### Поля

| Поле | Тип | Nullable | Default | Описание |
|------|-----|----------|---------|----------|
| `id` | UUID | NO | `gen_random_uuid()` | PK |
| `name` | VARCHAR(255) | NO | — | Название проекта |
| `description` | TEXT | YES | — | Описание (для контекста агентов) |
| `git_provider` | VARCHAR(50) | NO | `'local'` | `github` / `gitlab` / `bitbucket` / `local` |
| `git_url` | VARCHAR(1024) | YES | — | URL репозитория |
| `git_default_branch` | VARCHAR(255) | NO | `'main'` | Основная ветка |
| `git_credentials_id` | UUID | YES | — | FK → git_credentials |
| `vector_collection` | VARCHAR(255) | YES | — | Коллекция Weaviate |
| `tech_stack` | JSONB | NO | `'{}'` | Языки, фреймворки |
| `status` | VARCHAR(50) | NO | `'active'` | `active` / `paused` / `archived` |
| `settings` | JSONB | NO | `'{}'` | Настройки проекта |
| `user_id` | UUID | NO | — | FK → users, владелец |
| `created_at` | TIMESTAMPTZ | NO | `NOW()` | |
| `updated_at` | TIMESTAMPTZ | NO | `NOW()` | |

### Детали полей

**`git_provider`** — тип провайдера. В отличие от `git_credentials.provider`, здесь есть `local`:
- `local` — репозиторий без remote (только git init)
- Для `local` поля `git_url` и `git_credentials_id` будут NULL

**`git_url`** — URL для клонирования:
- HTTPS: `https://github.com/user/repo.git`
- SSH: `git@github.com:user/repo.git`
- NULL для `local`

**`git_credentials_id`** — ссылка на credentials:
- NULL для публичных репозиториев (read-only clone без авторизации)
- NULL для `local` провайдера
- Обязательна для приватных репозиториев и для push/PR

**`vector_collection`** — имя коллекции в Weaviate:
- Формат: `DevTeam_Project_{short_id}`
- Генерируется при первой индексации проекта
- NULL до первой индексации

**`tech_stack`** (JSONB):
```json
{
  "languages": ["go", "dart"],
  "frameworks": ["gin", "flutter"],
  "databases": ["yugabytedb"],
  "tools": ["docker", "make"]
}
```

**`settings`** (JSONB):
```json
{
  "code_style": "google",
  "auto_review": true,
  "auto_test": true,
  "max_concurrent_tasks": 3,
  "default_code_backend": "claude-code"
}
```

| `idx_projects_status` | `status` | Фильтрация активных |

## SQL миграция

### UP

```sql
-- +goose Up
-- +goose StatementBegin

-- 1. Таблица git_credentials (создаётся первой, т.к. projects ссылается на неё)
CREATE TABLE git_credentials (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    user_id         UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    provider        VARCHAR(50) NOT NULL,
    auth_type       VARCHAR(50) NOT NULL DEFAULT 'token',
    encrypted_value BYTEA NOT NULL,
    label           VARCHAR(255) NOT NULL,
    created_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at      TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    CONSTRAINT chk_git_credentials_provider CHECK (provider IN ('github', 'gitlab', 'bitbucket')),
    CONSTRAINT chk_git_credentials_auth_type CHECK (auth_type IN ('token', 'ssh_key', 'oauth'))
);

CREATE INDEX idx_git_credentials_user_id ON git_credentials(user_id);
CREATE INDEX idx_git_credentials_provider ON git_credentials(user_id, provider);

-- 2. Таблица projects
CREATE TABLE projects (
    id                  UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    name                VARCHAR(255) NOT NULL,
    description         TEXT,
    git_provider        VARCHAR(50) NOT NULL DEFAULT 'local',
    git_url             VARCHAR(1024),
    git_default_branch  VARCHAR(255) NOT NULL DEFAULT 'main',
    git_credentials_id  UUID REFERENCES git_credentials(id) ON DELETE SET NULL,
    vector_collection   VARCHAR(255),
    tech_stack          JSONB DEFAULT '{}',
    status              VARCHAR(50) NOT NULL DEFAULT 'active',
    settings            JSONB DEFAULT '{}',
    user_id             UUID NOT NULL REFERENCES users(id) ON DELETE CASCADE,
    created_at          TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    updated_at          TIMESTAMP WITH TIME ZONE DEFAULT NOW(),
    CONSTRAINT chk_projects_git_provider CHECK (git_provider IN ('github', 'gitlab', 'bitbucket', 'local')),
    CONSTRAINT chk_projects_status CHECK (status IN ('active', 'paused', 'archived'))
);

CREATE INDEX idx_projects_user_id ON projects(user_id);
CREATE INDEX idx_projects_status ON projects(status);
CREATE INDEX idx_projects_git_provider ON projects(git_provider);

-- +goose StatementEnd
```

### DOWN

```sql
-- +goose Down
-- +goose StatementBegin

-- Обратный порядок: сначала projects (зависит от git_credentials), потом git_credentials
DROP INDEX IF EXISTS idx_projects_git_provider;
DROP INDEX IF EXISTS idx_projects_status;
DROP INDEX IF EXISTS idx_projects_user_id;
DROP TABLE IF EXISTS projects;

DROP INDEX IF EXISTS idx_git_credentials_provider;
DROP INDEX IF EXISTS idx_git_credentials_user_id;
DROP TABLE IF EXISTS git_credentials;

-- +goose StatementEnd
```

package models

import (
    "time"

type GitCredentialProvider string

const (
    GitCredentialProviderGitHub    GitCredentialProvider = "github"
    GitCredentialProviderGitLab    GitCredentialProvider = "gitlab"
    GitCredentialProviderBitbucket GitCredentialProvider = "bitbucket"

    switch p {
    case GitCredentialProviderGitHub, GitCredentialProviderGitLab, GitCredentialProviderBitbucket:
        return true
    default:
        return false
    }

type GitCredentialAuthType string

const (
    GitCredentialAuthToken  GitCredentialAuthType = "token"
    GitCredentialAuthSSHKey GitCredentialAuthType = "ssh_key"
    GitCredentialAuthOAuth  GitCredentialAuthType = "oauth"
)

func (a GitCredentialAuthType) IsValid() bool {
    switch a {
    case GitCredentialAuthToken, GitCredentialAuthSSHKey, GitCredentialAuthOAuth:
        return true
    default:
        return false
    }
}

type GitCredential struct {
    ID             uuid.UUID            `gorm:"type:uuid;primaryKey;default:gen_random_uuid()" json:"id"`

    User           *User                `gorm:"foreignKey:UserID" json:"user,omitempty"`
    Provider       GitCredentialProvider `gorm:"type:varchar(50);not null" json:"provider"`
    AuthType       GitCredentialAuthType `gorm:"type:varchar(50);not null;default:'token'" json:"auth_type"`
    EncryptedValue []byte               `gorm:"type:bytea;not null" json:"-"`
    Label          string               `gorm:"type:varchar(255);not null" json:"label"`
    CreatedAt      time.Time            `gorm:"type:timestamp with time zone;default:now()" json:"created_at"`
    UpdatedAt      time.Time            `gorm:"type:timestamp with time zone;default:now()" json:"updated_at"`

    return "git_credentials"
}

## Важные решения

### FK `git_credentials_id` → ON DELETE SET NULL
При удалении credentials проект **не удаляется**, а теряет привязку к credentials. Это безопаснее, чем CASCADE — проект остаётся, но push/PR перестанут работать до привязки новых credentials.

### `git_credentials.provider` vs `projects.git_provider`
- `git_credentials.provider` — `github`, `gitlab`, `bitbucket` (без `local`, т.к. для локальных credentials не нужны)
- `projects.git_provider` — `github`, `gitlab`, `bitbucket`, `local` (расширенный набор)
- Валидация в service layer: если `project.git_provider = local`, то `git_credentials_id` должен быть NULL

### `encrypted_value` — BYTEA, а не TEXT
- Шифрованные данные — это бинарные данные (AES-256-GCM output)
- BYTEA корректно хранит бинарные данные без проблем с encoding
