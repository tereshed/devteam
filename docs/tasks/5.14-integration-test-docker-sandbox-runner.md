**Sprint:** 5 — Sandbox Runner (Backend, Docker)  
**Файлы (план):**

Только тесты, которые вызывают **Docker Engine API** через **`github.com/docker/docker/client`** и публичный API **`DockerSandboxRunner`** (**5.5**), без подмены HTTP-транспорта (в отличие от **5.13**).

## Описание

Добавить **интеграционные** тесты пакета `sandbox`, которые прогоняют **реальный** жизненный цикл **`DockerSandboxRunner`** против локального **Docker Engine** (как в CI/на машине разработчика с `docker.sock`).

**Зачем отдельно от 5.13:** юнит-тесты (**`docker_runner_test.go`**) имитируют Engine через **`httptest.Server`** и **не** проверяют: реальный **`NetworkCreate`/`ContainerCreate`/`CopyToContainer`/`ContainerStart`**, cgroup/предупреждения ядра, поведение **`network: none`**, тайминги **`postStartSanity`**, совместимость с **trap EXIT** в **`deployment/sandbox/claude/entrypoint.sh`**, фактический drain **`ContainerWait`** на настоящем движке.

**Граница с `entrypoint_integration_test.go`:** существующий файл проверяет **контракт entrypoint** через **`docker` CLI** (`docker create`, `docker start`, `docker cp`). Задача **5.14** — проверить **именно Go-раннер** (**`NewDockerSandboxRunner` → `RunTask` → `Wait` / `GetStatus` → `Cleanup`**). Дублировать сценарии «как в entrypoint-тестах» без вызова раннера **не** нужно; допустимо **вынести общие** `sandboxImage(t)`, проверку наличия образа, `t.Skip` — в общий `integration_helpers_test.go` (**оба** файла под `//go:build integration`).

**«Простая задача»** в смысле дорожной карты: **один** детерминированный сценарий без реального вызова Claude API и без неограниченного ожидания агента. Рекомендуемый контракт:

1. **Предпосылки:** образ **`devteam/sandbox-claude:local`** (или переопределение через **`SANDBOX_CLAUDE_IMAGE`**, согласованное с **`entrypoint_integration_test.go`** / **`make sandbox-build`**).  
2. **Клиент:** `client.NewClientWithOpts(client.FromEnv, client.WithAPIVersionNegotiation())` + **`client.Ping`** в начале или **`t.Skip`**, если демон недоступен (как паттерн «docker в PATH» в entrypoint-тестах — здесь логичнее **`Ping`**).  
3. **Раннер:** `NewDockerSandboxRunner(cli, nil /* или явный allowlist */)` с при необходимости **`WithDefaultTaskTimeout(...)`** короче дефолта **30m** для теста (например **2–5m**), чтобы зависший прогон не блокировал CI.  
   - **Связь с README (строка про `SANDBOX_*`):** переменные окружения **`SANDBOX_*`**, **`config.Load()`** и структура **`SandboxConfig`** живут в **`backend/internal/config/sandbox_config.go`** (задача **5.10**). Интеграционный тест **5.14** в пакете **`sandbox`** **по умолчанию не обязан** вызывать **`config.Load()`**: он проверяет **раннер**, задавая таймауты/политику лимитов явно через **`RunnerOption`** (**`WithDefaultTaskTimeout`**, **`WithResourceLimitPolicy`**). При необходимости «как в проде» можно один раз сопоставить числа с дефолтами из **`loadSandboxConfig`**, либо в отдельном подтесте вызвать **`config.Load()`** и пробросить значения в опции раннера — это **опционально**, не блокер MVP.  
   - **`SANDBOX_MAX_CONCURRENT`** — лимит на уровне приложения/очереди (**не** поле **`DockerSandboxRunner`**); проверка этого параметра **не входит** в минимальный объём **5.14**, если не добавляется отдельный сценарий уровня **`cmd/api`**.  
4. **Сценарий «быстрый фейл после старта» (рекомендуемый MVP):** валидные **`SandboxOptions`** (реальный публичный **`RepoURL`**, валидная **`Branch`**, непустой **`Instruction`**, **`Backend: claude-code`**, в **`EnvVars`** — **`ANTHROPIC_API_KEY`** с **плейсхолдером**, проходящим проверку entrypoint до clone), плюс **`DisableNetwork: true`**. Ожидание: **`git clone`** в контейнере не удаётся без сети; entrypoint завершает работу с ошибкой, **`status.json`** всё равно пишется (**trap EXIT`**).  
   - **`RunTask`** должен успешно вернуть **`SandboxInstance`** (контейнер к моменту **`postStartSanity`** успевает перейти в **`running`**).  
   - **`Wait`** — завершение с ожидаемым **`SandboxStatus`**; сравнение статуса и ошибок — см. § «Типизированные статусы и ошибки» ниже (не строковые литералы вроде **`"failed"`**).  
   - **`Cleanup`** — идемпотентность, отсутствие «висячих» контейнеров/сетей с префиксом **`devteam-`** / **`devteam-task-`** для данного **`TaskID`**.  
5. **Уникальный `TaskID`:** на каждый вызов — новый UUID (или иной формат из **`ValidateTaskID`**), чтобы не ловить **`ErrSandboxRunConflict`** по имени контейнера.

### Контексты с таймаутом (обязательно)

Все вызовы **`RunTask`**, **`Wait`**, **`GetStatus`**, **`Cleanup`** (и при необходимости **`Stop`/`StopTask`**) в интеграционном тесте **обязаны** использовать **`context.WithTimeout`** / **`context.WithDeadline`** (или контекст, унаследованный от **`t.Context()`** в Go 1.21+ с явным таймаутом выше по стеку), с запасом согласованным с **`WithDefaultTaskTimeout`** и ожидаемой длительностью сценария. Цель: при «залипшем» Docker Engine тест **падает по дедлайну контекста**, а не зависает до глобального таймаута CI.

### Типизированные статусы и ошибки (обязательно)

- Статус жизненного цикла: сравнивать **`SandboxStatus.Status`** с константами пакета **`sandbox`** (например **`SandboxStatusFailed`**, **`SandboxStatusCompleted`**, **`SandboxStatusTimedOut`** — **актуальный список** в **`types.go`**), а не со строками.  
- Ошибки из API раннера: **`errors.Is`** / **`errors.As`** к **`sentinel`**-ошибкам из **`errors.go`** (или принятому в пакете контракту), без проверки **`err.Error() == "..."`** как единственного критерия.

### Безопасность: инъекции в `Branch` / `RepoURL` (см. `.cursor/rules/review.mdc`)

**`RepoURL`** и **`Branch`** попадают в **`git clone`** / ref-логику внутри контейнера; регресс **Command/Flag Injection** (ветка **`--upload-pack=`**, ведущий **`-`** в **`Branch`**, и т.п.) должен быть **отбит до Docker** в **`SandboxOptions.Validate`** / **`ValidateBranchName`** / **`ValidateRepoURL`**.

- **Минимум:** таблица негативных значений с ожиданием **быстрого фейла** и **`errors.Is(..., ErrInvalidBranchName)`** / **`ErrInvalidRepoURL`** / **`ErrInvalidOptions`** — **достаточно в юнит-слое (5.13)**, если там уже покрыто; тогда в **5.14** — **краткий комментарий** в коде со ссылкой на тесты **5.13**.  
- **Усиление для merge (рекомендуется ревью):** один **`t.Run`** в **`integration_test.go`**, который вызывает **`RunTask`** с заведомо плохим **`Branch`** (например ведущий дефис) и проверяет, что **контейнер не создаётся** (mux в юните не нужен — ошибка **до** **`ContainerCreate`**). Так фиксируется связка «валидатор + реальный раннер» на одной сборке.

### `defer Cleanup` (обязательно)

После **успешного** **`RunTask`** (получен **`SandboxInstance`** и **`sandboxID`**) **`Cleanup(ctx, sandboxID)`** **обязан** быть зарегистрирован через **`t.Cleanup`** или **`defer`** **сразу** же (до длинных **`Wait`** и до **`require`/`Fatal`**-веток), с передачей **своего** контекста с таймаутом. Иначе при **`t.Fatalf`** / панике в ассертах после **`RunTask`** останутся висячие контейнеры **`devteam-task-*`** и сети **`devteam-sbx-*`**, что засоряет локальную машину и CI.

**Паттерн (ориентир):** вычислить **`sandboxID := inst.ID`**, затем **`t.Cleanup(func() { … Cleanup(cleanupCtx, sandboxID) … })`**; **`Wait`** и проверки — после регистрации cleanup. Убедиться, что **`Cleanup`** вызывается и при **ошибке `Wait`** (cleanup не должен зависеть от успеха сценария).

**Опционально (вне минимального MVP, по времени):** второй тест с **`DisableNetwork: false`**, публичным репо и подменой **`claude`** через bind-mount **сейчас недоступен** без расширения **`SandboxOptions`/`HostConfig`**; не блокирует закрытие **5.14**, но может быть отдельной задачей на «полный happy-path под SDK».

## Надёжность и `-race`

- Пакет `sandbox` уже использует **`goleak.VerifyTestMain`** (**`docker_test_helpers_test.go`**). Интеграционные тесты **обязаны** завершать цикл **`Wait`** (или явный **`Stop`/`StopTask`**) и **`Cleanup`**, не оставляя горутин из **`containerWaitLoop`/`StreamLogs`**.  
- **`Cleanup` + `defer`/`t.Cleanup`:** см. § «`defer Cleanup`» выше — **критично** для отсутствия мусора в Docker после падения теста.  
- **`t.Parallel()`** — только если каждый тест изолирован уникальным **`TaskID`** и нет гонок по общему глобальному состоянию Docker; при флейках на shared Engine — последовательный прогон с комментарием **`// sequential: single Docker daemon`**.
