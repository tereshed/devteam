## Описание

Определить все структуры данных, используемые в интерфейсе `GitProvider` (задача 4.1). Все типы размещаются в `pkg/gitprovider/types.go` и **НЕ ЗАВИСЯТ** от `internal/models`. Это — контракт данных между интерфейсом, реализациями (4.3, 4.4) и потребителями (4.6, Sprint 6).

**Правило:** Ни один тип не импортирует пакеты из `internal/`. Маппинг из доменных/персистентных моделей приложения в `gitprovider` — ответственность вызывающего слоя (например, `ProjectService`).

## Определения типов

### 1. Credentials — аутентификация

```go
// Credentials содержит расшифрованные данные для доступа к Git-провайдеру.
// Вызывающий код отвечает за расшифровку (AES-256-GCM, Sprint 4.7)
// и маппинг из слоя приложения (сущность учётных данных БД) в эту структуру.
type Credentials struct {
    // Token — Personal Access Token / OAuth token.
    // GitHub: ghp_xxx, GitLab: glpat-xxx.
    Token string

    // SSHKey — приватный SSH-ключ (PEM-формат).
    // Используется вместо Token для SSH-клонирования.
    SSHKey string

    // SSHKeyPassphrase — пароль для SSH-ключа (если есть).
    SSHKeyPassphrase string

    // Username — имя пользователя (для Basic Auth).
    // В качестве пароля в этом случае используется поле Token.
    // Большинство провайдеров используют Token как единственный способ.
    Username string
}
```

**Дизайн-решение:** Все поля — строки (не `*string`). Пустая строка = не задано. Это упрощает использование и сериализацию. Реализация выбирает приоритет: Token > SSHKey > Username+Token (Basic Auth).

### 2. Author — автор коммита

```go
// Author задаёт автора git-коммита.
// Разделение Name/Email вместо строки "Name <email>" гарантирует корректный
// парсинг при передаче в `git commit --author` или go-git Signature.
type Author struct {
    Name  string // "DevTeam Bot"
    Email string // "bot@devteam.io"
}

// String форматирует Author для git CLI: "Name <email>".
func (a Author) String() string {
    return fmt.Sprintf("%s <%s>", a.Name, a.Email)
}
```

### 3. CloneOptions — параметры клонирования

```go
// CloneOptions задаёт параметры для GitProvider.Clone.
// repoURL передаётся отдельным аргументом метода (не в структуре).
type CloneOptions struct {
    // DestPath — локальная директория, куда будет клонирован репозиторий.
    // После Clone эта директория становится workDir для остальных операций.
    // Директория НЕ ДОЛЖНА существовать (Clone создаёт её).
    DestPath string

    // Branch — ветка для клонирования. Пустая строка = default branch провайдера.
    Branch string

    // Depth — глубина клонирования (shallow clone).
    // 0 = полный клон (все коммиты). >0 = последние N коммитов.
    // Для импорта проекта (индексация): 0 (полный).
    // Для sandbox: 50 (достаточно для diff с main).
    Depth int
}
```

### 5. CommitOptions — параметры коммита

```go
// CommitOptions задаёт параметры для GitProvider.Commit и CommitAndPush.
// workDir передаётся отдельным аргументом метода (не в структуре).
type CommitOptions struct {
    // Message — текст коммит-сообщения.
    Message string

    // Пустой слайс / nil = `git add -A` (все изменения).

    Files []string
}
```

### 12. PRMergeOptions — слияние PR

```go
// PRMergeOptions задаёт параметры для GitProvider.MergePullRequest.
type PRMergeOptions struct {
    // CommitTitle — заголовок merge-коммита (для squash/merge).
    // Пустая строка = автоматический заголовок провайдера.
    CommitTitle string

    // CommitMessage — тело merge-коммита.
    CommitMessage string

    // MergeMethod — метод слияния: "merge", "squash", "rebase".
    // Пустая строка = "merge" (по умолчанию).
    MergeMethod string

    // SHA — SHA последнего коммита head-ветки.

    // с HEAD ветки, мерж будет отклонён. Предотвращает мерж чужого коммита,
    // запушенного после нашего аппрува.
    SHA string
}
```

**Валидация MergeMethod:**

```go
// MergeMethod определяет допустимые методы слияния PR.

    MergeMethodMerge  = "merge"
    MergeMethodSquash = "squash"
    MergeMethodRebase = "rebase"
)
```

```go
    Number int

    // Title — заголовок PR.

    ChangedFiles int
```

## Дизайн-решения

### 1. Идентификаторы — аргументы, payload — в Options (⚠️ КРИТИЧНО)

**Правило:** `repoURL` и `workDir` **НЕ ЯВЛЯЮТСЯ** полями Options-структур. Они передаются **отдельными аргументами** метода. Options содержат только payload — параметры самой операции.

Это обеспечивает единообразие API: разработчику не нужно гадать, где искать идентификатор ресурса — он **всегда** в сигнатуре метода.

### 2. Flat structs, не nested

**Выбор:** Все Options-структуры — плоские (без вложенных sub-struct'ов).

**Обоснование:** Типы предназначены для одного вызова метода. Вложенность (`opts.Git.Branch`) усложняет конструирование без выгоды. Исключение — `Author` в `CommitOptions` (логическая пара Name+Email).

### 3. Nullable поля: `*string` только для partial update

**Правило:**
- **Options (вход):** Обязательные поля — значения. Опциональные — значения с «zero = не задано» (пустая строка, 0, false).
- **PRUpdateOptions** — исключение: `*string` для partial update (nil = не менять). Это единственная структура с указателями.
- **PullRequest (выход):** `*bool` для `Mergeable` (три состояния: true/false/unknown).

### 4. Строковые enum-ы вместо `iota`

**Выбор:** `State`, `MergeMethod`, `Event`, `Status`, сторона diff в review (`PRDiffSideLeft` / `PRDiffSideRight`) — строковые константы (`"open"`, `"squash"`, `"APPROVE"`, `"modified"`, `"LEFT"`).

**Обоснование:**
- Совместимость с JSON API GitHub/GitLab (строки приходят из REST API).
- Человекочитаемость при отладке и логировании.
- Нет необходимости в маппинге `int → string`.
- Константы определены рядом с типами для документации допустимых значений.

### 5. `time.Time` вместо `*time.Time`

**Выбор:** Даты в `PullRequest` (`CreatedAt`, `MergedAt`, `ClosedAt`) — `time.Time`, не `*time.Time`.

**Обоснование:** Zero time (`time.Time{}`) — валидный индикатор «не задано» (PR не смержен → `MergedAt.IsZero() == true`). Указатели добавляют nil-проверки без выгоды.

### 6. `Patch` в `PRFile`

Поле `Patch` содержит unified diff для конкретного файла. Это позволяет ревьюеру анализировать изменения файла без отдельного вызова `GetDiff`. Может быть пустым для бинарных файлов или если провайдер не поддерживает inline-diff.

### 6a. `PRFile.Changes` и поле `changes` в API провайдеров

По контракту пакета `Changes` = **Additions + Deletions**. У GitHub в ответе list PR files есть своё поле `changes`; семантика часто совпадает, но реализации (4.3/4.4) должны **явно** приводить значение к контракту `types.go`, а не копировать поле API без проверки.

### 7. `PRComment.Type` — единая модель для двух видов комментариев

GitHub/GitLab различают Issue Comments (общие) и Review Comments (к строкам кода). Вместо двух типов используется один `PRComment` с полем `Type`. Поля `Path`, `Line`, `CommitSHA`, `Side` заполняются только для `review_comment` (`Side` — константы `PRDiffSide*`).

**Плюсы:** Один слайс `[]PRComment` вместо двух. Агенту-ревьюеру не нужно мержить два списка.

### 8. `io.ReadCloser` не фигурирует в types.go

Методы `GetDiff`, `GetLocalDiff`, `GetFileContent`, `GetLocalFileContent` возвращают `io.ReadCloser` напрямую (стандартная библиотека). Обёрточные типы не нужны.
