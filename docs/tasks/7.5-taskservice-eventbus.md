# 7.5 Интеграция: TaskService → EventBus при изменении статуса

**Sprint:** 7 — WebSocket и Реалтайм (Backend)
**Файл:** `backend/internal/service/task_service.go`
**Статус:** ⬜

---

## Описание

Внедрить **`EventBus` (7.4)** в **`TaskService`** (3.3) и **публиковать доменные события** при двух классах операций:

1. **Изменение статуса задачи** — `TaskStatusChanged` (любой переход машины состояний 3.6, включая `pending → planning`, `in_progress → review`, `* → failed`, `* → cancelled`, `* → paused` и т.д.).
2. **Создание сообщения задачи** — `TaskMessageCreated` (через `AddMessage`/эквивалент, см. 3.2 / `main.mdc` §2.5).

Цель — **на каждое доменное событие в `TaskService` появляется уведомление в WS-канале** проекта без прямой зависимости сервиса на `ws.Hub`. Зависимость идёт **только** на интерфейс `ws.EventBus`.

---

## Контекст использования

```
Handler (POST /tasks/:id/cancel)
        │
        ▼
TaskService.Cancel(ctx, taskID)
        │
        ├─ tx Begin
        │     ├─ ChangeStatus(... → cancelled)   (репозиторий)
        │     └─ AddMessage(... "task cancelled")
        ├─ tx Commit  ◄────── ВАЖНО: события публикуются ПОСЛЕ Commit
        │
        ├─ bus.Publish(ctx, ws.TaskStatusChanged{...})
        └─ bus.Publish(ctx, ws.TaskMessageCreated{...})
                │
                ▼
        EventBus → HubBridge (7.4) → ws.Hub.SendToProject (7.1) → клиенты
```

Согласно `README.md` (Sprint 7): без 7.5 фронт **не получает** статусы задач, и весь WS-канал бесполезен. Это критическая интеграция.

## Функциональные требования

### 1. Внедрение зависимости (DI)

- В `TaskService` добавить поле `bus ws.EventBus` (интерфейс из 7.4 — **не** конкретная in-memory реализация).
- Расширить конструктор:
  ```go
  func NewTaskService(
      taskRepo TaskRepository,
      messageRepo TaskMessageRepository,
      stateMachine TaskStateMachine,
      bus ws.EventBus,             // ← новое
      logger Logger,
  ) *TaskService
  ```
- В `cmd/api/main.go` Bus уже существует (создан в 7.4), просто пробросить в конструктор.
- **Глобальные переменные / `init()` запрещены** (`backend.mdc` §2.2).

### 2. Точки публикации `TaskStatusChanged`

Опубликовать **на каждом** успешном переходе статуса. Это включает (но не ограничивается):

| Метод TaskService | Триггер |
|-------------------|---------|
| `Create` | `pending` → начальный статус |
| `Update` (если меняет статус) | любой переход |
| `ChangeStatus` (если есть) | любой переход |
| `Pause` | `* → paused` |
| `Cancel` | `* → cancelled` |
| `Resume` | `paused → previous` |
| `MarkFailed` (если есть) | `* → failed` (с `ErrorMessage`) |
| Внутренние переходы из Orchestrator (6.x) через TaskService | `planning → in_progress`, `in_progress → review`, и т.д. |

**DRY:** Вместо копирования блока `s.bus.Publish` с маппингом 10+ полей в каждый из 8 методов — используй приватный хелпер `publishEvents(ctx, task, prevStatus, msg)` (см. раздел 4). При добавлении нового поля в payload хелпер правится в одном месте.

Payload `ws.TaskStatusChanged`:

| Поле | Источник | Примечание |
|------|----------|------------|
| `ProjectID` | `task.ProjectID` | обязателен, из БД |
| `TaskID` | `task.ID` | |
| `ParentTaskID` | `task.ParentTaskID` | |
| `Previous` | `previousStatus` | снимок ДО перехода |
| `Current` | `task.Status` | |
| `AssignedAgentID` | `task.AssignedAgentID` | |
| `ErrorMessage` | `task.ErrorMessage` | только для failed |
| `OccurredAt` | `time.Now().UTC()` | |

### 3. Точки публикации `TaskMessageCreated`

Опубликовать на каждом **успешно сохранённом** `TaskMessage`.

Payload `ws.TaskMessageCreated`:

| Поле | Источник | Примечание |
|------|----------|------------|
| `ProjectID` | `task.ProjectID` | обязателен, из БД |
| `TaskID` | `msg.TaskID` | |
| `MessageID` | `msg.ID` | |
| `SenderType` | `string(msg.SenderType)` | |
| `SenderID` | `msg.SenderID` | |
| `SenderRole` | `msg.SenderRole` | |
| `MessageType` | `string(msg.MessageType)` | |
| `Content` | `msg.Content` | raw, маскирование — в HubBridge (7.4) |
| `Metadata` | `msg.Metadata` | raw, whitelist — в HubBridge (7.4) |
| `OccurredAt` | `msg.CreatedAt` | |

### 4. Транзакционность — публикация ПОСЛЕ Commit (КРИТИЧНО)

**Запрещено** публиковать события **внутри транзакции** или **до** её коммита. Иначе подписчик получит уведомление о факте, который потом откатится — клиент увидит «task → completed», а в БД останется `in_progress`.

**Также запрещено** публиковать событие без проверки на реальное изменение состояния — иначе спам идентичными ивентами (например, при повторном вызове `Pause` на уже приостановленной задаче).

Правильный паттерн:

```go
func (s *TaskService) Cancel(ctx context.Context, taskID uuid.UUID) error {
    var (
        task           *Task
        previousStatus string
        msg            *TaskMessage
    )
    err := s.repo.WithTx(ctx, func(tx Tx) error {
        t, err := s.repo.GetByID(ctx, tx, taskID)
        if err != nil { return err }
        previousStatus = t.Status

        if err := s.stateMachine.Apply(t, StatusCancelled); err != nil { return err }
        if err := s.repo.Update(ctx, tx, t); err != nil { return err }

        m, err := s.messageRepo.Create(ctx, tx, &TaskMessage{...})
        if err != nil { return err }

        task = t
        msg = m
        return nil
    })
    if err != nil { return err }

    // ↓ ПОСЛЕ Commit, с защитой от nil и дублей
    s.publishEvents(ctx, task, previousStatus, msg)
    return nil
}

// publishEvents — приватный хелпер для DRY-публикации событий.
// task — обязателен (не nil), так как из него берётся ProjectID для роутинга в EventBus.
// prevStatus — предыдущий статус до перехода (для TaskStatusChanged).
// msg — может быть nil, если сообщение не создавалось (TaskMessageCreated не публикуется).
func (s *TaskService) publishEvents(ctx context.Context, task *Task, prevStatus string, msg *TaskMessage) {
    // Отвязываем от родительского ctx, чтобы закрытая вкладка клиента
    // не заблокировала доставку остальным подписчикам проекта.
    pubCtx := context.WithoutCancel(ctx)

    // Публикуем TaskStatusChanged ТОЛЬКО при реальном переходе
    if task != nil && task.Status != prevStatus {
        s.bus.Publish(pubCtx, ws.TaskStatusChanged{
            ProjectID: task.ProjectID, TaskID: task.ID,
            ParentTaskID: task.ParentTaskID,
            Previous: prevStatus, Current: task.Status,
            AssignedAgentID: task.AssignedAgentID,
            ErrorMessage:    task.ErrorMessage,
            OccurredAt:      time.Now().UTC(),
        })
    }

    // Публикуем TaskMessageCreated ТОЛЬКО если сообщение создано
    // task != nil обязательна — нужен ProjectID для роутинга
    if msg != nil && task != nil {
        s.bus.Publish(pubCtx, ws.TaskMessageCreated{
            ProjectID: task.ProjectID, TaskID: msg.TaskID, MessageID: msg.ID,
            SenderType: string(msg.SenderType), SenderID: msg.SenderID,
            SenderRole: msg.SenderRole, MessageType: string(msg.MessageType),
            Content:    msg.Content,   // raw, маскирование — в HubBridge (7.4)
            Metadata:   msg.Metadata,  // raw, whitelist — в HubBridge (7.4)
            OccurredAt: msg.CreatedAt,
        })
    }
}
```

**Снимок «previous status»** делается **до** `Apply`/`Update`, иначе после `Update` он уже потерян.

### 4.1. Edge Cases & Validation (обязательно к обработке)

При реализации необходимо предусмотреть:

1. **nil pointer при отсутствии сообщения.** Если метод меняет только статус и не создаёт `TaskMessage`, `msg` остаётся `nil`. Без проверки `msg.TaskID` вызовет panic. `publishEvents` проверяет `msg != nil` перед публикацией `TaskMessageCreated`.
2. **nil pointer для task (ProjectID).** Если `task == nil` при ненулевом `msg`, `task.ProjectID` вызовет panic. Условие `task != nil` добавлено в оба блока Publish.
3. **Повторный переход без изменения статуса.** При вызове `Pause` на уже приостановленной задаче `previousStatus == task.Status`. Публикация `TaskStatusChanged` при отсутствии реального перехода — спам идентичными ивентами. `publishEvents` пропускает публикацию, если `task.Status == prevStatus`.
4. **Контекст отменён после Commit, но до Publish.** HTTP-запрос отменён (пользователь закрыл вкладку), но БД уже закоммичена. `context.WithoutCancel(ctx)` гарантирует доставку остальным подписчикам.
5. **Попытка отменить уже отменённую задачу.** Машина состояний вернёт ошибку → транзакция откатится → `Publish` не вызывается. Корректное поведение.

### 5. Outbox vs in-process — явный выбор

В **MVP — публикация прямо в Bus после Commit** (см. п. 4). Это означает:

- Если приложение упадёт **между Commit и Publish**, событие потеряется. Клиент через REST увидит свежий статус, но live-уведомления не будет.
- Это **сознательный compromise** для MVP. Пользователь всегда может перезапросить состояние задачи через `GET /tasks/:id` и `GET /tasks/:id/messages`.
- **Будущее (вне 7.5):** transactional outbox-таблица + воркер-релэй. Зафиксировать как ADR/TODO в коде:
  ```go
  // TODO(7.x): перейти на transactional outbox для гарантии at-least-once доставки.
  ```

### 6. Контекст и отмена

- Используем **`context.WithoutCancel(ctx)`** для Publish. Это отвязывает lifecycle HTTP-запроса от lifecycle доставки события: если клиент закрыл вкладку и ctx отменён, остальные подписчики проекта всё равно получат уведомление.
- **НЕ** делать `context.Background()` — теряется trace_id из оригинального ctx. `WithoutCancel` сохраняет Values (trace, baggage), но сбрасывает Done-канал.
- Пример:
  ```go
  pubCtx := context.WithoutCancel(ctx)
  s.bus.Publish(pubCtx, ev)
  ```

### 7. Безопасность и приватность

- **Никаких raw-секретов в `TaskMessage.Content`/`Metadata`** на стороне TaskService нет — маскирование делает HubBridge (7.4). Здесь обязанность — **не превращать ошибки слоя в WS-сообщения с stacktrace**: `ErrorMessage` для `TaskStatusChanged` берётся из контролируемого `task.ErrorMessage` поля, не из `err.Error()` бизнес-ошибок.
- **Кросс-проектная утечка:** в payload **обязательно** ставится `ProjectID` именно из загруженной из БД сущности, а не из request-параметра — иначе IDOR через подмену.
- ⚠️ **Внимание: Content и Metadata передаются raw в EventBus.** Если к Bus позже подключатся новые подписчики (аналитика, аудит-логгер), они получат сырые данные без маскировки. Для MVP это приемлемо, но все новые подписчики EventBus должны маскировать Content перед логированием.

### 8. Обработка ошибок Publish

- `Publish` сам по себе **не возвращает ошибку** (см. контракт 7.4). Он либо доставит, либо дропнет с метрикой/логом.
- `TaskService` **не должен** падать на нём в любом случае.
- Если в Bus переполнение — клиент перепрочитает состояние через REST, see п. 5.

### 9. Без изменения контрактов API

- Этот тикет **НЕ** меняет HTTP-API (DTO, эндпоинты, Swagger) — только внутреннюю проводку. `make swagger` запускать не обязательно, если public API не менялся.
- MCP-инструменты (`backend.mdc` §7) — без изменений (event publishing не является публичным эндпоинтом).

## Анти-паттерны (что ЗАПРЕЩЕНО)

| Анти-паттерн | Почему плохо |
|--------------|--------------|
| `bus.Publish` внутри транзакции | Откат → клиент видит несуществующий факт |
| `ws.NewHub()`/`ws.Hub` в конструкторе TaskService | Нарушение Clean Architecture: домен → инфра |
| `panic(err)` при ошибке Publish | Bus не возвращает ошибки публикации (см. 7.4) |
| Глобальный `var bus ws.EventBus` | `backend.mdc` §2.2 — DI обязателен |
| Прямой `hub.SendToProject(...)` из TaskService | Sanitize-pipeline размажется (см. 7.4 §11) |
| `context.Background()` для Publish | Потеря trace_id, разрыв tracing |
| Восстановление `previous status` через второй `GetByID` | Лишний запрос + race с конкурентным `Update` |

---

## Дизайн-решения

1. **Зависимость на интерфейс `ws.EventBus`, а не на конкретный тип.** Позволяет мокать в unit-тестах и подменять реализацию (Redis Pub/Sub) без правок сервиса.
2. **Publish ПОСЛЕ Commit:** единственный безопасный паттерн при отсутствии outbox. Гарантирует, что события не «обманывают» клиента.
3. **Снимок previous_status до Apply:** дешевле и корректнее, чем повторное чтение из БД.
4. **`context.WithoutCancel`:** разделяет lifecycle HTTP-запроса и lifecycle публикации события. Закрытая вкладка пользователя ≠ повод не уведомить остальных подписчиков проекта.
5. **MVP без outbox + явный TODO:** прагматичный компромисс. Outbox — отдельный architectural concern с миграциями и воркерами, не вписывается в Sprint 7.

- [ ] `TaskService` принимает `ws.EventBus` через конструктор; глобальных переменных нет.
- [ ] На каждом успешном переходе статуса задачи публикуется `ws.TaskStatusChanged` с корректными `ProjectID`, `Previous`, `Current`, `OccurredAt`.
- [ ] На каждом успешном создании `TaskMessage` публикуется `ws.TaskMessageCreated`.
- [ ] **Все** Publish-вызовы происходят **после** `Commit` транзакции; при `Rollback` публикации нет.
