# 6.1 Интерфейс `AgentExecutor`

**Sprint:** 6 — Orchestrator Agent (Backend)
**Файл:** `backend/internal/agent/executor.go`
**Статус:** ⬜ Не выполнена

---

## Описание

Определить Go-интерфейс **`AgentExecutor`** — контракт между **`OrchestratorService`** (задача **6.4**) и **конкретными способами выполнения шага агента** по задаче: вызов LLM с промптом и инструментами (**`LLMAgentExecutor`**, **6.2**) либо запуск изолированного sandbox (**`SandboxAgentExecutor`**, **6.3**, поверх **`SandboxRunner`** из Sprint 5).

**Ключевой принцип:** Интерфейс описывает **один атомарный «прогон агента»** для уже подготовленного контекста: оркестратор загрузил данные из БД, собрал входную структуру и вызывает **`Execute`**. Исполнитель **не** подменяет собой `TaskService` / репозитории: он **не меняет статус задачи в БД** сам по себе (это ответственность оркестратора после разбора **`ExecutionResult`**), если только в отдельной задаче явно не решено иное.

**Граница ответственности:**

| Компонент | Загрузка `Task` / `Agent` / `Project` из БД | Вызов LLM / Docker | Обновление `tasks.status`, артефактов |
|-----------|---------------------------------------------|--------------------|----------------------------------------|
| `OrchestratorService` | ✅ | через **`AgentExecutor`** | ✅ |
| `LLMAgentExecutor` / `SandboxAgentExecutor` | ❌ (только то, что передано во входе) | ✅ | ❌ (MVP) |

## Зависимости

**Требует:** **Sprint 3** — доменные модели `models.Task`, `models.Agent`, `models.Project` (для **сборки** входа в оркестраторе; см. раздел «Зависимость от `models`»). **Sprint 5** — для **`SandboxAgentExecutor` (6.3)** понадобится инжект **`sandbox.SandboxRunner`**; сам интерфейс **`AgentExecutor`** **не** импортирует пакет `sandbox` (избегаем циклов и протекающих абстракций).

**Используется в:** **6.2** (`LLMAgentExecutor`), **6.3** (`SandboxAgentExecutor`), **6.4** (`OrchestratorService` — выбор реализации по роли агента / `code_backend` / политике пайплайна), **6.10** (юнит-тесты оркестратора с моком **`AgentExecutor`**).

**Порядок с типами входа/выхода:** В README **6.1** идёт первой в спринте. Если в одном файле неудобно уместить интерфейс и объёмные **`ExecutionInput` / `ExecutionResult`**, **допустимо** в одном PR добавить **`executor.go` + `execution_types.go`** (или `types.go` в пакете `agent`), сохранив смысловую нумерацию: контракт интерфейса — этот документ; поля структур — согласованы с **6.2** и **6.3**.

## Контекст использования

Согласно `README.md` (Sprint 6, линейный MVP-пайплайн):

```
User Message → Orchestrator (LLM) → Planner (LLM) → Developer (Sandbox)
    → Reviewer (LLM) → Tester (Sandbox) → Completed
```

Оркестратор на каждом шаге:

1. Определяет **какого агента** вызывать и с какой **задачей** (`Task`).
2. Собирает **все строки, JSON-контекст, секреты и ссылки на репо**, нужные выбранному исполнителю.
3. Вызывает **`AgentExecutor.Execute(ctx, input)`** у нужной реализации (или у **фасада**, если введёте маршрутизацию внутри одного типа).
4. По **`ExecutionResult`** и ошибкам обновляет БД, ставит следующий статус, решает ветвления (**6.5–6.7**).

**Разделение слоёв (Clean Architecture):**

| Слой | Знание о `AgentExecutor` | Знание о деталях LLM / Docker |
|------|---------------------------|-------------------------------|
| `handler` | ❌ | ❌ |
| `OrchestratorService` | ✅ вызывает интерфейс | ❌ только через результат абстракции |
| `internal/agent` | ✅ определение + реализации **6.2–6.3** | ✅ внутри конкретных типов |

## Интерфейс и типы

### executor.go (и при необходимости execution_types.go)

Ниже — целевой контракт. Точный набор полей **`ExecutionInput` / `ExecutionResult`** уточняется совместно с **6.2** и **6.3**; минимум для MVP зафиксирован так, чтобы **LLM** и **Sandbox** могли разделить одну сигнатуру без «условных полей для всех».

---

### Безопасность (согласно `.cursor/rules/review.mdc`, §1)

**Command / Flag Injection (критично):** Поля **`GitURL`**, **`GitDefaultBranch`**, **`BranchName`** попадают в цепочку, где **`SandboxAgentExecutor` (6.3)** и/или entrypoint формируют вызовы **`git`** и оболочки. Значения могут исходить из БД, пользователя или «галлюцинаций» LLM. **Обязательное требование к реализациям 6.2–6.3:**

- Валидировать формат (ветка: допустимый набор символов ref без ведущего `-`; URL: разбор как URL без произвольных аргументов командной строки — политика в одной функции на пакет, early return).
- При любой передаче пользовательских/внешних строк в **CLI** после фиксированных флагов использовать **разделитель `--`**, после которого идёт только позиционный аргумент (пример: `git checkout -- <branch>`), чтобы значения вида `-h`, `--upload-pack=...` не интерпретировались как флаги `git`. То же относится к скриптам и обёрткам, где строки попадают в argv.
- Ссылка на правило: **«1. Безопасность и Инъекции (Command/Flag Injection)»** в `review.mdc`.

**Утечка секретов (Token Leakage):** Случайный дамп входа (`zap.Any("input", in)`, `fmt.Printf("%+v", in)`) раскроет **`EnvSecrets`**. **Обязательное требование:** тип **`ExecutionInput`** реализует **`fmt.Stringer`** (`String() string`): во всех значениях карты **`EnvSecrets`** при выводе подставляется маска **`***`** (ключи можно оставить читаемыми для отладки имён переменных). Так снижается риск утечки при типичных логгерах. Дополнительно: по-прежнему **запрещено** явно логировать сырые значения секретов и включать их в **`ExecutionResult`**. Ссылка на правило: **«1. Утечка секретов»** в `review.mdc`.

**`GitURL` и секреты в URL:** **`GitURL` не должен** содержать пароль/токен в **userinfo** (`https://x-access-token:SECRET@github.com/...`): учётные данные передаются через **`EnvSecrets`** (и далее в sandbox по контракту **5.2** / **5.4**), чтобы **`String()`** и логи не унаследовали утечку из URL (см. также предупреждения в **5.4** про безопасный вывод `SandboxOptions`).

---

### Семантика JSON и пустых значений (согласно `review.mdc`, §3 и §7)

Поля **`ContextJSON`**, **`StructuredContext`**, **`ArtifactsJSON`** (вход/выход) имеют тип **`json.RawMessage`** (`encoding/json`): это тот же `[]byte` под капотом, но явно документирует «сырой JSON» и корректно встраивается в маршалинг без двойного экранирования.

**Контракт для `ContextJSON` и `StructuredContext`:** если значение **`nil`** или **`len == 0`**, реализации **6.2–6.3** обязаны трактовать это как **пустой JSON-объект `{}`** при логике парсинга: **не** считать это ошибкой ввода и **не** вызывать `json.Unmarshal` от «пустого смысла» без ветки — сначала нормализовать к `{}` либо пропустить merge в структуру. **Early return с ошибкой** — только при **ненулевой** длине и **синтаксически невалидном** JSON (ошибка уровня контракта / `ErrInvalidExecutionInput` по политике ниже).

**Контракт для `ArtifactsJSON` в результате:** `nil` / пустой **`json.RawMessage`** — «нет артефактов»; при записи в БД оркестратор может сохранить как `{}` или `NULL` по политике **6.4**.

### Жёсткий контракт: `error` vs `Success` (согласно `review.mdc`, §3)

Интерфейс **не** оставляет выбор «на потом»: **`LLMAgentExecutor` (6.2)** и **`SandboxAgentExecutor` (6.3)** ведут себя одинаково.

| Возврат | Смысл |
|---------|--------|
| **`error != nil`** | Только **инфраструктурные и системные** сбои: сеть недоступна, **`ctx.Done()`** (отмена/дедлайн), Docker Engine / API, упавший провайдер LLM, **невалидный вход** (в т.ч. невалидный JSON при ненулевой длине), не сконфигурированный исполнитель (`ErrExecutorNotConfigured`). |
| **`error == nil` и `Success == false`** | Агент / sandbox **отработали штатно до конца**, но **бизнес-цель шага не достигнута**: тесты упали, сборка не прошла, LLM явно вернула отказ, ненулевой exit code контейнера по **политике результата** — при этом stderr/stdout/diff уже собраны и отражены в **`Output`** / **`ArtifactsJSON`**. |

**Запрещено:** возвращать **`error != nil`** для «чисто бизнесового» провала (чтобы оркестратор не смешивал ветки «система сломалась» и «задача не решена»). **Запрещено:** возвращать **`error == nil` и `Success == true`**, если по смыслу шага агент зафиксировал фатальный провал (например exit code тестов ≠ 0) — такой случай → **`Success == false`**.

Комментарии к **`AgentExecutor.Execute`** в коде **обязаны** дословно отражать эту таблицу (кратко, в godoc).

```go
package agent

import (
	"context"
	"encoding/json"
	"fmt"
	"strings"
)

// AgentExecutor — один шаг выполнения логики агента над подготовленным контекстом задачи.
// Реализации: LLMAgentExecutor (6.2), SandboxAgentExecutor (6.3).
//
// Контракт отмены: ctx отмены/дедлайна должен прерывать сетевой вызов LLM и инициировать
// остановку sandbox (через SandboxRunner.Stop/Cleanup) в реализации 6.3 — детали в задаче 6.3.
//
// Контракт ошибок (жёстко, одинаково для 6.2 и 6.3):
//   - error != nil: только инфраструктура/система, отмена ctx, невалидный вход (в т.ч. JSON при len>0), не сконфигурированный executor.
//   - error == nil && Success == false: штатное завершение, но бизнес-цель не достигнута; детали в Output / ArtifactsJSON.
//   - error == nil && Success == true: шаг считается успешным по политике исполнителя.
type AgentExecutor interface {
	// Execute выполняет работу агента один раз (один «тик» пайплайна для данной задачи).
	// Не пишет в БД и не меняет Task.Status — это обязанность OrchestratorService (6.4+).
	Execute(ctx context.Context, in ExecutionInput) (*ExecutionResult, error)
}

// ExecutionInput — данные, которые оркестратор собрал до вызова исполнителя.
// Рекомендация: не передавать *gorm.DB и не передавать целые ORM-графы без необходимости;
// копируйте строки/UUID в поля ввода, чтобы юнит-тесты исполнителя не тянули БД.
//
// String реализует fmt.Stringer: значения EnvSecrets маскируются как "***" (защита от случайного логирования).
// Запрещено логировать структуру через reflect или %#v в обход String().
type ExecutionInput struct {
	TaskID    string
	ProjectID string

	Title       string
	Description string
	// ContextJSON — сырой JSON задачи (models.Task.Context). См. раздел «Семантика JSON и пустых значений».
	ContextJSON json.RawMessage

	AgentID   string
	AgentName string
	Role      string

	Model        string
	PromptSystem string
	PromptUser   string

	// GitURL, GitDefaultBranch, BranchName — см. раздел «Безопасность»: валидация + git CLI только с "--" перед пользовательскими ref/путями.
	GitURL           string
	GitDefaultBranch string
	BranchName       string

	CodeBackend string

	EnvSecrets map[string]string

	// StructuredContext — сырой JSON доп. контекста роли. См. раздел «Семантика JSON и пустых значений».
	StructuredContext json.RawMessage
}

// String маскирует значения EnvSecrets как "***" (fmt.Stringer). Усечение очень длинных
// полей (PromptUser, Description) для логов — по усмотрению PR, но секреты никогда не в открытом виде.
func (in ExecutionInput) String() string {
	var b strings.Builder
	b.WriteString("ExecutionInput{TaskID:")
	b.WriteString(in.TaskID)
	b.WriteString(" ProjectID:")
	b.WriteString(in.ProjectID)
	b.WriteString(" Role:")
	b.WriteString(in.Role)
	b.WriteString(" GitURL:")
	b.WriteString(in.GitURL)
	b.WriteString(" BranchName:")
	b.WriteString(in.BranchName)
	b.WriteString(" EnvSecrets:")
	if len(in.EnvSecrets) == 0 {
		b.WriteString("{}")
	} else {
		b.WriteByte('{')
		first := true
		for k := range in.EnvSecrets {
			if !first {
				b.WriteString(", ")
			}
			first = false
			fmt.Fprintf(&b, "%q:***", k)
		}
		b.WriteByte('}')
	}
	b.WriteByte('}')
	return b.String()
}

// ExecutionResult — нормализованный выход одного Execute для записи в Task / TaskMessage оркестратором.
// Success: см. таблицу «Жёсткий контракт: error vs Success».
type ExecutionResult struct {
	Success bool
	Summary string
	Output  string

	// ArtifactsJSON — сырой JSON для models.Task.Artifacts; nil/пусто — нет артефактов.
	ArtifactsJSON json.RawMessage

	PromptTokens     int
	CompletionTokens int
	SandboxInstanceID string
}
```

**Замечания по семантике:**

- **`Execute` и идемпотентность:** Повторный вызов с тем же `ExecutionInput` может привести к повторному запуску контейнера или повторному списанию токенов. Обеспечение идемпотентности (тот же `sandboxID`, дедупликация) — зона **6.4** / политики оркестратора, не интерфейса.
- **`error` vs `Success`:** См. таблицу выше; отдельные «договорённости» между реализациями **не допускаются** — оркестратор (**6.4**) опирается только на этот контракт.

## Зависимость от `models`

**Два допустимых варианта (выбрать один в реализации PR):**

| Вариант | Плюсы | Минусы |
|--------|--------|--------|
| **A.** `ExecutionInput` содержит только примитивы (`string`, `json.RawMessage`, `map`) | Пакет `agent` не импортирует `internal/models`; проще моки | Дублирование полей при сборке в оркестраторе |
| **B.** В состав входа включаются `*models.Task`, `*models.Agent` | Меньше копирования | Тесты и реализации теснее связаны с GORM-моделями |

Для соответствия духу **`SandboxRunner`** (без доменных типов в контракте инфраструктуры) рекомендуется **вариант A** для полей, уходящих в sandbox и внешние API; при необходимости отдельные указатели на `models` допускаются **только** если команда явно зафиксирует это в **6.4**.

## Ошибки

Sentinel-ошибки в пакете `agent` (в `executor.go` или `errors.go`):

| Ошибка | Когда |
|--------|--------|
| `ErrInvalidExecutionInput` | Пустой `TaskID`, отсутствует обязательное поле для данной реализации (например пустой `PromptSystem` у LLM), невалидный JSON |
| `ErrExecutionCancelled` | Опционально: обёртка над `context.Canceled` / `DeadlineExceeded` при отмене `ctx` (**возврат как `error != nil`**, не как `Success == false`) — единый стиль в **6.2–6.3** |
| `ErrExecutorNotConfigured` | Реализация не получила зависимость (нет API-ключа, не задан `SandboxRunner`) |

Маппинг на HTTP **не** в пакете `agent` (оркестратор не HTTP-слой; HTTP — в `handler` при отдельных админ-ручках, если появятся).

## Дизайн-решения

1. **DI:** Конкретные исполнители и (при необходимости) фасад маршрутизации передаются в `OrchestratorService` через конструктор; **без** глобальных переменных и `init()`.
2. **Context:** `context.Context` — первый аргумент `Execute`; отмена обязана доходить до LLM HTTP и Docker (**6.2**, **6.3**).
3. **Без Gin / HTTP** в интерфейсе и в типах входа/выхода.
4. **Один метод на шаг:** `Execute` — минимальный контракт; стриминг токенов/логов в UI через **Sprint 7** (EventBus / WS), а не расширение интерфейса **6.1**, если только не появится отдельная задача на `ExecuteStreaming`.
5. **Секреты:** только в `EnvSecrets`; запрещено логировать сырые значения; **`ExecutionInput` реализует `fmt.Stringer`** с маскированием значений как **`***`** (см. раздел «Утечка секретов»). Предпочтительно логировать `in.String()`, а не `%+v` / `reflect` без маски.
6. **Git-строки:** валидация + **`--`** в argv перед ref/путями из **`GitURL` / `BranchName` / `GitDefaultBranch`** в любых вызовах CLI (**6.3**, entrypoint — согласовать с **5.2**).
7. **Связь с Sandbox (6.3):** `SandboxAgentExecutor` маппит `ExecutionInput` → `sandbox.SandboxOptions` (строковые ID, ветки, `Instruction`/`Context` по политике **5.2** / **5.4**).

Согласовано с `main.mdc` (разделы **3** «Система AI-агентов», **4** «Sandbox Runner»): оркестратор управляет задачами и агентами; **изолированное выполнение кода** остаётся за **`SandboxRunner`**; **одиночный вызов рассуждения** — за LLM-исполнителем. Интерфейс **`AgentExecutor`** — точка расширения для смены провайдера LLM или раннера без переписывания **`OrchestratorService`**.
