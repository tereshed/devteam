## JSONB в теле запроса (wire-формат)

В Go у полей вроде **`Context`**, **`Metadata`** стоит **`swaggertype:"string"`** только для **Swagger 2.0**; на wire при **`ShouldBindJSON`** Gin ожидает **вложенный JSON-объект** (`{}` / произвольная структура), **не** строку с экранированным JSON внутри.

**На клиенте:** в **`Map<String, dynamic>`** класть обычные Map/List; Dio сериализует тело как **JSON-объект**. **Запрещено:** `jsonEncode(map)` в строку и отправка одной строкой вместо объекта — сломает биндинг на сервере.

## Дополнение `domain/requests.dart` (request-модели + фильтры)

Файл **уже** содержит **`TaskListResponse`** / **`TaskMessageListResponse`** и директивы **`part`** для codegen. **Добавить в конец** (без дублирования `part` / лишних импортов) Freezed-классы **1:1** с Go:

| Go (`task_dto.go`) | Dart |
|--------------------|------|
| `CreateTaskRequest` | `CreateTaskRequest` |
| `UpdateTaskRequest` | `UpdateTaskRequest` |
| `CorrectTaskRequest` | `CorrectTaskRequest` |
| `CreateTaskMessageRequest` | `CreateTaskMessageRequest` |

**Обязательно:** immutable **`TaskListFilter`** (Freezed в том же **`requests.dart`** или отдельный файл с тем же `part` — на выбор, но **один** тип, через который **`listTasks`** строит query — **только фильтры**, см. следующий абзац), чтобы **12.3** не собирала query вручную на экранах.

**Состав `TaskListFilter` vs параметры `listTasks`:** в **`TaskListFilter`** входят **только** query-фильтры из **`ListTasksRequest`**, совпадающие по смыслу с DTO: **`status`**, **`statuses`**, **`priority`**, **`assignedAgentId`**, **`createdByType`**, **`createdById`**, **`parentTaskId`**, **`rootOnly`**, **`branchName`**, **`search`**, **`orderBy`**, **`orderDir`** (направление — через **`TaskOrderDir`** / канон **`'ASC'`** / **`'DESC'`**, см. § ниже). **`limit`** и **`offset`** — **не** поля фильтра: они **только** параметры метода **`listTasks(..., limit:, offset:)`**, чтобы дефолты пагинации (**50 / 0**) не дублировались и не спорили с **`TaskListFilter`**. Поле **`search`** — **серверная** семантика полнотекста; локально «входит ли задача в выдачу» не воспроизвести — после мутаций при активном поиске см. **12.3** § **50** (**`invalidate`**, не локальный upsert).

**Ограничение query (явно):** в HTTP-контракте **`ListTasksRequest`** (`task_dto.go`) **нет** поля вроде **`updated_at_before`**. Оно есть во внутреннем **`repository.TaskFilter`**, но **хендлер его не биндит** — клиент **не** отправляет **`updated_at_before`** и не «добавляет по аналогии с репозиторием Go».

### Поля **`CreateTaskRequest`**

- `title` — `required`, **max 500** символов (binding на бэкенде).  
- `description` — строка (может быть пустой).  
- `priority` — строка; **пустая / отсутствие ключа** → на сервисе **`medium`** (`parseTaskPriority` в **`task_service.go`**).  
- `parent_task_id`, `assigned_agent_id` — опциональные UUID как **`String?`** в JSON.  
- **`context`** — **`@Default(<String, dynamic>{}) Map<String, dynamic>`**; на wire — **объект**, см. § **JSONB**.

### Поля **`UpdateTaskRequest`** (частичное обновление)

Для **`toJson`** — **`@JsonSerializable(includeIfNull: false)`** на классе (прецедент **`UpdateProjectRequest`**, **10.2**).

| JSON-ключ (Go) | Поле Dart | Примечание |
|----------------|-----------|------------|
| `title` | `String?` | omitempty |
| `description` | `String?` | |
| `priority` | `String?` | |
| `status` | `String?` | omitempty; обязательный **`///` dartdoc** на геттере/поле — см. абзац сразу под таблицей |
| `assigned_agent_id` | `String?` | UUID строкой |
| `clear_assigned_agent` | `bool` | **`@Default(false)`**, тип **не nullable** — в JSON уходит **`false`** и **`true`**; **`includeIfNull: false`** отсекает только **`null`**, не bool. Делать поле **`bool?`** «для оптимизации» **нельзя**: ломается явная семантика сброса агента на сервере. |
| `branch_name` | `String?` | **обязательно** в модели (**`task_dto.go:52`**) |

**Dartdoc на `UpdateTaskRequest.status` (обязательно в коде):** обычно изменение статуса — через **`pauseTask` / `cancelTask` / `resumeTask`**. Прямое поле **`status`** в **`updateTask`** использовать только когда фича явно требует (например, ручная пометка **`failed`** админом); иначе легко обойти ожидаемый поток и проверки переходов на бэкенде (**12.3** / **12.4**).

**Семантика `clear_assigned_agent` + `assigned_agent_id` на бэкенде:** в **`task_service.Update`** (**~562–568**) при **`clear_assigned_agent == true`** агент **сбрасывается**, ветка **`assigned_agent_id` не выполняется**; при **`false`** и ненулевом id — назначение. Одновременная передача **обоих** не даёт **400**: победит **`clear_assigned_agent`**. Отдельной серверной валидации «нельзя оба» **нет**.

**Клиент (опционально, превентивно):** если **`assigned_agent_id != null && clear_assigned_agent == true`** → **`ArgumentError`** до сети — **только** защита вызывающего кода от противоречивого запроса; это **не** «зеркало» серверной ошибки и **не** часть HTTP-контракта.

### Поля **`CorrectTaskRequest`**

- **`text`**: `required`, min 1 символ; лимит размера — **`utf8.encode(text).length ≤ 256 * 1024`** (`UserCorrectionMaxBytes`). Ранний **`ArgumentError`** в **`correctTask`** при превышении — **рекомендуется** (UX + тест из матрицы).

### Поля **`CreateTaskMessageRequest`**

- `content`, `message_type` — required (типы сообщений — **12.1** / `task_message_model.dart`).  
- **`metadata`** — **`@Default(<String, dynamic>{})`**; wire — объект, см. § **JSONB**.

После правок **`requests.dart` / фильтров:** один раз **`make frontend-codegen`**; в чек-листе приёмки — **нет незакоммиченного diff** в `*.g.dart` / `*.freezed.dart` после прогона.

## Направление сортировки (`order_dir`)

На бэкенде **`sanitizeOrderDir`** (`internal/repository/helpers.go`):

```go
if strings.ToUpper(dir) == "ASC" {
    return "ASC"
}
return "DESC"
```

Итого: **`asc`**, **`ASC`**, **`Asc`** → сортировка **ASC**; пустая строка, **`desc`**, **`descending`**, **`foo`** → **DESC** (без ошибки 400).

**Клиент:** в **`TaskListFilter`** не принимать «сырой» **`String?`** от UI без нормализации. Канон: **`enum TaskOrderDir { asc, desc }`** с методом **`String toWire()`** → ровно **`'ASC'`** или **`'DESC'`**, либо два публичных литерала **`kTaskOrderDirAsc` / `kTaskOrderDirDesc`**. Запрещено прокидывать произвольные строки вроде **`descending`** — иначе бэкенд молча даст **DESC**, а ожидания UI разойдутся без сигнала об ошибке.

На **`TaskHandler`** — **`normalizeTaskListPagination`** (`task_handler.go`):

| Вход | После нормализации |
|------|---------------------|
| `limit <= 0` | **50** |
| `limit > 200` | **200** |

**Клиент:** те же правила в статиках репозитория. **Дефолт** аргумента **`limit`** в **`listTasks`** / **`listTaskMessages`** — **`50`**.

**Важно:** у задач **50 / 200**; у чата (**11.3**) — **20 / 100**.

## Фильтры списка задач (`ListTasksRequest` → query)

Имена query — как **`form`-теги** в **`task_dto.go`** (их собирает **`TaskListFilter.toQueryParameters()`** + **`limit`/`offset`** из аргументов **`listTasks`**):  
`status`, **`statuses`** (multi), `priority`, `assigned_agent_id`, `created_by_type`, `created_by_id`, `parent_task_id`, `root_only`, `branch_name`, `search`, плюс **`limit`**, **`offset`**, **`order_by`**, **`order_dir`**.

**`statuses` (multi) и `ListFormat`:** ожидаемый wire для Gin — **`statuses=A&statuses=B`** (повтор ключа). В проекте **`ListFormat`** в **`BaseOptions`** / **`dioClientProvider`** **сейчас не задан** явно (дефолт Dio — **`multiCompatible`**); **глобально** выставлять **`ListFormat.multi`** **нельзя** — затронет чат, проекты и прочие фичи, где формат query может отличаться.

**Канон:** только **per-request** — в **`listTasks`** при вызове **`_dio.get(..., queryParameters: map, options: Options(listFormat: ListFormat.multi))`** (или эквивалент: **`queryParameters`** как **`Map<String, dynamic>`** со значением-**`List<String>`** для ключа **`statuses`**, если команда проверила, что Dio сериализует это в повтор ключей **без** смены глобальных настроек). **`dioClientProvider` / `dio_client`** — **не** менять ради задач.

**Обязательный** юнит-тест: два значения **`statuses`** → matcher на итоговый query (см. матрицу).

Допустимые **`order_by`:** `created_at`, `updated_at`, `title`, `status`, `priority`, `started_at` (`allowedTaskOrderColumns` в **`task_repository.go`**).

### Сетевые ошибки (без HTTP-кода)

Как **`ConversationRepository._handleDioError`**: если **`parseDioApiError`** даёт **`statusCode == null`** (таймауты **`connectionTimeout` / `receiveTimeout` / `sendTimeout`**, **`connectionError`** и т.д.), кидать **`TaskApiException`** с **`isNetworkTransportError: true`** (и **`statusCode: null`**, **`apiErrorCode: null`**), текст из **`p.sanitizedMessage`**. Отдельного **`NetworkException`** в **`core/`** нет — канон тот же, что у чата: **`ConversationApiException`** / здесь **`TaskApiException`**. **Не** пробрасывать **`DioException`** наружу.

## Дисамбигуация 404

- **`/projects/` … `/tasks`** (только список или создание задач под проектом) → **`ProjectNotFoundException`**

**Жёсткое правило для `/tasks/...`:** любой **404** на путях вида **`/tasks/{id}`**, **`/tasks/{id}/pause`**, **`…/messages`** и т.д. маппится в **`TaskNotFoundException`**, **независимо** от того, отражён ли внутренне ответ как «задача не найдена» или «проект/доступ» (на практике сервис отдаёт **404** с **`not_found`**; для клиента семантика — «ресурс по этому URL отсутствует»). **Не** пытаться различать **`ErrTaskNotFound`** vs **`ErrProjectNotFound`** по телу ответа.

---

## 403 (без дисамбигуации по пути)

Любой **403** на эндпоинтах задач — и **`/projects/{id}/tasks`**, и **`/tasks/{id}`**, и **`…/messages`** — маппится в **`TaskForbiddenException`**. Отдельного **`ProjectForbiddenException`** в **`api_exceptions.dart`** **нет** (**только** **`UnauthorizedException`** и **`ProjectNotFoundException`** среди общих) — **новый** общий тип **не** вводить; вся **403**-семантика задач — в **`TaskForbiddenException`**.

## Исключения фичи (**обязательный набор классов**)

По стилю **11.3** / **`conversation_exceptions.dart`** — **отдельные** типы (не один только **`TaskApiException`** по кодам):

| Класс | Когда |
|--------|--------|
| **`TaskCancelledException`** | **`p.isCancellation`** (`CancelToken`, **`DioExceptionType.cancel`**) |
| **`TaskNotFoundException`** | **404** на путях **`/tasks/...`** |
| **`TaskForbiddenException`** | **403** на **любых** путях задач (**включая** **`/projects/{id}/tasks`**, **`/tasks/{id}`**, **`…/messages`**) — см. § 403 |
| **`TaskConflictException`** | **409** (`stableErrorCode` часто **`conflict`** из `apierror.ErrConflict` в **`backend/pkg/apierror/apierror.go`**) |
| **`TaskUnprocessableException`** | **422** (`unprocessable` / **`ErrUnprocessable`**) |
| **`TaskApiException`** | прочие **4xx/5xx** с телом, **400**, **500**, а также **сеть без HTTP** (`isNetworkTransportError`) — с полями **`statusCode?`**, **`apiErrorCode?`** (= **`p.stableErrorCode`**), **`isNetworkTransportError`** по аналогии с **`ConversationApiException`** |

Общие: **`UnauthorizedException`**, **`ProjectNotFoundException`** — из **`api_exceptions.dart`**; в подклассы задач передавать **`apiErrorCode: p.stableErrorCode`** там, где поле предусмотрено (как у **`ConversationForbiddenException`**).

Стабильные строки **`error`** в JSON — константы в **`backend/pkg/apierror/apierror.go`** (**`forbidden`**, **`not_found`**, **`conflict`**, **`unprocessable`**, …); для точечной логики UI опираться на **`statusCode`** + опционально **`apiErrorCode`**, **не** на **`message`**.
