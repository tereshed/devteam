# 9.2 Индексатор кода: чанкинг файлов

**Sprint:** 9 — Vector DB & Knowledge Base
**Файл:** `backend/internal/indexer/code_indexer.go`
**Статус:** ✅ Завершено (Все архитектурные и security правки внесены)

---

## Описание

Задача заключается в реализации компонента `CodeIndexer`, который отвечает за разбиение исходного кода на смысловые фрагменты (чанки) для последующей индексации в векторной базе данных. Эффективный чанкинг критичен для качества поиска по коду: фрагменты должны быть достаточно большими, чтобы сохранять контекст, но достаточно малыми, чтобы поиск был точным.

Основная стратегия:
- **Для малых файлов:** индексация файла целиком (с учетом лимитов токенов).
- **Для больших файлов:** разбиение по функциям/методам (semantic chunking) через AST.

---

## Зависимости

**Требует реализации:**
- `vectordb.Client` (для сохранения чанков, поддержка **Batch Upsert**)
- `parser.CodeParser` (интерфейс для разбора структуры файлов через tree-sitter)
- `indexer.SyncStateRepository` (интерфейс для хранения хешей и стейта индексации)

**Библиотеки:**
- `github.com/smacker/go-tree-sitter` (использование CGO, требуется переиспользование парсеров через `sync.Pool`)
- `github.com/google/uuid`
- `github.com/pkoukk/tiktoken-go` (для точного подсчета токенов)

## Функциональные требования

### 1. Безопасность и Валидация (Security First)
- **Path Traversal Protection:** Строгая валидация `file_path`. Использовать `filepath.Rel(baseDir, targetPath)` и проверять, что путь не начинается с `../`.
- **Path Normalization:** Все пути в метаданных должны быть нормализованы (использовать `/` как разделитель) для консистентности поиска между разными ОС.
- **Secret Masking (Pre-processing):** Маскировка секретов (regex) должна применяться **ко всему тексту файла ДО чанкинга**, чтобы избежать утечек на границах чанков. Для файлов > 1MB рекомендуется потоковая обработка или оптимизированное чтение для минимизации спайков аллокаций.
- **Command Injection Protection:** Если для обхода файлов или вычисления хешей используются вызовы внешних команд (например, `git`), обязательно использовать разделитель `--` перед путями.
- **ReDoS Protection:** Использовать безопасные паттерны или таймауты для регулярных выражений.
- **OOM Prevention (Fast Filtering):** 
    - Проверка MIME-типа и эвристика минификации только на первых **512 байтах** (`io.LimitReader`).
    - **Hard Limit:** Игнорировать файлы размером более **5MB** (защита AST-парсера от OOM на гигантских сгенерированных файлах).
    - **Long Lines Check:** Игнорировать или обрезать файлы с аномально длинными строками (>10KB без переносов) для защиты от ReDoS и зависания парсера.
- **File Filtering (TOCTOU Safe):** Использовать `os.Lstat` для проверки символических ссылок. Игнорировать симлинки, пустые файлы (size == 0) и бинарники.

### 2. Стратегия чанкинга и Лимиты
- **Token-based Limits:** Решение о стратегии принимается на основе количества токенов (`tiktoken`). Максимальный размер чанка должен соответствовать `Context Window` модели эмбеддингов.
- **Small Files:** Если файл целиком укладывается в лимит токенов — индексировать одним чанком.
- **Large Files (Semantic Chunking):** Разбиение на основе AST (функции, методы, типы). 
    - **Max Recursion Depth:** Ограничить глубину рекурсивного разбиения (например, до 10 уровней). 
    - **Fallback:** При превышении глубины или ошибках парсинга — разбиение по фиксированному количеству токенов с перекрытием (overlap).
- **Processing Timeout:** Установить `context.WithTimeout` (например, 5-10 сек) на обработку одного файла.

### 3. Обогащение метаданных
Каждый чанк должен содержать:
- `content`: Текст кода (после маскировки секретов).
- `metadata`:
    - `file_path`: Очищенный и нормализованный относительный путь.
    - `language`: Язык программирования.
    - `start_line`, `end_line`: Диапазон строк.
    - `symbol_name`: Имя функции/метода.
    - `project_id`: ID проекта.
    - `content_hash`: SHA-256 для инкрементальных обновлений.

### 4. Производительность и Архитектура
- **Inversion of Control (SyncState):** Работа со стейтом идет через интерфейс `SyncStateRepository`.
- **Cleanup Logic:** Автоматическое удаление чанков из VectorDB для файлов, которые были удалены из репозитория.
- **Batch Processing:** Запись в `vectordb.Client` должна выполняться батчами.
    - **Batch Limits:** Учитывать не только количество (50-100), но и суммарный размер батча в байтах (например, не более 4MB).
- **Error Handling Strategy:**
    - **Rate Limited Logging:** Лимитировать логирование ошибок парсинга для предотвращения Disk Exhaustion (например, не более 100 ошибок на проект).
    - **TOCTOU & Race Conditions:** Ошибки `os.ErrNotExist` или `os.ErrPermission` при открытии файла после `Lstat` должны обрабатываться как нормальные краевые случаи (логировать и пропускать).
    - Ошибки инфраструктуры (БД, VectorDB) — **Fail Fast** через отмену контекста.
- **Resource Management:** 
    - `sync.Pool` для парсеров tree-sitter. 
    - **CGO Safety:** Обязательный вызов `parser.Reset()` перед возвратом парсера в пул.
    - `defer f.Close()`, Graceful Shutdown через `context.Context` и `sync.WaitGroup`.

## Критерии приёмки

- [ ] Индексатор работает через интерфейс `SyncStateRepository` и поддерживает Cleanup удаленных файлов.
- [ ] Реализован Hard Limit (5MB) на размер файла и Fast Filtering (512 байт).
- [ ] Запись в VectorDB выполняется батчами.
- [ ] Пути в метаданных нормализованы (разделитель `/`).
- [ ] Ошибки инфраструктуры вызывают Fail Fast, ошибки файлов — только логируются.
- [ ] Маскировка секретов защищена от ReDoS и применяется ко всему файлу до чанкинга.
- [ ] Unit-тесты покрывают: Hard Limits, нормализацию путей, батчинг и стратегию обработки ошибок.
