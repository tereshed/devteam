## Описание

Unit-тесты для `OrchestratorService` — ядра оркестратора, которое управляет полным циклом выполнения задачи: от анализа запроса пользователя до завершения пайплайна. Тесты покрывают линейный pipeline, логику ретраев, отмены, краевые случаи, безопасность и конкурентность.

> **Критично:** Мы пишем тесты не ради покрытия, а чтобы ядро системы не развалилось в проде. Пропуск дыр на этапе проектирования = инциденты в production.

**Pipeline (линейный MVP):**
```
User Message
    → Orchestrator (LLM: анализ запроса)
        → Planner (LLM: декомпозиция задачи)
            → Developer (Sandbox: Claude Code CLI)
                → Reviewer (LLM: ревью diff)
                    → [если changes_requested → Developer]
                    → Tester (Sandbox: запуск тестов)
                        → [если failed → Developer]
                        → Completed
```

## Требования к качеству тестов

### Table-Driven Tests — ОБЯЗАТЕЛЬНО

Для всех групп тестов (Error Handling, Branching, Pipeline Execution) **обязательно** использовать table-driven подход. Иначе 40+ тестов превратятся в копипаст с идентичной инициализацией моков.

**Структура хелперов:**
```go
// Фабрика моков — один раз, переиспользуется везде
func newTestOrchestratorHarness(t *testing.T, cfg *OrchestratorConfig) (*OrchestratorService, *mockAgents)

// Табличные тесты для Error Handling
var errorHandlingTests = []struct {
    name           string
    executorError  error
    expectedStatus TaskStatus
    expectErr      bool
}{
    {"timeout", context.DeadlineExceeded, StatusFailed, true},
    {"cancelled", context.Canceled, StatusFailed, true},
    {" executor error", errors.New("模型API错误"), StatusFailed, true},
}
```

### Проверка на утечки горутин — через t.Cleanup()

Для всех асинхронных тестов (Cancel, Pause, Timeout, Race Conditions) **обязательно** использовать `goleak` (uber-go/goleak) через `t.Cleanup()` внутри каждого теста. Если executor завис и проигнорировал контекст отмены, горутина оркестратора потечёт.

```go
import "go.uber.org/goleak"

func TestCancel_DuringDeveloper(t *testing.T) {
    t.Cleanup(func() { goleak.VerifyNone(t) })
    // ... тело теста
}
```

**Не допускается** выносить проверку утечек в отдельный тест — утечка может произойти в любом асинхронном тесте, а отдельный тест её не поймает (или упадёт ложно).

## Функциональные требования

### 1. Полный pipeline (5 шагов)

- **Orchestrator → Planner → Developer → Reviewer → Tester → Completed**
- Проверка корректной последовательности вызовов executor'ов
- Проверка передачи результата каждого шага как входа следующего
- Проверка финального статуса `completed`

### 2. Ветвление: `changes_requested` → Developer

- Reviewer возвращает `changes_requested`
- Orchestrator отправляет задачу обратно Developer (повторный вызов)
- После исправления → снова Reviewer
- Цикл продолжается до `approved` или лимита ретраев

### 3. Ветвление: `failed` → Developer

- Tester возвращает `failed`
- Orchestrator отправляет задачу обратно Developer
- После исправления → снова Tester
- Цикл продолжается до `passed` или лимита ретраев

### 4. Ретраи (Retry Logic)

- Лимит ретраев на шаг (настраиваемый, дефолт 3)
- Счётчик попыток увеличивается при `changes_requested` или `failed`
- При превышении лимита → пайплайн завершается с `failed`
- Проверка сброса счётчика при успешном прохождении шага

### 5. Отмена (Cancellation)

- Пользователь отменяет задачу через `cancel`
- Проверка: контейнер Developer/Tester останавливается
- Проверка: пайплайн завершается с `cancelled`
- Проверка: no further agent calls после cancel
- **Критично:** Проверка отсутствия утекших горутин (goleak)

### 6. Пауза (Pause)

- Пользователь приостанавливает задачу
- Проверка: текущий executor останавливается
- Проверка: задача переходит в `paused`
- Resume восстанавливает выполнение

### 7. Обработка ошибок executor'ов

- Executor вернул ошибку → пайплайн завершается с `failed`
- Timeout executor'а → пайплайн завершается с `failed`
- **Критично:** Error wrapping сохраняет контекст (`fmt.Errorf("planner failed: %w", err)`)
- **Критично:** Проверка отсутствия утекших горутин (goleak)

### 8. Валидация графа до начала выполнения

- Если `is_active: false` у базового агента → явная ошибка до старта
- Если отсутствует обязательный агент → явная ошибка
- Проверка: пайплайн не запускается с невалидным конфигом
