# 5.2 Entrypoint-скрипт sandbox: `clone → branch → agent → diff → result`

**Sprint:** 5 — Sandbox Runner (Backend, Docker)
**Файл:** `deployment/sandbox/claude/entrypoint.sh`
**Образ:** используется в `devteam/sandbox-claude` (задача **5.1**)
**Статус:** ✅ Выполнена

---

## Описание

Реализовать **точку входа контейнера** (`ENTRYPOINT`) для sandbox-образа Claude: shell-скрипт, который последовательно выполняет жизненный цикл задачи **без участия Go-бэкенда** — только через **переменные окружения** и **фиксированные пути артефактов** в `/workspace`, чтобы оркестратор (**5.5–5.7**) мог стабильно забрать результат (`docker.CopyFromContainer`, парсинг JSON/diff).

Этапы (логическая цепочка из README):

1. **clone** — клонирование репозитория в согласованный каталог (например `/workspace/repo`).
2. **branch** — checkout базовой ветки и создание рабочей ветки для задачи.
3. **agent** — запуск **Claude Code CLI** с инструкцией и контекстом; вывод — в предсказуемые файлы логов/результата.
4. **diff** — после **`git add -A`** сбор **unified diff** индекса относительно базы (ветка/remote), чтобы бэкенд не вызывал `git` на хосте и не терял **untracked** артефакты агента.
5. **result** — запись **итоговых артефактов** в заранее оговорённые пути (например `result.json`, `status.json`, `full.diff`, сокращённые логи), с **детерминированными кодами выхода** при фатальных ошибках.

Скрипт **не** содержит бизнес-логики домена DevTeam (проекты, задачи в БД) — только «как выполнить агента в изоляции и что положить на диск». Как **PID 1** он обязан корректно реагировать на **SIGTERM** (**`docker stop`** / отмена задачи пользователем) и не оставлять оркестратор без **`status.json`**.

## Требования к `entrypoint.sh`

### Валидация входа (Early Return) — обязательно

**Сразу после shebang и минимальной инициализации**, до `git clone` и любых операций с внешним вводом:

- Проверить **все критичные** переменные: непустые строки, отсутствие «только пробелы» (по политике команды).
- **Быстрый фейл по правам на рабочий каталог** (правило 3 code review: не полагаться на «системную» ошибку от `rm`/`git clone` при `EACCES`):

  ```bash
  if [ ! -w "/workspace" ]; then
    echo "Error: /workspace is not writable" >&2
    exit 1
  fi
  ```

  Так оркестратор сразу получает **контролируемое** сообщение и код, если volume примонтирован от `root` с неверными правами, вместо часов дебага по обрывочным errno в середине пайплайна.

- Пример обязательного стиля для env (имена переменных — по финальному контракту **5.4**):

  ```bash
  if [ -z "${REPO_URL:-}" ]; then
    echo "entrypoint: REPO_URL is required" >&2
    exit 1
  fi
  ```

- Если обязательная переменная пуста — **понятное сообщение в stderr** и **ненулевой код выхода**; итог всё равно фиксируется через **`finalize` + `trap EXIT`** (см. ниже), чтобы **5.7** не получал полностью пустой контейнер.

Без этого сценарий `git clone ""` даёт нечитаемую ошибку от git и усложняет отладку оркестратора.

### Общее

- **`set -euo pipefail`** — базовая строгость; ошибки команд не игнорировать без явной политики.
- Рабочий каталог: согласован с **5.1** (`/workspace`); клон — в подкаталог с **фиксированным литеральным** именем (не из env), чтобы не было path injection.
- Глобально **`set -x` запрещён** (см. раздел «Безопасность»): при отладке трассировку включать **локально** и **никогда** во время вывода команд с `REPO_URL` / токенами.
- Скрипт в типичной конфигурации Docker — **PID 1** в контейнере: поведение при **`docker stop`** (SIGTERM → ожидание → SIGKILL) — см. раздел **«Сигналы: SIGTERM и отмена задачи»**.

### Идемпотентная подготовка каталога перед `git clone` — обязательно

Оркестратор может смонтировать volume в **`/workspace/repo`** (кэш клона). Тогда каталог **уже существует** (пустой, с частичными данными или «мусором» с прошлого запуска), и **`git clone … /workspace/repo`** завершится ошибкой вида *destination path already exists*.

**Требование:** перед `git clone` выполнить **идемпотентный** шаг подготовки целевого пути, согласованный с политикой данных:

- например **`rm -rf /workspace/repo`** (или эквивалент очистки **только** фиксированного литерального пути клона), **либо**
- явная проверка: если каталог непуст и не является валидным git work tree — удалить/очистить по правилам команды.

Цель: повторный запуск и volume-кэш **не ломают** сценарий без ручного вмешательства.

### Единый финализатор и `trap EXIT` (DRY) — обязательно

Дублировать запись `status.json` в «успешной» ветке и в обработчике ошибок **нельзя** (нарушение DRY, расхождение полей).

**Обязательный паттерн:**

- Одна функция **`finalize`** (или эквивалентное имя), которая по **текущему состоянию** (сохранённый код выхода, фаза пайплайна, флаг отмены — см. SIGTERM) **идемпотентно** записывает `status.json` / `result.json`.
- **`trap 'finalize' EXIT`** (или эквивалент с передачей кода — см. ниже) — обработчик на **`EXIT`**: он вызывается **при любом** нормальном завершении shell-скрипта: код **0**, ненулевой код из‑за **`set -e`**, явный `exit N`. В момент входа в обработчик **`EXIT`** в bash переменная **`$?`** обычно уже соответствует **коду завершения сценария**; первая строка **`finalize`** может сохранить его в локальную переменную. Альтернатива: **`trap 'finalize "$?"' EXIT`** — если в вашей версии bash **`$?`** в теле `trap` вычисляется **при срабатывании** (проверить в образе **5.1**); при сомнениях использовать явное сохранение: **`trap 'LAST_EXIT=$?; finalize' EXIT`** и читать **`LAST_EXIT`** внутри **`finalize`**.

Требование: **одна точка записи** JSON при завершении процесса (кроме крайних случаев, когда запись на диск невозможна — OOM ядром; тогда — код контейнера и логи).

**Не использовать** отдельный **`trap … ERR`** как основной механизм записи успеха: при успехе `ERR` не срабатывает; дублирование «успех в конце скрипта + ERR при ошибке» запрещено.

### Сигналы: SIGTERM и отмена задачи (PID 1) — обязательно

При **`docker stop`** Docker посылает контейнеру **SIGTERM** процессу **PID 1** (нашему `entrypoint.sh`). Если bash **не обрабатывает** SIGTERM, скрипт будет завершён через **SIGKILL** по таймауту (часто ~10 с), без корректной записи метаданных.

**Требование:**

- **`trap 'handle_term' SIGTERM`** — обработчик, который:
  - выставляет флаг отмены (например **cancelled** / **user_cancelled**);
  - **направляет завершение дочернему процессу** агента (`kill -TERM` к известному **PID** `claude`, при необходимости — группе процессов; конкретика в реализации);
  - **обязательно ждёт реального завершения** дочернего процесса: **`wait "$CLAUDE_PID"`** (или эквивалент) после сигнала. **Запрещено** сразу после `kill` вызывать `exit` и надеяться, что контейнер «сам дождётся» — при немедленном выходе PID 1 Docker может уничтожить контейнер, пока **claude** ещё сбрасывает буферы, закрывает файлы и завершает сессию → **битые/усечённые логи**, частично записанный стейт, гонки при чтении артефактов с хоста (**5.7**).
  - при необходимости — **верхняя граница ожидания** (например цикл с таймаутом и затем `KILL` дочернего процесса), если процесс не завершается после SIGTERM, чтобы не висеть дольше политики **5.5**; после этого всё равно **`wait`** для освобождения зомби.
  - только **после** завершения дочернего процесса (или контролируемого таймаута + `KILL` + `wait`) — выставить код/флаг отмены и завершить скрипт так, чтобы сработал **`trap EXIT`** → **`finalize`**;
  - гарантировать, что **`status.json`** содержит статус **cancelled** (или эквивалентное поле по контракту **5.7**), согласованный **`exit_code`**, а не «пустой» контейнер.

**Избегать двойной записи:** `finalize` идемпотентен; порядок: дождаться агента → записать метаданные через **`finalize`** при `EXIT`.

После обработки SIGTERM скрипт завершается через **`exit`**, когда дочерний процесс уже не работает — тогда **`trap EXIT`** → **`finalize`** отражает реальное состояние файлов на диске.

Интеграционный тест (раздел «Тестирование») **желательно** включает сценарий отправки SIGTERM и проверку полей отмены в `status.json`.

### Гарантированная запись `status.json` при ошибках (`set -e`)

При **`set -e`** падение команды завершает скрипт и приводит к **`EXIT`** — отсюда единый **`finalize`** через **`trap EXIT`**. Отдельный «ручной» `write_status` в десяти местах не нужен.

### Git: разделитель `--` и защита от command/flag injection — обязательно

Значения из env (`REPO_URL`, `BRANCH_NAME`, `BASE_REF`, имена remote и т.д.) считаются **внешним вводом**. Строки вида `BRANCH_NAME='-q'`, `--orphan`, `--upload-pack=…` иначе интерпретируются git как флаги.

**Обязательно:** во всех вызовах `git`, где после опций следуют такие значения, использовать **конец опций** — **`--`** — сразу перед аргументом из переменной:

| Операция | Правильный шаблон (пример) |
|----------|----------------------------|
| Клон | `git clone <согласованные опции клонирования> -- "$REPO_URL" /workspace/repo` |
| Создание ветки | `git checkout -b -- "$BRANCH_NAME"` (после checkout базы — по сценарию) |
| Checkout по имени | `git checkout -- "$BRANCH_NAME"` |
| Fetch ref | `git fetch origin -- "$BASE_REF"` (или согласованный remote/ref; см. shallow ниже) |

Аналогично: любые пути или ref из env передавать **после `--`**, если команда это допускает. Литеральные флаги (`--depth=1`) задаются **константами в скрипте**, не склеиваются из env.

**Запрещено:** `eval`, подстановка целой команды из env, «динамический» `git $GIT_ARGS` из невалидированной строки.

### Git: shallow clone и корректный `git diff` — обязательно продумать

Если используется **`git clone --depth=…`** (или иной shallow-сценарий), локальный репозиторий может **не содержать** объекта, соответствующего `origin/main` (или выбранному `BASE_REF`), в объёме, достаточном для сравнения с текущим `HEAD` после коммитов агента — типичный сбой: `git diff` / сравнение с **неполной** историей remote.

**Требование к спеке реализации:**

- Явно зафиксировать: **как** собирается база для diff (имя ref: ветка, tag, SHA).
- Если остаётся shallow clone — **перед** финальным `git diff` (и при необходимости `git log`) выполнить **`git fetch`** с параметрами, достаточными для появления нужного ref у `origin`, например:

  `git fetch origin --depth=<N> -- "$BASE_REF"`

  где `<N>` согласовать (один коммит часто недостаточно; возможно несколько fetch или unshallow — trade-off размера сети vs корректности).

- Альтернатива: **не** использовать shallow для сценария, где нужен стабильный diff относительно удалённой базы — тогда документировать это и согласовать с **5.5** (время clone).

Цель: **`full.diff` и метаданные** не должны «молча» быть пустыми или ошибочными из‑за недостающих объектов при depth=1.

### Claude Code CLI: ARG_MAX, файлы вместо argv, неинтерактивный режим — обязательно

**Лимит argv (ARG_MAX):** `TASK_INSTRUCTION` и особенно **`TASK_CONTEXT`** (контекст из векторной БД) могут занимать **десятки и сотни килобайт**. Передача через командную строку (`claude -p "$TASK_INSTRUCTION"` и аналоги) приводит к ошибке ОС **`Argument list too long`**.

**Строго запрещено:** передавать **`TASK_INSTRUCTION`** и **`TASK_CONTEXT`** **напрямую** в аргументы командной строки вызова `claude` (и любых обёрток), если объём может превысить безопасный предел или вообще для контекста — по политике ниже.

**Обязательно:**

- Сохранять содержимое во **временные/фиксированные файлы** под **`/workspace`** с **литеральными** путями (например `prompt.txt`, `context.txt` — имена зафиксировать в контракте **5.4/5.7**), например безопасной записью из shell (с учётом `printf`/heredoc и размера) — конкретный механизм согласовать с ограничениями shell.
- Передавать агенту **пути к файлам** или использовать **режим чтения из файла**, **поддерживаемый конкретной версией** `@anthropic-ai/claude-code` (изучить `claude --help` / документацию для версии из **5.1** и зафиксировать в комментарии к скрипту).

**Неинтерактивность (deadlock):** Claude Code рассчитан на терминал с человеком. В Docker без TTY агент может **запросить подтверждение** («Run this? [Y/n]») и **зависнуть** до таймаута оркестратора, расходуя ресурсы.

**Обязательно:** вызов `claude` должен быть **гарантированно неинтерактивным**: изучить и **зафиксировать в скрипте** флаги автоматического согласия / отключения интерактивных промптов / non-interactive режим для версии CLI из **5.1** (обновить список при bump версии в **5.1**). При отсутствии подходящих флагов — явная политика в задаче **5.2** + согласование с командой (обёртка `script`, `expect` — только как крайняя мера с обоснованием).

Прочее:

- Вызов с флагами, совместимыми с установленной версией из **5.1**.
- Запрет **`eval`** сырого текста инструкции; путь к файлу — литерал или проверенная строка.
- Вывод агента — в файлы по контракту; код завершения CLI учитывать в состоянии для **`finalize`** / `status.json`.
- Дочерний PID агента сохранять для обработки **SIGTERM** и для **`wait`** (см. выше).

### Git identity для коммитов агента — обязательно до запуска агента

В «чистом» образе (**5.1**, `node:*-slim`) **не** заданы **`user.name`** / **`user.email`**. Агенты (Claude Code, Aider и др.) могут делать **промежуточные `git commit`** для сохранения контекста. Первая же попытка коммита завершится фатально: *Please tell me who you are* — задача падает без понятного восстановления изнутри агента.

**Обязательно до первого запуска агента** (после `cd` в репозиторий, в контексте пользователя **`sandbox`**):

```text
git config --global user.name "DevTeam Agent"
git config --global user.email "agent@devteam.local"
```

(Строки-константы можно согласовать с продуктом; главное — **детерминированные** значения, не из env без валидации.) Так коммиты агента не ломают пайплайн из‑за отсутствия identity.

### Diff и результат: untracked-файлы и индекс — обязательно

**Проблема:** агент часто создаёт **новые файлы**, но **не** делает `git add`. Обычный **`git diff`** (и сравнение с **`origin/main`**) по умолчанию **не включает untracked** файлы. Итог: сотни строк в новом `service.go`, а **`full.diff` пустой** — бэкенд забирает пустой результат, токены сгорели впустую.

**Обязательно перед сбором diff:**

1. В каталоге репозитория выполнить **`git add -A`** (или эквивалент, гарантирующий включение новых и изменённых файлов в индекс — политика команды должна быть одна).
2. **Только после этого** формировать unified diff относительно согласованной базы, сравнивая **индекс (staged)** с базовым ref — например **`git diff --staged`** / **`git diff --cached`** против **`"$BASE_REF"`** или **`origin/main`** (в современном git **`--staged`** ≡ **`--cached`**). Точную команду зафиксировать в скрипте и **проверить** на репозитории с новыми файлами без `git add` со стороны агента; согласовать с разделом про **shallow fetch** (нужные объекты у `origin` — **до** diff).

Цель: **`full.diff`** и производные артефакты **гарантированно** отражают все изменения, оставленные агентом на диске, включая новые файлы.

- **`git log`** / статистика — по согласованному сценарию (после индексации при необходимости).
- Финальный JSON/status: **только** через единый **`finalize`** и **`trap EXIT`** (и обновление состояния перед `exit` при отмене), чтобы **5.7** не зависел от «счастливого» пути без ошибок.

### Права и `WORKDIR`

- Явная проверка **`[ ! -w "/workspace" ]`** — в разделе **«Валидация входа»** выше; это **первичный** контролируемый фейл.
- Дополнительно скрипт выполняется от пользователя **`sandbox`** (**5.1**). Политика **5.5** по UID/GID и volume — задокументировать в согласовании с **5.5**; при ошибках записи после старта — по возможности отражать в **`status.json`** через **`finalize`**, а не только сырой errno в логах.

## Безопасность и эксплуатация

### Command / flag injection

- Соблюдать раздел **«Git: разделитель `--`»** выше — это **обязательное** требование приёмки, не рекомендация.

### ARG_MAX и логирование больших тел

- В логи **не** выводить полное содержимое `TASK_CONTEXT` без необходимости (объём, утечка данных). Диагностика — усечённо или хеш/размер файла.

### Утечка секретов через трассировку и логи

- **Запрещено** включать **`set -x` глобально** в `entrypoint.sh`. С `set -euo pipefail` часто «для дебага» добавляют `set -x` — тогда в stderr попадают **полные** строки команд, включая  
  `git clone 'https://x-access-token:SECRET@github.com/...'`  
  и токены в URL-encoded виде (см. правила code review: маскирование и учёт `+` vs `%20` при логировании).

- Если нужна отладка: локальные блоки `set +x` перед чувствительными командами, либо отдельная сборка образа с отладочным слоем **не в production CI**.

- Любой **намеренный** вывод URL в лог (диагностика) — только **после маскирования** userinfo (`https://***@host/...`) и без повторной утечки в дочерние процессы, если это возможно.

### Прочее

- Секреты не дублировать в `result.json` / `status.json` без необходимости.
- **Сеть:** политика исходящих соединений задаётся хостом/Docker.

## Тестирование

Ручной `docker run` **не является** достаточной проверкой качества.

**Обязательно:** интеграционный тест на **Go** в пакете `backend/internal/sandbox` (тег `//go:build integration` — по правилам репозитория), который:

- поднимает образ `devteam/sandbox-claude` (после **5.12** — через собранный тег; локально — сборка из `deployment/sandbox/claude`) — **Testcontainers** или **Docker API** напрямую;
- передаёт **контролируемые** env (в т.ч. заведомо невалидные для проверки валидации и `status.json`);
- проверяет, что в контейнере по фиксированным путям появляются ожидаемые артефакты: минимум **`status.json`** (или `result.json`) с ожидаемыми полями при ошибке, и при позитивном сценарии — согласованность **`full.diff`** / кода выхода;
- **желательно:** сценарий **SIGTERM** (аналог `docker stop`) и проверка, что **`status.json`** отражает **отмену** (cancelled) и согласованный код, а не «пустой» результат.

Объём может быть объединён с задачей **5.14** (интеграционный тест реального контейнера), но **критерий приёмки 5.2** требует, чтобы сценарий **entrypoint** был явно покрыт, а не «только ручной прогон».

Дополнительно разработчик может использовать ручную сборку до **5.12** (см. ниже).

### Проверка вручную (дополнительно)

До **5.12** — сборка образа вручную с контекстом `deployment/sandbox/claude`.

```bash
docker build -t devteam/sandbox-claude -f deployment/sandbox/claude/Dockerfile deployment/sandbox/claude
docker run --rm \
  -e REPO_URL="https://github.com/octocat/Hello-World.git" \
  -e BRANCH_NAME="sandbox-test" \
  -e TASK_INSTRUCTION="Say hello in README" \
  ... \
  devteam/sandbox-claude
```

После **5.12** — использовать `make sandbox-build` (или принятое имя цели).
