## Описание

Создать сервис `ConversationService`, который является высокоуровневым слоем бизнес-логики для работы с чатами (Conversations). Сервис должен объединять работу с репозиториями чатов и сообщений, а также интегрироваться с `OrchestratorService` для обработки входящих сообщений от пользователя и генерации ответов агентов.

---

## Интерфейс

```go
type ConversationService interface {
    // CreateConversation создает новый чат для указанного проекта.
    // Обязательно проверяет права доступа userID к projectID.
    CreateConversation(ctx context.Context, userID, projectID uuid.UUID, title string) (*models.Conversation, error)

    // GetConversation возвращает чат по ID.
    // Обязательно проверяет права доступа userID к чату.
    GetConversation(ctx context.Context, userID, id uuid.UUID) (*models.Conversation, error)

    // ListConversations возвращает список чатов проекта с пагинацией.
    // Обязательно проверяет права доступа userID к проекту.
    ListConversations(ctx context.Context, userID, projectID uuid.UUID, limit, offset int) ([]*models.Conversation, int64, error)

    // SendMessage отправляет сообщение пользователя в чат и запускает процесс оркестрации.
    // Обязательно проверяет права доступа userID к чату.
    SendMessage(ctx context.Context, userID, conversationID uuid.UUID, content string) (*models.ConversationMessage, error)

    // GetHistory возвращает историю сообщений чата.
    // Обязательно проверяет права доступа userID к чату.
    GetHistory(ctx context.Context, userID, conversationID uuid.UUID, limit, offset int) ([]*models.ConversationMessage, int64, error)

    // DeleteConversation удаляет чат и все его сообщения.
    // Обязательно проверяет права доступа userID к чату.
    DeleteConversation(ctx context.Context, userID, id uuid.UUID) error
}
```

## Безопасность и Валидация

### 1. Защита от IDOR (DRY)
- Все методы принимают `userID`. Проверка прав доступа должна быть вынесена в **приватные хелперы** (например, `checkProjectAccess`, `checkConversationAccess`), чтобы избежать дублирования кода и ошибок.
- Если доступа нет, возвращать `ErrForbidden` или `ErrNotFound` (согласно политике безопасности).

### 2. Безопасность логов (PII Protection)
- **Критически важно:** При логировании ошибок или событий внутри сервиса **запрещено** выводить полный текст сообщения пользователя (`content`). 
- Логировать только метаданные: `userID`, `conversationID`, `projectID`, длину сообщения (`len(content)`), время выполнения.

### 3. Валидация входных данных
- **SendMessage:**
    - `content`: не пустой, max **4096 символов**.
    - **Идемпотентность (MVP):** Если последнее сообщение в чате от того же пользователя имеет идентичный контент и отправлено менее 1 секунды назад — игнорировать (защита от двойного клика).
- **CreateConversation:**
    - `title`: не пустой, max **255 символов**.
- **Пагинация:**
    - `limit`: диапазон `[1, 100]`, дефолт `20`.
    - `offset`: не может быть отрицательным.

## Архитектура и Жизненный цикл

### Асинхронность и Graceful Shutdown
- Вызов `OrchestratorService.ProcessMessage` выполняется в **защищенной горутине**.
- **Panic Recovery:** Внутри горутины обязателен `defer recover()`, чтобы паника в оркестраторе не обрушила весь сервис. Ошибка паники должна быть залогирована (без `content`).
- **Жизненный цикл:** Для корректного завершения работы (Graceful Shutdown) горутины должны использовать `context.Context` или системный `sync.WaitGroup`, чтобы сервис дождался завершения активных процессов оркестрации при получении SIGTERM.

### Транзакции
- **DeleteConversation:** Удаление чата и сообщений должно быть атомарным. Рекомендуется использовать `ON DELETE CASCADE` в БД. Если это невозможно, использовать транзакцию через репозиторий.
